## 1. Document Purpose

This document delivers the **`Counter` entity** end-to-end: the API endpoints that operate on it, the admin UI pages that allow super-admins to configure it, the Zod schemas that validate it, the TypeScript types that describe it, and the business rules that govern it. The counter listing page also surfaces the current operational status as a read-only indicator (the toggle to change the status is Phase 4.2.1 — this document only reads and displays the state).

When this document is complete, a super-admin can:

- Navigate to `/counters` and see all counters in a data table with their operational status.
- Click "Create Counter" and create a new counter with a unique number, a name, an optional display label, and a description.
- Edit any counter (change name, display label, description, active/inactive status).
- Deactivate a counter (soft-delete) — it disappears from the active list but remains in the database for historical ticket references.

A non-admin cannot access these pages or APIs.

This document can be developed in **parallel** with 2.1.1 (Service Entity & Management API). The two documents touch different files and different database tables.

### Step 2: Create `src/schemas/counter.schema.ts` — The Zod Schemas

**Goal:** Validate every counter create/update payload.

**Files affected:** `src/schemas/counter.schema.ts` (new file).

**Specific instructions:**

Create `src/schemas/counter.schema.ts` containing:

- **`createCounterSchema`**:
  - `name`: required string, 1-100 characters, trimmed.
  - `number`: required positive integer, min 1, max 9999 (the document specifies the upper bound; the master plan allows any positive integer — the agent may relax this).
  - `description`: optional string, max 500 characters.
  - `displayLabel`: optional string, max 100 characters. If null or absent, the API defaults `displayLabel` to `name` on the read response.
  - `isActive`: optional boolean, defaults to `true`.
- **`updateCounterSchema`**:
  - All fields optional (PATCH is partial).
  - At least one field must be present (use Zod's `.refine`).
- **`listCountersQuerySchema`**:
  - `page`: optional integer, default 1, min 1.
  - `limit`: optional integer, default 20, min 1, max 100.
  - `search`: optional string, max 100 characters (matches against `name`, `displayLabel`, `description`).
  - `isActive`: optional enum string (`'true'`, `'false'`, or absent).
- Export inferred TypeScript types: `CreateCounterInput`, `UpdateCounterInput`, `ListCountersQuery`.

### Step 3: Create `src/types/counter.types.ts` — The API Types

**Goal:** Define the response shapes for the counter API endpoints.

**Files affected:** `src/types/counter.types.ts` (new file).

**Specific instructions:**

Create `src/types/counter.types.ts` containing:

- **`CounterListItem`**: the shape returned by `GET /api/counters`. Fields: `id`, `name`, `number`, `description`, `displayLabel`, `isActive`, `assignedServicesCount`, `operationalStatus`, `createdAt`, `updatedAt`.
  - `assignedServicesCount` is an integer count of services assigned to this counter (computed via Prisma `_count`).
  - `operationalStatus` is a string from a closed enum: `'OPEN' | 'CLOSED' | 'OFFLINE' | 'NO_OFFICER_ON_DUTY'`. The agent must compute this by joining through `CounterOfficer` for any officer where `isOnDuty = true`. If multiple officers are on duty, any single status is acceptable (the document specifies: pick the status of any on-duty officer; if no officer is on duty, the status is `'NO_OFFICER_ON_DUTY'`).
- **`CounterDetail`**: the shape returned by `GET /api/counters/[counterId]`. Includes everything in `CounterListItem` plus `services: { id, name, code, ticketPrefix, isActive }[]` (the assigned services).
- **`CreateCounterResponse`**: returns the created `CounterListItem`.
- **`CounterListMeta`**: pagination metadata.

### Step 4: Create `src/app/api/counters/route.ts` — List and Create

**Goal:** Provide the listing endpoint (with operational status) and the create endpoint.

**Files affected:** `src/app/api/counters/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`GET` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_COUNTER_READ)`.
  - Parses `page`, `limit`, `search`, `isActive` from query parameters using `listCountersQuerySchema`.
  - Builds a Prisma `where` clause:
    - If `search` is present: `OR: [{ name: { contains: search } }, { displayLabel: { contains: search } }, { description: { contains: search } }]`.
    - If `isActive` is present: `{ isActive }`.
  - Executes Prisma `findMany` with the following `include`:
    - `services: { include: { service: { select: { id, name, isActive } } } }` (to count assigned services).
    - `officers: { where: { isOnDuty: true }, select: { currentStatus: true } }` (to determine operational status).
  - Ordered by `number asc` (the display board order).
  - Maps each result to `CounterListItem`. For operational status:
    - If at least one officer has `currentStatus: 'OPEN'`, status is `'OPEN'`.
    - Else if at least one officer has `currentStatus: 'CLOSED'`, status is `'CLOSED'`.
    - Else if at least one officer has `currentStatus: 'OFFLINE'`, status is `'OFFLINE'`.
    - Else, status is `'NO_OFFICER_ON_DUTY'`.
  - Counts total for pagination.
  - Returns `{ success: true, data: <mapped CounterListItem[]>, meta }`.
- **`POST` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_COUNTER_CREATE)` (or `PERMISSION_COUNTER_MANAGE`).
  - Parses and validates the body with `createCounterSchema`.
  - Checks for existing counter with the same `number`. If exists, returns `CONFLICT` (HTTP 409).
  - Creates the counter with `db.counter.create`. If `displayLabel` is absent in the input, do NOT set it (leave it null); the API applies a default at read time only (the database stores null, the API returns `displayLabel ?? name`).
  - On success, calls `writeAuditLog({ action: 'COUNTER_CREATED', actorId, targetUserId: counter.id, description: \`Created counter "\${counter.name}" (#\${counter.number}).\`, metadata: { name, number, isActive } })`.
  - Returns `{ success: true, data: <CounterListItem> }` with HTTP 201.
- All error responses use the master plan envelope.

### Step 5: Create `src/app/api/counters/[counterId]/route.ts` — Get, Update, Delete

**Goal:** Provide per-counter endpoints.

**Files affected:** `src/app/api/counters/[counterId]/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`GET` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_COUNTER_READ)`.
  - Reads `counterId` from route params.
  - Queries the counter with `include`:
    - `services: { include: { service: { select: { id, name, code, ticketPrefix, isActive } } } }`.
    - `officers: { where: { isOnDuty: true }, select: { currentStatus: true, user: { select: { id, name } } } }`.
  - If not found, returns `NOT_FOUND` (HTTP 404).
  - Maps to `CounterDetail` (with `services` flattened to the service array).
  - Returns `{ success: true, data: <CounterDetail> }`.
- **`PATCH` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_COUNTER_UPDATE)` (or `PERMISSION_COUNTER_MANAGE`).
  - Captures the "before" state.
  - Validates body with `updateCounterSchema`.
  - If `number` is being changed, checks for conflicts. If conflict, returns `CONFLICT`.
  - Updates the counter.
  - On success, calls `writeAuditLog({ action: 'COUNTER_UPDATED', actorId, targetUserId: counterId, description: \`Updated counter "\${counter.name}".\`, metadata: { before, after } })`. If status changed to inactive, use `'COUNTER_DEACTIVATED'` instead.
  - Returns `{ success: true, data: <CounterListItem> }`.
- **`DELETE` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_COUNTER_DELETE)` (or `PERMISSION_COUNTER_MANAGE`).
  - Soft-delete: sets `isActive = false`. **Note:** the agent should verify (via a Prisma check) that no `Ticket` records reference this counter. If any do, the soft-delete is still allowed (since soft-delete only flips the flag), but a console warning should be logged.
  - On success, calls `writeAuditLog({ action: 'COUNTER_DEACTIVATED', actorId, targetUserId: counterId, description: \`Deactivated counter "\${counter.name}".\`, metadata: { name, number } })`.
  - Returns `{ success: true, data: { message: 'Counter deactivated.' } }`.
- All error responses use the master plan envelope.

## 8. Common Pitfalls

### 8.1 Not Handling Multiple Officers on Duty

If multiple `CounterOfficer` records have `isOnDuty = true` for the same counter (the schema allows this), the operational status logic must handle all of them. The document specifies the priority order: OPEN > CLOSED > OFFLINE > NO_OFFICER_ON_DUTY. An agent that picks only the first officer may report the wrong status.

### 8.2 Forgetting the Operational Status Display

The counter listing is more useful to admins when it shows whether a counter is currently open, closed, or offline. Omitting this column means the admin has to click into each counter to see its status.

### 8.3 Confusing `name` and `displayLabel`

`name` is the administrative name (used in internal lists, dropdowns, etc.). `displayLabel` is what the display board shows (e.g., "Counter 2" → "Information Desk"). The form must make this distinction clear. Forgetting to default `displayLabel` to `name` on read means the display shows nothing when `displayLabel` is null.

### 8.4 Hard-Deleting Counters with Ticket History

The `DELETE` endpoint must be a soft-delete. A counter that has had tickets served at it must never be hard-deleted — the tickets would lose their `counterId` reference.

### 8.5 Allowing `number` to Be Zero or Negative

The counter number is shown on the display board. Zero or negative numbers would be confusing. The Zod schema must enforce `min: 1`.

### 8.6 Not Returning `assignedServicesCount`

The listing should show how many services are assigned to each counter. Forgetting to include the count (via Prisma's `_count`) means admins have to click into each counter to see this.

### 9.3 Verification Tip

The fastest way to verify the counter entity is to create two counters with different numbers, verify the listing shows them in numeric order with the operational status as "No officer on duty", then attempt to create a third with a duplicate number and verify the conflict error.
