## 1. Sub-Phase Purpose

Sub-Phase 2.1 establishes the **structural backbone** of the queue system. It defines the two foundational entities of the domain — **Services** (the categories of work the system handles, such as "General Inquiry", "Medical", "Finance") and **Counters** (the physical or virtual points of service where officers serve customers) — and the **many-to-many assignment** between them (a counter can handle multiple services, and a service can be served at multiple counters).

This sub-phase does not issue tickets. No customer-facing UI is built here. The work is purely **configuration**: defining what services exist, what counters exist, and which counters handle which services. Every later sub-phase in Phase 2 (ticket issuance in 2.2, ticket lifecycle in 2.3) is impossible without these two entities and their relationship in place.

The three problems this sub-phase solves, in order:

1. **Defining services** — The `Service` entity, its CRUD API, its admin UI pages, and the business rules that govern it (unique code, unique ticket prefix, active/inactive toggling, atomic ticket-number sequencing). (Document 2.1.1.)
2. **Defining counters** — The `Counter` entity, its CRUD API, its admin UI pages, and the business rules that govern it (unique counter number, active/inactive toggling, name vs display-label distinction). (Document 2.1.2.)
3. **Defining the many-to-many assignment** — The `CounterService` join table, the assignment API, the "at least one service per counter" invariant, the cascade behaviour on deactivation, and the assignment UI integrated with both the counter detail and service detail pages. (Document 2.1.3.)

When this sub-phase is complete, a super-admin can configure the queue system's structural entities end-to-end through the admin UI, and every later sub-phase can read the configuration from a fully-populated, validated, audited database.

---

## 1.1 Why This Sub-Phase Comes First in Phase 2

Sub-Phases 2.2 and 2.3 cannot function without services and counters in place:

- A ticket issued in Sub-Phase 2.2 must reference a `Service` (and an assigned counter must exist for that service).
- A counter call in Sub-Phase 2.3 must be issued by an officer assigned to a `Counter`, and the counter must have at least one assigned `Service`.
- The daily reset in Sub-Phase 2.3 iterates over all services and creates a `QueueDailySnapshot` per service.
- The `currentTicketNumber` field on the `Service` entity (populated and reset to 0 by 2.2/2.3, but defined in 2.1.1's schema) is the foundation of ticket-number sequencing.

Sub-Phase 2.1 produces the **data foundation** that 2.2 and 2.3 build on. Any misstep here (wrong unique constraints, missing relations, ambiguous business rules) propagates through every later sub-phase.

---

## 1.2 What This Sub-Phase Is NOT

- It is **not** a ticket issuance implementation. That is Sub-Phase 2.2.
- It is **not** a kiosk UI. The kiosk is the customer-facing entry point; this sub-phase is the admin-facing configuration surface.
- It is **not** an officer-side dashboard or counter-status toggle. The "I am going on a break" counter-closure button is Phase 4.2.1. This sub-phase only **displays** the current operational status (read-only, sourced from the `CounterOfficer.currentStatus` field) on the counter listing.
- It is **not** real-time or SSE-driven. No events are emitted in this sub-phase.
- It is **not** a reporting or analytics layer. The `QueueDailySnapshot` is created in 2.3.3, not here.
- It is **not** a self-service kiosk interaction surface. The kiosk only reads the active services; it does not configure them.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to **distribute the complexity** across the three task documents. The split is:

- **2.1.1** owns the **Service** entity — its API, its business rules, its admin pages, and the `broadcastEvent()` stub (since 2.1.1 is the first Phase 2 document, it establishes the seam).
- **2.1.2** owns the **Counter** entity — its API, its business rules, its admin pages. It runs in parallel with 2.1.1.
- **2.1.3** owns the **Service-Counter Assignment** — the join table, the assignment API, the "at least one service per counter" rule, the cascade behaviour, and the assignment UI integrated with both the counter and service detail pages.

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 2.1.1 — Service Entity & Management API

**Owns:** The `Service` entity, its CRUD API, its admin UI, its business rules, and the `broadcastEvent()` function stub (per Phase 2 cross-cutting standard 8.2).

**Does NOT own:** The Counter entity (owned by 2.1.2), the assignment join table (owned by 2.1.3), ticket issuance (owned by 2.2.x).

**Key outputs:**

- `src/lib/schemas/service.schema.ts`: Zod schemas for service create/update payloads.
- `src/types/service.types.ts`: TypeScript types for service API request/response shapes.
- API endpoints:
  - `GET /api/services` — paginated, searchable list of services. Guarded with `withPermission(PERMISSION_SERVICE_READ)`.
  - `POST /api/services` — create a new service. Guarded with `withPermission(PERMISSION_SERVICE_MANAGE)`.
  - `GET /api/services/[serviceId]` — get a single service. Guarded with `withPermission(PERMISSION_SERVICE_READ)`.
  - `PATCH /api/services/[serviceId]` — update a service. Guarded with `withPermission(PERMISSION_SERVICE_MANAGE)`.
  - `DELETE /api/services/[serviceId]` — soft-delete (deactivate) a service. Guarded with `withPermission(PERMISSION_SERVICE_MANAGE)`.
- `src/lib/events.ts`: the `broadcastEvent(channel, eventType, payload)` stub. Exports the function signature locked in by Phase 2 standard 8.2. The body is a no-op that logs to `console.debug` in development.
- `src/types/sse.types.ts`: TypeScript types for all SSE event payloads (`TICKET_ISSUED`, `TICKET_CALLED`, `TICKET_RECALLED`, `TICKET_NO_SHOW`, `BROADCAST_MESSAGE`, `DAILY_RESET`, `COUNTER_OPENED`, `COUNTER_CLOSED`). 2.1.1 defines the envelope type and the per-event-type payload types. The actual event payloads are filled in by 2.2.1 and 2.3.x.
- UI pages:
  - `src/app/(dashboard)/services/page.tsx` — listing page.
  - `src/app/(dashboard)/services/new/page.tsx` — create page.
  - `src/app/(dashboard)/services/[serviceId]/page.tsx` — edit page (includes the reverse view: which counters handle this service).
- UI components:
  - `src/app/(dashboard)/services/_components/service-table.tsx` — data table.
  - `src/app/(dashboard)/services/_components/service-form.tsx` — shared create/edit form.
- **Service business rules:**
  - `code`: 3-10 uppercase alphanumeric characters, unique across all services.
  - `ticketPrefix`: a single uppercase letter A-Z (the document specifies the exact allowed character set).
  - `isActive`: defaults to `true` on create.
  - `currentTicketNumber`: defaults to `0` on create. **2.1.1 only defines the field and the seed value**; atomic increment is the responsibility of 2.2.1.
  - `sortOrder`: integer, defaults to `0`. Lower numbers appear first in the kiosk grid.
  - `averageServiceTime`: optional integer (minutes per customer). When `null`, the wait-time calculation in 2.2.1 falls back to a system default (configured in 2.3.3).
- Audit log writes for every mutating endpoint (create, update, deactivate). The audit log policy from Phase 1.3.3 applies unchanged.
- Route protection: pages and API endpoints are guarded with the appropriate `withPermission()` calls.

**Master Plan sections implemented:** Sections 6.4, 6.5, 8.2 (`Service` model), 9.3 (services API endpoints).

### 4.2 Document 2.1.2 — Counter Entity & Management API

**Owns:** The `Counter` entity, its CRUD API, its admin UI, and its business rules.

**Does NOT own:** The Service entity (owned by 2.1.1), the assignment join table (owned by 2.1.3), counter temporary-closure toggle (owned by Phase 4.2.1).

**Key outputs:**

- `src/lib/schemas/counter.schema.ts`: Zod schemas for counter create/update payloads.
- `src/types/counter.types.ts`: TypeScript types for counter API request/response shapes.
- API endpoints:
  - `GET /api/counters` — list counters. Includes the current operational status (from `CounterOfficer.currentStatus` of any officer currently on duty at this counter). Guarded with `withPermission(PERMISSION_COUNTER_READ)`.
  - `POST /api/counters` — create a new counter. Guarded with `withPermission(PERMISSION_COUNTER_MANAGE)`.
  - `GET /api/counters/[counterId]` — get a single counter, including its assigned services. Guarded with `withPermission(PERMISSION_COUNTER_READ)`.
  - `PATCH /api/counters/[counterId]` — update a counter. Guarded with `withPermission(PERMISSION_COUNTER_MANAGE)`.
  - `DELETE /api/counters/[counterId]` — soft-delete (deactivate) a counter. Guarded with `withPermission(PERMISSION_COUNTER_MANAGE)`.
- UI pages:
  - `src/app/(dashboard)/counters/page.tsx` — listing page (with operational status column).
  - `src/app/(dashboard)/counters/new/page.tsx` — create page.
  - `src/app/(dashboard)/counters/[counterId]/page.tsx` — edit page.
- UI components:
  - `src/app/(dashboard)/counters/_components/counter-table.tsx` — data table.
  - `src/app/(dashboard)/counters/_components/counter-form.tsx` — shared create/edit form.
- **Counter business rules:**
  - `number`: unique integer (the number shown on the display board).
  - `name`: required string (the administrative name, e.g., "Counter 1" or "Information Desk").
  - `displayLabel`: optional string. If null, defaults to `name`. If set, used on the display board (e.g., admin name "Counter 2" becomes "Information" on the display).
  - `isActive`: defaults to `true`. Deactivated counters are excluded from assignment UI but remain in the database.
- **Operational status display rule:** the counter listing shows the current operational status as a read-only badge, sourced from the `CounterOfficer.currentStatus` field of any officer currently on duty at this counter. If no officer is on duty, the badge reads "No officer on duty". This sub-phase **does not implement the toggle** (the toggle is Phase 4.2.1); it only displays the current state.
- Audit log writes for every mutating endpoint.
- Route protection: pages and API endpoints are guarded.

**Master Plan sections implemented:** Sections 6.4, 6.5, 8.2 (`Counter` model), 9.3 (counter API endpoints).

### 4.3 Document 2.1.3 — Service-Counter Assignment Logic

**Owns:** The `CounterService` join table, the assignment API, the "at least one service per counter" invariant, the cascade behaviour, and the assignment UI.

**Does NOT own:** The Service or Counter entities themselves (owned by 2.1.1 and 2.1.2), any business logic for tickets (Phase 2.2+), counter temporary-closure toggle (Phase 4.2.1).

**Key outputs:**

- API endpoints:
  - `GET /api/counters/[counterId]/services` — list services assigned to a counter. Guarded with `withPermission(PERMISSION_COUNTER_READ)`.
  - `POST /api/counters/[counterId]/services` — assign a service to a counter (body: `{ serviceId: string }`). Guarded with `withPermission(PERMISSION_COUNTER_MANAGE)`.
  - `DELETE /api/counters/[counterId]/services/[serviceId]` — unassign a service from a counter. Guarded with `withPermission(PERMISSION_COUNTER_MANAGE)`. Enforces the "at least one service per counter" rule.
- UI components:
  - `src/app/(dashboard)/counters/[counterId]/services/page.tsx` — the counter's services assignment page.
  - `src/app/(dashboard)/counters/_components/counter-service-assignment.tsx` — the assignment widget.
- **Service detail page** (in `src/app/(dashboard)/services/[serviceId]/page.tsx`, owned by 2.1.1) is extended in this document to show the reverse view: which counters handle this service, with links to each counter's detail page.
- **Business rules:**
  - Composite unique constraint on `(counterId, serviceId)` — duplicate assignments are rejected with `CONFLICT` (HTTP 409).
  - "At least one service per counter" rule: unassigning the last service from a counter is rejected with a clear `VALIDATION_ERROR` (HTTP 422).
  - The inverse rule (a service must be in at least one counter) is **not** enforced. Services can exist as configuration before any counter handles them.
- **Cascade behaviour:**
  - Deactivating a service (`isActive = false`) does NOT remove any `CounterService` rows. The assignments remain; the service just becomes hidden from the kiosk-active list.
  - Deactivating a counter (`isActive = false`) does NOT remove any `CounterService` rows. The assignments remain; the counter just stops appearing in the "active counters" lists.
  - **No hard delete.** Services and counters are never hard-deleted while ticket history references them. The `DELETE` endpoint is a soft-delete (status change).
- Audit log writes for every assignment change. The action types are added to the `AuditAction` union: `SERVICE_ASSIGNED_TO_COUNTER`, `SERVICE_UNASSIGNED_FROM_COUNTER`.
- Route protection: assignment pages and endpoints are guarded with `counter:manage`.

**Master Plan sections implemented:** Sections 6.5 (`CounterServiceAssignment` component), 8.2 (`CounterService` join table), 9.3 (assignment API endpoints), 17 (decisions on cascade behaviour).

## 5. Document Dependency Map

```
Phase 1.3 RBAC (withPermission, usePermission, Can, audit-log helper)
   │
   ▼
2.1.1  Service Entity & Management API     ──┐
                                              │
2.1.2  Counter Entity & Management API     ──┤──> 2.1.3  Service-Counter Assignment
                                              │
                                              ▼
                              (Phase 2.2/2.3 builds on the configuration)
```

**Build order:** `2.1.1` and `2.1.2` can be developed **in parallel** after Phase 1.3 is complete. `2.1.3` requires both `2.1.1` and `2.1.2`.

**Why this order:**

- **2.1.1 and 2.1.2 in parallel.** They touch different entities (`Service` and `Counter`) and different files. They share the `withPermission` guard, the audit-log helper, and the response envelope, but do not share business logic.
- **2.1.3 after both 2.1.1 and 2.1.2.** The assignment join table references both entities; the assignment API and UI need both APIs to exist.
- **Sub-Phases 2.2 and 2.3 after 2.1.3.** Sub-Phase 2.2 cannot issue a ticket without a service; Sub-Phase 2.3 cannot call a ticket without a counter that has at least one service assigned.

**What depends on this sub-phase:**

- **Sub-Phase 2.2.1** reads `Service` records (to display in the kiosk) and uses `CounterService` to determine which counter will serve an issued ticket.
- **Sub-Phase 2.3.1** reads `Counter` records and their assigned services (the officer can only call tickets for services their counter handles).
- **Sub-Phase 2.3.3** iterates over all services to perform the daily reset.
- **Phase 3 display board** reads services and counters to render the "Now serving X at Counter Y" view.
- **Phase 4 officer dashboard** reads the counter's services to know which tickets are eligible to be called.

## 6. Cross-Document Standards

These standards apply to every document in Sub-Phase 2.1. The conventions from Phase 1 (folder naming, import paths, TypeScript, Zod validation, error handling, env vars, git commits, audit logging) all carry forward unchanged. The standards below are **new or specific to Phase 2**.

### 6.1 Folder and File Naming

- All folders use kebab-case for multi-word names.
- Component files use PascalCase (`ServiceForm.tsx`, `CounterTable.tsx`, `CounterServiceAssignment.tsx`).
- Non-component modules use camelCase (`service.ts`, `counter.ts`, `events.ts`, `service-service.ts`).
- Schema files use the pattern `*.schema.ts` (e.g., `src/schemas/service.schema.ts`).
- Type files use `*.types.ts`.
- Route handlers follow Next.js App Router conventions (`route.ts`).

### 6.2 Import Conventions

- All project-internal imports use the `@/` alias.
- The `ServiceForm` and `CounterForm` components may be reused across pages.
- The `broadcastEvent` function is imported from `@/lib/events` (per Phase 2 standard 8.2).
- The `writeAuditLog` function is imported from `@/lib/audit-log` (from Phase 1.3.3).
- The `withPermission` guard is imported from `@/lib/guards` (from Phase 1.3.2).
- The permission constants are imported from `@/lib/permissions` (from Phase 1.3.1).

### 6.3 TypeScript Standards

- All exported functions and components have explicit type annotations.
- No `any` types in production code.
- The `Service` and `Counter` Prisma types are re-exported as types from `@/types/service.types.ts` and `@/types/counter.types.ts` for use in client components (so client components do not import the Prisma client directly).
- The `SseEvent` envelope type (defined in 2.1.1's `src/types/sse.types.ts`) is exported and consumed by 2.2/2.3.
- Discriminated unions are used where appropriate (e.g., the SSE event payload types).

### 6.4 Response Envelope Standard

Every API route in this sub-phase must return the master plan's standard envelope (Section 9.1) and use the standard error codes (Section 9.2):

| Code               | HTTP Status | When                                                                                                           |
| ------------------ | ----------- | -------------------------------------------------------------------------------------------------------------- |
| `UNAUTHORIZED`     | 401         | No valid authentication token                                                                                  |
| `FORBIDDEN`        | 403         | Authenticated but lacks permission                                                                             |
| `NOT_FOUND`        | 404         | Service or counter does not exist                                                                              |
| `VALIDATION_ERROR` | 422         | Request body fails Zod validation; or the "at least one service per counter" rule is violated                  |
| `CONFLICT`         | 409         | Unique constraint violation (duplicate `code`, `ticketPrefix`, `number`); or duplicate (counter, service) pair |
| `INTERNAL_ERROR`   | 500         | Unexpected server error                                                                                        |

### 6.5 Validation Standards

- Every form on the client has a Zod schema in `src/schemas/`.
- The same Zod schema is reused on the server (where applicable) for server-side validation.
- Validation error responses use `VALIDATION_ERROR` (HTTP 422) with the Zod error details in `error.details`.
- Schemas for this sub-phase:
  - `src/schemas/service.schema.ts`: `createServiceSchema`, `updateServiceSchema`, `listServicesQuerySchema`.
  - `src/schemas/counter.schema.ts`: `createCounterSchema`, `updateCounterSchema`, `listCountersQuerySchema`.
  - `src/schemas/counter-service.schema.ts`: `assignServiceSchema` (with `serviceId`).

### 6.6 Error Handling Standards

- Server actions and API routes return the master plan's standard envelope.
- Client-side error display uses shadcn `Toast` for transient errors and inline alerts for form-field errors.
- Every API route is wrapped in a try/catch that logs the error (in development) and returns `INTERNAL_ERROR` (HTTP 500) for unhandled exceptions.

### 6.7 Audit Logging Standard (Phase 2 Policy)

- Every mutating action in this sub-phase writes an `AuditLog` entry via `writeAuditLog()`:
  - **Service**: `SERVICE_CREATED`, `SERVICE_UPDATED`, `SERVICE_DEACTIVATED`.
  - **Counter**: `COUNTER_CREATED`, `COUNTER_UPDATED`, `COUNTER_DEACTIVATED`.
  - **Assignment**: `SERVICE_ASSIGNED_TO_COUNTER`, `SERVICE_UNASSIGNED_FROM_COUNTER`.
- These action types are added to the `AuditAction` union in `@/lib/audit-log.ts` (a Phase 2 update to Phase 1.3.3's module — the existing audit-log module must be extended, not duplicated).
- Each entry contains the actor's user ID (from the session), the action, a human-readable description, and a metadata JSON blob with change details.
- **Best-effort writes** continue from Phase 1.3.3: audit log failures do not fail the user-facing action.

### 6.8 Broadcast Event Seam (Phase 2 → Phase 3)

- A single utility function `broadcastEvent(channel: SseChannel, eventType: SseEventType, payload: SseEventPayload): Promise<void>` is established in 2.1.1 at `src/lib/events.ts`.
- During Phase 2, the function body is a no-op (or a `console.debug` log). Phase 3.1.3 replaces it with the real SSE-based implementation.
- 2.1.1 defines the function signature, the `SseChannel` type, the `SseEventType` union, and the `SseEventPayload` discriminated union in `src/types/sse.types.ts`.
- 2.1.1 does **not** call `broadcastEvent()` itself (no events are emitted in this sub-phase).
- 2.2.1, 2.3.1, 2.3.2, and 2.3.3 will call `broadcastEvent()` for their respective events.
- This seam is the only acceptable way to introduce the Phase 3 dependency in Phase 2 code.

### 6.9 Environment Variable Standards

- No new environment variables are introduced by this sub-phase beyond what Phase 1 added.
- The `APP_TIMEZONE` variable may be introduced in 2.3.3 (for the business date boundary). If 2.1.1 finds it referenced (it should not), it should be added to `.env.example` with a default value.

### 6.10 Commit Standards

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`.
- Each task plan document corresponds to one or more commits. A single commit should not span multiple task plan documents.
- Suggested commit boundaries:
  - End of 2.1.1: `feat: add Service entity with CRUD API, admin UI, and broadcast event stub`
  - End of 2.1.2: `feat: add Counter entity with CRUD API and admin UI`
  - End of 2.1.3: `feat: add service-counter assignment logic with cascade behavior`
- Updating the `AuditAction` union (Phase 2 extension of the Phase 1.3.3 module) may be split into its own commit or bundled with the first Phase 2 task plan that uses the new actions (2.1.1).

## 7. Sub-Phase Exit Criteria

Sub-Phase 2.1 is complete when **all** of the following are true:

### 7.1 Service Management (from 2.1.1)

- [ ] `GET /api/services` returns a paginated, searchable list of services.
- [ ] `POST /api/services` creates a service with a unique code and a unique ticket prefix.
- [ ] Duplicate `code` is rejected with `CONFLICT` (HTTP 409).
- [ ] Duplicate `ticketPrefix` is rejected with `CONFLICT` (HTTP 409).
- [ ] `ticketPrefix` outside the allowed character set is rejected with `VALIDATION_ERROR` (HTTP 422).
- [ ] `GET /api/services/[serviceId]` returns a single service.
- [ ] `PATCH /api/services/[serviceId]` updates a service.

## 8. AI Agent Implementation Guidance

This section is for the AI agents that will implement the task plan documents. It is guidance that applies across all three task plan documents in this sub-phase.

### 8.1 How to Use the Task Plan Documents

Each task plan document follows the same internal structure:

1. **Scope** — what this document owns and what it does not.
2. **Pre-requisites** — what must already be true (previous documents completed).
3. **Step-by-step implementation plan** — the actual work, in order.
4. **File-level specifications** — for every file the agent will create or modify.
5. **Verification checklist** — how the agent proves the work is done.
6. **Commit instructions** — what commit(s) to make at the end.
7. **Common pitfalls** — mistakes to avoid.
8. **AI agent guidance** — document-specific guidance.

The agent should follow the steps in order. Each step lists the **goal** of the step, the **files affected**, and any **specific instructions** that resolve the ambiguity the agent would otherwise have to guess about.

### 8.2 General Guidance

- **Do not skip steps.** Sub-Phase 2.1 is mostly sequential within each document (each step's output is the next step's input), but 2.1.1 and 2.1.2 can be developed in parallel.
- **Do not add features beyond the document's scope.** The temptation to "just add ticket issuance now" or "just add the daily reset" is real — resist it. Those are owned by later sub-phases.
- **Verify after every step.** The verification checklist at the end of each document is a hard gate.
- **When the master plan is ambiguous, choose the simpler option.** If a configuration choice has two reasonable interpretations, pick the one that is easier to change later.
- **Commit at the end of each document** using the commit message specified in the document.
- **Document any deviation.** If the agent makes a choice that differs from this document, it must note the deviation in the final summary so the human reviewer can approve or reject it.
- **Always use the master plan's response envelope.** Every API route must return `{ success, data | error }`.
- **Always use `withPermission()` from Phase 1.3.2** for route guards. Do not write inline session checks.
- **Always use `writeAuditLog()` from Phase 1.3.3** for mutating actions. The action types are extended in this sub-phase.
- **Establish the broadcast event seam in 2.1.1** even though 2.1.1 does not call it. The seam must exist so 2.2/2.3 can call it without modification.

### 8.3 What the AI Agent Should NOT Do

- Should not issue any tickets (owned by 2.2.x).
- Should not implement the kiosk UI (owned by 2.2.2).
- Should not implement counter temporary-closure toggle (owned by Phase 4.2.1).
- Should not implement ticket call/recall/no-show (owned by 2.3.x).
- Should not implement the daily reset (owned by 2.3.3).
- Should not hard-delete any service or counter. Always soft-delete (deactivate).
- Should not modify the Prisma schema (it was finalised in Phase 1.1.3).
- Should not implement rate limiting, security headers, or production hardening (Phase 5).
- Should not add PostgreSQL configuration.
- Should not implement real-time or SSE delivery (Phase 3).
- Should not implement push notifications (Phase 4).
- Should not implement reporting (Phase 5).
- Should not implement an audit log viewer UI (Phase 5).
- Should not allow a service or counter to be hard-deleted while ticket history references it (use soft-delete).

### 8.4 When to Ask the Human

The AI agent should ask the human reviewer (not guess) if any of the following situations arise:

- The master plan and the phase 2 overview contradict each other. The master plan wins, but the agent should confirm.
- The exact allowed character set for `Service.ticketPrefix` is ambiguous (the document specifies the default; the agent should confirm if there is any doubt).
- The current operational status display logic for counters is unclear (e.g., multiple officers on duty at the same counter — which status wins? The document specifies "any officer currently on duty"; if this is ambiguous, the agent should confirm).
- The audit log action type names are not specified (the document specifies them; the agent should use the names exactly).
- A schema field is missing or ambiguous (e.g., the `currentTicketNumber` initial value, the `sortOrder` default).
- The verification step fails for a reason that the document does not anticipate.
