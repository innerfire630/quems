## 1. Document Purpose

This document delivers the **many-to-many assignment** between services and counters via the `CounterService` join table. It implements the assignment API, the "at least one service per counter" invariant, the cascade behaviour on deactivation, the assignment UI integrated with both the counter detail page and the service detail page, and the audit log writes for every assignment change.

When this document is complete, a super-admin can:

- Navigate to a counter's services page (`/counters/[counterId]/services`) and see the list of currently assigned services.
- Add a new service to a counter from a dropdown of available (active and inactive) services.
- Remove a service from a counter — unless it would leave the counter with no assigned services (in which case the action is rejected with a clear error).
- View the reverse direction from a service's edit page (already created in 2.1.1): see which counters handle this service, with links to each counter's detail page.

A non-admin cannot perform any of these actions.

  - `src/app/(dashboard)/counters/[counterId]/services/page.tsx` — the full counter services page (replaces the placeholder from 2.1.2).
  - `src/app/(dashboard)/counters/_components/counter-service-assignment.tsx` — the assignment widget.
- Reverse view on the service detail page:
  - Extend `src/app/(dashboard)/services/[serviceId]/page.tsx` (created in 2.1.1) to render the list of assigned counters with links.
- Audit log writes:
  - Extend `src/lib/audit-log.ts` to add `SERVICE_ASSIGNED_TO_COUNTER` and `SERVICE_UNASSIGNED_FROM_COUNTER` to the `AuditAction` union.
- Zod schemas for assignment:
  - `src/schemas/counter-service.schema.ts` with `assignServiceSchema` (just `serviceId: string`).
- Business rules:
  - Composite unique on `(counterId, serviceId)` enforced at the database level.
  - "At least one service per counter" rule enforced at the API level.
  - Soft-delete cascade: deactivating a service or counter preserves join table rows.
- Route protection: all endpoints guarded with `PERMISSION_COUNTER_MANAGE`.

### 2.2 Out of Scope (Deferred to Other Documents)

- The Service entity, its API, and its UI. Owned by 2.1.1.
- The Counter entity, its API, and its UI. Owned by 2.1.2.
- Counter temporary-closure toggle. Owned by Phase 4.2.1.
- Ticket issuance and the ticket model. Owned by 2.2.x.
- Ticket call/recall/no-show (which depend on the assignment). Owned by 2.3.x.

### 2.3 What This Document Does NOT Touch

- It does **not** implement the Service or Counter entities themselves.
- It does **not** implement any ticket-related logic.
- It does **not** modify the Prisma schema.
- It does **not** call `broadcastEvent()` (assignment changes are not SSE events; they are configuration changes consumed only by the admin UI and 2.2/2.3).

### Step 3: Create `GET /api/counters/[counterId]/services` — List Assigned Services

**Goal:** Provide an endpoint that returns the list of services assigned to a counter.

**Files affected:** `src/app/api/counters/[counterId]/services/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`GET` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_COUNTER_READ)`.
  - Reads `counterId` from route params.
  - Verifies the counter exists. If not, returns `NOT_FOUND` (HTTP 404).
  - Queries `db.counterService.findMany({ where: { counterId }, include: { service: true } })`.
  - Maps the result to an array of `{ id, serviceId, service: { id, name, code, ticketPrefix, isActive } }`.
  - Returns `{ success: true, data: <array> }`.
- All error responses use the master plan envelope.

---

### Step 4: Create `POST /api/counters/[counterId]/services` — Assign a Service

**Goal:** Provide an endpoint to assign a service to a counter.

**Files affected:** same file as Step 3 (`src/app/api/counters/[counterId]/services/route.ts`).

**Specific instructions:**

- Add a **`POST` handler** to the file:
  - Wrapped with `withPermission(handler, PERMISSION_COUNTER_MANAGE)`.
  - Reads `counterId` from route params.
  - Verifies the counter exists. If not, returns `NOT_FOUND`.
  - Validates the body with `assignServiceSchema`.
  - Verifies the service exists. If not, returns `NOT_FOUND` with a clear message ("Service not found.").
  - Checks for an existing `CounterService` row with this `(counterId, serviceId)`. If exists, returns `CONFLICT` (HTTP 409) with message "Service is already assigned to this counter."
  - Creates the `CounterService` row.
  - On success, calls `writeAuditLog({ action: 'SERVICE_ASSIGNED_TO_COUNTER', actorId, targetUserId: counterId, description: \`Assigned service "\${service.name}" to counter "\${counter.name}".\`, metadata: { counterId, counterName, serviceId, serviceName } })`.
  - Returns `{ success: true, data: <assignment> }` with HTTP 201.
- Note: the document specifies that assignment is **not restricted** to active services or active counters. An admin may want to assign a service to an inactive counter for a future re-activation, or assign an inactive service for similar reasons. The UI shows a warning if either side is inactive, but the action is allowed.

---

### Step 5: Create `DELETE /api/counters/[counterId]/services/[serviceId]` — Unassign

**Goal:** Provide an endpoint to remove a service from a counter, enforcing the "at least one service" rule.

**Files affected:** `src/app/api/counters/[counterId]/services/[serviceId]/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`DELETE` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_COUNTER_MANAGE)`.
  - Reads `counterId` and `serviceId` from route params.
  - Verifies the counter exists. If not, returns `NOT_FOUND`.
  - Verifies the assignment exists. If not, returns `NOT_FOUND` with message "Service is not assigned to this counter."
  - **Enforces the "at least one service per counter" rule:**
    - Counts the total `CounterService` rows for this counter.
    - If the count is `1` (this is the last service), returns `VALIDATION_ERROR` (HTTP 422) with message "Cannot remove the last service from a counter. A counter must have at least one assigned service."
  - Deletes the `CounterService` row.
  - On success, calls `writeAuditLog({ action: 'SERVICE_UNASSIGNED_FROM_COUNTER', actorId, targetUserId: counterId, description: \`Removed service "\${service.name}" from counter "\${counter.name}".\`, metadata: { counterId, counterName, serviceId, serviceName } })`.
  - Returns `{ success: true, data: { message: 'Service unassigned.' } }`.
- All error responses use the master plan envelope.

### Step 6: Create the Counter Service Assignment Widget

**Goal:** A client component that allows adding and removing services from a counter.

**Files affected:** `src/app/(dashboard)/counters/_components/counter-service-assignment.tsx` (new file).

**Specific instructions:**

Create the file containing:

- A "use client" directive.
- **Props:** `{ counterId: string; assignedServices: AssignedService[]; availableServices: AvailableService[]; onChange?: () => void }`.
  - `AssignedService`: `{ id, serviceId, service: { id, name, code, ticketPrefix, isActive } }` (the current assignments).
  - `AvailableService`: `{ id, name, code, ticketPrefix, isActive }` (all services in the system, used for the add dropdown).
- Renders:
  - A header: "Assigned Services" with a count badge.
  - A list of currently assigned services, each rendered as a `Card` or list item showing the service's name, code, ticket prefix, and active status badge. Each item has a "Remove" button.
  - Below the list, an "Add Service" affordance:
    - A `Select` (shadcn) dropdown listing the services not already assigned (filtered from `availableServices`).
    - A "Add" button that submits the assignment.
- **Behaviour:**
  - On "Add": calls `POST /api/counters/[counterId]/services` with `{ serviceId }`. On success, the page refreshes (via `router.refresh()` or `onChange` callback). On error, a Toast is shown.
  - On "Remove": a confirmation dialog (shadcn `AlertDialog`) asks "Remove this service from the counter?". On confirm, calls `DELETE /api/counters/[counterId]/services/[serviceId]`. On success, the page refreshes. On the "last service" error, a Toast shows the clear message.
- **Inactive handling:**
  - Inactive services are shown in the assigned list with an "Inactive" badge. They can still be removed.
  - Inactive services are shown in the add dropdown with an "Inactive" label. They can still be assigned (with a warning Tooltip: "This service is currently inactive. It will appear on the kiosk only when reactivated.").

## 8. Common Pitfalls

### 8.1 Race Condition on "Last Service" Check

The "at least one service" check (count = 1, then delete) is not atomic. Two concurrent DELETE requests may both see count=1 and both succeed, leaving the counter with zero services. **The agent should wrap the count + delete in a transaction** (or use Prisma's transaction API) to make this atomic.

### 8.2 Forgetting the Cascade Behavior

Deactivating a service or counter must NOT remove join table rows. A `prisma.service.update({ where: { id }, data: { isActive: false } })` does not cascade to `CounterService`. The agent should verify this manually (no `onDelete: Cascade` is set on `CounterService`).

### 8.3 Hard-Deleting Services or Counters with Active Assignments

The 2.1.1 and 2.1.2 documents use soft-delete only. Hard-deleting an entity with assignments (and possibly tickets) would break foreign-key references.

### 8.4 Not Showing the Reverse View on the Service Detail Page

Without the reverse view, an admin editing a service has no way to see which counters handle it without navigating to each counter individually. The reverse view is essential.

### 8.5 Confusing Assignment Direction

The document is **counter-centric**: assignment is managed from the counter's services page. The reverse view on the service page is read-only. An agent that adds an "assign counters to service" UI on the service page would create two divergent ways to manage the same relationship.

### 8.6 Missing the Confirmation Dialog

Removing a service is destructive (the assignment is deleted). A confirmation dialog prevents accidental removal. Omitting it leads to frustration.

### 8.7 Inactive Service Shown Without Warning

The kiosk does not show inactive services. If an admin assigns an inactive service to a counter, the counter's "ticket creation" will still work but the service won't appear on the kiosk UI. A warning Tooltip prevents this confusion.

## 9. AI Agent Guidance

### 9.1 How to Approach This Document

The agent should treat this document as a **join table management task**. The patterns are identical to 2.1.1 and 2.1.2 (schemas → types → API → UI), but the focus is the relationship, not the entities.

The agent should:

1. **Start with the audit-log extension** (Step 1) — completes the Phase 2 audit log additions.
2. **Build the assignment API** (Steps 2-5): schema, GET, POST, DELETE. Test the "last service" rule with curl.
3. **Build the UI** (Steps 6-8): widget, page, reverse view.
4. **Verify** (Steps 9-10).

### 9.2 What the Agent Should NOT Do

- Should not implement service or counter CRUD (2.1.1 / 2.1.2).
- Should not implement ticket issuance (2.2.x).
- Should not modify the Prisma schema.
- Should not use `as any`.
- Should not call `broadcastEvent()` (assignment changes are not events).

### 9.3 Verification Tip

The "last service" rule is the trickiest part. Test it with two concurrent DELETE requests via parallel curl calls. The transaction should prevent both from succeeding. If both succeed, the transaction is missing.
