## 1. Document Purpose

This document delivers the **`Service` entity** end-to-end: the API endpoints that operate on it, the admin UI pages that allow super-admins to configure it, the Zod schemas that validate it, the TypeScript types that describe it, and the business rules that govern it. It also establishes the **`broadcastEvent()` function stub** and the **`SseEvent` types** that the Phase 2 → Phase 3 seam requires (per Phase 2 cross-cutting standard 8.2).

When this document is complete, a super-admin can:

- Navigate to `/services` and see all services in a paginated, searchable data table.
- Click "Create Service" and create a new service with a unique code, a unique ticket prefix, a description, an icon, a color, an estimated service time, and a sort order.
- Edit any service (change any field including active/inactive status).
- Deactivate a service (soft-delete) — it disappears from the active list but remains in the database for historical ticket references.
- See the current ticket number for each service (the value is `0` until 2.2.1 introduces atomic increment).

A non-admin cannot access these pages or APIs.

  - `DELETE /api/services/[serviceId]`

  - `src/app/(dashboard)/services/_components/service-table.tsx`
  - `src/app/(dashboard)/services/_components/service-form.tsx`

  - `code` unique, 3-10 uppercase alphanumeric characters.
  - `ticketPrefix` unique, single uppercase letter A-Z (no other characters allowed).
  - `isActive` defaults to `true`.

- The Service-Counter assignment. Owned by 2.1.3.

### Step 3: Create `src/types/sse.types.ts` — The SSE Event Types

**Goal:** Define the TypeScript types for all SSE event payloads that Phase 2 will emit and Phase 3 will deliver.

**Files affected:** `src/types/sse.types.ts` (new file).

**Specific instructions:**

Create `src/types/sse.types.ts` containing:

- **`type SseChannel = 'global' | 'counter' | 'security'`** — the three SSE channel names from Master Plan Section 11.1. The full channel path strings (`/api/sse/global`, `/api/sse/counter/[counterId]`, `/api/sse/security`) are not part of the type — the channel enum is the identifier.
- **`type SseEventType = 'TICKET_ISSUED' | 'TICKET_CALLED' | 'TICKET_RECALLED' | 'TICKET_NO_SHOW' | 'TICKET_SERVED' | 'TICKET_COMPLETED' | 'BROADCAST_MESSAGE' | 'DAILY_RESET' | 'COUNTER_OPENED' | 'COUNTER_CLOSED' | 'QUEUE_UPDATED' | 'NOTIFICATION_RECEIVED' | 'OFFICER_REPLY'`** — the full registry of event types. This includes events emitted in Phase 2 (the ticket lifecycle events, daily reset, broadcast) AND events emitted in Phase 3/4 that share the same envelope structure.
- **`type SseEventPayload = <discriminated union keyed on `type`>`** — a discriminated union with one variant per event type. 2.1.1 defines the envelope shape and the `TICKET_ISSUED` placeholder payload (since 2.1.1 does not emit the event itself but locks the type). The full payload types for the other events are filled in by 2.2.1, 2.3.1, 2.3.2, 2.3.3, and Phase 3.
  - The envelope type itself: `{ type: SseEventType; id: string; timestamp: string (ISO 8601); payload: <event-specific> }`. This matches Master Plan Section 11.2.
- **`type SseEvent = { channel: SseChannel; envelope: SseEventPayload }`** — the structure passed to `broadcastEvent()`. The function receives the channel and the full envelope (or builds the envelope from the event type + payload — the document specifies the exact shape: the function receives `channel`, `eventType`, and `payload` and constructs the envelope internally with a generated `id` and `timestamp`).
- Document that the payload types for events that 2.1.1 does not emit (TICKET_CALLED, TICKET_RECALLED, etc.) are placeholders in 2.1.1. The full payload shapes are defined by the document that introduces the emitting endpoint.

### Step 4: Create `src/schemas/service.schema.ts` — The Zod Schemas

**Goal:** Validate every service create/update payload on both client and server.

**Files affected:** `src/schemas/service.schema.ts` (new file).

**Specific instructions:**

Create `src/schemas/service.schema.ts` containing:

- **`createServiceSchema`**:
  - `name`: required string, 1-100 characters, trimmed.
  - `code`: required string, 3-10 characters, uppercase alphanumeric only (regex: `^[A-Z0-9]{3,10}$`).
  - `ticketPrefix`: required string, exactly 1 character, uppercase letter A-Z only (regex: `^[A-Z]$`).
  - `description`: optional string, max 500 characters.
  - `iconName`: optional string, must be a valid Lucide icon name (the document specifies this — for now, accept any string; Phase 2 may add a stricter validator).
  - `color`: optional string, must match the regex `^#[0-9A-Fa-f]{6}$` (hex color).
  - `isActive`: optional boolean, defaults to `true` (the API applies the default if the field is absent).
  - `currentTicketNumber`: optional integer, defaults to `0` (the API applies the default; the form does not expose this field).
  - `averageServiceTime`: optional positive integer (minutes), max 120.
  - `sortOrder`: optional integer, defaults to `0`.
- **`updateServiceSchema`**:
  - All fields optional (PATCH is partial).
  - At least one field must be present (use Zod's `.refine` to enforce).
  - `currentTicketNumber`: optional integer. **The form does not expose this field**; only 2.2.1's increment logic modifies it. If a malicious request includes it, the API may accept it (or may explicitly reject it — the document specifies the behaviour: accept it, since administrative bulk-fixes may be needed; document this).
- **`listServicesQuerySchema`**:
  - `page`: optional integer, default 1, min 1.
  - `limit`: optional integer, default 20, min 1, max 100.
  - `search`: optional string, max 100 characters.
  - `isActive`: optional enum string (`'true'`, `'false'`, or absent for both). Parsed to boolean on the server.
- Export inferred TypeScript types: `CreateServiceInput`, `UpdateServiceInput`, `ListServicesQuery`.

### Step 6: Create `src/app/api/services/route.ts` — List and Create

**Goal:** Provide the listing endpoint (paginated, searchable, filterable by active status) and the create endpoint.

**Files affected:** `src/app/api/services/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`GET` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_SERVICE_READ)`.
  - Parses `page`, `limit`, `search`, `isActive` from query parameters using `listServicesQuerySchema`.
  - Builds a Prisma `where` clause:
    - If `search` is present: `OR: [{ name: { contains: search } }, { code: { contains: search } }, { description: { contains: search } }]` (case-insensitive where the database supports it — SQLite is case-insensitive by default for `LIKE`; if Postgres, use `mode: 'insensitive'`).
    - If `isActive` is present (parsed to boolean): `{ isActive }`.
  - Executes Prisma `findMany` ordered by `sortOrder asc, name asc` (the kiosk's display order is the primary sort; name is the secondary).
  - Counts total for pagination.
  - Returns `{ success: true, data: <mapped ServiceListItem[]>, meta: { page, limit, total, totalPages } }`.
- **`POST` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_SERVICE_CREATE)` (or `PERMISSION_SERVICE_MANAGE` if the agent prefers the single permission; document the choice).
  - Parses the request body as JSON, validates with `createServiceSchema`. On validation failure, returns `VALIDATION_ERROR` (HTTP 422).
  - Checks for existing service with the same `code` or `ticketPrefix`. If either exists, returns `CONFLICT` (HTTP 409) with a clear message indicating which field.
  - Creates the service with a Prisma `create` call. The default for `isActive` (true) and `currentTicketNumber` (0) and `sortOrder` (0) are applied if the input does not include them.
  - On success, calls `writeAuditLog({ action: 'SERVICE_CREATED', actorId: session.user.userId, targetUserId: service.id, description: \`Created service "\${service.name}" (\${service.code}).\`, metadata: { name, code, ticketPrefix, isActive } })`.
  - Returns `{ success: true, data: <ServiceListItem> }` with HTTP 201.
- All error responses use the master plan envelope. Non-success paths are wrapped in try/catch.

### Step 7: Create `src/app/api/services/[serviceId]/route.ts` — Get, Update, Delete

**Goal:** Provide per-service endpoints for retrieval (with the reverse view of assigned counters), update, and soft-delete.

**Files affected:** `src/app/api/services/[serviceId]/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`GET` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_SERVICE_READ)`.
  - Reads `serviceId` from route params.
  - Queries the service with `include: { counters: { include: { counter: true } } }`.
  - If the service does not exist, returns `NOT_FOUND` (HTTP 404).
  - Maps the result to `ServiceDetail` (with the `counters` field flattened to the counter array).
  - Returns `{ success: true, data: <ServiceDetail> }`.
- **`PATCH` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_SERVICE_UPDATE)` (or `PERMISSION_SERVICE_MANAGE`).
  - Reads `serviceId` from route params.
  - Captures the "before" state of the service.
  - Parses and validates the body with `updateServiceSchema`. On validation failure, returns `VALIDATION_ERROR`.
  - If `code` or `ticketPrefix` is being changed, checks for conflicts. If conflict, returns `CONFLICT`.
  - Updates the service with `db.service.update`. The agent should decide whether to allow `currentTicketNumber` updates here (the document specifies: allow it for administrative use, but document this clearly).
  - On success, calls `writeAuditLog({ action: 'SERVICE_UPDATED', actorId, targetUserId: serviceId, description: \`Updated service "\${service.name}".\`, metadata: { before, after } })`. If status changed from `true`to`false`, the action is `'SERVICE_DEACTIVATED'`(the description should reflect this:`"Deactivated service ..."`).
  - Returns `{ success: true, data: <ServiceListItem> }`.
- **`DELETE` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_SERVICE_DELETE)` (or `PERMISSION_SERVICE_MANAGE`).
  - Reads `serviceId` from route params.
  - **Soft-delete**: sets `isActive = false` (NOT a hard delete). Uses the same `PATCH` logic under the hood, or a dedicated `db.service.update({ where: { id }, data: { isActive: false } })`.
  - On success, calls `writeAuditLog({ action: 'SERVICE_DEACTIVATED', actorId, targetUserId: serviceId, description: \`Deactivated service "\${service.name}".\`, metadata: { name, code } })`.
  - Returns `{ success: true, data: { message: 'Service deactivated.' } }`.
  - **Important:** hard-deleting a service that has ticket history is forbidden. The agent should verify (via a Prisma check) that no tickets reference the service before allowing a hard delete. Since this sub-phase uses soft-delete only, this check is defensive — the API does not expose a hard-delete endpoint.
- All error responses use the master plan envelope.

## 8. Common Pitfalls

The following mistakes are common when implementing the Service entity and the broadcast event seam, and are worth avoiding.

### 8.1 Hard-Deleting Services Instead of Soft-Deleting

Services with ticket history must never be hard-deleted. The `DELETE` endpoint is a soft-delete (sets `isActive = false`). Forgetting this would break ticket history references.

### 8.2 Allowing `ticketPrefix` Outside A-Z

The `ticketPrefix` field is used directly in ticket numbers (e.g., `A001`). Allowing lowercase, digits, or multi-character prefixes would either produce invalid ticket numbers or require complex escaping. The Zod regex `^[A-Z]$` enforces this at the API level; the form must also enforce it on the client.

### 8.3 Not Handling the `currentTicketNumber` Field on Create

The `currentTicketNumber` field defaults to `0` on create. If the Prisma schema defaults to `0`, the API does not need to set it explicitly. If it does not, the API must set it to `0` explicitly to avoid `null` values.

### 8.4 Forgetting to Extend the `AuditAction` Union

If the agent adds the new action types to the API endpoints but forgets to extend the `AuditAction` union in `audit-log.ts`, TypeScript will reject the `writeAuditLog()` calls with a type error. The extension must come first.

### 8.5 Confusing Sort Order with Display Order

The `sortOrder` field uses **ascending** order (lower numbers appear first). The default is `0`. Forgetting to order by `sortOrder` in the listing endpoint would defeat the purpose of the field.

### 8.6 Putting Server Logic in Client Components

The `ServiceTable` and `ServiceForm` are client components. They must not import the Prisma client or any server-only modules. They call the API via `fetch`. The page components (server components) fetch data and pass it as props.

### 8.7 Returning Passwords or Sensitive Fields in API Responses

The `Service` model does not include sensitive fields (no password, no token). However, the agent should still verify that the response mapping does not accidentally include any field that should not be exposed (e.g., if the schema evolves in a later phase to include a sensitive field, the response should omit it).

### 8.8 Calling `broadcastEvent()` Without Awaiting

`broadcastEvent()` returns `Promise<void>`. If 2.2/2.3 call it without `await`, the calling code may complete before the (no-op in Phase 2, but eventually real) broadcast is done. The agent should `await` the call. The stub is async for this reason.

### 8.9 Defining Only Some Event Types in `SseEventType`

The union must include all event types from Master Plan Section 11.2 (and the additional events defined in 2.2/2.3/Phase 3/4). If 2.1.1 defines only the ticket events, the other documents will need to extend the union — but the seam's principle is that the **signature** is locked. 2.1.1 should include the full registry of types as placeholders, even though the payloads are filled in later.

### 8.10 Using `as any` to Bypass Schema Validation

If the agent finds themselves reaching for `as any` to silence a type error in the API handler (e.g., when validating the request body), the issue is usually that the Zod schema is missing a field or the inferred type is not exported. Fix the schema or the import — never use `as any`.

### 8.11 Allowing the Form to Submit Without the Code Being Uppercased

The `code` field is constrained to uppercase alphanumeric. The form input should transform the user's input to uppercase on blur (or as they type). Forgetting this means users may type `gen` and receive a confusing validation error.

### 8.12 Not Handling the Pagination Total Count

The `GET /api/services` endpoint returns `meta.total`. The pagination controls depend on this. If the agent computes the count incorrectly (e.g., forgetting the `where` clause), the pagination will show incorrect page counts.
