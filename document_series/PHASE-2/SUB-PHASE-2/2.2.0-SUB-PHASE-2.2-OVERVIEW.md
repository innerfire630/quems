## 1. Sub-Phase Purpose

Sub-Phase 2.2 is the **customer-facing entry point** of the Smart Queue Management System. It introduces the **self-service kiosk** — the touch-screen terminal that anonymous customers use to take a queue ticket — and the **ticket generation engine** that produces each ticket. The kiosk loads the catalogue of services configured in Sub-Phase 2.1, lets the customer tap a service, calls the ticket issuance API to atomically assign the next sequential ticket number for that service, displays the ticket number plus an estimated wait time, and prints the ticket silently on a thermal printer without showing a browser dialog.

This sub-phase is the first sub-phase in the system that produces **a ticket in the customer's hand**. Everything that came before (Phase 1 foundation, Phase 1.3 RBAC, Sub-Phase 2.1 service & counter configuration) was preparation. Everything that comes after (Sub-Phase 2.3 call/recall/no-show/reset, Phase 3 display, Phase 4 notifications, Phase 5 reports) consumes the tickets produced here.

The three problems this sub-phase solves, in order:

1. **Generating tickets safely** — The `Ticket` model already exists in the database from Phase 1.1.3, but the `POST /api/tickets/issue` endpoint, the atomic `Service.currentTicketNumber` increment, the ticket number formatting (prefix + zero-padded sequence), the estimated wait time calculation, the business date assignment, and the `TicketEvent` audit row are all implemented in this sub-phase. (Document 2.2.1.)
2. **Showing the kiosk to customers** — The full-screen kiosk UI, the service selection grid, the confirmation screen with the issued ticket, and the auto-reset inactivity timer that returns the kiosk to the home screen after a configured timeout. (Document 2.2.2.)
3. **Printing the ticket silently** — The hidden iframe + dedicated printable ticket template strategy, the print CSS targeting 80mm and 58mm thermal paper, the post-issuance trigger mechanism, and the fallback path when printing fails. (Document 2.2.3.)

When this sub-phase is complete, a customer can walk up to a kiosk, tap a service, watch the system print a unique ticket with a known number and an estimated wait, and step aside to wait. The system is not yet announcing the ticket to a display (Phase 3), notifying officers (Phase 4), or reporting on the day (Phase 5), but the issuance path — the entire chain from a tap on the screen to a printed ticket in the customer's hand — is fully working.

---

## 1.1 Why This Sub-Phase Comes After 2.1

Sub-Phase 2.2 cannot function without services and counters configured:

- A ticket must reference a `Service` (from 2.1.1). The kiosk's service grid shows the catalogue of active services.
- The atomic ticket number sequence lives on `Service.currentTicketNumber` (the field is defined and seeded in 2.1.1; this sub-phase owns the increment).
- A ticket is technically eligible to be called only by a counter that handles the ticket's service (validated in Sub-Phase 2.3, but the dependency is set here).
- The estimated wait time calculation uses `Service.averageServiceTime` (an optional field from 2.1.1).

Sub-Phase 2.1 produces the **service catalogue** that the kiosk displays and the **ticket number sequence** that the issuance endpoint increments. Without 2.1, 2.2 has nothing to show the customer and nothing to increment.

---

## 1.2 What This Sub-Phase Is NOT

- It is **not** the ticket call/recall/no-show lifecycle. That is Sub-Phase 2.3.
- It is **not** the real-time display board, the audio announcement system, or any SSE consumer. Phase 3.
- It is **not** the push notification pipeline or counter status toggle. Phase 4.
- It is **not** the daily reset of the per-service ticket counters. Sub-Phase 2.3.3.
- It is **not** the officer-side dashboard. Phase 4.2.3.
- It is **not** reporting or analytics. Phase 5.
- It is **not** rate limiting, security headers, or production hardening. Phase 5.

This sub-phase **only** does issuance: a customer taps a service, a ticket is created, the ticket is shown to the customer, the ticket is printed, and the kiosk resets. The ticket then sits in the database in `WAITING` state until Sub-Phase 2.3 takes over.

## 2. Sub-Phase Scope and Boundaries

### 2.1 In Scope

| Area | What Sub-Phase 2.2 Delivers |
|---|---|
| Ticket issuance API | `POST /api/tickets/issue` — the only unauthenticated ticket-creation endpoint in the system |
| Atomic ticket number sequencing | The `Service.currentTicketNumber` field is atomically incremented inside the same database transaction that creates the new `Ticket` record |
| Ticket number formatting | The full `ticketNumber` (prefix + zero-padded sequence, e.g. `A001`) is computed from the service's `ticketPrefix` and the incremented integer |
| Queue position calculation | The new ticket's `waitPosition` is set to (count of `WAITING` tickets for the same service) + 1 at the moment of issuance |
| Estimated wait time calculation | `estimatedWaitMinutes = waitPosition * service.averageServiceTime` (or a system default when the service's `averageServiceTime` is null) |
| Business date assignment | Each ticket is stamped with the business date (a configurable system timezone defaulting to the server local time) so the daily reset can group tickets correctly |
| `TicketEvent` audit record | Every issuance writes an `ISSUED` `TicketEvent` row in the same transaction as the ticket creation |
| SSE event emission (stub) | On issuance, the locked-in `TICKET_ISSUED` SSE event payload is broadcast via `broadcastEvent()` (which is a no-op in Phase 2) |
| Kiosk UI — service grid | Full-screen kiosk interface displaying the active services as tappable cards, filtered by the kiosk's `restrictedServiceIds` if set |
| Kiosk UI — confirmation screen | After issuance, shows the issued ticket number, the service name, the estimated wait time, and the counter guidance text |
| Kiosk UI — auto-reset timer | After `KioskConfig.autoResetSeconds` of inactivity, the kiosk returns to the home (service grid) screen |
| Kiosk configuration loading | The kiosk page reads its `KioskConfig` by URL query (`?kioskId=...`) or falls back to the default kiosk config |
| Silent ticket printing | After issuance, a hidden iframe is mounted with the printable ticket template, `print()` is invoked, and the iframe is unmounted on completion or error |
| Print CSS for thermal paper | `@page { size: 80mm auto; margin: 4mm; }` plus a 58mm variant; styled for 80mm default and 58mm fallback |
| Print failure fallback | If the print promise rejects, the kiosk displays a clear fallback message ("Show this screen to the counter") and the ticket remains valid in the database |
| Audio file prerequisite | Place `bell.mp3` in `public/sounds/` so Phase 3 does not need to add it later (this sub-phase does not play audio) |

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to **distribute the complexity** across the three task documents. The split is:

- **2.2.1** owns the **ticket generation engine and queue logic** — the API endpoint, the atomic increment, the ticket number formatting, the wait time calculation, the business date concept, the `TicketEvent` audit row, and the `TICKET_ISSUED` SSE event payload definition. This document is the only one that touches the `Ticket` and `TicketEvent` tables in this sub-phase.
- **2.2.2** owns the **kiosk user-facing UI** — the full-screen kiosk layout, the service grid, the confirmation screen, the auto-reset timer, the kiosk configuration loading, and the routing for multiple kiosk instances. This document does not implement the silent printing flow (owned by 2.2.3) but does **trigger** the print after a successful issuance.
- **2.2.3** owns the **silent ticket printing flow** — the hidden iframe strategy, the print CSS, the `PrintTicket` component, the trigger mechanism post-issuance, the fallback UX, and the `bell.mp3` static asset placement. The kiosk confirmation screen in 2.2.2 mounts the print iframe via a component defined in 2.2.3.

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 2.2.1 — Ticket Generation Engine & Queue Logic

**Owns:** The `Ticket` issuance business logic. The `Ticket` and `TicketEvent` Prisma models already exist from Phase 1.1.3; this document implements the API endpoint and business logic that operates on them.

**Does NOT own:** The kiosk UI (owned by 2.2.2), silent printing (owned by 2.2.3), the call/recall/no-show endpoints (owned by 2.3.x), the daily reset (owned by 2.3.3).

**Key outputs:**

- `src/lib/schemas/ticket.schema.ts`: Zod schemas for ticket issuance (`issueTicketSchema`, `ticketNumberQuerySchema` for read endpoints).
- `src/types/ticket.types.ts`: TypeScript types for ticket API request/response shapes — `IssueTicketInput`, `IssuedTicketResponse`, `TicketDetail`, `TicketListItem`, `TicketEventLog`.
- `src/lib/ticket-service.ts`: the business logic module — pure functions for `formatTicketNumber(prefix, sequence)`, `calculateEstimatedWaitMinutes(waitPosition, averageServiceTime)`, `getCurrentBusinessDate(now?)`, `calculateWaitPosition(serviceId)`, plus the orchestrating `issueTicket(input)` function that wraps the transaction. **This is the module that owns the atomic `Service.currentTicketNumber` increment.**
- API endpoint: `POST /api/tickets/issue` (and the read endpoints `GET /api/tickets` queue list and `GET /api/tickets/[ticketId]` detail that the kiosk / officer surfaces consume).
- **Ticket number generation algorithm:**
  - `ticketNumber` = `<Service.ticketPrefix>` + `<zero-padded displayNumber>`, where the padding width is 3 by default and expands to 4 when the daily count exceeds 999.
  - `displayNumber` = the integer sequence (1, 2, 3, …).
  - `Service.currentTicketNumber` is atomically incremented within the same `$transaction` that creates the `Ticket` row — no other write touches this field except the daily reset (2.3.3).
- **Wait position calculation:** `waitPosition = (count of `WAITING` tickets for the same `serviceId` within the current business date) + 1`. Calculated inside the transaction with a row lock to prevent concurrent duplicate positions.
- **Estimated wait time formula:**
  - Default: `estimatedWaitMinutes = waitPosition * service.averageServiceTime` (rounded up to the nearest integer).
  - Fallback: if `service.averageServiceTime` is null, use a `SystemSetting` named `queue.default_average_service_time_minutes` (default value: 5 minutes). The `SystemSetting` lookup is performed by this document but the seeding of the default setting is owned by 2.3.3.
- **Business date:** the business date is the date the ticket belongs to, used for daily grouping. Implementation: `businessDate` is a `DateTime` set to midnight (00:00:00.000) of the day the ticket was issued, in the system's configured timezone (`APP_TIMEZONE` env var, defaulting to the server's local timezone). The helper `getCurrentBusinessDate()` returns this value. The document specifies the exact timezone resolution.
- **`TicketEvent` audit row:** every issuance creates a `TicketEvent` with `eventType = 'ISSUED'`, `ticketId` set, `counterId = null`, `officerId = null`, `metadata = { businessDate, waitPosition, estimatedWaitMinutes }`, all in the same transaction.
- **`TICKET_ISSUED` SSE event payload:** the payload type is **fully defined** in this document in `src/types/sse.types.ts` (extending the discriminated union seeded by 2.1.1). Fields: `{ ticketId, ticketNumber, displayNumber, serviceId, serviceName, issuedAt, businessDate, waitPosition, estimatedWaitMinutes, priority }`. The event is emitted via `broadcastEvent('global', 'TICKET_ISSUED', payload)` — the function is the no-op stub from 2.1.1.
- **Read endpoints** consumed by the kiosk / officer:
  - `GET /api/tickets` — list the current queue (filtered by service, counter, status). Paginated. Guarded by `ticket:view`.
  - `GET /api/tickets/[ticketId]` — get a single ticket detail with its event log. Guarded by `ticket:view`.
- **Audit log policy for issuance:** ticket issuance does **NOT** write to the `AuditLog` table. The `TicketEvent` row is the complete audit trail for the ticket. (Per Phase 2 cross-cutting standard 8.4.)
- **Route protection:** the `POST /api/tickets/issue` endpoint is **public** (no auth required, no permission check) but is rate-limited by IP — Phase 5 will implement the rate limiter, but the route is structured so it can be rate-limited without modification. The read endpoints (`GET`) require `PERMISSION_TICKET_VIEW`.

**Master Plan sections implemented:** Sections 6.5 (`TicketBadge`, `StatusChip`), 8.2 (`Ticket` and `TicketEvent` models), 9.3 (ticket issuance and read endpoints), 11.2 (SSE envelope structure), 11.3 (`TICKET_CALLED` payload format — applied here for `TICKET_ISSUED`), 17 (decisions on issuance atomicity).

### 4.2 Document 2.2.2 — Kiosk UI & Service Selection Flow

**Owns:** The kiosk page UI — the full-screen customer-facing interface, the service selection grid, the confirmation screen, the auto-reset inactivity timer, and the kiosk configuration loading.

**Does NOT own:** The ticket issuance endpoint (owned by 2.2.1), the silent printing flow (owned by 2.2.3), any authentication for the kiosk (it remains anonymous), the kiosk configuration CRUD admin UI (out of scope for this sub-phase; admin UI for `KioskConfig` is a future enhancement).

**Key outputs:**

- `src/app/kiosk/page.tsx`: the full-screen kiosk page. Supports an optional `?kioskId=<id>` URL query; falls back to the default `KioskConfig` (where `isDefault = true` and `isActive = true`).
- `src/app/kiosk/layout.tsx`: a minimal kiosk layout — no sidebar, no top bar, full-screen white background, 32px padding per Master Plan Section 6.4.
- `src/lib/kiosk-config.ts`: a server-side helper that loads the active `KioskConfig` by ID (or the default), caches it briefly for performance, and returns it as a typed object. Returns a fallback `KioskConfig` (or throws a controlled error displayed in the kiosk UI) if no active config is found.
- `src/hooks/useKioskReset.ts`: a client-side hook that starts a timer for `autoResetSeconds` and resets the kiosk to the home screen on inactivity. "Inactivity" is defined as no user interaction (tap, scroll, keypress, touch) and no in-flight API call. The hook exposes a `reset()` function the UI can call to explicitly reset (e.g., after a successful issuance, the confirmation screen schedules a reset).
- UI components:
  - `src/app/kiosk/_components/kiosk-home.tsx` — the service selection grid (client component).
  - `src/app/kiosk/_components/service-card.tsx` — a single tappable service card.
  - `src/app/kiosk/_components/kiosk-service-grid.tsx` — the grid wrapper (handles loading and empty states).
  - `src/app/kiosk/_components/ticket-confirmation.tsx` — the post-issuance confirmation screen (mounts the print iframe via the `PrintTicket` component from 2.2.3, schedules the auto-reset).
  - `src/app/kiosk/_components/kiosk-no-config.tsx` — the fallback message shown when no active kiosk config is found.
- **Service grid behaviour:**
  - Displays only services where `isActive = true`.
  - If `KioskConfig.restrictedServiceIds` is set, only those services are displayed (intersected with `isActive = true`).
  - Sorted by `Service.sortOrder` ascending, then by `name` ascending.
  - Each card shows: service `iconName` (rendered via the lucide-react icon component), service `name`, service `color` (as a left accent border), and an optional short description.
  - Layout: 2 columns on tablet (>= 768px wide), 1 column on phone, large tap targets (minimum 48px tall, typically 120px+ for the card body).
- **Confirmation screen behaviour:**
  - Shows the ticket number (rendered very large, monospace, bold).
  - Shows the service name.
  - Shows the estimated wait time (if `KioskConfig.showEstimatedWait` is true).
  - Shows the counter guidance text: "Please wait to be called" (configurable via `KioskConfig.footerMessage`).
  - Mounts the `PrintTicket` component (from 2.2.3) into a hidden iframe and triggers the silent print.
  - Schedules an auto-reset via `useKioskReset` to return to the home screen after `autoResetSeconds`.
- **Auto-reset timer:**
  - On any tap, touch, keypress, or scroll, the timer is reset to `autoResetSeconds`.
  - During an in-flight issuance API call, the timer is paused (so a slow response does not cause a premature reset).
  - On the confirmation screen, the timer is shorter (or fixed at `autoResetSeconds`) so the next customer can use the kiosk promptly.
  - The kiosk returns to the home screen (`/kiosk`) when the timer expires, or immediately when a "Done" button is tapped.
- **No-auth behaviour:** the kiosk page is public (no auth, no middleware enforcement). The `POST /api/tickets/issue` endpoint is also public (as documented in 2.2.1). The page does not check the session.

**Master Plan sections implemented:** Sections 6.4 (kiosk layout), 6.5 (kiosk components), 8.2 (`KioskConfig` model), 9.3 (ticket issuance from the kiosk's perspective).

### 4.3 Document 2.2.3 — Silent Ticket Printing Implementation

**Owns:** The silent printing strategy, the `PrintTicket` component, the print CSS targeting 80mm and 58mm thermal paper, the trigger mechanism, the fallback UX when printing fails, and the placement of `bell.mp3` as a static asset (the asset is not played by this sub-phase; it's staged for Phase 3.3.1).

**Does NOT own:** The kiosk page layout and the service grid (owned by 2.2.2), the ticket issuance API (owned by 2.2.1), the audio playback logic (Phase 3.3).

**Key outputs:**

- `src/components/kiosk/print-ticket.tsx`: the printable ticket template component. Receives the ticket data as props. Renders the ticket using a dedicated print stylesheet (NOT the global Tailwind utility classes — print styles are scoped).
- `src/app/kiosk/_components/silent-print-trigger.tsx`: a client component that, after the confirmation screen mounts, injects a hidden `<iframe>` into the DOM, renders `<PrintTicket>` inside the iframe via a same-origin document write or React portal into the iframe's document, waits for the iframe content to load, and calls `iframe.contentWindow.print()`.
- `src/app/globals.css` (or a new `src/styles/print.css`): the print CSS — `@page { size: 80mm auto; margin: 4mm; }` and a `@media print` block that resets all non-essential styling and renders the ticket for 80mm thermal paper. A 58mm variant is defined and selected via a CSS media query or a `KioskConfig` setting (`paperWidth` enum of `80MM` or `58MM`, default `80MM`).
- `public/sounds/bell.mp3`: a placeholder MP3 file (or a real chime sound if the team provides one — the document specifies a path and naming convention; Phase 3.3.1 may replace the file with the real chime).
- `public/print-ticket.html` is **NOT** created. The print iframe is dynamically generated in JavaScript and removed after printing. This avoids the document from going stale.
- **Silent printing strategy:**
  - The iframe is `display: none` on the host page (so the customer does not see a second screen flash).
  - The iframe is `sandbox="allow-same-origin allow-modals allow-popups-to-escape-sandbox"` (or a narrower set per the document's exact specification). The agent must specify the exact sandbox attributes.
  - The iframe's content is a fully styled HTML document with the print CSS applied.
  - `iframe.contentWindow.print()` is called after the iframe's `onload` fires (or `iframe.onload` is the print trigger directly).
  - Chrome's `--kiosk-printing` flag combined with an OS-level default printer suppresses the print dialog.
- **Print CSS specifics:**
  - `@page { size: 80mm auto; margin: 4mm; }` (default).
  - `@page { size: 58mm auto; margin: 3mm; }` (variant when `paperWidth = '58MM'`).
  - Reset all margins, padding, and borders to zero on `@media print` (so the ticket fills the printable area).
  - Font: a single sans-serif stack with explicit `font-family`, `font-size`, `line-height` for predictable rendering.
  - Ticket number: very large (e.g., 48px), bold, monospace (or a print-safe tabular figure font).
  - Service name: medium-large (e.g., 18px), bold.
  - Estimated wait time (if `showEstimatedWait`): medium (14px), regular weight.
  - Footer (issue datetime, guidance text): small (10–11px), regular weight.
  - Reserved space for a future QR code (a placeholder `<div>` with fixed dimensions, currently empty).
- **Trigger mechanism (in the confirmation screen, owned by 2.2.2):**
  - The confirmation screen mounts the silent-print component after the ticket issuance API call resolves successfully.
  - The silent-print component creates the iframe, renders the ticket, and calls `print()`.
  - The silent-print component listens for the `afterprint` event (or a timeout fallback of e.g. 5 seconds) and unmounts the iframe.
- **Fallback behaviour when printing fails:**
  - The `print()` call may reject, the iframe may fail to load, or the `afterprint` event may not fire within the timeout.
  - On any failure, the kiosk displays a yellow warning banner ("Print failed — please show this screen to the counter") with a "Reprint" button (re-mounts the iframe and retries) and a "Done" button (returns to the home screen).
  - The ticket remains valid in the database; the customer can present the on-screen ticket number at the counter if printing fails.

**Master Plan sections implemented:** Sections 6.5 (`PrintTicket` component), 8.2 (`KioskConfig` model usage), 12 (bell.mp3 file location), 13 (silent printing architecture).

```

                                                    ▼
2.2.3  Silent Ticket Printing Implementation     ── (depends on 2.2.2; PrintTicket mounted by 2.2.2)
```

**Build order:** 2.2.1 first (it produces the issuance API and the locked-in payload type). 2.2.2 second (it consumes 2.2.1's API and the kiosk components). 2.2.3 third (its `PrintTicket` component is mounted by 2.2.2, so 2.2.3's exports must exist when 2.2.2 builds the confirmation screen).

**Why this order:**

- **2.2.1 first.** The ticket issuance endpoint and its business logic are foundational — the kiosk UI in 2.2.2 calls this endpoint, and the SSE event payload type is locked here for Phase 3 to consume.
- **2.2.2 after 2.2.1.** The kiosk UI cannot confirm a ticket before the API exists. However, 2.2.2 can be developed in parallel with 2.2.3 once 2.2.1 is complete (2.2.2's confirmation screen imports 2.2.3's `PrintTicket` component, so 2.2.3 must export the component signature before 2.2.2 builds the screen).
- **2.2.3 after 2.2.1, can be developed in parallel with 2.2.2.** The `PrintTicket` component and the silent-print trigger are independent of the kiosk layout. The two documents share the API surface (the `PrintTicket` component's props) but not the implementation.

**What depends on this sub-phase:**

- **Sub-Phase 2.3.1** reads the `Ticket` records and the `TicketEvent` log produced by 2.2.1. The call/recall endpoints update tickets created here.
- **Sub-Phase 2.3.2** reads the `Ticket` records produced here. The no-show endpoint operates on the same model.
- **Sub-Phase 2.3.3** iterates over all services and resets their `currentTicketNumber` (the field incremented by 2.2.1). The `QueueDailySnapshot` archive includes the tickets issued here.
- **Phase 3 display board** consumes the `TICKET_ISSUED` event (and subsequent lifecycle events) via SSE.
- **Phase 4 push notification system** is triggered by the `TICKET_ISSUED` event for officers whose counters handle the ticket's service.
- **Phase 5 reporting dashboard** queries the historical tickets and `TicketEvent` records produced here.

## 6. Cross-Document Standards

These standards apply to every document in Sub-Phase 2.2. The conventions from Phase 1 and Phase 2.1 (folder naming, import paths, TypeScript, Zod validation, error handling, env vars, git commits, audit logging, broadcast event seam) all carry forward unchanged. The standards below are **new or specific to Sub-Phase 2.2**.

### 6.1 Folder and File Naming

- All folders use kebab-case for multi-word names.
- Component files use PascalCase (`TicketConfirmation.tsx`, `KioskServiceGrid.tsx`, `PrintTicket.tsx`).
- Non-component modules use camelCase (`ticket-service.ts`, `kiosk-config.ts`).
- Schema files use the pattern `*.schema.ts` (e.g., `src/schemas/ticket.schema.ts`).
- Type files use `*.types.ts`.
- The kiosk pages live under `src/app/kiosk/` (NOT under the `(dashboard)` route group — the kiosk is unauthenticated).
- The `_components` sub-folder under `src/app/kiosk/` mirrors the existing pattern from `src/app/(dashboard)/services/_components/`.

### 6.2 Import Conventions

- All project-internal imports use the `@/` alias.
- The `broadcastEvent` function is imported from `@/lib/events` (from 2.1.1).
- The `SseChannel`, `SseEventType`, `SseEventPayload` types are imported from `@/types/sse.types` (from 2.1.1).
- The `writeAuditLog` function is imported from `@/lib/audit-log` (from Phase 1.3.3). **Not used by 2.2.1 or 2.2.2** — issuance and kiosk actions do not write to the `AuditLog` table per Phase 2 cross-cutting standard 8.4.
- The `withPermission` guard is imported from `@/lib/guards` (from Phase 1.3.2). **Used by the read endpoints in 2.2.1** (which require `ticket:view`) but NOT by `POST /api/tickets/issue` (which is public).
- The `db` Prisma client is imported from `@/lib/db` (from Phase 1.1.3).
- The `auth()` helper is imported from `@/lib/auth` (from Phase 1.2.1) for read endpoints that need session data.

### 6.3 TypeScript Standards

- All exported functions and components have explicit type annotations.
- No `any` types in production code.
- The `Ticket` and `TicketEvent` Prisma types are re-exported as types from `@/types/ticket.types.ts` for use in client components (so client components do not import the Prisma client directly).
- The `SseEventPayload` discriminated union (defined in 2.1.1's `src/types/sse.types.ts`) is **extended** in 2.2.1 to include the `TICKET_ISSUED` payload variant. The extension is additive — 2.1.1's existing variants are preserved.

### 6.4 Response Envelope Standard

Every API route in this sub-phase must return the master plan's standard envelope and use the standard error codes (Master Plan Sections 9.1 and 9.2):

| Code | HTTP Status | When in this sub-phase |
|---|---|---|
| `UNAUTHORIZED` | 401 | No valid auth on the read endpoints |
| `FORBIDDEN` | 403 | Authenticated but lacks `ticket:view` on the read endpoints |
| `NOT_FOUND` | 404 | Ticket ID does not exist (on `GET /api/tickets/[ticketId]`) |
| `VALIDATION_ERROR` | 422 | Request body fails Zod validation; or the service is inactive; or the ticket number overflow check fails |
| `CONFLICT` | 409 | Race condition on atomic increment (extremely rare with Prisma's `increment`; mapped here for safety) |
| `RATE_LIMITED` | 429 | Issued by the Phase 5 rate limiter; the route is structured to accept it without modification |
| `INTERNAL_ERROR` | 500 | Unexpected server error |

### 6.5 Validation Standards

- Every API request body has a Zod schema in `src/schemas/`.
- The schemas are reusable across the client and the server where applicable (issuance is server-only, but the response shapes are typed for the client).
- Schemas for this sub-phase:
  - `src/schemas/ticket.schema.ts`: `issueTicketSchema`, `ticketNumberQuerySchema`, `listTicketsQuerySchema`, `getTicketByIdParamsSchema`.

### 6.6 Error Handling Standards

- Server actions and API routes return the master plan's standard envelope.
- Client-side error display uses shadcn `Toast` for transient errors and inline alerts for form-field errors.
- The kiosk UI uses a single full-screen error state for any unrecoverable error (e.g., "Issuance failed — please try again" with a "Try Again" button).
- Every API route is wrapped in a try/catch that logs the error (in development) and returns `INTERNAL_ERROR` (HTTP 500) for unhandled exceptions.

### 6.7 Audit Logging Standard (Sub-Phase 2.2 Specific)

- **Ticket issuance does NOT write to the `AuditLog` table.** The `TicketEvent` row is the complete audit trail for ticket lifecycle events. (Per Phase 2 cross-cutting standard 8.4.)
- **No administrative actions** happen in this sub-phase (no service creation, no counter creation, no user creation, no settings changes). So no `AuditLog` writes occur in this sub-phase at all.
- The `AuditAction` union in `@/lib/audit-log.ts` does **not** need extension for this sub-phase.

### 6.8 Broadcast Event Seam (Sub-Phase 2.2 Specific)

- 2.2.1 calls `broadcastEvent('global', 'TICKET_ISSUED', payload)` after a successful issuance.
- 2.2.1 **extends** `src/types/sse.types.ts` (the file from 2.1.1) to add the `TICKET_ISSUED` payload type to the discriminated union. The extension is additive.
- 2.2.2 and 2.2.3 do **not** call `broadcastEvent()`. They are UI-layer documents that consume 2.2.1's API.

### 6.9 Environment Variable Standards

- New env vars introduced in this sub-phase:
  - `APP_TIMEZONE` — the system's timezone for business date computation. Default: the server's local timezone (no env var required). If set (e.g., `APP_TIMEZONE=Asia/Colombo`), it overrides the default. The env validator in `src/lib/env.ts` (from Phase 1.1.1) is extended to include this optional variable.
- The `KioskConfig` table (from Master Plan Section 8.2) stores per-kiosk configuration; no env vars are required for kiosk configuration.

### 6.10 Commit Standards

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`.
- Each task plan document corresponds to one or more commits. A single commit should not span multiple task plan documents.
- Suggested commit boundaries:
  - End of 2.2.1: `feat: add ticket issuance engine with atomic increment and TICKET_ISSUED event`
  - End of 2.2.2: `feat: add kiosk UI with service grid, confirmation screen, and auto-reset timer`
  - End of 2.2.3: `feat: add silent ticket printing with hidden iframe and thermal print CSS`

### 6.11 Print-Specific Conventions (Sub-Phase 2.2 Specific)

- The print iframe is created and destroyed dynamically by JavaScript. There is no `public/print-ticket.html` static file.
- The print CSS is scoped — it does not use Tailwind utility classes inside the iframe's document. Plain CSS (or CSS modules) is used to keep the print output predictable across browsers and printers.
- The `bell.mp3` file is placed in `public/sounds/` as a small placeholder (e.g., a 1-second silent MP3 or a real chime if the team has one). It is **not played** by this sub-phase. Phase 3.3.1 may replace it with the production chime. The placeholder must be a valid MP3 file (even if silent) so Phase 3 does not need to add a binary file later.

### 6.12 Transaction Boundaries (Sub-Phase 2.2 Specific)

- `issueTicket` (the core issuance function in `lib/ticket-service.ts`) wraps **exactly one** `$transaction` block that contains:
  1. The atomic `Service.currentTicketNumber` increment.
  2. The wait position count query.
  3. The `Ticket` creation.
  4. The `TicketEvent` creation.
- All four operations are in one Prisma `$transaction`. The broadcast event emission (`broadcastEvent('global', 'TICKET_ISSUED', ...)`) is **outside** the transaction (after the transaction commits successfully) — the event must not be emitted if the transaction rolls back.

## 7. Sub-Phase Exit Criteria

Sub-Phase 2.2 is complete when **all** of the following are true:

### 7.1 Ticket Issuance Engine (from 2.2.1)

- [ ] `POST /api/tickets/issue` accepts a body with `serviceId` and returns a fully populated `Ticket` object.
- [ ] The `Service.currentTicketNumber` field is atomically incremented inside the same transaction that creates the `Ticket`.
- [ ] Two concurrent issuance requests produce sequential `displayNumber` values (e.g., 1 and 2), never duplicates.
- [ ] The `ticketNumber` is correctly formatted as `<prefix><zero-padded sequence>` (e.g., `A001`, `A002`, …, `A999`, `A1000`).
- [ ] The `waitPosition` field is set to the count of currently waiting tickets for the service + 1 at issuance time.
- [ ] The `estimatedWaitMinutes` is correctly calculated using the formula and the service's `averageServiceTime` (or the system default when null).
- [ ] The `businessDate` field is set to midnight of the day of issuance in the configured timezone.
- [ ] An `ISSUED` `TicketEvent` row is created in the same transaction.
- [ ] A `TICKET_ISSUED` SSE event is emitted via `broadcastEvent()` after the transaction commits (Phase 3 will deliver; Phase 2 emits a no-op stub).
- [ ] `GET /api/tickets` returns a paginated queue list filtered by service / counter / status.
- [ ] `GET /api/tickets/[ticketId]` returns a single ticket with its event log.
- [ ] All read endpoints are guarded with `PERMISSION_TICKET_VIEW`.
- [ ] The issuance endpoint is **not** guarded by `withPermission()` (kiosks are anonymous) but is structured for Phase 5's rate limiter.
- [ ] No `AuditLog` rows are written by issuance (per Phase 2 cross-cutting standard 8.4).

### 7.2 Kiosk UI (from 2.2.2)

- [ ] `GET /kiosk` renders the kiosk layout (full-screen, white background, 32px padding).
- [ ] `GET /kiosk?kioskId=<id>` loads the matching `KioskConfig`.
- [ ] When no `kioskId` is provided, the default `KioskConfig` (`isDefault = true`, `isActive = true`) is used.
- [ ] When no active `KioskConfig` is found, a fallback message is shown.
- [ ] The service grid shows only `isActive = true` services, optionally filtered by `restrictedServiceIds`, sorted by `sortOrder` then `name`.
- [ ] Tapping a service card calls `POST /api/tickets/issue` with the `serviceId`.
- [ ] On success, the confirmation screen shows the ticket number (large, monospace, bold), the service name, the estimated wait (if enabled), and the counter guidance text.
- [ ] On error, the kiosk shows a clear error state with a "Try Again" button.
- [ ] After `autoResetSeconds` of inactivity on the confirmation screen, the kiosk returns to the home screen.
- [ ] Any tap, touch, keypress, or scroll during the home screen resets the inactivity timer.
- [ ] The kiosk does not require authentication (no session, no redirect).

### 7.3 Silent Ticket Printing (from 2.2.3)

- [ ] After a successful issuance, the confirmation screen mounts the `PrintTicket` component into a hidden iframe.
- [ ] The iframe triggers `print()` after its content loads.
- [ ] The print CSS targets 80mm thermal paper by default (`@page { size: 80mm auto; margin: 4mm; }`).
- [ ] A 58mm variant is defined and selectable via `KioskConfig.paperWidth` (or a similar configuration field).
- [ ] In Chrome kiosk mode (`--kiosk-printing` flag) with a configured default printer, no browser print dialog appears.
- [ ] If the print fails (iframe fails to load, `print()` rejects, or `afterprint` does not fire within timeout), a fallback message is shown with a "Reprint" and "Done" button.
- [ ] The `bell.mp3` file is present at `public/sounds/bell.mp3` (placeholder OK).
- [ ] The print iframe is removed from the DOM after printing completes (success, failure, or timeout).

### 7.4 Event Emission (Phase 3 Hand-off)

- [ ] `src/types/sse.types.ts` has been extended with the `TICKET_ISSUED` payload variant (additive extension, no existing variants removed).
- [ ] `broadcastEvent('global', 'TICKET_ISSUED', payload)` is called from the issuance endpoint after the transaction commits.
- [ ] In development, the `broadcastEvent()` stub logs the payload to `console.debug` so the call site can be verified.

### 7.5 Code Quality

- [ ] All multi-write operations (issuance) are inside a single Prisma `$transaction`.
- [ ] The event emission is **outside** the transaction.
- [ ] `yarn lint`, `yarn type-check`, and `yarn build` all pass.
- [ ] No `any` types exist in the ticket, kiosk, or print modules.

## 8. AI Agent Implementation Guidance

This section is for the AI agents that will implement the task plan documents. It is guidance that applies across all three task plan documents in this sub-phase.

### 8.1 How to Use the Task Plan Documents

Each task plan document follows the same internal structure:

1. **Document Purpose** — what this document delivers.
2. **Scope and Boundaries** — what this document owns and does not.
3. **Pre-requisites** — what must already be true (previous documents completed).
4. **Step-by-step implementation plan** — the actual work, in order.
5. **File-level specifications** — for every file the agent will create or modify.
6. **Verification checklist** — how the agent proves the work is done.
7. **Commit instructions** — what commit(s) to make at the end.
8. **Common pitfalls** — mistakes to avoid.
9. **AI agent guidance** — document-specific guidance.

The agent should follow the steps in order. Each step lists the **goal** of the step, the **files affected**, and any **specific instructions** that resolve the ambiguity the agent would otherwise have to guess about.

### 8.2 General Guidance

- **Do not skip steps.** Sub-Phase 2.2 is mostly sequential within each document (each step's output is the next step's input), but 2.2.2 and 2.2.3 can be developed in parallel after 2.2.1 is complete.
- **Do not add features beyond the document's scope.** The temptation to "just add call/recall now" or "just add the daily reset" is real — resist it. Those are owned by Sub-Phase 2.3.
- **Verify after every step.** The verification checklist at the end of each document is a hard gate.
- **When the master plan is ambiguous, choose the simpler option.** If a configuration choice has two reasonable interpretations, pick the one that is easier to change later.
- **Commit at the end of each document** using the commit message specified in the document.
- **Document any deviation.** If the agent makes a choice that differs from this document, it must note the deviation in the final summary so the human reviewer can approve or reject it.
- **Always use the master plan's response envelope.** Every API route must return `{ success, data | error }`.
- **Always use the master plan's standard error codes.** When in doubt about the right code, refer to Master Plan Section 9.2.
- **Always use `withPermission()` for read endpoints** that require auth. The issuance endpoint does NOT use `withPermission()` (kiosks are anonymous).
- **Use `broadcastEvent()` from 2.1.1** for the `TICKET_ISSUED` event. The function is the locked-in seam; the agent must not reimplement it.
- **Wrap multi-write operations in a single Prisma `$transaction`.** This is non-negotiable.

### 8.3 What the AI Agent Should NOT Do

- Should not implement ticket call/recall/no-show endpoints (owned by 2.3.x).
- Should not implement the daily reset (owned by 2.3.3).
- Should not implement the SSE manager or any real SSE delivery (owned by Phase 3).
- Should not implement the display board UI (owned by Phase 3).
- Should not implement the audio announcement system (owned by Phase 3).
- Should not implement push notifications (owned by Phase 4).
- Should not implement the counter temporary-closure toggle (owned by Phase 4).
- Should not implement the officer dashboard layout (owned by Phase 4).
- Should not implement reporting or analytics (owned by Phase 5).
- Should not implement rate limiting, security headers, or production hardening (owned by Phase 5).
- Should not write to the `AuditLog` table from this sub-phase (issuance does not log there — the `TicketEvent` row is sufficient per Phase 2 cross-cutting standard 8.4).
- Should not modify the `Service.currentTicketNumber` field outside of the issuance endpoint and the daily reset.
- Should not emit events before the transaction commits. The broadcast event call must be outside the `$transaction` block.

### 8.4 When to Ask the Human

The AI agent should ask the human reviewer (not guess) if any of the following situations arise:

- The master plan and the phase 2 overview contradict each other. The master plan wins, but the agent should confirm.
- The exact zero-padding width for ticket numbers beyond 999 is ambiguous (the document specifies 4 digits; the agent should confirm if there is any doubt).
- The business date timezone (`APP_TIMEZONE`) value for the deployment environment is not specified (the agent should default to server local time and document the choice).
- The system default for `averageServiceTime` (when null) is unclear (the document specifies 5 minutes; the agent should confirm).
- The print iframe sandbox attributes are ambiguous (the document specifies a baseline set; the agent should confirm if more restrictive or permissive attributes are required).
- The verification step fails for a reason that the document does not anticipate.
