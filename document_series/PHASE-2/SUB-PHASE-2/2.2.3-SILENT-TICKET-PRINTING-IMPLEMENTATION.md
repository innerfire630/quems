## 1. Document Purpose

This document delivers the **silent ticket printing flow** end-to-end: the `PrintTicket` component that renders the printable ticket template, the hidden iframe strategy that triggers `window.print()` without showing a browser dialog, the print CSS targeting 80mm and 58mm thermal paper, the trigger mechanism (mounted by the kiosk confirmation screen in 2.2.2), the fallback UX when printing fails, and the placement of `bell.mp3` as a static asset (the asset is staged for Phase 3.3.1 but is **not played** by this sub-phase).

When this document is complete:

- After a customer taps a service on the kiosk and the ticket is issued, the kiosk confirmation screen mounts the `PrintTicket` component inside a hidden iframe.
- The iframe triggers `print()` after its content loads, producing a thermal-printed ticket on the configured default printer (in Chrome kiosk mode with `--kiosk-printing`).
- The print CSS is tuned for 80mm thermal paper (`@page { size: 80mm auto; margin: 4mm; }`) with a 58mm variant selectable via the kiosk's `paperWidth` setting.
- If the print fails (iframe fails to load, `print()` rejects, or `afterprint` does not fire within a timeout), a clear fallback message is shown with "Reprint" and "Done" buttons.
- The `bell.mp3` file is placed at `public/sounds/bell.mp3` as a placeholder so Phase 3 does not need to add a binary file later.

This document **does not** implement the kiosk page UI (owned by 2.2.2), the ticket issuance API (owned by 2.2.1), the audio playback logic (Phase 3.3.1), or the display board (Phase 3.2).

### Step 1: Create the Print CSS — `src/styles/print.css` (or Extend `globals.css`)

**Goal:** Define the print stylesheet targeting 80mm and 58mm thermal paper with predictable rendering across printers.

**Files affected:** `src/styles/print.css` (new file), `src/app/layout.tsx` (modified to import the CSS).

**Specific instructions:**

Create `src/styles/print.css` containing:

- **`@page` rule for 80mm paper:**
  - `size: 80mm auto;` — width 80mm, height auto (the height is determined by the content).
  - `margin: 4mm;` — uniform 4mm margins.
  - This is the **default** style applied to the iframe document.
- **`@page` rule for 58mm paper:**
  - `size: 58mm auto;` — width 58mm.
  - `margin: 3mm;` — uniform 3mm margins.
  - This is applied when the kiosk config specifies `paperWidth = '58MM'` (the document specifies how this is selected — see Step 2).
- **`@media print` block:**
  - Reset all margins, paddings, and borders on `body`, `html`, and the ticket root element to zero (the `@page` margin handles the outer padding).
  - Set the body font to a single sans-serif stack with explicit `font-family` (e.g., `'Helvetica Neue', Helvetica, Arial, sans-serif`).
  - Set the body `font-size` to `12px` and `line-height` to `1.4`.
  - Set the body color to pure black (`#000`) and background to pure white (`#fff`).
  - Disable any `box-shadow`, `text-shadow`, or `border-radius` that may render unpredictably on thermal printers.
- **Ticket element styles:**
  - `.ticket-number`: `font-size: 48px;`, `font-weight: 900;`, `font-family: 'Courier New', Courier, monospace;`, `letter-spacing: -0.02em;`, `text-align: center;`, `margin: 8px 0;`.
  - `.service-name`: `font-size: 18px;`, `font-weight: 700;`, `text-align: center;`, `margin: 4px 0;`.
  - `.estimated-wait`: `font-size: 14px;`, `text-align: center;`, `margin: 4px 0;`.
  - `.footer-text`: `font-size: 10px;`, `text-align: center;`, `margin: 8px 0 0;`, `color: #333;` (slightly muted).
  - `.metadata`: `font-size: 10px;`, `text-align: center;`, `margin: 4px 0;`, `color: #666;` (timestamps, business date).
  - `.divider`: a single-pixel border (`border-top: 1px dashed #999;`) used to separate sections on the ticket.
  - `.qr-placeholder`: a fixed-size empty `<div>` (e.g., `width: 80px; height: 80px;`) reserved for a future QR code. Currently empty.
- **`@media print and (max-width: 58mm)` block** (or a class-based selector):
  - Reduce `.ticket-number` font size to `36px` for 58mm paper.
  - Reduce `.service-name` font size to `16px`.
  - Adjust margins to `2px 0` for tighter spacing.

In `src/app/layout.tsx`, add an import for the print CSS. **Note:** the print CSS is intended for the iframe document, NOT the main app document. The main app does not need this CSS loaded. The agent should consider whether to import the CSS globally (and rely on the browser not applying `@page` rules outside `@media print`) or to import the CSS only inside the iframe document (via dynamic creation of a `<style>` element when the iframe is created). **The recommended approach is the latter — inject the CSS into the iframe document when the iframe is created in Step 3.** The CSS file is the single source of truth for the styles, but it is loaded into the iframe, not the main app.

    - A welcome text "Welcome!" or the kiosk's `welcomeMessage`.
  - **Ticket number section:**
    - `<div className="ticket-number">{ticket.ticketNumber}</div>` — the prominent number.
    - A small label above or below: "Your number".
  - **Service section:**
    - `<div className="service-name">{ticket.serviceName}</div>` — the service name.
  - **Estimated wait section** (if `kioskConfig.showEstimatedWait` and `ticket.estimatedWaitMinutes !== null`):
    - `<div className="estimated-wait">Estimated wait: ~{ticket.estimatedWaitMinutes} min</div>`.
  - **Counter guidance section:**
    - `<div className="footer-text">{kioskConfig.footerMessage ?? 'Please wait to be called.'}</div>`.
  - **Metadata section:**
    - `<div className="metadata">Issued: {formatted issuedAt}</div>`.
    - `<div className="metadata">Position: {ticket.waitPosition}</div>`.
  - **QR placeholder section** (reserved for future QR code):
    - `<div className="qr-placeholder"></div>`.
  - **Divider:**
    - `<div className="divider"></div>` between sections as appropriate.
- **Date formatting:**
  - The `issuedAt` is formatted as `YYYY-MM-DD HH:mm` (no seconds) in the configured `APP_TIMEZONE`.
- **No client-side interactivity:**
  - The component is a pure presentational component. No buttons, no event handlers.
  - The parent (`SilentPrintTrigger`) handles the iframe lifecycle and the print invocation.

### Step 3: Create `src/app/kiosk/_components/silent-print-trigger.tsx` — The Silent Print Trigger

**Goal:** Manage the hidden iframe lifecycle and trigger the silent print after the ticket is issued.

**Files affected:** `src/app/kiosk/_components/silent-print-trigger.tsx` (new file).

**Specific instructions:**

Create the component:

- A "use client" directive (it runs in the main app context, not in the iframe).
- **Props:**
  - `ticket: IssuedTicketResponse` — the ticket to print.
  - `kioskConfig: LoadedKioskConfig` — the kiosk configuration.
  - `businessLogoUrl?: string` — optional brand logo URL.
  - `onPrintSuccess?: () => void` — optional callback when the print completes successfully.
  - `onPrintFailure?: (error: Error) => void` — optional callback when the print fails.
- **State:**
  - `printStatus: 'pending' | 'printing' | 'success' | 'failed'` — tracks the print lifecycle.
  - `errorMessage: string | null` — the error message if `printStatus = 'failed'`.
- **Behaviour (on mount):**
  1. Create a hidden `<iframe>` element and append it to `document.body`.
  2. The iframe has `style.display = 'none'` so it is not visible to the customer.
  3. The iframe has `sandbox` attributes: `allow-same-origin` is required (the iframe's `contentWindow.print()` must be callable; without `allow-same-origin`, the print call may be blocked). `allow-modals` and `allow-popups-to-escape-sandbox` are not required for the print flow. **The exact sandbox attributes are: `allow-same-origin`.** (The agent must use the minimum required attributes for security — `allow-same-origin` is necessary; `allow-scripts` is not because the React rendering uses `createRoot` which is invoked from the parent context, not from inside the iframe.)
  4. Get the iframe's `contentDocument`. Write a minimal HTML document structure into the iframe (`<!DOCTYPE html><html><head>...</head><body><div id="root"></div></body></html>`).
  5. Inject the print CSS (from `src/styles/print.css`) into the iframe's `<head>` as a `<style>` element. The CSS content is imported as a string at build time (or fetched as a static asset; the document specifies the approach — the agent uses the import approach via the bundler's CSS-as-string support).
  6. Use ReactDOM's `createRoot` to render the `<PrintTicket>` component into the iframe's `#root` element. The React tree runs inside the iframe's document context.
  7. After the iframe's content loads (wait for the `load` event on the iframe or a short delay), call `iframe.contentWindow.print()`.
  8. Listen for the `afterprint` event on the iframe's `contentWindow`. When the event fires (or after a 5-second timeout, whichever comes first), set `printStatus = 'success'`.
  9. If the timeout elapses without the `afterprint` event, set `printStatus = 'failed'` with an error message indicating the timeout.
  10. If `iframe.contentWindow.print()` throws, catch the error and set `printStatus = 'failed'`.
  11. After the print completes (success or failure), remove the iframe from `document.body` to clean up.
- **Fallback UI (when `printStatus = 'failed'`):**
  - Render a yellow warning banner using shadcn `Alert` with the warning variant.
  - Display the message: "Print failed — please show this screen to the counter."
  - Display two buttons:
    - **"Reprint"** button: re-mounts the iframe and retries the print.
    - **"Done"** button: dismisses the banner and triggers the parent's `onDone` callback (or navigates back to the home screen).
- **Cleanup:**
  - On unmount, remove the iframe if it still exists.
  - Clear any pending timeouts.

## 8. Common Pitfalls

The following are common mistakes the agent should avoid:

- **Showing a browser print dialog.** The whole point of the hidden iframe is to avoid the dialog. If the dialog appears, the kiosk is broken. The most common causes are: (1) the iframe is not sandboxed correctly, (2) the browser is not in kiosk mode (acceptable in development), (3) the default printer is not configured.
- **Using Tailwind classes inside the print component.** Tailwind's utility classes may render unpredictably on thermal printers. Use plain CSS classes defined in `print.css` for the print iframe.
- **Forgetting the `allow-same-origin` sandbox attribute.** Without it, `iframe.contentWindow.print()` may be blocked by the browser.
- **Forgetting to inject the print CSS into the iframe.** The iframe's document does not inherit CSS from the main app. The CSS must be injected into the iframe's `<head>`.
- **Memory leaks from orphan iframes.** Each unsuccessful print leaves an iframe in the DOM if not cleaned up. The kiosk runs for hours; orphan iframes accumulate and slow down the page.
- **Not handling the `afterprint` timeout.** Some browsers do not fire the `afterprint` event reliably. The 5-second timeout is a safety net.
- **Playing the bell file.** This sub-phase places the file but does NOT play it. Playing the bell is Phase 3.3.1's responsibility.
- **Using `window.print()` directly on the main app.** This would print the entire kiosk page, not just the ticket. The hidden iframe isolates the print to the ticket contents.
- **Loading the print CSS into the main app's `<head>`.** This may cause the `@page` rule to apply to the main app's print preview (if the user prints the page manually). The recommended approach is to inject the CSS only into the iframe document.

### 9.1 The Silent Print Is the Most Failure-Prone Flow in This Sub-Phase

Silent printing relies on a combination of:

- Chrome's `--kiosk-printing` command-line flag.
- An OS-level default printer configuration.
- A properly sandboxed iframe.
- A correctly styled print document.

Any one of these failing breaks the flow. The agent must:

- Test the print in the actual kiosk environment, not just in Chrome DevTools.
- Verify the `@page` rule applies by inspecting Chrome's print preview.
- Confirm the `bell.mp3` file is in `public/sounds/` (not `public/`) and is committed to the repo.

### 9.2 The Iframe Lifecycle Must Be Bulletproof

A kiosk runs for hours. If the iframe is not cleaned up after each print, the DOM accumulates hundreds of hidden iframes over a day. The agent must:

- Remove the iframe in all exit paths (success, failure, timeout).
- Clear any pending timeouts on unmount.
- Use a `useEffect` cleanup function to handle the case where the kiosk confirmation screen unmounts before the print completes.

### 9.3 The Fallback UX Is Critical

Printers fail. When they do, the customer must be guided clearly:

- "Print failed — please show this screen to the counter."
- A "Reprint" button to retry (transient failures may resolve).
- A "Done" button to dismiss and return to the home screen.

The ticket remains valid in the database regardless of print success — the customer can present the on-screen ticket number at the counter.

### 9.4 Coordinate with 2.2.2 for Component Imports

This document exports `PrintTicket` from `@/components/kiosk/print-ticket` and `SilentPrintTrigger` from `@/app/kiosk/_components/silent-print-trigger`. The 2.2.2 confirmation screen imports both components. The agent must:

- Confirm the import paths match 2.2.2's expectations.
- Ensure the component interfaces (props) are stable so the imports do not break when one document is updated.
- Document in the final summary the exact export paths and prop shapes.

### 9.5 The Bell File Is a Placeholder, Not the Real Chime

This sub-phase places a silent MP3 (or a placeholder chime) at `public/sounds/bell.mp3`. The file is **not played** by this sub-phase. Phase 3.3.1 will:

- Replace the file with the production chime (the team must provide the actual audio).
- Implement the Web Audio API integration.
- Implement the audio unlock pattern.
- Implement the bell + TTS announcement queue.

The placeholder is staged so Phase 3 does not need to add a binary file later. The file must remain a valid MP3 (even if silent) to avoid 404 errors during Phase 3 development.

### 9.6 When in Doubt, Ask the Human

The agent should ask the human reviewer (not guess) if any of the following arise:

- The exact thermal printer model and its print capabilities are specified and conflict with the document's assumptions (e.g., the printer requires a different paper size).
- The `bell.mp3` placeholder file should be a specific chime (the team may have a real audio file to use).
- The kiosk deployment environment has specific Chrome flags or printer configurations that conflict with the document's approach.
- The fallback UX wording should match the team's customer-service tone.
