## 1. Document Purpose

This document delivers the **ticket generation engine and queue logic** end-to-end: the `POST /api/tickets/issue` endpoint that creates a new ticket atomically, the ticket number formatting algorithm (prefix + zero-padded sequence), the queue position calculation, the estimated wait time formula, the business date assignment, the `TicketEvent` audit row, the `TICKET_ISSUED` SSE event payload definition, and the supporting read endpoints (`GET /api/tickets` and `GET /api/tickets/[ticketId]`) that the kiosk and the officer surfaces consume.

When this document is complete:

- A request to `POST /api/tickets/issue` with a valid service ID returns a fully populated ticket with a unique ticket number, a queue position, and an estimated wait time.
- The `Service.currentTicketNumber` field is atomically incremented inside the same transaction that creates the new `Ticket`, so two concurrent issuances always produce sequential numbers (never duplicates).
- An `ISSUED` `TicketEvent` row is written in the same transaction.
- A `TICKET_ISSUED` SSE event is emitted via `broadcastEvent('global', 'TICKET_ISSUED', payload)` after the transaction commits. The event payload type is fully defined in this document so Phase 3 can consume it without modification.
- The read endpoints provide a paginated queue view and a single-ticket detail with its event log, gated by `PERMISSION_TICKET_VIEW`.

This document **does not** implement the kiosk UI (owned by 2.2.2), the silent printing flow (owned by 2.2.3), the call/recall/no-show lifecycle (owned by 2.3.x), or the daily reset (owned by 2.3.3). It is the **issuance** layer only.

## 2. Scope and Boundaries

### 2.1 In Scope

- Creating `src/schemas/ticket.schema.ts` with `issueTicketSchema`, `listTicketsQuerySchema`, `getTicketByIdParamsSchema`.
- Creating `src/types/ticket.types.ts` with the API request/response types — `IssueTicketInput`, `IssuedTicketResponse`, `TicketDetail`, `TicketListItem`, `TicketEventLogEntry`.
- Creating `src/lib/ticket-service.ts` with the business logic — `formatTicketNumber(prefix, sequence)`, `calculateEstimatedWaitMinutes(waitPosition, averageServiceTime)`, `getCurrentBusinessDate(now?)`, `calculateWaitPosition(tx, serviceId, businessDate)`, and the orchestrating `issueTicket(input)` function.
- Creating the API endpoints:
  - `POST /api/tickets/issue` — public (no auth, no permission check; rate-limited by Phase 5).
  - `GET /api/tickets` — paginated queue list, guarded with `PERMISSION_TICKET_VIEW`.
  - `GET /api/tickets/[ticketId]` — ticket detail with event log, guarded with `PERMISSION_TICKET_VIEW`.
- Extending `src/types/sse.types.ts` (from 2.1.1) to add the `TICKET_ISSUED` payload variant to the `SseEventPayload` discriminated union.
- Extending `src/lib/env.ts` (from Phase 1.1.1) to optionally include the `APP_TIMEZONE` environment variable (optional, defaults to server local timezone).
- Ticket number generation algorithm:
  - `ticketNumber` = `<Service.ticketPrefix>` + `<zero-padded displayNumber>`.
  - Default padding width is 3 digits (so the first 999 tickets are `A001` to `A999`).
  - When `displayNumber > 999`, padding width expands to 4 digits (`A1000`, `A1001`, …).
  - `displayNumber` is the raw integer sequence (1, 2, 3, …).
- Atomic increment of `Service.currentTicketNumber` inside a single Prisma `$transaction` that also creates the `Ticket` row and the `TicketEvent` row.
- Queue position calculation: `waitPosition = (count of tickets with `status = 'WAITING'` for the same `serviceId` on the same `businessDate`) + 1`, evaluated inside the transaction.
- Estimated wait time formula: `estimatedWaitMinutes = ceil(waitPosition * service.averageServiceTime)`; when `averageServiceTime` is null, fall back to the `SystemSetting` named `queue.default_average_service_time_minutes` (default 5).
- Business date assignment: `businessDate` is set to midnight (00:00:00.000) of the day the ticket is issued, in the configured `APP_TIMEZONE` (default server local timezone).
- `TicketEvent` audit row: every issuance writes a `TicketEvent` with `eventType = 'ISSUED'`, `ticketId` set, `counterId = null`, `officerId = null`, `metadata = { businessDate, waitPosition, estimatedWaitMinutes }`.
- SSE event emission: after the transaction commits successfully, `broadcastEvent('global', 'TICKET_ISSUED', payload)` is called. The payload is the full ticket object with selected denormalised fields (service name, ticket number, etc.).

### Step 5: Create `src/lib/ticket-service.ts` — The Business Logic Module

**Goal:** Encapsulate all ticket issuance and queue logic in pure functions (where possible) plus one orchestrating function that wraps the transaction.

**Files affected:** `src/lib/ticket-service.ts` (new file).

**Specific instructions:**

Create `src/lib/ticket-service.ts` containing:

- **`formatTicketNumber(prefix: string, sequence: number): string`**:
  - Pure function. Returns the formatted ticket number.
  - If `sequence <= 999`, padding width is 3: `A001`, `A042`, `A999`.
  - If `sequence > 999` and `sequence <= 9999`, padding width is 4: `A1000`, `A1234`, `A9999`.
  - If `sequence > 9999` and `sequence <= 99999`, padding width is 5.
  - If `sequence > 99999`, the function throws an `Error` with the message "Ticket number overflow — manual intervention required". The issuance endpoint maps this to `VALIDATION_ERROR` (HTTP 422).
  - The prefix is not padded — the document specifies the prefix is exactly 1 character (enforced by 2.1.1's schema).
  - Implementation: compute the padding width as `max(3, floor(log10(sequence)) + 1)` clamped between 3 and 5; use `String(sequence).padStart(width, '0')` to pad; concatenate with the prefix.

- **`calculateEstimatedWaitMinutes(waitPosition: number, averageServiceTime: number | null): number | null`**:
  - Pure function. Returns the estimated wait in minutes, or `null` if no input is available.
  - If `waitPosition <= 0`, returns `0`.
  - If `averageServiceTime` is null, the function returns `null` (the caller is responsible for resolving the system default before calling — see `resolveAndCalculateEstimatedWaitMinutes` below).
  - Otherwise: `return Math.ceil(waitPosition * averageServiceTime)`.
  - The result is always an integer (rounded up).

- **`resolveAndCalculateEstimatedWaitMinutes(waitPosition: number, averageServiceTime: number | null, fallbackMinutes: number): number`**:
  - Calls `calculateEstimatedWaitMinutes`; if the result is `null`, uses `fallbackMinutes * waitPosition` (rounded up).
  - The caller fetches `fallbackMinutes` from the `SystemSetting` table (key `queue.default_average_service_time_minutes`); if no setting is found, hard-codes `5` as the fallback.
  - This is the function the issuance flow calls.

- **`getCurrentBusinessDate(now: Date = new Date()): Date`**:
  - Pure function (given the same `now`, returns the same value).
  - Returns midnight (00:00:00.000) of the day of `now` in the configured timezone (`APP_TIMEZONE`).
  - Implementation note (describe in prose): construct a `Date` object representing midnight in the configured timezone. The agent can use `Intl.DateTimeFormat` to compute the year/month/day in the timezone, then construct a `Date` object using `Date.UTC(year, month - 1, day)` and store it as the business date. The resulting `Date` is **the UTC instant of midnight in that timezone** — Prisma stores it as a UTC timestamp.
  - Document that this function is called inside the issuance transaction to ensure consistency.

- **`calculateWaitPosition(tx: Prisma.TransactionClient, serviceId: string, businessDate: Date): Promise<number>`**:
  - Queries the database (within the transaction) for the count of `Ticket` records where `serviceId` matches, `businessDate` matches, and `status = 'WAITING'`.
  - Returns `count + 1`.
  - The Prisma query is a simple `count` operation scoped to the transaction client (`tx`).

- **`issueTicket(input: IssueTicketInput): Promise<IssuedTicketResponse>`**:
  - The orchestrating function. Wraps everything in a single Prisma `$transaction`.
  - Steps (in order, all inside the transaction):
    1. **Fetch the service.** Look up the service by `serviceId`. If not found, throw an error that maps to `NOT_FOUND` (HTTP 404). If found but `isActive = false`, throw an error that maps to `VALIDATION_ERROR` (HTTP 422) with the message "Service is not active".
    2. **Compute the business date.** Call `getCurrentBusinessDate()`.
    3. **Atomic increment.** Use Prisma's `db.service.update({ where: { id: serviceId }, data: { currentTicketNumber: { increment: 1 } } })` to atomically increment and return the new value. The returned value is the new `displayNumber`. (Note: this is inside the transaction; Prisma's `increment` is atomic at the database level.)
    4. **Calculate wait position.** Call `calculateWaitPosition(tx, serviceId, businessDate)`. This is inside the transaction; the count is accurate at this moment.
    5. **Resolve estimated wait.** Fetch the `SystemSetting` named `queue.default_average_service_time_minutes`. If not found, use `5`. Call `resolveAndCalculateEstimatedWaitMinutes(waitPosition, service.averageServiceTime, fallbackMinutes)`.
    6. **Format the ticket number.** Call `formatTicketNumber(service.ticketPrefix, displayNumber)`.
    7. **Create the Ticket.** Use `tx.ticket.create({ data: { ticketNumber, displayNumber, serviceId, status: 'WAITING', priority: input.priority ?? 0, waitPosition, estimatedWaitMinutes, businessDate, customerPhone: input.customerPhone ?? null, issuedAt: new Date() } })`.
    8. **Create the TicketEvent.** Use `tx.ticketEvent.create({ data: { ticketId: ticket.id, eventType: 'ISSUED', counterId: null, officerId: null, metadata: { businessDate: businessDate.toISOString(), waitPosition, estimatedWaitMinutes, displayNumber, ticketPrefix: service.ticketPrefix } } })`.
  - **After the transaction commits successfully** (NOT inside the transaction — the broadcast is outside):
    - Build the SSE payload (the `TICKET_ISSUED` payload type from Step 1).
    - Generate an `sseEventId` (use `randomUUID()` from Node's `crypto`).
    - Call `broadcastEvent('global', 'TICKET_ISSUED', { ...payload, _eventId: sseEventId })`.
  - **Return** the full `IssuedTicketResponse` with the ticket, the event log (a single `ISSUED` entry), and the `sseEventId`.
  - **Error handling:** if any step inside the transaction throws, the transaction rolls back and the error propagates to the API route handler. The handler maps Prisma errors and known business errors to the standard response envelope.
  - **No `AuditLog` writes** (per Phase 2 standard 8.4).

### Step 6: Create `src/app/api/tickets/issue/route.ts` — The Issuance Endpoint

**Goal:** Expose the issuance business logic via the public `POST /api/tickets/issue` endpoint.

**Specific instructions:**

Create the file containing:

## 8. Common Pitfalls

The following are common mistakes the agent should avoid:

- **Calling `broadcastEvent()` inside the `$transaction`.** If the transaction rolls back, the event has already been emitted. The agent must call `broadcastEvent()` **after** the transaction commits.
- **Forgetting the `TicketEvent` row.** The `TicketEvent` table is the audit trail for ticket lifecycle. Skipping it means the ticket has no history of how it was issued.
- **Manual `currentTicketNumber` increment instead of Prisma's `increment`.** Manual increment is not atomic under concurrent load. Always use Prisma's `update({ data: { currentTicketNumber: { increment: 1 } } })`.
- **Reading `currentTicketNumber` outside the transaction and then writing back manually.** The atomicity is lost. The increment must happen inside the transaction with Prisma's `increment`.
- **Computing `waitPosition` outside the transaction.** A concurrent issuance can change the count between read and write. The count must be inside the transaction.
- **Using `Math.round` instead of `Math.ceil` for estimated wait.** Rounding down can produce a wait time of 0 for the first ticket in a queue with `averageServiceTime = 1`. Always round up.
- **Wrong padding for ticket numbers above 999.** The default padding is 3 digits; it must expand to 4 at 1000 and to 5 at 10000. A fixed-width pad produces `A1000` formatted as `A1000` (4 digits), which is correct only if the padding expands dynamically.
- **Writing to `AuditLog` from issuance.** Per Phase 2 standard 8.4, issuance does **not** write to `AuditLog`. The `TicketEvent` row is the complete audit trail.
- **Guarding `POST /api/tickets/issue` with `withPermission()`.** The endpoint is public. The kiosk is anonymous. The agent must not add a `withPermission` wrapper.
- **Using `auth()` in `POST /api/tickets/issue`.** The endpoint does not read the session. The actor of an issuance is the kiosk (anonymous). The `TicketEvent.officerId` is `null`.
- **Forgetting the `APP_TIMEZONE` extension in `env.ts`.** Without this, the `getCurrentBusinessDate()` function cannot resolve the timezone from configuration.
- **Including `AuditLog` writes "for safety" or "for traceability".** The Phase 2 standard is explicit: high-volume ticket lifecycle events use `TicketEvent` only. Adding `AuditLog` writes doubles the database load and is out of policy.

## 9. AI Agent Guidance

This section provides document-specific guidance for the AI agent implementing this task plan.

### 9.1 The Most Important Rule

The single most important rule for this document is **atomicity**. Every ticket issuance must produce sequential numbers under concurrent load. The mechanism is the Prisma `$transaction` block in `issueTicket()` — every write happens inside that block, and the `Service.currentTicketNumber` increment is the database-level atomic operation that guarantees uniqueness.

If the agent is unsure whether to put a write inside or outside the transaction, the answer is **inside**. The only thing outside the transaction is the `broadcastEvent()` call (which is not a database write).

### 9.2 What "Sequential Under Concurrency" Means

When two requests arrive at `POST /api/tickets/issue` at the same time (e.g., 50ms apart), the two responses must have sequential `displayNumber` values (e.g., 1 and 2). The Prisma `increment` operation guarantees this by acquiring a row lock on the `Service` record during the update. Both transactions wait for each other to complete, then increment the counter by 1 each.

If the agent uses a read-then-write pattern (`const current = await db.service.findUnique(...); await db.service.update({ data: { currentTicketNumber: current.currentTicketNumber + 1 } } })`), the increment is **not** atomic — both requests can read the same value and write the same increment, producing duplicate numbers.

### 9.3 The `TicketEvent` Row Is Not Optional

Every issuance writes a `TicketEvent` row. This row is the audit trail for the ticket's lifecycle. If the agent forgets the `TicketEvent`, the ticket has no record of being issued, and the daily reset (2.3.3) and reporting (Phase 5) cannot compute statistics.

### 9.4 The SSE Payload Type Is the Contract with Phase 3

The `TICKET_ISSUED` payload type defined in this document is **the contract** that Phase 3 consumes. The shape is locked here. Phase 3 will write the SSE delivery code, but it relies on the payload type being correct.

If the agent changes the payload shape later (e.g., to add a field), Phase 3 may not consume the new field, leading to silent failures on the display board. The agent should not add fields after the fact — additions should be coordinated with Phase 3.

### 9.5 The Ticket Number Format Is the Customer's First Impression

The ticket number is the first thing the customer sees on the printout. It must be formatted correctly:

- `A001`, `A042`, `A999` (3-digit padding for the first 999 tickets).
- `A1000`, `A1234`, `A9999` (4-digit padding for tickets 1000–9999).
- `A10000`, `A99999` (5-digit padding for tickets 10000–99999).
- Beyond 99999: the issuance fails with a clear error (the system cannot generate a ticket number in this range without a manual intervention).

The agent should test the padding transitions explicitly: issue enough tickets to reach `A1000` and verify the format is correct.

### 9.6 The Business Date Is a UTC Instant

The `businessDate` field is stored as a UTC timestamp in the database (Prisma's default for `DateTime`). The `getCurrentBusinessDate()` function returns the UTC instant of midnight in the configured timezone. This is intentional — Prisma stores datetimes as UTC, and the daily reset (2.3.3) groups tickets by their UTC `businessDate` value.

If the deployment is in `Asia/Colombo` (UTC+5:30) and a ticket is issued at 23:30 local time on day D, the `businessDate` is the UTC instant of midnight on day D in Colombo time (which is `2026-06-18T18:30:00Z` if D is 2026-06-19 in Colombo). The daily reset at midnight Colombo time creates a snapshot for this business date.

The agent should not store the business date as a local date string or a "date-only" value. It must be a `Date` (UTC instant).

### 9.7 The Wait Position Is Calculated, Not Stored Incrementally

The `waitPosition` is calculated at issuance time as `(count of WAITING tickets for this service on this business date) + 1`. It is **not** stored incrementally (i.e., the agent should not increment a separate `lastIssuedWaitPosition` field). The count is accurate at the moment of calculation inside the transaction.

If a ticket is called (status changes to `CALLED`) between issuances, the wait position of a newly issued ticket reflects the reduced queue length — only `WAITING` tickets count.

### 9.8 Verification of the Concurrent Issuance Test Is Critical

Test 3 (concurrent issuance) is the most important verification. The agent must actually issue two requests concurrently (using `Promise.all` on two `fetch` calls) and verify that the responses have different `displayNumber` values. If the test is skipped, a subtle race condition may slip through and produce duplicate numbers under load.

### 9.9 When in Doubt, Ask the Human

The agent should ask the human reviewer (not guess) if any of the following arise:

- The `PERMISSION_TICKET_VIEW` permission does not exist in Phase 1.3.1's permission list.
- The `Ticket.ticketNumber` field has a unique constraint in the database (the document assumes it does **not** — the uniqueness comes from the atomic increment, not from a database constraint, but if the schema has a unique constraint, the agent must consider the error mapping for `P2002` Prisma errors).
- The `SystemSetting` model is missing or named differently.
- The `TicketStatus` enum has different values than expected.
- The deployment timezone is specified and conflicts with the `APP_TIMEZONE` default.
