## 1. Document Purpose

This document delivers the **customer-facing kiosk UI** end-to-end: the full-screen kiosk layout, the service selection grid showing the active services as tappable cards, the confirmation screen that displays the issued ticket number with the estimated wait time, the auto-reset inactivity timer that returns the kiosk to the home screen after a configured timeout, and the kiosk configuration loading (by URL query or default fallback).

When this document is complete:

- A customer visits `/kiosk` and sees the kiosk's full-screen layout (white background, brand logo, current date/time, welcome message).
- The service selection grid displays the active services (filtered by the kiosk's `restrictedServiceIds` if set, sorted by `sortOrder` then `name`) as large tappable cards.
- Tapping a service calls `POST /api/tickets/issue` with the `serviceId`. On success, the confirmation screen shows the issued ticket number (large, monospace, bold), the service name, the estimated wait time (if enabled), and the counter guidance text.
- The confirmation screen mounts the silent print component (from 2.2.3) to trigger the print.
- After `KioskConfig.autoResetSeconds` of inactivity, the kiosk returns to the home screen. Any tap, touch, keypress, or scroll during the home screen resets the inactivity timer.
- The kiosk does not require authentication. Multiple kiosk instances are supported via the `?kioskId=<id>` URL query.

This document **does not** implement the silent printing flow itself (owned by 2.2.3), the ticket issuance API (owned by 2.2.1), the ticket call/recall/no-show endpoints (owned by 2.3.x), or the daily reset (owned by 2.3.3).

### Step 1: Create `src/lib/kiosk-config.ts` — The Kiosk Configuration Loader

**Goal:** Provide a server-side helper that loads the active `KioskConfig` by ID or falls back to the default.

**Files affected:** `src/lib/kiosk-config.ts` (new file).

**Specific instructions:**

Create `src/lib/kiosk-config.ts` containing:

- **`type LoadedKioskConfig`**: the in-memory shape returned by the loader. Fields: `id`, `name`, `welcomeMessage`, `footerMessage`, `printerName` (nullable), `autoResetSeconds`, `showEstimatedWait`, `restrictedServiceIds` (a typed `string[]` or `null`), `paperWidth` (an enum or constant inferred from a future field — for now, hard-code `'80MM'` and document the future extension).

- **`loadKioskConfig(kioskId?: string | null): Promise<LoadedKioskConfig | null>`**:
  - If `kioskId` is provided:
    - Query the `KioskConfig` table where `id = kioskId AND isActive = true`.
    - If found, return the `LoadedKioskConfig` shape.
    - If not found, return `null` (the page renders the no-config fallback).
  - If `kioskId` is not provided:
    - Query the `KioskConfig` table where `isDefault = true AND isActive = true`. There must be at most one such record (the document specifies this constraint).
    - If found, return the `LoadedKioskConfig` shape.
    - If not found, return `null`.
  - The function is **not** wrapped in `unstable_cache` — Phase 5.2.2 will add caching for slowly-changing data. The document specifies that this loader is called once per page load (the kiosk page is server-rendered).

- **`getActiveServicesForKiosk(kiosk: LoadedKioskConfig): Promise<ServiceForKiosk[]>`**:
  - Queries the `Service` table where `isActive = true`.
  - If `kiosk.restrictedServiceIds` is set, intersects with that list (filter to only the IDs in the array).
  - Returns the services ordered by `sortOrder asc, name asc`.
  - The returned shape (`ServiceForKiosk`) includes: `id`, `name`, `code`, `description`, `iconName`, `color`.
  - The function is server-only.

### Step 5: Create `src/app/kiosk/_components/kiosk-home.tsx` — The Service Selection Grid

**Goal:** The main interactive component that shows the service grid, handles the issuance flow, and manages the auto-reset timer.

**Files affected:** `src/app/kiosk/_components/kiosk-home.tsx` (new file).

**Specific instructions:**

Create the component:

- A "use client" directive.
- **Props:** `{ services: ServiceForKiosk[]; kioskConfig: LoadedKioskConfig }`.
- **State:**
  - `currentView`: `'grid' | 'loading' | 'confirmation' | 'error'`. Determines what is rendered.
  - `currentTicket`: `IssuedTicketResponse | null`. The ticket returned by the issuance API.
  - `errorMessage`: `string | null`. The error to display on failure.
- **Behaviour:**
  - Initial state: `currentView = 'grid'`.
  - On service card tap:
    - Set `currentView = 'loading'`.
    - Call `POST /api/tickets/issue` with `{ serviceId: <service.id> }`.
    - On success: store the response in `currentTicket`, set `currentView = 'confirmation'`.
    - On failure: store the error message in `errorMessage`, set `currentView = 'error'`.
  - On auto-reset: set `currentView = 'grid'`, clear `currentTicket` and `errorMessage`.
  - On "Done" button tap on confirmation screen: trigger auto-reset manually.
- **Renders:**
  - If `currentView = 'grid'`: render `<KioskServiceGrid services={services} onSelect={handleServiceSelect} />` (Step 7).
  - If `currentView = 'loading'`: render a loading spinner with the message "Issuing your ticket...".
  - If `currentView = 'confirmation'`: render `<TicketConfirmation ticket={currentTicket} kioskConfig={kioskConfig} onDone={handleDone} />` (Step 9).
  - If `currentView = 'error'`: render an error state with the message and a "Try Again" button.
- **Auto-reset timer integration:**
  - Uses the `useKioskReset` hook (from Step 6) to manage the inactivity timer on the home screen.
  - On the confirmation screen, the timer is shorter (typically 10–15 seconds, or the kiosk config's value) so the next customer can use the kiosk promptly.
  - The hook is initialised with the `autoResetSeconds` from `kioskConfig`.

### Step 6: Create `src/hooks/use-kiosk-reset.ts` — The Auto-Reset Hook

**Goal:** Provide a client-side hook for the auto-reset inactivity timer.

**Files affected:** `src/hooks/use-kiosk-reset.ts` (new file).

**Specific instructions:**

Create the hook:

- A "use client" directive (the hook is a client-only hook).
- **Signature:** `useKioskReset(options: { inactivitySeconds: number; onReset: () => void; paused?: boolean }): { reset: () => void }`.
- **Behaviour:**
  - On mount, starts a timer for `inactivitySeconds`.
  - On any of the following user interaction events on `document`: `click`, `touchstart`, `keydown`, `scroll`, the timer is reset to `inactivitySeconds`.
  - When the timer expires, calls `onReset()`.
  - The `paused` option pauses the timer (used during in-flight API calls so a slow response does not cause a premature reset).
  - When `paused` is `true`, the timer is suspended; when `false`, it resumes from the elapsed time (or restarts at `inactivitySeconds` — the document specifies restart for simplicity).
  - The returned `reset()` function can be called to explicitly trigger `onReset()` (e.g., on the "Done" button tap).
- **Cleanup:**
  - On unmount, the event listeners are removed and the timer is cleared.
  - The hook must not leak listeners (a common mistake in kiosk apps that run for hours).

## 8. Common Pitfalls

The following are common mistakes the agent should avoid:

- **Adding authentication to the kiosk.** The kiosk is anonymous. Any session check or redirect to `/login` will break the kiosk for real customers. The page must be fully public.
- **Forgetting the `"use client"` directive on interactive components.** Components using `useState`, `useEffect`, or `useKioskReset` must be client components. Server components cannot use hooks.
- **Using `window` or `document` in a server component.** The `useKioskReset` hook uses `document` event listeners — it can only be called from a client component.
- **Memory leak from event listeners.** The `useKioskReset` hook must remove its event listeners on unmount. A kiosk app that runs for hours without proper cleanup will slow down over time.
- **Inconsistent view state.** The `currentView` state in `KioskHome` is the single source of truth for what is rendered. Mixing the grid and confirmation in the same render tree (without state) leads to confusing UX.
- **Not handling the `PrintTicket` import when 2.2.3 is not yet implemented.** If 2.2.3 is developed in parallel, the confirmation screen must import a placeholder component or handle the missing component gracefully. The kiosk should not crash if the print component is missing.
- **Reading `searchParams` in a client component.** The `searchParams` are only available in server components (Next.js 14+). The page (`page.tsx`) is a server component; the home component (`kiosk-home.tsx`) is a client component that receives the resolved config and services as props.
- **Hardcoding the welcome message or the footer text.** These come from the `KioskConfig` and must be configurable per kiosk.
- **Using `Math.random()` for anything that affects ticket numbering.** The ticket number is computed server-side by the issuance endpoint. The kiosk only displays the number returned by the API.
- **Forgetting the empty state in the service grid.** If no services are active (e.g., after a mass deactivation), the kiosk must show a clear message instead of an empty grid.

## 9. AI Agent Guidance

This section provides document-specific guidance for the AI agent implementing this task plan.

### 9.1 The Kiosk Is a Public-Facing Touch App

The kiosk is the most user-facing component in the system. It will be touched by hundreds of customers who have never used the system before. The design principles are:

- **Touch-first.** Every tappable element is at least 48px tall (preferably 100px+). Visual feedback on press (scale, color change) is essential.
- **Full-screen.** No sidebar, no top bar, no browser chrome. The kiosk should look like a dedicated app, not a web page.
- **Forgiving.** Errors are shown clearly. The "Try Again" button is always available. The auto-reset prevents the kiosk from getting stuck.
- **Fast.** Taps respond instantly. The loading state is brief. The print happens automatically.

### 9.2 The Auto-Reset Timer Is Critical

A kiosk that gets stuck on a previous customer's screen is a real problem. The auto-reset timer is the safety mechanism. The agent must:

- Test the timer thoroughly (start the timer, wait for expiry, verify the screen returns to home).
- Ensure the timer resets on any interaction (tap, touch, keypress, scroll).
- Pause the timer during the in-flight API call so a slow response does not cause a premature reset.
- Clean up event listeners on unmount to prevent memory leaks.

### 9.3 The Confirmation Screen Is Where the Customer's Experience Peaks

The confirmation screen is the moment the customer sees their ticket number for the first time. It must be:

- **Large and obvious.** The ticket number is the dominant element on the screen.
- **Readable from across a waiting area.** Use a large font (120px+) for the number.
- **Printed silently.** The print must happen automatically after the confirmation screen mounts. The customer should not need to tap "Print".

### 9.4 Coordinate with 2.2.3 for the Print Component

This document imports the `PrintTicket` component from `@/components/kiosk/print-ticket` (owned by 2.2.3). If 2.2.3 is developed in parallel, the component file may be a placeholder initially. The agent should:

- Confirm the import path matches 2.2.3's export.
- Ensure the placeholder component has the same interface (props) as the real component, so the import does not break when 2.2.3 is implemented.
- Document in the final summary if the print component is a placeholder at the time of this document's completion.

### 9.5 The Kiosk Does Not Need to Know About SSE or Phase 3

This document does not subscribe to SSE events. The kiosk is a fire-and-display surface — it issues a ticket and shows the confirmation. Real-time updates (e.g., "your ticket is being called") are owned by Phase 3 and the display board, not by the kiosk.

The kiosk does not need to:

- Subscribe to SSE.
- Listen for ticket call events.
- Update the displayed ticket after issuance.

The auto-reset is the only timer the kiosk manages.
