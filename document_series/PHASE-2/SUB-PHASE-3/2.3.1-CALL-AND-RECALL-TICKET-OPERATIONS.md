## 1. Document Purpose

This document delivers the **counter-side call and recall operations** — the two most frequent ticket state transitions in the queue. It introduces the **ticket status state machine** (the single source of truth for all legal ticket transitions), the `POST /api/tickets/[ticketId]/call` and `/recall` endpoints, the validation rules that govern officer authorisation (officer must be on duty, counter must be active, ticket must be in a state that allows the action, ticket's service must be assigned to the counter), the `TicketEvent` audit row for each transition, the `TICKET_CALLED` and `TICKET_RECALLED` SSE event payload definitions, and the `resolveCallingOfficer` server-side helper.

When this document is complete:

- An officer with the correct counter assignment can call any waiting ticket for one of that counter's services. The ticket's status transitions to `CALLED`, `Ticket.calledAt` is set, `Ticket.counterId` and `Ticket.calledByOfficerId` are populated, and a `TicketEvent` of type `CALLED` is written in the same transaction. A `TICKET_CALLED` event is emitted after the transaction commits.
- The same officer can recall a ticket currently in `CALLED` or `RECALLED` state. The status transitions to `RECALLED`, `Ticket.recalledAt` is updated, an internal `recallCount` is incremented, and a `TicketEvent` of type `RECALLED` is written. A `TICKET_RECALLED` event is emitted.
- All illegal ticket state transitions are rejected with `VALIDATION_ERROR` (HTTP 422) and a clear message identifying the offending rule.
- The state machine is a pure function exported from `src/lib/ticket-state-machine.ts` and is the single source of truth for transition rules — no inline status checks exist elsewhere in the codebase.

This document does **not** implement no-show handling (owned by 2.3.2), the daily reset (owned by 2.3.3), the temporary officer action panel UI (owned by 2.3.2), the real SSE delivery (Phase 3.1.3), or the full officer dashboard layout (Phase 4.2.3).

### Step 1: Decide `recallCount` Storage Approach

**Goal:** Choose between (a) adding a `recallCount` column to the `Ticket` Prisma model, or (b) deriving it from `TicketEvent` rows at read time.

**Files affected:** None (decision step).

**Specific instructions:**

- The agent must pick ONE approach and document the choice at the top of `src/lib/ticket-state-machine.ts` (the chosen approach affects how the `recallTicket` function and the recall metadata are written).
- **Recommended approach (b) — derive from `TicketEvent` rows:** Do NOT modify the Prisma schema. The Prisma schema was finalised in Phase 1.1.3 and changing it now means a new migration. Instead, the `recallCount` is computed at write time as the count of `TicketEvent` rows with `eventType = 'RECALLED'` for this ticket BEFORE the new event is written. The new event's `metadata.recallCount = priorCount + 1`. The agent writes the recall function so that it counts existing recall events inside the transaction (which is safe because the transaction snapshot is consistent).
- **Alternate approach (a) — add a column:** If the human reviewer prefers, the agent may add an `Int @default(0)` field named `recallCount` to the `Ticket` model via a Prisma migration. This is simpler to read but requires a schema change. Document this choice clearly.
- The verification steps assume approach (b). If approach (a) is chosen, the agent adjusts the recall function to `$increment` the column instead of counting events.

### Step 2: Create `src/lib/ticket-state-machine.ts` — The State Machine

**Goal:** Establish the pure ticket status state machine — the single source of truth for all legal ticket transitions implemented in Phase 2.

**Files affected:** `src/lib/ticket-state-machine.ts` (new file).

**Specific instructions:**

Create `src/lib/ticket-state-machine.ts` containing:

- **Type definitions:**
  - `type TicketAction = 'CALL' | 'RECALL' | 'NO_SHOW' | 'SERVE' | 'COMPLETE' | 'CANCEL' | 'TRANSFER'` — the set of possible actions. Only `CALL`, `RECALL`, and `NO_SHOW` are implemented in Phase 2; the others are listed for future phases.
  - `interface TicketTransition { from: TicketStatus; action: TicketAction; to: TicketStatus }` — a single rule.
  - `interface InvalidTicketTransitionError { kind: 'INVALID_TRANSITION'; from: TicketStatus; action: TicketAction; message: string }` — a typed error.
- **Constant: `LEGAL_TRANSITIONS: readonly TicketTransition[]`** — the full set of legal transitions for Phase 2. The list:
  - `{ from: 'WAITING', action: 'CALL', to: 'CALLED' }`
  - `{ from: 'RECALLED', action: 'CALL', to: 'CALLED' }`
  - `{ from: 'NO_SHOW', action: 'CALL', to: 'CALLED' }` (giving another chance)
  - `{ from: 'CALLED', action: 'RECALL', to: 'RECALLED' }`
  - `{ from: 'RECALLED', action: 'RECALL', to: 'RECALLED' }` (recall again)
  - `{ from: 'CALLED', action: 'NO_SHOW', to: 'NO_SHOW' }` (added in 2.3.2 but listed here for completeness — 2.3.1 only references `CALL` and `RECALL`; the full list enables 2.3.2 to use the same state machine without modification)
  - `{ from: 'RECALLED', action: 'NO_SHOW', to: 'NO_SHOW' }`
  - The transitions `SERVING` / `COMPLETED` / `CANCEL` / `TRANSFER` are NOT in this list — they are deferred.
- **Function: `canTransition(from: TicketStatus, to: TicketStatus, action: TicketAction): boolean`** — returns `true` if a transition matching the parameters exists in `LEGAL_TRANSITIONS`.
- **Function: `transitionTicket(from: TicketStatus, action: TicketAction): TicketStatus`** — returns the resulting status. Throws an `InvalidTicketTransitionError` (with a clear message including the offending `from` and `action`) if no matching transition exists. The function never returns `undefined`; the throw is the signal.
- **Constant: `IMPLEMENTED_ACTIONS: readonly TicketAction[]`** — `['CALL', 'RECALL', 'NO_SHOW']`. Used by the agent and future agents to know which actions are wired up.
- Document at the top of the file that this is the **single source of truth** for ticket transitions. Any new transition (in a future sub-phase) must be added to `LEGAL_TRANSITIONS` AND wired into the corresponding orchestrating function. Inline status checks elsewhere in the codebase are forbidden.

  - `interface OfficerNotOnDutyError { kind: 'OFFICER_NOT_ON_DUTY'; userId: string; counterId: string; message: string }`.

### Step 7: Create `src/lib/ticket-service.ts` Extensions — `callTicket` and `recallTicket`

**Goal:** Implement the orchestrating functions for call and recall, each wrapping a single Prisma `$transaction`.

**Files affected:** `src/lib/ticket-service.ts` (modified).

**Specific instructions:**

- Open the existing `src/lib/ticket-service.ts` (created in 2.2.1).
- **Add** the following functions:

**`async function callTicket(input: TicketCallInput, officer: ResolvedOfficer): Promise<TicketCallResponse>`**:
- Wraps everything in a single Prisma `$transaction`.
- Steps (in order, all inside the transaction):
  1. **Fetch the ticket.** Use `tx.ticket.findUnique({ where: { id: input.ticketId } })` with `include: { service: true }`. If not found, throw an error mapped to `NOT_FOUND` (HTTP 404). If `serviceId` is the joined service, use that name in the event payload.
  2. **Verify ticket's service is assigned to the counter.** Use `tx.counterService.findUnique({ where: { counterId_serviceId: { counterId: officer.counterId, serviceId: ticket.serviceId } } })`. If no record, throw `ServiceNotAssignedToCounterError`.
  3. **Compute the new status.** Call `transitionTicket(ticket.status, 'CALL')` from the state machine. If the transition is illegal, throw `InvalidTicketTransitionError`. The new status is almost always `CALLED`, but the state machine validates that the current status is one of `WAITING` / `RECALLED` / `NO_SHOW`.
  4. **Update the ticket.** Use `tx.ticket.update({ where: { id: input.ticketId }, data: { status: 'CALLED', calledAt: new Date(), counterId: officer.counterId, calledByOfficerId: officer.id, recalledAt: ticket.recalledAt /* preserve previous recall timestamp if any */ } })`.
  5. **Create the TicketEvent.** Use `tx.ticketEvent.create({ data: { ticketId: input.ticketId, eventType: 'CALLED', counterId: officer.counterId, officerId: officer.id, metadata: { previousStatus: ticket.status, calledAt: new Date().toISOString(), recallCount: ticket.recalledAt ? <derived from event count> : 0 } } })`.
- **After the transaction commits successfully** (NOT inside):
  - Build the `TICKET_CALLED` SSE payload using the updated ticket data, the counter's name + number (fetched in Step 1 or via a separate pre-transaction query), and the officer's name (fetched from `User` via `userId`).
  - Generate an `sseEventId` via `randomUUID()`.
  - Call `broadcastEvent('global', 'TICKET_CALLED', { ...payload, _eventId: sseEventId })`.
- **Return** the `TicketCallResponse` with the updated ticket, the event log (which now includes the new `CALLED` event), the `sseEventId`, and `previousStatus`.
- **Error handling:** if any step throws, the transaction rolls back. The route handler maps known errors to standard error codes; unexpected errors to `INTERNAL_ERROR`.

**`async function recallTicket(input: TicketRecallInput, officer: ResolvedOfficer): Promise<TicketRecallResponse>`**:
- Wraps everything in a single Prisma `$transaction`.
- Steps (in order, all inside):
  1. **Fetch the ticket.** As in `callTicket`.
  2. **Verify the ticket's service is assigned to the counter.** As in `callTicket`.
  3. **Compute the new status.** Call `transitionTicket(ticket.status, 'RECALL')`. The legal `from` statuses are `CALLED` or `RECALLED`. If illegal, throw `InvalidTicketTransitionError`.
  4. **Count existing recall events.** Use `tx.ticketEvent.count({ where: { ticketId: input.ticketId, eventType: 'RECALLED' } })`. The new `recallCount` is `count + 1`. (This is the recommended approach (b) from Step 1.)
  5. **Update the ticket.** Use `tx.ticket.update({ where: { id: input.ticketId }, data: { status: 'RECALLED', recalledAt: new Date() } })`. (Other fields — `calledAt`, `counterId`, `calledByOfficerId` — are preserved from the original call.)
  6. **Create the TicketEvent.** Use `tx.ticketEvent.create({ data: { ticketId: input.ticketId, eventType: 'RECALLED', counterId: officer.counterId, officerId: officer.id, metadata: { previousStatus: ticket.status, recalledAt: new Date().toISOString(), recallCount: <count + 1> } } })`.
- **After the transaction commits**:
  - Build the `TICKET_RECALLED` SSE payload (extends the call payload with `recalledAt` and `recallCount`).
  - Emit via `broadcastEvent('global', 'TICKET_RECALLED', { ...payload, _eventId: sseEventId })`.
- **Return** the `TicketRecallResponse` with the updated ticket, the event log, the `sseEventId`, and `recallCount`.

**Imports to add to `src/lib/ticket-service.ts`:**
- `SseEventPayload`, `SseChannel`, `SseEventType` from `@/types/sse.types`.
- `broadcastEvent` from `@/lib/events`.
- `canTransition`, `transitionTicket`, `InvalidTicketTransitionError` from `@/lib/ticket-state-machine`.
- `resolveCallingOfficer` from `@/lib/ticket-officer` is NOT called inside the transaction (the officer is resolved by the route handler before calling `callTicket` / `recallTicket`).
- `randomUUID` from `crypto` (Node built-in).

## 8. Common Pitfalls

- **Forgetting to validate officer-on-duty.** The call/recall endpoints MUST resolve the officer via `resolveCallingOfficer` BEFORE calling `callTicket` / `recallTicket`. Skipping this check allows any authenticated user with `counter:call` to call tickets from a counter they aren't assigned to.
- **Inline status checks outside the state machine.** A common mistake is writing `if (ticket.status !== 'WAITING') return error(...)` directly in the route handler. This bypasses the state machine and creates two sources of truth. Always go through `transitionTicket()`.
- **Calling `broadcastEvent()` inside the transaction.** If the transaction rolls back after the event is emitted, downstream consumers see an event for a ticket that doesn't exist. Always emit AFTER `await db.$transaction(...)` returns.
- **Reading `isOnDuty` inside the transaction.** This is fine in principle, but the recommended pattern (resolving the officer BEFORE the transaction) keeps the transaction focused on data integrity and reduces lock contention.
- **Forgetting to update `Ticket.counterId` and `Ticket.calledByOfficerId` on call.** These fields are the source of truth for "who is currently serving this ticket". Without them, the recall path and the display board have nothing to show.
- **Not preserving `recalledAt` on a subsequent call.** If a ticket is `RECALLED` and then re-called, the `recalledAt` timestamp should be preserved (the agent may choose to either keep it or clear it — the document specifies preserve).
- **Storing `recallCount` as a Prisma column without a migration.** If the agent chooses approach (a) from Step 1, they MUST also add a migration. The recommended approach (b) — counting `TicketEvent` rows — avoids this risk.
- **Missing `include: { service: true }` when fetching the ticket.** The SSE event payload needs `serviceName`, which requires the joined service. Forgetting the include causes a runtime null-pointer or a runtime fetch outside the transaction.

## 9. AI Agent Guidance

This section is for the AI agent that will implement this document. It supplements the general guidance in Sub-Phase 2.3 Section 8.

### 9.1 Recommended Approach for `recallCount`

The agent SHOULD use approach (b) — derive `recallCount` from the count of `TicketEvent` rows with `eventType = 'RECALLED'` for the ticket. This avoids a Prisma schema change. The count is performed inside the transaction (with `tx.ticketEvent.count`), so it sees a consistent snapshot. The new event's `metadata.recallCount = count + 1`.

### 9.2 Why the `counterId` Is in the Body, Not the URL

The endpoints are `POST /api/tickets/[ticketId]/call` (NOT `/api/tickets/[ticketId]/call/[counterId]`). The counter ID is in the request body because:

- The officer's "current counter" is determined by their `CounterOfficer.isOnDuty` assignment, not by the URL.
- A super-admin (or an automated test) can call a ticket on behalf of any officer at any counter by providing the counter ID explicitly.
- The URL stays focused on the resource being acted on (the ticket).

This pattern matches the master plan's endpoint registry (Section 9.3) where `POST /api/tickets/[id]/call` is the documented path.

### 9.3 Why `broadcastEvent()` Goes on the Global Channel

The master plan Section 11.1 lists `TICKET_CALLED`, `TICKET_RECALLED`, and `TICKET_NO_SHOW` as events on the `global` channel. The `counter` channel is reserved for officer-specific events (Phase 4.1 / 4.2 territory). Sub-Phase 2.3 only emits to `global`. Future sub-phases may add counter-specific events.

### 9.4 How to Verify Without a Browser UI

Since 2.3.2's action panel doesn't exist yet, the agent verifies the endpoints via `curl` or HTTPie. Use the super-admin's session cookie (obtained via `/api/auth/[...nextauth]` login) for all requests. The agent must NOT bypass the permission check by using an admin token — every test must use a user with the correct permission.

### 9.5 When to Stop and Ask

- If the master plan's `TICKET_CALLED` payload (Section 11.3) contradicts the field shape specified in Step 4, the master plan wins; the agent updates Step 4 to match. The current shapes are aligned (the document specifies the additions to the master plan's baseline).
- If the recallCount storage approach (a vs b) is unclear, default to (b) — derive from `TicketEvent` count.
- If the human reviewer wants `TICKET_RECALLED` to omit the `previousStatus` field, the agent removes it. The default includes it for downstream consumers.
