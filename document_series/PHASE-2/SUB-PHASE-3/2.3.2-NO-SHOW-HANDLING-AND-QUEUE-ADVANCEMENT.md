## 1. Document Purpose

This document delivers the **no-show handling** for the queue system ŌĆö the third counter-side operation an officer performs on a ticket. When a customer who was called does not appear at the counter, the officer marks the ticket as no-show. The system enforces a **configurable grace period** (preventing premature no-show marks immediately after calling), writes a `TicketEvent` audit row, optionally **auto-advances to the next waiting ticket** for the same service, and emits a `TICKET_NO_SHOW` SSE event. This document also delivers the **temporary officer action panel** ŌĆö the minimal interaction surface (Call / Recall / No-Show buttons) used by officers until Phase 4.2.3's full dashboard layout replaces it.

When this document is complete:

- An officer on duty at a counter can mark a `CALLED` or `RECALLED` ticket as no-show. The ticket's status transitions to `NO_SHOW`, `Ticket.noShowAt` is set, a `TicketEvent` of type `NO_SHOW` is written with grace-period metadata, and a `TICKET_NO_SHOW` event is emitted.
- A no-show attempt within the grace period (default 60 seconds after `calledAt`) is rejected with `VALIDATION_ERROR` and a clear message indicating the elapsed time and the required wait.
- When `SystemSetting.queue.auto_advance_on_no_show` is `true`, the next waiting ticket for the same service is automatically called after the no-show commits. The auto-advance is itself a `callTicket` operation (so it re-uses the state machine and emits a `TICKET_CALLED` event for the auto-advanced ticket).
- The temporary `TicketActionPanel` component renders Call, Recall, and No-Show buttons with correct enable/disable logic, in-flight spinners, and error toasts.
- The temporary stub page at `/(officer)/counter/[counterId]/page.tsx` shows the current serving ticket and the action panel, and exercises the API correctly.

This document does **not** implement the `SERVING` / `COMPLETED` ticket transitions (deferred), the real SSE delivery (Phase 3.1.3), the full officer dashboard layout (Phase 4.2.3), FCM push notifications (Phase 4.1), or rate limiting (Phase 5).

### Step 5: Extend `src/lib/ticket-service.ts` with the `noShowTicket` Orchestrating Function

**Goal:** Implement the no-show business logic, including the grace period check inside the transaction.

**Files affected:** `src/lib/ticket-service.ts` (modified).

**Specific instructions:**

- Open the existing `src/lib/ticket-service.ts` (extended in 2.3.1 with `callTicket` and `recallTicket`).
- **Add** the following function:

**`async function noShowTicket(input: TicketNoShowInput, officer: ResolvedOfficer): Promise<TicketNoShowResponse>`**:
- Wraps the no-show status update + `TicketEvent` write in a single Prisma `$transaction`. The auto-advance is OUTSIDE the transaction (Step 4).
- Steps (in order, all inside the transaction):
  1. **Fetch the ticket.** Use `tx.ticket.findUnique({ where: { id: input.ticketId }, include: { service: true } })`. If not found, throw an error mapped to `NOT_FOUND`.
  2. **Verify ticket's service is assigned to the counter.** Use `tx.counterService.findUnique({ where: { counterId_serviceId: { counterId: officer.counterId, serviceId: ticket.serviceId } } })`. If no record, throw `ServiceNotAssignedToCounterError`.
  3. **Compute the new status.** Call `transitionTicket(ticket.status, 'NO_SHOW')` from the state machine. The legal `from` statuses are `CALLED` or `RECALLED`. If illegal (e.g., no-show on a `WAITING` ticket ŌĆö the customer never had a chance to appear), throw `InvalidTicketTransitionError`.
  4. **Verify grace period.** Compute `elapsedSeconds = (now - ticket.calledAt) / 1000`. Read the grace period via `getNoShowGracePeriodSeconds()` (NOTE: this is called BEFORE the transaction starts because it does a DB read; the document specifies this ŌĆö the setting is read once per request, not per row). If `elapsedSeconds < gracePeriod`, throw `{ kind: 'GRACE_PERIOD_NOT_ELAPSED', elapsedSeconds, requiredSeconds: gracePeriod, message: \`Ticket was just called ŌĆö please wait \${gracePeriod - elapsedSeconds} more seconds before marking no-show.\` }`. The grace period is re-checked inside the transaction by re-fetching the ticket under the transaction's read snapshot to prevent TOCTOU races (the document specifies this defence-in-depth approach: the grace period is checked twice ŌĆö once before the transaction to fail fast on obvious cases, once inside to guarantee correctness).
  5. **Update the ticket.** Use `tx.ticket.update({ where: { id: input.ticketId }, data: { status: 'NO_SHOW', noShowAt: new Date() } })`.
  6. **Create the TicketEvent.** Use `tx.ticketEvent.create({ data: { ticketId: input.ticketId, eventType: 'NO_SHOW', counterId: officer.counterId, officerId: officer.id, metadata: { previousStatus: ticket.status, gracePeriodSeconds: gracePeriod, elapsedSeconds, autoAdvanced: false /* will be updated by the route handler after the auto-advance */ } } })`.
- **After the transaction commits successfully**:
  - Build the `TICKET_NO_SHOW` SSE payload using the updated ticket data, the service name (from the joined service), the counter's name + number (queried outside the transaction or via a separate pre-transaction query), and the officer's name.
  - Generate an `sseEventId`.
  - Emit `broadcastEvent('global', 'TICKET_NO_SHOW', { ...payload, _eventId: sseEventId })` ŌĆö with `autoAdvanced: false` and `autoAdvancedTicketNumber: null` for now.
  - **The auto-advance is performed by the route handler AFTER the no-show function returns**, not inside `noShowTicket`. This separation keeps the no-show transaction small and makes the auto-advance optional and replaceable.
- **Return** the `TicketNoShowResponse` with the updated ticket, the event log (which now includes the new `NO_SHOW` entry), the `sseEventId`, `previousStatus`, `gracePeriodSeconds`, `elapsedSeconds`, and `autoAdvanced: false` (the route handler may update this field after the auto-advance).
- **Imports to add:**
  - `getNoShowGracePeriodSeconds` from `@/lib/ticket-advance`.

## 8. Common Pitfalls

- **Forgetting the grace period inside the transaction.** The grace period MUST be re-checked inside the transaction (re-fetch the ticket). Otherwise, two concurrent no-show attempts within the grace period could both succeed ŌĆö the second one reading stale data.
- **Nesting the auto-advance inside the no-show transaction.** The auto-advance uses `callTicket`, which has its own transaction. Nesting transactions is a Prisma anti-pattern. The auto-advance is performed sequentially AFTER the no-show commits.
- **Re-emitting `TICKET_CALLED` for the auto-advanced ticket manually.** `callTicket` already emits the event. The agent does NOT re-emit.
- **Using `useState` without cleanup in the action panel.** The in-flight request tracking must be cleaned up on unmount; otherwise, React warnings about state updates on unmounted components appear.
- **Forgetting to handle the `autoAdvanced` failure case.** If `callTicket` throws inside `advanceToNextWaitingTicket`, the function catches and logs but does NOT re-throw. The no-show itself is unaffected.
- **Importing the state machine module into a client component while it transitively imports server-only modules.** The state machine module (`src/lib/ticket-state-machine.ts`) is a pure module with no server-only imports ŌĆö it is safely importable from the client. If the agent adds server-only imports to it (e.g., `@/lib/db`), the build will fail.
- **Setting the grace period to 0 in tests to "disable" it.** This is incorrect; the system requires the grace period to be respected even in tests. The agent tests the grace period rejection separately and uses realistic values (e.g., 60 seconds).
- **Forgetting to update `Ticket.status` when auto-advancing.** The auto-advance uses `callTicket`, which handles the status update. The agent does NOT update the status manually.

## 9. AI Agent Guidance

This section is for the AI agent that will implement this document. It supplements the general guidance in Sub-Phase 2.3 Section 8.

### 9.1 Why the Grace Period Is Checked Twice

The grace period is checked BEFORE the transaction (to fail fast on obvious cases) AND inside the transaction (to guarantee correctness under TOCTOU races). The agent must implement both checks ŌĆö the inner check is the authoritative one. The outer check is an optimisation.

### 9.2 Why Auto-Advance Is Outside the No-Show Transaction

Auto-advance uses `callTicket`, which is itself a transactional operation. Nesting transactions in Prisma is technically possible but creates lock contention and is hard to reason about. The recommended pattern is sequential transactions: no-show commits ŌåÆ auto-advance runs as a separate transaction. If auto-advance fails, the no-show itself remains committed ŌĆö this is correct because the no-show is the officer's intentional action, and the auto-advance is a system convenience.

### 9.3 Why the `TicketEvent.metadata.autoAdvanced` Is `false` at Write Time

The `TicketEvent` is written inside the no-show transaction, but the auto-advance result is not known until AFTER the transaction. The agent has two options:

- **(a)** Write `autoAdvanced: false` in the event metadata, then UPDATE the event row after the auto-advance to set the correct value.
- **(b)** Write `autoAdvanced: false` and don't update the event. The `TicketNoShowResponse` carries the correct `autoAdvanced` value, and the SSE event payload includes the correct value.

The document recommends **(b)** ŌĆö the `TicketEvent.metadata.autoAdvanced` is informational and may be `false` even when auto-advance succeeded (the canonical truth lives in the response object and the SSE event). The agent MAY choose (a) if it improves observability ŌĆö both are acceptable.

### 9.4 Why the Stub Page Lives Under `(officer)`

The `(officer)` route group is reserved for officer-side pages. The full officer dashboard (Phase 4.2.3) will share this layout. By placing the stub here, the migration to Phase 4.2.3 is a layout swap, not a route migration.

### 9.5 How to Verify the SSE Event Without a Display Board

The Phase 2 `broadcastEvent()` is a no-op stub that logs to `console.debug` in development. The agent verifies the call by reading the server's stdout. When Phase 3 implements the real SSE manager, the agent re-runs the verification to confirm the event is actually delivered.
