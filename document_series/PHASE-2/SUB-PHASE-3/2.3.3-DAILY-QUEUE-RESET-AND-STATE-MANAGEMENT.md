## 1. Document Purpose

This document delivers the **daily reset mechanism** — the business process that closes out each day's queue at the boundary of a business day, archives the day's statistics, and starts a fresh per-service ticket sequence for the next business day. It introduces the **scheduled reset** (a `setTimeout`-based scheduler that fires at a configurable time, default midnight in `APP_TIMEZONE`), the **`QueueDailySnapshot` archive** with all eight statistics (`totalIssued`, `totalServed`, `totalNoShow`, `totalCancelled`, `totalWaiting`, `averageWaitMinutes`, `averageServiceMinutes`, `peakHour`), the **per-service `currentTicketNumber` reset to 0**, the **in-progress ticket preservation policy** (tickets in `CALLED`, `RECALLED`, or `SERVING` are NOT modified), the **manual reset override** endpoint (a super-admin-only `POST /api/admin/reset-queue?confirm=RESET_TODAY`), the **`DAILY_RESET` SSE event payload**, the **`DAILY_RESET_MANUAL` audit log action**, and the **seeding of the four queue-related `SystemSetting` defaults** that 2.2.1 and 2.3.2 consume.

When this document is complete:

- At the configured daily reset time (default `00:00` in `APP_TIMEZONE`), the system automatically:
  1. Iterates over every service and computes the day's statistics from the day's `Ticket` and `TicketEvent` records.
  2. Creates or upserts a `QueueDailySnapshot` per service for the previous business date.
  3. Resets `Service.currentTicketNumber` to `0` for every service.
  4. Emits a `DAILY_RESET` event on the global channel with a summary payload.
  5. Logs the reset to the console (in development) for operator visibility.
- Tickets in `CALLED`, `RECALLED`, or `SERVING` state at the moment of reset are NOT modified — the officer can still complete or no-show them.
- A super-admin can manually trigger the reset via `POST /api/admin/reset-queue?confirm=RESET_TODAY` for the previous business date (or any specified date via an optional body parameter).
- The reset is idempotent — running it twice for the same business date updates existing snapshots rather than creating duplicates.
- The four `SystemSetting` defaults (`queue.daily_reset_time`, `queue.no_show_grace_period_seconds`, `queue.auto_advance_on_no_show`, `queue.default_average_service_time_minutes`) are seeded on first boot.

This document does **not** implement the reports dashboard that queries `QueueDailySnapshot` (Phase 5.1), the real SSE delivery (Phase 3.1.3), any cron / scheduled-job framework beyond a simple in-process timer (future concern), or rate limiting on the manual reset endpoint (Phase 5).

### 2.2 Out of Scope (Deferred to Other Documents)

- The reports dashboard that queries `QueueDailySnapshot` for analytics. Owned by Phase 5.1.
- The real SSE delivery. Owned by Phase 3.1.3.
- A cron / scheduled-job framework (BullMQ, node-cron, etc.) — Phase 2.3.3 uses an in-process `setTimeout`. Multi-server deployments are a future concern (Phase 5 or later).

- Rate limiting on the manual reset endpoint. Owned by Phase 5.2.1.
- The `SERVING` / `COMPLETED` ticket transitions that would populate `totalServed` and `averageServiceMinutes` — these transitions are deferred, so the snapshot fields are `0` and `null` respectively until a later phase implements them.

### 2.3 What This Document Does NOT Touch

- It does **not** modify the Prisma schema.
- It does **not** implement the `SERVING` / `COMPLETED` ticket transitions (the snapshot fields are documented as `0` / `null` until those transitions exist).
- It does **not** modify or delete any ticket. The reset only writes to `QueueDailySnapshot` and resets `Service.currentTicketNumber`.
- It does **not** touch tickets in `CALLED`, `RECALLED`, or `SERVING` state.
- It does **not** write to the `AuditLog` table from the SCHEDULED reset. Only the manual reset writes an `AuditLog` row.
- It does **not** implement the real SSE delivery (the `broadcastEvent()` stub from 2.1.1 is called but is a no-op in Phase 2).

### Step 6: Create `src/lib/queue-reset.ts` — The Core Reset Logic

**Goal:** Implement the per-service reset logic — snapshot statistics, snapshot upsert, counter reset — as the core business function that both the scheduler and the manual endpoint call.

**Files affected:** `src/lib/queue-reset.ts` (new file).

**Specific instructions:**

Create `src/lib/queue-reset.ts` containing:

- **Helper function: `async function getTicketsForBusinessDate(tx: Prisma.TransactionClient, serviceId: string, businessDate: Date): Promise<Ticket[]>`**:
  - Returns all tickets for the given `serviceId` and `businessDate` (inclusive of `WAITING`, `CALLED`, `RECALLED`, `SERVING`, `COMPLETED`, `NO_SHOW`, `CANCELLED`, `TRANSFERRED`).
- **Helper function: `async function computeSnapshotStatistics(tx: Prisma.TransactionClient, serviceId: string, businessDate: Date): Promise<SnapshotStatistics>`**:
  - **Counts (single Prisma queries, scoped to `tx`):**
    - `totalIssued` = count of all tickets for `(serviceId, businessDate)`.
    - `totalServed` = count of tickets with `status = 'COMPLETED'`. **Note:** COMPLETED is not implemented in Phase 2, so this is `0` for now. The agent still writes the count query — it will produce `0` until later phases add the transition.
    - `totalNoShow` = count of tickets with `status = 'NO_SHOW'`.
    - `totalCancelled` = count of tickets with `status = 'CANCELLED'`. Also `0` in Phase 2.
    - `totalWaiting` = count of tickets with `status = 'WAITING'` (the remaining queue at day close; excludes in-progress tickets).
  - **`averageWaitMinutes`**:
    - Query all tickets where `status IN ('CALLED', 'RECALLED', 'SERVING', 'COMPLETED', 'NO_SHOW')` AND `calledAt IS NOT NULL` for this `(serviceId, businessDate)`.
    - For each ticket, compute `(calledAt - issuedAt) / 60000` (minutes).
    - Average the values (sum / count).
    - Return `null` if no tickets have been called.
    - Use plain JS arithmetic (no external library).
  - **`averageServiceMinutes`**:
    - Query all tickets where `status = 'COMPLETED'` AND `servedAt IS NOT NULL` AND `completedAt IS NOT NULL`.
    - Compute `(completedAt - servedAt) / 60000` per ticket; average.
    - Return `null` in Phase 2 (no COMPLETED tickets exist yet).
  - **`peakHour`**:
    - Query all `TicketEvent` records where `eventType = 'CALLED'` AND `ticket.serviceId = serviceId` AND the event's `createdAt` falls on the `businessDate`.
    - Group the events by hour of day (0-23, using `APP_TIMEZONE` for the hour computation).
    - Return the hour with the highest count, or `null` if no CALLED events exist.
    - Implementation note: use a plain JS object or `Map<number, number>` to count, then `reduce` to find the max.
- **Function: `async function resetServiceForBusinessDate(tx: Prisma.TransactionClient, serviceId: string, businessDate: Date): Promise<PerServiceResetResult>`**:
  - All work for one service runs inside ONE `$transaction` (atomicity per service).
  - Steps:
    1. Compute statistics via `computeSnapshotStatistics(tx, serviceId, businessDate)`.
    2. **Upsert** the snapshot: `tx.queueDailySnapshot.upsert({ where: { businessDate_serviceId: { businessDate, serviceId } }, update: { totalIssued, totalServed, totalNoShow, totalCancelled, totalWaiting, averageWaitMinutes, averageServiceMinutes, peakHour }, create: { businessDate, serviceId, ...same stats... } })`. This is the idempotency mechanism — running twice for the same date updates the snapshot rather than creating a duplicate.
    3. **Reset** `Service.currentTicketNumber` to `0`: `tx.service.update({ where: { id: serviceId }, data: { currentTicketNumber: 0 } })`.
  - Returns `{ serviceId, serviceName, snapshotUpserted: true, counterReset: true, error: null }` on success.
  - On error, returns `{ ..., error: <message> }`. The caller decides whether to continue or abort (the caller — `runDailyReset` — continues by default).
- **Function: `async function runDailyReset(options: ResetOptions): Promise<ResetResult>`**:
  - The orchestrating function. Called by the scheduler (with `trigger: 'SCHEDULED'`) and the manual route handler (with `trigger: 'MANUAL'`, `triggeredByUserId`).
  - Steps:
    1. Query all services: `db.service.findMany({ select: { id: true, name: true } })`.
    2. For each service, call `resetServiceForBusinessDate(...)` INSIDE its own `$transaction`. Use a try/catch around each service — failures in one service do NOT block the others.
    3. Aggregate the per-service results into the `ResetResult`.
    4. **After all services are processed** (NOT inside any transaction):
       - Build the `DAILY_RESET` SSE payload.
       - Generate an `sseEventId`.
       - Emit `broadcastEvent('global', 'DAILY_RESET', { ...payload, _eventId: sseEventId })`.
       - Log to `console.info` in development: `Daily reset completed at <resetAt>: <totalSnapshotsUpserted> snapshots upserted, <totalCountersReset> counters reset, <N> errors.`
    5. Return the `ResetResult`.

### Step 7: Create `src/lib/reset-scheduler.ts` — The In-Process Scheduler

**Goal:** Schedule the daily reset to run at the configured time, re-scheduling itself after each fire.

**Files affected:** `src/lib/reset-scheduler.ts` (new file).

**Specific instructions:**

Create `src/lib/reset-scheduler.ts` containing:

- **Module-level state:**
  - `let currentTimer: NodeJS.Timeout | null = null` — the active `setTimeout` handle. Used for cancellation on shutdown (if needed; for Phase 2, the timer runs for the life of the process).
  - `let isRunning = false` — guard against re-entrance (if a previous reset is still running when the next fire time arrives).
- **Function: `async function getDailyResetTimeToday(now: Date): Promise<Date>`**:
  - Reads `SystemSetting.queue.daily_reset_time` (default `"00:00"`).
  - Parses the value as `HH:MM`.
  - Computes the next reset time as a `Date`:
    - Convert `now` to the `APP_TIMEZONE` (via `Intl.DateTimeFormat`).
    - Extract year, month, day.
    - Construct the reset time as a `Date` representing `HH:MM:00.000` on that day in `APP_TIMEZONE` (convert to a UTC `Date` instant using the timezone offset).
    - If `now >= resetTimeToday`, the next reset is TOMORROW at the same time.
  - Returns the computed `Date`.
- **Function: `scheduleNextReset(): void`**:
  - If `isRunning = true`, skip (a reset is currently in progress).
  - Compute the next reset time via `getDailyResetTimeToday(new Date())`.
  - Compute `msUntilReset = nextResetTime - now`.
  - If `msUntilReset < 0`, log a warning and re-schedule for tomorrow (defensive).
  - Call `setTimeout(async () => { ... }, msUntilReset)`. Store the handle in `currentTimer`.
  - When the timer fires:
    1. Compute `previousBusinessDate` (midnight of the day that just ended in `APP_TIMEZONE`).
    2. Set `isRunning = true`.
    3. Call `runDailyReset({ previousBusinessDate, now: new Date(), trigger: 'SCHEDULED', triggeredByUserId: null })`.
    4. Catch and log any error (the scheduler must not crash on a failed reset — it tries again tomorrow).
    5. Set `isRunning = false`.
    6. Call `scheduleNextReset()` again to schedule the next day.
- **Function: `startResetScheduler(): void`**:
  - Public entry point. Called once from `src/instrumentation.ts` on app boot.
  - Calls `scheduleNextReset()`.
  - Logs to `console.info` in development: "Daily reset scheduler started. Next reset at <nextResetTime>."
- **Function: `stopResetScheduler(): void`** (optional):
  - Clears `currentTimer`. Called on graceful shutdown (Next.js hooks into this via `process.on('SIGTERM')` if needed — for Phase 2, this is best-effort).

## 8. Common Pitfalls

- **Forgetting the `instrumentation.ts` registration.** Next.js only runs `src/instrumentation.ts` if the file exists at the root of `src/`. If the agent places it in a sub-folder (e.g., `src/lib/instrumentation.ts`), the scheduler never starts.
- **Edge runtime crash.** `process.env.NEXT_RUNTIME !== 'nodejs'` guard is required. The scheduler uses `setTimeout` which is not available in the edge runtime.
- **Re-emitting `DAILY_RESET` outside `runDailyReset`.** The event is emitted inside `runDailyReset` after all services are processed. The route handler does NOT re-emit — doing so produces duplicate events.
- **Forgetting `APP_TIMEZONE` in the reset time computation.** The reset time is in `APP_TIMEZONE`, not UTC. Without timezone awareness, the reset may fire at the wrong hour for non-UTC deployments.
- **Running the reset inside a single `$transaction` for ALL services.** This is a single-point-of-failure design — one slow service blocks the others. The recommended pattern is per-service transactions.
- **Hardcoding the snapshot statistics queries to read fields that don't exist.** The COMPLETED status transitions are not implemented in Phase 2. The agent's queries for `totalServed`, `averageServiceMinutes`, and `totalCancelled` correctly return `0` / `null` for now; the queries exist but produce empty results. Do NOT skip writing them.
- **Using `setInterval` instead of `setTimeout`-with-reschedule.** `setInterval` drifts over time. The recommended pattern is to compute the exact next fire time and use `setTimeout`, re-scheduling after each fire.
- **Forgetting to test the scheduler firing in development.** A test that sets the reset time to 30 seconds in the future is the easiest way to verify the scheduler works. Without this test, the scheduler might silently fail to fire.
- **Allowing `previousBusinessDate` to be in the future.** The manual reset endpoint does NOT enforce that the date is in the past, but the agent should document that resetting the future is meaningless and may produce empty snapshots. The verification step does NOT include a future-date test.
- **Re-seeding settings on every app restart.** The `upsert` with empty `update` is the fix — existing settings are not overwritten.

## 9. AI Agent Guidance

This section is for the AI agent that will implement this document. It supplements the general guidance in Sub-Phase 2.3 Section 8.

### 9.1 Why the Scheduler Uses `setTimeout` Instead of `setInterval`

`setInterval(fn, ms)` fires at fixed intervals from the START of the timer, not from the END of the previous fire. Over time, this drifts. `setTimeout(fn, ms)` with re-scheduling after each fire is drift-free — the next fire is computed from the current time + the configured interval.

### 9.2 Why Per-Service Transactions Instead of One Big Transaction

A single transaction for all services holds locks on every service row for the duration of the reset. If one service's statistics computation is slow (e.g., a large `TicketEvent` query), all other services are blocked. Per-service transactions isolate the failures and reduce contention.

### 9.3 Why the Manual Reset Writes an `AuditLog` Row But the Scheduled Reset Does Not

The manual reset is an **administrative action** (a user with `system:configure` clicked a button). The scheduled reset is a **system action** (a timer fires). Per Phase 2 cross-cutting standard 8.4 and the master plan's audit log guidance, administrative actions are logged; system-driven recurring actions are not.

### 9.4 Why the Reset Uses `Intl.DateTimeFormat` Instead of a Timezone Library

Node's built-in `Intl.DateTimeFormat` is sufficient for converting a `Date` to a year/month/day/hour in a specific IANA timezone. A library like `date-fns-tz` or `luxon` is overkill for this single use case. The agent can introduce a library later if complexity grows (Phase 5 or later).

### 9.5 How to Test the Scheduled Reset Without Waiting for Midnight

The fastest test is to set `SystemSetting.queue.daily_reset_time` to a time 30 seconds in the future, wait 60 seconds, and verify the console logs the reset message and a snapshot is created. The agent restores the setting after the test.

### 9.6 How to Verify the Reset is Idempotent

Run `runDailyReset` twice in a row (e.g., via the manual endpoint, then again immediately). Verify that the `QueueDailySnapshot` row count for the `(businessDate, serviceId)` pair is `1`, not `2`. The `upsert` pattern guarantees this.

### 9.7 What "In-Progress Preservation" Means in Practice

A ticket issued on day D, called on day D, and still `CALLED` at the reset time on day D+1:
- Its `businessDate` is D (unchanged).
- It does NOT appear in the day D snapshot's `totalWaiting` (only `WAITING` tickets count).
- It DOES appear in day D's snapshot's `totalIssued` (it was issued on D).
- The officer can still no-show or recall it on day D+1.

The reset never modifies ticket rows except via the `Service.currentTicketNumber` reset and the snapshot upsert.

### 9.8 Why the `DAILY_RESET` Event Is on the Global Channel

The display board (Phase 3.2) and the kiosks need to know when the reset happens so they can clear their in-memory queue state and start fresh. The global channel is the right place for system-wide events. The `counter` channel is reserved for officer-specific events.

### 9.9 When to Stop and Ask

- If the `peakHour` computation method (counting `CALLED` events per hour, not `ISSUED` events) is not what the operator wants, the agent should ask before implementing.
- If the human reviewer wants the scheduled reset to fire at a different default time (not midnight), the agent updates `QUEUE_DEFAULT_SETTINGS[0].value` and documents the change.
- If the audit log action name (`DAILY_RESET_MANUAL`) is not what the reviewer wants, the agent asks before committing.
