## 1. Sub-Phase Purpose

Sub-Phase 2.3 is the **counter-side of the queue** — and the final sub-phase of Phase 2. It closes the loop on the ticket lifecycle that began at the kiosk (Sub-Phase 2.2) by implementing every operation a counter officer performs on a ticket: **calling** a waiting ticket to the counter, **recalling** a ticket that was previously called (the customer didn't appear in time, the officer wants to re-cue), and **marking a ticket as no-show** (the customer definitely did not appear). It also implements the **daily reset** mechanism — the business process that closes out each day's queue, archives statistics, and starts a fresh sequence for the next business day.

When this sub-phase is complete, the system has a **fully functional queue at the API and database level**. Every ticket state transition (issue → wait → call → recall → no-show, plus the daily reset transition) is guarded, audited, and emits an event with a documented payload shape ready for Phase 3 to deliver to display boards and Phase 4 to deliver to officer devices. No client yet consumes these events in Phase 2, but the data shape and call sites are locked in by the end of this sub-phase.

The three problems this sub-phase solves, in order:

1. **Calling and recalling tickets** — The `POST /api/tickets/[ticketId]/call` and `/recall` endpoints, the ticket status state machine that governs every legal transition, the validation rules (officer must be on duty at the counter, counter must be active, ticket must be in a state that allows the operation), the `TicketEvent` audit row for each transition, and the `TICKET_CALLED` and `TICKET_RECALLED` SSE event payload definitions. (Document 2.3.1.)
2. **No-show handling and queue advancement** — The `POST /api/tickets/[ticketId]/no-show` endpoint, the configurable grace period (the officer cannot mark a freshly-called ticket as no-show), the optional automatic advancement to the next waiting ticket (a system setting), the `TicketEvent` audit row with grace-period metadata, the `TICKET_NO_SHOW` SSE event payload, and a minimal temporary officer ticket action panel that exposes Call / Recall / No-Show buttons. (Document 2.3.2.)
3. **Daily reset and queue state management** — The scheduled reset mechanism (a daily timer that runs at a configurable time, default midnight in `APP_TIMEZONE`), the per-service ticket counter reset to 0, the `QueueDailySnapshot` archive with daily statistics, the manual reset override (a super-admin endpoint with a confirmation parameter), the in-progress ticket preservation policy (tickets in CALLED/RECALLED/SERVING state are NOT modified by the reset), and the `DAILY_RESET` SSE event payload. (Document 2.3.3.)

When this sub-phase is complete:

- A counter officer can call the next waiting ticket for any service assigned to their counter, recall it, and mark it no-show — with full validation, audit, and event emission at every step.
- The system rejects every illegal ticket state transition with a clear error.
- A super-admin can trigger a manual reset, and the system automatically resets at the configured daily time.
- Every emitted event has its payload shape locked in `src/types/sse.types.ts`, ready for Phase 3.1.3 to implement real SSE delivery.
- The `QueueDailySnapshot` table is populated with the day's statistics for every service, ready for Phase 5's reporting layer.

After Sub-Phase 2.3, **Phase 2 is complete**. The system has a working queue engine; it does not yet display anything, announce anything, notify officers, or report on performance — but all the data needed for those features is being produced and persisted.

## 1.1 Why This Sub-Phase Closes Phase 2

Phase 2's strategic goal is to **build the working queue engine**: the structural configuration (services, counters), the customer-facing ticket issuance (kiosk), the counter-side ticket lifecycle (this sub-phase), and the daily business cycle (also this sub-phase). The seam between Phase 2 and Phase 3 is the **broadcast event interface** — Phase 2 produces events, Phase 3 delivers them. Sub-Phase 2.3 produces the four most important event types for the queue experience (`TICKET_CALLED`, `TICKET_RECALLED`, `TICKET_NO_SHOW`, `DAILY_RESET`), all of which Phase 3 needs to render the display board and trigger the audio announcement system.

Without Sub-Phase 2.3:

- Issued tickets in Sub-Phase 2.2 would sit in `WAITING` state forever — no officer could ever call them.
- There would be no ticket state transitions to demonstrate the system working end-to-end.
- The `currentTicketNumber` sequence would never reset, so day-old tickets would mix with new tickets.
- The display board (Phase 3) would have no events to listen to.

This sub-phase is therefore not optional — it is the **operational heartbeat** of the queue.

## 1.2 What This Sub-Phase Is NOT

- It is **not** a real-time / SSE delivery implementation. The `broadcastEvent()` function is still the no-op stub from 2.1.1; Phase 3.1.3 replaces it. Sub-Phase 2.3 only **calls** the function and **defines** the payload shapes.
- It is **not** the display board, audio announcements, or any display-side UI. Phase 3.
- It is **not** push notifications, FCM dispatch, or the counter temporary-closure toggle. Phase 4.
- It is **not** the full officer dashboard. The minimal `TicketActionPanel` from 2.3.2 is a **temporary** interaction surface — it is replaced by Phase 4.2.3's full officer dashboard layout (with sidebar, notification toggle, counter status toggle, recent activity feed, queue depth indicator).
- It is **not** the `SERVED` / `COMPLETED` / `CANCELLED` / `TRANSFERRED` ticket state transitions. The `TicketStatus` enum supports them, but this sub-phase does not implement the transitions or the corresponding endpoints — they are deferred to a later phase (the master plan does not allocate these transitions to any specific sub-phase; the document specifies them as out of scope here).
- It is **not** reporting or analytics. The `QueueDailySnapshot` is populated, but the reports dashboard is Phase 5.
- It is **not** rate limiting, security headers, or production hardening. Phase 5.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to **distribute the complexity** across the three task documents. The split is:

- **2.3.1** owns the **ticket status state machine and the call/recall lifecycle**. It introduces the state machine pure function in `lib/ticket-service.ts`, the call/recall endpoints, the validation rules for officer-on-duty and counter-active, the `TicketEvent` audit rows for CALL and RECALL, the `TICKET_CALLED` and `TICKET_RECALLED` SSE event payload variants, and the minimal officer-call context (a server-side helper that fetches "next waiting ticket for a counter" for the temporary action panel). This document does **not** implement no-show (owned by 2.3.2) or the daily reset (owned by 2.3.3).
- **2.3.2** owns the **no-show handling and the temporary officer action panel**. It introduces the no-show endpoint, the grace period logic, the auto-advance logic (the optional automatic call of the next waiting ticket), the `TicketEvent` audit row for NO_SHOW with grace-period metadata, the `TICKET_NO_SHOW` SSE event payload, and the minimal `TicketActionPanel` component + stub page at `/(officer)/counter/[counterId]/page.tsx`. This document **depends on 2.3.1** (it reuses the state machine, the `callTicket` orchestrating function for auto-advance, and the SSE event registry).
- **2.3.3** owns the **daily reset and state management**. It introduces the scheduled reset mechanism (`setInterval`-based), the `QueueDailySnapshot` population logic with all eight statistics, the per-service counter reset, the in-progress ticket preservation policy, the manual reset override endpoint, the `DAILY_RESET` SSE event payload, and the seeding of the four queue-related `SystemSetting` defaults. This document depends on 2.2.1 (for `Service.currentTicketNumber` and `TicketEvent` queries for statistics) and 2.3.1 + 2.3.2 (for the state machine and event registry).

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 2.3.1 — Call & Recall Ticket Operations

**Owns:** The ticket status state machine for call/recall transitions, the call/recall endpoints, the validation rules for officer-on-duty and counter-active, the audit rows, and the SSE event payload definitions for `TICKET_CALLED` and `TICKET_RECALLED`.

**Does NOT own:** The no-show transition (owned by 2.3.2), the daily reset (owned by 2.3.3), the kiosk UI (owned by 2.2.x), the real SSE delivery (Phase 3.1.3).

**Key outputs:**

- `src/lib/ticket-state-machine.ts`: the pure state machine module — `canTransition(from: TicketStatus, to: TicketStatus, action: TicketAction): boolean` (or `transitionTicket(from, to, action)` returning the new status / throwing on illegal). The module exports the full set of legal transitions for this sub-phase as a typed constant array, so other documents (and the future Android app via the API) can read the rules.
- `src/types/ticket.types.ts` (extension, from 2.2.1): the `TicketCallInput` / `TicketRecallInput` types, the `TicketCallResponse` / `TicketRecallResponse` types, the `TicketActionResult` discriminated union.
- `src/schemas/ticket.schema.ts` (extension, from 2.2.1): `callTicketParamsSchema`, `recallTicketParamsSchema`, plus a shared `ticketActionParamsSchema` (since both routes take only `ticketId` from the path).
- `src/lib/ticket-service.ts` (extension, from 2.2.1): the orchestrating `callTicket(input, actor)` and `recallTicket(input, actor)` functions. Each wraps a single `$transaction` that contains: (1) the `Ticket` row update (status + `calledAt` / `recalledAt` + `counterId` + `calledByOfficerId`), (2) the `TicketEvent` row creation. The `broadcastEvent('global', 'TICKET_CALLED' | 'TICKET_RECALLED', payload)` call is **outside** the transaction.
- API endpoints:
  - `POST /api/tickets/[ticketId]/call` — guarded with `withPermission(handler, PERMISSION_COUNTER_CALL)`.
  - `POST /api/tickets/[ticketId]/recall` — guarded with `withPermission(handler, PERMISSION_COUNTER_CALL)`.
- `src/types/sse.types.ts` (extension, from 2.1.1): the `TICKET_CALLED` and `TICKET_RECALLED` payload variants added to the `SseEventPayload` discriminated union. Field shapes specified by Master Plan Section 11.3 (with the recall variant adding a `recallCount` field).
- `src/lib/ticket-officer.ts`: a server-side helper that resolves the calling officer from the session — `resolveCallingOfficer(userId, counterId)`. Returns the `CounterOfficer` record if the officer is on duty at the counter; throws a specific error otherwise.
- **Validation rules:**
  - **Officer must be on duty.** The calling user's session must resolve to a `CounterOfficer` record with `isOnDuty = true` for the counter ID provided in the request body (the document specifies: the counter ID comes from the request body for both `/call` and `/recall`, NOT from the URL — the officer's counter is part of the action context, not the ticket's identity).
  - **Counter must be active.** The `Counter.isActive` must be `true`. Deactivated counters cannot receive calls.
  - **Ticket must be in a state that allows the action.**
    - For `call`: `from` must be `WAITING`, `RECALLED` (a re-call), or `NO_SHOW` (giving another chance). It must NOT be `CALLED`, `SERVING`, `COMPLETED`, `TRANSFERRED`, or `CANCELLED`.
    - For `recall`: `from` must be `CALLED` or `RECALLED`. The recall increments an internal `recallCount` counter on the `Ticket` (added in this document or in 2.2.1's TicketEvent metadata — the document specifies exactly where the count is stored).
  - **Ticket's service must be assigned to the counter.** The endpoint checks the `CounterService` join table to ensure the counter handles the ticket's service. This prevents an officer at Counter 1 (which only handles "General Inquiry") from calling a "Medical" ticket.
- **Ticket state machine specification (textual):**
  - Legal transitions implemented in this sub-phase:
    - `WAITING → CALLED` (on call)
    - `CALLED → RECALLED` (on recall)
    - `RECALLED → CALLED` (on call again)
    - `NO_SHOW → CALLED` (on call, giving another chance)
  - Deferred (NOT in this sub-phase; the `TicketStatus` enum supports them but they are not implemented):
    - `CALLED → SERVING` and `SERVING → COMPLETED` — the master plan does not allocate these to any specific sub-phase; they are out of scope here.
    - `* → CANCELLED` and `* → TRANSFERRED` — also deferred.
- **`TicketEvent` audit row:**
  - For `call`: `eventType = 'CALLED'`, `ticketId`, `counterId`, `officerId`, `metadata = { previousStatus, calledAt }`.
  - For `recall`: `eventType = 'RECALLED'`, `ticketId`, `counterId`, `officerId`, `metadata = { previousStatus, recalledAt, recallCount }`.
- **No `AuditLog` writes** (per Phase 2 standard 8.4 — ticket lifecycle is high-volume; `TicketEvent` is the authoritative audit trail).
- **No `withRole()` override.** The `counter:call` permission is sufficient. Super-admins with `counter:manage` can also call (the `withPermission` helper checks for any of the listed permissions).

**Master Plan sections implemented:** Sections 6.5 (`TicketActionPanel` — stubs the props here, full implementation in 2.3.2), 8.2 (`Ticket` and `TicketEvent` model usage), 9.3 (call/recall endpoints), 11.2 (SSE envelope), 11.3 (`TICKET_CALLED` payload).

### 4.2 Document 2.3.2 — No-Show Handling & Queue Advancement

**Owns:** The no-show endpoint, the grace period enforcement, the optional auto-advance to the next waiting ticket, the audit row, the `TICKET_NO_SHOW` SSE event payload, and the temporary `TicketActionPanel` component + stub officer page.

**Does NOT own:** The call/recall endpoints (owned by 2.3.1), the daily reset (owned by 2.3.3), the kiosk UI (owned by 2.2.x), the full officer dashboard layout (owned by Phase 4.2.3).

**Key outputs:**

- `src/lib/ticket-service.ts` (extension, from 2.2.1 and 2.3.1): the orchestrating `noShowTicket(input, actor)` function. Wraps a `$transaction` that: (1) verifies the grace period (computed from `Ticket.calledAt` against `now`), (2) updates the `Ticket` (status → `NO_SHOW`, `noShowAt`), (3) creates the `TicketEvent` audit row.
- `src/lib/ticket-advance.ts`: a small helper that finds the next waiting ticket for a service and calls it (reuses `callTicket` from 2.3.1). The auto-advance orchestration lives here so the no-show function stays focused on the no-show itself.
- API endpoint:
  - `POST /api/tickets/[ticketId]/no-show` — guarded with `withPermission(handler, PERMISSION_COUNTER_CALL)`.
- `src/types/sse.types.ts` (extension, from 2.1.1 and 2.3.1): the `TICKET_NO_SHOW` payload variant.
- **Grace period logic:**
  - Reads `SystemSetting.queue.no_show_grace_period_seconds` (default 60).
  - Computes elapsed seconds since `Ticket.calledAt`.
  - If elapsed < grace period, the action is rejected with `VALIDATION_ERROR` (HTTP 422) and a clear message: "Ticket was just called — please wait at least N seconds before marking no-show."
  - The check happens inside the transaction (with `Ticket` re-fetched under the transaction's read snapshot) to prevent TOCTOU races.
- **Auto-advance logic:**
  - Reads `SystemSetting.queue.auto_advance_on_no_show` (default `true`).
  - If `true`: after the no-show transaction commits, the no-show route handler calls `advanceToNextWaitingTicket(serviceId, counterId, officerId)` which re-uses `callTicket` from 2.3.1 to call the next waiting ticket. If no waiting ticket exists, the auto-advance is a no-op.
  - The auto-advance is logged: the `noShowTicket` response includes an `autoAdvancedTicket: IssuedTicket | null` field so the officer UI knows whether the next ticket was automatically called.
  - The auto-advance is performed **sequentially after** the no-show transaction (NOT inside the no-show transaction). If the auto-advance fails, the no-show itself remains committed — the failure is logged via `console.error`.
- **`TicketEvent` audit row:**
  - `eventType = 'NO_SHOW'`, `ticketId`, `counterId`, `officerId`, `metadata = { previousStatus, gracePeriodSeconds, elapsedSeconds, autoAdvanced: boolean, autoAdvancedTicketId: string | null }`.
- **`TICKET_NO_SHOW` SSE event payload (additive variant on `SseEventPayload`):**
  - Fields: `ticketId`, `ticketNumber`, `serviceId`, `serviceName`, `counterId`, `counterNumber`, `calledByOfficerId`, `calledByOfficerName`, `noShowAt`, `gracePeriodSeconds`, `elapsedSeconds`, `autoAdvanced` (boolean), `autoAdvancedTicketNumber` (string | null).
- **Temporary `TicketActionPanel` component:**
  - File: `src/components/counter/ticket-action-panel.tsx`.
  - "use client" directive.
  - **Props:** `{ ticket: TicketDetail | null; counterId: string; officerOnDuty: boolean }`.
  - Renders three buttons: **Call**, **Recall**, **No-Show**.
  - Each button is enabled only when the corresponding state transition is legal (uses the same state machine rules from 2.3.1, applied client-side as a UX hint).
  - Buttons call the corresponding API endpoints (`POST /api/tickets/[ticketId]/call`, `/recall`, `/no-show`).
  - After each successful action, the panel re-fetches the current ticket state (calls `GET /api/tickets?counterId=<id>&status=CALLED` or `status=SERVING` to find what's now in progress at the counter).
  - Loading state per button (spinner inside the button during in-flight request).
  - Error state: Toast on failure.
- **Temporary stub page:** `src/app/(officer)/counter/[counterId]/page.tsx`.
  - Server component.
  - Guards with `auth()` + manual `counter:read` check (the officer must at least see the counter).
  - Resolves the current serving ticket for the counter (`GET /api/tickets?counterId=<id>&status=CALLED`).
  - Renders the page header (counter name + number), the current serving ticket display (using `<TicketBadge>` and `<StatusChip>` from 2.2.1), and the `<TicketActionPanel>`.
  - If no serving ticket, shows the next waiting ticket preview and a single "Call Next" button (which calls the call endpoint directly without a `ticketId` — this is the "next ticket" path that the document specifies: it accepts an optional `serviceId` body parameter; if absent, the counter's first assigned service is used).
  - **Note:** this page is a TEMPORARY stub. Phase 4.2.3 replaces it with the full officer dashboard layout.
- **No `AuditLog` writes** (per Phase 2 standard 8.4).

**Master Plan sections implemented:** Sections 6.5 (`TicketActionPanel`), 8.2 (`TicketEvent` for no-show), 9.3 (no-show endpoint), 11.2 (`TICKET_NO_SHOW` event).

### 4.3 Document 2.3.3 — Daily Queue Reset & State Management

**Owns:** The scheduled daily reset mechanism, the `QueueDailySnapshot` population with all eight statistics, the per-service `currentTicketNumber` reset to 0, the in-progress ticket preservation policy, the manual reset override endpoint, the `DAILY_RESET` SSE event payload, and the seeding of the four queue-related `SystemSetting` defaults.

**Does NOT own:** The call/recall/no-show endpoints (owned by 2.3.1, 2.3.2), the kiosk UI (owned by 2.2.x), any cron / scheduled-job framework (the document specifies a simple in-process `setInterval` approach — a real cron or BullMQ is a future concern), the reports dashboard (Phase 5.1), any timezone library (uses Node's built-in `Intl.DateTimeFormat`).

**Key outputs:**

- `src/lib/queue-reset.ts`: the core reset logic — `runDailyReset({ now: Date, previousBusinessDate: Date }): Promise<ResetResult>`. The function is idempotent (re-running for the same `previousBusinessDate` updates existing snapshots rather than creating duplicates). It iterates over all services, computes the day's statistics, creates or upserts a `QueueDailySnapshot`, and resets `Service.currentTicketNumber` to 0.
- `src/lib/reset-scheduler.ts`: the in-process scheduler that triggers the reset at the configured time. Implementation: on app boot, compute the next reset time; set a `setTimeout` for that time; when it fires, run the reset, then compute the next reset time and schedule again. The scheduler is started by a `Next.js` instrumentation hook (`src/instrumentation.ts`).
- `src/instrumentation.ts`: the Next.js instrumentation hook that starts the reset scheduler on server boot. The hook is registered only when `process.env.NEXT_RUNTIME === 'nodejs'` (not during edge runtime) and only when the app boots once per process (not on hot-reloads).
- API endpoint:
  - `POST /api/admin/reset-queue?confirm=RESET_TODAY` — guarded with `withPermission(handler, PERMISSION_SYSTEM_CONFIGURE)` (super-admin only).
- `src/types/sse.types.ts` (extension, from 2.1.1, 2.3.1, 2.3.2): the `DAILY_RESET` payload variant.
- **Default `SystemSetting` seeding:**
  - On app boot (in the same `src/instrumentation.ts` hook that starts the scheduler), this document seeds the following default settings if they do not exist:
    - `queue.daily_reset_time` — value `"00:00"`, type `STRING`, default "00:00" (24-hour format in `APP_TIMEZONE`).
    - `queue.no_show_grace_period_seconds` — value `"60"`, type `INTEGER`, default 60.
    - `queue.auto_advance_on_no_show` — value `"true"`, type `BOOLEAN`, default `true`.
    - `queue.default_average_service_time_minutes` — value `"5"`, type `INTEGER`, default 5 (this is the fallback 2.2.1 reads when a service's `averageServiceTime` is null).
  - The seeding uses `upsert` (by `key`) so it never duplicates.
- **`runDailyReset` detailed behavior:**
  - **Input:** `previousBusinessDate` (a UTC `Date` representing midnight of the day that just ended, in `APP_TIMEZONE`).
  - **For each service:**
    1. Query all `Ticket` records where `serviceId = <serviceId>` and `businessDate = <previousBusinessDate>`. **Only `WAITING` tickets are considered "remaining" — the in-progress tickets (CALLED, RECALLED, SERVING) are NOT modified by the reset and NOT counted in the snapshot.** (Per the in-progress preservation policy below.)
    2. Compute statistics from the day's ticket records:
       - `totalIssued` = count of all tickets on that date.
       - `totalServed` = count of tickets with `status = 'COMPLETED'` (note: COMPLETED is not implemented in Phase 2, so this is 0 until later phases add the COMPLETED transition; the field is still populated as 0, not null).
       - `totalNoShow` = count of tickets with `status = 'NO_SHOW'`.
       - `totalCancelled` = count of tickets with `status = 'CANCELLED'` (also not implemented in Phase 2; field is 0).
       - `totalWaiting` = count of tickets with `status = 'WAITING'` (the remaining queue at day close).
       - `averageWaitMinutes` = average of `(calledAt - issuedAt)` in minutes for tickets that were called. Null if no tickets were called.
       - `averageServiceMinutes` = average of `(completedAt - servedAt)` in minutes for tickets that were completed. Null if no tickets were completed.
       - `peakHour` = the hour (0-23) with the highest count of `CALLED` `TicketEvent` records for this service on this business date. Null if no tickets were called.
    3. **Upsert** the `QueueDailySnapshot` with composite unique on `(businessDate, serviceId)`:
       - If a snapshot exists for this `(businessDate, serviceId)`, update its fields.
       - Otherwise, create a new snapshot.
    4. **Reset** `Service.currentTicketNumber` to `0`.
  - **In-progress ticket preservation:** tickets in `CALLED`, `RECALLED`, or `SERVING` state are NOT touched by the reset. Their `businessDate` is unchanged. The officer at the counter can still complete, recall, or no-show them.
  - **Per-service transaction:** the four steps (compute statistics, upsert snapshot, reset counter) for a single service are wrapped in one `$transaction` for atomicity. Failures in one service's reset do not block other services (the iteration uses a try/catch that logs and continues).
- **`DAILY_RESET` SSE event payload (additive variant on `SseEventPayload`):**
  - Fields: `resetAt`, `previousBusinessDate`, `affectedServiceIds: string[]`, `totalSnapshotsUpserted: number`, `totalCountersReset: number`.
- **Manual reset override endpoint:**
  - `POST /api/admin/reset-queue?confirm=RESET_TODAY`.
  - Guarded with `PERMISSION_SYSTEM_CONFIGURE` (super-admin only).
  - Body: optional `{ previousBusinessDate: string }` (ISO date). If absent, uses yesterday in `APP_TIMEZONE`.
  - The `?confirm=RESET_TODAY` query parameter is required — without it, the request is rejected with `VALIDATION_ERROR` (HTTP 422) and message "Confirmation parameter missing — pass `?confirm=RESET_TODAY` to confirm this destructive action."
  - Calls `runDailyReset()` synchronously, returns the `ResetResult` data.
  - Writes an `AuditLog` row: `action = 'DAILY_RESET_MANUAL'`, `entity = 'QueueDailySnapshot'`, `userId = session.user.userId`, `description = \`Manually triggered daily reset for business date \${previousBusinessDate}.\``, `metadata = { affectedServiceIds, totalSnapshotsUpserted }`.
  - Emits the `DAILY_RESET` SSE event after the reset completes.
- **Audit logging policy for this document:**
  - **Manual reset:** writes an `AuditLog` row (administrative action).
  - **Scheduled reset:** does NOT write an `AuditLog` row (system-driven, high-frequency; the `TicketEvent` and `QueueDailySnapshot` records are sufficient). The reset is logged via `console.info` in development so the operator can verify it ran.

**Master Plan sections implemented:** Sections 8.2 (`QueueDailySnapshot` model — initial population), 9.3 (admin reset endpoint), 11.1 / 11.2 (`DAILY_RESET` event in the channel registry), 11.2 (broadcast on daily reset).

**Build order:** `2.3.1` first (it produces the state machine and the call/recall endpoints that 2.3.2 re-uses for auto-advance). `2.3.2` second (it adds the no-show path on top of call/recall). `2.3.3` third (it iterates over the ticket history produced by the previous sub-phases and resets the counters; it also seeds the `SystemSetting` defaults that 2.2.1 and 2.3.2 expect).

**Why this order:**

- **2.3.1 first.** The state machine is the foundation for 2.3.2's auto-advance path (which re-calls the next waiting ticket via the call endpoint). 2.3.2 cannot work without the call endpoint.
- **2.3.2 after 2.3.1.** The no-show endpoint depends on the call endpoint (for auto-advance) and on the same officer-on-duty validation. Both are 2.3.1 outputs.
- **2.3.3 after both.** The daily reset iterates over ticket history (produced by all of 2.2 and 2.3.1+2.3.2). It can run with only 2.3.1 in place (the reset itself does not need no-show to work), but it produces the most accurate snapshot when 2.3.2 is also done (so the snapshot includes `totalNoShow`). Building 2.3.3 last also means the `SystemSetting` seeding in 2.3.3 happens last — which is correct because 2.2.1 and 2.3.2 read those settings before 2.3.3 is done; they just fall back to the hard-coded defaults.

**What depends on this sub-phase:**

- **Phase 3 (display & audio)** consumes every event produced here: `TICKET_CALLED` / `TICKET_RECALLED` / `TICKET_NO_SHOW` for the announcement system; `DAILY_RESET` to clear the display between business days.
- **Phase 4 (mobile & counter management)** uses the call/recall/no-show endpoints from the Android app and the full officer dashboard. Phase 4 also adds the `SERVING` / `COMPLETED` ticket transitions.
- **Phase 5 (reports)** queries the `QueueDailySnapshot` records produced by 2.3.3 for the analytics dashboard.

- **System setting seeding (in 2.3.3) does NOT write to the `AuditLog` table.** It is a system bootstrap action; the settings table itself is the record.

- 2.3.3 calls `broadcastEvent('global', 'DAILY_RESET', payload)` after the daily reset completes (both scheduled and manual paths).
- All event calls are OUTSIDE the transaction (after commit).
- The event payload types are additive extensions to `src/types/sse.types.ts`.

### 6.10 Transaction Boundaries

Every multi-write operation in this sub-phase must run inside a single Prisma `$transaction`:

- **Call** (2.3.1): `Ticket` update + `TicketEvent` creation. In one transaction.
- **Recall** (2.3.1): `Ticket` update + `TicketEvent` creation. In one transaction.
- **No-show** (2.3.2): grace-period re-check + `Ticket` update + `TicketEvent` creation. In one transaction.
- **Auto-advance** (2.3.2): a SEPARATE `callTicket` transaction (not nested inside the no-show transaction).
- **Per-service daily reset** (2.3.3): snapshot upsert + `Service.currentTicketNumber` reset. In one transaction per service.
- The `broadcastEvent()` call is ALWAYS outside the transaction.

### 6.11 Officer Permission Discipline

- All officer-action endpoints (call, recall, no-show) are guarded with `withPermission(handler, PERMISSION_COUNTER_CALL)`.
- The endpoint logic then resolves the session user to a `CounterOfficer` record at the counter (via `resolveCallingOfficer`) and rejects if `isOnDuty = false`.
- The endpoint does NOT check `isClosed` on the `CounterOfficer` (an officer can call tickets even if their counter is temporarily closed — the system treats this as an override; the temporary-closure feature itself is Phase 4.2.1, out of scope here).
- Super-admins with `counter:manage` permission CAN call tickets (their session is allowed through `withPermission` because the `counter:call` permission is in their role's permission set, OR because Phase 1.3.2's `withPermission` helper accepts any-of-multiple permissions — the document specifies the exact permission check pattern).

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`.
- Each task plan document corresponds to one or more commits. A single commit should not span multiple task plan documents.
- Suggested commit boundaries:
  - End of 2.3.1: `feat: add ticket call and recall endpoints with state machine and TICKET_CALLED/RECALLED events`

### 8.2 General Guidance

- **Do not skip steps.** Sub-Phase 2.3 is mostly sequential within each document (each step's output is the next step's input). 2.3.1 must be done before 2.3.2; 2.3.1 and 2.3.2 must be done before 2.3.3.

Do not inline `if (status === 'WAITING') ...` checks scattered across the codebase. If you find yourself writing such a check, add a new transition rule to the state machine instead.

- **Document any deviation.** If the agent makes a choice that differs from this document, it must note the deviation in the final summary so the human reviewer can approve or reject it.

- Should not implement security headers, audit log viewer UI, or PostgreSQL migration (Phase 5).
- Should not write to the `AuditLog` table from call/recall/no-show paths. The `TicketEvent` row is sufficient per Phase 2 standard 8.4.
- Should not modify the Prisma schema (it was finalised in Phase 1.1.3 — all the fields needed for this sub-phase already exist).
- Should not hard-delete any ticket. Tickets are never deleted; they transition through statuses until `COMPLETED` / `NO_SHOW` / `CANCELLED` (latter two are not in Phase 2).

### 8.4 When to Ask the Human

The AI agent should ask the human reviewer (not guess) if any of the following situations arise:

- The master plan and the phase 2.3 overview contradict each other. The master plan wins, but the agent should confirm.
- The exact list of ticket state transitions to support in 2.3.1 is ambiguous (the document specifies the legal transitions; if the human wants more, they can request a different scope).
- The grace period default value (60 seconds) is not what the operator wants.
- The default daily reset time (`00:00` in `APP_TIMEZONE`) is not what the operator wants.
- The auto-advance default (`true`) is not what the operator wants.
- The peak hour calculation method (counting `CALLED` events per hour, not `ISSUED` events) is not what the operator wants.
- The audit log action type name for manual reset (`DAILY_RESET_MANUAL`) is not what the reviewer wants (the document specifies it; if there is a conflict with existing names, ask).
- The reset scheduler strategy (`setInterval` vs `setTimeout`-recompute vs a third-party library) is unacceptable. The document specifies `setTimeout` with re-scheduling on each fire.
- A verification step fails for a reason that the document does not anticipate.
