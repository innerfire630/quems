## 1. Sub-Phase Purpose

Sub-Phase 1.3 transforms the application from **"we know who you are"** (the state left by Sub-Phase 1.2) to **"we know what you are allowed to do"**. It defines the complete role and permission catalogue, wires enforcement through every layer of the application (server, API, client UI, middleware), and provides the user management interface that administrators will use.

This is the **final sub-phase of Phase 1**. When it completes, the application is fully role-aware: every protected route is gated by both authentication and authorization, the user-management foundation is in place, and the audit log is being written to so that every mutating user action leaves a trace.

The three problems this sub-phase solves, in order:

1. **Defining the permission system** — A complete catalogue of all permissions the system will ever check, grouped by module (USER, COUNTER, SERVICE, TICKET, NOTIFICATION, REPORT, SYSTEM). A complete catalogue of all roles the system ships with (SUPER_ADMIN, ADMIN, COUNTER_OFFICER, SECURITY_OFFICER, KIOSK) and the default mapping from roles to permissions. A default super-admin user seeded with a known credential so that the application is immediately usable. (Document 1.3.1.)
2. **Enforcing permissions everywhere** — A `withPermission()` server-side guard that wraps API routes and server actions. A `withRole()` companion guard for role-name checks. A `usePermission()` client-side hook for conditional UI rendering. A `<Can>` component for declarative permission checks in JSX. The Next.js `middleware.ts` is extended to enforce role-based route protection at the edge. The standard error response shape (401 vs 403) is applied consistently. (Document 1.3.2.)
3. **Operating the user management surface** — A super-admin-facing UI for listing, creating, editing, deactivating, and resetting passwords for users. API endpoints that back the UI, protected by `withPermission('user:manage')`. An audit log helper that writes an `AuditLog` entry for every mutating user-management action. (Document 1.3.3.)

When this sub-phase is complete, every later phase can guard its own API routes with `withPermission('<module>:<action>')` and hide its own UI behind `<Can permission="...">...</Can>` without re-inventing the enforcement layer. The Phase 1 foundation is fully closed.

---

## 1.1 Why This Sub-Phase Closes Phase 1

Every later phase (2 through 5) needs three things from Sub-Phase 1.3:

- **Display UI conditionally based on the user's permissions.** For example, only users with `report:view` should see the "Reports" link in the navigation. This is the `usePermission()` hook and the `<Can>` component.
- **Guard API routes against unauthorised access.** For example, only users with `ticket:manage` should be able to delete a ticket. This is the `withPermission()` guard.
- **Audit who did what.** The `AuditLog` table (already in the schema from 1.1.3) needs to be populated by every mutating action in 1.3.3 (user created, role assigned, password reset) so that Phase 5's audit log viewer can display meaningful history. This is the `lib/audit-log.ts` helper written in 1.3.3.

Sub-Phase 1.3 establishes the seed data (1.3.1), the enforcement layer (1.3.2), and the operating surface (1.3.3) so that all three of these needs are met for every later phase.

---

## 1.2 What This Sub-Phase Is NOT

- It is **not** a new authentication system. Authentication (login, refresh, logout) was completed in Sub-Phase 1.2. This sub-phase assumes `getServerSession()` works and returns a typed session with `userId`, `roles`, and `permissions`.
- It is **not** a redesign of the `User` or `Role` schema. The schema was finalised in 1.1.3 (User, Account, Session, RefreshToken, Role, Permission, UserRole, RolePermission). This sub-phase writes data to those tables; it does not change their structure.
- It is **not** an audit log viewer. The `AuditLog` rows written in 1.3.3 are queryable directly via Prisma (e.g., from `prisma studio`) for verification, but the UI to browse them is in Phase 5 (5.2.3).
- It is **not** OAuth or social login. The master plan provisions the `Account` table for future OAuth providers, but this sub-phase uses only the credentials provider from 1.2.1.
- It is **not** a forgotten-password flow. The admin-initiated password reset in 1.3.3 is the only password reset path in Phase 1. Self-service password reset (via email link) is a future feature.
- It is **not** rate-limited. Rate limiting on auth/user-management endpoints is Phase 5.
- It is **not** production-hardened. Security headers, CORS tightening for production, and CI/CD are all Phase 5.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to **distribute the complexity** across the three task documents. The split is:

- **1.3.1** owns the **data layer** of RBAC: the full role and permission catalogue, the role-to-permission mappings, the seed script content, and the default super-admin user. After 1.3.1, the database has all the data the rest of the system needs; 1.3.2 and 1.3.3 read from it.
- **1.3.2** owns the **enforcement layer** of RBAC: the `withPermission()` guard, the `withRole()` guard, the `usePermission()` hook, the `<Can>` component, the permission constants in `lib/permissions.ts`, and the extension of `middleware.ts` to enforce role-based route prefixes. After 1.3.2, every layer of the application can ask "does this user have permission X?" with one function call.
- **1.3.3** owns the **operating surface** of RBAC: the user CRUD API endpoints, the user management admin pages, the admin-initiated password reset, and the audit log helper plus all audit log writes for user management. After 1.3.3, a super-admin can manage users through a working UI and every action is audit-logged.

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 1.3.1 — Role & Permission Data Seeding

**Owns:** The complete RBAC data catalogue, the seed logic that populates it, and the default super-admin user. The `prisma/seed.ts` script's substantive content lives here (the script's plumbing was set up in 1.1.3).

**Does NOT own:** Any UI, any permission enforcement, any API endpoints, any audit log writes.

**Key outputs:**

- `src/lib/permissions.ts` (the constants module, but with only the permission strings defined — the guard functions are added in 1.3.2).
- `prisma/seed.ts` updated to include:
  - Upsert logic for all permissions (keyed on `Permission.name`).
  - Upsert logic for all roles (keyed on `Role.name`).
  - Upsert logic for all role-to-permission mappings (keyed on the `RolePermission` composite key of `roleId` + `permissionId`).
  - Upsert logic for the default super-admin user (keyed on `User.email`) with the SUPER_ADMIN role assigned.
  - A console warning printed when the default password is in use, instructing the operator to change it immediately in any non-local environment.
- A documented default super-admin credential (email and password) in the document body. The default password is bcrypt-hashed before storage.
- A documented list of system-locked roles (all five default roles have `isSystem = true`).
- An idempotency guarantee: the seed can be run repeatedly without producing duplicates.
- A short verification script (e.g., a temporary `scripts/verify-seed.ts` or a Prisma query in the verification checklist) that confirms the seed produced the expected counts.

**Master Plan sections implemented:** Section 10.4 (full RBAC specification — role catalogue, permission catalogue, default role-to-permission mappings).

### 4.2 Document 1.3.2 — RBAC Middleware & API Guard

**Owns:** The enforcement layer for permissions, both server-side (API routes, server actions, server components) and client-side (conditional UI rendering), plus the role-based extension of the middleware.

**Does NOT own:** The permission data itself (owned by 1.3.1), any user management UI (owned by 1.3.3), the audit log writes on user-management actions (owned by 1.3.3).

**Key outputs:**

- `src/lib/permissions.ts` extended with:
  - The permission constants (the strings themselves are defined in 1.3.1; this document re-exports them so consumers have a single import).
  - A typed `Permission` union type derived from the constants (so `withPermission()` can enforce its argument is a known permission).
- `src/lib/guards.ts` (or `src/lib/rbac.ts`) containing:
  - `withPermission(handler, required, options?)`: a higher-order function that wraps a route handler. `required` is a single permission string or `{ any: string[] }` or `{ all: string[] }`. The wrapper:
    - Reads the current session via `auth()` (from `@/lib/auth`).
    - Returns `{ success: false, error: { code: 'UNAUTHORIZED', message: '...' } }` with HTTP 401 if no session.
    - Returns `{ success: false, error: { code: 'FORBIDDEN', message: '...' } }` with HTTP 403 if the session lacks the required permission.
    - Otherwise calls the wrapped handler with the typed session injected.
  - `withRole(handler, requiredRole)`: a companion guard for role-name checks. Less commonly used; intended for system-locked role checks (e.g., only SUPER_ADMIN can manage KioskConfig in a future phase).
- `src/hooks/usePermission.ts` containing:
  - `usePermission()`: reads `useSession()` from `next-auth/react` and returns a function `can(permission: Permission): boolean`.
- `src/components/can.tsx` containing:
  - `<Can permission={...} fallback={...}>...</Can>`: a declarative component that renders its children only if `can(permission)` returns true. The `fallback` prop is optional and renders when the check fails.
- `src/middleware.ts` updated to:
  - Read the session's `roles` and `permissions` from the JWT cookie (no DB access — middleware runs on the Edge runtime).
  - Allow the request to proceed if the user has the required permission for the route prefix.
  - Redirect to a `/403` page (or to `/?error=forbidden`) if the user lacks the permission for a role-protected route prefix.
  - The route-prefix rules are documented in the document body (e.g., `/users` requires `user:manage`, `/settings` requires `system:configure`, `/audit-log` requires `system:audit`).
- A temporary verification route (e.g., `src/app/api/_dev/permission-check/route.ts` gated by `NODE_ENV !== 'production'`) that exercises the guard end-to-end.
- Standardised error responses everywhere a guard returns: master plan envelope (`{ success: false, error: { code, message, details? } }`) with the correct HTTP status (401 or 403).

**Master Plan sections implemented:** Sections 9.1 (response envelope), 9.2 (error codes — particularly the 401 vs 403 distinction), 10.1 (permission propagation in the JWT), and 15 (security enforcement surface).

### 4.3 Document 1.3.3 — User Management Admin Panel

**Owns:** The user-facing UI for managing users and roles, the supporting API endpoints, the admin-initiated password reset flow, and the audit log helper plus all audit log writes for user-management actions.

**Does NOT own:** The permission constants or guard functions (owned by 1.3.1/1.3.2), the login page (owned by 1.2.3), any business domain CRUD (Phase 2+).

**Key outputs:**

- API endpoints:
  - `GET /api/users`: list users with pagination and search by name or email. Guarded with `user:read`.
  - `POST /api/users`: create a new user with role assignment. Guarded with `user:create` (or `user:manage` — the document specifies which).
  - `GET /api/users/[userId]`: get a single user. Guarded with `user:read`.
  - `PATCH /api/users/[userId]`: update a user (name, email, status, role assignments). Guarded with `user:update` (or `user:manage`).
  - `DELETE /api/users/[userId]`: deactivate a user (soft delete via `UserStatus.INACTIVE`). Guarded with `user:delete` (or `user:manage`).
  - `POST /api/users/[userId]/reset-password`: admin-initiated password reset. Guarded with `user:manage`. Returns the new temporary password in the response once.
- `src/lib/audit-log.ts` containing:
  - `writeAuditLog(entry)`: a single function that writes an `AuditLog` row. Accepts action type, actor user ID, target user ID (if applicable), and arbitrary metadata. Best-effort: failures are logged to the console but do not fail the user-facing action.
- `src/app/(dashboard)/users/page.tsx`:
  - A data table of users with columns: name, email, status, roles, last updated, actions.
  - A search input that filters by name or email.
  - A "Create User" button that opens a dialog or navigates to the create page.
  - Loading (skeleton rows), error (toast or alert), and empty (empty-state component) states.
- `src/app/(dashboard)/users/new/page.tsx`:
  - Form fields: name, email, initial password, status, role multi-select.
  - Client-side validation using a Zod schema in `src/schemas/user.schema.ts`.
  - On submit, calls `POST /api/users`.
- `src/app/(dashboard)/users/[userId]/page.tsx`:
  - Edit form with the same fields plus a "Reset Password" button.
  - Role multi-select displays each role's display name and description. System-locked roles are visually marked but can still be assigned (they just cannot be deleted — deletion is not exposed in Phase 1 at all; status change to INACTIVE is the soft-delete).
  - On submit, calls `PATCH /api/users/[userId]`.
- `src/app/(dashboard)/users/[userId]/reset-password-form.tsx` (or a dialog within the edit page):
  - Form to trigger `POST /api/users/[userId]/reset-password`.
  - On success, displays the temporary password to the admin once with a clear "Copy and send securely to the user — this is the only time it will be shown" warning.
- Route protection:
  - All pages are guarded with `user:read` (for listing) and `user:manage` (for create/edit/deactivate/reset).
  - All API endpoints are guarded with the same permissions via `withPermission()`.
- Audit log writes:
  - `writeAuditLog()` is called by every mutating endpoint (create, update, deactivate, password reset).
  - The action types are a stable enum (e.g., `USER_CREATED`, `USER_UPDATED`, `USER_DEACTIVATED`, `PASSWORD_RESET_BY_ADMIN`).
  - Each entry includes the actor's user ID (from the session), the target user ID, a human-readable description, and a JSON metadata blob with the change details.
- UI components:
  - `user-table.tsx`: the data table component.
  - `user-form.tsx`: the shared create/edit form component.
  - `role-multi-select.tsx`: the role assignment widget.
- Loading and error states: the listing page handles loading (skeleton rows), error (toast or alert), and empty (empty-state component) states. The form pages handle validation errors inline.

**Master Plan sections implemented:** Sections 6.4 (admin layout), 6.5 (`UserForm` and `UserTable` components), 8.2 (`User`, `Role`, `UserRole` model usage, plus `AuditLog` writes), 9.3 (user API endpoints), 10 (RBAC in action — specifically the guards from 1.3.2 wrapping these endpoints), and the `AuditLog` model from Section 8.2 (writes only — the viewer UI is Phase 5).

## 6. Cross-Document Standards

These standards apply to every document in Sub-Phase 1.3. They are conventions established now so that Phase 2+ inherits a consistent RBAC layer.

### 6.1 Folder and File Naming

| Item | Convention | Example |
|---|---|---|
| Multi-word folders | kebab-case | `user-management/`, `audit-log/` |
| Component files | PascalCase | `UserForm.tsx`, `UserTable.tsx`, `Can.tsx` |
| Non-component modules | camelCase | `permissions.ts`, `guards.ts`, `audit-log.ts`, `use-permission.ts` |
| Route handlers | `route.ts` (Next.js App Router convention) | `src/app/api/users/route.ts` |
| Type files | kebab-case with `.types.ts` suffix when colocated | `src/types/user.types.ts` |

### 6.2 Import Conventions

- All project-internal imports use the `@/` alias.
- External package imports are listed first (alphabetically), then internal imports grouped by `@/components`, `@/lib`, `@/hooks`, `@/types`, `@/schemas`.
- The guard functions (`withPermission`, `withRole`) are imported from `@/lib/guards` (or `@/lib/rbac`). The permission constants are imported from `@/lib/permissions`. The audit log helper is imported from `@/lib/audit-log`. The `usePermission` hook is imported from `@/hooks/use-permission`. The `Can` component is imported from `@/components/can`.
- **No deep relative imports** (`../../../lib/...`) inside the `src/` tree — always use `@/`.

### 6.3 TypeScript Standards

- All exported functions and components have explicit type annotations.
- **No `any` types in production code.** Use `unknown` and narrow with type guards where the type cannot be statically determined.
- The `Permission` type is a TypeScript string-literal union derived from the permission constants. The `withPermission()` function accepts only this type (or an array of this type). This is enforced at compile time.
- The `Role` type is similarly derived from the role name constants.
- The `AuditLog` write function accepts a strongly-typed entry — no `any` in the metadata blob (use `Record<string, unknown>` if arbitrary keys are needed).
- The `Session` type (from NextAuth, augmented in 1.2.1) carries `userId`, `roles`, and `permissions` strongly. All guards and hooks consume the augmented session type.
- Discriminated unions are used for guard outcomes (e.g., `{ ok: true, session } | { ok: false, code: 'UNAUTHORIZED' } | { ok: false, code: 'FORBIDDEN' }`).

### 6.4 Response Envelope Standard

Every API route in this sub-phase must return the master plan's standard envelope (Section 9.1):

- Success: `{ success: true, data: <payload>, meta?: <pagination> }`
- Error: `{ success: false, error: { code, message, details? } }`

And must use the master plan's standard error codes (Section 9.2):

| Code | HTTP Status | When |
|---|---|---|
| `UNAUTHORIZED` | 401 | No valid authentication token (the guard finds no session) |
| `FORBIDDEN` | 401 vs 403 — see note below | Authenticated but lacks the required permission (or role) for the route |
| `VALIDATION_ERROR` | 422 | Form body fails Zod validation (details contain the Zod error tree) |
| `NOT_FOUND` | 404 | The requested user does not exist (used by `GET /api/users/[userId]`) |
| `CONFLICT` | 409 | The email is already in use when creating a user, or the user is already deactivated |
| `RATE_LIMITED` | 429 | Too many requests (rate limiting is Phase 5 — this code is reserved) |
| `INTERNAL_ERROR` | 500 | Unexpected server error |

**The 401 vs 403 distinction (critical):**

- `UNAUTHORIZED` (HTTP 401): the request has no valid session. The user is not authenticated.
- `FORBIDDEN` (HTTP 403): the request has a valid session, but the user lacks the required permission (or role) for the route.

The middleware (`middleware.ts`) returns 401 for missing sessions (redirecting to `/login`) and 403 for missing permissions on role-protected routes (redirecting to `/?error=forbidden` or to a dedicated `/403` page — the document specifies). API routes return the JSON envelope with the same distinction.

### 6.5 Validation Standards

- Every form on the client has a Zod schema in `src/schemas/`.
- The same Zod schema is reused on the server (where applicable) for server-side validation.
- Validation error responses use `VALIDATION_ERROR` (HTTP 422) with the Zod error details in `error.details`.
- Schemas for this sub-phase:
  - `src/schemas/user.schema.ts`: `createUserSchema` (name, email, password, status, roleIds), `updateUserSchema` (name, email, status, roleIds), `resetPasswordSchema` (newPassword), `listUsersQuerySchema` (page, limit, search).
  - Email format is validated against an RFC 5322 subset.
  - Password minimum length is 8 characters (a documented choice — a stronger policy is a future phase).

### 6.6 Error Handling Standards

- Server actions and API routes return the master plan's standard envelope.
- Client-side error display uses shadcn `Toast` for transient errors (toast notifications are added in this sub-phase if not already present) and inline alerts for form-field errors.
- Every API route is wrapped in a try/catch that logs the error (in development) and returns `INTERNAL_ERROR` (HTTP 500) for unhandled exceptions. The log includes the request path and a sanitized error message.
- The guard functions (`withPermission`, `withRole`) return their own structured error envelope — they never throw to the caller. The wrapped handler is responsible for its own try/catch.

### 6.7 Audit Logging Standard

- Every mutating action in user management (Sub-Phase 1.3) writes an `AuditLog` entry via `writeAuditLog()`.
- The `lib/audit-log.ts` module exports a single function and is reusable by every later phase.
- **Best-effort writes:** Audit log writes must not block the main response — they are best-effort. Any failure in writing a log is caught and logged to the console (`console.error`) but does not fail the user-facing action. The user-management action still succeeds; only the audit row is missing.
- Audit log entries contain:
  - `action`: a stable string enum (e.g., `'USER_CREATED'`, `'USER_UPDATED'`, `'USER_DEACTIVATED'`, `'PASSWORD_RESET_BY_ADMIN'`, `'ROLE_ASSIGNED'`, `'ROLE_REMOVED'`).
  - `actorId`: the user ID who performed the action (from the session).
  - `targetUserId`: the affected user ID (when applicable).
  - `description`: a human-readable string suitable for display in a future audit log viewer.
  - `metadata`: a JSON blob with change details (e.g., `{ before: {...}, after: {...} }` for updates).

### 6.8 Environment Variable Standards

- No new environment variables are introduced by this sub-phase beyond what 1.2.x already added (`NEXTAUTH_SECRET`, `NEXTAUTH_URL`, `REFRESH_TOKEN_EXPIRY_DAYS`).
- If a default super-admin email or password needs to be configurable, they should be added to `.env.example` with defaults — but the document body specifies that the defaults are sufficient for local development.

### 6.9 Commit Standards

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `style:`, `test:`.
- Each task plan document corresponds to one or more commits. A single commit should not span multiple task plan documents.
- Suggested commit boundaries:
  - End of 1.3.1: `feat: seed role catalogue, permission catalogue, and default super-admin user`
  - End of 1.3.2: `feat: add RBAC guards, client permission hook, and role-based middleware`
  - End of 1.3.3: `feat: add user management admin panel with audit logging`
- The initial migration file from 1.1.3 and the seed script from 1.3.1 may be committed together as a single `feat: initial database schema and RBAC seed` commit if not yet committed. Otherwise, 1.3.1's commit stands alone.

## 8. AI Agent Implementation Guidance

This section is for the AI agents that will implement the task plan documents. It is guidance that applies across all three task plan documents in this sub-phase.

### 8.1 How to Use the Task Plan Documents

Each task plan document follows the same internal structure:

1. **Scope** — what this document owns and what it does not.
2. **Pre-requisites** — what must already be true (previous documents completed).
3. **Step-by-step implementation plan** — the actual work, in order.
4. **File-level specifications** — for every file the agent will create or modify, a description of its purpose and contents (no code, but enough to know exactly what to write).
5. **Verification checklist** — how the agent proves the work is done.
6. **Commit instructions** — what commit(s) to make at the end.
7. **Common pitfalls** — mistakes to avoid.
8. **AI agent guidance** — document-specific guidance.

The agent should follow the steps in order. Each step lists the **goal** of the step, the **files affected**, and any **specific instructions** that resolve the ambiguity the agent would otherwise have to guess about.

### 8.2 General Guidance

- **Do not skip steps.** Sub-Phase 1.3 is mostly sequential because each step's output is the next step's input. Skipping forces backtracking.
- **Do not add features beyond the document's scope.** The temptation to "just add the audit log viewer now" or "just add the user deactivation confirmation dialog with email notification" is real — resist it. Those are owned by later phases.
- **Verify after every step.** The verification checklist at the end of each document is a hard gate.
- **When the master plan is ambiguous, choose the simpler option.** Sub-Phase 1.3 is about closing the foundation, not optimising. If a configuration choice has two reasonable interpretations, pick the one that is easier to change later.
- **Commit at the end of each document** using the commit message specified in the document.
- **Document any deviation.** If the agent makes a choice that differs from this document, it must note the deviation in the final summary so the human reviewer can approve or reject it.
- **Always use the master plan's response envelope.** Every API route created in this sub-phase must return `{ success, data | error }` — not raw objects, not custom shapes.
- **Use the permission constants, not string literals.** Every permission check should reference a constant from `lib/permissions.ts`. This is what enables IDE refactoring and prevents typos.
- **Distinguish 401 from 403 carefully.** A missing session is 401. A session without the required permission is 403. Both use the standard envelope but different `code` values and different HTTP statuses.

### 8.3 What the AI Agent Should NOT Do

- Should not implement any business domain logic (queue, counter, ticket, display, notification). All of that is Phase 2+.
- Should not seed roles, permissions, or users with anything beyond what Master Plan Section 10.4 specifies. Do not invent additional roles or permissions; if a need arises, document it for future consideration but do not add it to the seed.
- Should not change the `User`, `Role`, `Permission`, `UserRole`, `RolePermission`, or `AuditLog` schema. These were finalised in 1.1.3 and must not be modified by Phase 1.
- Should not implement an audit log viewer UI. Only `writeAuditLog()` is in scope; the viewer is Phase 5.
- Should not implement rate limiting, security headers, or production hardening. That is Phase 5.
- Should not add PostgreSQL configuration. The database is SQLite for development.
- Should not implement OAuth / social login. The master plan provisions the `Account` table for future OAuth providers, but this sub-phase uses only the credentials provider.
- Should not implement self-service password reset. Only the admin-initiated password reset (from 1.3.3) is in scope.
- Should not implement email-based user invitation. The admin creates users directly with a known temporary password.
- Should not implement OAuth / social account linking. The `Account` table exists in the schema but is unused.
- Should not add a 2FA flow. That is a future feature.

### 8.4 When to Ask the Human

The AI agent should ask the human reviewer (not guess) if any of the following situations arise:

- The master plan and the phase 1 overview contradict each other. The master plan wins, but the agent should confirm.
- A required permission string is missing from Master Plan Section 10.4 (e.g., a Phase 2 dependency requires a permission that 10.4 does not list). The agent should request clarification rather than invent one.
- The default super-admin email or password is rejected by the env validator or by some other constraint (the agent should adjust the env validator, not weaken the credential).
- The verification step fails for a reason that the document does not anticipate.
- The agent is unsure whether to commit the migration file from 1.1.3 together with the seed from 1.3.1 (the document says this is acceptable — confirm with the human if there is any doubt).
