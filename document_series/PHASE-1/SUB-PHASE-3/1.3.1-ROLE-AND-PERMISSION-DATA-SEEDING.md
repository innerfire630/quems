## 1. Document Purpose

This document populates the database with the **complete RBAC catalogue** that every later layer of the application will read from. It delivers:

- The full **permission catalogue** as a typed constants module (`src/lib/permissions.ts`) and as database rows in the `Permission` table.
- The full **role catalogue** (five system-locked roles) as database rows in the `Role` table.
- The **role-to-permission mappings** as database rows in the `RolePermission` table, matching Master Plan Section 10.4 exactly.
- A **default super-admin user** seeded with a known email and a documented default password (bcrypt-hashed), with the `SUPER_ADMIN` role assigned.
- An **idempotent seed** that can be run multiple times without producing duplicate rows.

When this document is complete, `yarn prisma:seed` produces a database in which every later guard, hook, and route handler can immediately check permissions and roles without conditional "what if the seed hasn't run" logic.

## 2. Scope and Boundaries

### 2.1 In Scope

- Creating `src/lib/permissions.ts` containing:
  - Permission string constants grouped by module (`USER`, `COUNTER`, `SERVICE`, `TICKET`, `NOTIFICATION`, `REPORT`, `SYSTEM`).

  - A `ROLE` constants object with all five system role names.

- Documenting the system-locked role list (all five default roles have `isSystem = true`).

### Step 3: Update `prisma/seed.ts` â€” Upsert All Permissions

**Goal:** Populate the `Permission` table with every permission string from Master Plan Section 10.4.

**Files affected:** `prisma/seed.ts` (modified).

**Specific instructions:**

- At the top of `prisma/seed.ts`, import the `ALL_PERMISSIONS` array and `PERMISSION_DESCRIPTIONS` from `@/lib/permissions`.
- Add a function `seedPermissions()` that:
  - Iterates over `ALL_PERMISSIONS`.
  - For each permission, performs a Prisma `upsert` on the `Permission` table keyed on `name`:

  - Returns a count of permissions upserted.
- Call `seedPermissions()` as the first step of the seed.
- Use the `db` client from `@/lib/db` (the Prisma singleton).

**Goal:** Populate the `Role` table with all five system-locked roles.

**Files affected:** `prisma/seed.ts` (modified).

**Specific instructions:**

- Import `ALL_ROLES` from `@/lib/permissions`.
- Add a function `seedRoles()` that:
  - Iterates over `ALL_ROLES`.
  - For each role, performs a Prisma `upsert` on the `Role` table keyed on `name`:

### Step 5: Update `prisma/seed.ts` â€” Upsert Role-to-Permission Mappings

**Goal:** Populate the `RolePermission` table with the mappings from Master Plan Section 10.4.

**Files affected:** `prisma/seed.ts` (modified).

**Specific instructions:**

- Import `ROLE_PERMISSIONS` and `ALL_PERMISSIONS` from `@/lib/permissions`.
- Add a function `seedRolePermissions()` that:
  - For each role name in `ROLE_PERMISSIONS`:
    - Look up the role in the database by `name` (use `db.role.findUnique({ where: { name } })`).
    - For each permission string in the role's permission list:
      - Look up the permission in the database by `name`.
      - Perform a Prisma `upsert` on the `RolePermission` table keyed on the composite `{ roleId, permissionId }`:
        - If the mapping does not exist, create it.
        - If the mapping exists, do nothing (mappings are immutable â€” re-running the seed must not error).
    - For each permission in `ALL_PERMISSIONS` that is **not** in the role's list:
      - If a `RolePermission` mapping exists for this `{ roleId, permissionId }`, **delete it** (so re-running the seed after a permission removal cleans up stale mappings).
  - Returns a count of mappings created/deleted.
- Call `seedRolePermissions()` after `seedRoles()`.

**Important:** The deletion step is what guarantees idempotency when the source-of-truth mapping changes. If a role's permission list is edited in `permissions.ts`, the next seed run removes the stale mappings.

## 8. Common Pitfalls

The following mistakes are common when seeding RBAC data and are worth avoiding.

### 8.1 Hardcoding Permission Strings Outside `permissions.ts`

Every reference to a permission string must come from `@/lib/permissions`. No raw strings like `'user:manage'` scattered across the codebase. This is what enables IDE refactoring when permissions are renamed.

### 8.2 Forgetting the `module` Field on Permissions

Each `Permission` row in the database has a `module` field (e.g., `user`, `counter`, `system`). The seed must populate this from the prefix of the permission string (`user:manage` â†’ module `user`). Forgetting this breaks any UI that groups permissions by module.

### 8.3 Not Cleaning Up Stale `RolePermission` Rows

If a role's permission list is edited in `ROLE_PERMISSIONS`, the next seed run must **delete** stale mappings. Without this step, the database would still grant the removed permission to the role â€” a security bug.

### 8.4 Using `create` Instead of `upsert`

If the seed uses `create` instead of `upsert`, running the seed a second time will fail with a unique-constraint violation. Always use `upsert` keyed on the unique field (`name` for `Role` and `Permission`; composite `{ roleId, permissionId }` for `RolePermission`).

### 8.5 Storing the Default Password in Plaintext

The password must be bcrypt-hashed before being stored. A plaintext password in the database would allow anyone with database access to log in as the super-admin without knowing the actual password.

### 8.6 Forgetting the Console Warning

The console warning about the default password is the only safeguard against an operator deploying the default credential to a non-local environment. Omitting it is a security oversight.

### 8.7 Using the Wrong `UserRole` Composite Key

`UserRole` has a composite primary key of `{ userId, roleId }`, not a single auto-incrementing id. The seed's upsert must use `where: { userId_roleId: { userId, roleId } }` (Prisma's convention for composite keys) or use `findFirst` + `create`/`update`. Get this wrong and the seed will fail with a unique-constraint error.

### 8.8 Modifying the Schema in This Document

The `Role`, `Permission`, `RolePermission`, `UserRole`, `AuditLog` schema is finalised in 1.1.3. If a field appears to be missing (e.g., `RolePermission` lacks a `grantedAt` timestamp), do not add it here â€” note it for a future phase. Adding schema changes in 1.3.1 creates migration drift.

### 8.9 Hardcoding the Bcrypt Cost Factor Inconsistently

The cost factor used here (for the default super-admin's password) must match the cost factor used in 1.2.1 for `verifyCredentials`. If they differ, `bcrypt.compare` will still work but the seed will hash at a different cost than login verifies. Use the same value (10 or 12) throughout.

### 8.10 Inconsistent Role Name Casing

Role names are exported as uppercase strings (`SUPER_ADMIN`, not `super_admin` or `SuperAdmin`). The seed must use the same casing. Mismatched casing causes the `Role.findUnique({ where: { name: 'SUPER_ADMIN' } })` lookup to miss.
