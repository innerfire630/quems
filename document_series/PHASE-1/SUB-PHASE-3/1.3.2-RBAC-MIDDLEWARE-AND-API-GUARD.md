## 1. Document Purpose

This document builds the **enforcement layer** for the RBAC system. It ensures that every layer of the application ‚Äî API routes, server actions, server components, client components, and the Next.js middleware ‚Äî can ask "does the current user have permission X?" with a single, strongly-typed function call or component.

It delivers:

- The `withPermission()` server-side guard function (and its `withRole()` companion) that wraps API route handlers and server actions.
- The `usePermission()` client-side hook for conditional UI rendering.
- The `<Can>` component for declarative permission-gated JSX.
- The extension of `middleware.ts` (from 1.2.3) to enforce role-based route prefixes at the edge (before the request hits any route handler).
- The consistent application of the 401 vs 403 distinction across every protected surface.

When this document is complete, a counter officer who somehow reaches a `/users` page is redirected, who calls a `/api/users` endpoint receives `FORBIDDEN` (HTTP 403) with the standard envelope, and who views the navigation does not see the "Users" link. A super-admin sees and can do everything.

## 2. Scope and Boundaries

### 2.1 In Scope

- Creating `src/lib/guards.ts` (or `src/lib/rbac.ts`) containing:
  - `withPermission(handler, required, options?)`: higher-order function for API routes and server actions. `required` is a single `Permission` string or an object `{ any: Permission[] }` / `{ all: Permission[] }`. Returns a wrapped handler that:
    - Reads the session via `auth()` from `@/lib/auth`.
    - Returns `UNAUTHORIZED` (HTTP 401) if no session.
    - Returns `FORBIDDEN` (HTTP 403) if the session lacks the required permission.
    - Otherwise invokes the wrapped handler with the typed session injected.
  - `withRole(handler, requiredRole)`: companion guard for role-name checks. Accepts a single `Role` string.
  - Helper type `GuardedHandler<P>` for the wrapped handler signature.
- Creating `src/hooks/usePermission.ts` containing:
  - `usePermission()`: a hook that reads `useSession()` from `next-auth/react` and returns a function `can(permission: Permission): boolean`. While the session is loading, `can` returns `false`.
  - Optional: a `useRole()` companion that returns a function for role-name checks.
- Creating `src/components/can.tsx` containing:
  - `<Can permission={...} fallback={...}>...</Can>`: declarative component. While the session is loading, renders nothing (or the fallback, if provided).
  - `<CanRole role={...} fallback={...}>...</CanRole>`: companion for role checks.
- Extending `src/middleware.ts` (created in 1.2.3) to:
  - Read the session's `roles` and `permissions` from the JWT cookie (no DB access ‚Äî middleware runs on Edge).
  - Apply role-based route prefix rules (e.g., `/users` requires `user:manage`; `/settings` requires `system:configure`; `/audit-log` requires `system:audit`).
  - Return a redirect to `/login?callbackUrl=...` for missing sessions.
  - Return a redirect to `/?error=forbidden` (or a `/403` page) for missing permissions.
  - Keep the existing authentication-only checks from 1.2.3 intact.
- A temporary verification route (e.g., `src/app/api/_dev/permission-check/route.ts`) gated by `NODE_ENV !== 'production'` that exercises the guard end-to-end.
- Standardised error responses everywhere a guard returns: master plan envelope with the correct HTTP status (401 or 403).

### 2.2 Out of Scope (Deferred to Other Documents)

- The seed data (roles, permissions, default super-admin user). Owned by 1.3.1.
- The user management UI and user CRUD API. Owned by 1.3.3.
- The audit log helper (`lib/audit-log.ts`) and audit log writes. Owned by 1.3.3.
- The login page. Owned by 1.2.3 (the middleware assumes it exists).
- Any business domain logic (services, counters, tickets, displays, notifications). All of that is Phase 2+.
- Any role or permission beyond what Master Plan Section 10.4 specifies.

### 2.3 What This Document Does NOT Touch

- It does **not** seed any data.
- It does **not** implement any UI beyond the `<Can>` component itself (no pages, no forms).
- It does **not** implement any API routes beyond the temporary verification route.
- It does **not** modify the Prisma schema.

### Step 1: Create `src/lib/guards.ts` ‚Äî The Guard Functions

**Goal:** Provide the server-side enforcement primitives (`withPermission`, `withRole`) that every API route and server action in Phase 2+ will wrap.

**Files affected:** `src/lib/guards.ts` (new file).

**Specific instructions:**

Create `src/lib/guards.ts` containing (no code in this document ‚Äî only descriptions):

- **Imports:** `NextResponse` from `next/server`; `auth` from `@/lib/auth`; `Permission`, `Role` from `@/lib/permissions`; the augmented `Session` type from `next-auth`.
- **`type PermissionRequirement = Permission | { any: Permission[] } | { all: Permission[] }`** ‚Äî a discriminated union for what `withPermission` accepts.
- **`type GuardedContext = { session: Session }`** ‚Äî the object injected into the wrapped handler, carrying the typed session.
- **`type GuardedHandler<P extends PermissionRequirement> = (req: Request, ctx: GuardedContext) => Promise<Response>`** ‚Äî the wrapped handler's signature.
- **`withPermission<P extends PermissionRequirement>(handler: GuardedHandler<P>, required: P, options?: { onUnauthorized?: () => Response; onForbidden?: () => Response })`** ‚Äî the higher-order function:
  - Returns an `async` function that takes the original `Request` (and any other arguments Next.js passes ‚Äî the agent must use the appropriate Next.js route handler signature).
  - Calls `auth()` to read the session.
  - **If no session:** return a `NextResponse.json({ success: false, error: { code: 'UNAUTHORIZED', message: 'Authentication required.' } }, { status: 401 })`. The `onUnauthorized` option allows the caller to override this (rarely used).
  - **If session but lacks required permission:**
    - If `required` is a single `Permission`: check `session.user.permissions.includes(required)`.
    - If `required` is `{ any: [...] }`: check that at least one permission in the array is in the session's permissions.
    - If `required` is `{ all: [...] }`: check that every permission in the array is in the session's permissions.
    - If the check fails, return `NextResponse.json({ success: false, error: { code: 'FORBIDDEN', message: 'You do not have permission to access this resource.' } }, { status: 403 })`.
  - **If session and permission passes:** call `handler(req, { session })` and return its response.
- **`withRole(handler: GuardedHandler<Role>, requiredRole: Role, options?)`** ‚Äî companion for role checks. Same shape, but checks `session.user.roles.includes(requiredRole)`. Less commonly used; intended for system-locked role checks.
- **Type exports:** Export `GuardedHandler`, `GuardContext`, `PermissionRequirement` so consumers can type their wrapped handlers.
- The file must not have a `"use client"` directive ‚Äî guards are server-only.
- The functions must be **synchronous in construction** (the wrapper function is what is async) ‚Äî this allows `withPermission(...)` to be used as a higher-order function in module-level route handler definitions.

### Step 2: Create `src/hooks/usePermission.ts` ‚Äî The Client Hook

**Goal:** Provide a client-side hook so React components can conditionally render UI based on the current session's permissions.

**Files affected:** `src/hooks/usePermission.ts` (new file).

**Specific instructions:**

Create `src/hooks/usePermission.ts` containing:

- The file **must** have a `"use client"` directive at the top.
- **Imports:** `useSession` from `next-auth/react`; `Permission` from `@/lib/permissions`.
- **`usePermission()` hook:**
  - Calls `useSession()` from NextAuth.
  - Returns an object with two members:
    - `can(permission: Permission): boolean` ‚Äî returns `true` if the current session is loaded and contains the permission, `false` otherwise (including while loading).
    - `canAny(permissions: Permission[]): boolean` ‚Äî returns `true` if the session contains at least one of the permissions.
    - `canAll(permissions: Permission[]): boolean` ‚Äî returns `true` if the session contains all of the permissions.
  - Also exports `isLoading: boolean` (true while the session is loading) and `session: Session | null` for callers that need direct access.
- **`useRole()` hook (companion):**
  - Returns an object with:
    - `hasRole(role: Role): boolean`
    - `isLoading: boolean`
    - `session: Session | null`
- **Important:** When the session is `undefined` (still loading), `can` returns `false`. This prevents a flash of unauthorised content. Components should handle the loading state with a skeleton or fallback if needed.

### Step 3: Create `src/components/can.tsx` ‚Äî The Declarative Component

**Goal:** Provide a JSX component for declarative permission-gated rendering, complementing the hook.

**Files affected:** `src/components/can.tsx` (new file).

**Specific instructions:**

Create `src/components/can.tsx` containing:

- The file **must** have a `"use client"` directive at the top.
- **Imports:** `usePermission`, `useRole` from `@/hooks/usePermission`; `Permission`, `Role` from `@/lib/permissions`; React types.
- **`<Can>` component:**
  - Props: `{ permission?: Permission | Permission[]; any?: Permission[]; all?: Permission[]; fallback?: React.ReactNode; children: React.ReactNode }`.
  - Behaviour: if `permission` is provided (single string or array ‚Äî array means "any of"), call `can(permission)`. If `any` is provided, call `canAny(any)`. If `all` is provided, call `canAll(all)`. If the check returns true, render `children`; otherwise render `fallback` (default: `null`).
  - While the session is loading, render `fallback` (or `null`).
- **`<CanRole>` component:**
  - Props: `{ role: Role | Role[]; fallback?: React.ReactNode; children: React.ReactNode }`.
  - Array means "any of".
  - Same loading behaviour as `<Can>`.
- Export both components (named exports).

### Step 4: Extend `src/middleware.ts` for Role-Based Route Prefixes

**Goal:** Enforce role/permission-based route prefixes at the edge, so unauthorised users are redirected before the request hits any server component or API handler.

**Files affected:** `src/middleware.ts` (modified).

**Specific instructions:**

- Open `src/middleware.ts` (created in 1.2.3 with authentication-only checks).
- **Read the JWT cookie** to extract the session's `roles` and `permissions`. The middleware cannot use Prisma (Edge runtime constraint) and cannot use `auth()` (which requires Node APIs). Instead:
  - Use NextAuth's `getToken()` helper from `next-auth/jwt` to decode the JWT cookie. This is the standard NextAuth v5 pattern for middleware.
  - The decoded token will have `roles` and `permissions` arrays (assuming 1.2.1's JWT enrichment callback populates them).
- **Define the route prefix ‚Üí permission mapping** as a top-level constant:
  - `/users` ‚Üí `PERMISSION_USER_MANAGE` (or `PERMISSION_USER_READ` for the listing page ‚Äî see note below).
  - `/settings` ‚Üí `PERMISSION_SYSTEM_CONFIGURE`.
  - `/audit-log` ‚Üí `PERMISSION_SYSTEM_AUDIT`.
  - (Additional prefixes may be added by later phases ‚Äî the data structure must be a record so new entries can be appended.)
- **Logic for each request:**
  - If the request URL matches a protected route prefix:
    - If no JWT cookie is present ‚Üí redirect to `/login?callbackUrl=<original path>` (authentication failure).
    - If JWT cookie is present but the user's permissions do not include the required permission ‚Üí redirect to `/?error=forbidden` (authorisation failure). The `/` page can render a "You do not have permission" message based on the query parameter.
    - Otherwise, allow the request to proceed.
  - If the request URL does not match any protected prefix ‚Üí use the existing logic from 1.2.3 (redirect authenticated users away from `/login` if applicable; otherwise allow).
- **The matcher regex** (from 1.2.3) may need to be updated to include the new role-protected prefixes. The matcher continues to exclude `/api/auth/*`, `/api/sse/*`, `/display/*`, `/kiosk/*`, `/security/*`, `/api/tickets/issue`, `/api/health`, `/login`, and static assets.
- **Edge runtime constraint:** The middleware file must not import anything that requires Node.js APIs (no Prisma, no `bcrypt`, no Node `crypto`). Only `next-auth/jwt`'s `getToken` is allowed for token decoding.
- **Important:** The middleware's role-based check is a **first line of defence**. The `withPermission()` guard in API routes and server actions is the authoritative check ‚Äî the middleware can be bypassed (e.g., by direct API calls from a mobile app), so the API guards must always run.

## 8. Common Pitfalls

The following mistakes are common when building the RBAC enforcement layer and are worth avoiding.

### 8.1 Confusing 401 and 403

- `UNAUTHORIZED` (HTTP 401) = no session, or session is invalid/expired.
- `FORBIDDEN` (HTTP 403) = session exists but lacks the required permission (or role).

This distinction is critical. A 401 implies "log in to continue"; a 403 implies "you are logged in but cannot do this". Conflating them breaks client-side error handling.

### 8.2 Using the Guard from a Client Component

`withPermission` and `withRole` are server-only. Importing them into a client component will fail the build (or worse, leak server-only code to the client). Use `usePermission` / `<Can>` on the client; reserve `withPermission` for API routes, server actions, and server components.

### 8.3 Re-fetching Roles/Permissions in `withPermission`

The session already contains `roles` and `permissions` (populated by the JWT enrichment callback in 1.2.1). `withPermission` reads from the session ‚Äî it does **not** query the database. If the agent adds a database query inside `withPermission`, every protected API call becomes a database call.

### 8.4 Putting Edge-Incompatible Code in `middleware.ts`

The middleware runs on the Edge runtime. It cannot use Prisma, Node `crypto`, or other Node-only APIs. The agent must use `getToken()` from `next-auth/jwt` (which is Edge-compatible) and must not import the Prisma client, bcrypt, or any Node-only package. A build error or runtime error at the Edge will block deployment.

### 8.5 Checking Permissions in Only One Layer

The middleware redirect is a UX optimization (catches unauthorised users early). The API guard (`withPermission`) is the security boundary. Both must run. A counter officer who bypasses the middleware (e.g., by hitting the API directly from a script) must still receive 403 from the API.

### 8.6 Using `<Can>` Without Handling the Loading State

While the session is loading, `<Can>` renders nothing (or the fallback). Components that need to show a skeleton during loading must handle it explicitly. A common bug is to assume `<Can>` always reflects the current state ‚Äî it does not, while loading.

### 8.7 Storing Roles or Permissions as Strings Outside `permissions.ts`

Every reference to a role or permission must come from the constants in `@/lib/permissions`. A guard call like `withPermission(handler, 'user:manage')` (string literal) instead of `withPermission(handler, PERMISSION_USER_MANAGE)` (constant) defeats the type safety the `Permission` union provides.

### 8.8 Returning the Password Field from Session

The session must never carry the user's password. The 1.2.1 augmentation and `verifyCredentials` are responsible for omitting it. The agent should verify that the session type's `user` interface does not include `password`.

### 8.9 Redirecting Authorisation Failures to `/login`

A user who lacks the required permission is **not** unauthenticated ‚Äî they are authenticated but unauthorised. Redirecting them to `/login` is wrong (they are already logged in). They should be redirected to `/?error=forbidden` or to a dedicated `/403` page.

### 8.10 Hardcoding the API Permission Map in `middleware.ts`

The route prefix ‚Üí permission map should live in `ROUTE_PERMISSION_MAP` (in `lib/guards.ts` or a related module), not hardcoded inside the middleware function. This makes it easy to add new prefixes later without modifying middleware logic.

## 9. AI Agent Guidance

### 9.1 How to Approach This Document

The agent should treat this document as an **enforcement layer task**. Most of the work is in three files: `src/lib/guards.ts`, `src/hooks/usePermission.ts`, `src/components/can.tsx`. The middleware extension is a smaller change to an existing file.

### 9.3 Verification Tip

The fastest way to verify the entire enforcement layer end-to-end is:

1. Start the dev server.
2. Log in as the super-admin via the browser. Verify the session has all permissions.
3. Use `prisma studio` or a one-off script to create a second user with the `COUNTER_OFFICER` role.
4. Log in as the counter officer. Verify the session has only the counter officer's permissions.
5. As the counter officer, attempt to visit `/users`. Verify the redirect.
6. As the counter officer, attempt to call `/api/_dev/permission-check`. Verify the 403 response.
7. As the super-admin, repeat steps 5 and 6. Verify success.

If any of these fail, the failure mode will point to which layer is broken (middleware vs guard vs session enrichment).
