## 1. Document Purpose

This document builds the **operating surface** of the RBAC system. It provides the user-facing UI for managing users and roles, the supporting API endpoints, the admin-initiated password reset flow, and the audit log helper plus all audit log writes for user-management actions.

It delivers:

- A complete set of user CRUD API endpoints at `/api/users` and `/api/users/[userId]`, all guarded with `withPermission()`.
- A listing page at `/(dashboard)/users` with a data table, pagination, and search.
- Create and edit form pages at `/(dashboard)/users/new` and `/(dashboard)/users/[userId]`.
- An admin-initiated password reset endpoint and UI affordance.
- The `lib/audit-log.ts` helper that writes `AuditLog` rows for every mutating action.
- Strongly-typed Zod schemas for all user-management forms.

When this document is complete, a super-admin can open `/users`, see all users, create a new user with assigned roles, edit an existing user (change roles or status), deactivate a user, and reset a user's password. Every action is recorded in the `AuditLog` table. A non-admin cannot reach the pages or APIs.

## 2. Scope and Boundaries

### 2.1 In Scope

- Creating `src/lib/audit-log.ts` containing:
  - `writeAuditLog(entry)`: the single entry point for writing audit log rows. Accepts action, actor ID, target user ID (optional), description, and metadata. Best-effort (failures are logged but do not fail the calling action).
- Creating API endpoints:
  - `GET /api/users`: list users with pagination and search. Guarded with `withPermission(PERMISSION_USER_READ)`.
  - `POST /api/users`: create a new user with role assignment. Guarded with `withPermission(PERMISSION_USER_CREATE)` (or `PERMISSION_USER_MANAGE` if a single `manage` permission is preferred).
  - `GET /api/users/[userId]`: get a single user. Guarded with `withPermission(PERMISSION_USER_READ)`.
  - `PATCH /api/users/[userId]`: update a user. Guarded with `withPermission(PERMISSION_USER_UPDATE)` (or `PERMISSION_USER_MANAGE`).
  - `DELETE /api/users/[userId]`: deactivate a user (soft delete via `status: INACTIVE`). Guarded with `withPermission(PERMISSION_USER_DELETE)` (or `PERMISSION_USER_MANAGE`).
  - `POST /api/users/[userId]/reset-password`: admin-initiated password reset. Guarded with `withPermission(PERMISSION_USER_MANAGE)`.
- Creating UI pages:
  - `/(dashboard)/users/page.tsx`: listing page.
  - `/(dashboard)/users/new/page.tsx`: create form.
  - `/(dashboard)/users/[userId]/page.tsx`: edit form.
- Creating UI components:
  - `src/app/(dashboard)/users/_components/user-table.tsx`: data table.
  - `src/app/(dashboard)/users/_components/user-form.tsx`: shared create/edit form.
  - `src/app/(dashboard)/users/_components/role-multi-select.tsx`: role assignment widget.
  - `src/app/(dashboard)/users/_components/reset-password-dialog.tsx`: admin password reset UI.
- Creating Zod schemas:
  - `src/schemas/user.schema.ts`: `createUserSchema`, `updateUserSchema`, `resetPasswordSchema`, `listUsersQuerySchema`.
- Creating types:
  - `src/types/user.types.ts`: shared TypeScript types for the API request/response shapes.
- Audit log writes:
  - Every mutating endpoint calls `writeAuditLog()` with the appropriate action type and metadata.
- Route protection:
  - All pages are guarded with `user:read` (listing) and `user:manage` (create/edit/deactivate/reset).
  - All API endpoints are guarded via `withPermission()`.
- Loading, error, and empty states for all pages and components.

### 2.2 Out of Scope (Deferred to Other Documents)

- The permission constants and guard functions. Owned by 1.3.1 (constants) and 1.3.2 (guards).
- The login page. Owned by 1.2.3.
- The dashboard layout shell. Owned by 1.2.3 (extended by 1.3.2).
- The audit log viewer UI. The `AuditLog` rows written here are queryable via Prisma for verification, but the UI to browse them is Phase 5.
- Email-based user invitation. The admin creates users directly with a known temporary password; invitation emails are a future feature.
- Self-service password reset. Only admin-initiated reset is in scope.
- Bulk operations (e.g., bulk deactivation, bulk role assignment). Out of scope.
- User profile page (where a user edits their own profile). Out of scope.
- Any business domain logic (services, counters, tickets, displays, notifications). All of that is Phase 2+.

### 2.3 What This Document Does NOT Touch

- It does **not** implement the audit log viewer UI.
- It does **not** send emails (no email service integration).
- It does **not** implement any service, counter, ticket, display, or notification logic.
- It does **not** modify the Prisma schema.
- It does **not** implement rate limiting (Phase 5).

### Step 1: Create `src/lib/audit-log.ts` — The Audit Log Helper

**Goal:** Provide a single, best-effort function for writing `AuditLog` rows that every mutating endpoint in this document and every later phase will use.

**Files affected:** `src/lib/audit-log.ts` (new file).

**Specific instructions:**

Create `src/lib/audit-log.ts` containing (no code in this document — only descriptions):

- **Imports:** the Prisma client (`db`) from `@/lib/db`.
- **Exported types:**
  - `type AuditAction = <union of action string literals>` — a string-literal union listing every action type the system will write. For Phase 1, this includes:
    - `'USER_CREATED'`, `'USER_UPDATED'`, `'USER_DEACTIVATED'`, `'USER_REACTIVATED'`, `'PASSWORD_RESET_BY_ADMIN'`, `'ROLE_ASSIGNED'`, `'ROLE_REMOVED'`.
    - (Phase 2+ adds actions like `'SERVICE_CREATED'`, `'COUNTER_OPENED'`, etc. The type should be designed to be extensible — either via a single union or via separate per-phase files.)
  - `type AuditLogEntry = { action: AuditAction; actorId: string; targetUserId?: string; description: string; metadata?: Record<string, unknown> }`.
- **Exported function `writeAuditLog(entry: AuditLogEntry): Promise<void>`:**
  - Wraps the `db.auditLog.create` call in a try/catch.
  - On success, resolves silently.
  - On failure (e.g., database write error), logs the error to `console.error` with the action name and entry details — **but does not throw**. The caller (the mutating endpoint) sees a successful response even if the audit log write failed. This is the "best-effort" guarantee.
- The file must not have a `"use client"` directive — `writeAuditLog` is server-only.

### Step 2: Create `src/schemas/user.schema.ts` — The Zod Schemas

**Goal:** Validate every user-management form body and query parameter on both client and server.

**Files affected:** `src/schemas/user.schema.ts` (new file).

**Specific instructions:**

Create `src/schemas/user.schema.ts` containing:

- **`createUserSchema`**:
  - `name`: required, non-empty, max length 100.
  - `email`: required, valid email format (RFC 5322 subset), max length 255.
  - `password`: required, min length 8, max length 128. (The complexity rule — e.g., must contain digit and symbol — is documented as a future phase enhancement; for Phase 1, length is the only rule.)
  - `status`: optional, must be one of `ACTIVE`, `INACTIVE`, `SUSPENDED` (use the `UserStatus` enum from Prisma).
  - `roleIds`: array of strings (role IDs). May be empty (a user with no roles is technically valid but cannot do anything). Each ID must be a valid CUID (Prisma's default).
- **`updateUserSchema`**:
  - `name`: optional, non-empty if provided.
  - `email`: optional, valid email format if provided.
  - `status`: optional, must be a valid `UserStatus`.
  - `roleIds`: optional array of role IDs.
  - All fields optional (PATCH is partial). At least one field must be present (use `.refine` or similar to enforce this).
- **`resetPasswordSchema`**:
  - `newPassword`: required, min length 8, max length 128.
- **`listUsersQuerySchema`**:
  - `page`: optional integer, default 1, min 1.
  - `limit`: optional integer, default 20, min 1, max 100.
  - `search`: optional string, max length 100.
- Export the inferred TypeScript types: `CreateUserInput`, `UpdateUserInput`, `ResetPasswordInput`, `ListUsersQuery`.

### Step 3: Create `src/types/user.types.ts` — The API Types

**Goal:** Define the response shapes for the user API endpoints so client and server share a single type contract.

**Files affected:** `src/types/user.types.ts` (new file).

**Specific instructions:**

Create `src/types/user.types.ts` containing:

- **`UserListItem`**: the shape returned by `GET /api/users`. Includes `id`, `name`, `email`, `status`, `roles: { id, name, description }[]`, `createdAt`, `updatedAt`. **Excludes `password`.**
- **`UserDetail`**: the shape returned by `GET /api/users/[userId]`. Includes everything in `UserListItem` plus a `permissions: string[]` field (computed from the user's roles).
- **`CreateUserResponse`**: the shape returned by `POST /api/users`. Returns the created `UserListItem`.
- **`ResetPasswordResponse`**: the shape returned by `POST /api/users/[userId]/reset-password`. Includes `temporaryPassword: string` (the plaintext temporary password, shown to the admin once) and `userId: string`.
- **`UserListMeta`**: pagination metadata: `{ page: number; limit: number; total: number; totalPages: number }`.
- All types wrap in the master plan's standard envelope at the route level (the route returns `{ success: true, data: UserListItem, meta: UserListMeta }`).

### Step 4: Create `src/app/api/users/route.ts` — List and Create Endpoints

**Goal:** Provide the listing endpoint with pagination and search, and the create endpoint.

**Files affected:** `src/app/api/users/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`GET` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_USER_READ)`.
  - Parses `page`, `limit`, `search` from query parameters using `listUsersQuerySchema`.
  - Builds a Prisma `where` clause: if `search` is present, filters by `name` OR `email` containing the search string (case-insensitive — SQLite uses `LIKE` with `mode: 'insensitive'` only on Postgres, so use `contains` without mode for SQLite, or use `mode: 'insensitive'` if the master plan specifies Postgres for development).
  - Executes a Prisma `findMany` with `include: { userRoles: { include: { role: true } } }`.
  - Counts the total with the same `where` clause for pagination.
  - Returns `{ success: true, data: <mapped UserListItem[]>, meta: { page, limit, total, totalPages } }`.
- **`POST` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_USER_CREATE)` (or `PERMISSION_USER_MANAGE`).
  - Parses the request body as JSON, validates with `createUserSchema`. On validation failure, returns `VALIDATION_ERROR` (HTTP 422) with Zod error details.
  - Checks for existing user with the same email. If found, returns `CONFLICT` (HTTP 409).
  - Hashes the password with bcrypt (cost factor matching 1.2.1 / 1.3.1).
  - Creates the user with a Prisma transaction:
    - Create the `User` row.
    - Create `UserRole` rows for each provided `roleId`.
  - On success, calls `writeAuditLog({ action: 'USER_CREATED', actorId: session.user.userId, targetUserId: user.id, description: \`Created user \${user.email}.\`, metadata: { email, roleIds } })`.
  - Returns `{ success: true, data: <UserListItem> }` with HTTP 201.
- All error responses use the master plan envelope. All non-success paths are wrapped in try/catch with `INTERNAL_ERROR` on unexpected errors.

### Step 5: Create `src/app/api/users/[userId]/route.ts` — Get, Update, Deactivate Endpoints

**Goal:** Provide per-user endpoints for retrieval, update, and deactivation.

**Files affected:** `src/app/api/users/[userId]/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`GET` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_USER_READ)`.
  - Reads `userId` from the route params.
  - Queries the user with `include: { userRoles: { include: { role: { include: { rolePermissions: { include: { permission: true } } } } } } }`.
  - Computes the `permissions` array by flattening role → rolePermissions → permission across all the user's roles.
  - If the user does not exist, returns `NOT_FOUND` (HTTP 404).
  - Returns `{ success: true, data: <UserDetail> }`.
- **`PATCH` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_USER_UPDATE)` (or `PERMISSION_USER_MANAGE`).
  - Reads `userId` from the route params.
  - Parses and validates the body with `updateUserSchema`.
  - If `email` is being changed, checks for conflicts with other users. If conflict, returns `CONFLICT` (HTTP 409).
  - Captures the "before" state of the user for the audit log metadata.
  - Updates the user in a Prisma transaction:
    - Update the `User` row with the provided fields (use `update` with the changed fields only).
    - If `roleIds` is provided, delete all existing `UserRole` rows for the user and create new ones for the provided IDs. Capture the "before roles" and "after roles" for the audit log.
  - On success, calls `writeAuditLog({ action: 'USER_UPDATED', actorId, targetUserId: userId, description: \`Updated user \${user.email}.\`, metadata: { before, after, rolesBefore, rolesAfter } })`.
  - If status was changed from `ACTIVE` to `INACTIVE`, use action `'USER_DEACTIVATED'` instead. If from `INACTIVE` to `ACTIVE`, use `'USER_REACTIVATED'`. This split keeps audit log queries by action meaningful.
  - Returns `{ success: true, data: <UserListItem> }`.
- **`DELETE` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_USER_DELETE)` (or `PERMISSION_USER_MANAGE`).
  - Reads `userId` from the route params.
  - Soft-deletes by setting `status: INACTIVE` (NOT a hard delete).
  - If the user is already `INACTIVE`, returns `CONFLICT` (HTTP 409) — or succeeds idempotently; the agent chooses and documents.
  - **Important:** The `verifyCredentials` function in 1.2.1 already rejects `INACTIVE` and `SUSPENDED` users, so deactivated users cannot log in. The "deletion" here is a status change, not a row deletion.
  - On success, calls `writeAuditLog({ action: 'USER_DEACTIVATED', actorId, targetUserId: userId, description: \`Deactivated user \${user.email}.\`, metadata: { email } })`.
  - Returns `{ success: true, data: { message: 'User deactivated.' } }`.
- All error responses use the master plan envelope.

### Step 6: Create `src/app/api/users/[userId]/reset-password/route.ts` — Admin Password Reset

**Goal:** Allow an admin to set a new temporary password for any user.

**Files affected:** `src/app/api/users/[userId]/reset-password/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`POST` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_USER_MANAGE)`.
  - Reads `userId` from the route params.
  - The server generates a random temporary password (e.g., 12 characters, mixed case, digits, symbols). The agent should use `crypto.randomBytes` or a similar secure generator. The exact algorithm is the agent's choice but must be cryptographically random.
  - Alternatively (the agent's choice), accepts a `newPassword` in the body validated with `resetPasswordSchema`. This is useful if the admin wants to set a specific password rather than a random one.
  - **Document the choice clearly:** The recommended approach is to **always generate a random password** server-side and return it in the response. This avoids the admin typing a weak password and ensures the temporary password is strong.
  - Hashes the new password with bcrypt.
  - Updates the user's password.
  - On success, calls `writeAuditLog({ action: 'PASSWORD_RESET_BY_ADMIN', actorId, targetUserId: userId, description: \`Reset password for user \${user.email}.\`, metadata: { email } })`.
  - Returns `{ success: true, data: { temporaryPassword, userId } }`.
  - **The temporary password is shown to the admin once in this response.** It is never returned again. The agent must document this clearly in the response and in the UI: "Copy this password and send it to the user securely. It will not be shown again."

## 8. Common Pitfalls

The following mistakes are common when building the user management surface and are worth avoiding.

### 8.1 Forgetting to Filter the Password Field Out of API Responses

The `User` table includes a `password` field. Every API response that returns user data must exclude this field. Forgetting to omit it is a critical security bug — the bcrypt hash should never leave the server.

### 8.2 Hard-Deleting Users Instead of Soft-Deleting

Users must be deactivated by setting `status: INACTIVE`, not by deleting the row. Hard-deleting would break audit log references (the `AuditLog.targetUserId` would point to a non-existent user) and would prevent investigation of historical actions. Always soft-delete.

### 8.3 Not Wrapping Multi-Step Operations in a Transaction

Updating a user with role changes involves two operations: update the `User` row and replace the `UserRole` rows. If these are not wrapped in a Prisma transaction, a failure between them leaves the database in an inconsistent state (user with old roles but new email, etc.).

### 8.4 Returning the Temporary Password in Subsequent API Calls

The temporary password returned from `/api/users/[userId]/reset-password` is shown **once**. It must not be stored in a field on the `User` row (it is hashed before storage, like any password). Subsequent calls to `GET /api/users/[userId]` must not return the temporary password.

### 8.5 Calling the Audit Log Helper Without Awaiting

The audit log helper is async. If the agent calls `writeAuditLog()` without awaiting, the API response may return before the audit log write completes (or the write may be silently dropped). The agent should `await writeAuditLog(...)` — but the helper's best-effort design means a failure will not fail the response.

### 8.6 Using the Wrong Permission Constant

The `withPermission()` function accepts only the `Permission` type. Passing a string literal like `'user:manage'` works at runtime but loses the type safety the union provides. Always import the constant: `withPermission(handler, PERMISSION_USER_MANAGE)`.

### 8.7 Allowing a User to Deactivate Themselves

A user with `user:manage` permission should not be able to deactivate their own account (this would lock them out). The `DELETE /api/users/[userId]` handler must check `userId !== session.user.userId` and return `CONFLICT` (HTTP 409) with a clear message if they attempt self-deactivation.

### 8.8 Allowing a User to Remove Their Own SUPER_ADMIN Role

Similar to self-deactivation: if a user removes their own SUPER_ADMIN role assignment and they are the only super-admin, the system becomes un-manageable. The `PATCH /api/users/[userId]` handler must enforce a "at least one super-admin must remain" invariant. This is a documented but optional safeguard in Phase 1 — the agent may implement it or document it as a future hardening step.

### 8.9 Forgetting to Seed Roles for the Form to Work

The create/edit form's role multi-select requires the list of available roles. If the page does not fetch roles, the multi-select will be empty. The page must query `db.role.findMany()` and pass the result to the form.

### 8.10 Putting Server Logic in Client Components

The `UserTable`, `UserForm`, `RoleMultiSelect`, and `ResetPasswordDialog` are client components (they have interactivity). They must not directly query Prisma. They call the API endpoints via `fetch`. The page components (server components) fetch data and pass it as props to the client components.

### 8.11 Inconsistent Status Display

The status badge in the table uses colour variants. Ensure that:

- `ACTIVE` = default (or success) variant.
- `INACTIVE` = secondary (or muted) variant.
- `SUSPENDED` = destructive variant.

Inconsistent status display confuses admins about which users are usable.

### 8.12 Missing Toast Notifications

Success and error toasts are the primary feedback mechanism for the user management UI. Missing toasts leave the admin wondering whether their action succeeded. Every mutation must show a Toast on success and on failure.

### 8.13 Exposing Audit Log Metadata with Sensitive Data

The audit log metadata JSON blob may contain before/after values for user updates. Be careful not to include the user's plaintext password (it should never be in the metadata) or other sensitive fields. The agent should explicitly construct the metadata object with safe fields.

### 8.14 Not Handling the "User Not Found" Case in the Edit Page

If the `userId` in the URL does not match any user, the edit page must handle this gracefully (404 page or redirect). The agent should not let the page crash with an unhandled error.
