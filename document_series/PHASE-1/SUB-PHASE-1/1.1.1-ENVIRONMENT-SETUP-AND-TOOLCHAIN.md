## 1. Document Purpose

This document takes the workspace from **"empty folder"** to **"Next.js project created with a working toolchain"**. It establishes every tool, configuration file, and convention that exists **outside** the application's source code — the developer workstation, the package manager, the linting and formatting toolchain, the pre-commit hook, and the environment variable conventions.

When this document is complete, an agent or developer can clone the repository, run the documented install command, and have a runnable application that lints, formats, type-checks, and accepts commits through a working pre-commit hook.

### 2.1 In Scope

## 4. Step-by-Step Implementation Plan

The agent must execute these steps in order. Each step lists its **goal**, the **files affected**, and **specific instructions** to remove ambiguity.

### Step 1: Verify the Node.js Version

**Goal:** Confirm that the developer's machine has a supported Node.js version installed.

**Files affected:** None (read-only check).

**Specific instructions:**

- The master plan specifies Next.js 14+. Next.js 14 requires Node.js 18.17 or later. Document the requirement as **Node.js 20.x LTS** (the current LTS at the time of writing) as the supported version. Node.js 22 LTS is also acceptable.
- The agent should run `node -v` to check the installed version.
- If the version is below 18.17, the agent should stop and instruct the human reviewer to install a supported version (via nvm, fnm, or the official installer) before continuing.
- Document the supported version in the `README.md` (created at the end of this document) under a "Requirements" section.

### Step 2: Verify Yarn is Installed

**Goal:** Confirm Yarn is the active package manager.

**Files affected:** None (read-only check).

**Specific instructions:**

- The master plan mandates Yarn (Section 3.1 explicitly: "Package Manager: Yarn — Specified requirement").
- Run `yarn -v` to verify Yarn is installed. Yarn 1.22+ or Yarn Berry (3.x/4.x) is acceptable.
- If Yarn is not installed, the agent should install it via `npm install -g yarn` (the agent may need to ask the human to run this if the environment does not allow global installs).
- The agent should NOT use npm or pnpm. The master plan's choice is Yarn.

### Step 3: Create the Next.js Project

**Goal:** Use the official `create-next-app` scaffolder to create a new Next.js 14+ project in the current directory.

**Files affected:** The entire project root (created by the scaffolder).

**Specific instructions:**

- Run the `create-next-app` command with the following flags (the agent must use a non-interactive command since it cannot respond to prompts):

  - `--typescript` — enables TypeScript.
  - `--tailwind` — enables Tailwind CSS.
  - `--eslint` — enables ESLint.
  - `--app` — uses the App Router.
  - `--src-dir` — places source under `src/` (this is required by the master plan's folder structure in Section 7).
  - `--import-alias "@/*"` — sets the path alias (this is required by the master plan and Phase 1 standards).
  - `--use-yarn` — explicitly uses Yarn.
  - `--no-turbopack` — keeps the default Webpack-based dev server for now (Turbopack is optional and can be added later if needed).
  - The project name should be `smart-queue-system` (matching the master plan) or the agent should create the project in the current directory using `.` as the name.

- If the workspace already has a `document_series/` folder that should be preserved, the agent should create the project in a sub-folder (e.g., `app/`) and adjust the path expectations. The recommended structure is to keep the application code and the document series as siblings, with the application at the workspace root or in a clearly named sub-folder. **The agent should ask the human reviewer** if this is unclear.

- Verify after creation that the project root contains: `package.json`, `tsconfig.json`, `next.config.ts` (or `.mjs`), `tailwind.config.ts`, `postcss.config.mjs`, `.eslintrc.json` (or `eslint.config.mjs` for newer versions), `.gitignore`, `src/app/`, `public/`.

- Confirm that `tsconfig.json` has `paths` configured with `"@/*": ["./src/*"]`.

### Step 4: Extend ESLint Configuration with Project-Specific Rules

**Goal:** Add project-specific lint rules beyond the Next.js defaults.

**Files affected:** `.eslintrc.json` (or `eslint.config.mjs` — the agent should detect which the scaffolder created and extend it).

**Specific instructions:**

- The agent should detect the ESLint config file format. Next.js 14+ may use either `.eslintrc.json` (legacy) or the new flat config `eslint.config.mjs`. The agent must extend whichever exists.
- Add the following rules / conventions:
  - **Import order:** External packages first (alphabetical), then internal imports grouped by `@/components`, `@/lib`, `@/hooks`, `@/types`, `@/schemas`.
  - **No unused imports** (the Next.js default already covers this; verify it is enabled).
  - **No `any` types:** the agent should install `eslint-plugin-no-relative-import-paths` if needed and configure rules to disallow TypeScript `any` (e.g., via `@typescript-eslint/no-explicit-any` set to `error`).
  - **Consistent naming:** components PascalCase, modules camelCase (this is a convention enforced by code review, not by ESLint; the agent may add a comment in `.eslintrc` documenting it).
  - **No deep relative imports:** the agent should configure `eslint-plugin-no-relative-import-paths` to disallow relative imports that go up more than one level (i.e., disallow `../../` and deeper). All internal imports must use `@/`.

- Install any required ESLint plugins via `yarn add -D` and update the config file accordingly.

### Step 5: Configure Prettier

**Goal:** Install and configure Prettier with project-wide formatting rules, integrated with ESLint so they do not conflict.

**Files affected:** `.prettierrc` (new), `.prettierignore` (new), `.eslintrc.json` (updated to disable conflicting rules).

**Specific instructions:**

- Install Prettier as a dev dependency: `yarn add -D prettier`.
- Install `eslint-config-prettier` to disable ESLint rules that conflict with Prettier: `yarn add -D eslint-config-prettier`.
- Create `.prettierrc` with the following settings:
  - `printWidth: 100`
  - `tabWidth: 2`
  - `useTabs: false`
  - `semi: true`
  - `singleQuote: true`
  - `trailingComma: "all"`
  - `bracketSpacing: true`
  - `arrowParens: "always"`
  - `endOfLine: "lf"`
- Create `.prettierignore` containing: `node_modules`, `.next`, `out`, `dist`, `build`, `coverage`, `public`, `prisma/migrations`, `*.lock`, `yarn.lock`, `package-lock.json`.
- Update the ESLint config to add `"prettier"` to the `extends` array (or the equivalent for flat config) so conflicting rules are disabled.
- Add a `prettier` script to `package.json`: `"prettier": "prettier --write ."` and `"prettier:check": "prettier --check ."` (the document will use the term `format` for the lint script — see Step 9).

### Step 6: Install and Configure Husky and lint-staged

**Goal:** Set up a pre-commit hook that runs ESLint and Prettier on staged files only.

**Files affected:** `.husky/pre-commit` (new), `package.json` (updated), `lint-staged.config.js` or `package.json` `lint-staged` field (new/updated).

**Specific instructions:**

- Install Husky: `yarn add -D husky`.
- Install lint-staged: `yarn add -D lint-staged`.
- Enable Husky: `yarn husky init` (this creates `.husky/` and adds a `prepare` script to `package.json`).
- Create `.husky/pre-commit` containing the command: `yarn lint-staged`.
- Add a `lint-staged` configuration to `package.json` (preferred) or as `lint-staged.config.js`:

  - For `*.{ts,tsx,js,jsx}` files: run `eslint --fix` and `prettier --write`.
  - For `*.{json,md,css}` files: run `prettier --write`.

- Verify the hook works by staging a small intentional formatting change and committing. The hook should reformat the file and allow the commit to proceed.

### Step 7: Create the Environment Variable Files

**Goal:** Establish the `.env` file conventions and the initial variable list.

**Files affected:** `.env` (new, gitignored), `.env.example` (new, committed), `.env.local` (new, gitignored), `.gitignore` (updated).

**Specific instructions:**

- **Initial variable list** (documented in `.env.example` with placeholder values and inline comments):

  | Variable          | Purpose                                        | Example Value (placeholder)                          |
  | ----------------- | ---------------------------------------------- | ---------------------------------------------------- |
  | `DATABASE_URL`    | Prisma datasource URL (SQLite)                 | `file:./dev.db`                                      |
  | `NEXTAUTH_SECRET` | JWT signing secret                             | `change-me-in-production-please-use-32-bytes-random` |
  | `NEXTAUTH_URL`    | Public base URL for NextAuth callbacks         | `http://localhost:3000`                              |
  | `AUTH_URL`        | NextAuth v5 preferred alias for `NEXTAUTH_URL` | `http://localhost:3000`                              |
  | `APP_ENV`         | Application environment identifier             | `development`                                        |

- `.env` should contain a copy of `.env.example` with the same placeholder values, so a fresh clone has working defaults out of the box. The agent should generate a real `NEXTAUTH_SECRET` using `openssl rand -base64 32` and put the result in `.env` (not in `.env.example` — `.env.example` must contain a clear placeholder so humans know to replace it).

- `.env.local` should be created as an empty file with a single comment line: `# Local overrides go here. This file is gitignored.` This is created for future per-developer overrides.

- All three files must be added to `.gitignore` except `.env.example`:
  - `.env*` is gitignored.
  - `!.env.example` is the exception that keeps `.env.example` tracked.

### Step 8: Create the `.gitignore`

**Goal:** Ensure all generated, local, and sensitive files are excluded from version control.

**Files affected:** `.gitignore` (the agent should extend the one created by `create-next-app`).

**Specific instructions:**

- The scaffolder creates a starter `.gitignore`. The agent should verify it covers the following and add any missing entries:

  - `node_modules/`
  - `.next/`
  - `out/`
  - `build/`
  - `dist/`
  - `coverage/`
  - `.env*` (with `!.env.example` exception)
  - `.vercel/` (if using Vercel)
  - `*.tsbuildinfo`
  - `next-env.d.ts` (this is auto-generated, but should be committed for type safety — verify the scaffolder's choice and follow the same convention)
  - `.husky/_/` (Husky internal directory)
  - `prisma/dev.db*` (the SQLite database file and its journal)
  - `prisma/migrations/dev.db*` (in case the migration folder ever contains a local DB)
  - IDE folders: `.vscode/` (except `.vscode/settings.json` and `.vscode/extensions.json` if the team uses them — leave the user's choice), `.idea/`
  - OS metadata: `.DS_Store`, `Thumbs.db`, `desktop.ini`

### Step 9: Add the `package.json` Scripts

**Goal:** Provide a complete, consistent set of scripts that cover all common operations.

**Files affected:** `package.json` (scripts section only).

**Specific instructions:**

- The `create-next-app` scaffolder creates a basic scripts section. The agent should add the following scripts (verify they work):

  | Script            | Command                                 | Purpose                                       |
  | ----------------- | --------------------------------------- | --------------------------------------------- |
  | `dev`             | `next dev` (default)                    | Start the development server                  |
  | `build`           | `next build` (default)                  | Build for production                          |
  | `start`           | `next start` (default)                  | Start the production server                   |
  | `lint`            | `next lint`                             | Run ESLint (no fix)                           |
  | `lint:fix`        | `next lint --fix`                       | Run ESLint with auto-fix                      |
  | `format`          | `prettier --write .`                    | Format all files                              |
  | `format:check`    | `prettier --check .`                    | Verify formatting without writing             |
  | `type-check`      | `tsc --noEmit`                          | Run TypeScript type checking without emitting |
  | `prisma:generate` | `prisma generate`                       | Generate the Prisma client                    |
  | `prisma:migrate`  | `prisma migrate dev`                    | Apply migrations in development               |
  | `prisma:seed`     | `prisma db seed`                        | Run the seed script                           |
  | `db:reset`        | `prisma migrate reset`                  | Reset the database (development only)         |
  | `prepare`         | `husky` (already added by `husky init`) | Husky install hook                            |

- The agent should also add the `prisma.seed` field in `package.json` (separate from the scripts) so `yarn prisma:seed` knows what command to run. For now, configure it as `tsx prisma/seed.ts` (the `prisma/seed.ts` file itself is created by document 1.1.3, but the script wiring is owned by 1.1.1). If `tsx` is not yet installed, the agent should install it as a dev dependency: `yarn add -D tsx`.

- The agent should NOT add scripts for things outside this sub-phase's scope (e.g., `test`, `e2e`, `storybook` are not part of Sub-Phase 1.1).

### Step 10: Create the `README.md`

**Goal:** Document the project setup so a new developer can onboard quickly.

**Files affected:** `README.md` (new, at the project root).

**Specific instructions:**

- The README should contain, at minimum:
  - Project name and one-line description.
  - **Requirements:** Node.js 20.x LTS, Yarn 1.22+ or Berry.
  - **Getting Started** section: `yarn install`, `cp .env.example .env`, `yarn prisma:migrate` (note: this will fail until 1.1.3 is complete — add a note), `yarn dev`.
  - **Available Scripts** section: a table of all the scripts from Step 9 with one-line descriptions.
  - **Project Structure** section: a brief tree of the top-level folders with one-line descriptions. Note that the full structure is filled in by 1.1.2 — keep this section minimal at this point.
  - **Conventions** section: link to the document series (briefly) and note the path alias, naming conventions, and commit message format.

### Step 11: Verify the Toolchain Works

**Goal:** Confirm every tool and script works as expected.

**Files affected:** None (verification step).

**Specific instructions:**

- The agent should run the following commands in order and verify each exits with code 0 and produces the expected output:
  1. `yarn install` — should succeed.
  2. `yarn lint` — should pass with zero errors and zero warnings.
  3. `yarn format:check` — should pass (the format step has not been run yet, so files may not be formatted; if this fails, run `yarn format` once and verify the result).
  4. `yarn type-check` — should pass with zero errors.
  5. `yarn build` — should succeed.
  6. `yarn dev` — should start the dev server. The agent should briefly verify the root URL responds (HTTP 200) and then stop the server.

- The agent should also test the Husky pre-commit hook by:
  - Creating a small test change (e.g., adding a comment to a file).
  - Running `git add` on the change.
  - Running `git commit -m "test: verify pre-commit hook"`.
  - Verifying the hook fires and the commit proceeds.
  - The agent should reset this test commit with `git reset --soft HEAD~1` to keep the history clean.

## 8. Common Pitfalls

The following are mistakes the agent is likely to make. Each is listed with how to avoid it.

| Pitfall                                                                                    | How to Avoid                                                                                                                                                                     |
| ------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Using `npm` or `pnpm` instead of `yarn`                                                    | The master plan mandates Yarn. The agent should verify `yarn -v` works before any other command.                                                                                 |
| Forgetting to commit the Prisma seed wiring in 1.1.3's commit boundary                     | The `prisma.seed` field in `package.json` and the `tsx` dependency are owned by 1.1.1, but the `prisma/seed.ts` file itself is owned by 1.1.3. Keep them in separate commits.    |
| Creating folders inside `src/` that 1.1.2 will own                                         | This document should only touch the project root, config files, and the scaffolding-generated `src/app/` content. Do not preemptively create `src/components/`, `src/lib/`, etc. |
| Installing Prisma "to save a step"                                                         | Resist this. Prisma is owned by 1.1.3 and installing it here will cause duplicate work and possible version drift.                                                               |
| Using the legacy ESLint config when the scaffolder created the flat config (or vice versa) | The agent should `cat` the existing ESLint config file first, then extend it in whatever format it uses.                                                                         |
| Forgetting to enable Husky's `prepare` script                                              | The `husky init` command should add it automatically, but the agent should verify it is present. Without it, new clones will not have hooks installed.                           |
| Setting `NEXTAUTH_SECRET` in `.env.example` to a real value                                | `.env.example` must contain a placeholder so it is obvious that the value must be replaced. The real value goes in `.env` only.                                                  |
| Creating a Turbopack-specific dev script before confirming the team wants Turbopack        | Use the default Webpack-based dev server. Turbopack is opt-in.                                                                                                                   |
| Committing the `.env` file                                                                 | `.env` must be gitignored. Verify with `git check-ignore .env` after the commit (it should report "ignored").                                                                    |
| Adding `test` or `e2e` scripts                                                             | Not part of Sub-Phase 1.1. Defer to Phase 5 or a later sub-phase that needs them.                                                                                                |

## 9. AI Agent Guidance

This section contains additional guidance for the AI agent that is not specific to any one step.

### 9.1 Behavioural Reminders

- The agent must follow the steps in order. The order is the dependency order.
- If a step fails (e.g., `yarn lint` reports an error), the agent should fix the issue and re-run before proceeding. The agent must not skip a failed verification.
- If the agent is uncertain about a configuration choice, it should pick the simpler option and document the choice in the commit body.
- The agent should not modify any file outside the list in Section 5.

### 9.2 What the Agent Should Not Guess

- **Do not guess** the project's intended name. If unclear, use the master plan's name `smart-queue-system` or ask the human reviewer.
- **Do not guess** the `DATABASE_URL` value. The default is `file:./dev.db` (SQLite at the project root).
- **Do not guess** ESLint rule values. The values in Step 4 are prescriptive; deviate only with justification.
- **Do not guess** the Husky hook name. The hook directory is `.husky/` and the file is `pre-commit`.

### 9.3 Escalation Path

If the agent encounters any of the following, it should stop and ask the human reviewer:

- The `create-next-app` scaffolder fails or asks interactive questions.
- The Node.js or Yarn version is unsupported and cannot be upgraded.
- An existing `.eslintrc.json` exists in a format the agent does not recognize.
- The `prisma.seed` field has already been configured (this suggests 1.1.3 has partially been done and the agent should check git history).

### 9.4 Success Signal

This document is complete when the verification checklist in Section 6 is fully checked off and the single commit described in Section 7 is on the `main` branch with a clean working tree.
