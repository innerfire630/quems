## 1. Document Purpose

This document transforms the bare Next.js project (created by 1.1.1) into a **fully structured application skeleton** with:

- Every folder and route group from the master plan, present and reachable.
- Path aliases configured and verified.
- Global, auth, and dashboard layouts (with stub content).
- The complete Tailwind theme extension (all colour tokens from Master Plan Section 6.2).
- shadcn/UI initialized with the Phase 1 component set.

When this document is complete, the agent can navigate to any of the placeholder routes listed in the master plan and see a styled page that uses the project's design tokens.

---

## 2. Scope and Boundaries

### 2.1 In Scope

- Creating every folder in `src/` per Master Plan Section 7.
- Creating placeholder `page.tsx` files for every route group so all URLs resolve.
- Creating structural `layout.tsx` files for `(auth)`, `(dashboard)`, `(officer)`, and the root.
- Extending `tailwind.config.ts` with the full colour token system from Master Plan Section 6.2.
- Configuring the primary font via `next/font` (Google Font) or a self-hosted fallback.
- Initializing shadcn/UI: creating `components.json`, installing the base components, and the `lib/utils.ts` `cn()` helper.
- Verifying the path alias `@/*` works.

### 2.2 Out of Scope (Deferred to Other Documents)

- Any Prisma, database, or seed code (owned by 1.1.3).
- Any authentication, login form, or session logic (owned by 1.2.x).
- Any RBAC, permission checks, or user management UI (owned by 1.3.x).
- Any business domain logic (queue, counter, ticket, kiosk, display, notification — all Phase 2+).
- Any real component implementations beyond structural placeholders and shadcn/UI primitives.

### 2.3 What This Document Does NOT Touch

- It does **not** install Prisma, NextAuth, or any authentication library.
- It does **not** create any database connection code.
- It does **not** create any business domain components (counter, display, kiosk, etc.) — only the empty folders for them.
- It does **not** implement any conditional UI based on user roles or permissions.

## 4. Step-by-Step Implementation Plan

The agent must execute these steps in order. Each step lists its **goal**, the **files affected**, and **specific instructions** to remove ambiguity.

### Step 1: Verify the Path Alias

**Goal:** Confirm the `@/*` path alias is configured in `tsconfig.json`.

**Files affected:** `tsconfig.json` (read-only verification; modify only if missing).

**Specific instructions:**

- The agent should open `tsconfig.json` and verify the following:
  - `compilerOptions.baseUrl` is set (commonly to `.`).
  - `compilerOptions.paths` contains `"@/*": ["./src/*"]`.
  - The `paths` mapping is at the same level as `baseUrl`.
- If any of these is missing, the agent should add them. The `create-next-app` scaffolder usually sets this correctly when `--import-alias "@/*"` is used, so this step is usually a verification.
- The agent should also verify the Next.js config does not override the alias. If `next.config.ts` or `next.config.mjs` has a `webpack` config that ignores the alias, the agent should remove that override.
- Quick functional test: the agent should add a temporary import like `import { cn } from '@/lib/utils';` (after the `cn` helper is created in Step 9) to confirm the alias resolves. The temporary import is removed in a later step.

### Step 4: Write the Placeholder Pages

**Goal:** Every route group has at least one reachable `page.tsx` that renders a styled placeholder.

**Files affected:** All `page.tsx` files created in Step 3.

**Specific instructions:**

Each placeholder page should render:

- The route group's name as a heading (e.g., "Login" for `(auth)/login`, "Dashboard" for `(dashboard)`, "Counter 1" for `(officer)/counter/[counterId]` with a friendly note that the counter ID is dynamic).
- A short one-line description of what this page will eventually contain.
- A note that this is a placeholder: e.g., "This page is a placeholder. Full implementation is in [document reference]."
- Use semantic HTML (one `<main>`, one `<h1>`, brief content).
- Apply the Tailwind theme tokens (e.g., `bg-background text-text-primary`) so the design system is visible.

**For dynamic routes** (e.g., `[counterId]`, `[userId]`, `[serviceId]`), the placeholder should display the dynamic parameter from the route (using the appropriate Next.js mechanism) so it is clear the routing is working.

**Example patterns (described, not coded):**

- `src/app/(auth)/login/page.tsx`: A centred card with a heading "Login", a subheading "Authentication is implemented in 1.2.3", and a single disabled button labelled "Sign in (placeholder)".
- `src/app/(dashboard)/page.tsx`: A page with heading "Dashboard" and a paragraph "Welcome to the admin dashboard. Full overview widgets are implemented in 1.3.3 / Phase 5."
- `src/app/(dashboard)/users/page.tsx`: A heading "Users" and a paragraph "User management UI is implemented in 1.3.3."
- `src/app/display/page.tsx`: A heading "Display Board" on a `bg-display-bg text-display-text` background, with a placeholder message "Display board is implemented in Phase 3."
- `src/app/kiosk/page.tsx`: A heading "Kiosk" on a white background, with "Kiosk is implemented in Phase 2."
- `src/app/security/page.tsx`: A heading "Security Officer", with "Security officer screen is implemented in Phase 4."
- `src/app/(officer)/counter/[counterId]/page.tsx`: A heading "Officer Dashboard" and a paragraph showing the dynamic counter ID, with "Officer dashboard is implemented in Phase 4."

The agent should keep the placeholders consistent: same layout structure, same heading style, same placeholder note format.

### Step 5: Create the Layouts

**Goal:** Each route group has a structural `layout.tsx` that defines the visual frame (sidebar, top bar, centred card, etc.) but contains only stub content.

**Files affected:** All `layout.tsx` files for the route groups.

**Specific instructions:**

**Root layout `src/app/layout.tsx`:**

- HTML structure with `<html lang="en">` and `<body>`.
- Font configuration: the agent should add a Google Font via `next/font/google` (the recommended choice is `Inter` for body text and optionally `JetBrains Mono` for ticket numbers; the master plan suggests `Inter` as the primary). Import the font, apply the className to `<body>`, and set up CSS variables for the font family in `globals.css`.
- Global metadata: title "Smart Queue Management System", description matching the master plan.
- Renders `{children}` only.

**Auth layout `src/app/(auth)/layout.tsx`:**

- A full-viewport container with `min-h-screen flex items-center justify-center bg-background p-4`.
- Renders `{children}` inside a centred card-shaped container (use a shadcn `Card` if available, otherwise a `div` with the appropriate border and padding).
- This is the structural frame for the login page (and any future auth-related pages).

**Dashboard layout `src/app/(dashboard)/layout.tsx`:**

- Two-column layout: a left sidebar (240px wide, white background, `border-r border-border`) and a main content area.
- **Stub sidebar:** the sidebar shows a brand logo placeholder at the top, a vertical list of nav items (Dashboard, Users, Counters, Services, Reports, Audit Log, Settings), and a logout button at the bottom. Each nav item is rendered as a `Link` to the corresponding route (the links are functional even though the destinations are placeholders). The active nav item should be highlighted using the `primary-light` background and `primary` text colour, with a left border indicator. The agent may use `usePathname()` for the active state.
- **Stub top bar:** above the content area, a 64px-tall bar with a search input placeholder (left or centre) and a user avatar placeholder (right). The user avatar can be a circular div with initials "SA" (super admin) for now.
- The content area has `p-6` padding and renders `{children}`.
- The agent should create the components `AppSidebar` and `TopBar` in `src/components/layout/` as part of this step. The components should be functional shells — they accept no props and render the stub content described above. They will be enhanced by later sub-phases.

**Officer layout `src/app/(officer)/layout.tsx`:**

- Similar to the dashboard layout but with a simplified sidebar (just "Counter" link and "Logout").
- The agent should create `OfficerSidebar` in `src/components/layout/`.

**Display layout:** No layout file needed (no shared frame — the display page is self-contained).

**Kiosk layout:** No layout file needed (kiosk is a single full-screen page).

**Security layout:** No layout file needed (the security page is its own full-screen view).

### Step 6: Extend `tailwind.config.ts` with the Design Tokens

**Goal:** Make every colour token from Master Plan Section 6.2 available as a Tailwind class (e.g., `bg-primary`, `text-text-primary`, `bg-display-bg`).

**Files affected:** `tailwind.config.ts` (modified), `src/app/globals.css` (modified to add CSS variables).

**Specific instructions:**

The agent should add a `theme.extend.colors` object in `tailwind.config.ts` with the following mappings. Each colour should be defined as a CSS variable reference (preferred) or a hex literal.

| Token Name           | Hex Value (from master plan) |
| -------------------- | ---------------------------- |
| `primary`            | `#2563EB`                    |
| `primary-light`      | `#EFF6FF`                    |
| `primary-foreground` | `#FFFFFF`                    |
| `background`         | `#F3F4F6`                    |
| `surface`            | `#FFFFFF`                    |
| `border`             | `#E5E7EB`                    |
| `text-primary`       | `#111827`                    |
| `text-secondary`     | `#6B7280`                    |
| `success`            | `#10B981`                    |
| `success-light`      | `#ECFDF5`                    |
| `warning`            | `#F59E0B`                    |
| `warning-light`      | `#FFFBEB`                    |
| `destructive`        | `#EF4444`                    |
| `destructive-light`  | `#FEF2F2`                    |
| `display-bg`         | `#0F172A`                    |
| `display-accent`     | `#3B82F6`                    |
| `display-text`       | `#F8FAFC`                    |

The agent should also add the typography scale from Master Plan Section 6.3 to `theme.extend.fontSize` (e.g., `display-ticket: "5rem"`, `h1: "1.875rem"`, `h2: "1.25rem"`, `label: "0.875rem"`, `body: "1rem"`, `meta: "0.75rem"`).

**Recommended approach:** define the colours as CSS variables in `globals.css` under `:root` (e.g., `--color-primary: 37 99 235;` — using the RGB triplet so Tailwind's `rgb(var(--color-primary) / <alpha-value>)` pattern works) and reference them in `tailwind.config.ts`. This enables runtime theme switching in the future.

The agent should also add `fontFamily` extensions for the display font (e.g., `display: ["var(--font-display)", "system-ui", "sans-serif"]`).

### Step 7: Configure `globals.css`

**Goal:** Set up the global stylesheet with Tailwind directives, the CSS variables for the theme, and a minimal reset.

**Files affected:** `src/app/globals.css` (modified).

**Specific instructions:**

- Replace the default globals.css with:
  - Tailwind's three directives (`@tailwind base; @tailwind components; @tailwind utilities;`).
  - A `:root` block with all the colour CSS variables.
  - A minimal global reset (consistent box-sizing, no default margins on body, smooth scrolling).
  - The `font-family` applied to `body` using the variable from `next/font`.
  - Any additional global utility classes the project needs (e.g., `.scrollbar-hide` for the kiosk or display board's hide-scrollbar behaviour).
- Do NOT add component-level styles here — that is what Tailwind is for.

### Step 8: Initialize shadcn/UI

**Goal:** Set up shadcn/UI so that components can be added on demand by later sub-phases.

**Files affected:** `components.json` (new), `src/components/ui/` (new files), `src/lib/utils.ts` (new), `tailwind.config.ts` (modified by shadcn), `globals.css` (modified by shadcn).

**Specific instructions:**

- The agent should run the shadcn/UI initialization. The standard command is `npx shadcn@latest init` (or `shadcn-ui@latest init` for older versions). The agent should pass these flags to avoid interactive prompts:
  - `-y` (yes to defaults) — or pass all the configuration flags explicitly.
  - `--base-color slate` (matches the master plan's neutral palette).
  - TypeScript: yes.
  - Style: `default` (the standard shadcn style).
  - CSS variables: yes (so the Tailwind theme tokens integrate).
  - Path aliases: confirm `@/components` for components, `@/lib/utils` for utilities.
  - React server components: yes (Next.js App Router default).
- If the `shadcn init` command is interactive and the agent cannot pass flags, the agent should manually create `components.json` with the configuration above and then add components using the non-interactive flag pattern: `npx shadcn@latest add <component> --yes`.

- After init, the agent should verify:
  - `components.json` exists at the project root.
  - `src/lib/utils.ts` exists and exports a `cn()` function.
  - `tailwind.config.ts` has been updated by shadcn (it usually adds CSS variable colour references).
  - `globals.css` has been updated by shadcn (it usually adds the HSL variables shadcn expects).

- The agent should then add the **Phase 1 component set** using `npx shadcn@latest add`:

  - `button`
  - `card`
  - `input`
  - `label`
  - `form`
  - `dialog`
  - `table`
  - `toast` (this also installs `sonner` or the `toast` primitives)
  - `badge`
  - `dropdown-menu`
  - `separator`
  - `select`
  - `switch`
  - `alert`

- The agent should verify all components are in `src/components/ui/`.

- The agent should also install the supporting packages that shadcn requires but may not auto-install:
  - `sonner` (if shadcn chose sonner for toasts).
  - `react-hook-form` (used by the `form` component).
  - `zod` (used by the `form` component for validation).
  - `@hookform/resolvers` (Zod resolver for react-hook-form).

- These packages will be used by Sub-Phases 1.2 and 1.3 for the login form and user management form, so installing them now is justified.

## 8. Common Pitfalls

| Pitfall                                                                    | How to Avoid                                                                                                                                                                                           |
| -------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Forgetting the parentheses in route group names                            | `(auth)`, `(dashboard)`, `(officer)` are route groups — the parens are part of the folder name and must be preserved. They do not appear in the URL.                                                   |
| Confusing dynamic routes with static ones                                  | `[counterId]`, `[userId]`, `[serviceId]` are dynamic — the brackets are part of the folder name.                                                                                                       |
| Putting `page.tsx` inside the route group itself (e.g., `(auth)/page.tsx`) | This would create a `/` route inside the auth group. The agent should put the page under a named sub-folder (e.g., `(auth)/login/page.tsx`) or follow the master plan exactly.                         |
| Tailwind config not picking up the design tokens                           | Verify by adding a test class to a page and running `yarn dev`. If the colour does not apply, check that the `theme.extend.colors` keys match the class names being used.                              |
| shadcn init interactive prompt blocking the agent                          | Pass all flags non-interactively, or use `npx shadcn@latest init -y` if the version supports it. If neither works, manually create `components.json` and add components with the non-interactive flag. |
| Installing shadcn before Tailwind is set up                                | Tailwind must be working first (the scaffolder does this). The agent should verify Tailwind by adding a `bg-red-500` test class before running shadcn init.                                            |
| Creating folders inside `src/components/<domain>/`                         | The domain folders are owned by later sub-phases. This document only creates the empty folders.                                                                                                        |
| Using npm or pnpm commands instead of yarn                                 | Master plan mandates yarn. Verify every install command uses `yarn add`.                                                                                                                               |
| Forgetting the `src/lib/utils.ts` `cn()` helper                            | shadcn init creates it. If the agent bypassed shadcn init, the helper must be created manually.                                                                                                        |
| Adding too many shadcn components "just in case"                           | Only add the 14 components listed in Step 8. Other components are added on demand by their owning sub-phases.                                                                                          |
| API placeholders throwing errors instead of returning 501                  | The placeholders should always return a clean 501 JSON response with the standard error envelope, not throw or crash.                                                                                  |
| Removing the default `create-next-app` `page.tsx` content                  | The default page in `src/app/page.tsx` (or its equivalent after the route group refactor) should be replaced with a redirect to the dashboard or the login page, not removed entirely.                 |

- The agent should work top-down: folders first, then placeholder pages, then layouts, then Tailwind theme, then shadcn. This order minimises backtracking.

### 9.3 Escalation Path

- The placeholder pages conflict with each other (e.g., two routes both want to be `/`).
