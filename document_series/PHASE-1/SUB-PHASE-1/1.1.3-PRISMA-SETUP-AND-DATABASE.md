## 1. Document Purpose

This document adds the **complete database layer** to the project:

- Prisma is installed and configured.
- The `schema.prisma` file is written to contain **every enum and every model** from the master plan's Section 8, with all fields, types, defaults, indexes, unique constraints, and relations exactly as specified.
- The initial migration is generated and committed.
- The Prisma client singleton in `src/lib/db.ts` is in place.
- The seed script infrastructure (`prisma/seed.ts` and the `prisma.seed` field in `package.json`) is wired so that 1.3.1 can later add the substantive RBAC seed content.

When this document is complete, the database is in a state where every model exists, the Prisma client is importable from `@/lib/db`, and any later sub-phase can immediately read or write data without revisiting the schema.

## 2. Scope and Boundaries

### 2.1 In Scope

- Installing Prisma (`prisma` as dev dependency, `@prisma/client` as runtime dependency).
- Writing `prisma/schema.prisma` containing the SQLite datasource, the client generator, all enums, and all models from Master Plan Section 8.
- Running the initial migration to create the SQLite database file.
- Creating the Prisma client singleton in `src/lib/db.ts` with the hot-reload-safe pattern.
- Creating the seed script stub in `prisma/seed.ts` (logs a placeholder message; substantive content is in 1.3.1).
- Configuring the `prisma.seed` field in `package.json` (this was set up in 1.1.1, but the agent should verify it is correct).
- Adding Prisma scripts to `package.json` (these were also set up in 1.1.1, but the agent should verify they work end-to-end with the actual schema).
- Verifying the database is reachable and the client works.

### 2.2 Out of Scope (Deferred to Other Documents)

- The substantive content of the seed (which roles, permissions, and role-to-permission mappings to insert, and the default super-admin user). All of this is owned by document 1.3.1.
- Any authentication, login, or session logic. Owned by 1.2.x.
- Any RBAC enforcement, permission guards, or user management UI. Owned by 1.3.x.
- Any business domain logic (queue, counter, ticket, display, notification, etc.). Owned by Phase 2+.
- Any PostgreSQL configuration, production hardening, or migration strategy. Owned by Phase 5.

### 2.3 What This Document Does NOT Touch

- It does **not** add any business domain queries or mutations. The Prisma client is set up but no application code uses it (other than the verification step).
- It does **not** seed real data. The seed script stub only logs a message.
- It does **not** create any database views, stored procedures, or triggers. Prisma manages the schema.
- It does **not** configure connection pooling, read replicas, or any production database concerns.

## 4. Step-by-Step Implementation Plan

The agent must execute these steps in order. Each step lists its **goal**, the **files affected**, and **specific instructions** to remove ambiguity.

### Step 1: Install Prisma

**Goal:** Add Prisma and the Prisma client to the project.

**Files affected:** `package.json` (dependencies), `yarn.lock` (lock file).

**Specific instructions:**

- Install Prisma CLI as a dev dependency: `yarn add -D prisma`.
- Install the Prisma client as a runtime dependency: `yarn add @prisma/client`.
- Verify both are present in `package.json`:
  - `prisma` under `devDependencies`.
  - `@prisma/client` under `dependencies`.
- Verify `yarn install` still works (i.e., the lock file resolves cleanly).

### Step 2: Initialize Prisma

**Goal:** Create the `prisma/` directory and the `schema.prisma` file.

**Files affected:** `prisma/` (new folder), `prisma/schema.prisma` (new file).

**Specific instructions:**

- Run `npx prisma init --datasource-provider sqlite` to create the `prisma/` directory and a starter `schema.prisma` file.
- This command also adds a `DATABASE_URL` reference to the `.env` file (the agent should verify it is present and points to `file:./dev.db`).
- After `prisma init`, the `prisma/` directory should contain a `schema.prisma` file with a SQLite datasource and a placeholder `User` model.
- The agent should **replace** the entire `schema.prisma` content in Step 3. The starter content is discarded.

### Step 3: Write the Complete `schema.prisma` File

**Goal:** Write the full schema containing every enum and every model from Master Plan Section 8.

**Files affected:** `prisma/schema.prisma` (overwritten).

**Specific instructions:**

This is the most important step in the entire Sub-Phase 1.1. The schema file must match the master plan **verbatim** — every field, type, default, index, unique constraint, and relation. The agent should treat the master plan as the source of truth and copy the specifications exactly.

**3.1 — Datasource and Generator blocks:**

The file should start with:
- A `datasource db` block specifying `provider = "sqlite"` and `url = env("DATABASE_URL")`.
- A `generator client` block specifying `provider = "prisma-client-js"` and the default output location (or a custom output if the team prefers; the default is recommended).

**3.2 — Enums:**

Define every enum from Master Plan Section 8.1 in the exact order they appear:
- `UserStatus` (values: `ACTIVE`, `INACTIVE`, `SUSPENDED`)
- `Platform` (values: `ANDROID`, `IOS`, `WEB`)
- `CounterOfficerStatus` (values: `AVAILABLE`, `SERVING`, `CLOSED`, `OFFLINE`)
- `CounterEventStatus` (values: `OPENED`, `CLOSED_TEMPORARY`, `CLOSED_PERMANENT`, `REOPENED`)
- `TicketStatus` (values: `WAITING`, `CALLED`, `RECALLED`, `SERVING`, `COMPLETED`, `NO_SHOW`, `TRANSFERRED`, `CANCELLED`)
- `TicketEventType` (values: `ISSUED`, `CALLED`, `RECALLED`, `SERVED`, `COMPLETED`, `NO_SHOW`, `TRANSFERRED`, `CANCELLED`, `PRIORITY_CHANGED`)
- `NotificationType` (values: `TICKET_ISSUED`, `TICKET_RECALLED`, `SYSTEM_MESSAGE`, `BROADCAST`)
- `NotificationStatus` (values: `PENDING`, `SENT`, `DELIVERED`, `FAILED`, `READ`)
- `SettingType` (values: `STRING`, `INTEGER`, `BOOLEAN`, `JSON`)

Note: SQLite does not support native enums in Prisma. The agent has two options:
- **Option A (recommended for SQLite dev):** Define the enums in the Prisma schema using the `enum` keyword. Prisma will translate them to string columns with TypeScript-level type safety. This is the cleanest approach for the development database and will be migrated to PostgreSQL native enums in Phase 5.
- **Option B (more portable):** Use `String` columns with a comment in the schema documenting the allowed values. This loses some type safety but avoids any enum-related issues.

The agent should use **Option A** unless there is a specific reason not to (e.g., the team has decided to avoid enums for portability). Document the choice in the schema file with a comment.

**3.3 — Models:**

Define every model from Master Plan Section 8.2 in dependency order, with all fields, types, defaults, relations, indexes, and unique constraints as specified. The dependency order is:

1. `User`
2. `Role`
3. `Permission`
4. `RolePermission` (depends on Role and Permission)
5. `UserRole` (depends on User and Role)
6. `Session` (NextAuth, depends on User)
7. `Account` (NextAuth, depends on User)
8. `RefreshToken` (depends on User)
9. `Service`
10. `Counter`
11. `CounterService` (depends on Counter and Service)
12. `CounterOfficer` (depends on User and Counter)
13. `CounterStatusEvent` (depends on Counter and CounterOfficer)
14. `Ticket` (depends on Service, Counter, CounterOfficer)
15. `TicketEvent` (depends on Ticket, Counter, CounterOfficer)
16. `DisplayBoard`
17. `KioskConfig`
18. `DeviceToken` (depends on CounterOfficer)
19. `Notification` (depends on Ticket, CounterOfficer)
20. `NotificationReply` (depends on Notification, CounterOfficer)
21. `BroadcastMessage` (depends on CounterOfficer, NotificationReply, DisplayBoard, User)
22. `QueueDailySnapshot` (depends on Service)
23. `AuditLog` (depends on User)
24. `SystemSetting` (depends on User)

For each model, the agent should:
- Set the `id` field as `String @id @default(cuid())`.
- Add `createdAt DateTime @default(now())` and `updatedAt DateTime @updatedAt` where the master plan specifies them.
- Add `@@unique` composite constraints exactly as specified.
- Add `@@index` declarations exactly as specified.
- Use `?` for nullable fields.
- Use the `@@map` directive if the table name should differ from the model name (the master plan does not require any renames, so this is usually unnecessary).
- Define the relations with `@relation` and use named relations only if there are multiple relations between the same two models (the master plan does not appear to have any such cases, so named relations are usually unnecessary).

The agent should cross-check each model against the master plan's table after writing it. **Do not paraphrase or "improve" the schema — the master plan is authoritative.**

**3.4 — Validation:**

After writing the schema, the agent should run `npx prisma validate` to confirm the schema is syntactically and semantically valid. Any error must be fixed before proceeding.

**3.5 — Formatting:**

The agent should run `npx prisma format` to ensure the schema file is consistently formatted (Prisma's built-in formatter handles indentation, alignment, and section ordering).

### Step 4: Generate the Prisma Client

**Goal:** Generate the TypeScript client from the schema.

**Files affected:** `node_modules/@prisma/client/` (generated), `package.json` `prisma:generate` script (used).

**Specific instructions:**

- Run `yarn prisma:generate` (or `npx prisma generate`) to produce the typed client.
- The agent should verify the command exits with code 0 and the generated types are available. The simplest verification is to run `npx prisma` and confirm the CLI responds, or to attempt a typed import in the verification step (Step 9).
- If the client is not generated, subsequent steps will fail. The agent must not proceed until the client is generated successfully.

### Step 5: Create the Initial Migration

**Goal:** Generate the first migration file containing the entire schema and apply it to the SQLite database.

**Files affected:** `prisma/migrations/<timestamp>_init/migration.sql` (new), `prisma/dev.db` (new, the SQLite database file).

**Specific instructions:**

- Run `npx prisma migrate dev --name init` (the `--name init` flag gives the migration a descriptive name; the Prisma CLI will append a timestamp prefix automatically).
- The command will:
  1. Compare the schema to the previous migration (none, since this is the first).
  2. Generate the SQL migration file.
  3. Apply the migration to the SQLite database.
  4. Regenerate the Prisma client.
  5. Optionally run the seed script (the agent should decline this for now if prompted, since the seed is only a stub).
- The agent should verify:
  - A new folder `prisma/migrations/<timestamp>_init/` exists.
  - The folder contains a `migration.sql` file with all the `CREATE TABLE`, `CREATE INDEX`, `CREATE UNIQUE INDEX` statements.
  - The `prisma/dev.db` file exists at the project root (or wherever the `DATABASE_URL` points to).
- The agent should **commit the migration file** to the repository. The migration is part of the source of truth — without it, new clones cannot recreate the database.

### Step 6: Create the Prisma Client Singleton

**Goal:** Create `src/lib/db.ts` exporting a single Prisma client instance that survives hot-reloads in development.

**Files affected:** `src/lib/db.ts` (new).

**Specific instructions:**

- The file should export a `prisma` constant.
- The pattern is:
  - Declare a module-scope variable (e.g., `const globalForPrisma = globalThis as unknown as { prisma: PrismaClient | undefined };`).
  - In development, use the global-cached client if it exists; otherwise create a new one.
  - In production, always create a new client (no global caching needed).
  - Export the result as `prisma`.
- Include a brief comment explaining why the singleton pattern is used (Next.js hot reload in development can otherwise create many Prisma clients, exhausting the database connection pool).
- Configure the client with logging in development (e.g., `log: ['query', 'error', 'warn']`) but only `['error']` in production. The agent can use `process.env.NODE_ENV` or a project-specific environment variable to determine the mode.
- The export should be typed as `PrismaClient` (imported from `@prisma/client`).
- The file should NOT contain any business logic. It is purely a client export.

### Step 7: Create the Seed Script Stub

**Goal:** Wire up `prisma/seed.ts` so it can be run via `yarn prisma:seed`. The substantive seed content is added by 1.3.1; for now, the script only imports the Prisma client, connects, and logs a message.

**Files affected:** `prisma/seed.ts` (new).

**Specific instructions:**

- The file should import the Prisma client from `@/lib/db` (the singleton created in Step 6).
- The file should be structured as an async function (e.g., `async function main()`) that:
  1. Logs a clear message: `"Seed script stub — substantive content is added in 1.3.1"`.
  2. Performs a trivial Prisma query to verify connectivity (e.g., `await prisma.user.count()`, `await prisma.role.count()`).
  3. Logs the result (e.g., `"Database reachable. User count: 0, Role count: 0"`).
  4. Disconnects from the Prisma client (`await prisma.$disconnect()`).
- The function should be wrapped in a try/catch that logs errors and ensures `prisma.$disconnect()` is called in a `finally` block.
- The file should be executable via `tsx prisma/seed.ts` (the `package.json` `prisma.seed` field is already configured in 1.1.1).
- The agent should NOT seed any real data in this step. No roles, no permissions, no users. The 1.3.1 document owns all of that.

### Step 8: Verify the `prisma:seed` Script Works

**Goal:** Confirm the seed script can be run via the package.json script and that the database connection works.

**Files affected:** None (verification step).

**Specific instructions:**

- The agent should run `yarn prisma:seed`.
- The command should:
  1. Execute the seed script via `tsx prisma/seed.ts`.
  2. Print the stub message and the count values.
  3. Exit with code 0.
- If the command fails, the agent should diagnose the issue:
  - "Cannot find module @/lib/db" — the path alias is not resolving; check `tsconfig.json` and the `tsx` configuration.
  - "Database file not found" — the `DATABASE_URL` is incorrect or the migration was not applied.
  - "Prisma client not generated" — run `yarn prisma:generate` first.

## 8. Common Pitfalls

| Pitfall | How to Avoid |
|---|---|
| Missing a field, type, or default on a model | Cross-check every model against the master plan's Section 8.2 after writing it. The schema is the source of truth for the entire application. |
| Wrong relation direction or missing back-relation | Every relation in the master plan implies a back-relation. For example, `User.roles: UserRole[]` requires `UserRole.user: User`. The agent should verify both sides of every relation. |
| Composite unique constraints forgotten | The master plan explicitly calls out composite unique constraints on `RolePermission (roleId, permissionId)`, `UserRole (userId, roleId)`, `Account (provider, providerAccountId)`, `CounterService (counterId, serviceId)`, `CounterOfficer (userId, counterId)`, and `QueueDailySnapshot (businessDate, serviceId)`. The agent should add a `@@unique([...])` for each. |
| Indexes forgotten | The master plan lists indexes on many models (`User.email`, `Service.code`, `Service.ticketPrefix`, `Service.isActive`, `Counter.number`, `Counter.isActive`, `CounterOfficer.deviceTokens.token`, `CounterOfficer.notifications.userId`, `Ticket.serviceId`, `Ticket.counterId`, `Ticket.status`, `Ticket.businessDate`, `Ticket.ticketNumber`, `DeviceToken.token`, `DeviceToken.counterOfficerId`, `RefreshToken.token`, `RefreshToken.userId`, `AuditLog.userId`, `AuditLog.action`, `AuditLog.entity`, `AuditLog.createdAt`). The agent should add an `@@index([...])` for each. |
| Using `enum` in a way SQLite does not support | Prisma translates enums to string columns for SQLite, so the `enum` keyword is fine. The agent does not need to avoid it. |
| Generating the Prisma client before running `migrate dev` | The order is: write schema → `prisma generate` (optional) → `prisma migrate dev --name init` → client is auto-regenerated. The agent should not skip `migrate dev`. |
| Forgetting `prisma.$disconnect()` in the seed script | Without it, the script may hang or exit non-zero. The agent should use try/finally. |
| Forgetting to commit the migration file | The migration file is the source of truth for the database schema. New clones must run `prisma migrate dev` to recreate the database, and the migration file is what tells `prisma migrate dev` what to do. |
| Adding seed data in this step | Resist. The 1.3.1 document owns all seed content. The 1.1.3 seed script is a stub. |
| `@@id` vs `@@unique` confusion | The master plan's tables distinguish primary keys (`id`), unique constraints (`@unique` on a field, `@@unique` on a composite), and indexes (`@@index`). The agent should not conflate them. |
| Forgetting to update the README's "Database" section | The verification commands (`prisma studio`, `prisma migrate status`) should be documented so a fresh developer can verify the database. |
| Pre-creating migrations for future phases | The agent should create only the initial migration. Future schema changes are added as new migrations in their owning sub-phases. |

## 9. AI Agent Guidance

### 9.1 Behavioural Reminders

- The schema is the single most important artefact of this document. The agent should cross-check every line against the master plan.
- The agent should work in this order: install → init → write schema → validate → format → generate → migrate → client singleton → seed stub → verify. Skipping any step risks the next step failing in a non-obvious way.
- The agent should not "simplify" the schema. Every model, every field, every relation is needed by some later sub-phase.

### 9.2 What the Agent Should Not Guess

- **Do not guess** field types. If the master plan says `String`, the field is `String`. If it says `Int`, the field is `Int`. If it says `Json?`, the field is `Json?` with a question mark for nullability.
- **Do not guess** defaults. If the master plan says `default now()`, use `@default(now())`. If it says `default 0`, use `@default(0)`.
- **Do not guess** relation names. If there are multiple relations between two models, name them. If not, do not.
- **Do not guess** the migration name. Use `init` for the first migration.
