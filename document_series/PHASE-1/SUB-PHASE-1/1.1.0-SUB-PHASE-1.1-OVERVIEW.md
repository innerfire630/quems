## 1. Sub-Phase Purpose

Sub-Phase 1.1 transforms an **empty folder** into a **runnable Next.js application connected to a fully modelled SQLite database** that boots without errors, lints cleanly, type-checks cleanly, and exposes every planned route group as a reachable placeholder page.

The purpose is purely **foundational** — no business logic, no authentication flow, no role enforcement. The only "intelligence" introduced in this sub-phase is the database schema (defined in the master plan) being present, migrated, and verifiable.

### 1.1 Why This Sub-Phase Exists First

Every later sub-phase (1.2, 1.3, and all of Phases 2–5) requires:

- A working Next.js project structure with the correct folder layout, route groups, and path aliases.
- A Prisma schema containing every model the rest of the system depends on.
- A seeded database so the next sub-phases can immediately read/write users, roles, and permissions.
- Linting, formatting, and pre-commit hooks in place so that the codebase stays clean as it grows.

If any of these is missing or done incorrectly, every later sub-phase inherits the mess. Sub-Phase 1.1 isolates this foundational work so that the rest of the project can move quickly.

### 1.2 What This Sub-Phase Is NOT

- It is **not** an authentication implementation. That is Sub-Phase 1.2.
- It is **not** an RBAC implementation. That is Sub-Phase 1.3.
- It is **not** a business domain implementation. Queue, counter, ticket, display, notification logic are all in Phase 2–4.
- It is **not** a production hardening effort. Security headers, rate limiting, audit logging, PostgreSQL migration, CI/CD are all in Phase 5.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to **distribute the complexity** across the three task documents so that no single document is overwhelming. The split is:

- **1.1.1** owns the workstation + root config files (Node, Yarn, ESLint, Prettier, Husky, env, gitignore, package.json scripts, and the act of _creating_ the Next.js project).
- **1.1.2** owns everything that lives _inside_ the created project (folder structure, route groups, layouts, Tailwind theme, shadcn/UI, path aliases, placeholder pages).
- **1.1.3** owns the database (Prisma install, schema file, migration, client singleton, seed script infrastructure).

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 1.1.1 — Environment Setup & Toolchain Configuration

**Owns:** Everything that exists **before** any code is written — the developer workstation, the package manager, the code-quality toolchain, the environment variable conventions, and the initial act of creating the Next.js project.

**Does NOT own:** Folder structure inside the project, route groups, layouts, Tailwind theme, shadcn/UI, Prisma. (Those are 1.1.2 and 1.1.3.)

**Key outputs:**

- A clear Node.js version requirement (e.g., Node 20 LTS) and a verification step.
- Yarn as the package manager, with a `.yarnrc` if needed and a documented `nodeLinker` setting.
- A new Next.js 14+ project created using the official scaffolder, with TypeScript, Tailwind, App Router, ESLint, and the `@/*` import alias enabled.
- ESLint configuration extending the Next.js defaults with project-specific rules (import order, no `any`, no unused imports, consistent naming).
- Prettier configuration (print width, single quotes, trailing comma, semicolons) integrated with ESLint so the two never conflict.
- Husky + lint-staged installed and configured with a pre-commit hook that runs ESLint and Prettier on staged files.
- `.env`, `.env.example`, `.env.local` files in place, with `.env*` gitignored except `.env.example`. Initial variables: `DATABASE_URL`, `NEXTAUTH_SECRET` (placeholder), `NEXTAUTH_URL` (or `AUTH_URL`), `APP_ENV`.
- `package.json` scripts: `dev`, `build`, `start`, `lint`, `lint:fix`, `format`, `format:check`, `type-check`, `prisma:generate`, `prisma:migrate`, `prisma:seed`, `db:reset`, `prepare` (for Husky).
- A complete `.gitignore` covering `node_modules`, `.next`, `.env*` (except `.env.example`), Prisma generated client, IDE files, OS metadata.

**Master Plan sections implemented:** Section 3.1 (core framework), Section 3.4 (tooling), and the toolchain portion of Section 7.

  - **Every enum** from Master Plan Section 8.1: `UserStatus`, `Platform`, `CounterOfficerStatus`, `CounterEventStatus`, `TicketStatus`, `TicketEventType`, `NotificationType`, `NotificationStatus`, `SettingType`.

   │
   ▼
1.1.2  Next.js Scaffold & Folder Structure
   │
   ▼
1.1.3  Prisma Schema & Initial Migration

**Strict order:** 1.1.1 → 1.1.2 → 1.1.3. The documents must be executed in this sequence. There is no parallelism within Sub-Phase 1.1.

## 6. Cross-Document Standards

These standards apply to every document in Sub-Phase 1.1. They are not features — they are conventions established now so every later phase inherits a consistent codebase.

### 6.1 Folder and File Naming

| Item                  | Convention                         | Example                              |
| --------------------- | ---------------------------------- | ------------------------------------ |
| Multi-word folders    | kebab-case                         | `audit-log/`, `mobile-nav/`          |
| Component files       | PascalCase                         | `UserForm.tsx`, `CounterCard.tsx`    |
| Non-component modules | camelCase                          | `auth-utils.ts`, `ticket-service.ts` |
| Colocated type files  | kebab-case with `.types.ts` suffix | `auth.types.ts`                      |

### 6.2 Import Conventions

- All project-internal imports use the `@/` alias.
- External package imports are listed first (alphabetically), then internal imports grouped by `@/components`, `@/lib`, `@/hooks`, `@/types`, `@/schemas`.
- No deep relative imports inside `src/`. Never write `../../../lib/...` — always use `@/lib/...`.

### 6.3 TypeScript Standards

- All exported functions and components have explicit type annotations.
- No `any` types in production code. Use `unknown` and narrow with type guards.
- Domain-specific type files live in `src/types/<domain>.types.ts` and are re-exported from `src/types/index.ts`.

### 6.4 Environment Variables

- `.env` and `.env.local` are gitignored.
- `.env.example` is committed and contains every variable name with a placeholder value and an inline comment.
- All variable names use SCREAMING_SNAKE_CASE.

### 6.5 Commit Standards

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `style:`, `test:`.
- Each task plan document corresponds to one or more commits, but a single commit should not span multiple task plan documents.
- Suggested commit boundaries:
  - End of 1.1.1: `chore: initialize Next.js project with toolchain configuration`
  - End of 1.1.2: `feat: scaffold folder architecture and route groups`
  - End of 1.1.3: `feat: add complete Prisma schema and initial migration`

## 8. AI Agent Implementation Guidance

This section is for the AI agents that will implement the task plan documents. It is guidance that applies across all three task plan documents in this sub-phase.

### 8.1 How to Use the Task Plan Documents

Each task plan document follows the same internal structure:

1. **Scope** — what this document owns and what it does not.
2. **Pre-requisites** — what must already be true (previous documents completed).
3. **Step-by-step implementation plan** — the actual work, in order.
4. **File-level specifications** — for every file the agent will create or modify, a description of its purpose and contents (no code, but enough to know exactly what to write).
5. **Verification checklist** — how the agent proves the work is done.
6. **Commit instructions** — what commit(s) to make at the end.
7. **Common pitfalls** — mistakes to avoid.

The agent should follow the steps in order. Each step lists the **goal** of the step, the **files affected**, and any **specific instructions** that resolve the ambiguity the agent would otherwise have to guess about.

### 8.2 General Guidance

- **Do not skip steps.** Sub-Phase 1.1 is sequential because each step's output is the next step's input. Skipping a step forces the agent to backtrack.
- **Do not add features beyond the document's scope.** The temptation to "just add the login page now" or "just seed the roles now" is real — resist it. The other sub-phases own that work, and doing it here will cause merge conflicts and responsibility confusion later.
- **Verify after every step.** The verification checklist at the end of each document is a hard gate. If the verification fails, the document is not complete, even if the steps were followed.
- **When the master plan is ambiguous, choose the simpler option.** Phase 1 is about establishing foundations, not optimising. If a configuration choice has two reasonable interpretations, pick the one that is easier to change later.
- **Commit at the end of each document** using the commit message specified in the document. A clean commit history makes the diff reviewable later.
- **Document any deviation.** If the agent makes a choice that differs from this document (e.g., a different ESLint rule), it must note the deviation in the final summary so the human reviewer can approve or reject it.

### 8.3 What the AI Agent Should NOT Do

- Should not create any authentication code (NextAuth, login form, JWT logic). That is 1.2.
- Should not seed roles, permissions, or users with substantive content. The 1.1.3 seed stub only logs a message.
- Should not implement any business logic (queue, counter, ticket, display, notification). All of that is Phase 2+.
- Should not configure security headers, rate limiting, or any production hardening. That is Phase 5.
- Should not add PostgreSQL configuration. The database is SQLite for development.
- Should not add any dependency not explicitly mentioned in the document's step-by-step plan.

### 8.4 When to Ask the Human

The AI agent should ask the human reviewer (not guess) if any of the following situations arise:

- The master plan and the phase 1 overview contradict each other. The master plan wins, but the agent should confirm.
- A required package version is unavailable or has a known breaking change that affects the approach.
- A file or folder path listed in the master plan is empty in the latest spec (e.g., the master plan mentions a folder that is no longer relevant).
- The verification step fails for a reason that the document does not anticipate.

In all other cases, the agent should make the simplest reasonable choice, document it, and proceed.
