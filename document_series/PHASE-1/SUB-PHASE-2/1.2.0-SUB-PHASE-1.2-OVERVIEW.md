## 1. Sub-Phase Purpose

Sub-Phase 1.2 transforms the application from **"any visitor can access any page"** to **"only authenticated users with valid tokens can access protected pages, and tokens are issued, validated, and rotated correctly"**.

It establishes **identity** and **session continuity** but does **not yet** decide what an authenticated user is allowed to do. That second question — authorization — is the job of Sub-Phase 1.3.

The three problems this sub-phase solves, in order:

1. **Proving identity** — A user proves who they are with email + password. NextAuth v5 verifies the credentials against the `User` table (from 1.1.3) using bcrypt.
2. **Carrying identity** — A short-lived (15-minute) access token embedded in the NextAuth session is used for every subsequent request. The token is enriched at issuance with `userId`, `email`, `name`, `roles`, and `permissions`.
3. **Refreshing identity** — A long-lived (7-day), database-backed refresh token, stored hashed, is rotated on every refresh. Revoked tokens are rejected.

When this sub-phase is complete, a user can log in via the login page, navigate protected routes, log out, and have their access token refreshed transparently. Mobile clients can also authenticate via the dedicated mobile login endpoint.

---

## 1.1 Why This Sub-Phase Comes Before RBAC

Authentication must work before authorization. The RBAC system in Sub-Phase 1.3 needs to know "who is the current user?" on every protected request. That requires:

- A working login flow that produces a verifiable identity.
- A way to attach that identity to subsequent requests (the JWT session).
- A way to refresh that identity when it expires (the refresh token system).

These three requirements are exactly the scope of Sub-Phase 1.2. Sub-Phase 1.3 builds on top by adding role and permission enforcement.

---

## 1.2 What This Sub-Phase Is NOT

- It is **not** an RBAC implementation. Permission strings exist on the session, but no `withPermission()` guard or `<Can>` component is built in this sub-phase. That is Sub-Phase 1.3.
- It is **not** a user management interface. There is no admin page to create or assign roles here. That is 1.3.3.
- It is **not** a password reset / forgot password flow. Only the login + refresh + logout flows are in scope.
- It is **not** social/OAuth login. The master plan provisions the `Account` table for future OAuth providers, but this sub-phase uses only the credentials provider.
- It is **not** a session viewer, audit log viewer, or any admin-side observability. Those are Phase 5.

## 2. Sub-Phase Scope and Boundaries

### 2.1 In Scope

| Area | What Sub-Phase 1.2 Delivers |
|---|---|
| NextAuth v5 | Installed and configured with the credentials provider and Prisma adapter |
| JWT Session Strategy | Access tokens issued as JWTs, 15-minute expiry, signed with `NEXTAUTH_SECRET` |
| Token Enrichment | `jwt` callback adds `userId`, `email`, `name`, `roles`, `permissions` to the token |
| Session Shape | `session` callback exposes `userId`, `roles`, `permissions` to the client |
| Refresh Tokens | Generate, hash, store, rotate, revoke. Hashed with SHA-256, stored in `RefreshToken` table |
| Refresh Endpoint | `POST /api/auth/refresh` validates cookie/body, rotates, returns new access token + cookie |
| Mobile Login Endpoint | `POST /api/auth/mobile/login` returns JSON with access token, refresh token, and `expiresIn` |
| Mobile Refresh | Mobile clients send the refresh token in the request body to `/api/auth/refresh` |
| Login UI | Centred card with email/password, validation, loading state, error display |
| Route Protection | `middleware.ts` redirects unauthenticated users to `/login` with `callbackUrl` |
| Server-side Guards | Layout-level session check on `(dashboard)/layout.tsx` as defence-in-depth |
| Sign-out Flow | Logout button calls NextAuth sign-out and clears the session |
| Cookie Strategy | Refresh token is `HttpOnly`, `Secure` (in production), `SameSite=Strict`, env-aware |
| Password Hashing | bcrypt compare on login; bcrypt hash on user creation in the seed (delegated to 1.3.1 for content) |

### 2.2 Out of Scope (Explicitly Deferred)

| Deferred To | Item |
|---|---|
| 1.3.1 | Substantive RBAC seed (default super-admin user, role catalogue, permission catalogue). 1.2.1 reads from these tables but does not populate them. |
| 1.3.2 | `withPermission()`, `withRole()`, `usePermission()`, `<Can>` component, role-based middleware rules |
| 1.3.3 | User management UI, admin-initiated password reset, audit logging of user actions |
| Phase 2+ | All service, counter, ticket, kiosk, display, notification, reporting logic |
| Phase 5 | Rate limiting on login/refresh endpoints, security headers, audit log viewer, PostgreSQL migration, CI/CD |

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to **distribute the complexity** across the three task documents. The split is:

- **1.2.1** owns the NextAuth configuration, the credentials provider, the JWT/session callbacks, and the JWT enrichment. It produces a working login that returns a session with `userId`, `roles`, and `permissions`.
- **1.2.2** owns the refresh-token machinery and the mobile endpoints. It produces a `/api/auth/refresh` endpoint, a `/api/auth/mobile/login` endpoint, and the refresh-token utilities.
- **1.2.3** owns the user-facing login page, the middleware that protects routes, and the session-aware dashboard layout foundation.

An AI agent picking up any one of these three documents should be able to complete it independently, knowing that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 1.2.1 — NextAuth Configuration & JWT Strategy

**Owns:** The NextAuth v5 setup, the credentials provider, the Prisma adapter, the JWT callback, the session callback, the access-token expiry, and the environment variables NextAuth consumes.

**Does NOT own:** The refresh-token cookie logic (owned by 1.2.2), the login UI (owned by 1.2.3), the middleware route protection (owned by 1.2.3), and the permission/RBAC enforcement layer (owned by 1.3.2).

**Key outputs:**

- `next-auth` and `@auth/prisma-adapter` installed.
- `src/lib/auth.ts` containing the NextAuth configuration object:
  - Prisma adapter wired to the `src/lib/db.ts` client.
  - Credentials provider accepting email + password, looking up the user in the `User` table, verifying the password with bcrypt.
  - Session strategy set to `jwt` (not database sessions).
  - Session `maxAge` and JWT `maxAge` set to 15 minutes (the access-token expiry).
  - `signIn` callback that triggers refresh-token creation (calls into utilities in `src/lib/auth-utils.ts`, which is created in 1.2.2; in 1.2.1 the callback hook is wired but the actual token creation logic may be a thin pass-through to be filled by 1.2.2).
  - `jwt` callback that enriches the token with `userId`, `email`, `name`, `roles`, `permissions`. Roles and permissions are fetched from the database at sign-in time. On subsequent calls, the callback returns the token unchanged to avoid extra DB hits.
  - `session` callback that exposes `userId`, `roles`, `permissions` to the client in the `session.user` object.
- `src/app/api/auth/[...nextauth]/route.ts` exporting the NextAuth handler.
- `src/types/next-auth.d.ts` (or equivalent) extending the NextAuth type definitions so `session.user.userId`, `session.user.roles`, `session.user.permissions` are strongly typed (no `any`).
- Environment variables documented: `NEXTAUTH_SECRET`, `NEXTAUTH_URL` (or `AUTH_URL`), with a local-dev default and a production-secret generation note.
- A temporary verification route (e.g., `src/app/api/auth/debug-session/route.ts` protected by `APP_ENV !== 'production'`) that returns the current session shape so the agent can confirm enrichment is working.

**Master Plan sections implemented:** Sections 3.2 (backend stack — NextAuth.js v5), 10.1 (token strategy), 10.2 (web authentication flow, JWT portion), and the response envelope/standard error codes from Sections 9.1 and 9.2.

### 4.2 Document 1.2.2 — Access Token & Refresh Token Implementation

**Owns:** Refresh token utilities, the `RefreshToken` table usage, the cookie strategy, the `/api/auth/refresh` endpoint, and the `/api/auth/mobile/login` endpoint (and the mobile refresh flow).

**Does NOT own:** The NextAuth configuration object (owned by 1.2.1), the login UI (owned by 1.2.3), and any role/permission enforcement (owned by 1.3.x).

**Key outputs:**

- `src/lib/auth-utils.ts` exporting:
  - A function to generate a cryptographically secure random refresh token string (using `crypto.randomBytes`).
  - A function to hash the token string with SHA-256 before database storage.
  - A function to validate an incoming refresh token (hash → DB lookup → check `isRevoked === false` and `expiresAt > now` → return associated user or null).
  - A function to revoke a refresh token (set `isRevoked = true`, set `revokedAt`).
  - A function to issue a new access-token JWT (signed with `NEXTAUTH_SECRET`, expiring in 15 minutes, with the enriched payload). May reuse helpers from `src/lib/auth.ts`.
  - A function to rotate a refresh token (revoke old, create new, link `replacedByToken` if the master plan's `RefreshToken` model includes that field — verify against Section 8.2 of the master plan).
- Refresh token expiry: 7 days from issue.
- Cookie configuration: `HttpOnly`, `Secure` (in production only), `SameSite=Strict`, `Path=/`, `Max-Age` set to the refresh-token expiry in seconds. The configuration must be environment-aware.
- `src/app/api/auth/refresh/route.ts`:
  - Accepts the refresh token from the HttpOnly cookie (web flow) OR from the JSON request body (mobile flow — distinguished by a request header or content-type).
  - Validates the token.
  - On success: rotates the token, sets the new refresh-token cookie (web), and returns a new access token in the JSON body (with `expiresIn`).
  - On failure: returns the master plan's standard error envelope with `UNAUTHORIZED` (HTTP 401) and a clear message.
- `src/app/api/auth/mobile/login/route.ts`:
  - Accepts `{ email, password }` in the JSON body.
  - Validates credentials.
  - Returns `{ success: true, data: { accessToken, refreshToken, expiresIn } }` on success.
  - Returns the standard error envelope on failure.
- The `signIn` callback in `src/lib/auth.ts` (from 1.2.1) is updated to call the refresh-token creation utilities on successful web login so that the cookie is set automatically.
- All routes return the master plan's response envelope (Section 9.1) and use the standard error codes (Section 9.2).

**Master Plan sections implemented:** Sections 10.1 (refresh token details), 10.2 (refresh endpoint portion), 10.3 (entire mobile flow), 9.1 (response envelope), 9.2 (error codes), and 9.3 (the three `/api/auth/*` endpoints in the registry).

### 4.3 Document 1.2.3 — Authentication UI & Protected Routes

**Owns:** The login page UI, the `middleware.ts` for route protection, the unauthenticated-redirect logic with `callbackUrl`, the session-aware dashboard layout, the sign-out flow, and the `getServerSession()` server helper.

**Does NOT own:** NextAuth configuration (owned by 1.2.1), refresh token mechanics (owned by 1.2.2), and any role/permission enforcement (owned by 1.3.2).

**Key outputs:**

- `src/app/(auth)/login/page.tsx`:
  - A centred card layout using shadcn `Card`, `CardHeader`, `CardTitle`, `CardDescription`, `CardContent`, `CardFooter`.
  - Form fields for email (using `Input` + `Label`) and password (using `Input` + `Label` with `type="password"`).
  - Submit button (using `Button`) with loading state (disabled + spinner).
  - Error display using shadcn `Alert` with `variant="destructive"`.
  - On submit: call `signIn('credentials', { ... })` from `next-auth/react`. On success, redirect to `callbackUrl` (or `/` as default). On failure, display the returned error.
- `src/schemas/auth.schema.ts`:
  - A Zod schema validating email format (RFC 5322 subset) and non-empty password (minimum length 8 — matches the master plan's password policy from Section 15 if specified, otherwise choose 8 and document it).
  - Used by the login form for client-side validation.
- `src/middleware.ts` at the project root:
  - Exports a `default` middleware function and a `config.matcher` regex.
  - On every request to a protected route, reads the NextAuth session cookie.
  - If the session is missing or invalid, redirects to `/login?callbackUrl=<original-path>`.
  - If the session is present, allows the request to proceed.
  - The matcher includes: `/`, `/(dashboard)/:path*`, `/(officer)/:path*`, `/users/:path*`, `/settings/:path*`, `/audit-log/:path*`, and any other authenticated-only paths.
  - **Explicitly excluded** (pass-through): `/login`, `/display/:path*`, `/kiosk/:path*`, `/security/:path*`, `/api/auth/:path*`, `/api/sse/:path*`, `/api/tickets/issue`, `/api/health`, `/api/auth/refresh`.
  - In this document the middleware does **authentication only** — it does not check roles or permissions (that is 1.3.2).
- `src/app/(dashboard)/layout.tsx` (from 1.1.2) is updated to:
  - Call `getServerSession()` (a server helper exposed in `src/lib/auth.ts` or `src/lib/session.ts`).
  - If no session, call `redirect('/login?callbackUrl=...')`. This is defence-in-depth: middleware may not cover every edge case (e.g., server actions), so the layout enforces the gate at the React Server Component level.
- `src/app/(dashboard)/_components/dashboard-top-bar.tsx` (or similar):
  - Displays the logged-in user's name (from `session.user.name`).
  - Includes a "Logout" button that calls the NextAuth `signOut()` action and redirects to `/login`.
- `src/lib/auth.ts` (or a new `src/lib/session.ts`) exposes a `getServerSession()` typed helper that returns the session with `userId`, `roles`, `permissions` strongly typed.
- A loading state: while `signIn()` is in flight, the submit button shows a spinner and is disabled; while the dashboard layout is checking the session, a brief loading boundary (Next.js `loading.tsx`) prevents a flash of the unauthenticated UI.
- The `/api/health` endpoint from 1.1.2 returns 200 (and is now documented as not requiring auth).

**Master Plan sections implemented:** Section 6.4 (login card layout — using shadcn Card), Section 6.2 (colour tokens applied to the login form), the unauthenticated-route portions of Section 4.2 (session-aware layouts), and the security perimeter portions of Section 15 (cookie attributes, CSRF posture).

## 5. Document Dependency Map

```
1.1.3  Prisma Schema & Initial Migration     (must be complete — User, RefreshToken tables exist)
   │
   ▼
1.2.1  NextAuth & JWT Strategy
   │
   ├──────────────────────────────┐
   ▼                              ▼
1.2.2  Refresh Token System    1.2.3  Login UI & Protected Routes
   │                              │
   └──────────────────────────────┘
                  │
                  ▼
        (Sub-Phase 1.3 RBAC builds on this)
```

**Build order:** `1.2.1` must be completed first. `1.2.2` and `1.2.3` can be developed in parallel after `1.2.1` is complete.

**Why this order:**

- **1.2.1 first.** Both 1.2.2 and 1.2.3 depend on the NextAuth configuration object and the session helpers being in place. The login UI (1.2.3) cannot post to the credentials provider until 1.2.1 has registered it. The refresh endpoint (1.2.2) calls into the same `auth.ts` configuration for token signing.
- **1.2.2 and 1.2.3 in parallel.** They touch different files: 1.2.2 is server-side utilities and API routes; 1.2.3 is UI and middleware. They communicate through the `getServerSession()` helper and the `auth.ts` exports from 1.2.1, but they don't share files.
- **Both before 1.3.x.** Sub-Phase 1.3's RBAC enforcement layer (1.3.2) reads `session.user.permissions` from every request. The middleware from 1.2.3 sets up the route-protection plumbing that 1.3.2 extends with role checks.

**What depends on this sub-phase:**

- **1.3.1 (RBAC Seeding)** does not depend on this sub-phase directly — it only depends on 1.1.3. The seed creates the data that the JWT enrichment callback (1.2.1) reads from.
- **1.3.2 (RBAC Middleware)** depends on 1.2.1 (for the session type with permissions) and 1.3.1 (for permission data to exist in the DB).
- **1.3.3 (User Management UI)** depends on 1.2.3 (for the dashboard layout, the login session, and the auth-aware middleware pattern) and on 1.3.2 (for `withPermission()` guards used by the user API endpoints).
- **All of Phase 2+** assumes `getServerSession()` returns a typed session with `userId`, `roles`, and `permissions` on every authenticated request.

## 6. Cross-Document Standards

These standards apply to every document in Sub-Phase 1.2. They are conventions established now so that Sub-Phase 1.3 and Phase 2+ inherit a consistent authentication layer.

### 6.1 Folder and File Naming

| Item | Convention | Example |
|---|---|---|
| Multi-word folders | kebab-case | `auth-utils/`, `mobile-login/` |
| Component files | PascalCase | `LoginForm.tsx`, `DashboardTopBar.tsx` |
| Non-component modules | camelCase | `auth.ts`, `auth-utils.ts`, `session.ts` |
| Route handlers | `route.ts` (Next.js App Router convention) | `src/app/api/auth/refresh/route.ts` |
| Type augmentation files | kebab-case with `.d.ts` suffix | `next-auth.d.ts` |

### 6.2 Import Conventions

- All project-internal imports use the `@/` alias.
- External package imports are listed first (alphabetically), then internal imports grouped by `@/components`, `@/lib`, `@/hooks`, `@/types`, `@/schemas`.
- For NextAuth v5, imports come from `next-auth` (types) and `next-auth/react` (client helpers). For the server-only auth helpers, use `import { auth } from '@/lib/auth'` — never import `next-auth/react` from a server module.

### 6.3 TypeScript Standards

- All exported functions and components have explicit type annotations.
- No `any` types in production code. Use `unknown` and narrow with type guards where the type cannot be statically determined.
- **Module augmentation for NextAuth types is mandatory.** Create `src/types/next-auth.d.ts` to augment the `Session` and `JWT` interfaces with `userId`, `roles`, and `permissions`. This makes the session strongly typed everywhere it is consumed.
- All API request and response types live in `src/types/api.types.ts` or a domain-specific `.types.ts` file (e.g., `src/types/auth.types.ts`).
- Discriminated unions are used for auth state (e.g., `{ status: 'loading' } | { status: 'authenticated', session: Session } | { status: 'unauthenticated' }`).

### 6.4 Response Envelope Standard

Every API route in this sub-phase must return the master plan's standard envelope (Section 9.1):

- Success: `{ success: true, data: <payload>, meta?: <pagination> }`
- Error: `{ success: false, error: { code, message, details? } }`

And must use the master plan's standard error codes (Section 9.2):

| Code | HTTP Status | When |
|---|---|---|
| `UNAUTHORIZED` | 401 | No valid authentication token, expired refresh token, invalid credentials |
| `FORBIDDEN` | 403 | Authenticated but lacks permission (this sub-phase rarely returns 403 — RBAC enforcement in 1.3.2 does) |
| `VALIDATION_ERROR` | 422 | Login form body fails Zod validation |
| `RATE_LIMITED` | 429 | Too many requests (rate limiting is Phase 5 — this code is reserved) |
| `INTERNAL_ERROR` | 500 | Unexpected server error |

### 6.5 Validation Standards

- The login form uses a Zod schema (`schemas/auth.schema.ts`) on the client.
- The mobile login endpoint validates its request body with the same Zod schema on the server (re-imported).
- The refresh endpoint validates its input (cookie presence, body shape for mobile) before doing any database work.
- Validation errors return `VALIDATION_ERROR` (HTTP 422) with the Zod error details in `error.details`.

### 6.6 Error Handling Standards

- Server actions and API routes return the master plan's standard envelope.
- Client-side error display uses shadcn `Alert` (variant destructive) for inline form errors and shadcn `Toast` for transient errors (toasts are set up in 1.2.3 for future use).
- Every API route is wrapped in a try/catch that logs the error (in development) and returns `INTERNAL_ERROR` (in production) for unhandled exceptions. The log includes the request path and a sanitized error message.

### 6.7 Environment Variable Standards

- All environment variables are documented in `.env.example` (committed) with placeholder values and inline comments.
- All variable names use SCREAMING_SNAKE_CASE.
- New variables for this sub-phase:
  - `NEXTAUTH_SECRET` — long random string used to sign JWTs. Required in production. Must be generated with `openssl rand -base64 32` or equivalent.
  - `NEXTAUTH_URL` (or `AUTH_URL` if using the v5 canonical name) — base URL of the application (e.g., `http://localhost:3000` in dev). Optional in v5 but recommended for explicit clarity.
  - `REFRESH_TOKEN_EXPIRY_DAYS` — defaults to `7`. Documented as overrideable for testing.
- All variable access goes through `src/lib/env.ts` (from 1.1.1), which validates required variables at startup.

### 6.8 Cookie Standards

- The refresh token cookie is named `refresh-token` (or `__Secure-refresh-token` in production when `Secure` is on).
- Attributes in production: `HttpOnly`, `Secure`, `SameSite=Strict`, `Path=/`, `Max-Age=<7 days in seconds>`.
- Attributes in development: `HttpOnly`, `Secure=false` (because localhost is HTTP), `SameSite=Strict`, `Path=/`, `Max-Age=<7 days in seconds>`.
- The configuration is environment-aware (reads `process.env.NODE_ENV` or `APP_ENV`).

### 6.9 Commit Standards

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `style:`, `test:`.
- Each task plan document corresponds to one or more commits. A single commit should not span multiple task plan documents.
- Suggested commit boundaries:
  - End of 1.2.1: `feat: configure NextAuth v5 with credentials provider and JWT enrichment`
  - End of 1.2.2: `feat: add refresh token rotation and mobile auth endpoints`
  - End of 1.2.3: `feat: add login page, auth middleware, and session-aware dashboard layout`

## 7. Sub-Phase Exit Criteria

Sub-Phase 1.2 is complete when **all** of the following are true:

### 7.1 Authentication

- [ ] NextAuth v5 is configured with the credentials provider and the Prisma adapter.
- [ ] A user can submit email + password to the login page and receive a session.
- [ ] The session contains `userId`, `roles`, and `permissions` (verifiable via the debug route from 1.2.1 or via `getServerSession()`).
- [ ] The session is signed and verifiable (changing `NEXTAUTH_SECRET` invalidates the session).
- [ ] The access token has a 15-minute expiry.

### 7.2 Refresh Tokens

- [ ] Successful web sign-in creates a `RefreshToken` record in the database.
- [ ] The refresh token is stored hashed (SHA-256), not in plaintext.
- [ ] The refresh token is set as an `HttpOnly`, `Secure` (in production), `SameSite=Strict` cookie.
- [ ] `POST /api/auth/refresh` accepts a valid refresh token, rotates it, and returns a new access token + a new refresh token.
- [ ] After rotation, the previous refresh token is marked as `isRevoked = true` in the database.
- [ ] A revoked or expired refresh token returns `UNAUTHORIZED` (HTTP 401) and does not issue a new access token.
- [ ] The refresh token has a 7-day expiry.

### 7.3 Mobile Authentication

- [ ] `POST /api/auth/mobile/login` accepts `{ email, password }` and returns `{ success: true, data: { accessToken, refreshToken, expiresIn } }`.
- [ ] Invalid credentials return `UNAUTHORIZED` (HTTP 401) with the standard error envelope.
- [ ] The mobile refresh flow (sending the refresh token in the request body) works and rotates the token.
- [ ] The mobile flow does not set any cookies (the refresh token is returned in the JSON body only).

### 7.4 Authentication UI

- [ ] The login page renders at `/login` with a centred card layout using shadcn components.
- [ ] Form validation works (empty fields, invalid email format, short password all show errors).
- [ ] Submitting valid credentials logs the user in and redirects to `/` (or `callbackUrl` if present).
- [ ] Submitting invalid credentials shows an error alert.
- [ ] While the sign-in request is in flight, the submit button is disabled and shows a spinner.

### 7.5 Route Protection

- [ ] An unauthenticated user requesting `/` is redirected to `/login?callbackUrl=/`.
- [ ] An unauthenticated user requesting `/(dashboard)/...`, `/(officer)/...`, `/users`, `/settings`, `/audit-log`, or any protected path is redirected to `/login`.
- [ ] The redirect preserves the original path in the `callbackUrl` query parameter.
- [ ] Unauthenticated routes (`/login`, `/display/*`, `/kiosk/*`, `/security/*`, `/api/auth/*`, `/api/sse/*`, `/api/tickets/issue`, `/api/health`) are NOT redirected.
- [ ] After successful login, the user is redirected to the `callbackUrl` if present, or `/` otherwise.
- [ ] Logging out clears the session and returns the user to `/login`.
- [ ] Defence-in-depth: `/(dashboard)/layout.tsx` calls `getServerSession()` and redirects to `/login` if no session is present.

### 7.6 Standards and Quality

- [ ] `yarn lint` passes with zero errors and zero warnings.
- [ ] `yarn type-check` passes with zero errors.
- [ ] `yarn build` produces a successful production build.
- [ ] All API routes return the master plan's standard response envelope.
- [ ] No `any` types exist in the auth-related modules (`lib/auth.ts`, `lib/auth-utils.ts`, `lib/session.ts`, all `app/api/auth/**` routes, `middleware.ts`, `types/next-auth.d.ts`).
- [ ] All environment variables are documented in `.env.example`.

## 8. AI Agent Implementation Guidance

This section is for the AI agents that will implement the task plan documents. It is guidance that applies across all three task plan documents in this sub-phase.

### 8.1 How to Use the Task Plan Documents

Each task plan document follows the same internal structure:

1. **Scope** — what this document owns and what it does not.
2. **Pre-requisites** — what must already be true (previous documents completed).
3. **Step-by-step implementation plan** — the actual work, in order.
4. **File-level specifications** — for every file the agent will create or modify, a description of its purpose and contents (no code, but enough to know exactly what to write).
5. **Verification checklist** — how the agent proves the work is done.
6. **Commit instructions** — what commit(s) to make at the end.
7. **Common pitfalls** — mistakes to avoid.

The agent should follow the steps in order. Each step lists the **goal** of the step, the **files affected**, and any **specific instructions** that resolve the ambiguity the agent would otherwise have to guess about.

### 8.2 General Guidance

- **Do not skip steps.** Sub-Phase 1.2 is mostly sequential because each step's output is the next step's input. Skipping forces backtracking.
- **Do not add features beyond the document's scope.** The temptation to "just add the permission guard now" or "just add the user management UI now" is real — resist it. Those are owned by Sub-Phase 1.3.
- **Verify after every step.** The verification checklist at the end of each document is a hard gate.
- **When the master plan is ambiguous, choose the simpler option.** Sub-Phase 1.2 is about establishing identity and session continuity, not optimising. If a configuration choice has two reasonable interpretations, pick the one that is easier to change later.
- **Commit at the end of each document** using the commit message specified in the document.
- **Document any deviation.** If the agent makes a choice that differs from this document, it must note the deviation in the final summary so the human reviewer can approve or reject it.
- **Always use the master plan's response envelope.** Every API route created in this sub-phase must return `{ success, data | error }` — not raw objects, not custom shapes.

### 8.3 What the AI Agent Should NOT Do

- Should not seed roles, permissions, or users with substantive content. 1.2.1 reads from these tables but does not populate them. The seed is 1.3.1.
- Should not implement any RBAC enforcement (`withPermission()`, `<Can>`, role-based middleware). That is 1.3.2.
- Should not implement any user management UI. That is 1.3.3.
- Should not implement any business logic (queue, counter, ticket, display, notification). All of that is Phase 2+.
- Should not configure rate limiting, security headers, or production hardening. That is Phase 5.
- Should not add PostgreSQL configuration. The database is SQLite for development.
- Should not implement OAuth / social login. The master plan provisions the `Account` table for future OAuth providers, but this sub-phase uses only the credentials provider.
- Should not implement password reset or forgot-password flows. Those are future.
- Should not use database sessions. The master plan specifies JWT session strategy.
- Should not return the refresh token in the JSON response of a web sign-in (only the mobile login endpoint returns it in the body — web sets it as an HttpOnly cookie).

### 8.4 When to Ask the Human

The AI agent should ask the human reviewer (not guess) if any of the following situations arise:

- The master plan and the phase 1 overview contradict each other. The master plan wins, but the agent should confirm.
- A required package version is unavailable or has a known breaking change that affects the approach (e.g., NextAuth v5 API changed since this document was written).
- The `RefreshToken` model in the master plan's Section 8.2 has fields that the agent is unsure about (e.g., `replacedByToken` — confirm if it should be a relation).
- The verification step fails for a reason that the document does not anticipate.
- The agent is unsure whether the cookie name should be `refresh-token` or `__Secure-refresh-token` in production (default to `__Secure-refresh-token` when `Secure` is on, but document the choice).

In all other cases, the agent should make the simplest reasonable choice, document it, and proceed.
