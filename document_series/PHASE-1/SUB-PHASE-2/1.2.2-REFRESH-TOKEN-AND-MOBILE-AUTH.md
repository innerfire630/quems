## 1. Document Purpose

This document implements the **refresh token system** and the **mobile authentication endpoints**. It builds on the NextAuth configuration from document 1.2.1 by:

- Implementing the refresh-token utility functions (`generate`, `hash`, `validate`, `revoke`, `rotate`, and `issueAccessToken`).
- Creating the `/api/auth/refresh` endpoint that accepts a refresh token (from cookie or body), validates it, rotates it, and returns a new access token.
- Creating the `/api/auth/mobile/login` endpoint that accepts credentials in the JSON body and returns the access token, refresh token, and expiry in the response body (so mobile clients that cannot use cookies can use the same auth model).
- Wiring the `signIn` callback in `src/lib/auth.ts` (from 1.2.1) to create a refresh token on successful web login.
- Defining the cookie configuration for the web flow.

When this document is complete, the dual-token system is operational: the access token expires after 15 minutes, and the refresh token can be used (once, because of rotation) to obtain a new access token for 7 days after the original login.

### Step 3: Implement the Refresh Token Utilities in `src/lib/auth-utils.ts`

**Goal:** Add the full set of refresh-token utility functions.

**Files affected:** `src/lib/auth-utils.ts` (the file started in 1.2.1).

**Specific instructions:**

- Open `src/lib/auth-utils.ts` and add (alongside the existing `verifyCredentials` function) the following exports:
  - **`generateRefreshToken(): string`**
    - Uses Node's `crypto.randomBytes(48)` (or equivalent) to produce a 48-byte random buffer.
    - Converts it to a URL-safe base64 string (or hex string — the agent's choice; base64-url is recommended).
    - Returns the plaintext token. This value is given to the client exactly once and is never stored in plaintext.
  - **`hashRefreshToken(token: string): string`**
    - Uses Node's `crypto.createHash('sha256')` to compute the SHA-256 hash of the token.
    - Returns the hex digest (or base64 — be consistent).
    - The hash is what is stored in `RefreshToken.tokenHash`.
  - **`createRefreshTokenForUser(userId: string): Promise<{ token: string; expiresAt: Date }>`**
    - Generates a plaintext token via `generateRefreshToken`.
    - Hashes it via `hashRefreshToken`.
    - Computes `expiresAt` as `now + REFRESH_TOKEN_EXPIRY_DAYS * 24 * 60 * 60 * 1000` (or uses `Date.now()` math; whichever is cleaner).
    - Creates a `RefreshToken` record with `{ tokenHash, userId, expiresAt, isRevoked: false }`.
    - Returns `{ token: plaintextToken, expiresAt }` so the caller can give the plaintext token to the client and store the hash in the DB.
    - This function replaces the placeholder stub from 1.2.1.
  - **`validateRefreshToken(plaintextToken: string): Promise<RefreshTokenRecord | null>`**
    - Hashes the incoming token via `hashRefreshToken`.
    - Looks up the hash in `RefreshToken` by `tokenHash`.
    - If no record found, returns `null`.
    - If the record's `isRevoked === true`, returns `null`.
    - If `expiresAt <= now`, returns `null`.
    - Otherwise, returns the record (including the associated `user` object via Prisma's `include`).
  - **`revokeRefreshToken(tokenId: string, replacedByTokenId?: string): Promise<void>`**
    - Updates the `RefreshToken` record by `id`, setting `isRevoked: true`, `revokedAt: new Date()`, and (if `replacedByTokenId` is provided) `replacedByToken: replacedByTokenId`.
  - **`rotateRefreshToken(plaintextToken: string): Promise<{ newToken: string; newExpiresAt: Date; user: UserRecord } | null>`**
    - Validates the token via `validateRefreshToken`. If null, returns null (caller should respond with `UNAUTHORIZED`).
    - Creates a new refresh token for the same user via `createRefreshTokenForUser`.
    - Revokes the old token via `revokeRefreshToken`, passing the new token's id as `replacedByToken`.
    - Returns the new plaintext token, the new expiry, and the user record.
  - **`issueAccessToken(user: UserWithRolesAndPermissions): { accessToken: string; expiresIn: number }`**
    - Uses NextAuth's `encode` function (from `next-auth/jwt`) OR the agent's chosen JWT library to sign a new JWT.
    - The payload contains: `sub: user.id`, `userId: user.id`, `email: user.email`, `name: user.name`, `roles: user.roles`, `permissions: user.permissions`, plus standard `iat` and `exp` claims.
    - `exp` is set to `now + 15 * 60` (15 minutes).
    - Returns the signed JWT and `expiresIn: 900` (15 minutes in seconds).
- All functions are server-only and async (they touch the database).
- Add inline documentation comments explaining the security rationale (e.g., why SHA-256 and not bcrypt for refresh tokens — refresh tokens are already cryptographically random, they do not need a slow hash).

### Step 5: Create `/api/auth/refresh` Endpoint

**Goal:** Expose the refresh endpoint that handles both web (cookie) and mobile (body) flows.

**Files affected:** `src/app/api/auth/refresh/route.ts` (new file).

**Specific instructions:**

- Create a `POST` handler.
- **Determine the source of the refresh token:**
  - First, check the JSON request body for `{ refreshToken: string }`. If present, this is a mobile request.
  - If absent, read the cookie (via Next.js's `cookies()` helper from `next/headers`) using the cookie name from Step 4. If present, this is a web request.
  - If neither is present, return `UNAUTHORIZED` (HTTP 401) with the error envelope.
- **Validate the token** via `validateRefreshToken` from `src/lib/auth-utils.ts`. If null, return `UNAUTHORIZED`.
- **Rotate the token** via `rotateRefreshToken`. If returns null (validation failed during rotation), return `UNAUTHORIZED`.
- **Issue a new access token** via `issueAccessToken`. The `user` returned by `rotateRefreshToken` must have its roles and permissions loaded (the function should `include` them in the Prisma query).
- **Set the new refresh cookie** (web only — check whether the original request was from cookie or body):
  - If web: call `cookies().set(...)` with the cookie name and options from Step 4, and the new plaintext token as the value.
  - If mobile: do NOT set a cookie.
- **Return the response** in the master plan's envelope:
  - For web: `{ success: true, data: { accessToken, expiresIn } }`. The refresh cookie is in the `Set-Cookie` header, not the body.
  - For mobile: `{ success: true, data: { accessToken, refreshToken, expiresIn } }`. The new refresh token is in the body.
- **Error responses:**
  - Missing token: `UNAUTHORIZED` (401).
  - Invalid/expired/revoked token: `UNAUTHORIZED` (401).
  - Internal error: `INTERNAL_ERROR` (500) with the standard envelope.
- The handler must be wrapped in a try/catch that logs (in development) and returns `INTERNAL_ERROR` for unhandled exceptions.
- Document the dual-mode behaviour (web vs mobile) clearly in comments at the top of the file.

### Step 6: Create `/api/auth/mobile/login` Endpoint

**Goal:** Provide a credentials login endpoint for mobile clients that returns the tokens in the JSON body.

**Files affected:** `src/app/api/auth/mobile/login/route.ts` (new file).

**Specific instructions:**

- Create a `POST` handler.
- **Parse the request body** as JSON. Expect `{ email: string, password: string }`.
- **Validate the body** with a Zod schema (the agent should create `src/schemas/auth.schema.ts` here if it doesn't exist yet — note that the same schema is used by the login UI in 1.2.3, so the agent should create it now and 1.2.3 will reuse it).
  - The schema: `z.object({ email: z.string().email(), password: z.string().min(1) })` (allow empty password for the schema check; length check is enforced server-side via `verifyCredentials` which uses bcrypt).
- **If validation fails**, return `VALIDATION_ERROR` (HTTP 422) with the Zod error details in `error.details`.
- **Verify credentials** via `verifyCredentials` from `@/lib/auth-utils.ts`. If null, return `UNAUTHORIZED` (HTTP 401) with the message "Invalid email or password" (do not leak whether the email exists).
- **Fetch the user's roles and permissions** from the database (the same query as the `jwt` callback in 1.2.1).
- **Issue an access token** via `issueAccessToken`.
- **Create a refresh token** via `createRefreshTokenForUser`.
- **Return the response** in the master plan's envelope:
  - `{ success: true, data: { accessToken, refreshToken, expiresIn: 900 } }`.
- **Do NOT set any cookies.** Mobile clients store the tokens in their secure storage (Android Keystore / EncryptedSharedPreferences).
- **Error responses:**
  - Bad body: `VALIDATION_ERROR` (422).
  - Invalid credentials: `UNAUTHORIZED` (401).
  - Internal error: `INTERNAL_ERROR` (500).
- The handler must be wrapped in a try/catch with logging in development.

## 8. Common Pitfalls

### 8.1 Storing Refresh Tokens in Plaintext

The `RefreshToken.tokenHash` field must store the SHA-256 hash, never the plaintext. The plaintext is given to the client exactly once and never stored server-side. If an attacker dumps the database, they should not be able to use the refresh tokens.

### 8.2 Using bcrypt for Refresh Token Hashing

Refresh tokens are already cryptographically random (48 bytes of `crypto.randomBytes`). They do not need the slow hashing that bcrypt provides. SHA-256 is the correct choice — it is fast enough not to become a DoS vector and is collision-resistant for this use case.

### 8.3 Returning the Refresh Token in the Web Sign-In Response Body

The web flow returns the refresh token via an HttpOnly cookie, NOT via the JSON body. Only the mobile flow returns it in the JSON body (because mobile apps cannot set HttpOnly cookies). The agent must distinguish between the two flows based on the request source (cookie vs body).

### 8.4 Not Rotating the Refresh Token

If the refresh endpoint accepts an old refresh token without revoking it, an attacker who steals the token can keep using it. The endpoint MUST rotate: revoke the old, create a new, link them via `replacedByToken`.

### 8.5 Setting `Secure: true` on Cookies in Development

The `Secure` flag requires HTTPS. Setting it to `true` in development (where the app runs on HTTP localhost) will cause the browser to silently drop the cookie, and the user will appear to be perpetually unauthenticated. The cookie configuration must be environment-aware.

### 8.6 Setting `Secure: false` in Production

The opposite mistake is equally bad — setting `Secure: false` in production exposes the refresh token to network attackers. The configuration must check `process.env.NODE_ENV === 'production'`.

### 8.7 Forgetting the `__Secure-` Cookie Name Prefix

Modern browsers require the `__Secure-` prefix on cookies that have the `Secure` flag. Setting the cookie as `Secure: true` but named `refresh-token` (without the prefix) will cause browsers to silently reject it. The agent must use `__Secure-refresh-token` in production.

### 8.8 Returning Detailed Errors That Leak User Existence

The mobile login endpoint must NOT say "User not found" vs "Wrong password" — both must return the same generic `UNAUTHORIZED` message. Leaking user existence is an enumeration vulnerability.

### 8.9 Catching All Errors and Returning 200

The endpoints must return the correct HTTP status codes (401 for auth failures, 422 for validation, 500 for internal). A try/catch that swallows errors and returns 200 with `{ success: false }` breaks HTTP semantics and makes monitoring/debugging harder.

## 9. AI Agent Guidance

### 9.1 How to Approach This Document

The agent should treat this document as **two API endpoints + one set of utilities**. The utilities are pure functions that can be unit-tested in isolation (if tests are added later); the endpoints are thin wrappers that orchestrate the utilities and the database.

The verification is end-to-end: the agent must prove that the full cycle works — login → access token → token expires → refresh → new access token → old token revoked.

### 9.2 What the Agent Should NOT Do

- Should not modify the JWT enrichment logic from 1.2.1.
- Should not add any UI in this document.
- Should not implement RBAC enforcement.
- Should not seed users (use the test user from 1.2.1 or insert one manually for verification).
- Should not use bcrypt for refresh token hashing (use SHA-256).
- Should not store refresh tokens in plaintext.
- Should not return different error messages for "user not found" vs "wrong password".
- Should not set cookies from the mobile login endpoint.

### 9.3 Decision Points

| Decision                                                         | Recommendation                                                      | Rationale                                                               |
| ---------------------------------------------------------------- | ------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| Where to set the refresh cookie in the web flow                  | Server action called after `signIn` succeeds (Approach A in Step 8) | Cleaner separation; doesn't depend on NextAuth's callback context.      |
| Hash format (hex vs base64)                                      | Hex                                                                 | Easier to inspect in the database; both are equally secure for SHA-256. |
| Token encoding (hex vs base64url)                                | base64url                                                           | URL-safe; useful for future use in URL parameters.                      |
| Whether to verify the `replacedByToken` chain for audit purposes | Not in this document — just set the field                           | Phase 5 (audit log viewer) reads the chain.                             |
| Cookie name with or without `__Secure-` prefix                   | With prefix in production, without in development                   | Required by browsers for `Secure` cookies.                              |
| Where to define `getServerSession`                               | Re-export `auth` from `auth.ts`                                     | v5 pattern; less code.                                                  |

### 9.4 When to Stop and Ask the Human

The agent should stop and ask if:

- The `RefreshToken` model in the schema has fields with names that differ from the agent's assumptions (e.g., `token_hash` instead of `tokenHash`).
- The master plan specifies a token format (e.g., JWT refresh token vs opaque token) that conflicts with this document's approach.
- The NextAuth v5 `cookies()` helper is not available in the `signIn` callback context (in which case Approach A is mandatory and the agent should confirm this is acceptable).
- The verification step fails in a way that suggests the database, the JWT signing, or the cookie configuration is fundamentally wrong (rather than a small bug to fix).
