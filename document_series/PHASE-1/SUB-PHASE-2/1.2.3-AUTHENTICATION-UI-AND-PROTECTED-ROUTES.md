## 1. Document Purpose

This document builds the **user-facing login experience** and the **route protection layer**. It takes the NextAuth configuration from 1.2.1 and the refresh-token system from 1.2.2 and produces:

- A polished login page at `/login` with a centred card layout, form validation, loading state, and error display.
- A `middleware.ts` file that intercepts every request and redirects unauthenticated users to `/login?callbackUrl=...`.
- A session-aware `(dashboard)/layout.tsx` that performs a server-side session check as defence-in-depth.
- A dashboard top bar showing the logged-in user's name and a logout button.
- The `getServerSession()` server helper (or its re-export) is documented and used consistently.
- The `app/api/health` route is confirmed to require no authentication.

When this document is complete, the application has a complete authentication experience: the user can log in via a styled form, is redirected to the dashboard, navigates protected routes without re-authenticating, and is redirected to the login page if their session expires. Logging out returns them to the login page.

**Goal:** Build a reusable client component for the login form.

  - The card content contains:

    - The email field: `Label` "Email" + `Input` (type email, required, autofocus).
    - The password field: `Label` "Password" + `Input` (type password, required).
  - The card footer contains:
    - The submit `Button` (type submit, full width, disabled when `isSubmitting`).
    - When `isSubmitting`, the button shows a spinner icon + "Signing in..." text. When idle, it shows "Sign in".
- **Accessibility:**
  - All form fields have associated labels.
  - The error alert has `role="alert"` (shadcn `Alert` may already include this — verify).
  - The submit button has appropriate focus management.
- The component does NOT include the card's outer layout (centering, background) — that is the responsibility of the login page (Step 3).

### Step 5: Create the Next.js Middleware for Route Protection

**Goal:** Intercept every request and redirect unauthenticated users to the login page.

**Files affected:** `src/middleware.ts` (new file at the project root — NOT inside `src/app/`).

**Specific instructions:**

- Create `src/middleware.ts` at the project root, **outside** the `src/` directory (Next.js convention: `middleware.ts` must be at the same level as `src/` or in `src/` if using the `src/` layout; verify against the project's setup). The agent should verify where Next.js expects `middleware.ts` based on whether the project uses the `src/` directory convention. If `src/` is used, `middleware.ts` should be at `src/middleware.ts`.
- The file exports:
  - A `default` async function `middleware(request)` that:
    - Reads the NextAuth session cookie from the request. The cookie name is `next-auth.session-token` in development and `__Secure-next-auth.session-token` in production (NextAuth handles this naming internally, but the agent may need to know it).
    - If the session cookie is present, allows the request to proceed (`NextResponse.next()`).
    - If the session cookie is absent AND the request is for a protected route, redirect to `/login?callbackUrl=<original-path>`.
    - If the session cookie is absent AND the request is for an unprotected route, allows the request to proceed.
  - A `config` object with a `matcher` regex array:
    - **Protected (matched and checked):** `/`, `/(dashboard)/:path*`, `/(officer)/:path*`, `/users/:path*`, `/settings/:path*`, `/audit-log/:path*`.
      - Note: The matcher runs on every request matching the pattern. The middleware function decides whether to redirect.
    - **Excluded from the matcher** (so they are never checked):
      - `/login`
      - `/display/:path*`
      - `/kiosk/:path*`
      - `/security/:path*`
      - `/api/auth/:path*` (NextAuth's own routes — never redirect)
      - `/api/sse/:path*` (real-time — never redirect)
      - `/api/tickets/issue` (kiosk ticket issuance — never redirect)
      - `/api/health` (health check — never redirect)
      - Static assets (`_next/static/*`, `_next/image/*`, `favicon.ico`, images, etc.) — use a `negative lookahead` or list them in the matcher exclusions.
- **Important caveat:** The middleware does NOT verify the JWT's signature — it only checks for the cookie's presence. Signature verification happens in the API routes and server components. This is the standard NextAuth middleware pattern. The agent should document this as a known limitation: a forged cookie with the right name would pass the middleware check but fail the server-side session verification.
- The middleware does **not** perform role checks (that is 1.3.2).
- **Edge runtime caveat:** Middleware runs on the Edge runtime by default, which does NOT have access to the Prisma client. The middleware cannot call `verifyCredentials` or `validateRefreshToken`. It can only inspect cookies. This is by design.

## 8. Common Pitfalls

### 8.1 Open-Redirect Vulnerability in `callbackUrl`

If the agent accepts any `callbackUrl` value (including absolute URLs), an attacker can craft a `/login?callbackUrl=https://evil.com` link. After the user logs in, they are redirected to the attacker's site. The mitigation is to validate that `callbackUrl` is a relative path (starts with `/` and is not `//`).

### 8.2 Forgetting the Edge Runtime Caveat in Middleware

Next.js middleware runs on the Edge runtime by default, which does NOT have access to Prisma, Node's `crypto`, or other Node-only APIs. The middleware can only inspect cookies and headers. The agent must not call `verifyCredentials`, `validateRefreshToken`, or any Prisma function from the middleware — it only checks for the session cookie's presence.

### 8.3 Not Excluding NextAuth's Own Routes from the Middleware

If `/api/auth/*` is included in the protected routes, the NextAuth login flow itself is blocked. The matcher must exclude `/api/auth/*` (and other auth-related paths).

### 8.4 Not Excluding Static Assets

Including `_next/static/*` and `favicon.ico` in the protected routes causes a redirect on every static asset request, which is slow and may cause the application to break. The matcher should exclude static asset paths.

### 8.5 Middleware Redirecting the Login Page to Itself

If the matcher includes `/login`, the user is in an infinite redirect loop: visit `/login` → no session → redirect to `/login` → ... The matcher must exclude `/login`.

### 8.6 Using `useEffect` for Session Check in the Login Page

The login page should be a server component for the initial render, with the form as a client component. Using `useEffect` for session checking in the client creates a flash of the login form before the redirect happens.

### 8.7 Forgetting the Open Redirect on the Mobile Login Endpoint

The mobile login endpoint from 1.2.2 does not use `callbackUrl` (mobile clients do not navigate via browser redirects), so this document's open-redirect mitigation does not apply there. But the web login page must validate `callbackUrl` carefully.

### 8.8 Not Redirecting Already-Authenticated Users from `/login`

If a logged-in user visits `/login`, they see the login form again. The `(auth)/layout.tsx` (or the page itself) should redirect them to `/`.

### 8.9 Storing the Refresh Token in Client-Accessible Storage (Web)

The refresh token cookie is `HttpOnly`, so client-side JavaScript cannot read it. The agent should not attempt to read or store the refresh token in `localStorage` or `sessionStorage` on the web — this would be a downgrade in security.

### 8.10 Using the Refresh Token as the Access Token

The refresh token is NOT an access token. It cannot be sent as `Authorization: Bearer <refreshToken>`. Only the access token is valid for API authentication. The agent should document this clearly.

### 8.11 Removing the Refresh Cookie After Logout

NextAuth's `signOut()` clears the session cookie but does NOT clear the refresh cookie (the refresh cookie is set by the application, not by NextAuth). The agent must also clear the refresh cookie on logout. This is a small but important step — if the refresh cookie is not cleared, a user who logs out and then logs in as a different user on the same browser may inherit the old refresh token.

- **Mitigation:** The logout server action (or the `signIn` callback's cookie setting logic) should also delete the refresh cookie. The agent should add a `cookies().delete(refreshCookieName)` call in the logout flow.

## 9. AI Agent Guidance

### 9.1 How to Approach This Document

The agent should treat this document as **three orthogonal tasks**:

1. **The login page (Steps 2–4).** Pure UI work using shadcn components. Form validation using the existing Zod schema from 1.2.2.
2. **The middleware (Step 5).** A small, declarative file at the project root. The matcher regex is the most important part — getting it wrong causes infinite redirects or broken static assets.
3. **The dashboard layout (Steps 6–9).** Defence-in-depth session check + sidebar/top bar structure + logout flow.

The verification is end-to-end in a browser: log in, navigate, log out.

### 9.2 What the Agent Should NOT Do

- Should not modify the NextAuth configuration from 1.2.1.
- Should not modify the refresh token mechanics from 1.2.2.
- Should not add any RBAC enforcement in the middleware (that is 1.3.2).
- Should not add any business domain UI (queue, ticket, counter — those are Phase 2+).
- Should not implement password reset or forgot password.
- Should not implement social / OAuth login buttons.
- Should not implement "Remember me".
- Should not use database sessions.

### 9.3 Decision Points

| Decision                                                            | Recommendation                                           | Rationale                                                                  |
| ------------------------------------------------------------------- | -------------------------------------------------------- | -------------------------------------------------------------------------- |
| Where to place `middleware.ts`                                      | At the project root (same level as `src/`)               | Next.js convention for projects using the `src/` directory.                |
| How to detect the current path in the middleware                    | Use the request's `nextUrl.pathname`                     | Standard Next.js middleware pattern.                                       |
| How to validate `callbackUrl`                                       | Reject if not starting with `/` or starting with `//`    | Prevents open-redirect attacks.                                            |
| Whether to redirect from `(auth)/layout.tsx` or from the login page | From `(auth)/layout.tsx`                                 | Cleaner — every route in the auth group benefits.                          |
| Loading skeleton vs no skeleton                                     | Include the skeletons                                    | Improves perceived performance; prevents flash of unauthenticated content. |
| Client-side logout vs server-side logout                            | Both: client for the button, server action for forms     | Covers all use cases.                                                      |
| Where to clear the refresh cookie on logout                         | In the logout server action and in a client-side handler | Belt-and-suspenders; ensures the cookie is always cleared.                 |

### 9.4 When to Stop and Ask the Human

The agent should stop and ask if:

- The middleware's matcher pattern does not behave as expected (e.g., the project uses a non-standard directory layout that changes where `middleware.ts` must live).
- NextAuth v5's middleware helpers (`auth()` from middleware) are needed and the agent is unsure how to integrate them with the project's chosen cookie strategy.
- The `signIn` client helper's API has changed in a way that breaks the login form.
- The user wants a different login page layout (e.g., split-screen with branding) — this is a design decision that the master plan does not specify precisely.
