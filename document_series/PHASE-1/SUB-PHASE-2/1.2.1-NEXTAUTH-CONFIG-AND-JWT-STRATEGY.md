## 1. Document Purpose

This document configures **NextAuth.js v5** for the project. It establishes:

- The NextAuth v5 configuration object with the credentials provider and the Prisma adapter.
- The JWT session strategy (not database sessions) with a 15-minute access-token expiry.
- The `signIn`, `jwt`, and `session` callbacks that enrich the token and the session with `userId`, `email`, `name`, `roles`, and `permissions`.
- The NextAuth route handler at `app/api/auth/[...nextauth]/route.ts`.
- The TypeScript module augmentation so the session and JWT are strongly typed everywhere.
- A temporary debug route (development-only) that returns the current session shape so the enrichment can be verified.

When this document is complete, a user who submits valid credentials to NextAuth receives a signed JWT session that contains their identity and effective permissions. Documents 1.2.2 (refresh tokens) and 1.2.3 (login UI) build on top of this.

### Step 3: Create the NextAuth Type Augmentation File

**Goal:** Make the NextAuth `Session` and `JWT` types carry the enriched payload (`userId`, `roles`, `permissions`) with full type safety.

**Files affected:** `src/types/next-auth.d.ts` (new file), `src/types/index.ts` (may need re-export).

**Specific instructions:**

- Create a `.d.ts` file at `src/types/next-auth.d.ts` (the file uses module augmentation and contains no runtime code).
- Augment the `next-auth` module:
  - Extend the `Session` interface to add `user: { id, userId, email, name, image?, roles, permissions }`.
  - Extend the `User` interface to add `userId`, `roles`, `permissions`.
  - Extend the `JWT` interface to add `userId`, `email`, `name`, `roles`, `permissions`.
- The `roles` field is `string[]` (array of role names like `"SUPER_ADMIN"`, `"ADMIN"`).
- The `permissions` field is `string[]` (flattened array of permission strings like `"user:manage"`, `"counter:read"`).
- The `userId` field is `string` (the user's primary key — same as `id` but named explicitly to avoid confusion with NextAuth's `id`).
- Make sure `tsconfig.json` includes `src/types/**/*.d.ts` in its `include` array. The default Next.js `tsconfig.json` already includes `**/*.d.ts` and `**/*.ts` under `src/`, so this should work automatically — but the agent should verify.
- The augmentation should be picked up automatically by TypeScript; if it is not, the agent should check the `tsconfig.json` `include` and `typeRoots` settings.

### Step 4: Create the Credentials Provider Authorization Logic

**Goal:** Implement the email + password verification function used by the credentials provider.

**Files affected:** `src/lib/auth-utils.ts` (new file — note: this file is shared with 1.2.2; in this document only the `verifyCredentials` function is created, the refresh-token functions are added in 1.2.2).

**Specific instructions:**

- Create `src/lib/auth-utils.ts` as a non-React module that exports authentication-related utilities.
- Export a function `verifyCredentials(email: string, password: string)`:
  - Looks up the user in the `User` table by email (use `db.user.findUnique({ where: { email } })`).
  - If no user is found, returns `null`.
  - If the user's `status` is `INACTIVE` or `SUSPENDED` (the `UserStatus` enum from 1.1.3), returns `null` (treat as invalid credentials — do not leak whether the email exists).
  - Uses bcrypt compare to verify the password against `user.password`.
  - Returns the user object (without the password field) on success, `null` on failure.
- The function is server-only — it must not be importable from client components. The agent should verify by checking that no "use client" directive appears in this file.
- Document that the password field on the `User` model is hashed with bcrypt and the seed (in 1.3.1) is responsible for ensuring the default super-admin's password is hashed correctly.

### Step 5: Create `src/lib/auth.ts` — The NextAuth Configuration Object

**Goal:** Wire up NextAuth with the credentials provider, Prisma adapter, JWT strategy, and the enrichment callbacks.

**Files affected:** `src/lib/auth.ts` (new file).

**Specific instructions:**

This is the largest step. The agent should create `src/lib/auth.ts` containing:

- **Imports:** The NextAuth core, the Prisma adapter, the Prisma client (`@/lib/db`), the `verifyCredentials` function from `@/lib/auth-utils`, the env helpers from `@/lib/env`, the type augmentation file (auto-imported), and any other needed imports.
- **The `authOptions` object** that is passed to `NextAuth()`:
  - `adapter`: the Prisma adapter instantiated with the Prisma client.
  - `session`: `{ strategy: 'jwt', maxAge: 15 * 60 }` (15 minutes in seconds). Document why 15 minutes (master plan Section 10.1).
  - `jwt`: `{ maxAge: 15 * 60 }`.
  - `secret`: read from `process.env.NEXTAUTH_SECRET` via the env helper.
  - `pages`: optional `{ signIn: '/login' }` to direct NextAuth to the custom login page (this is wired in 1.2.3, but the agent may set the value here if certain).
  - `providers`: an array containing the credentials provider:
    - `name: 'credentials'`.
    - `credentials` schema: `{ email: { type: 'email' }, password: { type: 'password' } }`.
    - `authorize` function: calls `verifyCredentials`, returns the user object on success or `null` on failure.
  - `callbacks`:
    - `signIn`: this is invoked on successful credential verification. For this document, the callback should:
      - On successful web sign-in, call a function `createRefreshTokenForUser(user.id)` (which is exported from `@/lib/auth-utils.ts` but is a stub in this document — the agent creates the function as a placeholder that returns `null` or throws "not implemented", with a `// TODO: implemented in 1.2.2` comment, OR the agent can leave the callback hook empty and document that 1.2.2 will populate it).
      - Return `true` to allow sign-in.
      - **Recommended approach:** in 1.2.1, the `signIn` callback is set up to call the stub function so the wiring is in place, but the stub itself returns `undefined` (so no refresh token is created yet — login still works, just without a refresh cookie). Document clearly in the file comments that 1.2.2 fills this in.
    - `jwt`: this is invoked on every JWT issuance and on every request that reads the JWT.
      - If `token.userId` is already set (subsequent calls), return the token unchanged (do not re-fetch roles/permissions).
      - On the initial call (when `token.userId` is not set), look up the user by `token.sub` (NextAuth puts the user id in `sub` by default). Fetch the user's roles (via the join table) and flatten the role-to-permission mappings into a `string[]`. Set `token.userId`, `token.email`, `token.name`, `token.roles`, `token.permissions`.
      - **Note:** The fetch must respect the `UserRole` and `RolePermission` join tables from Master Plan Section 8.2. The query should use Prisma's `include` to traverse `UserRole → Role → RolePermission → Permission`.
    - `session`: this shapes the session object exposed to the client.
      - Set `session.user.id = token.userId`.
      - Set `session.user.userId = token.userId`.
      - Set `session.user.email = token.email`.
      - Set `session.user.name = token.name`.
      - Set `session.user.roles = token.roles`.
      - Set `session.user.permissions = token.permissions`.
      - Return the modified session.
- Export the NextAuth handler as `const handler = NextAuth(authOptions)` and the auth options as `export { authOptions, handler }` (or the v5 pattern `export const { handlers, auth, signIn, signOut } = NextAuth(authOptions)`).
- The agent should follow the NextAuth v5 export conventions — `handlers` is used by the route handler in Step 6, `auth` is used as the server-side session reader, `signIn` and `signOut` are used by server actions.

## 8. Common Pitfalls

The following mistakes are common when configuring NextAuth v5 and are worth avoiding:

### 8.1 Using Database Sessions Instead of JWT Sessions

The master plan explicitly requires JWT sessions (`session.strategy = 'jwt'`). Database sessions would break the custom token model and the mobile flow. Always set the strategy to `'jwt'`.

### 8.2 Forgetting the TypeScript Module Augmentation

If `src/types/next-auth.d.ts` is not created or is not picked up by `tsconfig.json`, then `session.user.userId`, `session.user.roles`, and `session.user.permissions` will be typed as `any` (or will be missing entirely, causing type errors). The augmentation must be in place **before** any code that consumes the session is written.

### 8.3 Calling the Credentials Provider from the Client Directly

The credentials provider's `authorize` function must only run on the server. Do not import it from a client component. The login UI in 1.2.3 uses the `signIn('credentials', { ... })` client helper from `next-auth/react`, which internally posts to the NextAuth API route — the agent does not call `authorize` from the client.

### 8.4 Storing the Password in Plaintext in the Database

This is a pre-requisite issue (the seed from 1.3.1 must bcrypt-hash passwords), but the agent should verify that any test user created in Step 8 has a properly hashed password. If the password is stored in plaintext, `bcrypt.compare` will always fail.

### 8.5 Wrong Session maxAge

The master plan's Section 10.1 specifies a 15-minute access-token expiry. Setting `maxAge` to a different value (e.g., 30 days, the default for many NextAuth setups) would silently break the security model. The agent must explicitly set both `session.maxAge` and `jwt.maxAge` to `15 * 60` seconds.

### 8.6 Re-fetching Roles and Permissions on Every JWT Call

The `jwt` callback runs on every request that touches the JWT. If the agent naively re-fetches roles and permissions on every call, it will cause a database query per request. The pattern must be: fetch on initial sign-in only (when `token.userId` is not yet set), and return the token unchanged on subsequent calls. Role updates only take effect on the next full sign-in — this is a known limitation documented in the file's comments.

### 8.7 Returning the Password Field from `verifyCredentials`

The user object returned from `authorize` (and from `verifyCredentials`) must not include the password. The Prisma query should either omit the password field or the function should destructure it out before returning.

### 8.8 Using `as any` to Bypass Type Errors

The TypeScript augmentation is the correct solution. If the agent finds themselves reaching for `as any` to silence a type error, it usually means the augmentation file is missing or not being picked up.

- Should not use database sessions.
- Should not return the password from `verifyCredentials`.
- Should not store passwords in plaintext.
- Should not use `as any` to bypass type errors.

### 9.3 Decision Points

The agent may need to make the following decisions. The recommendation is given for each:

| Decision | Recommendation | Rationale |
|---|---|---|
| `bcrypt` vs `bcryptjs` | `bcryptjs` | No native build, works on all platforms without a C compiler. Performance is acceptable for this use case. |
| NextAuth v5 install command | `yarn add next-auth@beta` | NextAuth v5 is currently on the beta channel. Pin the version after install. |
| Token refresh strategy (on JWT expiry) | Client-side: client detects expiry and calls `/api/auth/refresh` | Simpler than silent refresh; acceptable because the access token is short. |
| Test user creation | A small one-off script that inserts a user with a known email + bcrypt-hashed password + a role + a permission | Easier to repeat than `prisma studio`; the script is deleted after verification. |
| Whether to expose `signIn` / `signOut` from `auth.ts` | Yes, export them per the v5 pattern | Used by server actions in 1.2.3 and 1.3.3. |

### 9.4 When to Stop and Ask the Human

The agent should stop and ask the human if:

- The NextAuth v5 export pattern has changed in a way that makes the step instructions inapplicable.
- The Prisma schema's `User`, `UserRole`, `Role`, `RolePermission`, `Permission` models do not match the assumed relations.
- The master plan specifies a token field that is not covered here (e.g., a custom `sessionVersion` claim for forced logout).
- `yarn build` fails with a TypeScript error in `src/types/next-auth.d.ts` that cannot be resolved by adjusting the augmentation.
