## 1. Sub-Phase Purpose

Sub-Phase 4.3 is the **communication loop closer** of the Smart Queue Management System. It introduces the officer reply flow, the broadcast injection system that converts replies into display + security messages, and the security officer screen that receives those messages. This sub-phase completes the end-to-end communication chain that begins in Sub-Phase 4.1 (push notifications to officers) and ends in Sub-Phase 4.3 (replies reaching the display board and the security officer).

The strategic goal of this sub-phase is to enable **officers to communicate context and request assistance from their assigned counter without leaving it**. When a counter officer receives a push notification about a new ticket (a customer waiting) or a recall (a customer who did not respond), the officer can send a free-form text reply from their mobile device. That reply is then:

1. Recorded in the system as a `NotificationReply` (4.3.1).
2. Converted into a `BroadcastMessage` (4.3.2) and emitted as a `BROADCAST_MESSAGE` SSE event.
3. Pushed to the display board's `<BroadcastBanner>` slot (already reserved by 3.2.1) so waiting customers and on-site staff can see it.
4. Pushed to the security officer screen (4.3.3) via the `security` SSE channel so security personnel can act on officer requests.

When this sub-phase is complete:

- An officer can submit a text reply to any push notification they have received, via `POST /api/notifications/[notificationId]/reply`.
- The reply is validated (1-500 characters, non-empty), persisted in the `NotificationReply` table with the correct authorization and timestamps.
- The reply is converted to a `BroadcastMessage` (with sender display name, optional target display board, default 10-second display duration, default 5-minute expiry).
- A `BROADCAST_MESSAGE` SSE event is emitted on the `global` and `security` channels, with payload `{ broadcastId, message, senderName, displaySeconds }`.
- The display board's `<BroadcastBanner>` overlay renders the message for `displaySeconds`, then auto-dismisses. Multiple simultaneous broadcasts rotate (newest replaces current). Expired or inactive broadcasts are not displayed.
- The security officer screen at `/security` shows the new broadcast in a chronological feed within one second of the reply, with sender name, message text, timestamp, and an unread indicator.
- Only users with the `SECURITY_OFFICER` role can access the security screen. Non-security users are redirected or denied (403).
- An `OFFICER_REPLY` SSE event is also emitted on the `security` channel to give the security screen context that a reply was just sent (in addition to the `BROADCAST_MESSAGE` event that arrives moments later).

Three problems are solved in this sub-phase, in order:

1. **The officer reply API and message flow** — the `POST /api/notifications/[notificationId]/reply` endpoint, the `NotificationReply` record creation, the Zod validation (1-500 chars, non-empty, authorized reply), the `lib/notification-reply.ts` centralised module, the error classes (`NotificationNotFoundError`, `NotAuthorizedToReplyError`), the audit log write, and the post-commit invocation of the broadcast creation (4.3.2). (Document 4.3.1.)
2. **The display board message injection** — the conversion of a `NotificationReply` to a `BroadcastMessage` in `lib/broadcast.ts`, the `BROADCAST_MESSAGE` SSE event emission on the `['global', 'security']` channels, the wiring of the display board's existing `<BroadcastBanner>` slot (reserved by 3.2.1) to render the broadcast with proper positioning, animation, and dismissal logic, the multiple simultaneous broadcast handling (rotation strategy), the expiry logic on the client, and the display board's reducer (3.2.3) is updated additively to handle the new payload variant. (Document 4.3.2.)
3. **The security officer screen and broadcast receiver** — the `src/app/security/page.tsx` page, the `src/app/security/layout.tsx` minimal layout, the server-side data loader in `lib/security-dashboard.ts` for initial broadcasts, the client-side `<SecurityScreenClient>` component that owns the SSE subscription to the `security` channel, the `<BroadcastFeed>` component that renders the chronological feed, the `<UnreadIndicator>` logic (localStorage-backed), the `SECURITY_OFFICER` role check at both the page and SSE channel levels, and the empty/loading state handling. (Document 4.3.3.)

When this sub-phase is complete, **Phase 4 is finished**. The system is operationally complete for officers and ready for the analytics, hardening, and production-readiness work in Phase 5. The future Android app has a complete API surface for replies and broadcasts.

## 1.1 Why This Sub-Phase Closes Phase 4

Sub-Phase 4.3 is the final sub-phase of Phase 4. It completes the operational picture:

- Officers can receive notifications (4.1).
- Officers can manage their counter and notification state (4.2).
- Officers can reply to notifications, and the replies reach the right people (4.3).
- The display board and security officer screen become communication surfaces, not just status displays.

After Sub-Phase 4.3, the system is **operationally complete** for officers and ready for the analytics and hardening layer in Phase 5. The end-to-end event consumption chain from Phase 3 is now fully wired: notifications (FCM) → officer mobile app → officer reply → SSE broadcast → display board overlay + security officer screen.

## 1.2 Why This Sub-Phase Comes After 4.1 and 4.2

Sub-Phase 4.3 depends on 4.1 and 4.2 in the following ways:

- **4.3.1 (officer reply) requires 4.1.3:** the `Notification` table populated by 4.1.3's dispatch logic is the source of truth for "which notifications can be replied to". The reply is keyed off `notificationId`, and the authorization check verifies that the calling user owns the recipient `CounterOfficer` profile that 4.1.3 stored against the `Notification` row. The reply is the inverse direction of the dispatch.
- **4.3.2 (broadcast injection) requires 4.3.1:** the `BroadcastMessage` is created in response to a `NotificationReply`. The reply's `id` is stored on the broadcast as `sourceReplyId`. 4.3.2's broadcast creation is invoked as a post-commit step by 4.3.1's endpoint, NOT by 4.3.1's endpoint directly (separation of concerns — 4.3.1 knows nothing about broadcasts).
- **4.3.3 (security screen) requires 4.3.2:** the `BROADCAST_MESSAGE` SSE event is emitted by 4.3.2's broadcast creation. The security screen subscribes to the `security` channel and consumes the event.
- **4.3.3 (security screen) requires 3.1.1 + 3.1.2:** the `useSSE()` hook (3.1.2) and the `/api/sse/[channel]` route handler (3.1.1) are the consumption and transport layers.
- **4.3.3 (security screen) requires 3.1.1's `authorizeSseChannel`:** the `security` channel requires the calling user to have the `SECURITY_OFFICER` role. This is enforced server-side by the existing channel auth helper.
- **4.3.2 (broadcast) requires 3.2.1's `<BroadcastBanner>` slot:** the slot was reserved by 3.2.1 (the slot is in the layout, but no wiring was done). 4.3.2 wires the `BROADCAST_MESSAGE` SSE event consumption into the existing slot.
- **4.3.2 (broadcast) requires 3.2.3's display state reducer:** the `applyBROADCAST_MESSAGE` function in `src/lib/display-state.ts` (created by 3.2.3) already handles the broadcast event shape. 4.3.2 does not modify the reducer — the reducer was forward-designed by 3.2.3 to consume this exact event.

## 1.3 What This Sub-Phase Is NOT

- It is **not** the Android app. No Android code is written. The API surface is fully ready for the future app, but the app itself is not built in this phase.
- It is **not** the display board implementation. The display board (3.2) and its reducer (3.2.3) already exist. 4.3.2 only wires the broadcast event into the existing `<BroadcastBanner>` slot reserved by 3.2.1 — it does not implement the display board.
- It is **not** the SSE manager. The SSE manager (3.1.1) and `useSSE()` hook (3.1.2) already exist. 4.3 only consumes them.
- It is **not** the officer dashboard. The officer dashboard (4.2.3) is unaffected. The 4.3 reply button (if added to the dashboard in 4.2.3) is a UI sugar on top of the 4.3.1 endpoint.
- It is **not** the notification toggle or counter closure toggles. Those are 4.2.1 and 4.2.2.
- It is **not** the FCM push pipeline. The FCM pipeline (4.1) is upstream; 4.3 is downstream of the reply that comes back from the mobile app.
- It is **not** the reports dashboard. The `NotificationReply` and `BroadcastMessage` tables are populated by 4.3, but no reporting UI exists yet (Phase 5.1).
- It is **not** rate limiting. Phase 5.2.1.
- It is **not** PostgreSQL migration. Phase 5.3.
- It is **not** the audit log viewer. Phase 5.2.3. The `NotificationReply` and `BroadcastMessage` events are written to `AuditLog` from 4.3.1 and 4.3.2 respectively, but the admin UI to view them is Phase 5.

## 2. Sub-Phase Scope and Boundaries

### 2.1 In Scope

| Area | What Sub-Phase 4.3 Delivers |
|---|---|
| `POST /api/notifications/[notificationId]/reply` endpoint | Authenticated endpoint. Accepts `{ message: string }`. Validates non-empty, 1-500 characters, authorized reply (the calling user must be the recipient `CounterOfficer` for the notification, OR the assigned officer for the counter the notification is about). Creates a `NotificationReply` record. Triggers broadcast creation (4.3.2) as a post-commit step. Returns the created `NotificationReply` with the associated `BroadcastMessage` (denormalised for the mobile app to show "Reply broadcast sent" confirmation). |
| `NotificationReply` record creation | Fields populated per master plan Section 8.2: `{ id, notificationId, counterOfficerId, message, isDisplayBroadcast, isSecurityBroadcast, broadcastAt, createdAt }`. The `isDisplayBroadcast` and `isSecurityBroadcast` flags default to `true` for all replies (every reply is broadcast). `broadcastAt` is set immediately after the broadcast is created. |
| `lib/notification-reply.ts` centralised module | Exports `createReply`, `getNotificationForOfficer`, `validateReplyMessage`, plus `NotificationNotFoundError` and `NotAuthorizedToReplyError` classes. The module is the single source of truth for reply operations. |
| `src/schemas/notification-reply.schema.ts` Zod schemas | `notificationReplyCreateSchema` — `{ message: string (1-500 chars, non-empty, trimmed) }`. `notificationIdParamSchema` — `notificationId: string.cuid()`. |
| Reply authorization rules | The calling user must be the `CounterOfficer` recipient (i.e., the `Notification.counterOfficId` matches a `CounterOfficer` record where `CounterOfficer.userId = session.userId`). Alternative authorization: the calling user is an assigned `CounterOfficer` for the counter the notification is about (i.e., they could have received the notification had they been the on-duty officer). The two rules are equivalent in practice (the recipient IS the assigned officer at the time of dispatch) but the document captures both for clarity. |
| Reply character limit | Minimum 1 character (after trim), maximum 500 characters (matches the master plan's `NotificationReply.message` field constraint). Empty strings, whitespace-only strings, and over-length strings are rejected with `VALIDATION_ERROR` (422). |
| `BROADCAST_MESSAGE` SSE event emission | Called as a post-commit step by 4.3.1's endpoint via 4.3.2's `lib/broadcast.ts`. The event is emitted on the `['global', 'security']` channels per the existing `EVENT_ROUTING` entry in `src/lib/events.ts` (set up by 3.1.3). Payload: `{ broadcastId, message, senderName, displaySeconds }` per master plan Section 11.4. |
| `OFFICER_REPLY` SSE event emission | The reply endpoint also emits an `OFFICER_REPLY` event on the `security` channel (in addition to `BROADCAST_MESSAGE`). This event has a minimal payload `{ notificationId, replyId, repliedByOfficerName, repliedAt }`. The security screen uses this to highlight the originating reply (in addition to the broadcast). |
| `BroadcastMessage` record creation | On every reply, a `BroadcastMessage` is created with: `{ id, message, senderOfficerId, senderDisplayName, sourceReplyId, targetDisplayBoardId (null for "all"), targetSecurityUserId (null for "all"), displayDurationSeconds (default 10), expiresAt (default now + 5 minutes), isActive (default true), createdAt }`. The `senderDisplayName` is denormalised at creation time (so a later officer rename does not affect old broadcasts). |
| Display board `<BroadcastBanner>` wiring | The reserved slot in the display board page (3.2.1) is wired up. The banner subscribes to `BROADCAST_MESSAGE` events on the `global` channel, displays the message for `displaySeconds`, auto-dismisses via a `useEffect` timer, supports manual dismissal (clicking the banner), and handles multiple simultaneous broadcasts via the rotation strategy. |
| Display board multiple-broadcast handling | Default implementation: the newest broadcast replaces the currently displayed one and shows for its full duration. Older broadcasts arriving while one is active are queued in client state and shown after the current one expires. Older broadcasts that are still active (not yet expired) at the moment a new one arrives are discarded (ephemeral by design). The security screen (4.3.3) is the authoritative receiver for the full broadcast history. |
| Display board expiry logic | On receiving a `BROADCAST_MESSAGE` event, the client checks `isActive === true` AND `expiresAt > now()`. If either is false, the broadcast is not displayed (silent skip). The server also does not emit events for inactive broadcasts (the 4.3.2 module checks `isActive` before calling `broadcastRoutedEvent`). |

| Security screen initial data load | On page load, the server fetches the most recent active `BroadcastMessage` records (last 50, ordered by `createdAt desc`, filtered by `isActive = true` AND `expiresAt > now()`). Passed as `initialData` to the client component. New events arriving via SSE are prepended to the local state. |
| Security screen broadcast feed | Chronological list (most-recent-first) of broadcasts. Each row shows: sender display name, message text, relative timestamp via `Intl.RelativeTimeFormat`. Empty state: "No broadcasts yet". |
| Unread indicator | The security screen tracks which broadcasts have been "viewed" (tapped or scrolled into view). Unread broadcasts show a colored dot. The read state is persisted in `localStorage` (key: `quems:security:broadcasts:read:<broadcastId>` for each, or a single JSON map). New broadcasts arriving while the security officer is not on the page remain unread when they return. No server-side read tracking in 4.3 (it would require a new `BroadcastMessageRead` table — deferred to a future phase). |

Dark theme using the `display-bg` and `display-text` color tokens (per master plan Section 6.2) for visual consistency with the display board. Top bar: "Security Officer" label, current time, user name, logout button. |

Metadata: `{ notificationId, replyId, message, broadcastId, senderOfficerId, senderDisplayName, expiresAt }` for `BROADCAST_MESSAGE_SENT`. `{ notificationId, replyId, counterOfficerId, messageLength, broadcastTriggered }` for `NOTIFICATION_REPLIED`. |

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to distribute the complexity across the three task documents. The split is:

- **4.3.1** owns the **officer reply API and message flow**. It introduces the `POST /api/notifications/[notificationId]/reply` endpoint, the `lib/notification-reply.ts` module, the `src/schemas/notification-reply.schema.ts` Zod schemas, the `NotificationReply` record creation, the reply authorization rules, the audit log write for replies, and the post-commit invocation of 4.3.2's `createBroadcastFromReply` helper. It does **not** know about the display board (3.2), the security screen (4.3.3), or the broadcast rendering — it just creates a `NotificationReply` and asks 4.3.2 to broadcast it.
- **4.3.2** owns the **display board message injection**. It introduces the `lib/broadcast.ts` module (centralised broadcast operations), the `BROADCAST_MESSAGE` SSE event emission (via `broadcastRoutedEvent`), the wiring of the display board's existing `<BroadcastBanner>` slot (from 3.2.1) to render the broadcast, the multiple simultaneous broadcast handling (rotation strategy), the expiry logic, and the `OFFICER_REPLY` event emission on the `security` channel. It does **not** know about the officer reply authorization (4.3.1 owns that) or the security screen (4.3.3 owns that).
- **4.3.3** owns the **security officer screen and broadcast receiver**. It introduces the `src/app/security/page.tsx` page, the `src/app/security/layout.tsx` minimal layout, the `lib/security-dashboard.ts` server-side data loader, the `<SecurityScreenClient>` client component that owns the SSE subscription, the `<BroadcastFeed>` component that renders the chronological list, the unread indicator logic, the `SECURITY_OFFICER` role check, and the empty/loading state handling. It does **not** know about the reply creation (4.3.1) or the broadcast emission (4.3.2) — it is a pure consumer of `BROADCAST_MESSAGE` and `OFFICER_REPLY` events on the `security` channel.

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 4.3.1 — Officer Reply API & Message Flow

**Owns:** The `POST /api/notifications/[notificationId]/reply` endpoint, the `src/lib/notification-reply.ts` centralised module, the `src/schemas/notification-reply.schema.ts` Zod schemas, the `NotificationReply` record creation, the reply authorization rules, the reply character validation, the `NOTIFICATION_REPLIED` audit log write, the post-commit invocation of 4.3.2's `createBroadcastFromReply` helper, the `OFFICER_REPLY` SSE event emission, and the route protection.

**Does NOT own:** The broadcast creation logic (owned by 4.3.2), the display board's broadcast rendering (owned by 4.3.2), the security officer screen (owned by 4.3.3), the FCM push pipeline (owned by 4.1), the dashboard composition (owned by 4.2.3), the notification toggle (owned by 4.2.2), the counter closure (owned by 4.2.1), the SSE manager (owned by 3.1).

**Key outputs:**

- `src/lib/notification-reply.ts` — the centralised reply module:
  - `async function createReply({ notificationId, counterOfficerId, message }): Promise<{ reply: NotificationReply, broadcast: BroadcastMessage }>` — creates the reply in a `prisma.$transaction` containing: (1) the authorization re-check inside the transaction (the `Notification.counterOfficerId` belongs to the calling user's `CounterOfficer` profile), (2) the `NotificationReply` record creation, (3) the `isDisplayBroadcast` / `isSecurityBroadcast` flag setting (always `true` for 4.3.1's replies). Returns both the reply and the broadcast that 4.3.2 creates. After the transaction commits, calls 4.3.2's `createBroadcastFromReply(reply, officerName)` to create the broadcast.
  - `async function getNotificationForOfficer(notificationId, counterOfficerId): Promise<Notification | null>` — fetches the notification with its `counterOfficer` relation joined. Returns `null` if not found. Used in the authorization check.
  - `function validateReplyMessage(message: string): { valid: true; trimmed: string } | { valid: false; reason: string }` — pure function that checks length (1-500 chars after trim) and emptiness. Returns a discriminated result.
  - `class NotificationNotFoundError extends Error` — thrown when the `notificationId` does not match any `Notification` record.
  - `class NotAuthorizedToReplyError extends Error` — thrown when the calling user is not the recipient of the notification.
- `src/app/api/notifications/[notificationId]/reply/route.ts` — the POST endpoint:
  - `POST` handler. Authenticated via `auth()` + `withPermission(handler, PERMISSION_NOTIFICATION_REPLY)`. The `notification:reply` permission is consumed here for the first time (defined as a system permission in Phase 1.3.1 but not used until 4.3.1).
  - Reads `notificationId` from the URL params. Validates via `notificationIdParamSchema` (cuid format).
  - Resolves the calling user's `CounterOfficer` profile (the officer must be an assigned officer for at least one counter; if not, returns 403). The authorization check ensures that the notification's `counterOfficerId` matches the calling user's `CounterOfficer` profile ID.
  - Validates the request body via `notificationReplyCreateSchema` from `src/schemas/notification-reply.schema.ts`. Body shape: `{ message: string (1-500 chars, non-empty, trimmed) }`.
  - Calls `createReply` from `src/lib/notification-reply.ts`.
  - After the transaction commits, calls 4.3.2's `createBroadcastFromReply(reply, officerName)` to create the `BroadcastMessage` and emit the `BROADCAST_MESSAGE` SSE event.
  - Also emits an `OFFICER_REPLY` SSE event on the `security` channel (informational — gives the security screen the originating reply context).
  - Writes an `AuditLog` entry with action `NOTIFICATION_REPLIED` AFTER the transaction commits (best-effort, per Phase 1.3.3 discipline).
  - Returns `{ success: true, data: { reply, broadcast } }`. The `broadcast` is included so the mobile app can show "Reply broadcast sent" confirmation with the broadcast ID.
- `src/schemas/notification-reply.schema.ts` — Zod schemas for request bodies and types:
  - `notificationReplyCreateSchema` — `{ message: string (min 1, max 500, trimmed) }`. The `trim()` is applied inside the schema (`.transform()` or `.refine()` to reject whitespace-only).
  - `notificationIdParamSchema` — `notificationId: string.cuid()` (the URL parameter).
  - Type exports: `NotificationReplyCreateInput`, `NotificationReplyWithBroadcast`.
- **`NotificationReply` record creation** — happens inside the transaction in `createReply`. Fields populated per master plan Section 8.2. The `isDisplayBroadcast` and `isSecurityBroadcast` flags default to `true` (every reply triggers a broadcast). The `broadcastAt` field is set AFTER the broadcast is created (post-commit step in 4.3.2).
- **Reply authorization rules** — two equivalent rules enforced at the route handler:
  - **Rule 1 (recipient-based):** the calling user has a `CounterOfficer` profile whose ID matches `Notification.counterOfficerId`. This is the standard case — the officer who received the push notification.
  - **Rule 2 (counter-based):** the calling user has a `CounterOfficer` profile assigned to the counter the notification is about. This handles edge cases where the original recipient has gone off-duty and another officer is now serving the counter.
  - The route handler uses Rule 1 primarily; the document specifies both for documentation purposes.
- **Validation rules** — the `message` field must:
  - Be present (required).
  - Be non-empty after `.trim()`.
  - Be at most 500 characters.
  - Be at least 1 character (after trim, this is implied by the non-empty check).
  - Not contain control characters (Phase 5.2.2 may add stricter validation; 4.3.1 keeps it minimal).
- **SSE event payload for `OFFICER_REPLY`** — added to the `SseEventPayload` discriminated union in `src/types/sse.types.ts`:
  - `notificationId: string`.
  - `replyId: string`.
  - `repliedByOfficerName: string` — denormalised from `User.name` for display.
  - `repliedAt: string` — ISO 8601 timestamp.
  - The payload does NOT include the message text (the message is in the `BROADCAST_MESSAGE` event that follows; `OFFICER_REPLY` is informational, telling the security screen "a reply just happened", with a link back to the `notificationId` for context).
- **`OFFICER_REPLY` routing** — added to `EVENT_ROUTING` in `src/lib/events.ts`:
  - `OFFICER_REPLY: ['security']` — the security screen is the only consumer.
  - The 3.1.3 routing for `OFFICER_REPLY` is currently `['security']` (forward-seam entry, documented as "future sub-phase will emit" — that sub-phase is 4.3.1).
- **Audit log writes** — `NOTIFICATION_REPLIED` is a new value added to the `AuditAction` union (additive extension, per Phase 1.3.3 discipline). The `NotificationReply` is the primary record; the `AuditLog` captures the high-level event for the admin viewer. Metadata: `{ notificationId, replyId, counterOfficerId, messageLength, broadcastId, broadcastTriggered: true }`. The `broadcastId` allows the admin to trace the reply to the broadcast it triggered.
- **No `broadcastEvent` calls from the reply endpoint that the 4.3.1 document directly owns.** The reply endpoint delegates broadcast creation to 4.3.2's `createBroadcastFromReply` helper. The 4.3.1 document does NOT call `broadcastRoutedEvent` or `broadcastEvent` directly — that is 4.3.2's responsibility. - **No `TicketEvent` writes from 4.3.1 paths** — a reply is not a ticket state change.
- **Route protection** — the `POST /api/notifications/[notificationId]/reply` endpoint is guarded with `withPermission(handler, PERMISSION_NOTIFICATION_REPLY)`. The ownership check (the calling user must be the recipient `CounterOfficer`) is enforced inside the handler, AFTER the permission check.

### 6.4 Reply Authorization Discipline (NEW — Phase 4.3 specific)

The `POST /api/notifications/[notificationId]/reply` endpoint enforces the following authorization:

- The calling user MUST have a valid session.
- The calling user MUST have the `notification:reply` permission (enforced by `withPermission`).
- The calling user MUST be a `CounterOfficer` (have at least one `CounterOfficer` profile).
- The notification's `counterOfficerId` MUST match the calling user's `CounterOfficer` profile ID.
- The notification's parent counter MUST still be active (defensive check — defensive but documented for clarity).

If any of these checks fail, the endpoint returns:
- 401 (UNAUTHORIZED) for missing session.
- 403 (FORBIDDEN) for missing permission or non-matching `CounterOfficerId`.
- 404 (NOT_FOUND) for non-existent notification.

The endpoint is INTENTIONALLY strict — only the original recipient can reply. There is no "reply on behalf of" or "delegate" mechanism in 4.3.1.

### 6.5 Reply Idempotency Discipline (NEW — Phase 4.3 specific)

The `POST /api/notifications/[notificationId]/reply` endpoint is NOT idempotent by design. Each call creates a new `NotificationReply` record and a new `BroadcastMessage`. This is intentional:

- The officer may want to send multiple replies in succession (e.g., "Need assistance" then "Urgent — system issue"). Each is a separate message.
- The mobile app's UI should not double-submit (use a local "submitting" state to prevent double-clicks), but the server does not enforce idempotency.

Future enhancements (out of scope for 4.3.1) may add an `Idempotency-Key` header for true idempotency, but Phase 4 keeps it simple.

### 6.6 Broadcast Lifecycle Discipline (NEW — Phase 4.3 specific)

`BroadcastMessage` records have a well-defined lifecycle:

- **Created** when a `NotificationReply` triggers a broadcast (4.3.2's `createBroadcastFromReply`).
- **Active** for `displayDurationSeconds` (default 10) on the display board.
- **Expired** when `expiresAt` is reached (default 5 minutes from creation).
- **Inactive** when `isActive = false` (set manually by an admin in the future, or automatically by the cleanup job in 4.3.2 if a `cleanupExpiredBroadcasts` function is added — Phase 5.2.2 may add this).

The discipline:

- The display board checks `isBroadcastVisible(broadcast)` before displaying. Expired or inactive messages are not shown.
- The security screen shows all active messages in its feed; expired messages are not shown but may be retained in the database for history.
- The database keeps the `BroadcastMessage` records for audit and historical purposes; only the display logic filters by expiry.
- The server (`emitBroadcastEvent`) checks `isBroadcastVisible` before calling `broadcastRoutedEvent`. Expired broadcasts are not emitted.

### 6.7 SSE Channel Filter Discipline (NEW — Phase 4.3 specific)

The security screen subscribes to a STRICT filter: `['BROADCAST_MESSAGE', 'OFFICER_REPLY']`. It does NOT subscribe to:

- Ticket lifecycle events (`TICKET_CALLED`, `TICKET_RECALLED`, `TICKET_NO_SHOW`, `TICKET_QUEUED`, `QUEUE_UPDATED`).
- Counter status events (`COUNTER_OPENED`, `COUNTER_CLOSED`, `COUNTER_STATUS_CHANGED`).
- Daily reset events (`DAILY_RESET`).
- Notification events (`NOTIFICATION_RECEIVED`).

The security officer is a focused observer — they care about officer communications (broadcasts and replies), not queue activity. The strict filter reduces noise and keeps the feed clean.

### 6.8 Audit Logging Standard (Sub-Phase 4.3 Specific)

- `NOTIFICATION_REPLIED` (4.3.1, every reply creation) — written to `AuditLog`.
- `BROADCAST_MESSAGE_SENT` (4.3.2, every broadcast creation) — written to `AuditLog`.
- The `NotificationReply` table is the detailed audit trail for reply creation (capturing the message, recipient, timestamp).
- The `BroadcastMessage` table is the detailed audit trail for broadcast creation (capturing the sender, denormalised name, expiry, source reply).
- The `AuditLog` captures the high-level events for the admin viewer.
- All audit log writes are AFTER the action commits, best-effort (per Phase 1.3.3 discipline).
- The security page's SSE event consumption does NOT write to `AuditLog` (it's read-only).

### 6.9 Broadcast Event Seam (Sub-Phase 4.3 Specific)

- 4.3.1 emits `OFFICER_REPLY` SSE event via `broadcastRoutedEvent` (from 3.1.3) on the `security` channel.
- 4.3.2 emits `BROADCAST_MESSAGE` SSE event via `broadcastRoutedEvent` (from 3.1.3) on the `['global', 'security']` channels.
- The `EVENT_ROUTING` entries are already in place from 3.1.3 (`BROADCAST_MESSAGE: ['global', 'security']`, `OFFICER_REPLY: ['security']`). 4.3 does not change the routing.
- 4.3.3 consumes SSE events on `security` via the `useSSE()` hook. Events consumed: `BROADCAST_MESSAGE`, `OFFICER_REPLY`.
- The display board (3.2) consumes `BROADCAST_MESSAGE` on the `global` channel. The existing reducer (3.2.3) handles this event — 4.3.2's wiring is the only change.

### 6.10 Transaction Boundaries

- 4.3.1's `createReply` is wrapped in a single `prisma.$transaction` containing: (1) the authorization re-check inside the transaction (the `Notification.counterOfficerId` matches the calling user), (2) the `NotificationReply` record creation. Atomic — either all succeed or none do.
- The broadcast creation (4.3.2) is OUTSIDE the reply transaction (post-commit step). If the broadcast creation fails, the reply is still committed. The `NotificationReply.broadcastAt` field is updated AFTER the broadcast is created (best-effort).
- The `OFFICER_REPLY` SSE event emission is OUTSIDE the transaction (after commit) — per the 3.1.3 broadcast seam discipline.
- The `AuditLog` write is AFTER the transaction commits (best-effort, per Phase 1.3.3 discipline).
- 4.3.2's `createBroadcastFromReply` is a single `create` call (atomic via Prisma's `create`). The SSE event emission is AFTER the record is created.
- 4.3.3's security page is read-only — no transactions. The SSE event consumption is read-only.

### 6.11 Environment Variable Standards

No new environment variables are introduced in 4.3. The existing env vars from Phase 1.1 are sufficient.

### 6.12 Rate Limiting

- The `POST /api/notifications/[notificationId]/reply` endpoint is NOT rate-limited in Phase 4.3. Phase 5.2.1 adds the rate limit (5 requests per minute per officer — replies are not high-frequency).
- The `GET /api/security/dashboard` endpoint (if 4.3.3 adds one for refreshes) is NOT rate-limited in Phase 4.3. Phase 5.2.1 adds the rate limit (30 requests per minute per security officer).
- The SSE channel itself is not rate-limited (channels are persistent connections, not request-based).

### 8.3 Common Pitfalls

- **DO NOT** add `withPermission` calls in 4.3.2 — 4.3.2 is server-side but does not have a user-facing endpoint. The audit log write uses the same actor as the calling reply.
- **DO NOT** modify the display board's reducer (3.2.3). The `applyBROADCAST_MESSAGE` function already handles the event. 4.3.2 only updates the `<BroadcastBanner>` component to actually render the state.
- **DO NOT** create a separate `POST /api/broadcasts` endpoint. The reply endpoint (4.3.1) is the only entry point. The broadcast creation is invoked as a post-commit step.
- **DO NOT** write to the `Notification` table in 4.3.1. The reply does not modify the original notification (no `readAt` update, no `status` change). The reply is a separate record.
- **DO NOT** enforce idempotency on the reply endpoint. Each call creates a new reply. The mobile app should debounce double-clicks at the UI level.
- **DO NOT** use `broadcastEvent` directly. Use `broadcastRoutedEvent` from 3.1.3 — the routing is set up in `EVENT_ROUTING` and `broadcastRoutedEvent` resolves the templates.
- **DO NOT** mix up the `security` and `global` channels. The display board consumes `BROADCAST_MESSAGE` on `global`; the security screen consumes on `security`. Both events are broadcast to both channels.
- **DO NOT** modify the SSE infrastructure (3.1). The manager, hook, and channel auth are unchanged.
- **DO NOT** add a new `BroadcastMessageRead` table for read tracking. The unread state is localStorage-only in Phase 4.3.

### 8.5 Questions to Ask Before Starting

If anything is unclear, ask before starting. Common questions:

- Should the reply endpoint be a server action (Next.js 14+ Server Actions) or an API route handler? → **API route handler** for consistency with the rest of the project (all Phase 2, 3, 4 endpoints are API route handlers).
- Should the broadcast message text be the reply text verbatim, or formatted? → **Verbatim** for Phase 4.3.1. Future enhancements may add formatting.
- Should the security screen show historical broadcasts (e.g., the last 24 hours regardless of active status) or only currently active ones? → **Only active** (`isActive = true` AND `expiresAt > now()`). The database keeps the full history for the reports phase.
- Should the read state be per-officer (so a shift-change security officer doesn't see the previous officer's read state) or per-browser? → **Per-browser (localStorage)** for Phase 4.3. Per-officer is future.
