## 1. Document Purpose

This document delivers the **officer reply API endpoint and message flow**. The endpoint accepts a free-text reply from a counter officer in response to a push notification they have received, persists the reply as a `NotificationReply` record, and triggers the broadcast creation (4.3.2) as a post-commit step. The endpoint also emits an informational `OFFICER_REPLY` SSE event on the `security` channel so the security officer screen (4.3.3) can show that a reply was just sent.

This document is the **first document in Sub-Phase 4.3** because the reply endpoint is the entry point for the entire reply-broadcast chain. Without 4.3.1, no `NotificationReply` records are created, no broadcasts are triggered, and 4.3.2/4.3.3 have nothing to consume.

The endpoint's surface is small and focused — one `POST` handler, one Zod schema, one centralised module, and one audit log write. The complexity is in the **transaction boundaries** and the **post-commit orchestration** (the reply must commit before the broadcast is created, but the broadcast must be created even if some unrelated side-effect fails).

When this document is complete:

- A counter officer (via the future Android app or any client with a valid session) can submit a text reply to a notification they have received.
- The reply is validated (1-500 characters, non-empty, trimmed).
- The reply is persisted in the `NotificationReply` table with the correct `notificationId`, `counterOfficerId`, and `message`.
- The reply triggers a `BroadcastMessage` creation (via 4.3.2's `createBroadcastFromReply`).
- An `OFFICER_REPLY` SSE event is emitted on the `security` channel.
- An `NOTIFICATION_REPLIED` audit log entry is written.
- Invalid messages return 422 `VALIDATION_ERROR`.
- Non-recipient users receive 403 `FORBIDDEN`.
- Non-existent notifications return 404 `NOT_FOUND`.

This document does **not** implement the broadcast creation logic (owned by 4.3.2), the display board's broadcast rendering (owned by 4.3.2), the security officer screen (owned by 4.3.3), the FCM push pipeline (owned by 4.1), the dashboard composition (owned by 4.2.3), the SSE manager (owned by 3.1), or the notification toggle (owned by 4.2.2).

### Step 1: Create `src/schemas/notification-reply.schema.ts` — Zod Schemas

**Goal:** Define the Zod schemas for the reply request body and URL parameter.

**Files affected:** `src/schemas/notification-reply.schema.ts` (new file). NOTE: The `notification.schema.ts` file already exists from 4.1.2 with `deviceRegisterSchema`, `deviceTokenIdParamSchema`, and `notificationIdParamSchema`. The agent has two choices:

- **Option A (preferred — separate file):** Create a new file `notification-reply.schema.ts` for the reply-specific schemas. Keeps the schemas focused per feature.
- **Option B (append to existing file):** Add the reply-specific schemas to the existing `notification.schema.ts`. Reduces file count.

The document specifies Option A for clarity. The existing `notificationIdParamSchema` from 4.1.2 can be reused, but the new `notificationIdParamSchema` in this file is the same shape — the agent may either import from the old file or redefine (with the same schema). Importing is preferred.

**Specific instructions:**

Create the file with the following exports:

- **`notificationReplyCreateSchema`** — the request body schema:
  - `message: z.string().trim().min(1, "Message cannot be empty").max(500, "Message must not exceed 500 characters")`.
  - The `.trim()` is applied as a transform — the trimmed value is the stored value. The mobile app may submit `"  Need help  "` and the server stores `"Need help"`.
  - Use `z.string().min(1)` AFTER `.trim()` — the `min(1)` is applied to the trimmed value. Zod's `.min(1)` after `.trim()` works because Zod applies the trim transform first, then validates length.
  - No additional fields. The endpoint is intentionally minimal — the reply is a single text write.
  - Export the inferred TypeScript type: `type NotificationReplyCreateInput = z.infer<typeof notificationReplyCreateSchema>`.
- **`notificationIdParamSchema`** — the URL parameter schema (re-exported from the existing `notification.schema.ts` for clarity, or redefined if the agent chooses):
  - `notificationId: z.string().cuid("Invalid notification ID")`.
  - Export the inferred type: `type NotificationIdParam = z.infer<typeof notificationIdParamSchema>`.
- **`notificationReplyResponseSchema`** — the response envelope schema (for documentation and client-side validation):
  - `success: z.literal(true)`.
  - `data: z.object({ reply: ..., broadcast: ... })` where `reply` and `broadcast` are the Prisma return types with the denormalised officer name.
  - Optional. The agent may skip this for simplicity — the response is not validated by the mobile app against this schema (the mobile app trusts the server's response).

### Step 2: Create `src/lib/notification-reply.ts` — Centralised Reply Module

**Goal:** Provide the centralised reply operations (the single source of truth for reply creation).

**Files affected:** `src/lib/notification-reply.ts` (new file).

**Specific instructions:**

Create the file with:

- **`async function createReply(input: CreateReplyInput): Promise<{ reply: NotificationReply, broadcast: BroadcastMessage }>`**:
  - The main entry point. Called by the route handler.
  - **Input shape:** `{ notificationId: string, counterOfficerId: string, message: string }`. The `counterOfficerId` is the calling user's `CounterOfficer` profile ID (resolved by the route handler from the session).
  - **Step 1 — Authorisation re-check inside the transaction:** uses `tx.notification.findUnique({ where: { id: notificationId }, include: { counterOfficer: { include: { user: true } } } })`. If null → throw `NotificationNotFoundError`. If `counterOfficerId !== input.counterOfficerId` → throw `NotAuthorizedToReplyError`. (TOCTOU safety: the check is inside the transaction, so a concurrent modification cannot bypass it.)
  - **Step 2 — `NotificationReply` creation:** creates a record with `{ notificationId, counterOfficerId, message: message.trim(), isDisplayBroadcast: true, isSecurityBroadcast: true, broadcastAt: null (set later), createdAt: now() }`. The `isDisplayBroadcast` and `isSecurityBroadcast` flags default to `true` for all replies.
  - **Step 3 — Return the reply:** returns the created `NotificationReply` (with relations included: `counterOfficer: { include: { user: true, counter: true } }`). The user's `name` is needed for the broadcast's `senderDisplayName`.
  - **Post-commit step (OUTSIDE the transaction):** after the transaction commits, calls `createBroadcastFromReply(reply, officerName)` from `@/lib/broadcast` (4.3.2's module) to create the broadcast. The function does NOT throw if the broadcast creation fails — it logs via `console.error` and returns a partial response with `broadcast: null`. The reply is still recorded; the broadcast failure is recoverable.
  - **Returns:** `{ reply, broadcast }` where `broadcast` may be `null` if the broadcast creation failed. The route handler responds with the reply and (if available) the broadcast.
  - **Outer try/catch:** the entire function is wrapped in a try/catch. Unexpected errors are logged via `console.error` and re-thrown (so the route handler can return 500).
- **`async function getNotificationForOfficer(notificationId: string, counterOfficerId: string): Promise<Notification | null>`**:
  - Fetches the notification with `counterOfficer: { include: { user: true } }` joined. Returns `null` if not found OR if the notification's `counterOfficerId` does not match the provided `counterOfficerId`. The function is used by the route handler for the OUTSIDE-transaction pre-check (fail fast before opening the transaction).
  - Note: the in-transaction check in `createReply` is the authoritative one (TOCTOU-safe). This function is for the route handler's pre-flight check.
- **`function validateReplyMessage(message: string): { valid: true, trimmed: string } | { valid: false, reason: 'EMPTY' | 'TOO_LONG' }`**:
  - Pure function. Trims the message. Returns the validation result.
  - Used in the route handler BEFORE the Zod validation (defence in depth — Zod also validates, but this allows the route handler to return a more specific error code).
  - Optional. The Zod schema in Step 1 handles this validation. The agent may skip this function if the Zod schema is sufficient.
- **Custom error classes (exported):**
  - `class NotificationNotFoundError extends Error` — `{ code: 'NOT_FOUND', message: 'Notification not found' }`. Thrown when the `notificationId` does not match any record.
  - `class NotAuthorizedToReplyError extends Error` — `{ code: 'FORBIDDEN', message: 'You are not authorized to reply to this notification' }`. Thrown when the calling user is not the recipient of the notification.
  - Both classes have a `code` property (a string literal) for the route handler to map to the correct HTTP status code and error envelope.
- **Type exports:**
  - `interface CreateReplyInput { notificationId: string; counterOfficerId: string; message: string }`.
  - `interface CreateReplyResult { reply: NotificationReply & { counterOfficer: { user: { name: string }, counter: Counter } }; broadcast: BroadcastMessage | null }`.

**Important note on the post-commit broadcast creation:**

The 4.3.1 document does NOT define the `createBroadcastFromReply` function — that is 4.3.2's responsibility. The 4.3.1 file imports it from `@/lib/broadcast` (the file that 4.3.2 creates). This means:

- The agent implementing 4.3.1 needs to ALSO create a stub for `src/lib/broadcast.ts` (with the `createBroadcastFromReply` function signature) BEFORE running the build, OR
- The agent should implement 4.3.1 and 4.3.2 together (this is the recommended approach — they are tightly coupled).

The recommended approach: create `src/lib/broadcast.ts` as a stub with the function signature but the body returning `null` (or a placeholder), then implement 4.3.1 to use the stub. Then implement 4.3.2 to fill in the stub. This avoids build failures during the iteration.

### Step 3: Create `src/app/api/notifications/[notificationId]/reply/route.ts` — The POST Endpoint

**Goal:** Expose the `POST /api/notifications/[notificationId]/reply` endpoint.

**Files affected:** `src/app/api/notifications/[notificationId]/reply/route.ts` (new directory and file).

**Specific instructions:**

Create the file with:

- **Route configuration exports:**
  - `export const runtime = 'nodejs'` (required for Prisma + auth).
  - `export const dynamic = 'force-dynamic'` (required — Next.js must not attempt to cache the response).
- **`async function POST(request: NextRequest, { params }: { params: { notificationId: string } }): Promise<NextResponse>`**:
  - **Step 1 — Authentication:** calls `auth()` from `@/lib/auth`. If no session → return 401 `UNAUTHORIZED`.
  - **Step 2 — URL param validation:** validates `params.notificationId` via `notificationIdParamSchema`. If invalid → return 422 `VALIDATION_ERROR`.
  - **Step 3 — Body parsing and validation:** parses the request body as JSON, then validates via `notificationReplyCreateSchema`. If invalid → return 422 `VALIDATION_ERROR`.
  - **Step 4 — Resolve calling officer's `CounterOfficer` profile:** queries `db.counterOfficer.findFirst({ where: { userId: session.user.userId }, orderBy: { createdAt: 'asc' } })`. Returns the FIRST `CounterOfficer` record for the user (defensive — most officers have one). If none found → return 403 `FORBIDDEN` with code `NO_COUNTER_OFFICER`.
  - **Step 5 — Permission check:** wraps the handler with `withPermission(handler, PERMISSION_NOTIFICATION_REPLY)`. The wrapper is applied at module top level (e.g., `export const POST = withPermission(handlePost, PERMISSION_NOTIFICATION_REPLY)`). The handler logic is in a separate function `handlePost` to allow the wrapping.
  - **Step 6 — Pre-flight notification check:** calls `getNotificationForOfficer(params.notificationId, counterOfficer.id)`. Returns `null` → 404 `NOT_FOUND`. Returns a notification but with wrong `counterOfficerId` → 403 `FORBIDDEN` with code `NOT_RECIPIENT`. (This pre-flight is fast — the in-transaction check in `createReply` is the authoritative one.)
  - **Step 7 — Create the reply:** calls `createReply({ notificationId: params.notificationId, counterOfficerId: counterOfficer.id, message: body.message })` from `@/lib/notification-reply`. The function returns `{ reply, broadcast }` (broadcast may be `null` if 4.3.2's module failed).
  - **Step 8 — Emit `OFFICER_REPLY` SSE event:** calls `broadcastRoutedEvent('OFFICER_REPLY', { notificationId: reply.notificationId, replyId: reply.id, repliedByOfficerName: counterOfficer.user.name, repliedAt: reply.createdAt.toISOString() }, {})`. The existing `EVENT_ROUTING` entry routes this to the `security` channel. Wrapped in a try/catch — failures are logged via `console.error` but do not fail the request.
  - **Step 9 — Write audit log:** calls `writeAuditLog({ action: 'NOTIFICATION_REPLIED', actorId: session.user.userId, description: 'Officer replied to notification', metadata: { notificationId, replyId: reply.id, counterOfficerId: counterOfficer.id, messageLength: reply.message.length, broadcastId: broadcast?.id ?? null, broadcastTriggered: broadcast !== null } })`. The `broadcastTriggered` flag allows the admin to identify replies where the broadcast creation failed (recoverable — the reply is recorded but the broadcast may need manual recovery).
  - **Step 10 — Return success:** returns `{ success: true, data: { reply: { id, notificationId, counterOfficerId, message, isDisplayBroadcast, isSecurityBroadcast, broadcastAt: null (will be set by 4.3.2), createdAt, counterOfficer: { user: { name }, counter: { id, name, number, displayLabel } } }, broadcast } }` with HTTP 201 Created. The 201 status code indicates the resource was created.
- **Error handling:** the function catches `NotificationNotFoundError` → 404, `NotAuthorizedToReplyError` → 403, Zod errors → 422, unexpected errors → 500 `INTERNAL_ERROR`. All errors use the standard response envelope from master plan Section 9.1.
- **Route protection:** the handler is wrapped with `withPermission(handler, PERMISSION_NOTIFICATION_REPLY)`. The ownership check (calling user must be the recipient `CounterOfficer`) is enforced inside the handler AFTER the permission check. This is a "defence in depth" pattern — the permission check verifies the user has the right to reply to notifications in general, the ownership check verifies they are replying to their own notification.

## 8. Common Pitfalls

- **DO NOT** call `broadcastRoutedEvent('BROADCAST_MESSAGE', ...)` from this document. The `BROADCAST_MESSAGE` event is emitted by 4.3.2's broadcast creation, not by the reply endpoint. The 4.3.1 endpoint only emits `OFFICER_REPLY`.
- **DO NOT** modify the `Notification` table in this document. The reply is a separate record. The notification's `readAt` is NOT updated (it remains a future enhancement when the Android app reports read events).
- **DO NOT** create a `BroadcastMessage` record directly in this document. The broadcast creation is delegated to 4.3.2's `createBroadcastFromReply` function. The 4.3.1 endpoint does not know about broadcast payload structure, display duration, expiry, or routing — those are 4.3.2's concerns.
- **DO NOT** write the `NotificationReply.broadcastAt` field inside the transaction. It is set AFTER the broadcast is created (in 4.3.2's post-commit step OR in a subsequent 4.3.1 follow-up call if the broadcast creation succeeded). For Phase 4.3.1, `broadcastAt` is set to `null` at creation time and updated by 4.3.2.
- **DO NOT** enforce idempotency on the endpoint. Each call creates a new reply. The mobile app is expected to debounce at the UI level.
- **DO NOT** add `Idempotency-Key` header handling. Not in scope for Phase 4.3.1.
- **DO NOT** validate the `notificationId` against the `Notification` table inside the Zod schema (Zod cannot do database lookups). The Zod schema only validates the format (cuid). The existence check happens in the route handler and the transaction.
- **DO NOT** skip the pre-flight `getNotificationForOfficer` check. It provides a fast failure path (no transaction opened for non-existent or unauthorised notifications). The in-transaction check is the authoritative one for TOCTOU safety.
- **DO NOT** use `findFirst` without `orderBy` for the calling officer's `CounterOfficer` profile. While most officers have one profile, the schema allows multiple. Use `orderBy: { createdAt: 'asc' }` to deterministically pick the oldest (most likely the primary assignment). Alternative: query ALL `CounterOfficer` records and check if any matches `Notification.counterOfficerId`. The document specifies the `orderBy: { createdAt: 'asc' }` approach for simplicity.
- **DO NOT** write to the audit log INSIDE the transaction. Audit log writes are AFTER the action commits (per Phase 1.3.3 discipline).
- **DO NOT** throw a generic `Error` from the centralised module. Use the custom error classes (`NotificationNotFoundError`, `NotAuthorizedToReplyError`). The route handler maps these to the correct HTTP status codes and error envelopes.

### 9.3 Common Questions

- **Q: Should the endpoint be a server action or an API route handler?**
  A: API route handler (consistent with all other endpoints in Phase 2, 3, 4).

- **Q: Should the reply endpoint be rate-limited?**
  A: Not in Phase 4.3.1. Phase 5.2.1 adds rate limiting (5 requests per minute per officer).

- **Q: Should the reply be visible in the officer dashboard's recent activity feed (4.2.3)?**
  A: Yes, automatically. The dashboard queries `TicketEvent` and `CounterStatusEvent` (per 4.2.3's data loader). A reply is not in either, so the dashboard does not show past replies. If a future enhancement wants to show "you replied at 10:30" in the dashboard, the 4.2.3 data loader is extended to query `NotificationReply` records. Phase 4.3.1 does not modify 4.2.3.

- **Q: Can the reply be edited or deleted after creation?**
  A: No. Phase 4.3.1 is create-only. The reply is a permanent record. Future enhancements may add edit/delete for the original replier within a time window, but not in scope.

- **Q: Can an officer reply to a notification that was sent days ago?**
  A: Yes. There is no time limit on replies. The notification is a historical record; the reply attaches to it regardless of age. (In practice, the officer will reply within minutes of receiving the push notification, but the data model allows older replies.)
