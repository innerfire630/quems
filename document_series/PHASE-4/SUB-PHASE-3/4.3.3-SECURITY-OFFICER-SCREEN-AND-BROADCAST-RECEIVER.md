## 1. Document Purpose

This document delivers the **security officer screen and broadcast receiver** — the dedicated full-screen view at `/security` that security officers use to monitor incoming broadcast messages from counter officers. The page subscribes to the `security` SSE channel (which carries `BROADCAST_MESSAGE` and `OFFICER_REPLY` events) and displays the broadcasts in a chronological feed with an unread indicator.

This document is the **third and final document in Sub-Phase 4.3**. It is a pure consumer of the events that 4.3.1 and 4.3.2 produce — it does not create broadcasts, does not emit SSE events, and does not modify any data. Its sole purpose is to provide a focused, always-on view for security personnel to monitor officer communications.

The strategic value of this document is that it **completes the Phase 4 communication loop**. When a counter officer sends a reply via 4.3.1, the broadcast appears on the display board (4.3.2) AND on the security officer's screen (4.3.3). The security officer can then act on the request — dispatch assistance, investigate an issue, or simply acknowledge receipt.

The security officer screen is conceptually similar to the display board (3.2) — full-screen, dark theme, minimal chrome, designed to be displayed on a wall-mounted monitor at a security desk. But the content is different: instead of counter status and ticket calls, it shows officer-originated broadcast messages.

When this document is complete:

- A user with the `SECURITY_OFFICER` role can navigate to `/security` and see the page.
- The page shows the most recent 50 active broadcasts in a chronological feed.
- New `BROADCAST_MESSAGE` events arriving via SSE are prepended to the feed within 1 second.
- New `OFFICER_REPLY` events arriving via SSE are prepended as informational entries.
- The unread indicator correctly shows which broadcasts are unread.
- Clicking/tapping a broadcast marks it as read.
- The read state persists in `localStorage` across page reloads.
- A non-security user is denied (403 or redirect to `/?error=forbidden`).
- The SSE channel subscription enforces the same authorization at the server side (via 3.1.1's `authorizeSseChannel`).

This document does **not** implement the broadcast creation (owned by 4.3.2), the reply creation (owned by 4.3.1), the display board's broadcast rendering (owned by 4.3.2), the SSE manager (owned by 3.1.1), the `useSSE()` hook (owned by 3.1.2), or the SSE channel auth (owned by 3.1.1 — the `security` channel already requires the `SECURITY_OFFICER` role).

### Step 1: Create `src/types/security-dashboard.types.ts` — TypeScript Types

**Goal:** Define the TypeScript types for the security screen state.

**Files affected:** `src/types/security-dashboard.types.ts` (new file).

**Specific instructions:**

Create the file with the following exports:

- **`SecurityDashboardData`** — the full dashboard state shape returned by the server-side data loader:
  - `user: { id, name, email }`.
  - `recentBroadcasts: BroadcastEntry[]` — the most recent 50 active broadcasts, most-recent-first.
  - `initialReadBroadcastIds: string[]` — the list of broadcast IDs the calling user has already marked as read. Loaded from `localStorage` on the client (the server returns an empty array; the client hydrates from `localStorage` on mount).
- **`BroadcastEntry`** — a discriminated union of two entry types in the feed:
  - `type: 'BROADCAST'`, `broadcastId`, `message`, `senderName`, `createdAt: string` (ISO 8601), `expiresAt: string | null` (ISO 8601), `isRead: boolean`.
  - `type: 'OFFICER_REPLY'`, `notificationId`, `replyId`, `repliedByOfficerName`, `repliedAt: string` (ISO 8601), `isRead: boolean`.
  - The `isRead` flag is client-side only (set on the client based on the `localStorage` read state). The server does not track read state.
- **`SECURITY_EVENT_FILTER`** — the SSE event filter for the security screen. A readonly array: `['BROADCAST_MESSAGE', 'OFFICER_REPLY']` as `const`. Exported as a top-level constant. The client component passes this to `useSSE('security', { filter: SECURITY_EVENT_FILTER })`.
- **`READ_STATE_STORAGE_KEY`** — the `localStorage` key: `'quems:security:broadcasts:read'`. Exported as a top-level constant. The value is a JSON-serialised map: `{ [broadcastId: string]: readTimestamp: number }`.

### Step 2: Create `src/lib/security-dashboard.ts` — Server-Side Data Loader

**Goal:** Provide a single function that loads the initial security screen state.

**Files affected:** `src/lib/security-dashboard.ts` (new file).

**Specific instructions:**

Create the file with:

- **`async function getSecurityDashboardData(userId: string, userRoles: string[]): Promise<SecurityDashboardData>`**:
  - Validates that the calling user has the `SECURITY_OFFICER` role. If not, throws `UnauthorizedSecurityAccessError` (the page catches this and returns 403).

    - Recent active broadcasts: `getActiveBroadcasts(50)` from `@/lib/broadcast` (4.3.2's module).
  - Maps the broadcasts to `BroadcastEntry` format:
    - `type: 'BROADCAST'`, `broadcastId`, `message`, `senderName`, `createdAt: createdAt.toISOString()`, `expiresAt: expiresAt?.toISOString() ?? null`, `isRead: false` (the client hydrates from `localStorage`).
  - Returns the assembled `SecurityDashboardData` with `initialReadBroadcastIds: []` (the server doesn't track read state; the client hydrates from `localStorage` on mount).
- **Custom error class:**
  - `class UnauthorizedSecurityAccessError extends Error` — thrown when the calling user does not have the `SECURITY_OFFICER` role. The `code` property is `'FORBIDDEN'`.

### Step 5: Create `src/components/security/security-screen-client.tsx` — The Client Wrapper

**Goal:** The client component that owns the SSE subscription, manages the feed state, and hydrates the read state from `localStorage`.

**Files affected:** `src/components/security/security-screen-client.tsx` (new file).

**Specific instructions:**

Create the file with:

- A `"use client"` directive.
- **Props:** `{ initialData: SecurityDashboardData }`.
- **Behaviour:**
  - **Step 1 — State management:**
    - `entries`: `useState<BroadcastEntry[]>(initialData.recentBroadcasts)` — the feed.
    - `readState`: `useState<Record<string, number>>({})` — the read map (broadcastId → readTimestamp). Initialised on mount from `localStorage` (see Step 2).
    - `pendingReadBroadcasts`: `useState<Set<string>>(new Set())` — the set of broadcast IDs that have been passively read (5+ seconds on screen) but not yet committed to `localStorage`. Used for debounced `localStorage` writes.
  - **Step 2 — Hydrate read state from `localStorage` on mount:**
    - In a `useEffect` with empty deps, reads `localStorage.getItem(READ_STATE_STORAGE_KEY)`. If the value exists, parses it and merges with the current `entries` state (marking matching entries as `isRead: true`).
    - This is a "client hydration" pattern — the server does not have access to `localStorage`, so the client hydrates on mount.
  - **Step 3 — Subscribe to SSE:** `useSSE('security', { filter: SECURITY_EVENT_FILTER, onEvent: (envelope) => { ... } })`. The `onEvent` callback:
    - If `envelope.type === 'BROADCAST_MESSAGE'`: prepends a new entry of type `'BROADCAST'` to the feed. Builds the entry from the payload `{ broadcastId: payload.broadcastId, message: payload.message, senderName: payload.senderName, createdAt: envelope.timestamp, expiresAt: <computed from createdAt + displaySeconds>, isRead: false }`. The `expiresAt` is `new Date(new Date(envelope.timestamp).getTime() + payload.displaySeconds * 1000).toISOString()` (computed client-side; matches the server's `expiresAt = createdAt + displayDurationSeconds`).
    - If `envelope.type === 'OFFICER_REPLY'`: prepends a new entry of type `'OFFICER_REPLY'`. Builds from the payload `{ notificationId: payload.notificationId, replyId: payload.replyId, repliedByOfficerName: payload.repliedByOfficerName, repliedAt: payload.repliedAt, isRead: false }`.
    - Maintains a max length of 50 (oldest dropped when new entries arrive).
  - **Step 4 — Mark as read (explicit click):** when the user clicks/taps a broadcast, calls `markAsRead(broadcastId)`. This:
    - Updates the `readState` map: `readState[broadcastId] = Date.now()`.
    - Updates the matching entry's `isRead` flag to `true` in the `entries` state.
    - Persists the updated `readState` to `localStorage`.
  - **Step 5 — Mark as read (passive — 5+ seconds on screen):** for broadcasts that have been on screen for 5+ seconds (visible in the viewport), passively mark as read. Implementation: a `useEffect` that watches the `entries` state and, for each entry, sets a 5-second timer. When the timer fires, marks the entry as read. Cleared on entry removal or unmount. This handles the "user sees the broadcast without explicitly clicking" case.
  - **Step 6 — `localStorage` write throttling:** the `localStorage.setItem` call in `markAsRead` is throttled to once per second (via a `useRef<number>` for the last write timestamp). Multiple rapid clicks in the same second batch into a single write.
  - **Step 7 — Render layout:**
    - **Top header (server-renderable portion):** `<SecurityHeader user={initialData.user} />` — shows the user name, current time (client-side `useState` + `setInterval`, updated every second), and a logout button (a small client sub-component that calls `signOut({ callbackUrl: '/login' })` from `next-auth/react`).
    - **Main content:** `<BroadcastFeed entries={entries} onMarkRead={markAsRead} />` — the chronological list.
    - **Unread badge:** `<UnreadIndicator count={unreadCount} />` — small badge in the header showing the unread count.
- **Exports:** the `SecurityScreenClient` component as default, plus the `SecurityHeader` sub-component (or as a separate file `security-header.tsx` — designer's choice).

### Step 6: Create `src/components/security/broadcast-feed.tsx` — The Chronological Feed

**Goal:** The chronological list of broadcast entries with unread indicators.

**Files affected:** `src/components/security/broadcast-feed.tsx` (new file).

**Specific instructions:**

Create the file with:

- A `"use client"` directive.
- **Props:** `{ entries: BroadcastEntry[], onMarkRead: (broadcastId: string) => void }`.
- **Behaviour:**
  - Renders each entry as a card/row with:
    - **For `BROADCAST` entries:**
      - Sender name (bold, e.g., "Officer John Silva").
      - Message text (large, prominent).
      - Relative timestamp via `Intl.RelativeTimeFormat` (e.g., "2 min ago").
      - Unread indicator (colored dot or border) when `isRead === false`.
      - The whole row is clickable; clicking calls `onMarkRead(broadcastId)`.
    - **For `OFFICER_REPLY` entries:**
      - Smaller, less prominent style.
      - Icon (lucide-react `MessageSquare` or `Reply`).
      - Description: "Officer [name] just replied".
      - Relative timestamp.
      - The row is NOT clickable (the reply is informational only; no action).
  - **Empty state:** when `entries.length === 0`, render "No broadcasts yet" with a friendly placeholder (e.g., a lucide-react `Inbox` icon and the message "Waiting for officer broadcasts...").
  - **Scrollable:** when more than 10 entries are visible, the feed is scrollable (`overflow-y-auto max-h-[60vh]` or similar). New entries at the top are visible without scrolling.
  - **Animation:** new entries animate in with a brief CSS slide-down + fade-in (respects `prefers-reduced-motion`).
  - **Accessibility:**
    - `role="feed"` for the container.
    - `role="article"` for each entry.
    - `aria-live="polite"` so screen readers announce new entries (but the announcement is throttled to once per 5 seconds to avoid spam).
    - `aria-label` describing the entry ("Broadcast from [senderName] at [time]").

## 8. Common Pitfalls

- **DO NOT** add a new permission to the RBAC system. The `notification:broadcast` permission already exists for `SECURITY_OFFICER`. The 4.3.3 page checks the ROLE directly (not the permission) because the page is a full-screen view — the role check is the appropriate granularity.
- **DO NOT** emit any SSE events from the security page. It is a consumer, not a producer.
- **DO NOT** modify the SSE infrastructure (3.1). The manager, hook, and channel auth are unchanged.
- **DO NOT** create a server-side read tracking table. Read state is `localStorage`-only in Phase 4.3.3.
- **DO NOT** subscribe to ticket lifecycle events on the security page. The strict filter `['BROADCAST_MESSAGE', 'OFFICER_REPLY']` is intentional — security officers care about officer communications, not queue activity.
- **DO NOT** add a `BroadcastMessageRead` table or API. Future enhancement; not in scope.
- **DO NOT** write to the `AuditLog` from the security page. It is read-only. Audit logs for broadcasts are written by 4.3.2.
- **DO NOT** use `bcryptjs` or any hashing in this document. The `senderDisplayName` is plain text (denormalised in the `BroadcastMessage` record by 4.3.2).
- **DO NOT** skip the middleware update. Without the middleware allowing `/security` for security officers, the page will be blocked at the middleware level before reaching the page's own auth check.
- **DO NOT** read `localStorage` in the server component (`page.tsx`). `localStorage` is browser-only. The read state hydration must happen in the client component (`security-screen-client.tsx`) via `useEffect`.
- **DO NOT** render the page with the standard `(dashboard)` or `(officer)` layouts. The security page is at the project root (`/security`) with its own minimal layout. This is intentional — it is a dedicated full-screen view, not part of any other UI.
- **DO NOT** use `useEffect` with `[]` deps for SSE subscription if the SSE hook itself manages subscription lifecycles. The 3.1.2 `useSSE()` hook handles connection/disconnection internally. The 4.3.3 component just calls `useSSE()` and uses the `onEvent` callback.

### 9.3 Common Questions

- **Q: Should the security screen be at `/security` or inside a route group?**
  A: At the project root (`/security`) with its own minimal layout. This is intentional — it is a dedicated full-screen view, similar to the display board at `/display` and the kiosk at `/kiosk`. No route group.

- **Q: Should the security page use `withRole` or check the role directly?**
  A: The page can use `withRole` (from 1.3.2) for consistency with the rest of the project. Alternatively, the page can check the role manually after `auth()`. Both patterns are valid. The document specifies the manual check (in the page) for clarity — the role check is the only logic in the page, so a manual check is simpler than wrapping in `withRole`.

- **Q: Should the security page be a "use client" component entirely?**
  A: No. The page is a server component (for auth + initial data load). The client wrapper handles the SSE subscription and state. This is the standard pattern from 4.2.3 (officer dashboard) and 3.2.1 (display board).

- **Q: Should the read state be per-user or per-browser?**
  A: Per-browser (`localStorage`) for Phase 4.3.3. A shift-change security officer will see the previous officer's read state if they use the same browser. Per-user read tracking would require a `BroadcastMessageRead` table — deferred to a future phase.

- **Q: Can the security officer acknowledge a broadcast?**
  A: Not in Phase 4.3.3. The page is read-only. Future enhancements may add an "Acknowledge" button that sends a confirmation back to the broadcast sender. Out of scope for 4.3.3.

- **Q: Should the security screen show historical broadcasts beyond the 50 most recent?**
  A: Not in Phase 4.3.3. The feed shows the most recent 50 active broadcasts. Historical broadcasts (expired or older than 50) are kept in the database for the reports phase. The security officer does not have a "load more" or "view history" button.

- **Q: What happens if the SSE connection drops?**
  A: The `useSSE()` hook (3.1.2) implements exponential backoff reconnection. New events arriving after reconnection are processed. Historical events (during the disconnect) are NOT replayed. The security officer may miss broadcasts that arrived during the disconnect — this is acceptable for Phase 4.3.3 (the officer can refresh the page to re-fetch the initial state from the server, which includes the most recent 50 active broadcasts).
