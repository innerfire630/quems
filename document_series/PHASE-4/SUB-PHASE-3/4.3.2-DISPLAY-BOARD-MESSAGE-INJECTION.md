## 1. Document Purpose

This document delivers the **display board message injection** — the conversion of a `NotificationReply` (created by 4.3.1) into a `BroadcastMessage`, the emission of the `BROADCAST_MESSAGE` SSE event, and the wiring of the display board's existing `<BroadcastBanner>` slot (reserved by 3.2.1) to actually render the broadcast. It also handles the **multiple simultaneous broadcast** scenario (rotation strategy) and the **expiry logic** (`isActive` and `expiresAt > now()` check on both server and client).

This document is the **second document in Sub-Phase 4.3** and the **hub of the reply-broadcast-display chain**. Without 4.3.2, the reply from 4.3.1 is recorded but never reaches the display board or the security screen. The 4.3.3 security screen is also affected — its initial data comes from 4.3.2's `getActiveBroadcasts` helper.

The strategic value of this document is that it makes the display board a **communication surface**, not just a status display. When a counter officer sends a reply (e.g., "Need assistance at Counter 2"), the message appears on the public display board so waiting customers and on-site staff can see it. This is the first time the display board is used for two-way communication in the system.

When this document is complete:

- Every `NotificationReply` created by 4.3.1 triggers a `BroadcastMessage` creation.
- Every `BroadcastMessage` is emitted as a `BROADCAST_MESSAGE` SSE event on the `['global', 'security']` channels.
- The display board's `<BroadcastBanner>` renders the message for `displaySeconds` (default 10s), then auto-dismisses.
- The banner supports manual dismissal (clicking).
- Multiple simultaneous broadcasts are handled via the rotation strategy (newest replaces current, pending broadcasts queue).
- Expired or inactive broadcasts are not displayed.
- `BROADCAST_MESSAGE_SENT` audit log entry is written for every broadcast creation.
- A `getActiveBroadcasts(limit)` helper is available for 4.3.3's security screen initial data load.

This document does **not** implement the reply creation (owned by 4.3.1), the security screen (owned by 4.3.3), the display board's other features (owned by 3.2), the SSE manager (owned by 3.1), or the SSE infrastructure (owned by 3.1).

### Step 1: Create `src/lib/broadcast.ts` — Centralised Broadcast Module

**Goal:** Provide the centralised broadcast operations (the single source of truth for `BroadcastMessage` creation and emission).

**Files affected:** `src/lib/broadcast.ts` (new file). If 4.3.1 created a temporary stub for this file, REPLACE the stub with the real implementation.

**Specific instructions:**

Create the file with:

- **Module-scope constants (exported):**
  - `const DEFAULT_DISPLAY_DURATION_SECONDS = 10`.
  - `const DEFAULT_EXPIRY_MINUTES = 5`.
  - These are the defaults for `displayDurationSeconds` and `expiresAt` respectively. They match the master plan's design intent.
- **`async function createBroadcastFromReply(reply: NotificationReply, replyingOfficerDisplayName: string): Promise<BroadcastMessage | null>`**:
  - The main entry point. Called by 4.3.1's `createReply` as a post-commit step.
  - **Input:** the `NotificationReply` record (with relations included — `counterOfficer: { include: { user: true, counter: true } }`) and the denormalised officer display name (the user who made the reply, resolved by 4.3.1).
  - **Step 1 — Compute expiry and duration:** `expiresAt = new Date(Date.now() + DEFAULT_EXPIRY_MINUTES * 60 * 1000)`. `displayDurationSeconds = DEFAULT_DISPLAY_DURATION_SECONDS`. The values are computed at creation time.
  - **Step 2 — Create `BroadcastMessage` record:** uses `db.broadcastMessage.create({ data: { message: reply.message, senderOfficerId: reply.counterOfficerId, senderDisplayName: replyingOfficerDisplayName, sourceReplyId: reply.id, targetDisplayBoardId: null, targetSecurityUserId: null, displayDurationSeconds, expiresAt, isActive: true } })`. Returns the created record.
  - **Step 3 — Emit `BROADCAST_MESSAGE` SSE event:** calls `emitBroadcastEvent(broadcast)` (defined in this same file, see below). The event is emitted AFTER the record is created.
  - **Step 4 — Update `NotificationReply.broadcastAt`:** uses `db.notificationReply.update({ where: { id: reply.id }, data: { broadcastAt: new Date() } })`. Wrapped in a try/catch — failures are logged via `console.error` but do not fail the broadcast.
  - **Step 5 — Write audit log:** calls `writeAuditLog({ action: 'BROADCAST_MESSAGE_SENT', actorId: reply.counterOfficerId, description: 'Officer reply broadcast to display board and security screen', metadata: { broadcastId: broadcast.id, sourceReplyId: reply.id, notificationId: reply.notificationId, message: broadcast.message, senderOfficerId: broadcast.senderOfficerId, senderDisplayName: broadcast.senderDisplayName, expiresAt: broadcast.expiresAt.toISOString(), displayDurationSeconds: broadcast.displayDurationSeconds, isActive: broadcast.isActive } })`. Best-effort (per Phase 1.3.3 discipline).
  - **Returns:** the created `BroadcastMessage`.
  - **Outer try/catch:** the entire function is wrapped. Unexpected errors are logged via `console.error` and `null` is returned. The 4.3.1 caller treats `null` as "broadcast creation failed, but the reply is still recorded".
- **`async function createDirectBroadcast(input: CreateDirectBroadcastInput): Promise<BroadcastMessage | null>`**:
  - For future use (admin-initiated broadcasts in Phase 5). Not called in 4.3.2 by anyone except `createBroadcastFromReply` indirectly.
  - **Input shape:** `{ message: string, senderOfficerId: string, senderDisplayName: string, targetDisplayBoardId?: string | null, targetSecurityUserId?: string | null, displayDurationSeconds?: number, expiresAt?: Date }`. All optional fields default to `null` (no specific target) and the module-scope constants.
  - **Step 1 — Same as `createBroadcastFromReply` steps 1-3:** compute defaults, create record, emit event.
  - **Step 2 — Write audit log** (no `sourceReplyId` or `notificationId` in the metadata since the broadcast is admin-initiated).
  - **Returns:** the created `BroadcastMessage`.
  - Exists for forward-compat with Phase 5 admin features (Phase 5 may add an admin panel for sending direct broadcasts). The 4.3.2 document does not call this function except internally.
- **`async function emitBroadcastEvent(broadcast: BroadcastMessage): Promise<void>`**:
  - Internal helper. NOT exported (or exported as `_emitBroadcastEvent` for testability — designer's choice).
  - **Step 1 — Visibility check:** calls `isBroadcastVisible(broadcast, new Date())`. If `false`, logs via `console.warn('[broadcast] Skipping emit for inactive or expired broadcast', { broadcastId: broadcast.id, isActive: broadcast.isActive, expiresAt: broadcast.expiresAt })` and returns (silent skip).
  - **Step 2 — Build payload:** constructs the SSE payload per master plan Section 11.4: `{ broadcastId: broadcast.id, message: broadcast.message, senderName: broadcast.senderDisplayName, displaySeconds: broadcast.displayDurationSeconds }`.
  - **Step 3 — Emit:** calls `broadcastRoutedEvent('BROADCAST_MESSAGE', payload, {})`. The existing `EVENT_ROUTING` entry resolves the empty context to `['global', 'security']`. Wrapped in a try/catch — failures are logged via `console.error` but do not throw.
- **`async function getActiveBroadcasts(limit: number = 50): Promise<BroadcastMessage[]>`**:
  - Fetches the most recent active broadcasts.
  - **Query:** `db.broadcastMessage.findMany({ where: { isActive: true, expiresAt: { gt: new Date() } }, orderBy: { createdAt: 'desc' }, take: limit })`.
  - **Returns:** the array of active broadcasts, most-recent-first.
  - Used by 4.3.3's security screen initial data load.
- **`function isBroadcastVisible(broadcast: BroadcastMessage, now: Date = new Date()): boolean`**:
  - Pure function. Returns `true` if `broadcast.isActive === true` AND `broadcast.expiresAt > now`. Returns `false` otherwise.
  - Handles the edge case where `expiresAt` is `null` (defensive — treats `null` as "no expiry" → visible).
  - Used by both the server (in `emitBroadcastEvent`) and the client (in the display board's banner component and the security screen's feed component).
- **Type exports:**
  - `interface CreateDirectBroadcastInput { message: string; senderOfficerId: string; senderDisplayName: string; targetDisplayBoardId?: string | null; targetSecurityUserId?: string | null; displayDurationSeconds?: number; expiresAt?: Date }`.
  - `type BroadcastWithRelations = BroadcastMessage & { ... }` (optional — used by 4.3.3 for the typed initial data).

### Step 2: Update `src/components/display/broadcast-banner.tsx` — Wire Up the Broadcast Rendering

**Goal:** The display board's `<BroadcastBanner>` component (which was reserved as a slot by 3.2.1) is now wired up to actually render the broadcast received via the SSE event.

**Files affected:** `src/components/display/broadcast-banner.tsx` (modified — file was created by 3.2.1 as a placeholder).

**Specific instructions:**

Open the existing file (created by 3.2.1) and **replace** the placeholder content with the real implementation. The file should:

- Be a client component (`"use client"`).
- **Props:** `{ broadcast: BroadcastDisplayState | null }` where `BroadcastDisplayState` is the shape of the broadcast from the parent display board's state. The display board's page passes the current broadcast (from the SSE-managed state) as the prop.
- **Behaviour:**
  - If `broadcast === null`: render nothing (the slot is hidden).
  - If `broadcast !== null`:
    - Render a prominent banner overlay (positioned at the top of the display board, full width, slightly inset from the edges). The banner sits on top of the counter cards but does not block them visually.
    - Show the `message` text in large typography (use `text-2xl` or `text-3xl` for prominence). White text on the `primary` background.
    - Show the `senderName` below the message in smaller text (`text-sm`). Semi-transparent white.
    - Show a countdown indicator (small, in the corner) showing seconds remaining until auto-dismiss.
    - The banner animates in with a CSS slide-down transition (respects `prefers-reduced-motion`).
    - The banner auto-dismisses after `displaySeconds` via a `useEffect` timer. When the timer fires, it calls the parent's `onDismiss` callback to clear the broadcast from local state.
    - Clicking the banner also calls `onDismiss` (manual dismissal).
  - **Accessibility:**
    - `role="status"` for the banner container.
    - `aria-live="polite"` so screen readers announce the message.
    - `aria-atomic="true"` so the entire message is read on update.
    - `aria-label` describing the banner ("Broadcast message from [senderName]").
  - **Styling:** uses Tailwind + the `display-bg` color token for the background, `display-text` for the text, `primary` for the banner background. Position: `fixed top-4 left-1/2 transform -translate-x-1/2 z-40` so the banner is centered horizontally and at the top, with high z-index to overlay the counter cards.
- **The component is purely presentational.** The state management (queueing multiple broadcasts, picking the next one when the current dismisses) is handled by the parent display board page (the state is managed via the `useSSE` hook's `onEvent` callback that updates a React state holding a queue of pending broadcasts). The banner just renders the current broadcast and reports dismissal.
- **Exports:** the `BroadcastBanner` component as default, plus an optional `BroadcastDisplayState` type (for the parent page's convenience).

**Multiple broadcast handling strategy (implemented in the PARENT display board page, not in the banner component):**

- The display board's page (3.2.1) holds a queue of pending broadcasts in React state.
- On `BROADCAST_MESSAGE` SSE event, the new broadcast is added to the queue.
- The currently displayed broadcast (the head of the queue) runs to its `displaySeconds` (or until manually dismissed). Then the next broadcast from the queue takes its place.
- If multiple broadcasts arrive within a single second, the latest is shown first (FIFO among the arrivals in that second — implemented by always appending to the queue).
- The display board's `useEffect` watches the queue length and the current broadcast's `expiresAt`. When the current expires, it advances the queue.

**Note:** The parent display board page's queue management is NOT modified in this document. The 3.2.1 setup already manages state via the SSE event. The 4.3.2 document only updates the `<BroadcastBanner>` component to render the state correctly. If the parent page's queue management is insufficient, a follow-up enhancement can extend the parent page — but that is out of scope for 4.3.2.

## 8. Common Pitfalls

- **DO NOT** call `broadcastRoutedEvent` directly in this document's module except through the `emitBroadcastEvent` helper. The helper enforces the expiry check. Calling `broadcastRoutedEvent` directly bypasses the check.
- **DO NOT** modify the display board's reducer (`src/lib/display-state.ts`) or its SSE subscription (in `src/app/display/page.tsx`). Both are already correct from 3.2.1/3.2.3. 4.3.2 only updates the `<BroadcastBanner>` component.
- **DO NOT** create a new `POST /api/broadcasts` endpoint. The reply endpoint (4.3.1) is the only entry point. The broadcast creation is invoked as a post-commit step.
- **DO NOT** write the `expiresAt` field as a relative time. It must be an absolute `Date` object (the Prisma `DateTime` type).
- **DO NOT** skip the `isBroadcastVisible` check in `emitBroadcastEvent`. Without it, expired broadcasts are emitted to all subscribers — wasted work and confusing UX.
- **DO NOT** mutate the `BroadcastMessage` object in the `emitBroadcastEvent` helper. The object is the source of truth — emit a copy if transformations are needed.
- **DO NOT** enforce idempotency on `createBroadcastFromReply`. Each call creates a new broadcast. The 4.3.1 endpoint is non-idempotent (each reply is a separate message).
- **DO NOT** add the `BROADCAST_MESSAGE` event type to the `SseEventPayload` union. It was already added by 3.1.3 as a forward-seam entry. Only the `OFFICER_REPLY` variant (owned by 4.3.1) needs adding.
- **DO NOT** modify the `EVENT_ROUTING` constant. The `BROADCAST_MESSAGE: ['global', 'security']` entry is already in place from 3.1.3.
- **DO NOT** use `bcryptjs` or any hashing in this document. The `senderDisplayName` is denormalised plain text (the officer's user name at the time of the reply). No hashing needed.
- **DO NOT** denormalise the sender's user data beyond the `name`. The `User` table is the source of truth for the user's current data. Only the display name (for the broadcast) is denormalised because the broadcast is a historical record and a later officer rename should not affect old broadcasts.

### 9.3 Common Questions

- **Q: Should the display board's parent page be updated to manage a queue of broadcasts?**
  A: 3.2.1's existing setup handles the state via the SSE event and the reducer. The 4.3.2 document does not need to update the parent page unless the existing setup is insufficient. If the agent finds that the display board does NOT show the broadcast (e.g., the state is not passed to the banner), the agent may need to update the parent page — but this is a fallback, not the primary path.

- **Q: Should the `targetDisplayBoardId` field be used in 4.3.2?**
  A: For Phase 4.3.2, the field is set to `null` (broadcasts target all display boards). Future enhancements may allow per-board targeting, but the data model already supports it.

- **Q: What if the FCM dispatch and the SSE broadcast are out of sync?**
  A: They are independent. The FCM push (4.1.3) delivers the notification to the officer's mobile device. The SSE broadcast (4.3.2) delivers the reply to the display board and security screen. They are parallel actions, not interdependent.

- **Q: Should the broadcast message be translated or formatted?**
  A: For Phase 4.3.2, the broadcast message is the reply text verbatim. Future enhancements may add formatting (e.g., "[Officer John]: Need assistance at counter 2"). The current `senderDisplayName` is shown separately as the sender label, so no formatting is needed in the message text.

- **Q: Can the display board's banner be customised (e.g., per-board configuration)?**
  A: The default banner styling is fixed for Phase 4.3.2. The display board's `themeColor`, `logoUrl`, etc. (from the `DisplayBoard` model) are not applied to the broadcast banner — the banner uses the system-wide colors. Future enhancements may support per-board banner styling.
