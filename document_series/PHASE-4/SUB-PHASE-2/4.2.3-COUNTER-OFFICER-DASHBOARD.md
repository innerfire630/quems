## 1. Document Purpose

This document delivers the **full counter officer dashboard** — the composition layer that brings together every piece of the officer's operational surface into a single coherent page. The dashboard replaces the temporary stub page that Phase 2.3.2 put in place (`src/app/(officer)/counter/[counterId]/page.tsx` with just the `TicketActionPanel`) and elevates it to a complete operational home base.

The dashboard is fundamentally a **composition layer, not a re-implementation**. It imports and composes:

- The `<TicketActionPanel>` from 2.3.2 — Call, Recall, No-Show buttons.
- The `<CounterStatusToggle>` from 4.2.1 — open/close the counter with reason.
- The `<NotificationToggle>` from 4.2.2 — enable/disable push notifications.
- New dashboard-specific components: `<QueueDepthIndicator>`, `<NextTicketPreview>`, `<RecentActivityFeed>`.
- The `useSSE()` hook from 3.1.2 — real-time updates via the `counter:[counterId]` channel.

No business logic is duplicated from earlier phases. The dashboard only orchestrates pre-existing components and provides the layout, data loading, and real-time wiring.

The dashboard consumes events from the `counter:[counterId]` SSE channel: ticket lifecycle events (`TICKET_CALLED`, `TICKET_RECALLED`, `TICKET_NO_SHOW`), queue events (`TICKET_QUEUED`, `QUEUE_UPDATED`), counter status events (`COUNTER_OPENED`, `COUNTER_CLOSED` — emitted by 4.2.1's PATCH endpoint), and notification events (`NOTIFICATION_RECEIVED`). The display board (3.2) consumes these events separately on the `global` channel. The dashboard's consumption on the counter channel is a sibling, not a replacement.

When this document is complete:

- An officer assigned to a counter can navigate to `/counter/[counterId]` and see the full dashboard.
- The dashboard shows: counter header (name + number + status indicator), current serving ticket card, queue depth indicator, next ticket preview, action panel, counter status toggle, notification toggle, recent activity feed.
- The dashboard updates in real time via SSE — when a ticket is queued, called, recalled, or marked no-show, when the counter status changes (by this officer or another admin), when a push notification is dispatched.
- Loading and empty states are handled gracefully — skeleton on load, "No tickets waiting" empty state for the queue, disabled action buttons when the counter is closed.
- A non-assigned officer cannot access another counter's dashboard (403).

This document does **not** implement the toggle components (owned by 4.2.1 and 4.2.2), the action panel (owned by 2.3.2), the SSE infrastructure (owned by 3.1), the call/recall/no-show endpoints (owned by 2.3.1 and 2.3.2), or the FCM dispatch (owned by 4.1.3).

### Step 2: Create `src/lib/officer-dashboard.ts` — Server-Side Data Loader

**Goal:** Provide a single function that loads all the initial dashboard state in a single server-side call.

**Files affected:** `src/lib/officer-dashboard.ts` (new file).

**Specific instructions:**

Create the file with:

- **`async function getOfficerDashboardData(counterId: string, userId: string): Promise<OfficerDashboardData>`**:
  - Validates that the calling user is assigned to the counter. If not, throws `OfficerNotAssignedToCounterError` (the route handler catches this and returns 403).
  - Loads all dashboard data in PARALLEL via `Promise.all`:
    - `getCounterDetails(counterId)` — counter name, number, displayLabel, isActive.
    - `findCurrentServingTicketForCounter(counterId)` — the ticket currently being served (if any), with service name joined.
    - `findNextWaitingTicketForCounter(counterId)` — the next waiting ticket for the counter's services.
    - `getQueueDepth(counterId)` — count of `WAITING` tickets for the counter's services. Implementation: `db.ticket.count({ where: { status: 'WAITING', service: { counters: { some: { counterId } } } } })`.
    - `getRecentActivity(counterId, 10)` — last 10 `TicketEvent` rows for tickets assigned to this counter, ordered by `createdAt desc`. Includes ticket number and officer name via joins.
    - `getRecentStatusEvents(counterId, 5)` — last 5 `CounterStatusEvent` rows for the counter (from 4.2.1's module). Includes officer name.
    - `getNotificationsState(userId)` — full notification state for the user (from 4.2.2's module).
    - `getCurrentStatus(counterId)` — current counter status from 4.2.1's module.
    - `findCounterOfficerForUserAndCounter(userId, counterId)` — the user's `CounterOfficer` record for this counter (for `isOnDuty`, `notificationsEnabled`, etc.).
    - `getUserBasicInfo(userId)` — user name and email (simple `findUnique`).
  - All queries are efficient — uses `include` to avoid N+1.
  - Returns the assembled `OfficerDashboardData` object.
- **Internal helpers (not exported):** `getCounterDetails`, `getQueueDepth`, `getRecentActivity`, `getUserBasicInfo` — small focused functions that return subsets of the dashboard data.
- **Custom error class:**
  - `OfficerNotAssignedToCounterError` — thrown when the user is not assigned to the counter.

### Step 8: Create `src/components/officer/officer-dashboard-client.tsx` — The Client-Side Dashboard Wrapper

**Goal:** A client component that wraps the dashboard layout, owns the SSE subscription, and passes event updates down to the sub-components.

**Files affected:** `src/components/officer/officer-dashboard-client.tsx` (new file).

**Specific instructions:**

Create the file with:

- A `"use client"` directive.
- **Props:** `{ initialData: OfficerDashboardData }`.
- **Behaviour:**
  - Owns the SSE subscription: `useSSE('counter:<initialData.counter.id>', { filter: DASHBOARD_EVENT_FILTER })`.
  - Holds the dashboard state in React state (`useState<OfficerDashboardData>(initialData)`).
  - On SSE events:
    - `TICKET_QUEUED` → increment queue depth in state.
    - `QUEUE_UPDATED` → re-fetch queue depth + next ticket (or update from payload if available).
    - `TICKET_CALLED` → update current serving ticket in state. Refresh next ticket.
    - `TICKET_RECALLED` → update current serving ticket in state (same ticket, recalled status).
    - `TICKET_NO_SHOW` → clear current serving ticket (it will be re-populated by the subsequent `TICKET_CALLED` from auto-advance).
    - `COUNTER_OPENED` → update `officerContext.currentStatus` to `AVAILABLE` (mapped from the API's `OPENED`). Update the toggle component's state.
    - `COUNTER_CLOSED` → update `officerContext.currentStatus` to `CLOSED`. Update the toggle component's state. Disable action buttons.
    - `NOTIFICATION_RECEIVED` → visual indicator (e.g., a brief toast or a "Notification dispatched" entry in the recent activity feed).
  - **Layout:** renders the dashboard in a grid layout:
    - **Top row:** `<CounterHeader>` (counter name + number + status badge).
    - **Second row:** two-column grid:
      - Left: `<CurrentServingTicketCard>` (large ticket display).
      - Right: vertical stack of `<QueueDepthIndicator>`, `<NextTicketPreview>`, `<NotificationToggle>`.
    - **Third row:** two-column grid:
      - Left: `<TicketActionPanel>` (Call / Recall / No-Show buttons).
      - Right: `<CounterStatusToggle>` (open/close toggle).
    - **Fourth row:** full-width `<RecentActivityFeed>`.
  - On `counterStatus === 'CLOSED'`:
    - The action panel's buttons are disabled (pass a `disabled` prop or rely on the panel's own logic if it has it).
    - A clear banner or message is shown: "Counter is closed — reopen to serve tickets."
  - Uses Tailwind grid classes for the responsive layout (`grid grid-cols-1 md:grid-cols-2 gap-6`).
  - Uses shadcn `Card` for each sub-section.

### Step 12: End-to-End Verification

**Goal:** Confirm the dashboard renders correctly, consumes SSE events, and updates in real time.

**Files affected:** None (verification only).

**Specific instructions:**

- **Test 1 — Initial load:**
  - Log in as a counter officer assigned to counter X. Navigate to `/counter/X`.
  - Verify: the dashboard renders with all sections — header, current serving ticket (or empty state), queue depth, next ticket (or empty state), action panel, status toggle, notification toggle, recent activity feed.
  - Verify: the initial state is loaded from the server (no loading skeleton after the initial render).
- **Test 2 — Live SSE updates:**
  - From a separate browser session (as a customer or kiosk), issue a ticket for a service handled by counter X.
  - Verify: within 1 second, the queue depth increments in the dashboard.
  - From another session (as a different officer or admin), call the ticket.
  - Verify: within 1 second, the dashboard's "current serving ticket" updates to the new ticket.
  - Recall the ticket.
  - Verify: within 1 second, the dashboard reflects the recall.
  - Mark the ticket as no-show.
  - Verify: within 1 second, the dashboard clears the current serving ticket (and the auto-advance's `TICKET_CALLED` populates the next ticket).
- **Test 3 — Counter status reflection:**
  - From another session, close counter X (via the toggle).
  - Verify: within 1 second, the dashboard's status badge switches to "Temporarily Closed", the action buttons are disabled, and the "Counter is closed" message is shown.
  - Reopen the counter.
  - Verify: within 1 second, the dashboard returns to normal.

## 8. Common Pitfalls

1. **Re-implementing business logic.** The dashboard is a composition layer. Do NOT re-implement Call / Recall / No-Show logic, the state machine, the dispatch logic, the toggle logic, etc. Import and compose the existing components.

2. **Emitting SSE events from the dashboard.** The dashboard is a CONSUMER, not a producer. Do NOT call `broadcastEvent` or `broadcastRoutedEvent` from any dashboard path. The action panel and toggle components emit events via their respective API calls.

3. **Hardcoding event types in multiple places.** Use the `DASHBOARD_EVENT_FILTER` constant from `src/types/officer-dashboard.types.ts` as the single source of truth for which events the dashboard consumes. Do NOT inline the array in the `useSSE` call.

4. **Forgetting to re-fetch after `QUEUE_UPDATED`.** The `QUEUE_UPDATED` event signals that the queue has changed, but the payload may not include the full next ticket data. The agent should re-fetch the next ticket (and possibly the queue depth) on this event, not just increment/decrement locally.

5. **Disabling the action buttons via the wrong mechanism.** The action panel has its own enable/disable logic based on the ticket state machine. The dashboard should pass `disabled` to the panel based on `counterStatus === 'CLOSED'`. If the panel doesn't accept a `disabled` prop, the dashboard wraps it in a `<div>` with `pointer-events-none opacity-60` or similar.

6. **Not handling the `NOTIFICATION_RECEIVED` event.** This event is consumed on the counter channel by 4.2.3. The dashboard shows a brief visual indicator (toast or activity feed entry) when a push notification is dispatched. Skipping this event makes the dashboard feel disconnected from the mobile app experience.

7. **Using the wrong event payload fields.** The SSE payload for `COUNTER_OPENED` / `COUNTER_CLOSED` (added in 4.2.1) has specific field names: `counterId`, `counterNumber`, `counterName`, `changedByOfficerId`, `changedByOfficerName`, `changedAt`, `reason`. The dashboard uses these exact names.

8. **Confusing the counter channel with the global channel.** The dashboard subscribes to `counter:<counterId>` (the per-counter channel). The display board subscribes to `global`. Both receive `COUNTER_OPENED` / `COUNTER_CLOSED` events (because of the 4.2.1 routing extension). The agent should NOT subscribe to `global` from the dashboard — that would receive events for ALL counters, not just the officer's.

9. **Not initialising the SSE filter as `readonly`.** The `DASHBOARD_EVENT_FILTER` constant should be typed as `readonly SseEventType[]` (or `as const`) so that `useSSE`'s `filter` parameter type matches. Inlining a mutable array in the `useSSE` call works at runtime but breaks the type check.

10. **Forgetting the `runtime = 'nodejs'` and `dynamic = 'force-dynamic'` exports.** These are required for the page to work correctly. Without them, Next.js may attempt to cache or run in the edge runtime.

11. **Not handling the SSE hook's reconnection logic.** The `useSSE` hook from 3.1.2 has its own reconnection logic. The dashboard does NOT implement its own reconnection. Trusting the hook's behaviour.

12. **Not exporting `DASHBOARD_EVENT_FILTER` from the correct location.** If the constant is used in multiple files (e.g., the dashboard client, a potential unit test, a debug panel), it should be exported from a shared location like `src/types/officer-dashboard.types.ts` or `src/lib/sse-dashboard-filter.ts`. Do NOT duplicate the array in multiple files.
