## 1. Document Purpose

This document delivers the **temporary counter closure system** — the operational feature that lets a counter officer mark their counter as temporarily unavailable, with an optional reason, and have that closure reflected everywhere it matters: the display board that customers see, the officer's own dashboard, and the system that prevents new tickets from being called to the closed counter.

The closure is a deliberate, reversible state change. An officer closes their counter when they step away for a break, a meeting, or an emergency. The system records the closure as a `CounterStatusEvent`, broadcasts a `COUNTER_CLOSED` SSE event to both the `global` channel (which the display board consumes) and the `counter:[counterId]` channel (which the officer's own dashboard consumes), updates `CounterOfficer.currentStatus` from `AVAILABLE` to `CLOSED`, and prevents the call/recall/no-show endpoints from assigning new work to that counter. When the officer reopens, the inverse happens: a `CounterStatusEvent` is recorded, a `COUNTER_OPENED` event is broadcast, and normal operation resumes.

Tickets that are already waiting for the closed counter's services **continue to wait**. They are not auto-reassigned to other counters. This is a deliberate UX decision: reassignment would silently change which counter serves a customer, creating confusion. The display board's "Temporarily Closed" label informs waiting customers that the counter is temporarily unavailable.

When this document is complete:

- An officer assigned to a counter can close that counter via `PATCH /api/counters/[counterId]/status` with `{ status: 'CLOSED', reason: 'On break' }`.
- The closure is reflected on the display board within 1 second (the affected counter's card switches to opacity-50 with a "Temporarily Closed" label).
- The closure is reflected on the officer's own dashboard within 1 second (the action buttons disable, the status toggle switches to "Closed").
- A `CounterStatusEvent` is created with the correct status, reason, officer, and timestamp.
- No new tickets can be called to the closed counter — the call endpoint returns 403 with code `COUNTER_CLOSED`.
- Tickets already waiting for the closed counter's services continue to wait. They are not reassigned.
- When the officer reopens the counter, normal operation resumes — the inverse of all the above happens.

This document does **not** implement the notification toggle (Phase 4.2.2), the full officer dashboard composition (Phase 4.2.3), the officer reply flow (Phase 4.3.1), the broadcast system (Phase 4.3.2), the security officer screen (Phase 4.3.3), or rate limiting (Phase 5.2.1).

### Step 5: Create `src/lib/counter-status.ts` — Centralised Counter Status Module

**Goal:** Provide focused, reusable functions for all `CounterStatusEvent` and `CounterOfficer.currentStatus` operations.

**Files affected:** `src/lib/counter-status.ts` (new file).

**Specific instructions:**

Create the file with the following exports:

- **Type definitions:**
  - `interface SetCounterStatusInput { counterId: string; officerId: string; newStatus: 'AVAILABLE' | 'CLOSED'; reason?: string | null }`.
  - `interface CounterStatusSnapshot { status: 'OPENED' | 'CLOSED'; reason: string | null; lastChangedAt: Date; changedByOfficerId: string | null; changedByOfficerName: string | null }`.
  - `interface CounterStatusEventWithOfficer extends CounterStatusEvent { changedByOfficerName: string }`.
- **`async function getCurrentStatus(counterId: string): Promise<CounterStatusSnapshot | null>`**:
  - Looks up the most recent `CounterStatusEvent` for the counter (order by `createdAt desc`).
  - If found, returns `{ status: <mapped from event.status>, reason: event.reason, lastChangedAt: event.createdAt, changedByOfficerId: event.counterOfficerId, changedByOfficerName: <looked up via include> }`.
  - If no event exists, falls back to looking up the current `CounterOfficer` record (where `isOnDuty = true`) and reading `currentStatus` directly.
  - If no officer is on duty, returns `null`.
  - Used by the dashboard's initial render and by the toggle component's initial state.
- **`async function setCounterStatus(input: SetCounterStatusInput): Promise<CounterStatusEventWithOfficer>`**:
  - Validates the transition: cannot close an already-closed counter (idempotency check). Cannot open an already-open counter. Returns a clear error (`InvalidCounterStatusTransitionError`) if the transition is illegal.
  - Resolves the calling officer's `CounterOfficer` record for the counter. Throws `OfficerNotAssignedToCounterError` if the officer is not assigned.
  - Wraps in a `prisma.$transaction` containing:
    1. Re-check inside the transaction that the counter exists and the officer is assigned (TOCTOU safety).
    2. Update `CounterOfficer.currentStatus` to the new value.
    3. Create the `CounterStatusEvent` record with `{ counterId, counterOfficerId, status: newStatus, reason: input.reason ?? null, createdAt: new Date() }`.
    4. Return the created event with the officer name included.
  - The function does NOT emit the SSE event — the route handler does that after the transaction commits. This keeps the function pure and testable.
  - The function does NOT write the `AuditLog` — the route handler does that after the transaction commits.
- **`async function isCounterClosed(counterId: string): Promise<boolean>`**:
  - Lightweight check: returns `true` if any active `CounterOfficer` record with `isOnDuty = true` has `currentStatus = 'CLOSED'`. Returns `false` otherwise (including if no officer is on duty — a counter with no officer is treated as "not closed", since there's nothing to be closed).
  - Used by the call/recall/no-show/call-next endpoints in 2.3.1 and 2.3.2 (the integration step). The endpoints call this BEFORE the transaction (fail fast).
  - The function is intentionally simple — it's called on every ticket action and must not be expensive.
- **`async function getRecentStatusEvents(counterId: string, limit: number): Promise<CounterStatusEventWithOfficer[]>`**:
  - Returns the last `limit` `CounterStatusEvent` records for the counter, ordered by `createdAt desc`.
  - Includes the officer name via join.
  - Used by the recent activity feed in 4.2.3 (the dashboard). Not used directly by 4.2.1's UI, but available for the dashboard's composition.
- **Custom error classes:**
  - `InvalidCounterStatusTransitionError` — thrown when the requested transition is illegal (e.g., closing an already-closed counter).
  - `OfficerNotAssignedToCounterError` — thrown when the calling user has no `CounterOfficer` record for the specified counter.

### Step 6: Create `src/app/api/counters/[counterId]/status/route.ts` — The PATCH Endpoint

**Goal:** Expose the counter status change business logic via `PATCH /api/counters/[counterId]/status`.

**Files affected:** `src/app/api/counters/[counterId]/status/route.ts` (new file).

**Specific instructions:**

Create the file with:

- **Module configuration exports:**
  - `export const runtime = 'nodejs'`.
  - `export const dynamic = 'force-dynamic'`.
- **`PATCH` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_COUNTER_CLOSE)`.
  - Reads `counterId` from route params (Next.js dynamic segment).
  - Calls `auth()` to get the session.
  - Validates the request body with `counterStatusChangeSchema`. On failure, returns `VALIDATION_ERROR` (HTTP 422).
  - Maps the API status (`'OPENED'`) to the database enum (`'AVAILABLE'`) at the boundary. The API uses `OPENED` for human readability; the database uses `AVAILABLE`.
  - Calls `setCounterStatus({ counterId, officerId: <resolved from session.user.userId + CounterOfficer lookup>, newStatus: <mapped>, reason })`.
  - On success:
    - **AFTER the transaction commits**:
      - Build the SSE event payload (`COUNTER_OPENED` or `COUNTER_CLOSED`).
      - Emit `broadcastRoutedEvent('COUNTER_OPENED' | 'COUNTER_CLOSED', payload, { counterId })`. The `EVENT_ROUTING` extension from Step 2 handles the channel routing.
      - Write `AuditLog` entry with action `COUNTER_STATUS_CHANGED` and metadata `{ counterId, oldStatus, newStatus, reason, counterStatusEventId }`. AFTER the SSE emission (best-effort, per Phase 1.3.3 discipline).
    - Return `{ success: true, data: <event with officer name> }` with HTTP 200.
  - **Error mapping:**
    - On `InvalidCounterStatusTransitionError` → `VALIDATION_ERROR` (HTTP 422) with the message "Counter is already in the requested state".
    - On `OfficerNotAssignedToCounterError` → `FORBIDDEN` (HTTP 403).
    - On unknown error → `INTERNAL_ERROR` (HTTP 500).
  - **Defensive try/catch** wraps the entire handler — unexpected errors are caught and logged via `console.error`. The SSE emission and audit log write are also wrapped in try/catch — their failures do NOT fail the API response.

## 8. Common Pitfalls

1. **Modifying the call/recall/no-show transaction logic.** The 4.2.1 modification is ONLY a pre-transaction check. Do not modify the transaction body, the state machine, the SSE emission, or the audit log writes inside these endpoints. Only add the `isCounterClosed` check before the existing logic.

2. **Forgetting the EVENT_ROUTING context.** `broadcastRoutedEvent` requires `context = { counterId }` for the `<counterId>` placeholder to resolve. If the context is missing, the counter channel is skipped silently and the officer's own dashboard won't update.

3. **Mapping the API status to the database enum inconsistently.** The API uses `'OPENED'` and `'CLOSED'` (human-readable); the database uses `'AVAILABLE'` and `'CLOSED'`. The mapping happens at the API boundary, inside the route handler (or in a helper function). Do NOT pass `'OPENED'` to Prisma — Prisma will reject it because it's not in the enum.

4. **Emitting the SSE event inside the transaction.** The transaction commits, THEN the SSE event is emitted. If the transaction rolls back, the event should not have been sent. Always emit AFTER commit.

5. **Writing the AuditLog inside the transaction.** Same rationale as SSE — the audit log is a side effect of a successful action, not part of the action's atomicity.

6. **Forgetting the TOCTOU re-check inside the transaction.** The `setCounterStatus` function re-checks the counter existence and officer assignment inside the transaction to prevent TOCTOU race conditions. A pre-transaction check is necessary but not sufficient.

7. **Treating the `CLOSED` state as a permanent deactivation.** The 4.2.1 endpoint only toggles `CounterOfficer.currentStatus`. It does NOT set `Counter.isActive = false` (that's an admin action via 2.1.2). The display board's opacity-50 + "Temporarily Closed" label is a transient visual state, not a permanent deactivation.

8. **Auto-reassigning waiting tickets on closure.** This is explicitly NOT done. Tickets wait. The display board's "Temporarily Closed" label informs customers. Reassignment would silently change which counter serves a customer, creating confusion.

9. **Emitting the SSE event for the same status twice (idempotency bug).** `setCounterStatus` validates the transition and rejects "already closed" / "already open" attempts. The endpoint should NOT emit an SSE event if the transition is illegal (no state change occurred).

10. **Forgetting the `runtime = 'nodejs'` and `dynamic = 'force-dynamic'` exports on the new route.** These are required for the PATCH handler to work in the Next.js App Router. Without them, the route may attempt to cache or run in the edge runtime, causing subtle bugs.
