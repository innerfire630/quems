## 1. Document Purpose

This document delivers the **notification toggle feature** — the personal preference that lets an officer enable or disable push notifications for their devices, on a per-counter basis, with the dispatch system from Phase 4.1.3 respecting the toggle state in real time.

The toggle is intentionally independent of the counter's open/closed state. An officer can have their counter `AVAILABLE` (open) but `notificationsEnabled = false` (they don't want push notifications while at the counter). Or they can have their counter `CLOSED` (on break) but `notificationsEnabled = true` (they want to know what's happening in the queue even while on break). The two states are tracked in separate fields, updated by separate endpoints, and enforced by separate validation — a deliberate non-negotiable UX rule documented in the master plan Section 4.2.2.

The dispatch logic in 4.1.3 already reads `CounterOfficer.notificationsEnabled` before sending a push notification. If the field is `false`, no `Notification` record is created — the dispatch is skipped entirely. This document does NOT modify the dispatch logic; it only provides the endpoint and UI to write the field that the dispatch logic already reads.

When this document is complete:

- An officer assigned to a counter can toggle push notifications on or off for that counter via `PATCH /api/officers/me/notifications` with `{ counterId, notificationsEnabled }`.
- The toggle state is persisted to `CounterOfficer.notificationsEnabled`.
- Subsequent ticket issuance (Phase 2.2.1) and recall (Phase 2.3.1) events respect the toggle — an officer with `notificationsEnabled = false` does NOT receive push notifications.
- An officer assigned to multiple counters can have different notification preferences for each. The toggle is per-counter, not per-officer.
- The dashboard (Phase 4.2.3) renders the toggle with the correct initial state for each of the officer's counter assignments.
- The toggle is fully independent of the counter open/closed state. The two fields are tracked separately and enforced separately.

This document does **not** implement the counter closure system (Phase 4.2.1), the full officer dashboard composition (Phase 4.2.3), the FCM dispatch logic (Phase 4.1.3 — already reads `notificationsEnabled` but is unchanged here), the device registration endpoints (Phase 4.1.2), the officer reply flow (Phase 4.3.1), or rate limiting (Phase 5.2.1).

- Export inferred types: `OfficerNotificationsUpdateInput`, `OfficerNotificationsStateEntry`, `OfficerNotificationsUpdateResponse`.

### Step 3: Create `src/lib/officer-notifications.ts` — Centralised Module

**Goal:** Provide focused, reusable functions for reading and writing the `CounterOfficer.notificationsEnabled` field.

**Files affected:** `src/lib/officer-notifications.ts` (new file).

**Specific instructions:**

Create the file with the following exports:

- **Type definitions:**
  - `interface SetNotificationsEnabledInput { counterOfficerId: string; enabled: boolean }`.
  - `interface NotificationsStateEntry { counterOfficerId: string; counterId: string; counterName: string; counterNumber: number; notificationsEnabled: boolean }`.
- **`async function setNotificationsEnabled(input: SetNotificationsEnabledInput): Promise<CounterOfficer>`**:
  - Validates that the `CounterOfficer` record exists. Throws `CounterOfficerNotFoundError` if not.
  - Performs a single Prisma `update` call: `db.counterOfficer.update({ where: { id: counterOfficerId }, data: { notificationsEnabled: input.enabled } })`.
  - The update is atomic — no transaction needed for a single update.
  - Returns the updated `CounterOfficer` record (with the new `notificationsEnabled` value).
  - Does NOT write the `AuditLog` — the route handler does that after the update.
- **`async function getNotificationsState(userId: string): Promise<NotificationsStateEntry[]>`**:
  - Queries all `CounterOfficer` records for the given user, joined with the `Counter` table to denormalise the counter name and number.
  - Returns the array, ordered by `Counter.number asc` (consistent display order).
  - Used by the GET endpoint to return the full state, and by the dashboard (4.2.3) for initial state.
- **`async function findCounterOfficerForUserAndCounter(userId: string, counterId: string): Promise<CounterOfficer | null>`**:
  - Looks up the `CounterOfficer` record matching the given user and counter.
  - Returns `null` if not found.
  - Used by the route handler to enforce ownership (the calling user must be assigned to the specified counter).
- **Custom error class:**
  - `CounterOfficerNotFoundError` — thrown when the `CounterOfficer` record does not exist.

### Step 4: Create `src/app/api/officers/me/notifications/route.ts` — The PATCH and GET Endpoints

**Goal:** Expose the notification toggle business logic via HTTP PATCH and the current state via HTTP GET.

**Files affected:** `src/app/api/officers/me/notifications/route.ts` (new file).

**Specific instructions:**

Create the file with:

- **Module configuration exports:**
  - `export const runtime = 'nodejs'`.
  - `export const dynamic = 'force-dynamic'`.
- **`PATCH` handler:**
  - Wrapped with `auth()` only (no `withPermission` call — the toggle is a personal preference available to any counter officer).
  - Validates the request body with `officerNotificationsUpdateSchema`. On failure, returns `VALIDATION_ERROR` (HTTP 422).
  - Calls `auth()` to get the session, then `findCounterOfficerForUserAndCounter(session.user.userId, body.counterId)`.
  - If the lookup returns `null`, returns `FORBIDDEN` (HTTP 403) — the calling user is not assigned to this counter.
  - Calls `setNotificationsEnabled({ counterOfficerId: <found id>, enabled: body.notificationsEnabled })`.
  - On success:
    - Writes `AuditLog` entry with action `NOTIFICATIONS_TOGGLED` and metadata `{ counterId, counterOfficerId, oldValue, newValue }`. AFTER the update (best-effort, per Phase 1.3.3 discipline).
    - Returns `{ success: true, data: { counterId, notificationsEnabled } }` with HTTP 200.
  - **Error mapping:**
    - On `CounterOfficerNotFoundError` → `INTERNAL_ERROR` (HTTP 500). This should not happen if the ownership check passes, but defensive.
    - On unknown error → `INTERNAL_ERROR` (HTTP 500).
  - **Defensive try/catch** wraps the entire handler.
- **`GET` handler:**
  - Authenticated via `auth()` (no `withPermission` call).
  - Calls `auth()` to get the session.
  - Calls `getNotificationsState(session.user.userId)`.
  - Returns `{ success: true, data: <state array> }` with HTTP 200.
  - **No audit log write** — the GET is read-only.
  - **Defensive try/catch** wraps the entire handler.

### Step 5: Create `src/components/counter/notification-toggle.tsx` — The UI Component

**Goal:** Build the toggle UI for enabling and disabling push notifications.

**Files affected:** `src/components/counter/notification-toggle.tsx` (new file).

**Specific instructions:**

Create the file with:

- A `"use client"` directive at the top.
- **Props:** `{ counterId: string; counterName: string; counterNumber: number; initialEnabled: boolean; onToggle: (newValue: boolean) => void; disabled?: boolean }`.
- **Behaviour:**
  - Renders a clear toggle/switch with a label like "Push Notifications for Counter X" (where X is the counter number) or simply "Push Notifications".
  - The toggle has a visual indicator: green/on when enabled, grey/off when disabled.
  - On change:
    - If `disabled` is `true`, do nothing.
    - Otherwise, optimistically update the local state to the new value, then call `PATCH /api/officers/me/notifications` with `{ counterId, notificationsEnabled: newValue }`.
    - On success, call `onToggle(newValue)`. Show a success toast (e.g., "Push notifications enabled for Counter X").
    - On error, revert the local state to the previous value. Show an error toast with the API's error message.
  - **Loading state:** the toggle is disabled while the API call is in flight. A small spinner is shown next to the toggle.
  - **Disabled state:** when `disabled` is `true`, the toggle is greyed out and non-interactive (used when the user does not have a `CounterOfficer` record for the counter, or when the page is loading).
- **Visual design:** uses shadcn `Switch`. Uses shadcn `Toast` (or `Sonner`) for feedback.
- **Accessibility:** the toggle has a clear `aria-label` ("Toggle push notifications for Counter X, currently enabled/disabled"). The state change is announced via the toast.

  - Call `GET /api/officers/me/notifications` as the officer.
  - Verify: the response contains an array with one entry per `CounterOfficer` assignment, each with the counter details and the current toggle state.

## 8. Common Pitfalls

1. **Adding a `withPermission` call.** The toggle is a personal preference available to any counter officer. No `withPermission` is needed. Adding one would unnecessarily restrict the endpoint and create a coupling between the toggle and the permission system.

2. **Treating the toggle as a global (per-officer) setting.** The toggle is per-counter. An officer with multiple `CounterOfficer` records can have different preferences for each. The request body includes `counterId` to identify which assignment to update.

3. **Modifying the dispatch logic in 4.1.3.** The dispatch logic already reads `notificationsEnabled`. This document does NOT modify it. If the agent modifies 4.1.3, they are out of scope and may introduce regressions.

4. **Forgetting the audit log write.** The toggle is a user action that mutates state. Every successful toggle writes a `NOTIFICATIONS_TOGGLED` `AuditLog` row. Skipping the audit log breaks the compliance trail.

5. **Emitting an SSE event for the toggle.** The toggle does NOT need real-time fan-out. The dispatch reads the field on every event — there's no need to broadcast the change to other clients. Adding an SSE event would be unnecessary and could cause subtle bugs in the dashboard's event consumption.

6. **Optimistically updating without reverting on error.** The toggle component optimistically updates the local state on change. If the API call fails, the state MUST be reverted to the previous value. Failing to revert leaves the UI in an inconsistent state.

7. **Forgetting the ownership check.** The PATCH endpoint must verify that the calling user is assigned to the specified counter. Without this check, any authenticated user could toggle any other officer's notifications. The check is the `findCounterOfficerForUserAndCounter` call returning `null` → 403.

8. **Using the wrong field name.** The field is `notificationsEnabled`, not `notifications_enabled`, not `notificationsActive`, not `pushNotifications`. The Prisma client uses camelCase; the field name is exact.

9. **Mapping the API value to a database enum.** `notificationsEnabled` is a plain boolean, not an enum. The API and database both use `true` / `false`. There is no mapping step.

10. **Confusing this endpoint with the device registration endpoint.** The device registration endpoint (`POST /api/notifications/devices/register`) is from Phase 4.1.2. The notification toggle endpoint (`PATCH /api/officers/me/notifications`) is from this document. They have different paths, different methods, and different purposes. The device endpoint registers an FCM token; the toggle endpoint enables/disables dispatch for an officer-counter pair.
