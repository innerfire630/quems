## 1. Sub-Phase Purpose

Sub-Phase 4.1 is the **mobile notification foundation** of the Smart Queue Management System. It introduces Firebase Cloud Messaging (FCM) as the delivery channel that reaches counter officers on their mobile devices, the `NotificationService` server module that wraps the FCM HTTP v1 API, the device registration API that the future Android app will call, and the dispatch logic that fires notifications on ticket issuance and recall events.

Sub-Phase 4.1 is **the first sub-phase of Phase 4**. It is the producer side of the notification system — the part that creates and dispatches notifications. The consumer side (the officer dashboard, the security officer screen, the broadcast overlay) is built in later sub-phases of Phase 4 and depends on the API surface that 4.1 produces.

When this sub-phase is complete:

- The server can dispatch a push notification to a registered device token via FCM, or log to the console if FCM is not configured (so local development works without Firebase).
- A mobile device (future Android app) can register its FCM token via `POST /api/notifications/devices/register`, and the server stores the token against the calling officer's `CounterOfficer` profile.
- The system automatically deactivates invalid FCM tokens (returned by FCM with `INVALID_REGISTRATION` or `UNREGISTERED`) so that subsequent dispatches are not wasted on dead tokens.
- When a ticket is issued for a service that an on-duty officer's counter handles, that officer receives a push notification ("New ticket A005 for General").
- When a ticket is recalled, the original calling officer receives a push notification ("Ticket A005 has been recalled").
- Failed FCM dispatches are retried up to 3 times with exponential backoff (1s, 5s, 30s); non-retriable errors short-circuit immediately.

Four problems are solved in this sub-phase, in order:

1. **The FCM integration** — the Firebase project setup, the service account credential configuration, the `NotificationService` server module that wraps the FCM HTTP v1 API, the error code handling for `INVALID_REGISTRATION` / `UNREGISTERED` / `SENDER_ID_MISMATCH` / `QUOTA_EXCEEDED`, the console-log fallback for local development, and the token cleanup callback hook. (Document 4.1.1.)
2. **The device registration and mobile authentication API** — `POST /api/notifications/devices/register` (idempotent, deduplicates on token), `DELETE /api/notifications/devices/[tokenId]` (ownership-checked), the `lib/device-token.ts` module that centralises all `DeviceToken` operations, the FCM-triggered cleanup integration that deactivates tokens when FCM rejects them, and the finalisation of the `POST /api/auth/mobile/login` endpoint for mobile clients. (Document 4.1.2.)
3. **The notification dispatch and delivery system** — the trigger points (`TICKET_ISSUED` and `TICKET_RECALLED`), the recipient determination logic (find all on-duty officers with notifications enabled for the relevant counter's services), the `Notification` record creation in `PENDING` state, the FCM call wrapped in retry logic, the `Notification.status` transitions (`PENDING` → `SENT` / `FAILED`), and the integration into the existing ticket issuance (2.2.1) and recall (2.3.1) code paths via a `notifyOfficers()` helper. (Document 4.1.3.)

When this sub-phase is complete, **Phase 4.1 is finished**. The system has a complete push notification pipeline: FCM integration, device token management, and dispatch logic. The mobile app (future) is the consumer; the API surface is fully ready for it. Sub-Phases 4.2 (counter status, notification controls, officer dashboard) and 4.3 (notification reply, broadcast) build on top of the foundation that 4.1 produces.

## 1.1 Why This Sub-Phase Comes First

Sub-Phases 4.2 and 4.3 both depend on the notification system from 4.1:

- **4.2.2 (notification toggle)** writes to the `CounterOfficer.notificationsEnabled` field. The dispatch logic in 4.1.3 reads this field to decide whether to send a notification. Without 4.1.3, the toggle has no effect.
- **4.2.3 (officer dashboard)** shows the current notification toggle state and provides the toggle UI. The state is sourced from the `CounterOfficer` record that 4.1.3 reads during dispatch.
- **4.3.1 (officer reply)** is the inverse of 4.1.3 — it consumes notifications. The `Notification` and `DeviceToken` tables populated by 4.1.3 are the source of truth for what notifications exist and to whom.
- **4.3.2 (broadcast injection)** and **4.3.3 (security screen)** are independent of 4.1's server-side code, but they share the `NotificationReply` and `BroadcastMessage` data model that 4.1.3 establishes.

Sub-Phase 4.1 produces the notification foundation that the rest of Phase 4 builds on.

## 1.2 Why This Sub-Phase Comes After Phase 3

Phase 4.1 cannot exist without Phase 3:

- The SSE infrastructure from Phase 3.1 is the parallel consumer pipeline. Notifications (FCM) and real-time events (SSE) are two different delivery channels for the same domain events. 4.1.3 adds notification dispatch to existing ticket issuance and recall code paths that already broadcast SSE events; the SSE infrastructure is not modified.
- The `broadcastEvent()` function from Phase 3.1.3 is unchanged. 4.1.3 calls `notifyOfficers()` in addition to the existing `broadcastEvent()` call — they are sibling actions, not replacements.

The `DeviceToken`, `Notification`, `NotificationReply`, `BroadcastMessage`, and `CounterStatusEvent` tables are already defined in the Prisma schema (master plan Section 8.2) and migrated in Phase 1.1.3. Sub-Phase 4.1 only WRITES to these tables; it does not migrate the schema.

## 1.3 What This Sub-Phase Is NOT

- It is **not** a mobile app. No Android or iOS code is written. The future Android app is a consumer of the API surface defined here.
- It is **not** the officer dashboard UI. Phase 4.2.3.
- It is **not** the counter temporary closure system. Phase 4.2.1.
- It is **not** the broadcast message system. Phase 4.3.
- It is **not** the security officer screen. Phase 4.3.3.
- It is **not** rate limiting on the new endpoints. Phase 5.2.1.
- It is **not** PostgreSQL migration. Phase 5.3.
- It is **not** the officer reply flow. Phase 4.3.1.
- It is **not** the reports dashboard that visualises notification volume. Phase 5.1.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to distribute the complexity across the three task documents. The split is:

- **4.1.1** owns the **FCM transport integration**. It introduces the `NotificationService` server module, the FCM HTTP v1 API integration, the credential configuration, the error code mapping, the console-log fallback, and the token cleanup callback hook. It does **not** know about device registration, the `DeviceToken` table, the trigger points, or the mobile login — it just dispatches a notification when asked.
- **4.1.2** owns the **device registration and mobile authentication surface**. It introduces `lib/device-token.ts`, the `POST /api/notifications/devices/register` endpoint, the `DELETE /api/notifications/devices/[tokenId]` endpoint, the FCM-triggered cleanup integration (which subscribes to the callback from 4.1.1), and the mobile login finalisation. It depends on 4.1.1 for the callback but does not own the dispatch.
- **4.1.3** owns the **dispatch logic and trigger integration**. It introduces the `notifyOfficers()` helper, the trigger points wired into the existing ticket issuance (2.2.1) and recall (2.3.1) code paths, the recipient determination logic, the `Notification` record lifecycle, the retry wrapper, and the notification payload construction. It depends on 4.1.1 (for the service) and 4.1.2 (for the token lookup and the `Notification` model).

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 4.1.1 — FCM Integration Architecture

**Owns:** The `NotificationService` server module, the FCM HTTP v1 API integration, the credential configuration, the error code mapping, the console-log fallback, the token cleanup callback hook, and the `isFcmConfigured` flag.

**Does NOT own:** The device registration endpoints (owned by 4.1.2), the dispatch logic and trigger points (owned by 4.1.3), the `Notification` table writes (owned by 4.1.3), the mobile login endpoint (owned by 4.1.2).

**Key outputs:**

- `src/lib/notification-service.ts` — the FCM integration module:
  - `interface SendNotificationInput` — the input shape: `{ token: string; title: string; body: string; data?: Record<string, string> }`.
  - `type SendNotificationResult` — a discriminated union: `{ success: true; messageId: string; mode: 'fcm' | 'console' } | { success: false; error: 'INVALID_TOKEN' | 'CONFIGURATION_ERROR' | 'QUOTA_EXCEEDED' | 'TRANSIENT_ERROR'; message: string }`. The caller can match on `error` to decide whether to retry or deactivate the token.
  - `async function sendNotification(input: SendNotificationInput): Promise<SendNotificationResult>` — the main entry point. If `isFcmConfigured()` is false, logs the payload to `console.info` and returns a success with `mode: 'console'`. If true, calls FCM HTTP v1 API and maps the response to the result type.
  - `async function sendBatchNotifications(inputs: SendNotificationInput[]): Promise<SendNotificationResult[]>` — calls `sendNotification` for each input in parallel via `Promise.all`. Used when a single event has multiple recipients.
  - `function isFcmConfigured(): boolean` — returns true if both `FCM_SERVICE_ACCOUNT_JSON` and `FCM_PROJECT_ID` are present and the service account JSON parses successfully. Called at startup and before each send.
  - `function setOnInvalidTokenCallback(callback: (token: string) => void): void` — registers a callback that fires when FCM returns `INVALID_REGISTRATION` or `UNREGISTERED`. The callback is module-scope; setting it twice replaces the previous callback.
  - **Internal helpers** (not exported): `getAccessToken()` (exchanges service account JSON for a Google OAuth access token via JWT bearer flow), `getFcmEndpoint()` (returns the FCM v1 URL for the project), `buildFcmPayload()` (constructs the FCM request body from `SendNotificationInput`), `parseFcmError()` (maps FCM error response codes to the result `error` enum).
- **FCM HTTP v1 endpoint** — `https://fcm.googleapis.com/v1/projects/{FCM_PROJECT_ID}/messages:send`. The project ID is interpolated from the env var.
- **Service account JSON exchange** — the server creates a short-lived OAuth 2.0 access token (1 hour) by signing a JWT with the service account's private key and exchanging it at `https://oauth2.googleapis.com/token`. The token is cached in module scope and refreshed when within 5 minutes of expiry. Standard Google service account flow.
- **Error code mapping** — the FCM error responses are mapped as follows:
  - `INVALID_ARGUMENT` with `INVALID_REGISTRATION` or `UNREGISTERED` in the error details → `INVALID_TOKEN`. Invokes the cleanup callback with the token.
  - `INVALID_ARGUMENT` with `SENDER_ID_MISMATCH` → `CONFIGURATION_ERROR`. Logged via `console.error` with a "CRITICAL" prefix.
  - `RESOURCE_EXHAUSTED` with `QUOTA_EXCEEDED` → `QUOTA_EXCEEDED`. Logged via `console.warn`. The 4.1.3 retry layer treats this as deferred (no immediate retry).
  - `UNAVAILABLE`, `INTERNAL`, `DEADLINE_EXCEEDED` → `TRANSIENT_ERROR`. The 4.1.3 retry layer retries.
  - Any other 4xx → `CONFIGURATION_ERROR`. Logged with details.
  - Any other 5xx → `TRANSIENT_ERROR`. The 4.1.3 retry layer retries.
- **Console-log fallback** — when `isFcmConfigured()` is false (env vars missing or unparseable), `sendNotification` logs a one-line summary via `console.info` and returns `{ success: true, messageId: 'console-<random>', mode: 'console' }`. The `mode: 'console'` field allows callers to distinguish real FCM dispatch from logged dispatch (e.g., for tests).
- **No SSE events emitted from any 4.1.1 path.** The `NotificationService` is a one-way outbound call to FCM; it does not interact with the SSE infrastructure.

**Master Plan sections implemented:** Sections 3.3 (FCM technology choice), 4.5 (Android integration strategy), 14.1 (FCM integration overview), 14.4 (invalid token cleanup), 17 (decision on FCM).

### 4.2 Document 4.1.2 — Device Registration & Mobile Authentication API

**Owns:** The `lib/device-token.ts` module (centralised `DeviceToken` operations), the `POST /api/notifications/devices/register` endpoint (idempotent), the `DELETE /api/notifications/devices/[tokenId]` endpoint (ownership-checked), the FCM-triggered cleanup integration (subscribes to 4.1.1's callback), and the finalisation of the `POST /api/auth/mobile/login` endpoint to include the officer's full profile in the response.

**Does NOT own:** The FCM integration itself (owned by 4.1.1), the dispatch logic (owned by 4.1.3), the `Notification` record creation (owned by 4.1.3), the `AuditLog` writes for notification dispatch events (owned by 4.1.3).

**Key outputs:**

- `src/lib/device-token.ts` — the centralised `DeviceToken` module:
  - `async function registerToken({ counterOfficerId, token, platform, deviceInfo }): Promise<DeviceToken>` — idempotent registration. If a `DeviceToken` with the same `token` string already exists, the existing record is updated (`lastUsedAt` refreshed, `deviceInfo` updated if provided, `isActive` set to `true` to re-activate a previously deactivated token, `counterOfficerId` updated to the new owner if the token moved devices). The function returns the upserted record.
  - `async function removeToken(tokenId: string, requestingOfficerId: string): Promise<boolean>` — removes a `DeviceToken` by ID. Returns `true` if removed, `false` if not found. Throws an error if the token is owned by a different officer (the route handler catches this and returns 403).
  - `async function deactivateToken(token: string): Promise<void>` — sets `isActive = false` on the `DeviceToken` matching the given token string. Called by the FCM-triggered cleanup (subscribed via `setOnInvalidTokenCallback` from 4.1.1). Silent no-op if the token is not found.
  - `async function getActiveTokensForOfficer(counterOfficerId: string): Promise<DeviceToken[]>` — returns all active tokens for the given officer. Used by 4.1.3's dispatch logic to find recipients.
  - `async function getDeviceTokenByToken(token: string): Promise<DeviceToken | null>` — single-record lookup by token string. Used by the FCM-triggered cleanup to find the record to deactivate.
- `src/app/api/notifications/devices/register/route.ts` — the registration endpoint:
  - `POST` handler. Accepts `{ token, platform, deviceInfo? }`. Authenticated via `auth()` — requires a session.
  - Resolves the calling officer's `CounterOfficer` profile (the route requires the caller to have one; if not, returns 403).
  - Calls `registerToken` from `lib/device-token.ts`. Writes an `AuditLog` entry with action `DEVICE_TOKEN_REGISTERED` AFTER the upsert commits.
  - Returns the upserted `DeviceToken` record. Response: `{ success: true, data: DeviceToken }`.
- `src/app/api/notifications/devices/[tokenId]/route.ts` — the removal endpoint:
  - `DELETE` handler. Authenticated. Reads `tokenId` from the URL. Calls `removeToken` with the calling officer's `CounterOfficer` ID for ownership check. Writes an `AuditLog` entry with action `DEVICE_TOKEN_REMOVED` AFTER the delete commits.
  - Returns 204 on success, 403 if the token is owned by a different officer, 404 if not found.
- `src/app/api/auth/mobile/login/route.ts` — the mobile login finalisation:
  - The endpoint was implemented in Phase 1.2.2 (returns access + refresh tokens). Phase 4.1.2 extends the response to include the officer's full profile.
  - Response shape after finalisation: `{ accessToken, refreshToken, expiresIn, officer: { id, userId, name, email, counter: { id, name, number, displayLabel }, notificationsEnabled, currentStatus, roles, permissions } }`.
  - The officer profile is fetched in a single `findFirst` with nested `counter` and `user.roles.role.permissions` includes. The permissions array is flattened.
- `src/lib/fcm-cleanup-listener.ts` — a small bootstrap module that imports the FCM callback and subscribes the `deactivateToken` function. Called from `src/instrumentation.ts` (added by 4.1.2). Without this wiring, the FCM-triggered cleanup never fires.
- `src/schemas/notification.schema.ts` — Zod schemas for the request bodies:
  - `deviceRegisterSchema` — `{ token: string.min(10), platform: enum(['ANDROID', 'IOS', 'WEB']), deviceInfo: object.optional }`. The token must be at least 10 characters (FCM tokens are typically 100+ characters).
  - `notificationIdParamSchema` — `notificationId: string.cuid()` (for the future reply endpoint, owned by 4.3.1, but the file is created here as a forward-compatibility anchor).
- **Route protection** — the registration endpoint requires authentication (any authenticated user); it does NOT require `notification:toggle` permission at the endpoint level (any officer with a `CounterOfficer` profile can register their own devices). The removal endpoint requires authentication + ownership (the calling user must own the token). The mobile login endpoint is public.
- **Audit log writes** — `DEVICE_TOKEN_REGISTERED` and `DEVICE_TOKEN_REMOVED` are new values added to the `AuditAction` union (additive extension, per Phase 1.3.3 discipline). The FCM-triggered cleanup is a SYSTEM event, not a user action — it does NOT write an `AuditLog` entry; it just sets `isActive = false` and logs via `console.info` for diagnostics.
- **No `broadcastEvent` calls from any 4.1.2 path.** Device registration and removal are user actions that do not need real-time fan-out.

**Master Plan sections implemented:** Sections 4.5 (Android integration strategy), 9.3 (device register/delete endpoints), 10.3 (mobile authentication flow), 14.2 (FCM trigger points — referenced from 4.1.2, not implemented here), 14.4 (invalid token cleanup integration).

### 4.3 Document 4.1.3 — Notification Dispatch & Delivery System

**Owns:** The `notifyOfficers()` helper, the trigger point integration into ticket issuance (2.2.1) and recall (2.3.1), the recipient determination logic, the `Notification` record lifecycle (PENDING → SENT/FAILED), the retry wrapper with exponential backoff, the notification payload structure, and the audit log writes for dispatch events.

**Does NOT own:** The FCM transport (owned by 4.1.1), the `DeviceToken` operations (owned by 4.1.2), the `Notification` schema (owned by Phase 1.1.3), the officer dashboard UI (owned by Phase 4.2.3), the broadcast message creation (owned by Phase 4.3.2).

**Key outputs:**

- `src/lib/notification-dispatch.ts` — the dispatch logic:
  - `interface NotifyOfficersInput` — `{ ticketId, ticketNumber, serviceId, serviceName, counterId, counterName, type: 'TICKET_ISSUED' | 'TICKET_RECALLED', recipientCounterOfficerIds: string[] }`. The recipient list is pre-determined by the caller (2.2.1 or 2.3.1) and passed in.
  - `async function notifyOfficers(input: NotifyOfficersInput): Promise<NotificationDispatchSummary>` — the main entry point. For each recipient:
    1. Look up active device tokens via `getActiveTokensForOfficer` (from 4.1.2).
    2. If no active tokens, skip silently (no `Notification` record is created).
    3. If the recipient's `CounterOfficer.notificationsEnabled === false`, skip silently.
    4. Create a `Notification` record in `PENDING` state with the full payload in `data`.
    5. Call `sendNotificationWithRetry` (defined in this file) for each active token. The `Notification.status` is updated to `SENT` or `FAILED` based on the result.
  - `interface NotificationDispatchSummary` — `{ notificationsCreated: number, notificationsSent: number, notificationsFailed: number, recipientsSkipped: number, durationMs: number }`. Returned to the caller for logging.
  - `async function sendNotificationWithRetry(notificationId: string, token: string, payload: SendNotificationInput): Promise<{ success: boolean, error?: SendNotificationResult['error'] }>` — wraps `sendNotification` from 4.1.1 with retry logic. On `TRANSIENT_ERROR`, retries up to 3 times with exponential backoff (1s, 5s, 30s). On `INVALID_TOKEN`, `CONFIGURATION_ERROR`, or `QUOTA_EXCEEDED`, does NOT retry — returns the error immediately. Updates the `Notification.status` field and `fcmMessageId` after each attempt.
  - `function buildNotificationPayload(input: NotifyOfficersInput, notificationId: string): { title: string, body: string, data: Record<string, string> }` — constructs the title, body, and `data` object per master plan Section 14.3. The `data` object is the structured payload (`{ type, notificationId, ticketId, ticketNumber, serviceId, serviceName, counterId, counterName, replyUrl }`).
  - **Title and body text** — for `TICKET_ISSUED`: title = `'New Ticket'`, body = `'Ticket {ticketNumber} for {serviceName}'`. For `TICKET_RECALLED`: title = `'Ticket Recalled'`, body = `'Ticket {ticketNumber} has been recalled'`. The body is the same regardless of which counter is involved.
- `src/app/api/tickets/issue/route.ts` (modified — owned by 2.2.1): a new `notifyOfficers` call is added after the existing `broadcastEvent` call. The recipient list is determined by:
  1. Find all `Counter` records that have the issued service assigned (`CounterService` join).
  2. For each counter, find all `CounterOfficer` records with `isOnDuty = true`.
  3. Filter to officers with `notificationsEnabled = true` (the field is read here; the toggle UI to change it is Phase 4.2.2).
  4. Pass the resulting list of `CounterOfficer` IDs to `notifyOfficers`.
  5. The `notifyOfficers` call is **OUTSIDE the transaction** (after commit), per the Phase 2.2.1 discipline (broadcast events are not inside transactions).
- `src/app/api/tickets/[ticketId]/recall/route.ts` (modified — owned by 2.3.1): a new `notifyOfficers` call is added after the existing `broadcastEvent` call. The recipient is the original calling officer (`Ticket.calledByOfficerId`). If the calling officer is not set (shouldn't happen, but defensively), no notification is sent.
- **Recipient determination for issuance** — the logic in 2.2.1 is:
  - Query: `db.counterService.findMany({ where: { serviceId }, include: { counter: { include: { officers: { where: { isOnDuty: true, notificationsEnabled: true } } } } } })`.
  - Flatten the result to a list of `CounterOfficer` IDs. The list is deduplicated (an officer assigned to multiple counters handling the same service appears once).
  - If the list is empty, `notifyOfficers` is a no-op (no logs, no records).
- **Notification record creation** — happens in `notifyOfficers`, NOT in the trigger endpoint. The trigger endpoint only determines the recipient list and calls `notifyOfficers`. This separates concerns: the trigger code does not know about `Notification` records or FCM.
- **`Notification.status` transitions**:
  - On create: `PENDING` with no `sentAt`, no `fcmMessageId`, no `deliveredAt`, no `readAt`.
  - On FCM success (any attempt): `SENT` with `sentAt = now()` and `fcmMessageId = <from FCM response>`. If multiple active tokens exist for the same officer, the `Notification` is sent to each, but only the LAST success updates `fcmMessageId` (the first one wins — the FCM message ID is the unique identifier; subsequent successful sends are not separately tracked).
  - On final failure (after all retries): `FAILED` with `sentAt = now()`. The error message is logged via `console.warn` but not stored in the record (Phase 4.1 keeps the `Notification` model lean).
  - `DELIVERED` and `READ` are not transitioned in Phase 4.1 — they require a future callback endpoint that the Android app will call (Phase 5+).
- **Retry logic** — exponential backoff: `1s, 5s, 30s` between attempts. Implementation: `await new Promise((resolve) => setTimeout(resolve, delay))` between attempts. The first attempt has no delay. After the third attempt fails, the function gives up and returns failure.
- **Audit log writes** — `NOTIFICATION_DISPATCHED` (when at least one token in the dispatch succeeded) and `NOTIFICATION_FAILED` (when all tokens failed) are new values added to the `AuditAction` union. The `Notification` table itself is the detailed audit trail; the `AuditLog` captures aggregate outcomes.
- **No `broadcastEvent` calls from the dispatch path** — the dispatch is parallel to the SSE broadcast but not interleaved. The 2.2.1 and 2.3.1 routes call both `broadcastEvent` (for the display board) and `notifyOfficers` (for FCM) in sequence. The order is: broadcast first, then notify. If the broadcast fails, the notification still proceeds (they are independent).
- **No SSE events emitted from the dispatch path** — the FCM dispatch is outbound, not SSE.

### 6.4 FCM Credential Discipline (NEW — Phase 4.1 specific)

The FCM credentials are sensitive and must be handled carefully:

- The `FCM_SERVICE_ACCOUNT_JSON` env var contains the full service account JSON. It MUST be stored in a secrets manager in production (AWS Secrets Manager, GCP Secret Manager, Vercel Environment Variables) and NEVER committed to git.
- The service account JSON is loaded ONCE at module init. The parsed object is held in a module-scope variable. The private key is used only for JWT signing to obtain OAuth access tokens.
- The access token obtained from the JWT exchange is cached for 55 minutes (the Google token TTL is 60 minutes; the 5-minute buffer prevents expiry mid-request). On expiry, a new token is obtained.
- The `isFcmConfigured()` function is called at module init. If the env vars are missing or unparseable, the module logs a one-time `console.info` message ("FCM not configured — running in console-log mode") and sets a module-scope flag. Subsequent calls to `isFcmConfigured()` return false without logging.
- **In development**, the typical setup is to NOT set the FCM env vars. The console-log fallback handles this. Setting them is optional and only needed for end-to-end FCM testing.
- **In production**, the env vars MUST be set. The console-log fallback is NEVER a production configuration.

### 6.5 FCM Token Cleanup Discipline (NEW — Phase 4.1 specific)

The `DeviceToken` records have a strict lifecycle:

- **Created** via the device registration endpoint (4.1.2).
- **Refreshed** on every successful FCM dispatch (`lastUsedAt` is updated). The refresh happens in the dispatch flow (4.1.3), not in the service (4.1.1). The service returns the FCM message ID; the dispatch layer updates the `DeviceToken.lastUsedAt`.
- **Re-registered** when the same token is registered again (the existing record's `isActive` is set back to `true`, the `counterOfficerId` is updated to the new owner if the token moved devices, `lastUsedAt` is refreshed, `deviceInfo` is updated if provided). Idempotent.
- **Deactivated** when FCM returns `INVALID_REGISTRATION` or `UNREGISTERED`. The cleanup callback in 4.1.1 invokes `deactivateToken` in 4.1.2, which sets `isActive = false` and logs via `console.info`.
- **Deleted** via the device removal endpoint (4.1.2). Hard delete; the audit log captures the action.

The dispatch logic (4.1.3) **only sends to active tokens** (`isActive = true`). This is enforced at the query level in `getActiveTokensForOfficer` (which filters by `isActive: true`).

### 6.6 Notification Trigger Discipline (NEW — Phase 4.1 specific)

The `notifyOfficers()` helper is called from exactly two places:

- **Ticket issuance** (2.2.1, after the existing `broadcastEvent` call). The recipient list is all active, on-duty officers on counters handling the issued service, with `notificationsEnabled = true`.
- **Ticket recall** (2.3.1, after the existing `broadcastEvent` call). The recipient is the original calling officer (`Ticket.calledByOfficerId`).

The `notifyOfficers()` call is **OUTSIDE the database transaction** (per the Phase 2.2.1 discipline — broadcasts and notifications are not inside transactions). The transaction commits the ticket state change; then the SSE broadcast and the FCM notification are dispatched as separate steps.

If `notifyOfficers` throws an unexpected error, the error is caught and logged via `console.error` — the ticket state change still commits. The notification dispatch is best-effort relative to the ticket operation (the ticket is saved even if the notification fails).

### 6.7 Audit Logging Standard (Sub-Phase 4.1 Specific)

- `DEVICE_TOKEN_REGISTERED` and `DEVICE_TOKEN_REMOVED` are written to `AuditLog` from 4.1.2 (user actions).
- `NOTIFICATION_DISPATCHED` and `NOTIFICATION_FAILED` are written to `AuditLog` from 4.1.3 (aggregate outcomes).
- The FCM-triggered cleanup (token deactivation) is a SYSTEM event, NOT a user action — no `AuditLog` entry. Logged via `console.info` for diagnostics.
- All audit log writes are AFTER the action commits, best-effort (per Phase 1.3.3 discipline — failures don't fail the action).
- The `Notification` table itself is the detailed audit trail for individual dispatch attempts. It records the recipient, payload, status, and FCM message ID for each attempt. The `AuditLog` captures user actions and aggregate outcomes only.

### 6.8 Broadcast Event Seam (Sub-Phase 4.1 Specific)

- 4.1.3 does NOT emit any SSE events. The FCM dispatch is outbound, not SSE.
- 4.1.2 does NOT emit any SSE events. Device registration and removal are user actions that do not need real-time fan-out.
- 4.1.1 does NOT emit any SSE events. The FCM service is outbound.
- The existing SSE events from Phase 2 (TICKET_ISSUED, TICKET_CALLED, TICKET_RECALLED, etc.) are UNCHANGED. The 2.2.1 and 2.3.1 routes call `broadcastEvent` and `notifyOfficers` as sibling actions.

### 6.9 Environment Variable Standards

Two new env vars are introduced in 4.1.1:

- `FCM_SERVICE_ACCOUNT_JSON` — the full service account JSON as a string. Optional in development, REQUIRED in production. Example: `'{"type":"service_account","project_id":"my-project","private_key_id":"...","private_key":"-----BEGIN PRIVATE KEY-----\n...","client_email":"...","client_id":"...","auth_uri":"...","token_uri":"...","auth_provider_x509_cert_url":"...","client_x509_cert_url":"..."}'`. The newlines in `private_key` MUST be preserved as literal `\n` characters in the env var (most secrets managers handle this correctly).
- `FCM_PROJECT_ID` — the Firebase project ID. Optional in development, REQUIRED in production. Example: `'my-firebase-project'`.

Both env vars are added to `src/lib/env.ts` (extended from Phase 1.1) as optional strings with a clear comment block. The validator does NOT throw if they are missing — the service falls back to console-log mode. The validator parses the JSON at startup and logs a `console.info` message about the configuration state.

### 6.10 Transaction Boundaries

- 4.1.1 has no database transactions (it is a pure outbound service).
- 4.1.2's `registerToken` is a single `upsert` call (not a transaction — `upsert` is atomic). The `removeToken` is a single `delete` call.
- 4.1.3's dispatch logic creates a `Notification` record per recipient (one row each), then sends to FCM. The `Notification` writes are NOT in a transaction with each other (they are independent — partial success is acceptable; a failed write to one recipient's notification does not roll back others).
- The `notifyOfficers` call is OUTSIDE the ticket operation's transaction (per 6.6 above).

### 6.11 Rate Limiting

- The new notification endpoints (`/api/notifications/devices/register`, `/api/notifications/devices/[tokenId]`) are NOT rate-limited in Phase 4.1. Phase 5.2.1 adds the rate limits (10 requests per minute for device registration, 30 requests per minute for device removal — the exact values are in master plan Section 15.3 and the Phase 5.2.1 task plan).
- The `/api/auth/mobile/login` endpoint is rate-limited in Phase 5.2.1 (10 requests per minute per IP, per master plan Section 15.3). Phase 4.1.2 does not add rate limiting to this endpoint.
- The FCM HTTP v1 API has its own rate limits (600 requests per minute per project by default). The `sendNotification` function does not implement client-side rate limiting — FCM returns `QUOTA_EXCEEDED` if the limit is hit, and the dispatch layer treats this as deferred.

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`.
- Each task plan document corresponds to one or more commits.
- Suggested commit boundaries:
  - End of 4.1.1: `feat: add FCM NotificationService with error code handling and console-log fallback`.
  - End of 4.1.2: `feat: add device registration API and FCM-triggered token cleanup integration`.
  - End of 4.1.3: `feat: add notification dispatch with trigger points, retry logic, and audit logging`.

### 8.2 General Guidance

- **Do not skip steps.** Sub-Phase 4.1 is mostly sequential within each document (each step's output is the next step's input). 4.1.1 must be done before 4.1.2 (the cleanup callback is exported from 4.1.1). 4.1.1 and 4.1.2 must be done before 4.1.3 (the dispatch depends on both).
- **Do not add features beyond the document's scope.** The temptation to "just add an admin UI for viewing notifications" or "just add iOS support" is real — resist it. Those are owned by later phases or future enhancements.
- **The FCM service is server-side only.** No client-side code. The Android app (future) is the FCM receiver, but it is not built in this phase.
- **The console-log fallback is a first-class feature.** It allows local development without Firebase setup. Do not gate it behind an env var like `NODE_ENV === 'development'` — the absence of FCM env vars is the trigger.
- **Verify after every step.** The verification checklist at the end of each document is a hard gate.
- **When the master plan is ambiguous, choose the simpler option.** If a configuration choice has two reasonable interpretations, pick the one that is easier to change later.
- **Commit at the end of each document** using the commit message specified in the document.
- **Document any deviation.** If the agent makes a choice that differs from this document, it must note the deviation in the final summary so the human reviewer can approve or reject it.
- **The FCM error code handling is critical.** Get it right. The wrong mapping (e.g., treating `INVALID_REGISTRATION` as `TRANSIENT_ERROR` and retrying) would cause the system to waste FCM quota on dead tokens and never deactivate them.
- **The token cleanup callback is one-way.** 4.1.1 invokes the callback; 4.1.2 receives it. The callback is a simple `(token: string) => void` — no return value, no async (the cleanup is fire-and-forget). If 4.1.2's `deactivateToken` throws, the error is caught and logged in 4.1.1; it does not propagate.
- **The retry logic is in 4.1.3, not 4.1.1.** The service returns `TRANSIENT_ERROR` on retryable failures; the dispatch layer decides whether to retry. This separation keeps the service stateless and the retry policy easy to test.
- **The `notifyOfficers` call is OUTSIDE the transaction.** This is the same discipline as the SSE broadcast in Phase 2.2.1. The ticket operation commits first; then the broadcast and the notification are dispatched.

### 8.4 FCM Configuration Notes

- **The service account JSON must be valid.** A malformed JSON (e.g., missing `private_key` field) should be treated as "not configured" — the service falls back to console-log mode. The agent should validate the JSON at module init, not at first send.
- **The OAuth access token exchange is the standard Google service account flow.** The agent can use the `jsonwebtoken` library (already in dependencies via `next-auth/jwt`) to sign the JWT, then POST to `https://oauth2.googleapis.com/token` with `grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer&assertion=<jwt>`. The response is `{ access_token, expires_in, token_type }`.
- **The FCM HTTP v1 endpoint is `https://fcm.googleapis.com/v1/projects/{projectId}/messages:send`.** The request body has the shape `{ message: { token, notification: { title, body }, data: { ... } } }`. The `data` values are strings (FCM requires string values).
- **FCM error responses have the shape `{ error: { code, message, status, details: [{ errorCode, '@type' }] } }`.** The agent parses the `details[].errorCode` for the specific error (e.g., `messaging/registration-token-not-registered`).
- **The `mode: 'console'` field in the success result is for testability.** Tests can assert that a test environment runs in console mode and avoid the FCM HTTP call.

### 8.5 The Console-Log Fallback is a Real Feature

The console-log fallback is not a "TODO" or a "production hack" — it is a designed-in feature that enables local development without Firebase. The agent should:

- Log a one-time `console.info` message at module init: `"FCM not configured — running in console-log mode. Set FCM_SERVICE_ACCOUNT_JSON and FCM_PROJECT_ID to enable real dispatch."`
- Log each send in console mode with the full payload (ticket number, service, counter, recipient) via `console.info`. The log format is: `[FCM console] type=TICKET_ISSUED ticket=A005 service=General counter="Counter 2" mode=console`.
- Return `{ success: true, messageId: 'console-<random>', mode: 'console' }` so the dispatch layer can track the dispatch as a success in the `Notification` table.

### 8.6 The Recipient Determination for Ticket Issuance

The recipient list for ticket issuance is computed in the 2.2.1 endpoint (NOT in 4.1.3). The query is:

- Start with the issued `serviceId`.
- Find all `Counter` records that have this service assigned (via `CounterService` join).
- For each counter, find all `CounterOfficer` records with `isOnDuty = true` AND `notificationsEnabled = true`.
- Flatten to a deduplicated list of `CounterOfficer` IDs.

If the list is empty (no on-duty officers with notifications enabled for counters handling the service), `notifyOfficers` is a no-op. No `Notification` records are created. The ticket issuance itself still succeeds — the notification is a parallel action.

### 8.7 The Notification Record Schema

The `Notification` table (master plan Section 8.2) has these fields:

- `id` — primary key (cuid, auto-generated).
- `ticketId` — the ticket this notification is about.
- `counterOfficerId` — the recipient officer.
- `type` — `NotificationType` enum: `TICKET_ISSUED` or `TICKET_RECALLED` (the other values, `SYSTEM_MESSAGE` and `BROADCAST`, are future).
- `title` — string (e.g., "New Ticket").
- `body` — string (e.g., "Ticket A005 for General").
- `data` — JSON (the structured payload per master plan Section 14.3).
- `status` — `NotificationStatus` enum, default `PENDING`.
- `fcmMessageId` — nullable string (set on success).
- `sentAt` — nullable datetime (set on success or final failure).
- `deliveredAt` — nullable datetime (NOT set in Phase 4.1).
- `readAt` — nullable datetime (NOT set in Phase 4.1).
- `createdAt` — auto-set by Prisma.

### 8.8 The Token Cleanup Integration

The token cleanup flow is a chain of three components:

1. **4.1.1's `NotificationService`** invokes the cleanup callback when FCM returns `INVALID_REGISTRATION` or `UNREGISTERED`. The callback is `(token: string) => void` — fire-and-forget.
2. **4.1.2's `deactivateToken`** is the registered callback. It looks up the `DeviceToken` by token string and sets `isActive = false`.
3. **4.1.2's `fcm-cleanup-listener.ts`** wires the two together at module init. This file imports `setOnInvalidTokenCallback` from 4.1.1 and `deactivateToken` from 4.1.2, and calls `setOnInvalidTokenCallback(deactivateToken)`.

The `fcm-cleanup-listener.ts` is imported once in `src/instrumentation.ts` (so the subscription is active on app boot). The import is a side-effect import (`import '@/lib/fcm-cleanup-listener'`) — the module's top-level code runs the subscription.

### 8.9 Testing Notes

- The FCM service can be tested end-to-end WITHOUT a real Firebase project. The console-log fallback covers the case where env vars are missing. The agent should run the dev server without the env vars to verify the fallback.
- To test with real FCM, the agent needs:
  - A Firebase project (free tier is sufficient).
  - A service account JSON key.
  - A real device or emulator with a registered FCM token. For automated testing, this is impractical — the agent should rely on the console-log fallback for automated tests and use real FCM only for manual verification.
- The `lib/notification-service.ts` module is designed to be testable. The agent can write unit tests that mock the FCM HTTP call (using `fetch` mock or a dependency injection pattern). The verification checklist includes a "test the error code mapping" step.
- The `lib/notification-dispatch.ts` module can be unit tested with a mocked `sendNotification` function. The retry logic is testable with a fake timer.
