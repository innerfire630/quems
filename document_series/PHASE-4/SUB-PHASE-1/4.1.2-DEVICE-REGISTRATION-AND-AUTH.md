## 1. Document Purpose

This document delivers the **device token management surface** and the **mobile authentication finalisation** for the push notification system. It introduces the centralised `lib/device-token.ts` module that owns all `DeviceToken` table operations, the `POST /api/notifications/devices/register` endpoint (idempotent, deduplicates on token), the `DELETE /api/notifications/devices/[tokenId]` endpoint (ownership-checked), the FCM-triggered cleanup integration that subscribes to the invalid-token callback from 4.1.1, and the finalisation of the `POST /api/auth/mobile/login` endpoint to include the officer's full profile in the response (so the future Android app has everything it needs in one call after login).

When this document is complete:

- A device can be registered via `POST /api/notifications/devices/register`. Duplicate tokens are deduplicated (the same token string returns the same `DeviceToken` record). Previously-deactivated tokens are re-activated.
- A device can be removed via `DELETE /api/notifications/devices/[tokenId]`. Officers can only remove their own tokens.
- When FCM returns `INVALID_REGISTRATION` or `UNREGISTERED` for a token (per 4.1.1's mapping), the corresponding `DeviceToken.isActive` is set to `false` within 1 second, via the FCM cleanup callback.
- The mobile login endpoint returns a complete officer profile (name, email, assigned counter, notification toggle state, roles, permissions) so the Android app has no follow-up calls to make.
- All device registration and removal actions are written to `AuditLog` (with `DEVICE_TOKEN_REGISTERED` and `DEVICE_TOKEN_REMOVED` actions) after the database write commits.

This document does **not** build the dispatch logic (owned by 4.1.3), the FCM transport (owned by 4.1.1), the counter status endpoints (owned by 4.2.1), the notification toggle endpoint (owned by 4.2.2), the officer reply (owned by 4.3.1), or the broadcast system (owned by 4.3.2).

### Step 1: Define the Zod Schemas for Notification Endpoints

**Goal:** Establish the validation schemas for the device registration request and (forward-compatibility anchor) the future reply request.

**Files affected:** `src/schemas/notification.schema.ts` (new file).

**Specific instructions:**

Create `src/schemas/notification.schema.ts` with the following Zod schemas:

- **`deviceRegisterSchema`** — the request body for `POST /api/notifications/devices/register`:
  - `token: z.string().min(10).max(500)` — the FCM registration token. Minimum 10 characters (FCM tokens are typically 100+); maximum 500 (defensive upper bound).
  - `platform: z.enum(['ANDROID', 'IOS', 'WEB'])` — the device platform. Uses the Prisma `Platform` enum values as the Zod enum literals.
  - `deviceInfo: z.object({}).passthrough().optional()` — optional device metadata. The schema is `z.object({}).passthrough()` so any keys are allowed (e.g., `{ model: 'Pixel 7', osVersion: '14', appVersion: '1.0.0' }`).
- **`deviceTokenIdParamSchema`** — the URL parameter for `DELETE /api/notifications/devices/[tokenId]`:
  - `tokenId: z.string().min(1)` — the `DeviceToken.id` (a cuid). The schema does not validate the cuid format (cuids are arbitrary strings).
- **`notificationIdParamSchema`** — the URL parameter for the FUTURE `POST /api/notifications/[notificationId]/reply` endpoint (owned by 4.3.1). Created here as a forward-compatibility anchor:
  - `notificationId: z.string().min(1)` — the `Notification.id` (a cuid).
- Export all three schemas.

### Step 2: Create the `lib/device-token.ts` Module

**Goal:** Centralise all `DeviceToken` table operations in a single module that all device-token-related code paths use.

**Files affected:** `src/lib/device-token.ts` (new file).

**Specific instructions:**

Create `src/lib/device-token.ts` with the following exported functions:

- **`async function registerToken(input: { counterOfficerId: string; token: string; platform: 'ANDROID' | 'IOS' | 'WEB'; deviceInfo?: Record<string, unknown> }): Promise<DeviceToken>`**:
  - Performs an idempotent upsert on the `DeviceToken` table:
    - Search: `db.deviceToken.findUnique({ where: { token: input.token } })`.
    - If found:
      - Update the record: `db.deviceToken.update({ where: { id: existing.id }, data: { counterOfficerId: input.counterOfficerId, platform: input.platform, deviceInfo: input.deviceInfo ?? existing.deviceInfo, isActive: true, lastUsedAt: null } })`. The `isActive` is forced to `true` (re-activate a previously deactivated token). The `lastUsedAt` is set to `null` (it will be refreshed on the next successful FCM dispatch by 4.1.3, not at registration time).
    - If not found:
      - Create the record: `db.deviceToken.create({ data: { counterOfficerId: input.counterOfficerId, token: input.token, platform: input.platform, deviceInfo: input.deviceInfo, isActive: true } })`.
  - Returns the upserted `DeviceToken` record.
  - The function does NOT write to `AuditLog` (the route handler does that, after the function returns).
- **`async function removeToken(tokenId: string, requestingOfficerId: string): Promise<boolean>`**:
  - Looks up the token: `db.deviceToken.findUnique({ where: { id: tokenId }, include: { counterOfficer: true } })`.
  - If not found, returns `false`.
  - If found but the token's `counterOfficer.userId !== requestingOfficerId`, throws a `DeviceTokenOwnershipError` (a custom error class defined in this module). The route handler catches this and returns 403.
  - If owned, deletes: `db.deviceToken.delete({ where: { id: tokenId } })`.
  - Returns `true` on success.
- **`async function deactivateToken(token: string): Promise<void>`**:
  - Looks up the token: `db.deviceToken.findUnique({ where: { token } })`.
  - If found AND `isActive === true`, sets `isActive = false` via `db.deviceToken.update({ where: { id: existing.id }, data: { isActive: false } })`.
  - If not found OR already inactive, returns silently (no-op).
  - Logs the deactivation via `console.info`: `[FCM cleanup] Deactivated token: ${token.substring(0, 8)}...` (the token is truncated for log hygiene).
  - The function NEVER throws. Errors are caught and logged via `console.error`.
- **`async function getActiveTokensForOfficer(counterOfficerId: string): Promise<DeviceToken[]>`**:
  - Returns all active tokens for the officer: `db.deviceToken.findMany({ where: { counterOfficerId, isActive: true }, orderBy: { lastUsedAt: 'desc' } } )`.
  - The ordering is `lastUsedAt desc` — the most-recently-used token is first. This is useful for the dispatch layer (4.1.3) which sends to all active tokens; the ordering is a soft preference, not a strict rule.
- **`async function getDeviceTokenByToken(token: string): Promise<DeviceToken | null>`**:
  - Single-record lookup by token string. Returns the `DeviceToken` or `null`.
  - Used by the cleanup path and by tests that need to verify state.

Export all five functions and the `DeviceTokenOwnershipError` class.

### Step 3: Create the Device Registration Endpoint

**Goal:** Expose the `POST /api/notifications/devices/register` endpoint for the Android app to register FCM tokens.

**Files affected:** `src/app/api/notifications/devices/register/route.ts` (new file).

**Specific instructions:**

Create `src/app/api/notifications/devices/register/route.ts` with the `POST` handler:

- **Auth check:** Calls `auth()` from `@/lib/auth`. If no session, returns `{ success: false, error: { code: 'UNAUTHORIZED', message: 'Authentication required.' } }` with HTTP 401.
- **Officer profile lookup:** Resolves the calling user's `CounterOfficer` record: `db.counterOfficer.findFirst({ where: { userId: session.user.userId } })`. If not found, returns `{ success: false, error: { code: 'FORBIDDEN', message: 'Only counter officers can register devices.' } }` with HTTP 403. (Any user with a `CounterOfficer` record can register; no `withPermission` check needed — the permission system is for fine-grained module-level access, not for this self-service operation.)
- **Body validation:** Validates the request body via `deviceRegisterSchema.parse(body)`. On Zod failure, returns `{ success: false, error: { code: 'VALIDATION_ERROR', message: 'Invalid request body.', details: zodError.errors } }` with HTTP 422.
- **Registration call:** Calls `registerToken` from `@/lib/device-token` with the resolved `counterOfficerId`, the token, the platform, and the deviceInfo. Receives the upserted `DeviceToken` record.
- **Audit log write:** Calls `writeAuditLog({ action: 'DEVICE_TOKEN_REGISTERED', actorId: session.user.userId, entity: 'DeviceToken', entityId: deviceToken.id, description: \`Device token registered for officer \${session.user.name}\`, metadata: { platform: input.platform, isReactivation: <true|false> } })`AFTER the upsert commits. The`isReactivation`flag is`true`if the token previously existed with`isActive = false` and was re-activated. The audit log write is best-effort (failures are caught and logged but do not fail the registration).
- **Response:** Returns `{ success: true, data: deviceToken }` with HTTP 200. The response includes the full `DeviceToken` record (id, counterOfficerId, token, platform, deviceInfo, isActive, createdAt, updatedAt).
- **Route options:** `export const runtime = 'nodejs'` (uses Prisma); `export const dynamic = 'force-dynamic'` (no caching).

### Step 4: Create the Device Removal Endpoint

**Goal:** Expose the `DELETE /api/notifications/devices/[tokenId]` endpoint for the Android app to remove FCM tokens (e.g., on logout).

**Files affected:** `src/app/api/notifications/devices/[tokenId]/route.ts` (new file).

**Specific instructions:**

Create `src/app/api/notifications/devices/[tokenId]/route.ts` with the `DELETE` handler:

- **Auth check:** Calls `auth()`. If no session, returns 401.
- **URL parameter validation:** Validates the `tokenId` parameter via `deviceTokenIdParamSchema.parse(params)`. On Zod failure, returns 422.
- **Removal call:** Calls `removeToken` from `@/lib/device-token` with the `tokenId` and the calling user's ID. Handles the two failure cases:
  - `DeviceTokenOwnershipError` → returns `{ success: false, error: { code: 'FORBIDDEN', message: 'You can only remove your own device tokens.' } }` with HTTP 403.
  - Returns `false` (token not found) → returns `{ success: false, error: { code: 'NOT_FOUND', message: 'Device token not found.' } }` with HTTP 404.
- **Audit log write:** Calls `writeAuditLog({ action: 'DEVICE_TOKEN_REMOVED', actorId: session.user.userId, entity: 'DeviceToken', entityId: tokenId, description: \`Device token removed by officer \${session.user.name}\` })` AFTER the delete commits. Best-effort.
- **Response:** Returns HTTP 204 (No Content) on success. No body.
- **Route options:** `runtime = 'nodejs'`, `dynamic = 'force-dynamic'`.

### Step 8: Finalise the Mobile Login Endpoint

**Goal:** Extend the response of `POST /api/auth/mobile/login` to include the officer's full profile.

**Files affected:** `src/app/api/auth/mobile/login/route.ts` (modified).

**Specific instructions:**

- Open the existing route file (created in Phase 1.2.2).
- The existing handler validates credentials, generates access + refresh tokens, and returns `{ accessToken, refreshToken, expiresIn }`. Phase 4.1.2 does NOT change any of this — the credential validation and token generation logic is unchanged.
- **Add a profile fetch** BEFORE the response is returned:
  - Query: `db.counterOfficer.findFirst({ where: { userId: <userId from credential validation> }, include: { counter: true, user: { include: { roles: { include: { role: { include: { permissions: { include: { permission: true } } } } } } } } } })`.
  - If no `CounterOfficer` record is found (the user is not a counter officer), the response is the original `{ accessToken, refreshToken, expiresIn }` without the officer profile. This is the case for admin users, security officers, etc. — they can log in via the mobile endpoint but get no officer-specific data.
  - If a `CounterOfficer` record is found, build the `officer` object:
    - `id`: the `CounterOfficer.id`.
    - `userId`: the user ID.
    - `name`: the user's name.
    - `email`: the user's email.
    - `counter`: `{ id, name, number, displayLabel }` — the assigned counter (if any). If no counter, the field is `null`.
    - `notificationsEnabled`: the `CounterOfficer.notificationsEnabled` field.
    - `currentStatus`: the `CounterOfficer.currentStatus` enum value.
    - `roles`: an array of role names (e.g., `['COUNTER_OFFICER']`).
    - `permissions`: a flattened array of permission strings (deduplicated, e.g., `['counter:read', 'counter:call', 'counter:close', 'ticket:view', 'notification:toggle', 'notification:reply']`).
- **Update the response** to include the `officer` field (or omit it if no `CounterOfficer` profile).
- The response shape is now: `{ accessToken, refreshToken, expiresIn, officer: <OfficerProfile> | undefined }`. The `officer` field is always present in the response object (with `null` value if no profile) so the Android app has a stable shape.
- The `expiresIn` field is the same as before (900 seconds, matching the 15-minute access token TTL).

## 8. Common Pitfalls

The agent should be aware of the following common mistakes and avoid them:

- **Not making the registration endpoint idempotent.** A duplicate token (e.g., the Android app re-registers on every launch) MUST NOT create duplicate `DeviceToken` rows. The `registerToken` function uses `findUnique` + `update`/`create` to ensure exactly one row per token string.
- **Forgetting to re-activate a previously deactivated token.** When a user re-installs the Android app and re-registers the same FCM token, the token may have been previously deactivated. The `registerToken` function MUST set `isActive = true` on the upsert.
- **Not validating ownership on the removal endpoint.** Without the ownership check, one officer could remove another officer's device tokens, which is a security issue.
- **Throwing from `deactivateToken`.** The function is called from the FCM cleanup callback (a fire-and-forget invocation). If it throws, the error propagates back to `sendNotification` and may break the dispatch. The function catches all errors and logs them.
- **Writing the audit log BEFORE the database write commits.** The audit log must be written AFTER the action commits (per the Phase 1.3.3 discipline). Otherwise, a failed commit leaves an orphan audit log entry.
- **Modifying the credential validation in the mobile login endpoint.** The Phase 1.2.2 logic is correct and tested. Phase 4.1.2 ONLY extends the response. Any change to the credential validation is a regression.
- **Hardcoding the permission list in the mobile login response.** The permissions are derived from the `RolePermission` table via the `user.roles.role.permissions.permission` join. The list is the live set, not a hardcoded subset.
- **Not handling the case where the user has no `CounterOfficer` record.** The mobile login endpoint must work for all users (admin, security officer, etc.), not just counter officers. The `officer` field is `null` when no profile exists.
- **Importing `@/lib/fcm-cleanup-listener` from multiple files.** It should be imported ONCE in `instrumentation.ts`. Multiple imports do no harm (the subscription is idempotent — the last call to `setOnInvalidTokenCallback` wins), but a single import is cleaner.
- **Not preserving the existing mobile login response shape.** The existing `accessToken`, `refreshToken`, `expiresIn` fields are unchanged. The `officer` field is ADDED, not replacing anything. The Android app integration depends on this stability.

### 9.2 The Idempotent Registration Pattern

The `registerToken` function is the canonical example of an idempotent registration pattern in this codebase. The pattern is:

1. Look up the record by its unique key (in this case, the FCM token string).
2. If found, update the existing record (re-activate, update deviceInfo, update owner if the token moved devices).
3. If not found, create a new record.
4. Return the upserted record.

This pattern is used in other places (e.g., the system settings seed from 2.3.3). The agent should follow this pattern for any future idempotent registration endpoints.

### 9.3 The Ownership Check Pattern

The `removeToken` function demonstrates the canonical ownership check pattern for the codebase:

1. Look up the record.
2. If not found, return `false` (caller maps to 404).
3. If the record's owner does not match the requesting user, throw a custom error (caller maps to 403).
4. If owned, perform the action.

This pattern is reused in Phase 4.3.1 (officer reply — only the recipient officer can reply to a notification). The agent should use this pattern for any future ownership-checked endpoints.

### 9.4 The Mobile Login Response Shape

The mobile login response is the single most important API for the future Android app. After login, the app has:

- `accessToken` — for all subsequent API requests.
- `refreshToken` — for refreshing the access token.
- `expiresIn` — the access token TTL (900 seconds = 15 minutes).
- `officer` — the full profile (counter, notification toggle, roles, permissions).

The `officer` object is designed to give the app EVERYTHING it needs in one call. The app does NOT need to make a follow-up `GET /api/officers/me` call after login. This is a critical design decision that minimises the app's complexity.

### 9.5 The FCM Cleanup Chain

The FCM cleanup is a chain of three modules:

1. **`NotificationService` (4.1.1)** invokes the cleanup callback when FCM returns `INVALID_TOKEN`.
2. **`fcm-cleanup-listener` (4.1.2)** subscribes the `deactivateToken` function to the callback.
3. **`deactivateToken` (4.1.2)** sets `isActive = false` on the corresponding `DeviceToken` record.

The chain is wired by importing `fcm-cleanup-listener` in `instrumentation.ts` (which runs on app boot). Without this import, dead FCM tokens are never deactivated, and the dispatch layer (4.1.3) will waste FCM quota sending to them.

### 9.6 The Audit Log Best-Effort Pattern

The audit log writes (for `DEVICE_TOKEN_REGISTERED` and `DEVICE_TOKEN_REMOVED`) are best-effort: they happen AFTER the database action commits, and any failure in the audit log write is caught and logged but does not fail the user-facing action. This is the standard pattern from Phase 1.3.3.

The agent wraps the `writeAuditLog` call in a try/catch:

- The `writeAuditLog` function itself catches errors internally (per Phase 1.3.3).
- The agent's route handler does NOT need an additional try/catch around the `writeAuditLog` call.
- The agent MAY add a `console.warn` if `writeAuditLog` throws (for diagnostics), but the user-facing response is unaffected.

### 9.7 The `deviceInfo` JSON Field

The `DeviceToken.deviceInfo` field is a JSON column (nullable). The agent stores whatever the Android app sends — the schema is `z.object({}).passthrough()` to allow any keys. Typical fields include `model` (e.g., "Pixel 7"), `osVersion` (e.g., "14"), `appVersion` (e.g., "1.0.0"). The agent does NOT validate the contents (only the shape). The field is for diagnostics, not for logic.

### 9.8 When the Master Plan is Ambiguous

- The master plan does not specify whether device registration is idempotent. The agent makes it idempotent based on the requirement that re-registering the same token MUST NOT create duplicates (a natural consequence of the unique constraint on `token`).
- The master plan does not specify the response shape of the mobile login endpoint. The agent uses the documented shape (which includes the officer profile) based on the requirement that the Android app should have everything it needs in one call.
- The master plan does not specify the exact wording of the audit log description. The agent uses the documented wording (`"Device token registered for officer X"`, `"Device token removed by officer X"`).
- The master plan does not specify the error message for the ownership check. The agent uses the documented wording (`"You can only remove your own device tokens."`).
