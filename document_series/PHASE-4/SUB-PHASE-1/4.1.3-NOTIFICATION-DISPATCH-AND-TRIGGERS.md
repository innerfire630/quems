## 1. Document Purpose

This document delivers the **notification dispatch and delivery system** for the push notification pipeline. It introduces the `notifyOfficers()` helper that orchestrates the end-to-end dispatch flow, the trigger point integration into the existing ticket issuance (Phase 2.2.1) and recall (Phase 2.3.1) code paths, the recipient determination logic, the `Notification` record lifecycle (`PENDING` → `SENT` / `FAILED`), the retry wrapper with exponential backoff, the notification payload structure (per master plan Section 14.3), and the audit log writes for dispatch events.

When this document is complete:

- Issuing a ticket (`POST /api/tickets/issue`) creates `Notification` records in `PENDING` for all eligible recipients (active, on-duty officers on counters handling the issued service, with `notificationsEnabled = true`). The records are then dispatched to FCM.
- Recalling a ticket (`POST /api/tickets/[ticketId]/recall`) creates a `Notification` record in `PENDING` for the original calling officer. The record is then dispatched to FCM.
- `Notification.status` transitions to `SENT` on FCM success (with `sentAt` and `fcmMessageId` populated) or `FAILED` on final failure (with `sentAt` populated). The `DELIVERED` and `READ` states are not transitioned in Phase 4.1.
- Failed dispatches with `TRANSIENT_ERROR` are retried up to 3 times with delays of 1s, 5s, 30s between attempts. Non-retriable errors (`INVALID_TOKEN`, `CONFIGURATION_ERROR`, `QUOTA_EXCEEDED`) short-circuit immediately.
- Officers with `notificationsEnabled = false` are skipped entirely (no `Notification` record is created).
- Officers with no active device tokens are skipped silently (no `Notification` record is created).
- The `notifyOfficers()` call is OUTSIDE the ticket operation's transaction (per the Phase 2.2.1 discipline — broadcasts and notifications are not inside transactions).
- Audit log entries are written for `NOTIFICATION_DISPATCHED` (at least one success) and `NOTIFICATION_FAILED` (all failures).

This document does **not** build the FCM transport (owned by 4.1.1), the device registration endpoints (owned by 4.1.2), the counter status toggle (owned by 4.2.1), the notification toggle endpoint (owned by 4.2.2), the officer reply (owned by 4.3.1), or the broadcast system (owned by 4.3.2).

### Step 1: Define the Dispatch Types

**Goal:** Establish the typed interface that the trigger endpoints (2.2.1 and 2.3.1) will use to invoke the dispatch.

**Files affected:** `src/lib/notification-dispatch.ts` (new file, initial scaffold).

**Specific instructions:**

Create `src/lib/notification-dispatch.ts` with the following type declarations at the top:

- **`NotifyOfficersInput`** — the input shape for `notifyOfficers`:
  - `ticketId: string` — the ticket ID.
  - `ticketNumber: string` — the displayed ticket number (e.g., `'A005'`).
  - `serviceId: string` — the service ID.
  - `serviceName: string` — the service name (e.g., `'General Inquiry'`).
  - `counterId: string` — the counter ID (the counter the ticket was called to, or the counter the ticket was issued for; for issuance, this is the first eligible counter; for recall, this is the original counter).
  - `counterName: string` — the counter display name (e.g., `'Counter 2'`).
  - `type: 'TICKET_ISSUED' | 'TICKET_RECALLED'` — the notification category.
  - `recipientCounterOfficerIds: string[]` — the pre-determined list of `CounterOfficer` IDs to notify. The caller (2.2.1 or 2.3.1) is responsible for determining this list.
- **`NotificationDispatchSummary`** — the output shape for `notifyOfficers`:
  - `notificationsCreated: number` — the count of `Notification` records created.
  - `notificationsSent: number` — the count of dispatches that succeeded (status transitioned to `SENT`).
  - `notificationsFailed: number` — the count of dispatches that failed after all retries (status transitioned to `FAILED`).
  - `recipientsSkipped: number` — the count of recipients that were skipped (no active tokens OR `notificationsEnabled = false`).
  - `durationMs: number` — the total duration of the dispatch in milliseconds.
- **`RETRY_DELAYS_MS`** — a module-scope constant: `readonly number[] = [1000, 5000, 30000]` (1s, 5s, 30s).
- **`MAX_RETRY_ATTEMPTS`** — a module-scope constant: `number = 3` (one initial attempt + up to 3 retries = 4 total attempts).
- **`NOTIFICATION_TITLE_BY_TYPE`** — a constant: `{ TICKET_ISSUED: 'New Ticket', TICKET_RECALLED: 'Ticket Recalled' }`.
- **`buildNotificationPayload(input: NotifyOfficersInput, notificationId: string): { title: string; body: string; data: Record<string, string> }`**:
  - Constructs the title from `NOTIFICATION_TITLE_BY_TYPE[input.type]`.
  - Constructs the body based on the type:
    - `TICKET_ISSUED`: `'Ticket ${input.ticketNumber} for ${input.serviceName}'`.
    - `TICKET_RECALLED`: `'Ticket ${input.ticketNumber} has been recalled'`.
  - Constructs the `data` object per master plan Section 14.3:
    - `type`: input.type.
    - `notificationId`: notificationId.
    - `ticketId`: input.ticketId.
    - `ticketNumber`: input.ticketNumber.
    - `serviceId`: input.serviceId.
    - `serviceName`: input.serviceName.
    - `counterId`: input.counterId.
    - `counterName`: input.counterName.
    - `replyUrl`: `'/api/notifications/' + notificationId + '/reply'`.
  - All `data` values are strings (FCM requirement).
- Export all types and constants and the `buildNotificationPayload` function.

### Step 2: Implement the Retry Wrapper

**Goal:** Wrap the `sendNotification` function with retry logic for transient errors.

**Files affected:** `src/lib/notification-dispatch.ts` (extended).

**Specific instructions:**

- Add the function `async function sendNotificationWithRetry(notificationId: string, token: string, payload: SendNotificationInput): Promise<{ success: boolean; error?: SendNotificationResult['error']; fcmMessageId?: string }>` (NOT exported — internal helper):
  - **First attempt:** Calls `sendNotification(payload)` (from 4.1.1).
  - If the result is `success: true`, returns `{ success: true, fcmMessageId: result.messageId }`. Updates the `Notification.status` to `SENT` and sets `sentAt = new Date()` and `fcmMessageId = result.messageId`.
  - If the result is `success: false` AND `error === 'TRANSIENT_ERROR'`, waits `RETRY_DELAYS_MS[0]` = 1000ms and retries.
  - If the result is `success: false` AND `error` is `'INVALID_TOKEN'`, `'CONFIGURATION_ERROR'`, or `'QUOTA_EXCEEDED'`, does NOT retry. Returns `{ success: false, error: result.error }`. Updates the `Notification.status` to `FAILED` and sets `sentAt = new Date()`.
  - **Retry loop:** Up to `MAX_RETRY_ATTEMPTS` = 3 retries. Between attempts, waits `RETRY_DELAYS_MS[i]` ms (1s, 5s, 30s). The `i` index increments with each retry.
  - If all retries are exhausted and the last error is still `TRANSIENT_ERROR`, returns `{ success: false, error: 'TRANSIENT_ERROR' }`. Updates the `Notification.status` to `FAILED` and sets `sentAt = new Date()`.
- The function uses `setTimeout` wrapped in a Promise for the delays: `await new Promise((resolve) => setTimeout(resolve, delay))`.
- The function does NOT throw. All outcomes return a result object.
- The function logs via `console.warn` when a retry is attempted: `'[Notification] Retrying dispatch for notification ${notificationId} (attempt ${attempt + 1}/${MAX_RETRY_ATTEMPTS + 1}) after ${delay}ms'`.

### Step 3: Implement the `notifyOfficers` Function

**Goal:** Provide the main entry point that the trigger endpoints call.

**Files affected:** `src/lib/notification-dispatch.ts` (extended).

**Specific instructions:**

- Add the exported function `async function notifyOfficers(input: NotifyOfficersInput): Promise<NotificationDispatchSummary>`:
  - Records the start time: `const startTime = Date.now()`.
  - Initialises the summary: `{ notificationsCreated: 0, notificationsSent: 0, notificationsFailed: 0, recipientsSkipped: 0, durationMs: 0 }`.
  - **For each recipient `counterOfficerId` in `input.recipientCounterOfficerIds`:**
    1. Look up the `CounterOfficer` record: `db.counterOfficer.findUnique({ where: { id: counterOfficerId } })`.
    2. If the `CounterOfficer.notificationsEnabled === false`, increment `recipientsSkipped` and continue to the next recipient. (This check is defensive — the caller should have already filtered, but the dispatch layer enforces it too.)
    3. Look up active device tokens: `getActiveTokensForOfficer(counterOfficerId)` (from 4.1.2).
    4. If the array is empty, increment `recipientsSkipped` and continue to the next recipient.
    5. For each active token, create a `Notification` record:
       - `db.notification.create({ data: { ticketId: input.ticketId, counterOfficerId, type: input.type, title: <from buildNotificationPayload>, body: <from buildNotificationPayload>, data: <from buildNotificationPayload>, status: 'PENDING' } })`.
       - Increment `notificationsCreated`.
    6. For each created `Notification` record AND each active token (in case the officer has multiple devices):
       - Call `sendNotificationWithRetry(notification.id, token, payload)` (the internal helper from Step 2).
       - The `payload` is the `SendNotificationInput` constructed from `buildNotificationPayload` and the token.
       - Based on the result, increment `notificationsSent` or `notificationsFailed` and update the `Notification.status` (already done inside `sendNotificationWithRetry`).
       - Also update the `DeviceToken.lastUsedAt = new Date()` on success (this is the refresh from Phase 4.1 standards 6.5).
  - Computes `durationMs = Date.now() - startTime` and returns the summary.
- The function is wrapped in a try/catch at the outer level. Any unexpected exception is caught and logged via `console.error` with the message `'[Notification] Unexpected error in notifyOfficers:'` and the error. The summary is still returned (with whatever counts were accumulated before the error).
- The function does NOT throw to the caller. The caller (2.2.1 or 2.3.1) treats the dispatch as best-effort and continues regardless of the summary.

### Step 5: Add Audit Log Writes to the Dispatch Flow

**Goal:** Write `AuditLog` entries for dispatch outcomes.

**Files affected:** `src/lib/notification-dispatch.ts` (extended).

**Specific instructions:**

- At the end of the `notifyOfficers` function (after the per-recipient loop), determine the dispatch outcome:
  - If `notificationsSent > 0`, call `writeAuditLog({ action: 'NOTIFICATION_DISPATCHED', actorId: null, entity: 'Notification', entityId: null, description: \`Dispatched \${notificationsSent} notification(s) for \${input.type} ticket \${input.ticketNumber}\`, metadata: { type: input.type, ticketId: input.ticketId, ticketNumber: input.ticketNumber, notificationsSent, notificationsFailed, recipientsSkipped, durationMs } })`. The `actorId`is`null` because the dispatch is triggered by the system (the ticket issuance or recall endpoint), not by a specific user. (For ticket issuance, the actor is the kiosk; for recall, the actor is the calling officer — but in both cases, the dispatch itself is a system action, not a user action.)
  - Else if `notificationsCreated > 0` AND `notificationsSent === 0` AND `notificationsFailed > 0`, call `writeAuditLog({ action: 'NOTIFICATION_FAILED', ... })` with a similar description.
  - Else (no notifications created — all recipients skipped), no audit log entry. The skip is not a notable action.
- The audit log writes are best-effort: they happen after the dispatch loop, and any failure is caught and logged via `console.error` but does not fail the dispatch.

### Step 6: Modify the Ticket Issuance Endpoint to Call `notifyOfficers`

**Goal:** Add the `notifyOfficers` call to the ticket issuance flow.

**Files affected:** `src/app/api/tickets/issue/route.ts` (modified — owned by Phase 2.2.1).

**Specific instructions:**

- Open the existing route file (created in Phase 2.2.1).
- Locate the section after the transaction commits. The 2.2.1 endpoint commits the transaction (with the `Service.currentTicketNumber` increment, the `Ticket` creation, and the `TicketEvent` creation) and then calls `broadcastEvent` (from Phase 2.1.1 / Phase 3.1.3).
- **Add a new function `findEligibleRecipientsForIssuance(serviceId: string): Promise<string[]>`** (defined inline in the route file, or extracted to a helper at the top of the file). The function:
  - Queries: `db.counterService.findMany({ where: { serviceId }, include: { counter: { include: { officers: { where: { isOnDuty: true, notificationsEnabled: true } } } } } })`.
  - Flattens the result to a deduplicated list of `CounterOfficer` IDs. An officer assigned to multiple counters handling the same service appears once.
  - Returns the list.
- **After the `broadcastEvent` call** (NOT inside the transaction), add the following:
  1. Call `findEligibleRecipientsForIssuance(serviceId)` to get the recipient list.
  2. If the list is empty, skip the `notifyOfficers` call (no recipients).
  3. Else, call `await notifyOfficers({ ticketId, ticketNumber, serviceId, serviceName, counterId: <first counter from the query, if multiple; or null>, counterName: <first counter name, if multiple; or null>, type: 'TICKET_ISSUED', recipientCounterOfficerIds: recipients })`.
  4. The `notifyOfficers` call is awaited (not fire-and-forget) so the dispatch completes before the API returns. The dispatch is fast (typically <100ms in console mode, <2s with real FCM including retries). The 2.2.1 endpoint's response is delayed by this time, but the user experience is acceptable for a kiosk flow.
  5. Wrap the `notifyOfficers` call in a try/catch — any error is caught and logged via `console.error` but does NOT fail the ticket issuance. The ticket is issued regardless of the dispatch outcome.
- **The `counterId` and `counterName` for the notification payload:** the issuance is not tied to a specific counter (the ticket is in the `WAITING` state, not yet called to a counter). The agent uses the FIRST counter from the eligible recipients query (the first `CounterService` result's `counter` field) as the `counterId` and `counterName` in the notification payload. If no counters handle the service, the `notifyOfficers` call is skipped.
- **Do NOT modify the existing 2.2.1 transaction.** The transaction is unchanged. The `notifyOfficers` call is added AFTER the transaction commits.

### Step 11: Retry Logic Verification (Unit-Test Style)

## 8. Common Pitfalls

The agent should be aware of the following common mistakes and avoid them:

- **Calling `notifyOfficers` INSIDE the transaction.** The transaction is the database commit. The `notifyOfficers` call is a network call to FCM (or a console log) and must be OUTSIDE the transaction. A transaction with a network call inside is a recipe for timeouts and deadlocks.
- **Not wrapping the `notifyOfficers` call in a try/catch.** If the dispatch throws, the ticket issuance or recall fails. The ticket operation should succeed regardless of the dispatch outcome. The try/catch ensures this.
- **Forgetting to await the `notifyOfficers` call.** The call is awaited so the response includes the dispatch summary. If the call is fire-and-forget, the API returns before the dispatch completes, and the audit log entry may not be written before the process exits.
- **Retrying `INVALID_TOKEN` or `QUOTA_EXCEEDED`.** These are non-retriable in the 4.1.3 retry wrapper. The 4.1.1 service returns the error, and the wrapper does NOT retry. The `INVALID_TOKEN` cleanup is handled by the callback (4.1.1 → 4.1.2). The `QUOTA_EXCEEDED` is a rate-limit signal; the next dispatch will catch up.
- **Sending the notification to the wrong recipient on recall.** The recall notification is sent to the ORIGINAL calling officer (`Ticket.calledByOfficerId`), not the officer performing the recall. The 2.3.1 endpoint's `counterOfficerId` (the officer performing the recall) is NOT used in the dispatch.
- **Not deduplicating the recipient list for ticket issuance.** An officer assigned to multiple counters handling the same service must appear in the recipient list only once. The `findEligibleRecipientsForIssuance` query includes a `Set`-based deduplication (or a `[...new Set(ids)]` pattern).
- **Using the wrong `counterId` / `counterName` in the issuance payload.** The ticket is in the `WAITING` state, not yet called to a counter. The agent uses the FIRST counter from the eligible recipients query as the `counterId` / `counterName` in the notification payload. This is a reasonable approximation — the Android app shows the counter where the ticket is likely to be called (the first counter handling the service).
- **Writing the `Notification.data` as a typed object instead of `Prisma.InputJsonValue`.** Prisma requires the `data` field to be `Prisma.InputJsonValue`. The agent casts the plain object to this type.
- **Not updating `DeviceToken.lastUsedAt` on successful dispatch.** The token refresh is part of the dispatch flow. Without it, the 90-day cleanup heuristic (future) cannot determine which tokens are stale.
- **Treating the dispatch as a user action in the audit log.** The dispatch is a system action (triggered by a ticket operation, not by a user). The `actorId` in the audit log is `null`. The `AuditLog` entry is descriptive ("Dispatched N notifications for TICKET_ISSUED ticket A005"), not user-attributed.
- **Logging the full notification payload.** The payload contains the ticket number, service name, and counter name — these are not PII but are operational data. The agent logs only a summary (the FCM console log from 4.1.1 is the verbose path; the dispatch layer logs the summary).

### 9.2 The "OUTSIDE the Transaction" Discipline

The `notifyOfficers` call is OUTSIDE the database transaction. This is the same discipline as the SSE broadcast in Phase 2.2.1. The reasoning is:

- A transaction with a network call inside is dangerous — the transaction holds locks while waiting for FCM, which can cause deadlocks and timeouts.
- The ticket operation is the user-facing action. The notification is a parallel action. The ticket should succeed even if the notification fails (and vice versa).
- The transaction commits first; then the broadcast and the notification are dispatched as separate steps. If the broadcast fails, the SSE event is lost; if the notification fails, the FCM message is not delivered. The ticket state is correct in both cases.

The agent should verify the `notifyOfficers` call is placed AFTER the transaction's `commit` or `end` block in both the 2.2.1 and 2.3.1 routes.

### 9.3 The Retry Policy

The retry policy is in 4.1.3, not 4.1.1. The separation of concerns is:

- **4.1.1** is the FCM transport. It returns a typed result. It does NOT retry.
- **4.1.3** is the dispatch layer. It wraps 4.1.1 with retry logic. It decides the retry policy.

This separation allows:

- The 4.1.1 service to be tested without retry complexity.
- The 4.1.3 retry policy to be changed without modifying the service.
- Different dispatch contexts (e.g., a future admin broadcast system) to use 4.1.1 with different retry policies.

The retry policy in 4.1.3 is:

- `TRANSIENT_ERROR`: retry up to 3 times with delays of 1s, 5s, 30s.
- `INVALID_TOKEN`, `CONFIGURATION_ERROR`, `QUOTA_EXCEEDED`: no retry.
- Success on any attempt: stop retrying.

### 9.4 The `DeviceToken.lastUsedAt` Refresh

The `lastUsedAt` field is updated on successful dispatch. This serves two purposes:

1. **Diagnostic**: shows the agent / admin when a device last received a notification.
2. **Future cleanup**: the 90-day cleanup heuristic (Phase 4.2 or Phase 5) can identify tokens that haven't been used in 90 days and deactivate them.

The update is a single `db.deviceToken.update({ where: { id: token.id }, data: { lastUsedAt: new Date() } })` call. It is best-effort (the dispatch does not fail if the update fails). The agent wraps the update in a try/catch and logs failures via `console.warn`.

### 9.5 The Audit Log Strategy

The audit log captures aggregate dispatch outcomes, not individual `Notification` records. The rationale:

- The `Notification` table is the detailed audit trail for individual dispatches (one row per recipient per attempt).
- The `AuditLog` captures higher-level events ("a dispatch happened, here is the summary").
- This separation keeps the `AuditLog` small and queryable for compliance and reporting.

The two new audit actions are:

- `NOTIFICATION_DISPATCHED` — written when at least one notification in the dispatch batch succeeded. The `metadata` includes the type, ticket, counts, and duration.
- `NOTIFICATION_FAILED` — written when all notifications in the batch failed. The `metadata` is similar.

The agent does NOT write an audit log entry for skip-only dispatches (all recipients skipped). The skip is not a notable action.

### 9.6 The Counter ID and Name in the Issuance Payload

The ticket issuance notification's `counterId` and `counterName` are the FIRST counter from the eligible recipients query. This is a design decision:

- The ticket is in the `WAITING` state, not yet called to a specific counter.
- The notification is sent BEFORE the ticket is called.
- The Android app shows the counter where the ticket is most likely to be called.
- The first counter is a reasonable approximation (the ticket is typically called to the first available counter handling the service).

If the design is changed in the future (e.g., the notification is sent only after the ticket is called), the `counterId` and `counterName` would be the actual called counter. For Phase 4.1, the first-counter approximation is sufficient.

### 9.7 When the Master Plan is Ambiguous

- The master plan does not specify the exact retry delays. The agent uses 1s, 5s, 30s based on common FCM retry patterns and the Phase 2.3.2 auto-advance pattern.
- The master plan does not specify the exact wording of the audit log description. The agent uses the documented wording.
- The master plan does not specify whether the dispatch is awaited or fire-and-forget. The agent awaits the call (so the audit log is written before the API returns) but wraps in try/catch (so the dispatch failure does not fail the ticket operation).
- The master plan does not specify the `counterId` / `counterName` for the issuance notification. The agent uses the first counter from the eligible recipients query (the most reasonable approximation).
- The master plan does not specify the exact title and body text. The agent uses the documented wording (`"New Ticket"` / `"Ticket X for Y"`, `"Ticket Recalled"` / `"Ticket X has been recalled"`).
