## 1. Document Purpose

This document delivers the **server-side FCM transport** for the push notification system. It introduces the `NotificationService` server module at `src/lib/notification-service.ts` — a wrapper around the Firebase Cloud Messaging HTTP v1 API. The module exposes a small, typed interface (`sendNotification`, `sendBatchNotifications`, `setOnInvalidTokenCallback`, `isFcmConfigured`) that the dispatch layer (4.1.3) and the device registration layer (4.1.2) call. It also introduces the FCM credential configuration (the two new env vars), the OAuth access token exchange with Google's token endpoint, the error code mapping from FCM responses to typed result variants, the console-log fallback for local development, and the token cleanup callback hook that 4.1.2 subscribes to.

When this document is complete:

- The `NotificationService` module exists and exports the documented API surface.
- The two new env vars (`FCM_SERVICE_ACCOUNT_JSON` and `FCM_PROJECT_ID`) are added to the env validator.
- When the env vars are missing or unparseable, `isFcmConfigured()` returns `false` and `sendNotification` logs the payload to the server console and returns a successful result with `mode: 'console'`.
- When the env vars are present and valid, `isFcmConfigured()` returns `true` and `sendNotification` calls the FCM HTTP v1 API with a short-lived OAuth access token (cached in module scope, refreshed on expiry).
- FCM error responses are mapped to the four documented error variants: `INVALID_TOKEN`, `CONFIGURATION_ERROR`, `QUOTA_EXCEEDED`, `TRANSIENT_ERROR`. The `INVALID_TOKEN` mapping triggers the cleanup callback.
- The `setOnInvalidTokenCallback` registration works (subsequent `sendNotification` calls with bad tokens invoke the registered callback).
- A small test page or temporary route demonstrates the console-log fallback (no FCM credentials needed).

This document does **not** build the device registration endpoints (owned by 4.1.2), the dispatch logic (owned by 4.1.3), the mobile login finalisation (owned by 4.1.2), or the officer dashboard (owned by 4.2.3).

### Step 2: Define the `SendNotificationInput` and `SendNotificationResult` Types

**Goal:** Establish the typed interface that the dispatch layer (4.1.3) and the test code will use.

**Files affected:** `src/lib/notification-service.ts` (new file, initial scaffold).

**Specific instructions:**

Create `src/lib/notification-service.ts` with the following type declarations at the top:

- **`SendNotificationInput`** — the input shape for `sendNotification`:
  - `token: string` — the FCM device token. Must be at least 10 characters (enforced by the caller via Zod, not here).
  - `title: string` — the notification title (e.g., "New Ticket"). Max 100 characters (FCM limit; enforced by the caller).
  - `body: string` — the notification body text. Max 200 characters (FCM limit; enforced by the caller).
  - `data?: Record<string, string>` — the structured data payload. All values are strings (FCM requirement). The shape is `{ type, notificationId, ticketId, ticketNumber, serviceId, serviceName, counterId, counterName, replyUrl }` per master plan Section 14.3, but the service does not validate the keys (it is the caller's responsibility).
- **`SendNotificationResult`** — the output shape, a discriminated union:
  - Success variant: `{ success: true; messageId: string; mode: 'fcm' | 'console' }`. The `messageId` is the FCM-assigned message ID for `mode: 'fcm'`, or a generated `console-<random>` string for `mode: 'console'`.
  - Failure variants (each has `success: false`, an `error` enum, and a `message` string for diagnostics):
    - `error: 'INVALID_TOKEN'` — the token is no longer registered with FCM. The cleanup callback is invoked (if registered).
    - `error: 'CONFIGURATION_ERROR'` — the FCM configuration is invalid (SENDER_ID_MISMATCH, malformed response, etc.). The caller should NOT retry; this is a developer error.
    - `error: 'QUOTA_EXCEEDED'` — the FCM project has hit its rate limit. The caller may retry after a delay (the dispatch layer in 4.1.3 does not retry QUOTA_EXCEEDED).
    - `error: 'TRANSIENT_ERROR'` — a transient FCM error (UNAVAILABLE, INTERNAL, DEADLINE_EXCEEDED). The caller may retry with exponential backoff.
- Export both types.

### Step 3: Implement the FCM Configuration Detection Helper

**Goal:** Provide a single function that the rest of the module uses to determine whether FCM is configured.

**Files affected:** `src/lib/notification-service.ts` (extended).

**Specific instructions:**

- Add a module-scope variable `let fcmConfig: { configured: boolean; serviceAccount?: ServiceAccount; projectId?: string } | null = null`. Initialised to `null` (lazy init).
- Add a function `getFcmConfig()` (NOT exported) that:
  - If `fcmConfig !== null`, returns the cached value.
  - If `fcmConfig === null`:
    - Read `process.env.FCM_SERVICE_ACCOUNT_JSON` and `process.env.FCM_PROJECT_ID`.
    - If either is missing, set `fcmConfig = { configured: false }` and log a one-time `console.info` message: `"[FCM] Not configured — running in console-log mode. Set FCM_SERVICE_ACCOUNT_JSON and FCM_PROJECT_ID to enable real dispatch."`
    - If both are present, attempt to parse `FCM_SERVICE_ACCOUNT_JSON` as JSON. If parsing fails, set `fcmConfig = { configured: false }` and log a one-time `console.warn` message: `"[FCM] FCM_SERVICE_ACCOUNT_JSON is not valid JSON — falling back to console-log mode."`
    - If parsing succeeds, set `fcmConfig = { configured: true, serviceAccount: parsedJson, projectId: <env var> }`.
  - Returns the cached value.
- Add the exported function `isFcmConfigured(): boolean` that returns `getFcmConfig().configured`.
- The one-time log messages use a separate `let hasLoggedConfigMessage = false` flag (not stored in the config object) so they fire only once per process lifetime, not on every `getFcmConfig()` call.

### Step 4: Implement the OAuth Access Token Exchange

**Goal:** Obtain a short-lived OAuth 2.0 access token from Google's token endpoint using the service account's private key. Cache the token and refresh on expiry.

**Files affected:** `src/lib/notification-service.ts` (extended).

**Specific instructions:**

- Add module-scope state: `let cachedAccessToken: { token: string; expiresAt: number } | null = null`. The `expiresAt` is a Unix timestamp in milliseconds.
- Add a function `async function getAccessToken(): Promise<string>` (NOT exported) that:
  - If `cachedAccessToken !== null` AND `Date.now() < cachedAccessToken.expiresAt - (5 * 60 * 1000)` (5-minute buffer), returns `cachedAccessToken.token`.
  - Otherwise:
    - Read the service account from `getFcmConfig()`. If not configured, throw an error: `"FCM is not configured"`. (This should not happen if the caller checks `isFcmConfigured()` first, but the defensive check is documented.)
    - Construct a JWT with the following claims:
      - `iss`: the `client_email` from the service account.
      - `scope`: `'https://www.googleapis.com/auth/firebase.messaging'`.
      - `aud`: `'https://oauth2.googleapis.com/token'`.
      - `iat`: current Unix timestamp in seconds.
      - `exp`: current Unix timestamp + 3600 seconds (1 hour).
    - Sign the JWT with the `private_key` from the service account using the RS256 algorithm. Use the `jsonwebtoken` package's `jwt.sign()` function.
    - POST to `https://oauth2.googleapis.com/token` with the body `grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer&assertion=<signed_jwt>` (URL-encoded form data).
    - Parse the response. The response shape is `{ access_token: string, expires_in: number, token_type: 'Bearer' }`.
    - Set `cachedAccessToken = { token: access_token, expiresAt: Date.now() + (expires_in * 1000) }`.
    - Return the access token.
- The function does not catch errors — they propagate to the caller. The caller (`sendNotification`) catches and maps to the appropriate result variant.

### Step 5: Implement the FCM HTTP v1 Request

**Goal:** Construct the FCM request body and POST to the FCM HTTP v1 endpoint.

**Files affected:** `src/lib/notification-service.ts` (extended).

**Specific instructions:**

- Add a function `async function sendToFcm(input: SendNotificationInput, accessToken: string, projectId: string): Promise<Response>` (NOT exported) that:
  - Constructs the FCM endpoint URL: `https://fcm.googleapis.com/v1/projects/${projectId}/messages:send`.
  - Constructs the request body: `{ message: { token: input.token, notification: { title: input.title, body: input.body }, data: input.data || {} } }`. The `data` object is the caller's `Record<string, string>` (FCM requires all values to be strings; the service does NOT validate this).
  - Sends a POST request to the FCM endpoint with:
    - `Authorization: Bearer ${accessToken}` header.
    - `Content-Type: application/json` header.
    - JSON-stringified body.
  - Uses the `fetch` API (Node 18+ has it built-in; no import needed).
  - Returns the raw `Response` object (caller checks `ok` and parses the body).
- The function does not catch errors — they propagate.

### Step 6: Implement the FCM Error Code Mapping

**Goal:** Parse the FCM error response and map it to the `SendNotificationResult` failure variants.

**Files affected:** `src/lib/notification-service.ts` (extended).

**Specific instructions:**

- Add a function `function mapFcmErrorToResult(response: Response, errorBody: any): Extract<SendNotificationResult, { success: false }>` (NOT exported) that:
  - If `response.status === 404` OR the error body has `error.details[].errorCode` matching `messaging/registration-token-not-registered` OR `messaging/invalid-registration-token` → return `{ success: false, error: 'INVALID_TOKEN', message: '...' }`.
  - If the error body has `error.details[].errorCode` matching `messaging/mismatched-credential` OR `messaging/authentication-error` OR `error.status === 'UNAUTHENTICATED'` → return `{ success: false, error: 'CONFIGURATION_ERROR', message: '...' }`. The message includes "SENDER_ID_MISMATCH" or similar FCM wording for diagnostics.
  - If the error body has `error.details[].errorCode` matching `messaging/quota-exceeded` OR `error.status === 'RESOURCE_EXHAUSTED'` → return `{ success: false, error: 'QUOTA_EXCEEDED', message: '...' }`.
  - If the response status is 5xx (500, 502, 503, 504) OR the error body has `error.status` of `UNAVAILABLE` / `INTERNAL` / `DEADLINE_EXCEEDED` → return `{ success: false, error: 'TRANSIENT_ERROR', message: '...' }`.
  - For any other 4xx error → return `{ success: false, error: 'CONFIGURATION_ERROR', message: '...' }`. The dispatch layer does not retry this.
  - For any other 5xx error → return `{ success: false, error: 'TRANSIENT_ERROR', message: '...' }`.
- The function uses a `console.warn` for `QUOTA_EXCEEDED` and a `console.error` with a "CRITICAL" prefix for `CONFIGURATION_ERROR`. `INVALID_TOKEN` and `TRANSIENT_ERROR` do not log (they are expected or transient).

### Step 7: Implement the `sendNotification` Function

**Goal:** Tie together the configuration check, the access token exchange, the FCM request, the error mapping, and the console-log fallback.

**Files affected:** `src/lib/notification-service.ts` (extended).

**Specific instructions:**

- Add the exported function `async function sendNotification(input: SendNotificationInput): Promise<SendNotificationResult>` that:
  - Calls `getFcmConfig()`. If `!configured`:
    - Logs the payload via `console.info` with the format: `"[FCM console] type=${input.data?.type ?? 'unknown'} ticket=${input.data?.ticketNumber ?? 'n/a'} service=${input.data?.serviceName ?? 'n/a'} counter=${input.data?.counterName ?? 'n/a'} mode=console"`.
    - Returns `{ success: true, messageId: 'console-' + randomUUID(), mode: 'console' }`.
  - If `configured`:
    - Calls `getAccessToken()`. If it throws, returns `{ success: false, error: 'CONFIGURATION_ERROR', message: 'Failed to obtain access token: ' + error.message }`.
    - Calls `sendToFcm(input, accessToken, projectId)`.
    - If `response.ok` (status 200), parses the body: `{ name: 'projects/.../messages/<messageId>' }`. Extracts the `<messageId>` from the `name` field (split on `/` and take the last segment). Returns `{ success: true, messageId: <extracted>, mode: 'fcm' }`.
    - If `!response.ok`, parses the error body and calls `mapFcmErrorToResult`. If the result is `INVALID_TOKEN`, invokes the cleanup callback (if registered) with `input.token`. Returns the result.
- The function is wrapped in a try/catch that returns `{ success: false, error: 'TRANSIENT_ERROR', message: '...' }` for any unexpected exception. This defensive catch ensures the dispatch layer (4.1.3) never sees a thrown error from `sendNotification` — it always gets a typed result.

### Step 8: Implement the `sendBatchNotifications` Function

**Goal:** Provide a parallel batch dispatch that returns the result for each input.

**Files affected:** `src/lib/notification-service.ts` (extended).

**Specific instructions:**

- Add the exported function `async function sendBatchNotifications(inputs: SendNotificationInput[]): Promise<SendNotificationResult[]>` that:
  - Calls `sendNotification` for each input in parallel via `Promise.all`.
  - Returns the array of results in the same order as the inputs.
- The function does NOT aggregate results. The caller is responsible for inspecting each result and updating the `Notification.status` accordingly. (The 4.1.3 dispatch layer handles this.)

### Step 9: Implement the `setOnInvalidTokenCallback` Function

**Goal:** Allow the device registration layer (4.1.2) to register a callback that fires when FCM returns `INVALID_TOKEN`.

**Files affected:** `src/lib/notification-service.ts` (extended).

**Specific instructions:**

- Add a module-scope variable: `let onInvalidTokenCallback: ((token: string) => void) | null = null`.
- Add the exported function `function setOnInvalidTokenCallback(callback: (token: string) => void): void` that sets `onInvalidTokenCallback = callback`. Setting it twice replaces the previous callback.
- In the `sendNotification` function (from Step 7), when the result is `INVALID_TOKEN` AND `onInvalidTokenCallback !== null`, invoke the callback: `try { onInvalidTokenCallback(input.token); } catch (callbackError) { console.error('[FCM] Invalid token callback threw:', callbackError); }`. The try/catch ensures a buggy callback does not break the dispatch.

## 8. Common Pitfalls

The agent should be aware of the following common mistakes and avoid them:

- **Hardcoding the FCM endpoint URL.** The URL includes the `FCM_PROJECT_ID` and must be constructed at runtime from the env var. A hardcoded URL would fail when the project ID changes.
- **Not caching the OAuth access token.** Exchanging the JWT for an access token on every send is wasteful (the token exchange takes ~100ms; the cached token is instant). Cache it.
- **Forgetting the 5-minute expiry buffer.** The Google access token TTL is 60 minutes. The 4.1.1 module refreshes the token when within 5 minutes of expiry, not on exact expiry, to prevent in-flight requests from using an expired token.
- **Treating `INVALID_TOKEN` as `TRANSIENT_ERROR` and retrying.** This wastes FCM quota on dead tokens and never deactivates them. The error mapping MUST treat `INVALID_REGISTRATION` and `UNREGISTERED` as `INVALID_TOKEN` (non-retriable, triggers cleanup).
- **Logging the full service account JSON.** The JSON contains the private key. Never log it. Log only the project ID and the configuration state (configured vs. not).
- **Throwing from `sendNotification`.** The function ALWAYS returns a `SendNotificationResult`. Never throws. The dispatch layer (4.1.3) relies on this contract to avoid try/catch boilerplate at every call site. The catch-all in Step 7 ensures this.
- **Making the `mode: 'console'` field optional.** It is always present on the success variant. Tests and the dispatch layer rely on this for distinguishing real FCM from console fallback.
- **Setting `onInvalidTokenCallback` as a constant instead of a mutable variable.** The registration function replaces the previous callback. A `const` would not allow this.
- **Forgetting the `runtime = 'nodejs'` directive.** This module uses `crypto` (for JWT signing) and `process.env` (for env vars). The route handlers in 4.1.2 that import this module are server routes and should have `runtime = 'nodejs'`. The `notification-service.ts` module itself is a server module — it should not be imported by client components.
- **Treating the FCM `data` object as untyped.** The `data` values are always strings (FCM requirement). The `SendNotificationInput.data` type is `Record<string, string>` to enforce this at the TypeScript level. The 4.1.3 dispatch layer is responsible for ensuring all values are strings before passing them in.
- **Not using a try/catch around the cleanup callback invocation.** A buggy callback (e.g., one that throws) would prevent the dispatch result from being returned. The callback is best-effort; its errors are logged but not propagated.

### 9.2 The Console-Log Fallback is a First-Class Feature

The console-log fallback is not a "TODO" or a "production hack" — it is a designed-in feature that enables local development without Firebase setup. The agent should:

- Treat console-log mode as a real, supported mode of operation.
- Log a one-time `console.info` message at module init: `"[FCM] Not configured — running in console-log mode. Set FCM_SERVICE_ACCOUNT_JSON and FCM_PROJECT_ID to enable real dispatch."`
- Log each send in console mode with the full payload via `console.info` using the documented format.
- Return `{ success: true, messageId: 'console-<random>', mode: 'console' }` so the dispatch layer can track the dispatch as a success in the `Notification` table.

### 9.3 The FCM Error Code Mapping is Critical

Get the error code mapping right. The wrong mapping (e.g., treating `INVALID_REGISTRATION` as `TRANSIENT_ERROR`) would cause the system to waste FCM quota on dead tokens and never deactivate them. The four error categories are:

- **`INVALID_TOKEN`** — non-retriable, triggers cleanup callback. Use this for `INVALID_REGISTRATION` and `UNREGISTERED`.
- **`CONFIGURATION_ERROR`** — non-retriable, logs critical alert. Use this for `SENDER_ID_MISMATCH` and other auth errors.
- **`QUOTA_EXCEEDED`** — non-retriable in the dispatch layer (no immediate retry), but the dispatch layer's 4.1.3 retry wrapper does NOT retry this. The next event will "catch up" naturally.
- **`TRANSIENT_ERROR`** — retriable, the dispatch layer's 4.1.3 retry wrapper retries up to 3 times with exponential backoff (1s, 5s, 30s).

### 9.4 The Access Token Exchange

The OAuth 2.0 service account flow is the standard Google pattern:

1. Sign a JWT with the service account's private key. The JWT has the claims: `iss` (client_email), `scope` (`https://www.googleapis.com/auth/firebase.messaging`), `aud` (`https://oauth2.googleapis.com/token`), `iat`, `exp`.
2. Exchange the JWT at `https://oauth2.googleapis.com/token` with `grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer&assertion=<signed_jwt>`.
3. Receive an access token with a 60-minute TTL.
4. Use the access token as `Authorization: Bearer <access_token>` in the FCM request.

The `jsonwebtoken` package's `jwt.sign(payload, privateKey, { algorithm: 'RS256' })` function signs the JWT. The `privateKey` is the `private_key` field from the service account JSON. Note that the key is a PEM-encoded string with literal `\n` characters that need to be preserved.

### 9.5 The FCM HTTP v1 Request Body

The request body has the shape:

```
{
  "message": {
    "token": "<device-token>",
    "notification": {
      "title": "<title>",
      "body": "<body>"
    },
    "data": {
      "type": "TICKET_ISSUED",
      "notificationId": "...",
      "ticketId": "...",
      ...
    }
  }
}
```

The `data` values are ALL strings (FCM requirement). The 4.1.3 dispatch layer ensures this by constructing the payload with `String(value)` for non-string values.

### 9.6 FCM Error Response Shape

FCM error responses have the shape:

```
{
  "error": {
    "code": 400,
    "message": "Request contains an invalid argument.",
    "status": "INVALID_ARGUMENT",
    "details": [
      {
        "@type": "type.googleapis.com/google.firebase.fcm.v1.FcmError",
        "errorCode": "messaging/registration-token-not-registered"
      }
    ]
  }
}
```

The agent parses `error.details[].errorCode` to get the specific FCM error code. The `error.status` field provides the general category (e.g., `INVALID_ARGUMENT`, `RESOURCE_EXHAUSTED`, `UNAUTHENTICATED`, `UNAVAILABLE`, `INTERNAL`).

The mapping is:

- `errorCode` starts with `messaging/registration-token-not-registered` or `messaging/invalid-registration-token` → `INVALID_TOKEN`.
- `errorCode` is `messaging/mismatched-credential` or `error.status` is `UNAUTHENTICATED` → `CONFIGURATION_ERROR`.
- `errorCode` is `messaging/quota-exceeded` or `error.status` is `RESOURCE_EXHAUSTED` → `QUOTA_EXCEEDED`.
- `error.status` is `UNAVAILABLE` or `INTERNAL` or `DEADLINE_EXCEEDED` → `TRANSIENT_ERROR`.
- Any other 4xx → `CONFIGURATION_ERROR`.
- Any other 5xx → `TRANSIENT_ERROR`.

### 9.7 The `mode: 'console'` Field

The `mode: 'console'` field in the success result is for testability. Tests can assert that a test environment runs in console mode and avoid the FCM HTTP call. The dispatch layer in 4.1.3 does NOT use the `mode` field — it treats both modes as success and updates the `Notification.status` to `SENT` regardless. The field is for diagnostic purposes and future test infrastructure.

### 9.8 The Cleanup Callback is Fire-and-Forget

The cleanup callback is `(token: string) => void` — no return value, no async. The 4.1.2 `deactivateToken` function is synchronous (it returns a Promise but the callback ignores it). The 4.1.1 `sendNotification` invokes the callback inside a try/catch and ignores the return value. A buggy callback does not break the dispatch.
