## 1. Document Purpose

This document describes the complete strategy for migrating the Smart Queue Management System from SQLite (development) to PostgreSQL (production). It establishes the two-step Prisma migration process, the data migration tooling via `pgloader`, the PostgreSQL-specific query discipline (case sensitivity, JSONB, connection pooling, transaction isolation, backups), and the connection pool sizing for the Prisma client.

The deliverable is a **reproducible migration** that takes a fresh PostgreSQL instance and produces a fully-functional production database, plus a data migration path for any existing SQLite deployments. The migration is documented in `docs/postgresql-migration.md` as the single source of truth, verified by the `verify-migration.ts` script, and supported by the connection pool configuration in `src/lib/db.ts`.

**Why this matters:** A production queue management system cannot run on SQLite. SQLite has limitations around concurrent writes, transaction isolation, JSON column handling, and data volume that make it unsuitable for a production workload. PostgreSQL is the standard choice for production Next.js + Prisma applications and supports all the features the system needs: concurrent writes, ACID transactions, JSONB columns for flexible metadata, and a mature backup/restore ecosystem.

### Step 4: Document the `pgloader` Data Migration

**Files affected:** `docs/postgresql-migration.md` (updated), `scripts/migration/pgloader.config` (new file).

**What to do:** Document the data migration strategy for existing SQLite deployments. The data migration uses `pgloader`, a system-level tool that handles the SQLite-to-PostgreSQL schema conversion automatically.

**Specific instructions:**

- In the `docs/postgresql-migration.md` file, expand the **Data Migration Strategy** section with:
  - When to use the data migration: ONLY for existing deployments with data in SQLite. Fresh deployments skip this step.
  - The `pgloader` command: `pgloader scripts/migration/pgloader.config` (or similar). The exact invocation depends on the platform (system-level tool, not a Yarn script).
  - The behavior: `pgloader` reads the SQLite database, converts the schema (e.g., `TEXT` → `text`, `INTEGER` → `integer`, `DATETIME` → `timestamp`), converts the data (e.g., date format if different), and inserts it into PostgreSQL. The conversion is automatic for most types but may require custom rules for edge cases.
  - The expected output: a summary of tables migrated with row counts.
  - The failure modes: connection failures, type mismatches, constraint violations. The error message indicates which table and which row failed.
- Create the `scripts/migration/pgloader.config` file with the `pgloader` configuration:
  - The source: SQLite file path (e.g., `prisma/dev.db`).
  - The target: PostgreSQL connection string (read from `DATABASE_URL` env var or specified directly).
  - The schema mapping rules:
    - `TEXT` → `text` (no change).
    - `INTEGER` → `integer` (no change for most cases; may need `bigint` for large counters).
    - `DATETIME` → `timestamp` (PostgreSQL's `timestamp` is `timestamp without time zone` by default; the schema uses `@db.Timestamptz` which maps to `timestamp with time zone`).
    - `JSON` → `jsonb` (Prisma's `Json` type maps to PostgreSQL's `jsonb`).
    - `BLOB` → `bytea` (not used in the current schema, but documented for future).
  - The include/exclude rules: include all tables, exclude Prisma's internal `__prisma_migrations` table (handled by `prisma migrate deploy`, not by `pgloader`).
  - The worker count and batch size: default 4 workers, 2500 rows per batch.
  - The casting rules: any type mismatches that need explicit casts (e.g., SQLite's `0/1` for booleans to PostgreSQL's `true/false`).
- Add a sub-section **Running the Data Migration** — the step-by-step for the operator:
  1. Stop the application (or ensure no writes are happening).
  2. Take a backup of the SQLite database.
  3. Verify the PostgreSQL instance is running and accessible.
  4. Run `pgloader scripts/migration/pgloader.config`.
  5. Wait for the migration to complete (output shows progress).
  6. Run the verification script (`yarn migration:verify`) to confirm row counts match.
  7. Run the seed script if needed (for fresh deployments only).

**Verification:** The `pgloader.config` file exists and contains the source, target, schema mapping rules, include/exclude rules, worker count, and batch size.

**Specific instructions:**

- Create the `scripts/migration/` directory at the project root.
- Create `scripts/migration/verify-migration.ts` with the following structure:
  - **Two Prisma client instances:** one for the source (SQLite), one for the target (PostgreSQL). The source is read-only; the target is read-only.
  - **The list of tables to verify:** all tables in the schema (User, Role, Permission, RolePermission, UserRole, RefreshToken, Service, Counter, CounterService, CounterOfficer, CounterStatusEvent, Ticket, TicketEvent, DisplayBoard, KioskConfig, DeviceToken, Notification, NotificationReply, BroadcastMessage, QueueDailySnapshot, AuditLog, SystemSetting).

### Step 6: Extend the Prisma Client with PostgreSQL Connection Pool Configuration

**Files affected:** `src/lib/db.ts` (updated), `src/lib/env.ts` (updated).

**What to do:** Update the Prisma client to use the correct connection pool configuration for PostgreSQL in production. The pool size is environment-aware (small for development with SQLite, sized for production with PostgreSQL).

**Specific instructions:**

- Read the current `src/lib/db.ts` to understand the existing Prisma client setup. The existing setup likely uses a singleton pattern with `globalThis.__prisma` (like the SSE manager and the rate limit store) to survive Next.js dev hot-reload.
- Extend `src/lib/db.ts` with:
  - **Connection pool configuration:** read the `DATABASE_URL` and parse the connection limit. The Prisma client constructor accepts a `log` array and (via the connection string URL parameter) a `?connection_limit=N` query parameter.
  - **Environment-aware logic:** in development (SQLite), the connection limit is not applicable (SQLite is file-based, not a server). In production (PostgreSQL), the connection limit is set based on the number of application instances.
  - **Connection timeout:** set the `connect_timeout` URL parameter to 10 seconds (default is 5). Production databases may need more time to connect.
  - **Application name:** set the `application_name` URL parameter to `quems-app` for easier identification in PostgreSQL logs.
  - **The `DIRECT_URL` handling:** if `DIRECT_URL` is set, use it for migrations. The Prisma client uses `DATABASE_URL` for the application and `DIRECT_URL` for migrations (when set). This is a Prisma feature — the schema can specify `directUrl = env("DIRECT_URL")` in the `datasource db` block.
- Update `src/lib/env.ts` with the new env vars:
  - `DIRECT_URL` — optional string. If set, used for Prisma migrations.
  - `DATABASE_POOL_SIZE` — optional string (number). Default is computed from the formula in 6.4.
  - `DATABASE_CONNECT_TIMEOUT_MS` — optional string (number). Default `'10000'`.
- Document the connection pool sizing formula in `docs/postgresql-migration.md`:
  - `pool_size = floor((max_connections - reserved) / num_app_instances)`.
  - Default PostgreSQL `max_connections` is 100.
  - Reserve 10 for admin (e.g., `psql` connections for manual operations).
  - With 3 application instances: `pool_size = floor((100 - 10) / 3) = 30` per instance.
  - With 1 application instance: `pool_size = floor((100 - 10) / 1) = 90` per instance.

### Step 7: Document PostgreSQL-Specific Query Discipline

**Files affected:** `docs/postgresql-migration.md` (updated).

**What to do:** Add the comprehensive documentation of the PostgreSQL-specific query patterns that differ from SQLite. This is the operational guide for developers writing or reviewing queries.

**Specific instructions:**

- In the `docs/postgresql-migration.md` file, add a new section **PostgreSQL-Specific Query Discipline** with the following subsections:
  - **Case sensitivity:**
    - PostgreSQL string comparisons are case-sensitive by default. `WHERE email = 'John@Example.com'` does NOT match `'john@example.com'`.
    - The primary case-insensitive query in the system is the email lookup on login.
    - **Mitigation 1:** use Prisma's `mode: 'insensitive'` in the where clause. Example: `db.user.findFirst({ where: { email: { equals: email, mode: 'insensitive' } } })`.
    - **Mitigation 2:** change the `email` column to PostgreSQL's `citext` type. This requires a schema migration and is not recommended (citext has performance overhead).
    - **Recommendation:** use `mode: 'insensitive'` in Prisma queries. It's the standard approach.
  - **JSONB queries:**
    - Prisma's `Json` type maps to PostgreSQL's `jsonb`. Queries use Prisma's `Json` filter syntax: `path`, `string_contains`, `equals`, `array_contains`, etc.
    - Example: `db.auditLog.findMany({ where: { metadata: { path: ['settingKey'], equals: 'queue.daily_reset_time' } } })`.
    - JSONB is more efficient than JSON for queries (it's parsed once at insert time, not at query time).
  - **Date/time handling:**
    - All timestamps are stored as UTC. The Prisma `DateTime` type maps to PostgreSQL's `timestamp(3) with time zone`.
    - Application code must convert to local time for display via `APP_TIMEZONE` and `Intl.DateTimeFormat`. Never store local time.
    - The `businessDate` field on `Ticket` is a UTC `Date` representing midnight in `APP_TIMEZONE`. The conversion is handled in the application code.
  - **Connection pool sizing:**
    - The formula: `pool_size = floor((max_connections - reserved) / num_app_instances)`.
    - The default PostgreSQL `max_connections` is 100. Reserve 10 for admin. With 3 instances: 30 connections each.
    - The pool size is set via the `?connection_limit=N` query parameter on `DATABASE_URL`.
  - **Transaction isolation:**
    - The default `READ COMMITTED` is used. This is sufficient for the queue workload.
    - The system does not have high-contention serializable operations. Tickets are issued sequentially per service, calls are sequential per counter, no two officers can call the same ticket at the same time (the state machine enforces it).
    - If higher isolation is ever needed, `SERIALIZABLE` is available but adds overhead and may cause serialization conflicts. Document any future change to `SERIALIZABLE` in the runbook.
  - **Backup strategy:**
    - Daily `pg_dump` to a compressed file. Retain for 30 days locally, 1 year in cold storage.
    - Use `pg_dump -Fc -d quems_prod -f backup.dump` for custom-format dumps (smaller, faster restore).
    - Use `pg_restore -d quems_prod backup.dump` to restore.
    - Offsite backup: upload to S3 or equivalent cloud storage. Use encryption at rest.
    - Test the restore procedure quarterly. A backup that has never been restored is not a backup.
    - For managed services (AWS RDS, Supabase, Neon), use the managed backup feature and verify it's enabled.

**Verification:** The migration runbook contains all 6 subsections of the PostgreSQL-specific query discipline.

### Step 8: Document the Rollback Procedure

**Files affected:** `docs/postgresql-migration.md` (updated).

**What to do:** Document the steps to roll back the migration in case of failure. The rollback procedure is critical for production safety.

**Specific instructions:**

- In the `docs/postgresql-migration.md` file, add a **Rollback Procedure** section with:
  - **When to roll back:** if the migration fails partway through, if the verification checks fail, or if the application is not functional after the migration.
  - **The rollback steps:**
    1. **Revert the `provider` change** in `prisma/schema.prisma` from `postgresql` back to `sqlite`. This allows local development to continue.
    2. **Restore the previous PostgreSQL state** (if the migration was applied to a production PostgreSQL). Use `pg_restore` from the most recent backup, or use the managed service's point-in-time recovery feature.
    3. **Update the `DATABASE_URL`** back to the previous value (or to the SQLite path for local development).
    4. **Restart the application** with the previous configuration.
  - **What CANNOT be rolled back:** if data was migrated from SQLite to PostgreSQL and the migration is rolled back, the SQLite database is the source of truth. The PostgreSQL data is discarded. The operator must re-migrate from the SQLite backup.
  - **What to do if the data migration is partial:** if `pgloader` fails partway through, the PostgreSQL database may have some tables populated and some empty. The verification script will catch this. The fix is to drop the PostgreSQL database and re-run the migration from scratch.
  - **Communication:** notify the team immediately if a rollback is needed. Document the rollback in the deployment log.

**Verification:** The migration runbook contains the complete rollback procedure.

## 8. Common Pitfalls

- **Committing the `provider` change to `prisma/schema.prisma` in the repository.** The `provider` should remain `sqlite` in the repository. The change to `postgresql` is applied by the operator during the migration. Committing the change would break local development for everyone on the team.
- **Forgetting the `DIRECT_URL` for managed databases.** Some platforms (Supabase, Neon) require a `DIRECT_URL` for migrations that bypass the connection pooler. Without it, migrations fail with a "connection refused" or "too many connections" error.
- **Not sizing the connection pool correctly.** A pool size of 100 per application instance (the default) would exhaust the PostgreSQL connection limit (default 100) with a single application instance. The formula in 5.3.0 Section 6.4 must be followed.
- **Not handling the case-sensitivity of email lookups.** The login endpoint queries `db.user.findUnique({ where: { email } })`. After the migration, this query is case-sensitive. The mitigation is to use `mode: 'insensitive'` or change the column to `citext`. Forgetting this is a common bug.
- **Not testing the `pgloader` configuration against a sample export.** The `pgloader` configuration may have type mismatches that only appear with real data. Always test against a sample SQLite export before running the production migration.
- **Not running the verification script after the migration.** The `verify-migration.ts` script catches data integrity issues that are not visible by inspecting the tables. Always run it.
- **Not documenting the rollback procedure.** A migration without a documented rollback is a migration that the team is afraid to run. Always document the rollback.
- **Forgetting to test the `prisma migrate deploy` against a fresh PostgreSQL instance.** The migration may have SQL that works against SQLite but not against PostgreSQL (e.g., SQLite-specific functions, date handling differences). Always test against a fresh PostgreSQL instance.
- **Not updating the `CITEXT` extension or the `mode: 'insensitive'` for the email column.** The login endpoint is the most common user-facing query. Forgetting to handle the case sensitivity makes the login fail for users with mixed-case emails.
- **Not setting the `application_name` in the connection string.** The PostgreSQL logs show the application name for each connection. Without it, debugging connection issues is harder.

### 9.4 What to Do If You Get Stuck

- **The Prisma client connection pool is not working.** Check the `DATABASE_URL` format. The `?connection_limit=N` parameter must be at the end of the query string, not in the middle. Test with `psql $DATABASE_URL` to verify the connection string is correct.
- **The `pgloader` migration is failing.** Check the `pgloader` logs for the specific error. Common issues: type mismatches, constraint violations, missing tables. Adjust the configuration file and re-run.
- **The verification script is reporting FAIL.** Check the specific table and row that failed. The most common cause is a type mismatch (e.g., SQLite stores `0/1` for booleans, PostgreSQL stores `true/false`). The verification script should report the exact column and value that mismatched.
- **The email lookup is failing after the migration.** The query is case-sensitive in PostgreSQL. Use `mode: 'insensitive'` in the Prisma query. Update the login endpoint.
- **The `prisma migrate deploy` is failing.** Check the error message. Common issues: the database is unreachable, the credentials are wrong, a migration has a SQL syntax that PostgreSQL doesn't support. Test the migration against a fresh PostgreSQL instance.
