## 1. Document Purpose

This document describes the final verification of the Smart Queue Management System before production launch. It establishes the end-to-end integration test scenarios for every user role, the cross-browser compatibility checklist, the printer compatibility test, the FCM delivery verification procedure, the SSE stability test, the final security review checklist, and the updated `README.md`.

The deliverable is a **verified system** that has been tested end-to-end across every user role, every browser, every external system, and every security concern. The integration test scenarios are the manual walkthrough scripts. The security review checklist is the last gate before production. The README is the entry point for new developers and operators.

**Why this matters:** A production deployment without proper verification is a deployment that will fail in unexpected ways. Features that worked in isolation may fail when combined. External systems (printers, push notifications, browsers) may have quirks. Security vulnerabilities may be hidden. This document provides the verification procedures that catch all of these issues before the system is declared production-ready.

### Step 1: Document the End-to-End Integration Test Scenarios

**Files affected:** `docs/integration-test-scenarios.md` (new file).

**What to do:** Document the comprehensive end-to-end integration test scenarios for every user role. Each scenario has exact steps and expected outcomes. The scenarios are the manual walkthrough scripts for the QA team.

**Specific instructions:**

- Create the `docs/` directory if it doesn't exist.
- Create `docs/integration-test-scenarios.md` with the following sections:
  1. **Overview** — what these scenarios are, who runs them, when they're run (before every production deploy and after every major feature change), the expected duration (4-8 hours for a full pass).
  2. **Super-Admin Scenario** — log in with the seeded super-admin credentials → create a service (e.g., "General Inquiry", code "GEN", prefix "A") → create a counter (e.g., "Counter 1", number 1) → assign the service to the counter → create a user with the `ADMIN` role → log out → log in as the new admin → verify the admin can access the dashboard, the reports, and the audit log. The expected outcome: the admin can access everything except user management and system configuration.
  3. **Admin Scenario** — log in as the admin → view the reports dashboard → filter by service and counter → export a CSV → view the audit log → filter by action (e.g., "SERVICE_CREATED") → verify the service creation is recorded. The expected outcome: the CSV downloads, the audit log shows the entries.
  4. **Counter Officer Scenario** — log in as a counter officer assigned to Counter 1 → navigate to the officer dashboard → observe the queue depth → call a waiting ticket → observe the display board update → recall the ticket → mark as no-show (if grace period expired) → toggle the counter closed (with a reason) → observe the display board show the counter as closed → toggle notifications off → reopen the counter. The expected outcome: every action succeeds, the display board reflects the state, the audit log records the actions.
  5. **Security Officer Scenario** — log in as a security officer → observe the security screen → trigger an officer reply from a counter officer (via the counter officer's mobile app or via the admin UI) → observe the broadcast message arrive on the security screen → verify the message text and sender → click the broadcast to mark it as read. The expected outcome: the broadcast arrives within 2 seconds of the reply, the read state persists.
  6. **Kiosk Scenario** — open the kiosk in a browser (no authentication required) → select a service → observe the ticket issued and printed (silent print to the configured printer) → observe the confirmation screen with the estimated wait time → observe the auto-reset after the configured timeout (e.g., 30 seconds). The expected outcome: the ticket prints correctly, the confirmation screen shows the correct data, the auto-reset returns to the service selection.
  7. **Display Board Scenario** — open the display board in a browser → click to enable audio (the audio unlock overlay) → observe a ticket call from a counter officer → verify the "now serving" slot updates → verify the bell plays → verify the TTS announcement plays → observe a ticket recall → verify the recall re-plays the audio. The expected outcome: the visual and audio updates are synchronized, the audio is clear.
  8. **Daily Reset Scenario** — log in as super-admin → trigger a manual reset via the admin endpoint with the `confirm=RESET_TODAY` query parameter → verify the `Service.currentTicketNumber` resets to 0 for all services → verify the `QueueDailySnapshot` records are created for the previous business date → verify the snapshot contains the correct metrics. The expected outcome: the reset succeeds, the snapshots are correct.
  9. **Multi-Counter Scenario** — open 3+ counters → observe the display board showing all counters in a grid → issue tickets for different services → call tickets on different counters → verify the display board updates each counter independently. The expected outcome: the grid renders correctly, each counter shows its own "now serving" and recent calls.
  10. **Counter Closure Scenario** — open a counter → close it temporarily with a reason → verify the display board shows it as closed (with reduced opacity or a "Temporarily Closed" label) → verify the call/recall/no-show endpoints reject with 403 → reopen the counter → verify normal operation resumes. The expected outcome: the closure is visible, the endpoints reject, the reopening restores functionality.
  11. **Notification Toggle Scenario** — log in as a counter officer → toggle notifications off for a counter → issue a ticket for a service the counter handles → verify NO push notification is sent to the officer's device → toggle notifications back on → issue another ticket → verify the push notification IS sent. The expected outcome: the toggle is respected, the push notification is sent only when enabled.
  12. **Cross-Feature Scenario** — issue a ticket from the kiosk → observe the display board show the queue depth update → call the ticket from a counter → observe the display board show the call → reply from a counter officer's mobile (or simulate a reply) → observe the broadcast on the display board and the security screen → mark the ticket as completed. The expected outcome: the entire flow works end-to-end, every surface is consistent.

### Step 3: Document the Printer Compatibility Test

**Files affected:** `docs/printer-compatibility-test.md` (new file).

**What to do:** Document the printer compatibility test. The silent printing depends on the Chrome `--kiosk-printing` flag, the OS printer configuration, and the thermal printer's ESC/POS compatibility. All three must be tested together.

**Specific instructions:**

- Create `docs/printer-compatibility-test.md` with the following sections:
  1. **Overview** — what this test is, why silent printing is complex, the prerequisites (Chrome `--kiosk-printing` flag, OS printer configuration, target thermal printer).
  2. **Setup** —
  - Install Chrome (latest version).
  - Configure the target thermal printer as the default printer on the OS.
  - Set the paper width (80mm or 58mm) in the printer's OS settings.
  - Launch Chrome with the flags: `--kiosk --kiosk-printing --disable-pinch https://queue.example.com/kiosk`.
  - The `--kiosk-printing` flag enables silent printing (no dialog).
  - The `--kiosk` flag enables fullscreen mode.
  3. **Test Cases** —
  - **80mm paper, basic ticket:** issue a ticket, verify the ticket prints with the correct content (ticket number, service name, estimated wait, issue datetime). The ticket should fit within the 80mm width with 4mm margins.
  - **80mm paper, long service name:** issue a ticket for a service with a 50-character name. Verify the text wraps correctly and the ticket fits within the 80mm width.
  - **80mm paper, large queue depth:** issue a ticket when the queue depth is 99+. Verify the wait time estimate is correct and the ticket fits.
  - **58mm paper, basic ticket:** change the printer to 58mm, issue a ticket. Verify the ticket prints correctly with the narrower paper.
  - **58mm paper, truncation:** issue a ticket with a long service name. Verify the text truncates or wraps correctly.
  - **No printer configured:** launch Chrome without a configured printer. Verify the kiosk shows a "Printer not available" message (graceful degradation).
  4. **Supported Printer Specifications** —
  - **Type:** ESC/POS compatible thermal printer.
  - **Resolution:** 203 DPI minimum (8 dots/mm).
  - **Paper width:** 80mm (most common) or 58mm (smaller).
  - **Connection:** USB (most common) or network (via the OS's print server).
  - **Tested models:** document any specific printer models that have been tested and verified. The list is not exhaustive — any ESC/POS compatible thermal printer should work.
  5. **Troubleshooting** —
  - **Print dialog appears:** the `--kiosk-printing` flag is not set. Add it to the Chrome launch command.
  - **Nothing prints:** the printer is not configured as the default. Configure it in the OS settings.
  - **Text is cut off:** the paper width is incorrect. Set the correct width in the printer settings and in the print CSS.
  - **Print is slow:** the printer is slow. Acceptable for thermal printers; the silent print is triggered after the ticket issuance, not blocking the user.
  - **Garbled characters:** the printer's character encoding is incorrect. Set the encoding to UTF-8 or the printer's default (usually CP437 or CP850).

**Verification:** The `docs/printer-compatibility-test.md` file exists with all 5 sections. The test cases cover the common scenarios.

### Step 4: Document the FCM Delivery Verification Procedure

**Files affected:** `docs/fcm-delivery-verification.md` (new file).

**What to do:** Document the FCM delivery verification procedure. The procedure uses a real Android device to verify that push notifications are delivered correctly.

**Specific instructions:**

- Create `docs/fcm-delivery-verification.md` with the following sections:
  1. **Overview** — what this procedure is, why real-device testing is required (FCM delivery cannot be reliably tested in a simulator), the prerequisites.
  2. **Prerequisites** —
  - A real Android device with the FCM-enabled test app installed (or the future Android app when it's built).
  - The device is connected to the internet (Wi-Fi or mobile data).
  - The Google Play Services are installed and updated.
  - The device is registered as a device token for a counter officer (via `/api/notifications/devices/register`).
  3. **Test Procedure** —
  - **Test 1: Ticket Issued Notification.**
    1. Verify the counter officer is assigned to a counter that handles a service.
    2. Verify the officer's device token is registered and `isActive = true`.
    3. Issue a ticket for the service from the kiosk.
    4. Verify the push notification arrives on the device within 5 seconds.
    5. Verify the notification title is "New Ticket" and the body includes the ticket number and service name.
    6. Tap the notification. Verify the deep link opens the officer dashboard (or the future Android app's ticket detail screen).
  - **Test 2: Ticket Recalled Notification.**
    1. Call a ticket from the officer dashboard.
    2. Recall the ticket.
    3. Verify the push notification arrives on the device within 5 seconds (the officer who called the ticket is the recipient).
    4. Verify the notification title is "Ticket Recalled" and the body includes the ticket number.
  - **Test 3: Invalid Token Cleanup.**
    1. Manually deactivate the device token in the database (set `isActive = false`).
    2. Issue a ticket.
    3. Verify no push notification is sent (the deactivated token is skipped).
    4. Re-activate the token (set `isActive = true`).
    5. Issue another ticket.
    6. Verify the push notification is sent.
  - **Test 4: Multiple Devices.**
    1. Register 2+ device tokens for the same officer.
    2. Issue a ticket.
    3. Verify the push notification is sent to all registered devices.
  4. **Verification Checklist** —
  - [ ] The push notification arrives within 5 seconds of the trigger.
  - [ ] The notification title is correct.
  - [ ] The notification body is correct.
  - [ ] The notification data payload is correct (`type`, `notificationId`, `ticketId`, `ticketNumber`, `serviceId`, `serviceName`, `counterId`, `counterName`, `replyUrl`).
  - [ ] Tapping the notification opens the correct screen.
  - [ ] Invalid tokens are cleaned up automatically (via the FCM `INVALID_TOKEN` callback).
  - [ ] Multiple devices receive the notification.
  5. **Troubleshooting** —
  - **Notification not arriving:** check the device token is registered, the officer has `notificationsEnabled = true`, the service is assigned to a counter the officer handles, the FCM service account is configured, the FCM project ID is correct.
  - **Notification arrives but the data is wrong:** check the dispatch logic in 4.1.3, the `buildNotificationPayload` function, the `Notification` table.
  - **Notification arrives but the deep link is wrong:** check the FCM payload's `replyUrl` field, the Android app's intent filter.

**Verification:** The `docs/fcm-delivery-verification.md` file exists with all 5 sections. The test cases cover the common scenarios.

### Step 5: Document the SSE Stability Test

**Files affected:** `docs/sse-stability-test.md` (new file), `scripts/load-test-sse.ts` (new file, optional).

**What to do:** Document the SSE stability test that verifies the system can handle 50+ concurrent SSE connections and 100+ events over 5 minutes without dropping events or leaking memory.

**Specific instructions:**

- Create `docs/sse-stability-test.md` with the following sections:
  1. **Overview** — what this test is, why it's important (SSE is the real-time backbone of the system), the prerequisites.
  2. **Prerequisites** —
  - A staging environment that mirrors production.
  - A load testing script (e.g., `scripts/load-test-sse.ts`) that opens SSE connections from simulated clients.
  - A monitoring tool to track the server's heap usage (e.g., `node --inspect` or a Prometheus exporter).
  3. **Load Testing Script** —
  - Create `scripts/load-test-sse.ts` (optional but recommended) with:
    - An `EventSource` client that opens a connection to `/api/sse/global` (the public channel).
    - A counter for events received and events missed.
    - A heartbeat check (verify the heartbeat is received every 30 seconds).
    - A reconnect logic (if the connection drops, reconnect with exponential backoff).
    - The ability to spawn N concurrent clients.
  - The script is run with `tsx scripts/load-test-sse.ts --clients 50 --duration 300` (50 clients for 300 seconds).
  4. **Test Cases** —
  - **Test 1: 50 Concurrent Connections for 5 Minutes.**
    1. Start the load test with 50 clients.
    2. Wait 5 minutes.
    3. Verify all 50 connections are still open (no drops).
    4. Verify the heartbeats are received on schedule.
    5. Stop the load test.
    6. Check the server's heap usage — it should be stable (no upward trend).
  - **Test 2: 100 Concurrent Connections for 5 Minutes.**
    1. Same as Test 1 but with 100 clients.
    2. The server may need to be scaled up (more CPU, more memory) to handle 100 connections.
  - **Test 3: 50 Connections + 100 Events.**
    1. Start the load test with 50 clients.
    2. Trigger 100 ticket call events from a separate process (via the API or directly in the database).
    3. Verify every client receives every event (no drops).
    4. Verify the events arrive in the correct order (FIFO).
  - **Test 4: Abrupt Client Disconnect.**
    1. Start the load test with 50 clients.
    2. Abruptly close 25 clients (simulate browser tab close).
    3. Verify the server detects the disconnect and removes the clients from the manager.
    4. Verify the remaining 25 clients continue to receive events.
  - **Test 5: Network Timeout.**
    1. Start the load test with 50 clients.
    2. Simulate a 30-second network drop on 10 clients.
    3. Verify the clients reconnect after the network is restored.
    4. Verify no events are dropped (the events are buffered in the SSE manager, or the clients re-fetch the state).
  5. **Pass Criteria** —
  - [ ] Zero events dropped.
  - [ ] All clients reach the `open` state after reconnect attempts.
  - [ ] Heap usage is stable (no upward trend over 5 minutes).
  - [ ] No orphaned clients in the SSE manager (verified by checking the manager's internal state).
  6. **Troubleshooting** —
  - **Events are dropped:** check the SSE manager's `sendToClient` function, the `controller.enqueue` error handling, the connection cleanup on `cancel`.
  - **Heap usage is growing:** check for memory leaks in the SSE manager, the event payload (large payloads), the client registration.
  - **Clients fail to reconnect:** check the `useSSE` hook's reconnect logic, the exponential backoff, the `EventSource` API behavior.

**Verification:** The `docs/sse-stability-test.md` file exists with all 6 sections. The load test script (if created) opens connections and tracks events.

  12. **XSS Prevention** —

### Step 7: Create the SSE Load Test Script (Optional but Recommended)

**Files affected:** `scripts/load-test-sse.ts` (new file).

**What to do:** Create the TypeScript load test script for the SSE stability test. The script opens N concurrent SSE connections and tracks events received and missed.

**Specific instructions:**

- Create `scripts/load-test-sse.ts` with:
  - **Command-line arguments:** `--clients` (default 50), `--duration` (default 300 seconds), `--url` (default `http://localhost:3000/api/sse/global`).
  - **EventSource wrapper:** opens a connection to the SSE endpoint, tracks the connection state, tracks the events received, tracks the heartbeats.
  - **Reconnection logic:** on connection drop, reconnect with exponential backoff (1s, 2s, 4s, 8s, max 30s).
  - **Statistics:** total events received, total events missed, total reconnections, total time connected, average event latency.
  - **Output:** at the end of the test, print a summary table with the statistics. Exit with code 0 if no events were missed, code 1 otherwise.

## 8. Common Pitfalls

- **Skipping the FCM test on a real device.** FCM delivery cannot be reliably tested in a simulator. A real Android device is required. Skipping this test risks production push notifications failing silently.
- **Skipping the printer test on the target printer.** Silent printing depends on the Chrome `--kiosk-printing` flag, the OS printer configuration, and the thermal printer's ESC/POS compatibility. All three must be tested together. Skipping this test risks the kiosk failing to print tickets.
- **Not testing the SSE stability under load.** A few connections work fine; 50+ connections may expose leaks or efficiency issues that aren't visible in development. The load test catches these issues before production.
- **Not documenting the security review checklist.** The final security review is the last gate before production. A critical finding (e.g., a hardcoded secret, a missing security header, an open CORS policy) must be addressed before the system is deployed. Skipping the checklist is skipping the gate.
- **Not updating the README with the deployment instructions.** The README is the entry point for new developers and operators. A README without deployment instructions forces every new operator to figure out the deployment from scratch.
- **Not linking the documentation from the README.** The detailed documentation is useless if no one can find it. The README is the index.
- **Not running the integration tests against a staging environment that mirrors production.** The integration tests may pass against a development environment with SQLite but fail against a production environment with PostgreSQL. The staging environment must mirror production.
- **Not testing the email lookup with mixed-case emails after the PostgreSQL migration.** The login endpoint is case-sensitive in PostgreSQL. The mitigation is to use `mode: 'insensitive'`. Without testing, this bug may not be caught until production.
- **Not testing the security headers from a real browser.** The headers are set in `next.config.ts`, but the browser may strip them. Always test from a real browser.
- **Not testing the rate limit from a real network.** The rate limit is applied at the middleware, but the network may add latency or drop requests. Always test from a real network.
- **Not testing the CORS from a real origin.** The CORS headers are set in the middleware, but the browser may cache them or the preflight may behave differently. Always test from a real origin.
- **Not testing the audit log viewer with real data.** The viewer may render correctly with empty data but fail with 10,000 entries. Always test with realistic data volumes.
