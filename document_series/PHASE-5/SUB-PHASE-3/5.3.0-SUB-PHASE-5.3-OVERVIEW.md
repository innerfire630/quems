## 1. Sub-Phase Purpose

Sub-Phase 5.3 takes the **production-grade** system from Sub-Phase 5.2 and makes it **production-deployed**. Where Phases 1–4 built features, 5.1 made them observable, and 5.2 made them secure and performant, 5.3 takes the working system from local SQLite development to a real production environment with PostgreSQL, CI/CD automation, and a comprehensive integration test suite.

The strategic goal is to **transform the production-ready system into a production-deployed system**. This means:

- **PostgreSQL migration** (5.3.1) so the system runs against a production-grade database that supports concurrent writes, transactions, JSONB queries, and the data volumes of a real queue operation.
- **Environment configuration and CI/CD** (5.3.2) so every deployment is reproducible, secrets are managed securely, and the pipeline catches regressions before they reach production.
- **Final integration and production checklist** (5.3.3) so the system is verified end-to-end across every user role, every browser, every external system (printer, push notification), and every security concern before operations takes over.

When this sub-phase is complete, the system is **fully production-deployed** and the DDD document series is complete. The work transitions from development to operations: monitoring, incident response, and feature evolution.

### 1.1 Why This Sub-Phase Comes After 5.2

Sub-Phase 5.3 comes after 5.2 (hardening) because:

- **The CI/CD pipeline (5.3.2) needs to run the lint, type-check, and build steps.** If the rate limiting (5.2.1), caching (5.2.2), or audit log viewer (5.2.3) is broken, the pipeline fails. Building 5.3 after 5.2 ensures the codebase is in a known-good state before deployment automation is added.
- **The PostgreSQL migration (5.3.1) needs to apply cleanly against the optimised schema.** The database indexes from 5.2.2 must be valid in PostgreSQL. The case-sensitive string comparisons and JSONB column handling must be reviewed against the query patterns the application uses.
- **The smoke test checklist (5.3.3) needs to verify hardening.** The checklist includes rate limiting verification (`send 11 auth requests in 1 minute`), security header verification (`every response carries the headers`), audit log verification (`viewer renders entries`), and health check verification (`/api/health` returns 200). These checks depend on 5.2 being complete.
- **The migration is the very last technical concern.** Migrating to PostgreSQL before the system is hardened means re-migrating after security changes. Migrating after hardening means the production database is correct from the first deployment.

### 1.2 Why This Sub-Phase Is the Last

The DDD document series is complete at Sub-Phase 5.3. The system's feature set, operational surface, hardening layer, and deployment story are all in place. Future work after 5.3 falls into three categories, none of which are part of this series:

1. **Operations** — running the system in production, monitoring, incident response, capacity planning, scaling decisions.
2. **New features** — the master plan's Section 2.3 lists explicit "out of scope" items (native Android app, SMS/WhatsApp notifications, payment processing, etc.) that are future enhancements.
3. **Scaling** — when the system needs to handle more load than a single server can support (multi-server SSE with Redis Pub/Sub, horizontal scaling, connection pooling beyond a single PostgreSQL instance).

These are not Sub-Phase 5.4. They are post-series work that depends on the production deployment established by Phase 5.3.

### 1.3 What "Done" Means for Sub-Phase 5.3

Sub-Phase 5.3 is complete when:

1. The Prisma schema can be swapped from `sqlite` to `postgresql` with a single line change and `prisma migrate deploy` produces a working schema against a fresh PostgreSQL instance.
2. The complete `.env.production` variable list is documented, every variable is validated at application boot, and the CI/CD pipeline runs lint → type-check → build → migrate → deploy reproducibly.
3. The data migration path from any existing SQLite database to PostgreSQL is documented and tested (using `pgloader` or equivalent tool).
4. The post-deployment smoke test checklist has been executed against a staging environment that mirrors production.
5. All end-to-end integration test scenarios for every user role (super-admin, admin, counter officer, security officer, kiosk user, display viewer) have passed.
6. The cross-browser compatibility test (Chrome desktop, Chrome Android, Safari macOS) has passed.
7. The printer compatibility test (silent printing via Chrome `--kiosk-printing` flag) has passed against the target thermal printer.
8. The FCM delivery verification test (real Android device receiving a push notification) has passed.
9. The SSE stability test (50+ concurrent connections, 100+ events over 5 minutes) has passed without dropped events or memory leaks.
10. The final security review checklist (auth, RBAC, security headers, rate limiting, CORS, audit log, no secrets in repo) is complete with no critical findings.

When all of the above are true, the system is production-deployed and the DDD series is complete.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to distribute the complexity across the three task documents. The split is:

- **5.3.1** owns the **database layer**. It establishes the PostgreSQL provider swap, the `prisma migrate deploy` workflow, the `pgloader` data migration (if applicable), and the PostgreSQL-specific query patterns (case sensitivity, JSONB, connection pooling, transaction isolation, backups). It establishes the complete environment variable list, the Vercel/self-hosted deployment configuration, the CI/CD pipeline, the secrets management, and the post-deployment smoke test checklist. It does **not** know about the data migration (5.3.1) or the integration tests (5.3.3). It makes the deployment reproducible.
- **5.3.3** owns the **verification layer**. It establishes the end-to-end integration test scenarios for each user role, the cross-browser compatibility test, the printer compatibility test, the FCM delivery verification, the SSE stability test, and the final security review. It does **not** know about the database migration (5.3.1) or the CI/CD pipeline (5.3.2). It verifies the system is production-ready.

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 5.3.1 — PostgreSQL Migration Strategy

**Owns:** The PostgreSQL provider swap, the `prisma migrate deploy` workflow, the `pgloader` data migration, the PostgreSQL-specific query patterns (case sensitivity, JSONB, connection pooling, transaction isolation, backups), the connection pool sizing constants in `src/lib/db.ts`, the migration verification scripts.

**Does NOT own:** The CI/CD pipeline (owned by 5.3.2), the integration tests (owned by 5.3.3), the environment variables beyond `DATABASE_URL` and `DIRECT_URL` (owned by 5.3.2).

**Key outputs:**

- **`prisma/schema.prisma`** — DOCUMENTED single-line change to swap the `provider` from `sqlite` to `postgresql`. The change is NOT applied in the repository (the file remains `sqlite` for local development). The migration runbook (`docs/postgresql-migration.md`) documents the exact change and the rollback procedure.
- **`prisma/migrations/`** — All migrations from Phase 1.1.3 onward are present. They apply cleanly to either SQLite or PostgreSQL. New migration from 5.2.2 (`add_performance_indexes`) is included. The migration order is preserved.
- **`docs/postgresql-migration.md`** — the comprehensive migration runbook:
  - **Two-step migration process:** (1) change `provider` to `postgresql`, (2) run `prisma migrate deploy` against the production PostgreSQL instance.
  - **Environment variable changes:** `DATABASE_URL` from SQLite file path to PostgreSQL connection string. `DIRECT_URL` if needed for migration operations that bypass connection pooling.
  - **Data migration strategy:** for fresh deployments, no data migration. For existing deployments, the `pgloader` workflow with the configuration file and verification queries.
  - **PostgreSQL-specific considerations:** UUID vs CUID, JSONB column handling, case sensitivity (with `LOWER()` / `ILIKE` guidance), connection pool sizing, transaction isolation level, backup strategy.
  - **Rollback procedure:** documented for emergency rollback to a previous PostgreSQL state.
  - **Verification checklist:** the queries and commands the operator runs to verify the migration succeeded.
- **`scripts/migration/pgloader.config`** — the `pgloader` configuration file for migrating from SQLite to PostgreSQL:
  - Source: SQLite file path.
  - Target: PostgreSQL connection string.
  - Schema mapping rules (e.g., `TEXT` → `text`, `INTEGER` → `integer`, `DATETIME` → `timestamp`).
  - Casting rules for any type mismatches.
  - Include/exclude rules (e.g., skip `__drizzle_migrations` or other internal tables).
  - Worker count and batch size.
- **`scripts/migration/verify-migration.ts`** — the data migration verification script:
  - Connects to the source SQLite and the target PostgreSQL.
  - For each table, compares `COUNT(*)` between source and target.
  - For each table, samples 10 random rows and compares every column.
  - Outputs a pass/fail report.
  - Exits with non-zero code if any check fails.
- **`src/lib/db.ts`** — EXTENDED with PostgreSQL connection pool configuration:
  - Reads the `DATABASE_URL` and parses the connection limit.
  - Configures the Prisma client with the connection limit (e.g., 10 connections per application instance).
  - Configures the connection timeout (e.g., 10 seconds).
  - Configures the `DIRECT_URL` if set (for migrations that require a direct connection).
  - The configuration is environment-aware: in development, uses SQLite defaults; in production, uses the PostgreSQL pool.
- **PostgreSQL-specific query discipline documentation** in `docs/postgresql-query-discipline.md`:
  - **Case sensitivity:** all `WHERE` clauses that perform case-insensitive comparisons must use `LOWER()` or Prisma's `mode: 'insensitive'`. Email lookups are flagged as the primary case-insensitive query.
  - **JSONB queries:** use Prisma's `Json` filter syntax (`path`, `string_contains`, `equals`, etc.) which maps to PostgreSQL's `JSONB` operators.
  - **Date/time handling:** all timestamps are stored as UTC. Application code must convert to local time for display via `APP_TIMEZONE` and `Intl.DateTimeFormat`. Never store local time.
  - **Connection pool sizing:** documented formula: `pool_size = floor((max_connections - reserved) / num_app_instances)`. The default PostgreSQL `max_connections` is 100; reserve 10 for admin; if 3 app instances, each gets `floor((100 - 10) / 3) = 30` connections.
  - **Transaction isolation:** the default `READ COMMITTED` is used. Documented reasoning (sufficient for queue workload, avoids serialization conflicts).
  - **Backup strategy:** documented procedure using `pg_dump` for daily backups, retention period, offsite storage.

**Master Plan sections implemented:** Section 3.2 (database choice — production), Section 17 (decisions log — PostgreSQL choice).

### 4.3 Document 5.3.3 — Final Integration & Production Checklist

**Owns:** The end-to-end integration test scenarios for every user role, the cross-browser compatibility checklist, the printer compatibility test, the FCM delivery verification procedure, the SSE stability test, the final security review checklist, the updated `README.md`, the operations runbook content for verification.

  - **Super-admin scenario:** log in → create a service → create a counter → assign the service to the counter → create a user with `ADMIN` role → log out → log in as the new admin → verify access. The exact steps and expected outcomes.
  - **Admin scenario:** log in → view the reports dashboard → filter by service → export a CSV → view the audit log → filter by action. The exact steps and expected outcomes.
  - **Counter officer scenario:** log in → navigate to the officer dashboard → call a waiting ticket → recall the ticket → mark as no-show → toggle the counter closed (with a reason) → reopen the counter → toggle notifications off. The exact steps and expected outcomes.
  - **Security officer scenario:** log in as a security officer → observe a broadcast message arrive (triggered by a counter officer reply) → verify the message text and sender. The exact steps and expected outcomes.
  - **Kiosk scenario:** load the kiosk → select a service → observe the ticket issued and printed (silent print) → observe the confirmation screen → observe the auto-reset. The exact steps and expected outcomes.
  - **Display board scenario:** load the display board → click to enable audio → observe a ticket call update the board → trigger the bell + TTS announcement. The exact steps and expected outcomes.
  - **Daily reset scenario:** trigger a manual reset (via the admin endpoint with `confirm=RESET_TODAY`) → verify the ticket counters reset → verify the `QueueDailySnapshot` is created with correct metrics. The exact steps and expected outcomes.
  - **Multi-counter scenario:** 3+ counters open → observe the display board showing all counters in the grid → issue tickets for different services → call tickets on different counters → verify the display board updates correctly. The exact steps and expected outcomes.
  - **Counter closure scenario:** open a counter → close it temporarily with a reason → verify the display board shows it as closed → verify the call/recall/no-show endpoints reject with 403 → reopen the counter → verify normal operation resumes. The exact steps and expected outcomes.
- **`docs/cross-browser-checklist.md`** — the cross-browser compatibility checklist:

### 6.4 PostgreSQL-Specific Query Discipline

After the migration to PostgreSQL, all queries must be reviewed for:

- **Case-sensitive comparisons:** use `LOWER()` or Prisma's `mode: 'insensitive'` for case-insensitive matches. The primary case-insensitive query in the system is the email lookup on login (`db.user.findUnique({ where: { email } })`). After the migration, this query must be reviewed and either migrated to `db.user.findFirst({ where: { email: { equals: email, mode: 'insensitive' } } })` or the `email` column must be changed to a `citext` type.
- **Connection pool sizing:** the Prisma pool size must be set to stay within PostgreSQL's connection limit. The default PostgreSQL `max_connections` is 100. The formula: `pool_size = floor((max_connections - reserved) / num_app_instances)`. With 3 application instances and 10 reserved connections: `pool_size = floor((100 - 10) / 3) = 30` per instance.
- **JSON queries:** use Prisma's `Json` filter syntax (`path`, `string_contains`, `equals`, etc.) which maps to `JSONB` operators in PostgreSQL.
- **Date/time zones:** all timestamps are stored as UTC. Application code must convert to local time for display via `APP_TIMEZONE` and `Intl.DateTimeFormat`. Never store local time.
- **Transaction isolation:** the default `READ COMMITTED` is used. This is sufficient for the queue workload (no high-contention serializable operations). Documented in the migration runbook.

- The pipeline runs on every push to `main` and every pull request to `main`.

- The pipeline MUST run all steps: `yarn lint`, `yarn type-check`, `yarn build`.

### 6.8 Environment Variable Validation Discipline

- The production environment variables are validated at application startup.
- The `lib/env.ts` module (or similar) reads every required variable and throws a clear error if any are missing or malformed.
- The validation runs once on boot and is the first thing the application does.
- The validation is environment-aware: in development, missing variables get a console warning; in production, missing variables are fatal.
- The validation rules:

### 8.6 Common Pitfalls

- **Committing secrets to the repository.** The `.env` file is gitignored. The `.env.example` file uses placeholders. The `service-account.json` is gitignored. Verify with `git status` and `git log --all` before each commit.
- **Applying the `provider` change to `prisma/schema.prisma` in the repository.** The change is applied by the operator during the migration, NOT by the agent. Keeping the repository on SQLite allows local development to continue.
- **Forgetting to update `.env.example` with new variables.** Every variable the system reads must be in `.env.example` with a description.
- **Forgetting the `DIRECT_URL` for PostgreSQL migrations.** Some hosting platforms (e.g., Heroku, Supabase) require a `DIRECT_URL` for migrations that bypass connection pooling. Document this in the migration runbook.
- **Not setting `proxy_buffering off` in the nginx configuration.** The SSE stream requires `proxy_buffering off` in nginx. Without it, nginx buffers the SSE events and the client doesn't receive them in real-time.
- **Not running the Prisma migration during the Vercel build.** Vercel does NOT automatically run `prisma migrate deploy`. The migration must be triggered via a build hook or a custom build step.
- **Forgetting the `NEXTAUTH_URL` HTTPS requirement.** In production, `NEXTAUTH_URL` must be HTTPS. NextAuth requires this for secure cookies.
- **Not setting the `NEXTAUTH_SECRET` to a strong value.** The default development secret must NOT be used in production. The validator throws on the default value.
- **Not testing the FCM delivery against a real device.** FCM delivery cannot be reliably tested in a simulator. A real Android device is required.
- **Not testing the printer compatibility against the target thermal printer.** Silent printing depends on the Chrome `--kiosk-printing` flag, the OS printer configuration, and the thermal printer's ESC/POS compatibility. All three must be tested together.
- **Not testing the SSE stability under load.** A few connections work fine; 50+ connections may expose leaks or efficiency issues that aren't visible in development.
