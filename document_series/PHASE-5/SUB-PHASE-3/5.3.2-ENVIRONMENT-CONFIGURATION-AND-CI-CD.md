## 1. Document Purpose

This document describes the complete environment configuration and CI/CD pipeline for deploying the Smart Queue Management System to production. It establishes the comprehensive `.env.production` variable list, the env validator extensions, the GitHub Actions CI and deploy workflows, the Vercel deployment configuration, the self-hosted deployment runbook, the secrets management convention, the post-deployment smoke test checklist, and the operations runbook.

The deliverable is a **reproducible deployment** that the operations team can execute with confidence. The CI pipeline catches regressions before they reach production. The deploy workflow applies migrations and deploys with manual approval. The smoke test verifies the system is operational after every deployment. The operations runbook documents every procedure the operator might need.

**Why this matters:** A production deployment without proper configuration is a deployment that will fail. Environment variables are misconfigured, secrets are committed to the repository, the CI pipeline doesn't catch type errors, the deploy workflow doesn't apply migrations, and the smoke test is ad-hoc. This document prevents all of these failure modes by providing the configuration, the automation, and the verification procedures as the single source of truth.

### Step 1: Update `.env.example` with Every Variable

**Files affected:** `.env.example` (updated or created).

**What to do:** Create or update the `.env.example` file at the project root with every environment variable the system reads. The file is the template that developers copy to `.env` for local development. It is committed to the repository (unlike `.env`).

**Specific instructions:**

- Create or update `.env.example` with the following structure:
  - **Header comment** explaining what the file is and how to use it.
  - **Section: Database** with `DATABASE_URL` (example: `file:./prisma/dev.db` for SQLite dev, or `postgresql://user:password@host:5432/database?schema=public` for production) and `DIRECT_URL` (optional, example: `postgresql://user:password@host:5432/database?schema=public`).
  - **Section: Authentication** with `NEXTAUTH_SECRET` (example: a long random string), `NEXTAUTH_URL` (example: `http://localhost:3000` for dev, `https://queue.example.com` for production).
  - **Section: Push Notifications** with `FCM_SERVICE_ACCOUNT_JSON` (example: a JSON string or path) and `FCM_PROJECT_ID` (example: `my-firebase-project`).
  - **Section: CORS** with `ALLOWED_MOBILE_ORIGINS` (optional, example: `https://app.example.com,https://admin.example.com`).
  - **Section: Application** with `APP_TIMEZONE` (optional, example: `Asia/Colombo`), `NODE_ENV` (example: `development`).
  - **Section: Optional Hardening** with `RATE_LIMIT_ENABLED` (optional, default `true` in production), `MAX_REQUEST_BODY_SIZE` (optional, default `10485760`), `PRISMA_LOG_QUERIES` (optional, default `true` in development).
  - **Section: Optional Database Pool** with `DATABASE_POOL_SIZE` (optional, default computed), `DATABASE_CONNECT_TIMEOUT_MS` (optional, default `10000`).
- Every variable has a comment above it explaining what it is, an example value, and whether it's required or optional.
- No secrets are in the file. Every value is a placeholder.
- The file is in the standard `KEY=value` format, one variable per line.

**Verification:** The `.env.example` file exists at the project root and lists every variable with a comment.

### Step 2: Extend the Env Validator with Production Variables

**Files affected:** `src/lib/env.ts` (updated).

**What to do:** Update the env validator to validate every production variable. The validation is environment-aware — stricter in production, more lenient in development.

**Specific instructions:**

- Read the existing `src/lib/env.ts` to understand the current validation pattern (Zod schemas, per-environment validation).
- Add or extend the Zod schema for every variable:
  - `DATABASE_URL` — must be a valid URL. In production, must start with `postgresql://` or `postgres://`.
  - `DIRECT_URL` — optional. If set, must be a valid URL. In production, if set, must start with `postgresql://` or `postgres://`.
  - `NEXTAUTH_SECRET` — must be at least 32 characters. In production, must NOT be the development default value (a hard-coded string like `'development-secret-do-not-use-in-production'`).
  - `NEXTAUTH_URL` — must be a valid URL. In production, must be HTTPS.
  - `FCM_SERVICE_ACCOUNT_JSON` — optional in development, required in production. If set, must be valid JSON (use `z.string().refine((v) => { try { JSON.parse(v); return true; } catch { return false; } })`).
  - `FCM_PROJECT_ID` — optional in development, required in production. Must be a non-empty string.
  - `ALLOWED_MOBILE_ORIGINS` — optional. If set, must be a comma-separated list of valid URLs. Validate each URL with `z.string().url()`.
  - `APP_TIMEZONE` — optional. If set, must be a valid IANA timezone. Validate against `Intl.supportedValuesOf('timeZone')`.
  - `RATE_LIMIT_ENABLED` — optional. Default `'true'` in production, `'false'` in development. Coerce string to boolean.
  - `MAX_REQUEST_BODY_SIZE` — optional. Default `'10485760'` (10 MB). Coerce to number.
  - `PRISMA_LOG_QUERIES` — optional. Default `'true'` in development, `'false'` in production. Coerce to boolean.
  - `DATABASE_POOL_SIZE` — optional. If set, must be a positive integer.
  - `DATABASE_CONNECT_TIMEOUT_MS` — optional. If set, must be a positive integer.
- The validation throws on boot in production with a clear error message. Example: `[env] FATAL: NEXTAUTH_SECRET must be at least 32 characters. See docs/environment-configuration.md.`
- The validation logs a warning in development but does not throw. Example: `[env] WARNING: FCM_SERVICE_ACCOUNT_JSON is not set — running in console-log mode.`

**Verification:** The env validator throws on missing or malformed values in production. The validator logs warnings in development.

### Step 3: Create the GitHub Actions CI Workflow

**Files affected:** `.github/workflows/ci.yml` (new file).

**What to do:** Create the CI workflow that runs on every push and pull request to the main branch. The workflow runs lint, type-check, and build. The workflow must pass before any merge.

**Specific instructions:**

- Create the `.github/workflows/` directory at the project root.
- Create `.github/workflows/ci.yml` with:
  - **Name:** `CI`.
  - **Triggers:** `on.push.branches: [main]`, `on.pull_request.branches: [main]`.
  - **Concurrency:** cancel in-progress runs for the same PR (`concurrency.group: ${{ github.workflow }}-${{ github.ref }}`, `concurrency.cancel-in-progress: true`).
  - **Jobs:** one job named `build` that runs on `ubuntu-latest`.
  - **Steps:**
    1. **Checkout:** `actions/checkout@v4` (or latest stable).
    2. **Setup Node.js:** `actions/setup-node@v4` with `node-version: 20` (or the version pinned in `.nvmrc`).
    3. **Cache Yarn dependencies:** `actions/cache@v4` with `path: ~/.cache/yarn`, `key: ${{ runner.os }}-yarn-${{ hashFiles('**/yarn.lock') }}`, `restore-keys: ${{ runner.os }}-yarn-`.
    4. **Install dependencies:** `yarn install --frozen-lockfile`.
    5. **Lint:** `yarn lint`.
    6. **Type-check:** `yarn type-check`.
    7. **Build:** `yarn build`. Set the env var `DATABASE_URL` to a dummy SQLite URL for the build (`file:./prisma/dev.db`).
  - **Environment variables:** the build step needs `DATABASE_URL` to run `prisma generate` (which is part of `yarn build`). Set it to a dummy SQLite URL.
- The workflow uses Yarn 1.x (the standard Yarn version for this project). No Yarn 2/3/4 features.
- The workflow does NOT include any deploy steps. Deploy is a separate workflow.

**Verification:** The `.github/workflows/ci.yml` file exists with the correct steps. Pushing to a branch triggers the workflow. The workflow passes locally with `act` (if installed).

### Step 4: Create the GitHub Actions Deploy Workflow

**Files affected:** `.github/workflows/deploy.yml` (new file).

**What to do:** Create the deploy workflow that runs after CI passes on the main branch. The workflow applies the Prisma migration and deploys to the hosting platform. The production deploy requires manual approval.

**Specific instructions:**

- Create `.github/workflows/deploy.yml` with:
  - **Name:** `Deploy`.
  - **Triggers:** `on.push.branches: [main]` (after CI passes), `on.workflow_dispatch` (manual trigger).
  - **Concurrency:** do NOT cancel in-progress deploys (`concurrency.cancel-in-progress: false`). Deploys must complete.
  - **Jobs:** one job named `deploy` that runs on `ubuntu-latest`.
  - **Environment:** the job uses a GitHub Actions Environment (e.g., `production` or `staging`). The environment has a protection rule that requires manual approval for the `production` environment.
  - **Steps:**
    1. **Checkout:** `actions/checkout@v4`.
    2. **Setup Node.js:** `actions/setup-node@v4` with `node-version: 20`.
    3. **Cache Yarn dependencies:** same as the CI workflow.
    4. **Install dependencies:** `yarn install --frozen-lockfile`.
    5. **Apply Prisma migration:** `yarn prisma:migrate:deploy` with `DATABASE_URL` set from the GitHub Actions Secret.
    6. **Deploy to Vercel:** use the `amondnet/vercel-action@v25` (or latest stable) action with `vercel-token`, `vercel-org-id`, `vercel-project-id`, `vercel-args: '--prod'`. For staging, use `vercel-args: ''` (no `--prod` flag).
    7. **(Alternative) Deploy to self-hosted:** use `appleboy/ssh-action@v1` (or latest stable) to SSH into the server and run the deploy script.
  - **Secrets:** the workflow uses GitHub Actions Secrets for `DATABASE_URL`, `NEXTAUTH_SECRET`, `FCM_SERVICE_ACCOUNT_JSON`, `FCM_PROJECT_ID`, and any other production secrets. These are stored in the GitHub repository settings under `Settings > Secrets and variables > Actions`.
  - **Environment protection:** the `production` environment has a required reviewer (e.g., the team lead). The deploy step waits for approval before proceeding.
- The workflow does NOT include any test steps. Tests are run in the CI workflow.
- The workflow logs the deploy time, the commit SHA, and the actor for audit purposes.

**Verification:** The `.github/workflows/deploy.yml` file exists with the correct steps. Pushing to `main` triggers the workflow. The workflow requires manual approval for the `production` environment.

### Step 5: Create the Vercel Deployment Configuration

**Files affected:** `vercel.json` (new file, if not present) or Vercel project settings documentation in the runbook.

**What to do:** Document the Vercel deployment configuration. The application is a Next.js 14+ app, deployable to Vercel with zero configuration. The Vercel project settings handle the environment variables, the build command, and the postbuild Prisma migration.

**Specific instructions:**

- Create `vercel.json` at the project root with:
  - **`buildCommand`:** `yarn build` (or the Vercel default, which is `next build`).
  - **`installCommand`:** `yarn install --frozen-lockfile`.
  - **`framework`:** `nextjs` (Vercel auto-detects this, but explicit is better).
  - **`regions`:** the deployment region (e.g., `sin1` for Singapore). The default is the region closest to the user.
  - **`headers`:** the security headers from 5.2.1 are set in `next.config.ts` (already done). No additional headers in `vercel.json`.
  - **`redirects`:** the HTTP-to-HTTPS redirect is handled by Vercel by default. No additional redirects in `vercel.json`.
- Document the Vercel project settings in `docs/self-hosted-deployment.md` (or a separate `docs/vercel-deployment.md`):
  - **Environment Variables:** set in the Vercel project UI under `Settings > Environment Variables`. Add every variable from `.env.example` with the production values. Mark `NEXTAUTH_SECRET`, `DATABASE_URL`, and FCM credentials as sensitive (Vercel encrypts them).
  - **Build & Development Settings:** the defaults are correct. Confirm the build command is `yarn build` and the output directory is `.next`.
  - **Postbuild Prisma migration:** Vercel does NOT automatically run `prisma migrate deploy`. The migration must be triggered via a build hook or a custom build step. The recommended approach is to add the migration to the `buildCommand` in `vercel.json`:
    - `"buildCommand": "prisma migrate deploy && next build"`
    - OR use a `postinstall` script in `package.json` that runs `prisma generate` (for the client) and the deploy workflow runs the migration step separately.
  - **Domain:** the production domain (e.g., `queue.example.com`) is added under `Settings > Domains`. Vercel automatically provisions a TLS certificate via Let's Encrypt.
  - **Branch deployments:** every branch gets a preview deployment. The `DATABASE_URL` for preview deployments can be a separate staging database (or a shared staging database with a branch-specific schema).

**Verification:** The `vercel.json` file exists with the correct configuration. The Vercel project settings are documented in the runbook.

### Step 6: Create the Self-Hosted Deployment Runbook

**Files affected:** `docs/self-hosted-deployment.md` (new file).

**What to do:** Document the complete self-hosted deployment procedure. The runbook covers prerequisites, build, database migration, systemd service, nginx configuration, TLS setup, and process supervision.

**Specific instructions:**

- Create `docs/self-hosted-deployment.md` with the following sections:
  1. **Overview** — what this document is, who should read it, the prerequisites.
  2. **Prerequisites** — Node.js 20+, PostgreSQL 15+, nginx 1.20+, systemd, a domain name, a server with at least 2 GB RAM and 20 GB disk.
  3. **Initial Server Setup** — create a non-root user (`quems`), set up the directory structure (`/opt/quems` for the application, `/etc/quems` for the configuration, `/var/log/quems` for the logs), configure the firewall (allow 22, 80, 443).
  4. **Database Setup** — install PostgreSQL, create the `quems` database and user, set the password, configure `pg_hba.conf` for local connections, test the connection.
  5. **Application Build** — clone the repository, install dependencies with `yarn install --frozen-lockfile`, set the environment variables in `/etc/quems/secrets.env`, generate the Prisma client with `yarn prisma:generate`, build the application with `yarn build`.
  6. **Database Migration** — run `yarn prisma:migrate:deploy` against the production database.
  7. **Database Seeding** — run `yarn prisma:seed` to create the default roles, permissions, and super-admin user.
  8. **Systemd Service** — create `/etc/systemd/system/quems.service` with the service definition: `Type=simple`, `User=quems`, `WorkingDirectory=/opt/quems`, `EnvironmentFile=/etc/quems/secrets.env`, `ExecStart=/usr/bin/yarn start`, `Restart=on-failure`, `RestartSec=5s`. Enable and start the service with `systemctl enable quems && systemctl start quems`.
  9. **Nginx Configuration** — create `/etc/nginx/sites-available/quems.conf` with the reverse proxy config: `proxy_pass http://localhost:3000`, `proxy_http_version 1.1`, `proxy_set_header Upgrade $http_upgrade`, `proxy_set_header Connection 'upgrade'`, `proxy_set_header Host $host`, `proxy_set_header X-Real-IP $remote_addr`, `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for`, `proxy_set_header X-Forwarded-Proto $scheme`, **`proxy_buffering off`** (CRITICAL for SSE), `proxy_read_timeout 86400` (CRITICAL for SSE long-lived connections), `proxy_cache off`. Enable the site with `ln -s /etc/nginx/sites-available/quems.conf /etc/nginx/sites-enabled/`.
  10. **TLS Setup** — use Let's Encrypt with `certbot --nginx -d queue.example.com`. Certbot automatically configures nginx for HTTPS and sets up auto-renewal.
  11. **Process Supervision** — systemd restarts the service on failure. Logs are in `journalctl -u quems`.
  12. **Log Aggregation** — `journalctl` for systemd logs. Optional forwarding to a log aggregation service (e.g., Papertrail, Loggly).
  13. **Backup Setup** — daily `pg_dump` to `/var/backups/quems/` with a cron job. Offsite upload to S3. Retention: 30 days local, 1 year cold storage.
  14. **Monitoring Setup** — external monitoring (Pingdom, Datadog, UptimeRobot) calls `/api/health` every 60 seconds. Alert on 503 response.
  15. **Deployment Procedure** — the step-by-step for deploying a new version: pull the latest code, install dependencies, run the migration, restart the service, verify with the smoke test.
  16. **Rollback Procedure** — the step-by-step for rolling back: check out the previous commit, rebuild, restart the service, verify with the smoke test.

**Verification:** The `docs/self-hosted-deployment.md` file exists with all 16 sections. The systemd service file, nginx config, and TLS setup are documented.

### Step 8: Create the Operations Runbook

**Files affected:** `docs/operations-runbook.md` (new file).

**What to do:** Document the operations procedures that the operations team follows. The runbook covers the smoke test, incident response, backup/restore, secret rotation, scaling, and deployment rollback.

**Specific instructions:**

- Create `docs/operations-runbook.md` with the following sections:
  1. **Overview** — what this runbook is, who should read it, the relationship to the smoke test checklist and the self-hosted deployment runbook.
  2. **Smoke Test Execution** — the link to `docs/smoke-test-checklist.md` and the procedure for executing it (within 30 minutes of every deploy, recorded in the deployment log).
  3. **Incident Response** —
    - **When `/api/health` returns 503:** the database is unreachable. Check the database server, the connection string, the network. The error message in the health response indicates the cause.
    - **When a 5xx spike is observed:** check the application logs (`journalctl -u quems` for self-hosted, Vercel logs for hosted), check the database logs, check the error tracking system (if configured).
    - **When the database is unreachable:** fail over to the standby (if using a managed service with HA), or restore from the most recent backup.
    - **When the application is unresponsive:** check the systemd service status (`systemctl status quems`), check the application logs, check the resource usage (`htop`, `df -h`).
    - **Communication:** notify the team immediately. Document the incident in the incident log. Post-mortem within 48 hours.
  4. **Database Backup and Restore** —
    - **Manual backup:** `pg_dump -Fc -d quems_prod -f /var/backups/quems/backup-$(date +%Y%m%d).dump`.
    - **Scheduled backup:** a cron job runs `pg_dump` daily at 02:00 UTC. The backup is compressed and uploaded to S3.
    - **Restore:** `pg_restore -d quems_prod /var/backups/quems/backup-YYYYMMDD.dump`. Verify the restore with the verification script.
    - **Retention:** 30 days local, 1 year cold storage.
    - **Test the restore:** quarterly. A backup that has never been restored is not a backup.
  5. **Secret Rotation** —
    - **`NEXTAUTH_SECRET`:** rotate annually. Generate a new value with `openssl rand -base64 32`. Update the env var. Restart the service. All existing sessions are invalidated (users must log in again).
    - **Database password:** rotate quarterly. Generate a new value with `openssl rand -base64 24`. Update the password in PostgreSQL (`ALTER USER quems WITH PASSWORD 'new_password';`). Update the env var. Restart the service.
    - **FCM service account JSON:** rotate annually. Download a new key from the Firebase Console. Update the env var. Restart the service.
  6. **Scaling** —
    - **Vertical scaling:** increase the server size (CPU, RAM). Restart the service. Monitor the connection pool usage and the response time.
    - **Horizontal scaling:** add another application instance. Update the nginx config to load-balance. The PostgreSQL connection limit must be considered — each instance adds to the connection count.
    - **Database scaling:** move to a larger instance, add read replicas, or shard (future enhancement).
  7. **Deployment Rollback** —
    - **Self-hosted:** check out the previous commit, rebuild, restart the service. Verify with the smoke test.
    - **Hosted (Vercel):** use the Vercel dashboard to promote a previous deployment to production. The migration step is NOT rolled back (the schema change is forward-only). If a migration must be rolled back, follow the PostgreSQL migration rollback procedure.
  8. **Monitoring and Alerting** —
    - **External monitoring:** Pingdom, Datadog, or UptimeRobot calls `/api/health` every 60 seconds. Alert on 503 response.
    - **Log monitoring:** errors in the application logs trigger alerts. Use a log aggregation service (e.g., Papertrail) to set up alerts.
    - **Performance monitoring:** track the response time of the reports dashboard, the kiosk issuance endpoint, and the SSE connection count. Alert on degradation.
  9. **Disaster Recovery** —
    - **RPO (Recovery Point Objective):** 24 hours (the daily backup).
    - **RTO (Recovery Time Objective):** 4 hours (the time to restore from backup and verify).
    - **Procedure:** restore from the most recent backup, apply any post-backup migrations manually, verify with the verification script, deploy, run the smoke test.

**Verification:** The `docs/operations-runbook.md` file exists with all 9 sections.

## 8. Common Pitfalls

- **Committing secrets to the repository.** The `.env` file is gitignored. The `.env.example` file uses placeholders. The `service-account.json` is gitignored. Verify with `git status` and `git log --all` before each commit.
- **Forgetting the `NEXTAUTH_URL` HTTPS requirement.** In production, `NEXTAUTH_URL` must be HTTPS. NextAuth requires this for secure cookies. The env validator throws on HTTP in production.
- **Not setting the `NEXTAUTH_SECRET` to a strong value.** The default development secret must NOT be used in production. The validator throws on the default value. Generate with `openssl rand -base64 32`.
- **Forgetting `proxy_buffering off` in the nginx configuration.** The SSE stream requires `proxy_buffering off` in nginx. Without it, nginx buffers the SSE events and the client doesn't receive them in real-time. The display board will appear to "freeze" after the first event.
- **Forgetting `proxy_read_timeout 86400` in the nginx configuration.** The SSE connection is long-lived. The default nginx timeout is 60 seconds, which kills the SSE connection after 60 seconds of inactivity (despite the heartbeat). Set to 86400 (24 hours) for safety.
- **Not running the Prisma migration during the Vercel build.** Vercel does NOT automatically run `prisma migrate deploy`. The migration must be triggered via a build hook or a custom build step.
- **Not setting the `application_name` in the PostgreSQL connection string.** The PostgreSQL logs show the application name for each connection. Without it, debugging connection issues is harder.
- **Not testing the CI pipeline locally.** Push to a branch and let the CI run. If it fails, fix the issue. Don't push to `main` and hope.
- **Not setting the `production` environment protection rule.** Without the protection rule, any push to `main` deploys to production. Set the required reviewers.
- **Forgetting to update the `FCM_SERVICE_ACCOUNT_JSON` in the secrets manager.** The FCM credentials are per-Firebase-project. If the project changes, the credentials must be updated.
- **Not documenting the rollback procedure.** A deployment without a documented rollback is a deployment the team is afraid to run. Always document the rollback.
- **Forgetting to test the smoke test checklist against a staging environment.** The smoke test must be executed against staging first to verify it works. Then it can be used against production.
- **Not running the env check script before the deploy.** The `env:check` script catches missing or malformed env vars before the application boots. Always run it.
- **Not setting the `RATE_LIMIT_ENABLED` flag correctly.** In production, the rate limiter is enabled by default. In development, it's disabled by default (so developers aren't rate-limited during local testing). The env validator must respect this.
