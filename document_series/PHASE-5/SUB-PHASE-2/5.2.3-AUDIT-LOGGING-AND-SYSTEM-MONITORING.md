## 1. Document Purpose

This document delivers the **operational visibility and resilience layer** of the Smart Queue Management System. It finalizes the audit log policy (consolidating every write point across all 5 prior sub-phases into a single reference), builds the audit log viewer at `/audit-log` (so administrators can see every administrative action), implements the `/api/health` endpoint (so external monitoring tools can verify the system is operational), and adds application error boundaries (so client-side crashes don't blank the entire UI).

This document is the **third and final document in Sub-Phase 5.2**. It is the **operational layer**: while 5.2.1 protects the system from abuse and 5.2.2 makes it fast, 5.2.3 ensures operators have visibility and resilience. Without the audit log viewer, the `AuditLog` table is opaque. Without the health check endpoint, monitoring tools have nothing to poll. Without error boundaries, a single client-side crash takes down the entire page.

The strategic value of this document is that **it makes the system operable**. Production deployments require:

- Visibility into administrative actions (who changed what when).
- Health check endpoints for monitoring tools (Kubernetes liveness/readiness probes, Pingdom, Datadog, etc.).
- Resilience to client-side errors (graceful degradation, not blank pages).

When this document is complete:

- The audit log viewer at `/audit-log` shows every administrative action (user changes, service changes, counter changes, assignment changes, display board changes, counter status changes, system setting changes, report exports, daily resets) filterable by user, action, entity, and date range.
- The `GET /api/audit-log` endpoint returns paginated, filtered entries for programmatic access.
- The `/api/health` endpoint returns 200 with `{ status, timestamp, database, version, uptimeSeconds }` for monitoring tools.
- When the database is unreachable, `/api/health` returns 503 with `status: 'degraded'`.
- A client-side component throwing an unhandled error shows the error boundary fallback UI instead of blanking the page.
- The audit log policy is documented in `docs/audit-log-policy.md` as the single source of truth.

This document does **not** implement rate limiting (owned by 5.2.1), caching (owned by 5.2.2), or any new business logic. It is purely operational infrastructure.

### Step 1: Create `docs/audit-log-policy.md` ‚Äî Comprehensive Policy Document

**Goal:** Document every audit log write point in a single reference document.

**Files affected:** `docs/audit-log-policy.md` (new file).

**Specific instructions:**

Create the file with the following sections:

- **Section 1 ‚Äî Purpose.** Why the policy exists. Single source of truth for what goes in `AuditLog`.
- **Section 2 ‚Äî Policy.** The high-level rule: `AuditLog` captures LOW-VOLUME, ADMINISTRATIVE actions. `AuditLog` does NOT capture HIGH-VOLUME actions (those use dedicated tables like `TicketEvent`, `Notification`, `BroadcastMessage`, `CounterStatusEvent`).
- **Section 3 ‚Äî Audit Log Write Points.** A table of every `AuditAction` value with:
  - The action name (e.g., `USER_CREATED`).
  - The source phase (e.g., `1.3.3`).
  - The trigger (e.g., "admin creates a new user").
  - The metadata shape (e.g., `{ email, name, roles }`).
  - The actor (usually the calling user, sometimes the system).
- **Section 4 ‚Äî Actions NOT Logged.** A list of actions that use dedicated tables instead:
  - Ticket lifecycle events (`TicketEvent`).
  - Notification dispatch (`Notification`).
  - Officer replies (`NotificationReply`).
  - Broadcast messages (`BroadcastMessage`).
  - Counter status changes (`CounterStatusEvent`).
  - Login attempts (NextAuth session).
  - Token refreshes (NextAuth session).
- **Section 5 ‚Äî Best-Effort Discipline.** All audit log writes are AFTER the transaction commits, wrapped in try/catch. Failures are logged via `console.error` but do NOT fail the action.
- **Section 6 ‚Äî Adding New Write Points.** The procedure for evaluating a new administrative action:
  1. Is it an administrative action (changes configuration, not day-to-day operation)? If no, it does NOT go in `AuditLog`.
  2. Is it high-volume (could be hundreds per day)? If yes, it does NOT go in `AuditLog` ‚Äî use a dedicated table.
  3. If it passes both checks, add it to `AuditAction` union and write the audit log entry in the endpoint.
- **Section 7 ‚Äî Read Access.** The `system:audit` permission grants access to the audit log viewer and the `/api/audit-log` endpoint. Default role assignment: `SUPER_ADMIN` only.

The document should be comprehensive but concise ‚Äî about 100-150 lines.

### Step 4: Create `src/lib/audit-log-queries.ts` ‚Äî Query Helper

**Goal:** Provide the query function for the audit log viewer.

**Files affected:** `src/lib/audit-log-queries.ts` (new file).

**Specific instructions:**

Create the file with:

- **`async function queryAuditLogs(filters: AuditLogFilters, pagination: Pagination): Promise<AuditLogPageResult>`**:
  - Builds the Prisma `where` clause from the filters:
    - `userId?: { userId: filters.userId }`.
    - `action?: { action: filters.action }`.
    - `entity?: { entity: filters.entity }`.
    - `entityId?: { entityId: filters.entityId }`.
    - `startDate?: { createdAt: { gte: filters.startDate } }`.
    - `endDate?: { createdAt: { lte: filters.endDate } }`.
    - Combines with AND logic.
  - Calls `db.auditLog.findMany({ where, orderBy: { createdAt: 'desc' }, skip: (page - 1) * pageSize, take: pageSize, include: { user: { select: { id: true, name: true, email: true } } } })`.
  - Maps the results to `AuditLogEntry` format (denormalising the user fields).
  - Calls `db.auditLog.count({ where })` to get the total.
  - Returns `{ entries, total, page, pageSize, totalPages: Math.ceil(total / pageSize) }`.
- **`async function getAuditLogEntry(entryId: string): Promise<AuditLogEntry | null>`**:
  - Fetches a single entry by ID with the user joined.
  - Returns `null` if not found.

### Step 5: Create `src/app/api/audit-log/route.ts` ‚Äî GET Endpoint

**Goal:** Expose the audit log query via an API endpoint.

**Files affected:** `src/app/api/audit-log/route.ts` (new file).

**Specific instructions:**

Create the file with:

- **GET handler:**
  - Guarded with `withPermission(handler, PERMISSION_SYSTEM_AUDIT)`.
  - Reads query params from the request URL.
  - Validates via `auditLogQuerySchema.safeParse()`. If validation fails, returns 422 with `VALIDATION_ERROR`.
  - Calls `queryAuditLogs(filters, pagination)`.
  - Writes an `AUDIT_LOG_VIEWED` audit log entry AFTER the query succeeds (best-effort). Metadata: `{ filters, resultCount, durationMs }`.
  - Returns `{ success: true, data: entries, meta: { total, page, pageSize, totalPages } }`.
  - `runtime = 'nodejs'`, `dynamic = 'force-dynamic'`.

**Important note:** The `AUDIT_LOG_VIEWED` audit write is BEST-EFFORT. If the audit write fails, the response still succeeds. This prevents an infinite loop (viewing the audit log generates audit entries, which are viewed, which generate more audit entries...).

### Step 9: Create `src/lib/health-check.ts` ‚Äî Health Check Module

**Goal:** Implement the health check logic.

**Specific instructions:**

  - **Does NOT throw** ‚Äî returns the status as a string.
- **`async function getHealthStatus(): Promise<HealthStatus>`**:
  - Calls `checkDatabaseConnection()`.
  - Constructs the `HealthStatus` object.
  - Returns it.

## 8. Common Pitfalls

- **Creating an infinite loop with `AUDIT_LOG_VIEWED` writes.** If the audit log of audit log views is itself viewed, infinite audit entries are created. The 5.2.3 implementation writes `AUDIT_LOG_VIEWED` from the API endpoint (server-side), but the page does NOT generate `AUDIT_LOG_VIEWED` on every render ‚Äî only when the API endpoint is called explicitly.
- **Using `useEffect` for the SSE subscription in the error boundary context.** Error boundaries reset the React tree. Subscriptions must be re-established after an error. Test the recovery path.
- **Logging the full error stack trace in production.** Production users should see a friendly message, not a stack trace. Only show the trace in development.
- **Using `window.location.reload()` in SSR.** The reload button must only call this on the client. Use a client component for the button.
- **Setting the health check endpoint's rate limit too low.** 60 req/min is appropriate for external monitoring. Anything tighter causes false alarms.
- **Not handling the case where the database is slow (but not disconnected).** The 1-second threshold considers slow queries as 'degraded'. Adjust based on production performance.
- **Forgetting the per-route `error.tsx` files.** Next.js requires them for clean error handling at the route level. Without them, errors may bubble to the root layout's boundary, which is too coarse.
- **Not testing the error boundaries.** It is easy to forget. Add a temporary `Bomb` component to verify the fallback works.
- **Including the error message in the audit log viewer without sanitization.** Error messages may contain sensitive data. Consider truncating or sanitizing.

- **Per-route `error.tsx` not triggering?** Check the file is at the correct path and exports a default `Error` component.
- **`AUDIT_LOG_VIEWED` not being written?** Check the audit log helper is called after the query succeeds. Check the `AuditAction` union includes the new value.

### 9.4 Testing Strategy

- **Audit log viewer:** seed the database with a few `AuditLog` entries (using the existing mutation endpoints, which write audit entries automatically). Navigate to `/audit-log`. Verify the entries are shown.
- **Health check:** use `curl http://localhost:3000/api/health` to verify the response. Stop the database to verify the 503 response.
- **Error boundaries:** add a temporary `Bomb` component (e.g., `function Bomb() { throw new Error('Test') }`) to a surface. Load the surface. Verify the fallback. Remove the `Bomb`.
- **Per-route error.tsx:** same approach, but the test focuses on the `error.tsx` file specifically.

### 9.5 What NOT to Do

- Do not modify existing audit log write points. They are already in place from prior phases.
- Do not require authentication for `/api/health`. External monitoring tools need to call it.
- Do not use `useEffect` to set up the SSE subscription inside an error boundary ‚Äî it will be torn down on error.
- Do not log the error stack trace in production. Only in development.
- Do not create an infinite loop with `AUDIT_LOG_VIEWED` writes. The API endpoint writes the audit entry once per call, but the page render does not trigger a write.
