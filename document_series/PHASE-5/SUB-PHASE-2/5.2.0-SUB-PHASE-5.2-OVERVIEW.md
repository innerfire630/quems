## 1. Sub-Phase Purpose

Sub-Phase 5.2 takes the **working and observable** system from Sub-Phase 5.1 and transforms it into a **production-grade** system. Where Phases 1–4 built features and 5.1 made them observable, Sub-Phase 5.2 makes them secure, performant, and resilient under production load. This sub-phase is the **operational and security infrastructure** layer of the system.

The strategic goal is to **production-harden every endpoint, page, and background process** that exists after Phase 5.1. By the end of this sub-phase, the system must be ready for real customer traffic, real-world abuse attempts, and the operational visibility that production deployments require. The four pillars of hardening in this sub-phase are:

1. **API security** (5.2.1) — rate limiting per route group, CORS configuration, request size limits, security headers on every response. The system must reject abusive traffic at the edge (rate limiting) and be resilient against common web vulnerabilities (security headers, CORS).
2. **Performance optimization** (5.2.2) — `unstable_cache` for slowly-changing data, database query optimization (eliminating N+1 patterns), Prisma query logging in development. The system must remain responsive as data volume grows.
3. **Audit logging policy finalization** (5.2.3) — comprehensive enforcement of the audit log policy across every administrative action, plus a viewer UI that lets administrators see what changed and when.
4. **System monitoring** (5.2.3) — a `/api/health` endpoint for external monitoring tools, application error boundaries that prevent client-side crashes from blanking the entire UI, and clear visibility into the system's operational status.

The work in this sub-phase has three layers, matching the three task plan documents:

- **5.2.1 — API Security & Rate Limiting.** Establishes the rate limiting infrastructure (`lib/rate-limit.ts` module, in-memory `RateLimitBucket` store, per-route-group limits), wires the rate limiter into the middleware (so it runs at the edge before the route handler), configures CORS via `next.config.ts` and `ALLOWED_MOBILE_ORIGINS`, sets request size limits, and applies security headers to every response via `next.config.ts`. This document consumes the existing route protection stack (auth + RBAC) and adds rate limiting as the outermost layer.
- **5.2.2 — Performance Optimization & Caching.** Introduces `unstable_cache` for slowly-changing data (active services, active counters, display board configurations, system settings), establishes a `revalidateTag()` discipline so the cache is invalidated precisely when underlying data changes, reviews the SSE connection pooling for any leaks or efficiency issues, optimizes report queries for N+1 patterns, and enables Prisma query logging in development. This document makes the system fast under production load.
- **5.2.3 — Audit Logging & System Monitoring.** Finalizes the audit log policy (which actions write to `AuditLog` and which use dedicated tables like `TicketEvent`, `Notification`, etc.), builds the audit log viewer at `/audit-log` (server component with `withPermission(PERMISSION_SYSTEM_AUDIT)`), implements the `GET /api/health` endpoint (returns `{ status, timestamp, database, version, uptimeSeconds }`), and wraps the application in error boundaries (top-level + per-route) using a React error boundary. This document gives operators the visibility and resilience they need.

When this sub-phase is complete:

- Sending 11 auth requests in 1 minute from the same IP results in a `RATE_LIMITED` (429) response on the 11th request, with a `Retry-After` header.
- Every response from the application carries the four security headers: `X-Frame-Options: SAMEORIGIN`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy: camera=(), microphone=(), geolocation=()`.
- CORS allows the web app's same-origin requests (no CORS in production) and the future Android app's configured origin (via `ALLOWED_MOBILE_ORIGINS`).
- The reports dashboard loads in under 2 seconds with 10,000 tickets in the database.
- The audit log viewer at `/audit-log` shows all administrative actions (user changes, service changes, counter changes, assignment changes, display board changes, counter status changes, system setting changes, report exports, daily resets) filterable by user, action, entity, and date range.
- `/api/health` returns 200 with a JSON body indicating the operational status (database connectivity, version, uptime).
- A client-side component throwing an unhandled error shows the error boundary fallback instead of blanking the page.
- The audit log policy is single-sourced and enforced.

Three problems are solved in this sub-phase, in order:

1. **The API is hardened against abuse and common vulnerabilities.** Rate limiting prevents brute-force login attacks and denial-of-service attempts. Security headers prevent clickjacking, MIME sniffing, and excessive referrer leakage. CORS allows the mobile app to consume the API while blocking unauthorized origins. (Document 5.2.1.)
2. **The system is performant under production load.** Slowly-changing data is cached, queries are optimized, and developers can see slow queries via Prisma's query log. (Document 5.2.2.)
3. **Operators have visibility into administrative actions and the system's health.** The audit log viewer surfaces every administrative change. The health check endpoint enables external monitoring. Error boundaries prevent client-side crashes from taking down the entire UI. (Document 5.2.3.)

When this sub-phase is complete, **Phase 5.2 is finished**. The system is production-grade. Sub-Phase 5.3 (deployment & migration) can begin — it migrates the database to PostgreSQL, configures the CI/CD pipeline, and provides the final integration test checklist.

## 1.1 Why This Sub-Phase Comes After 5.1

Sub-Phase 5.2 comes after 5.1 (analytics) because:

- **The reports dashboard (5.1.2) needs hardening.** Rate limiting (5.2.1) protects the report endpoints from abuse. Security headers (5.2.1) protect the report page. Caching (5.2.2) speeds up the report queries. Building the reports first and hardening them afterwards is cleaner than hardening an empty page and then adding features.
- **The CSV export endpoint (5.1.3) needs rate limiting.** Building the export before the rate limiter means the export is temporarily unprotected. 5.2.1 adds a stricter rate limit (10 req/min per user) because CSV generation is more expensive than JSON responses.
- **The audit log writes from 5.1 use the existing helper.** 5.2.3's audit log viewer surfaces the `REPORT_GENERATED` and `REPORT_EXPORTED` entries.
- **The data collection in 5.1.1 is queried by the cache layer (5.2.2).** The cache invalidation tags (`['services']`, `['counters']`, etc.) are defined here, and the report endpoint queries through the cached data when possible.

## 1.2 Why This Sub-Phase Comes Before 5.3

Sub-Phase 5.2 comes before 5.3 (deployment) because:

- **The CI/CD pipeline (5.3.2) needs to run the lint, type-check, and build steps.** If the build is broken due to a rate limiting or caching bug, the pipeline fails. Building 5.2 before 5.3 ensures the codebase is in a known-good state before deployment automation is documented.
- **The PostgreSQL migration (5.3.1) needs to apply cleanly.** If the migration encounters unexpected query patterns from unoptimized code, the migration may fail or run slowly. Optimizing queries in 5.2.2 first ensures the migration is fast.
- **The smoke test checklist (5.3.3) needs to verify hardening.** The checklist includes rate limiting verification (`send 11 auth requests in 1 minute`), security header verification (`every response carries the headers`), audit log verification (`viewer renders entries`), and health check verification (`/api/health` returns 200). These checks depend on 5.2 being complete.

## 1.3 What This Sub-Phase Is NOT

- It is **not** new user-facing features. Phase 5.2 is hardening, not feature work. Any new feature is post-series.
- It is **not** Redis-backed rate limiting. The in-memory rate limit store is sufficient for the initial deployment. Redis is the documented upgrade path for horizontal scaling.
- It is **not** automated scheduled email reports. That's 5.1.3 (interface only) and future implementation.
- It is **not** real-time analytics. The dashboard is on-demand refresh.
- It is **not** the native Android app. The API is ready; the app is post-series.
- It is **not** the display board or the officer dashboard. These exist already (Phases 3 and 4). 5.2 makes them faster and more resilient, but does not change their behavior.
- It is **not** internationalization of the admin UI.
- It is **not** multi-server SSE with Redis Pub/Sub. The in-process singleton from 3.1 is the deployed architecture.
- It is **not** security penetration testing. The security headers and rate limiting are baseline defenses. Penetration testing is an operational concern post-deployment.
- It is **not** a full test automation suite. Manual smoke tests and integration walkthroughs are the testing strategy. Automated testing is a future enhancement.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to distribute the complexity across the three task documents. The split is:

- **5.2.1** owns the **API security layer**. It finalizes the audit log policy, builds the audit log viewer, implements the `/api/health` endpoint, and adds the application error boundaries. It does **not** know about rate limiting (5.2.1) or caching (5.2.2). It gives operators the visibility and resilience they need.

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 5.2.1 — API Security & Rate Limiting

**Owns:** The `lib/rate-limit.ts` module (rate limiting infrastructure), the in-memory `RateLimitBucket` store, the per-route-group `RATE_LIMITS` configuration, the rate limit key generation helpers, the `withRateLimit` higher-order function, the middleware integration, the CORS configuration via `next.config.ts`, the request size limits via `next.config.ts`, and the security headers via `next.config.ts`.

**Does NOT own:** The caching layer (owned by 5.2.2), the audit log viewer (owned by 5.2.3), the health check endpoint (owned by 5.2.3), the error boundaries (owned by 5.2.3), the rate limit UI or status (not in scope).

**Key outputs:**

- **`src/lib/rate-limit.ts`** — the rate limiting module:
  - **`interface RateLimitBucket`** — `{ count: number, windowStart: number (timestamp ms), limit: number, windowMs: number }`. The bucket tracks the request count within the current window.
  - **`interface RateLimitConfig`** — `{ limit: number, windowMs: number, keyStrategy: 'ip' | 'user' | 'ip-or-user' }`. The configuration for a route group's limit.
  - **`class RateLimitExceededError extends Error`** — thrown when a limit is exceeded. Has `retryAfterMs: number` and `code: 'RATE_LIMITED'` properties.
  - **`async function checkRateLimit(key: string, config: RateLimitConfig): Promise<{ allowed: boolean, remaining: number, resetMs: number }>`** — the core rate limit check. Looks up the bucket by key. If the bucket is missing OR the window has expired, creates a new bucket. If the count is within the limit, increments and returns `{ allowed: true, remaining, resetMs }`. If the count exceeds the limit, returns `{ allowed: false, remaining: 0, resetMs }`.
  - **`async function checkIpRateLimit(ip: string, routeGroup: RouteGroup): Promise<RateLimitResult>`** — convenience wrapper for IP-based limits.
  - **`async function checkUserRateLimit(userId: string, routeGroup: RouteGroup): Promise<RateLimitResult>`** — convenience wrapper for user-based limits.
  - **`function getRequestIp(request: Request): string`** — extracts the client IP from the request, considering X-Forwarded-For headers. Falls back to a default value if the IP cannot be determined.
  - **`function createRateLimitResponse(retryAfterMs: number): NextResponse`** — creates a 429 response with `Retry-After` header (in seconds, rounded up) and the standard JSON envelope.
  - **Module-scope state:** the `rateLimitStore` Map. Uses the `globalThis.__rateLimitStore` pattern (like the SSE manager from 3.1) to survive Next.js dev hot-reload. Periodically cleaned up (expired buckets removed).
  - **Bucket cleanup:** a function `cleanupExpiredBuckets(now)` that removes buckets whose window has expired. Called periodically (every 5 minutes via `setInterval`).
  - **Constants:** `RATE_LIMITS: Record<RouteGroup, RateLimitConfig>` — the per-route-group configuration.
- **`src/middleware.ts`** — the middleware is EXTENDED to invoke the rate limiter. The middleware:
  - Extracts the request path.
  - Determines the route group based on the path prefix.
  - Extracts the IP (or user ID if authenticated via the JWT cookie — the middleware has access to the JWT via `getToken()`).
  - Calls `checkRateLimit(key, config)` with the matching `RATE_LIMITS` entry.
  - If the limit is exceeded, returns a 429 response immediately (the route handler is not invoked).
  - If under the limit, passes the request through to the route handler.
  - The middleware's existing route protection rules (auth, RBAC) are preserved.
- **`src/lib/rate-limit-keys.ts`** — key generation helpers:
  - `function generateIpKey(ip: string, routeGroup: RouteGroup): string` — returns `ratelimit:ip:<ip>:<routeGroup>`.
  - `function generateUserKey(userId: string, routeGroup: RouteGroup): string` — returns `ratelimit:user:<userId>:<routeGroup>`.
  - `function generateSseKey(ip: string): string` — returns `ratelimit:sse:<ip>` (for concurrent connection counting).
- **`src/lib/guards.ts`** — EXTENDED to add `withRateLimit(handler, routeGroup)` helper. The helper wraps a route handler with rate limiting at the handler level (in addition to or instead of the middleware level). Useful for routes that need handler-level access to the request body for fingerprinting.
- **`src/app/api/auth/[...nextauth]/route.ts`** and other auth endpoints — wrapped with rate limiting via the middleware. The middleware enforces the 10 req/min per IP limit.
- **`src/app/api/tickets/issue/route.ts`** — wrapped with rate limiting via the middleware. The middleware enforces the 30 req/min per IP limit.
- **`next.config.ts`** — UPDATED to:
  - Set the `headers()` function to apply the four security headers to ALL responses.
  - Configure CORS headers on API responses (`/api/*`).
  - Set the `bodySizeLimit` to 10 MB (or the appropriate Next.js configuration for limiting body size).
  - The config is type-safe and documented.
- **`src/lib/security-headers.ts`** — the security headers constants module:
  - `const SECURITY_HEADERS: Record<string, string>` — the four headers from master plan Section 15.5.
  - `function applySecurityHeaders(response: NextResponse): NextResponse` — mutates a `NextResponse` to add the headers (utility for endpoints that construct responses directly).
- **`src/lib/cors.ts`** — the CORS configuration module:
  - `function getCorsHeaders(origin: string | null): Record<string, string>` — returns the CORS headers based on the request origin and `ALLOWED_MOBILE_ORIGINS`.
  - `function isOriginAllowed(origin: string | null): boolean` — checks if the origin is allowed (same-origin or in `ALLOWED_MOBILE_ORIGINS`).
- **Route group classification** — the middleware uses a `ROUTE_GROUP_PATTERNS` constant to classify paths:
  - `/api/auth/*` → `'AUTH'` group (10 req/min per IP).
  - `/api/tickets/issue` → `'TICKET_ISSUANCE'` group (30 req/min per IP).
  - `/api/tickets/[ticketId]/call|recall|no-show`, `/api/tickets/call-next` → `'OFFICER_ACTIONS'` group (60 req/min per user).
  - `/api/sse/*` → `'SSE'` group (10 concurrent per IP).
  - All other `/api/*` → `'GENERAL'` group (200 req/min per user or IP).
  - Non-API routes → no rate limiting (or `'GENERAL'` group — designer's choice; the document specifies).
- **Test endpoint** — `src/app/api/_test/rate-limit/route.ts` — a development-only test endpoint that returns rate limit headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`) for verification. Disabled in production (NODE_ENV check).

**Master Plan sections implemented:** Section 15 (entire Security Architecture — rate limits, CORS, security headers).

### 4.2 Document 5.2.2 — Performance Optimization & Caching

**Owns:** The `unstable_cache` strategy for slowly-changing data, the cache tag system, the revalidation discipline, the database query optimization for report queries, the Prisma query logging, and the database indexes.

**Does NOT own:** The rate limiting (owned by 5.2.1), the audit log viewer (owned by 5.2.3), the health check endpoint (owned by 5.2.3), the error boundaries (owned by 5.2.3), the SSE connection pooling (the manager is owned by 3.1; 5.2.2 only reviews for issues).

**Key outputs:**

- **`src/lib/cache-tags.ts`** — the cache tag constants:
  - `export const SERVICES_TAG = 'services'` — invalidates when any service changes.
  - `export const COUNTERS_TAG = 'counters'` — invalidates when any counter or counter status changes.
  - `export const DISPLAY_BOARDS_TAG = 'display-boards'` — invalidates when any display board changes.
  - `export const SETTINGS_TAG = 'settings'` — invalidates when any system setting changes.
  - `export const USERS_TAG = 'users'` — invalidates when any user changes.
  - `export const ROLE_PERMISSIONS_TAG = 'role-permissions'` — invalidates when role permissions change.
  - `export const OFFICERS_TAG = 'officers'` — invalidates when counter officer assignments change.

### 6.4 Rate Limiting Discipline

- Rate limit keys follow the `ratelimit:<strategy>:<identifier>:<scope>` pattern.
- IP-based limits are used for unauthenticated endpoints (auth, ticket issuance).
- User-based limits are used for authenticated endpoints (officer actions, general API).
- The middleware does rate limiting BEFORE auth (for IP-based endpoints) and AFTER auth (for user-based endpoints). The middleware uses `getToken()` to extract the user ID for user-based limits; if no token, falls back to IP.
- Rate limit headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`) are set on all responses (success and 429).
- The `Retry-After` header on 429 responses is in seconds, rounded up.
- The rate limit window is FIXED (not sliding). This is simpler and sufficient for the use case. Sliding window is a future enhancement.
- Rate limit buckets are cleaned up periodically (every 5 minutes) to prevent memory growth from one-off requests.

### 6.5 Cache Invalidation Discipline

- Every cache entry has a tag (or set of tags).
- Every mutation that changes cached data calls `revalidateTag()` with the matching tag(s).
- The cache tags are constants defined in `@/lib/cache-tags`.
- If a mutation changes multiple data types, ALL relevant tags are invalidated.
- The `revalidateTag()` call is OUTSIDE the database transaction (post-commit). A failure in `revalidateTag()` does NOT roll back the database change — it just means the next request will see stale data for up to the cache TTL (max 1 hour).
- The cache TTL is 1 hour (3600 seconds) as a safety net. Tags are the primary invalidation mechanism; TTL is the fallback.

### 6.7 Security Header Discipline

### 6.8 CORS Discipline

- The web app is same-origin in production (no CORS headers needed for same-origin requests).
- The Android app is allowed via `ALLOWED_MOBILE_ORIGINS` (comma-separated list of origins).
- CORS headers are set on `/api/*` responses only (not on page responses).
- The preflight OPTIONS request is handled by Next.js (via the `headers()` configuration in `next.config.ts`).
- Wildcard origins (`*`) are NOT used — the `ALLOWED_MOBILE_ORIGINS` must be a specific list.

### 6.9 Error Boundary Discipline

- A top-level error boundary wraps the entire app via the root layout.
- Per-route error boundaries catch errors specific to their surface.
- Error boundaries log via `console.error` (not via `writeAuditLog` — client-side errors are not user actions).
- Error boundaries render a fallback UI that includes a refresh button (for attended surfaces) or an auto-refresh timer (for unattended surfaces like the display board).
- The fallback UI hides the error message and stack trace in production (only shows "Something went wrong" + refresh button).
- In development, the fallback UI shows the error message and stack trace for debugging.

## 7. Sub-Phase Exit Criteria

Sub-Phase 5.2 is complete when **all** of the following are true:

### 7.1 API Security (from 5.2.1)

- [ ] Sending 11 auth requests in 1 minute from the same IP results in 429 on the 11th request.
- [ ] The 429 response includes a `Retry-After` header.
- [ ] The 429 response body is `{ success: false, error: { code: 'RATE_LIMITED', message: '...' } }`.
- [ ] Every response from the application includes the security headers: `X-Frame-Options: SAMEORIGIN`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy: camera=(), microphone=(), geolocation=()`.
- [ ] A request from an `ALLOWED_MOBILE_ORIGINS` origin succeeds (CORS headers are set).
- [ ] A request from a non-allowed origin does NOT receive CORS headers.
- [ ] A request with a 20MB body is rejected with 413 (Payload Too Large).
- [ ] A request from an IP that has hit the rate limit receives 429 even for legitimate requests.
- [ ] The rate limit headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`) are set on all responses.

### 7.2 Performance (from 5.2.2)

- [ ] The reports dashboard loads in under 2 seconds with 10,000 tickets in the database (via the benchmark script).
- [ ] Mutating a service (create/update/delete) invalidates the `services` cache within 1 second (verified by querying the service list immediately after).
- [ ] Mutating a counter invalidates the `counters` cache within 1 second.
- [ ] In development, Prisma queries are logged with timing.
- [ ] Slow queries (>100ms) are highlighted with a warning.
- [ ] The new database indexes are migrated (via `prisma migrate dev`).
- [ ] The cache tags are defined as constants in `src/lib/cache-tags.ts`.
- [ ] The cached data functions are exported from `src/lib/cached-data.ts`.

### 7.3 Audit Logging & Monitoring (from 5.2.3)

- [ ] The audit log viewer at `/audit-log` shows all administrative actions with filters by user, action, entity, and date range.
- [ ] The `GET /api/audit-log` endpoint returns paginated, filtered entries.

- [ ] The `/api/health` endpoint returns 200 with `{ status: 'ok', timestamp, database: 'connected', version, uptimeSeconds }`.
- [ ] When the database is unreachable, `/api/health` returns 503 with `{ status: 'degraded', database: 'disconnected' }`.

## 8. AI Agent Implementation Guidance

This section provides specific instructions to an AI agent implementing any of the 5.2.x documents. The goal is to remove ambiguity and reduce the need for guesswork.

### 8.1 General Guidance

- **Read the parent documents first.** Before starting 5.2.x, read [05-PHASE-5-OVERVIEW.md](../05-PHASE-5-OVERVIEW.md) and this document. Understand how 5.2 fits into Phase 5 and the overall DDD series.
- **Read the master plan sections.** Each 5.2.x document references master plan sections (15, 3.2, 8.2, 9.3, etc.). Read those sections for the precise specifications.
- **Follow the existing patterns.** The codebase has established patterns from Phases 1-4 and 5.1. Follow them. Do not introduce new patterns unless the document explicitly says to.
- **Verify before committing.** Run `yarn lint`, `yarn type-check`, and `yarn build` after each step. Fix any errors before proceeding to the next step.
- **Test incrementally.** After each step, test the new functionality manually. Verify it works end-to-end before proceeding.
- **Use `withRateLimit` for handler-level rate limits.** Do not bypass the rate limiter.
- **Use `revalidateTag` for cache invalidation.** Do not invalidate cache by manipulating the cache directly.
- **Use `writeAuditLog` for new audit write points.** Follow the Phase 1.3.3 discipline (best-effort, post-commit, metadata-rich).
- **Use the established error envelope.** `{ success: false, error: { code, message, details? } }` for errors. `{ success: true, data, meta? }` for successes.

### 8.2 Common Pitfalls to Avoid

- **Do not put rate limiting logic in the route handler.** Rate limiting happens at the middleware level (or via `withRateLimit`). Putting it in the handler means the limit is not enforced for unauthorized requests (which are rejected by `auth()` first).
- **Do not use a sliding window for rate limiting in 5.2.1.** Use a fixed window. Sliding window is a future enhancement.
- **Do not use Redis for rate limiting in 5.2.1.** The in-memory store is sufficient for the initial deployment. Redis is future.
- **Do not cache data that changes frequently.** Only cache slowly-changing data (active services, counters, display boards, settings). Tickets and notifications change constantly — do NOT cache them.
- **Do not `revalidateTag` inside a transaction.** Cache invalidation is post-commit.
- **Do not let cache invalidation failures block the response.** `revalidateTag` is best-effort. Wrap in try/catch.
- **Do not use `any` types.** Use proper TypeScript types.
- **Do not skip the migration when adding new indexes.** The Prisma migration must be applied.
- **Do not write audit log entries from error boundaries.** Client-side errors are not user actions.
- **Do not rate-limit the health check endpoint heavily.** 60 req/min is sufficient for external monitoring. Anything tighter causes false alarms.
- **Do not forget to test the rate limiter.** Send 11 auth requests in 1 minute and verify the 11th is rejected. If you cannot easily verify, the rate limiter is not correctly wired.
- **Do not use `globalThis.__rateLimitStore` without checking if it already exists.** The pattern is `globalThis.__rateLimitStore ??= new Map()`. Without the `??=`, the store is recreated on every module reload.
- **Do not log full Prisma query parameters.** They may contain sensitive data. Log only the query text and timing.
