## 1. Document Purpose

This document delivers the **API security and rate limiting layer** of the Smart Queue Management System. It establishes the infrastructure that protects every API endpoint from abuse (rate limiting per route group), enables secure cross-origin requests from the future Android app (CORS), limits request body sizes to prevent payload-based attacks, and sets the four security headers on every response per master plan Section 15.5.

This document is the **first of three documents in Sub-Phase 5.2**. It is the **outermost layer of production defense**: rate limiting runs at the middleware (Edge runtime where possible) BEFORE auth, RBAC, or business logic. The other 5.2 documents (5.2.2 caching, 5.2.3 audit log and health) consume the rate limiter when they expose new endpoints.

The strategic value of this document is that **it transforms the system from "functionally correct" to "production-safe"**. Without rate limiting, an attacker can brute-force login attempts or overwhelm the system with legitimate-looking requests. Without security headers, the system is vulnerable to clickjacking and MIME sniffing. Without CORS, the mobile app cannot consume the API. Without request size limits, an attacker can exhaust memory with large payloads.

When this document is complete:

- Sending 11 auth requests in 1 minute from the same IP results in `RATE_LIMITED` (429) on the 11th request, with a `Retry-After` header.
- Every response from the application carries the four security headers from Section 15.5.
- CORS allows requests from origins in the `ALLOWED_MOBILE_ORIGINS` env var and blocks all other cross-origin requests.
- A request body larger than 10 MB is rejected with 413 (Payload Too Large).
- Rate limit headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`) are set on all responses.
- The rate limit infrastructure is reusable: future endpoints can be rate-limited by adding them to the `RATE_LIMITS` configuration.

This document does **not** implement the caching layer (owned by 5.2.2), the audit log viewer (owned by 5.2.3), the health check endpoint (owned by 5.2.3), or the error boundaries (owned by 5.2.3). It only handles API security.

### Step 1: Create `src/lib/rate-limit.ts` — Core Rate Limiting Module

**Goal:** Establish the rate limiting infrastructure — bucket store, check function, IP/user key strategies, cleanup utilities.

**Files affected:** `src/lib/rate-limit.ts` (new file).

**Specific instructions:**

Create the file with the following exports:

- **`interface RateLimitBucket`** — the bucket shape: `{ count: number, windowStart: number (timestamp ms), limit: number, windowMs: number }`. The bucket tracks the request count within the current window.
- **`interface RateLimitConfig`** — the configuration: `{ limit: number, windowMs: number, keyStrategy: 'ip' | 'user' | 'ip-or-user' }`. The `keyStrategy` determines which identifier is used.
- **`interface RateLimitResult`** — the check result: `{ allowed: boolean, remaining: number, resetMs: number, retryAfterMs: number }`. `resetMs` is the timestamp when the window resets. `retryAfterMs` is how long to wait before retrying (0 if allowed).
- **`class RateLimitExceededError extends Error`** — thrown when a limit is exceeded. Properties: `retryAfterMs: number`, `code: 'RATE_LIMITED'`.
- **`enum RouteGroup`** — the route group enum (or string union type): `'AUTH'`, `'TICKET_ISSUANCE'`, `'OFFICER_ACTIONS'`, `'SSE'`, `'GENERAL'`, `'HEALTH'`. The values match the route groups defined in master plan Section 15.3.
- **`const RATE_LIMITS: Record<RouteGroup, RateLimitConfig>`** — the per-route-group configuration:
  - `AUTH`: `{ limit: 10, windowMs: 60_000, keyStrategy: 'ip' }` (10 req/min per IP).
  - `TICKET_ISSUANCE`: `{ limit: 30, windowMs: 60_000, keyStrategy: 'ip' }` (30 req/min per IP).
  - `OFFICER_ACTIONS`: `{ limit: 60, windowMs: 60_000, keyStrategy: 'user' }` (60 req/min per user).
  - `SSE`: `{ limit: 10, windowMs: 0, keyStrategy: 'ip' }` (10 concurrent per IP — `windowMs: 0` indicates concurrent count, not rate).
  - `GENERAL`: `{ limit: 200, windowMs: 60_000, keyStrategy: 'ip-or-user' }` (200 req/min per user or IP).
  - `HEALTH`: `{ limit: 60, windowMs: 60_000, keyStrategy: 'ip' }` (60 req/min per IP for health checks).
- **`async function checkRateLimit(key: string, config: RateLimitConfig): Promise<RateLimitResult>`** — the core check function:
  - Looks up the bucket by `key` in `rateLimitStore`.
  - If the bucket is missing OR the window has expired (`Date.now() - bucket.windowStart > config.windowMs`), creates a new bucket: `{ count: 0, windowStart: Date.now(), limit: config.limit, windowMs: config.windowMs }`.
  - If the bucket count is less than the limit, increments and returns `{ allowed: true, remaining: limit - count - 1, resetMs: windowStart + windowMs, retryAfterMs: 0 }`.
  - If the bucket count has reached the limit, returns `{ allowed: false, remaining: 0, resetMs: windowStart + windowMs, retryAfterMs: windowStart + windowMs - Date.now() }`.
- **`async function checkIpRateLimit(ip: string, routeGroup: RouteGroup): Promise<RateLimitResult>`** — convenience wrapper:
  - Calls `checkRateLimit(generateIpKey(ip, routeGroup), RATE_LIMITS[routeGroup])`.
- **`async function checkUserRateLimit(userId: string, routeGroup: RouteGroup): Promise<RateLimitResult>`** — convenience wrapper for user-based limits.
- **`async function checkIpOrUserRateLimit(request: Request, session: Session | null, routeGroup: RouteGroup): Promise<RateLimitResult>`** — convenience wrapper that prefers the user ID if the session exists, otherwise falls back to the IP. The `request` is needed for the IP extraction.
- **`function getRequestIp(request: Request): string`** — extracts the client IP:
  - Checks the `x-forwarded-for` header (the leftmost IP is the original client).
  - Falls back to `x-real-ip` header.
  - Falls back to a constant `'unknown'` (defensive — should not happen in production behind a proxy).
  - Returns the IP string.
- **`function createRateLimitResponse(result: RateLimitResult, routeGroup: RouteGroup): NextResponse`** — creates a 429 response:
  - Status: 429.
  - Headers: `Retry-After: Math.ceil(result.retryAfterMs / 1000)` (seconds, rounded up), `X-RateLimit-Limit: config.limit`, `X-RateLimit-Remaining: 0`, `X-RateLimit-Reset: Math.ceil(result.resetMs / 1000)`.
  - Body: `{ success: false, error: { code: 'RATE_LIMITED', message: 'Rate limit exceeded. Try again in N seconds.', details: { retryAfterMs, routeGroup } } }`.
- **Module-scope state:** the `rateLimitStore` Map. Uses the `globalThis.__rateLimitStore ??= new Map<string, RateLimitBucket>()` pattern to survive Next.js dev hot-reload.
- **`function cleanupExpiredBuckets(): void`** — removes buckets whose window has expired:
  - Iterates the store. Removes buckets where `Date.now() - bucket.windowStart > bucket.windowMs` AND `bucket.windowMs > 0` (skip concurrent buckets).
  - Called periodically (every 5 minutes via `setInterval`).
  - The interval is started in the `instrumentation.ts` file (alongside the reset scheduler).
- **`function getActiveBucketCount(): number`** — returns the current store size (used for diagnostics).

**Important note:** The store is in-memory. It is per-process. In a multi-server deployment, each server has its own store, so the effective rate limit is `limit * numServers`. For Phase 5.2, single-server deployment is assumed. Redis-backed distributed rate limiting is a future enhancement.

### Step 3: Create `src/lib/cors.ts` — CORS Configuration Helpers

**Goal:** Provide CORS configuration that allows the web app (same-origin) and the future Android app (via `ALLOWED_MOBILE_ORIGINS`).

**Files affected:** `src/lib/cors.ts` (new file).

**Specific instructions:**

Create the file with:

- **`function getAllowedOrigins(): string[]`** — reads the `ALLOWED_MOBILE_ORIGINS` env var (comma-separated) and returns the array of origins. Returns `[]` if the env var is not set.
- **`function isOriginAllowed(origin: string | null): boolean`** — returns `true` if the origin is allowed:
  - If `origin === null` (same-origin request), returns `true`.
  - If `origin` is in `getAllowedOrigins()`, returns `true`.
  - Otherwise, returns `false`.
- **`function getCorsHeaders(requestOrigin: string | null): Record<string, string>`** — returns the CORS headers to set on the response:
  - `Access-Control-Allow-Origin`: `requestOrigin` if `isOriginAllowed(requestOrigin)`, otherwise `'null'` (do NOT use `'*'` for security).
  - `Access-Control-Allow-Methods`: `'GET, POST, PATCH, DELETE, OPTIONS'`.
  - `Access-Control-Allow-Headers`: `'Content-Type, Authorization, X-Requested-With'`.
  - `Access-Control-Allow-Credentials`: `'true'` (so cookies can be sent for web auth).
  - `Access-Control-Max-Age`: `'86400'` (24 hours — preflight cache).
  - If the origin is NOT allowed, returns an empty object (no CORS headers).
- **`function handleCorsPreflight(request: Request): NextResponse | null`** — handles the OPTIONS preflight request:
  - If the request method is OPTIONS, returns a 204 response with the CORS headers.
  - Otherwise, returns `null` (let the request pass through).

### Step 5: Update `next.config.ts` — Apply Security Headers, CORS, and Body Size Limits Globally

**Goal:** Configure Next.js to apply the security headers and CORS to all responses, and limit request body size.

**Files affected:** `next.config.ts` (modified — file was created by Phase 1.1.1).

**Specific instructions:**

Update the file with:

- **`headers()` configuration** — returns an array of header rules. One rule applies to ALL paths (the `headers` field is `[{ key, value }, ...]` applied to the `{ source: '/(.*)', headers: [...] }` source):
  - For each entry in `SECURITY_HEADERS`, add a header entry.
  - The source is `'/(.*)'` (matches all paths).
  - The CORS headers are added with source `'/api/:path*'` (apply only to API routes).
  - The `Access-Control-Allow-Origin` value is dynamic — but Next.js config does not support dynamic headers. Solution: set the static value `'*'` for the preflight response, and let the middleware overwrite the actual `Access-Control-Allow-Origin` based on the request origin (see Step 6 for middleware integration). The `*` is acceptable here because the middleware ensures the actual origin is checked before the response is sent.
- **`experimental.serverActions` configuration** — set `bodySizeLimit: process.env.MAX_REQUEST_BODY_SIZE ?? '10mb'` (defaults to 10 MB).
- **The configuration is type-safe and documented.** Comments explain the purpose of each header.

**Example structure (described in prose, no code):** The `next.config.ts` exports a default object with `async headers()` that returns an array. Each element has `source` (path pattern) and `headers` (array of `{ key, value }`). The security headers are applied globally. The CORS preflight headers are applied to `/api/:path*`.

### Step 6: Update `src/middleware.ts` — Integrate the Rate Limiter and CORS

**Goal:** Make the middleware invoke the rate limiter and handle CORS preflight.

**Files affected:** `src/middleware.ts` (modified — file was created by Phase 1.2.3, extended by 1.3.2 and 4.3.3).

**Specific instructions:**

Update the middleware:

- **Import the rate limit helpers and CORS helpers** from the new modules.
- **Add CORS preflight handling at the top of the middleware:**
  - If the request method is OPTIONS, return a 204 response with the CORS headers.
  - This handles the preflight before any other middleware logic runs.
- **Add rate limiting logic:**
  - Extract the request path.
  - Classify the path into a route group using a `ROUTE_GROUP_PATTERNS` constant:
    - `'/api/auth/.*'` → `'AUTH'`.
    - `'/api/tickets/issue'` → `'TICKET_ISSUANCE'`.
    - `'/api/tickets/.*/(call|recall|no-show)'`, `'/api/tickets/call-next'` → `'OFFICER_ACTIONS'`.
    - `'/api/sse/.*'` → `'SSE'`.
    - `'/api/health'` → `'HEALTH'`.
    - All other `'/api/.*'` → `'GENERAL'`.
    - Non-API routes → no rate limiting.
  - Extract the IP (via `getRequestIp(request)`).
  - For IP-based limits (`AUTH`, `TICKET_ISSUANCE`, `SSE`, `HEALTH`): call `checkIpRateLimit(ip, routeGroup)`.
  - For user-based limits (`OFFICER_ACTIONS`): call `getToken()` from `next-auth/jwt`. If a token exists, call `checkUserRateLimit(userId, routeGroup)`. If no token, skip rate limiting (auth check will reject the request anyway).
  - For mixed limits (`GENERAL`): call `checkIpOrUserRateLimit(request, session, routeGroup)` where the session is from `getToken()`.
  - If the check returns `allowed: false`, return a 429 response using `createRateLimitResponse(result, routeGroup)`.
  - If allowed, set the rate limit headers on the response (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`) by wrapping the `NextResponse.next()` call.
- **Preserve existing middleware logic:**
  - The route protection (auth, RBAC) remains unchanged.
  - The redirect rules for unauthenticated users remain unchanged.
  - The role checks for `/security` (from 4.3.3) remain unchanged.
- **The `matcher` config** is updated to include `/api/:path*` (so the middleware runs on all API routes). Non-API routes (e.g., `/login`, `/dashboard`) continue to be matched by the existing matcher.

### Step 7: Update `src/lib/guards.ts` — Add `withRateLimit` Helper

**Goal:** Provide a higher-order function for handler-level rate limiting (in addition to the middleware-level).

**Files affected:** `src/lib/guards.ts` (modified — file was created by Phase 1.3.2).

**Specific instructions:**

Add the following to `src/lib/guards.ts`:

- **`async function withRateLimit<T>(handler: RouteHandler<T>, routeGroup: RouteGroup): RouteHandler<T>`** — wraps a route handler with rate limiting:
  - Extracts the IP from the request (via `getRequestIp`).
  - Calls `checkIpRateLimit(ip, routeGroup)`.
  - If not allowed, returns a 429 response.
  - If allowed, calls the wrapped handler.
  - Sets the rate limit headers on the success response.
- **Usage:** future endpoints that need handler-level rate limiting (e.g., for routes that need access to the request body for fingerprinting) can use `withRateLimit(handler, 'AUTH')`. The middleware-level rate limiting is the primary mechanism; this is for edge cases.

## 8. Common Pitfalls

- **Forgetting the `??=` pattern for the rate limit store.** Without it, every module reload in dev recreates the store, which breaks rate limit continuity. Use `globalThis.__rateLimitStore ??= new Map()`.
- **Putting rate limiting in the route handler instead of the middleware.** The middleware runs BEFORE auth, RBAC, and business logic. Putting rate limiting in the handler means the limit is not enforced for rejected requests (which are cheap to reject but still count).
- **Using `*` for `Access-Control-Allow-Origin` with credentials.** This is a security vulnerability. The middleware sets the actual origin dynamically.
- **Forgetting to set rate limit headers on successful responses.** Clients use these headers to know when to back off.
- **Forgetting to start the cleanup interval.** Without cleanup, the store grows indefinitely. Start it in `instrumentation.ts`.
- **Using `process.env.NEXT_RUNTIME` checks incorrectly.** The middleware runs in Edge runtime; the rate limit module should not import Prisma or other Node-only modules. Keep `rate-limit.ts` Edge-compatible (only uses `Map`, `Date`, primitives).
- **Not handling the case where the user is not authenticated for user-based limits.** The middleware skips rate limiting if no token exists — auth will reject the request anyway.
- **Forgetting the `NODE_ENV` check in the test endpoint.** Without it, the test endpoint is exposed in production.
- **Setting `Access-Control-Allow-Origin` to a static value in `next.config.ts`.** The middleware must overwrite it with the actual origin (or omit it for disallowed origins). The static value in `next.config.ts` is only for the OPTIONS preflight.
- **Setting the body size limit too low.** 10 MB is the recommended default. Lower limits may reject legitimate requests with large payloads (e.g., bulk operations).
- **Forgetting to handle the `SSE` route group with `windowMs: 0`.** The SSE limit is concurrent (not rate-based), so `windowMs: 0` is used to distinguish it. The `checkRateLimit` function must handle this case.

### 9.2 Implementation Sequence

2. **Step 2 (rate-limit-keys.ts).** Small helper module.
3. **Step 3 (cors.ts).** Independent of rate limiting.

10. **Step 10-11 (verification).** Build, lint, test.

### 9.3 What to Do If Stuck

- **Rate limiter not working?** Check the `globalThis.__rateLimitStore` pattern. If the store is recreated on every module reload, the buckets are lost.
- **Security headers missing on a response?** Verify the `next.config.ts` `headers()` function is correctly configured. Restart the dev server.
- **CORS preflight failing?** Check that the OPTIONS handler in the middleware is being invoked before the rate limiter.
- **429 response not including `Retry-After`?** Verify the `createRateLimitResponse` function sets the header in seconds, not milliseconds.

### 9.4 Testing Strategy

- Use `curl` for verification. The test scripts should:
  - Send 11 requests in a loop with `sleep 5` between them (to stay within the window).
  - Capture the response headers and body.
  - Assert the 11th response is 429.
- For SSE testing, use `curl -N` (no-buffer) to establish the connection and verify the 11th connection is rejected.
- For CORS testing, use `curl -X OPTIONS -H "Origin: https://test.com"`.

### 9.5 What NOT to Do

- Do not add rate limiting to non-API routes (e.g., `/login` page). The page has its own protection (login attempt limits are separate).
- Do not add rate limiting to static asset requests (`/_next/*`, `/sounds/*`).
- Do not skip the Edge runtime check in the middleware. The rate limit module is Edge-compatible, but if you accidentally import a Node-only module, the build fails.
- Do not use a third-party rate limit library (e.g., `next-rate-limit`). The in-memory implementation is sufficient and avoids dependencies.
