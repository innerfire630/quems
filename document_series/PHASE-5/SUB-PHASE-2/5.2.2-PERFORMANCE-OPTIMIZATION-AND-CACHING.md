## 1. Document Purpose

This document delivers the **performance optimization and caching layer** of the Smart Queue Management System. It establishes the infrastructure that keeps the system responsive as data volume grows: `unstable_cache` for slowly-changing data (active services, active counters, display board configurations, system settings) with `revalidateTag()` invalidation, database query optimization (eliminating N+1 patterns in report queries and adding missing indexes), Prisma query logging in development (with slow query detection), and a benchmark script that verifies the reports dashboard loads in under 2 seconds with 10,000 tickets.

This document is the **second of three documents in Sub-Phase 5.2**. It is the **performance layer**: while 5.2.1 protects the system from abuse (rate limiting, security headers), 5.2.2 ensures the system remains fast for legitimate users. The document also reviews the SSE connection pooling from Phase 3.1 (no changes — just a confirmation that the existing implementation is production-grade).

The strategic value of this document is that **it makes the system production-fast**. Without caching, every request to the reports dashboard re-queries the database for active services, counters, and settings. Without query optimization, the report queries can become N+1 nightmares as data grows. Without indexes, common queries (e.g., "tickets for service X on date Y") slow down to unacceptable levels. Without Prisma logging, developers cannot identify slow queries during development.

When this document is complete:

- The reports dashboard loads in under 2 seconds with 10,000 tickets in the database (verified by `scripts/benchmark-reports.ts`).
- Active services, counters, display boards, and system settings are cached with `unstable_cache` and invalidated precisely when underlying data changes (via `revalidateTag`).
- Database indexes are added (or verified) on frequently-queried fields: `Ticket.businessDate`, `Ticket.serviceId`, `Ticket.status`, `TicketEvent.eventType`, `CounterStatusEvent.counterId`, `Notification.counterOfficerId`.
- In development, Prisma queries are logged with timing. Slow queries (>100ms) are highlighted with a warning.
- The SSE connection pooling is reviewed and confirmed production-grade (heartbeat interval, concurrent connection limit).

This document does **not** implement rate limiting (owned by 5.2.1), the audit log viewer (owned by 5.2.3), the health check endpoint (owned by 5.2.3), or the error boundaries (owned by 5.2.3). It only handles performance.

### Step 1: Create `src/lib/cache-tags.ts` — Cache Tag Constants

**Goal:** Define the cache tags as constants for use across the application.

**Files affected:** `src/lib/cache-tags.ts` (new file).

**Specific instructions:**

Create the file with the following string constants:

- `export const SERVICES_TAG = 'services'` — invalidates when any service is created, updated, or deactivated. Affects: active services list, kiosk service list, display board service list, report service filter.
- `export const COUNTERS_TAG = 'counters'` — invalidates when any counter is created, updated, deactivated, or its status changes. Affects: active counters list, officer dashboard, display board counter list, report counter filter.
- `export const DISPLAY_BOARDS_TAG = 'display-boards'` — invalidates when any display board is created, updated, or deleted. Affects: display board configurations list, display board admin UI.

- `export const USERS_TAG = 'users'` — invalidates when any user is created, updated, or deactivated. Affects: user lists, user detail.
- `export const ROLE_PERMISSIONS_TAG = 'role-permissions'` — invalidates when role permissions are assigned or revoked. Affects: permission checks.
- `export const OFFICERS_TAG = 'officers'` — invalidates when counter officer assignments change. Affects: officer dashboard, notification dispatch.
- `export const QUEUE_SNAPSHOTS_TAG = 'queue-snapshots'` — invalidates when a daily reset runs. Affects: report cache (the daily reset flushes the snapshot data).

Also export the `CACHE_TAGS` object that aggregates all tags for easy reference:

- `export const CACHE_TAGS = { SERVICES_TAG, COUNTERS_TAG, DISPLAY_BOARDS_TAG, SETTINGS_TAG, USERS_TAG, ROLE_PERMISSIONS_TAG, OFFICERS_TAG, QUEUE_SNAPSHOTS_TAG } as const`.

### Step 2: Create `src/lib/cached-data.ts` — Cached Query Functions

**Goal:** Wrap the slowly-changing data queries with `unstable_cache`.

**Files affected:** `src/lib/cached-data.ts` (new file).

**Specific instructions:**

Create the file with the following functions:

- **`async function getActiveServices(): Promise<Service[]>`**:
  - Returns services where `isActive = true`, ordered by `sortOrder asc, code asc`.
  - Wrapped in `unstable_cache(fn, ['active-services'], { tags: [SERVICES_TAG], revalidate: 3600 })`.
  - The inner function `fn` is a Prisma query: `db.service.findMany({ where: { isActive: true }, orderBy: [{ sortOrder: 'asc' }, { code: 'asc' }] })`.
  - Used by the kiosk, display board, reports dashboard, and admin service management.
- **`async function getActiveCounters(): Promise<CounterWithServices[]>`**:
  - Returns counters where `isActive = true`, with their assigned services joined.
  - Wrapped in `unstable_cache(fn, ['active-counters'], { tags: [COUNTERS_TAG], revalidate: 3600 })`.
  - The inner function uses `include: { services: { include: { service: true } } }`.
  - Used by the officer dashboard, display board, reports dashboard, and admin counter management.
- **`async function getDisplayBoardConfigurations(): Promise<DisplayBoard[]>`**:
  - Returns all display board configurations, ordered by name.
  - Wrapped in `unstable_cache(fn, ['display-boards'], { tags: [DISPLAY_BOARDS_TAG], revalidate: 3600 })`.
  - Used by the display board page (to find the default board) and the admin display board UI.
- **`async function getSystemSetting(key: string): Promise<string | null>`**:
  - Reads a single system setting value.
  - Wrapped in `unstable_cache(fn, ['setting-' + key], { tags: [SETTINGS_TAG, `setting:${key}`], revalidate: 3600 })`.
  - The per-key tag allows invalidating specific settings without invalidating all settings.
  - Returns `null` if the setting is not found.
- **`async function getSystemSettings(keys: string[]): Promise<Record<string, string | null>>`**:
  - Bulk version of `getSystemSetting` — fetches multiple settings in one query (avoids N+1 in callers).
  - Wrapped in `unstable_cache(fn, ['settings-' + keys.sort().join(',')], { tags: [SETTINGS_TAG], revalidate: 3600 })`.
  - Returns a map of key → value.
- **`async function getUserRolePermissions(userId: string): Promise<Permission[]>`**:
  - Returns the flattened permission list for a user (resolved through role permissions).
  - Wrapped in `unstable_cache(fn, ['user-permissions-' + userId], { tags: [USERS_TAG, ROLE_PERMISSIONS_TAG], revalidate: 3600 })`.
  - Used by the middleware and `withPermission` to check permissions.
  - **WARNING:** caching permissions has a security implication: if a user's role is revoked, the cache may still grant access for up to 1 hour. For Phase 5.2, this is acceptable (the existing `withPermission` does a database lookup anyway). The cache is an optimization for the COMMON case (user has stable permissions). The cache hit rate should be high.

**Type imports:** `Service`, `Counter`, `DisplayBoard`, `Permission` from `@prisma/client` or `@/types/...` as appropriate.

### Step 4: Create `src/lib/prisma-logging.ts` — Prisma Query Logger

**Goal:** Provide the logging logic with slow query detection.

**Files affected:** `src/lib/prisma-logging.ts` (new file).

**Specific instructions:**

Create the file with:

- **`const SLOW_QUERY_THRESHOLD_MS = 100`** — the threshold for slow query warnings.
- **`function logQuery(event: Prisma.QueryEvent): void`** — handles the `'query'` event:
  - Extracts `event.query`, `event.params`, `event.duration` (in ms).
  - Logs via `console.debug` with the query text and duration: `[Prisma] ${duration}ms ${query}`.
  - If `duration > SLOW_QUERY_THRESHOLD_MS`, logs via `console.warn` with a `⚠️ SLOW QUERY` prefix: `[Prisma] ⚠️ SLOW QUERY (${duration}ms) ${query} - params: ${params}`.
  - **Does NOT log full params if they contain sensitive data** (e.g., passwords, tokens). The agent should sanitize the params — strip fields named `password`, `token`, `secret`, `hashedPassword`, `refreshToken`. This is a basic guard; for full security, the caller should pass sanitized params.
- **`function logWarn(event: Prisma.LogEvent): void`** — handles the `'warn'` event: logs via `console.warn`.
- **`function logError(event: Prisma.LogEvent): void`** — handles the `'error'` event: logs via `console.error`.
- **`function isLoggingEnabled(): boolean`** — returns `true` if logging is enabled. Checks `process.env.PRISMA_LOG_QUERIES` and `process.env.NODE_ENV`.

### Step 5: Update `prisma/schema.prisma` — Add Database Indexes

**Goal:** Add indexes on frequently-queried fields to optimize common queries.

**Files affected:** `prisma/schema.prisma` (modified — file was created by Phase 1.1.3).

**Specific instructions:**

Add `@@index([...])` declarations to the relevant models:

- **`Ticket` model** — add:
  - `@@index([businessDate])` — speeds up date-range queries (e.g., "tickets for service X between dates A and B").
  - `@@index([serviceId, businessDate])` — composite for the report queries.
  - `@@index([status, businessDate])` — composite for status filtering within a date range.
  - `@@index([counterId])` — speeds up counter-specific queries.
- **`TicketEvent` model** — add:
  - `@@index([eventType, eventTime])` — speeds up peak hour calculation and event-type filtering.
  - `@@index([ticketId, eventTime])` — speeds up event log queries for a single ticket.
- **`CounterStatusEvent` model** — add:
  - `@@index([counterId, createdAt])` — speeds up "recent status events for a counter" queries.
  - `@@index([status, createdAt])` — speeds up "all CLOSED events in a period" queries.
- **`Notification` model** — add:
  - `@@index([counterOfficerId, status])` — speeds up "active notifications for an officer" queries (used by 4.1.3 dispatch).
  - `@@index([createdAt])` — speeds up time-range queries.
- **`NotificationReply` model** — add:
  - `@@index([notificationId])` — speeds up reply lookup by notification.
- **`BroadcastMessage` model** — add:
  - `@@index([isActive, expiresAt])` — speeds up "active broadcasts" queries.
- **`AuditLog` model** — add (for 5.2.3's audit log viewer):
  - `@@index([createdAt])` — speeds up time-range queries.
  - `@@index([userId, createdAt])` — speeds up "audit log for user X" queries.
  - `@@index([action, createdAt])` — speeds up "audit log for action type X" queries.
  - `@@index([entity, entityId])` — speeds up "audit log for entity X" queries.

**Important:** Existing indexes should NOT be removed. Only ADD new indexes.

### Step 7: Update Existing Mutation Endpoints to Call `revalidateTag`

**Goal:** Ensure cache invalidation happens whenever underlying data changes.

**Files affected:** Multiple endpoint files. This step is a survey + update across existing endpoints.

**Specific instructions:**

For each mutation endpoint listed below, add `revalidateTag(TAG)` calls. The calls are placed AFTER the transaction commits (best-effort, wrapped in try/catch).

- **Service mutations:**
  - `POST /api/services` (create, from 2.1.1): `revalidateTag(SERVICES_TAG)`.
  - `PATCH /api/services/[serviceId]` (update, from 2.1.1): `revalidateTag(SERVICES_TAG)`.
  - `DELETE /api/services/[serviceId]` (deactivate, from 2.1.1): `revalidateTag(SERVICES_TAG)`.
- **Counter mutations:**
  - `POST /api/counters` (create, from 2.1.2): `revalidateTag(COUNTERS_TAG)`.
  - `PATCH /api/counters/[counterId]` (update, from 2.1.2): `revalidateTag(COUNTERS_TAG)`.
  - `DELETE /api/counters/[counterId]` (deactivate, from 2.1.2): `revalidateTag(COUNTERS_TAG)`.
  - `PATCH /api/counters/[counterId]/status` (status change, from 4.2.1): `revalidateTag(COUNTERS_TAG)` AND `revalidateTag(OFFICERS_TAG)`.
- **Service-counter assignment mutations:**
  - `POST /api/counters/[counterId]/services` (assign, from 2.1.3): `revalidateTag(SERVICES_TAG)` AND `revalidateTag(COUNTERS_TAG)`.
  - `DELETE /api/counters/[counterId]/services/[serviceId]` (unassign, from 2.1.3): `revalidateTag(SERVICES_TAG)` AND `revalidateTag(COUNTERS_TAG)`.
- **Display board mutations:**
  - `POST /api/display-boards` (create, from 3.2.3): `revalidateTag(DISPLAY_BOARDS_TAG)`.
  - `PATCH /api/display-boards/[boardId]` (update, from 3.2.3): `revalidateTag(DISPLAY_BOARDS_TAG)`.
  - `DELETE /api/display-boards/[boardId]` (delete, from 3.2.3): `revalidateTag(DISPLAY_BOARDS_TAG)`.
- **User mutations:**
  - `POST /api/users` (create, from 1.3.3): `revalidateTag(USERS_TAG)`.
  - `PATCH /api/users/[userId]` (update, from 1.3.3): `revalidateTag(USERS_TAG)`.
  - `POST /api/users/[userId]/reset-password` (reset, from 1.3.3): `revalidateTag(USERS_TAG)`.
- **Role permission mutations:**
  - `POST /api/roles/[roleId]/permissions` (assign, from 1.3.1): `revalidateTag(ROLE_PERMISSIONS_TAG)`.
  - `DELETE /api/roles/[roleId]/permissions/[permissionId]` (revoke, from 1.3.1): `revalidateTag(ROLE_PERMISSIONS_TAG)`.
- **System setting mutations (when implemented in a future admin UI):**
  - `PATCH /api/settings/[key]` (update): `revalidateTag(SETTINGS_TAG)`.
- **Daily reset (from 2.3.3, extended by 5.1.1):**
  - In `runDailyReset`, after the snapshot is written: `revalidateTag(QUEUE_SNAPSHOTS_TAG)`.

**Pattern for `revalidateTag` calls:**

- Wrap in try/catch. Failures are logged via `console.error` but do NOT fail the response.
- The call is AFTER the transaction commits (not inside the transaction).
- The call is synchronous (no `await` — `revalidateTag` is sync).

### Step 8: Create `scripts/benchmark-reports.ts` — Benchmark Script

**Goal:** Provide a script that verifies the reports dashboard loads in under 2 seconds with 10,000 tickets.

**Files affected:** `scripts/benchmark-reports.ts` (new file).

**Specific instructions:**

Create the file with:

- **Purpose:** Seed the database with 10,000 tickets and time the `GET /api/reports` request.
- **Structure:**
  - **Phase 1 — Setup:** connect to the database, check if 10,000 tickets already exist (skip seeding if they do).
  - **Phase 2 — Seed (if needed):** create 10,000 tickets spread across 5 services, 30 days, and various statuses (WAITING, CALLED, NO_SHOW, SERVED, COMPLETED). Use `db.ticket.createMany({ data: [...] })` in batches of 500. This may take 30-60 seconds.
  - **Phase 3 — Benchmark:** make a fetch request to `http://localhost:3000/api/reports?startDate=...&endDate=...&_token=<admin_token>`. Time the request. Log the time in ms.
  - **Phase 4 — Assert:** if the time exceeds 2000ms, log a `FAIL` message. Otherwise, log `PASS`.
  - **Phase 5 — Cleanup (optional):** delete the seeded tickets if a `--cleanup` flag is passed.
- **The script runs against the local development server.** It assumes `yarn dev` is running on `localhost:3000`.
- **The script requires an admin token** for the `GET /api/reports` request. The token can be obtained by logging in via the auth endpoint and extracting the access token from the response.
- **The script outputs a summary:**
  - Number of tickets in the database.
  - Number of services.
  - Benchmark time in ms.
  - Pass/fail status.

## 8. Common Pitfalls

- **Caching data that changes frequently.** Only cache slowly-changing data (services, counters, display boards, settings). Tickets, notifications, and broadcasts change constantly — do NOT cache them.
- **Calling `revalidateTag` inside a transaction.** Cache invalidation must be post-commit.
- **Letting cache invalidation failures block the response.** Wrap `revalidateTag` in try/catch. The cache is best-effort.
- **Using `unstable_cache` without tags.** Without tags, the cache cannot be invalidated by mutations. The TTL fallback (1 hour) is too slow for a user-facing admin UI.
- **Forgetting to import `revalidateTag` from `next/cache`.** The function is from Next.js, not a custom helper.
- **Using `unstable_cache` for authenticated data.** The cache is per-process, not per-user. Caching user-specific data (e.g., the user's notification list) leaks data between users. Only cache data that is the SAME for all users (services, counters, settings).
- **Caching permissions with a long TTL.** The cache is 1 hour. If a user's role is revoked, they retain access for up to 1 hour. For Phase 5.2, this is acceptable (the `withPermission` guard does a database lookup as a safety net). The cache is an optimization for the COMMON case. Document this trade-off.
- **Adding indexes that duplicate existing indexes.** Check the existing schema before adding. The migration should only CREATE new indexes.
- **Running the benchmark script against the production database.** The script is for development only. Add a `NODE_ENV` check.
- **Logging full Prisma query parameters in production.** The parameters may contain sensitive data. The logger sanitizes sensitive field names (`password`, `token`, etc.) but the agent should verify this is sufficient.
- **Forgetting to regenerate the Prisma client after adding indexes.** Run `yarn prisma:generate` after `yarn prisma migrate dev`.

### 9.3 What to Do If Stuck

- **Cache not invalidating?** Verify the `revalidateTag` call is AFTER the transaction commits and uses the correct tag constant.
- **Benchmark exceeding 2 seconds?** Check for N+1 patterns in the report queries. The optimization may need to add more indexes or refactor the queries.
- **Prisma logging not appearing?** Check that `PRISMA_LOG_QUERIES` is `'true'` (or unset in development). Verify the event listeners are registered in `db.ts`.
- **Migration failing?** Verify the SQL syntax is compatible with the database (SQLite for development, PostgreSQL for production).

### 9.4 What NOT to Do

- Do not cache user-specific data (notifications, replies, broadcasts). The cache is per-process.
- Do not use `unstable_cache` for the `/api/reports` endpoint itself — the data changes too frequently. Cache the slowly-changing data the endpoint queries.
- Do not skip the migration when adding new indexes. The Prisma migration must be applied.
- Do not log full query parameters without sanitization. Sensitive data may leak.
- Do not introduce Redis or any external caching system in Phase 5.2. The in-process cache is sufficient.
