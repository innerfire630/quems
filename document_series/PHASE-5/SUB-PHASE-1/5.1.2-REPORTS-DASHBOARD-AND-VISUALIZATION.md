## 1. Document Purpose

This document delivers the **presentation layer** for the analytics and reporting sub-phase — the `/reports` page at `src/app/(dashboard)/reports/page.tsx` and all the components that render the analytics data: KPI cards, tickets-by-hour bar chart, per-service performance table, per-counter performance table, date range picker, and service/counter filters.

This document is the **second document in Sub-Phase 5.1**. It consumes the `getReportData` function and the `ReportData` type defined in 5.1.1. The dashboard is a pure consumer of the analytics endpoint — it does not implement calculation logic, does not query the database directly (beyond the initial server-side fetch), and does not write any data.

The strategic value of this document is that it **makes the system observable to its operators**. The dashboard is the user-facing surface for the analytics layer. Before 5.1.2, the data collected by 5.1.1 is not visible to anyone. After 5.1.2, administrators can open `/reports` in their browser and immediately see how the queue is performing — KPIs, trends, and per-service / per-counter breakdowns — and change the date range or filters to slice the data differently.

When this document is complete:

- An administrator can navigate to `/reports` and see the analytics dashboard.
- The dashboard shows four KPI cards: total tickets, average wait time, no-show rate, busiest hour.
- The dashboard shows a tickets-by-hour bar chart with 24 hour buckets.
- The dashboard shows a per-service performance table with sortable columns.
- The dashboard shows a per-counter performance table with sortable columns.
- The date range picker allows the user to change the start and end dates.
- The service and counter filters allow the user to focus on a single service or counter.
- When any filter changes, the dashboard re-fetches the data and re-renders.
- Loading skeletons appear during the fetch.
- Empty state messages appear when no data exists for the selected filters.
- A non-admin user is denied (403 or redirect to `/?error=forbidden`).
- The "Reports" nav item is visible in the admin sidebar (only for users with `report:view`).
- Every successful report fetch writes a `REPORT_GENERATED` audit log entry (handled by the `GET /api/reports` endpoint from 5.1.1).

This document does **not** implement the data collection (owned by 5.1.1), the CSV export (owned by 5.1.3), the calculation logic (owned by 5.1.1's `lib/analytics-service.ts`), the audit log viewer (owned by 5.2.3), or the rate limiting (owned by 5.2.1).

### Step 2: Create `src/components/reports/date-range-picker.tsx` — Date Range Picker

**Goal:** Provide a date range picker that emits start and end dates to the parent.

**Files affected:** `src/components/reports/date-range-picker.tsx` (new file).

**Specific instructions:**

Create the file with:

- A `"use client"` directive.
- **Props:** `{ startDate: string, endDate: string, onChange: (range: { startDate: string, endDate: string }) => void }`.
- **Behaviour:**
  - Two date inputs (`<input type="date">`). Each shows the current value and emits `onChange` with the updated range.
  - The inputs are constrained: `min` is 90 days before today; `max` is today. The end date input's `min` is the start date.
  - Validation on change: if `endDate < startDate`, do not call `onChange` (or call with a flag indicating invalid). The Zod schema in the API will reject invalid ranges anyway.
  - A "Today" button that sets both dates to today and calls `onChange`.
  - A "Last 7 days" button that sets start to 7 days ago and end to today.
  - (Both preset buttons are simple convenience — the main UX is the date inputs.)
- **Styling:** uses shadcn `Input` for the date inputs and `Button` for the presets. Layout is a flex row.

### Step 8: Create `src/components/reports/reports-dashboard-client.tsx` — Client Wrapper

**Goal:** Own the filter state, fetch the data, render the sub-components.

**Files affected:** `src/components/reports/reports-dashboard-client.tsx` (new file).

**Specific instructions:**

Create the file with:

- A `"use client"` directive.
- **Props:** `{ initialData: ReportData, services: Array<{ id, code, name }>, counters: Array<{ id, name, number }> }`.
- **State:**
  - `startDate: string` — initial value `initialData.startDate`.
  - `endDate: string` — initial value `initialData.endDate`.
  - `serviceId: string | null` — initial value `initialData.serviceId`.
  - `counterId: string | null` — initial value `initialData.counterId`.
  - `reportData: ReportData` — initial value `initialData`.
  - `isLoading: boolean` — initial value `false`.
  - `error: string | null` — initial value `null`.
- **Behaviour:**
  - **`async function refetch()`**: builds the query string, calls `GET /api/reports?startDate=...&endDate=...&serviceId=...&counterId=...` via `fetch`. On success, updates `reportData`. On error, updates `error`. Sets `isLoading = true` during the fetch and `false` after.
  - **Debounced refetch:** uses a 300ms debounce on filter changes (so rapid typing in the date input doesn't trigger many requests). The implementation: a `useRef` for the timeout, cleared and reset on each filter change.
  - **Effect:** watches `startDate`, `endDate`, `serviceId`, `counterId`. On change, calls the debounced refetch.
- **Render:**
  - **Header bar:** title "Reports", subtitle showing the current date range (e.g., "Showing data from 2026-06-20 to 2026-06-20"). A button slot for the "Export CSV" button (5.1.3) — rendered as `{/* Export CSV button slot — implemented by 5.1.3 */}` with a comment.
  - **Filter row:** `<DateRangePicker>` and `<ReportFilters>` in a flex row.
  - **Loading state:** when `isLoading === true`, shows shadcn `Skeleton` placeholders for the KPI cards and the tables (the chart shows a loading spinner).
  - **Error state:** when `error !== null`, shows an error message at the top of the dashboard with a retry button.
  - **KPI cards row:** `<ReportKPICards summary={reportData.kpi} />`.
  - **Chart:** `<TicketsByHourChart data={reportData.hourly} />`.
  - **Per-service table:** `<ServicePerformanceTable rows={reportData.services} />`.
  - **Per-counter table:** `<CounterPerformanceTable rows={reportData.counters} />`.
  - **Empty state:** when `reportData.kpi.totalTickets === 0 && reportData.services.length === 0`, shows a centered message: "No data for the selected period" with a lucide-react `Inbox` icon.

### Step 9: Create `src/app/(dashboard)/reports/page.tsx` — The Reports Page

**Goal:** The server component that guards the page, loads initial data, and renders the client component.

**Files affected:** `src/app/(dashboard)/reports/page.tsx` (new file).

**Specific instructions:**

Create the file with:

- **Route configuration exports:**
  - `export const runtime = 'nodejs'` (required for Prisma + auth).
  - `export const dynamic = 'force-dynamic'` (required — Next.js must not attempt to cache the response).
- **`async function Page()`** (default export):
  - **Step 1 — Authentication:** calls `auth()` from `@/lib/auth`. If no session → `redirect('/login')`.
  - **Step 2 — Permission check:** calls `withPermission` server-side helper or manually checks the permission via `session.user.permissions`. If the user lacks `report:view` → `redirect('/?error=forbidden')`.
    - **Implementation note:** use the existing pattern from `(dashboard)/layout.tsx` (Phase 1.2.3) which checks session + role. For permissions, use `session.user.permissions.includes('report:view')` (the permissions are flattened into the session per Phase 1.2.1).
  - **Step 3 — Load services and counters (for the filter dropdowns):**
    - `const services = await db.service.findMany({ where: { isActive: true }, select: { id: true, code: true, name: true }, orderBy: { code: 'asc' } })`.
    - `const counters = await db.counter.findMany({ where: { isActive: true }, select: { id: true, name: true, number: true }, orderBy: { number: 'asc' } })`.
  - **Step 4 — Load initial report data:** today's snapshot.
    - `const today = new Date(); const todayStr = today.toISOString().split('T')[0]` (or use `formatDate` helper that respects `APP_TIMEZONE`).
    - `const initialData = await getReportData(new Date(todayStr + 'T00:00:00.000Z'), new Date(todayStr + 'T23:59:59.999Z'))`.
  - **Step 5 — Render:** returns `<ReportsDashboardClient initialData={initialData} services={services} counters={counters} />`.
- **Metadata:**
  - `export const metadata = { title: 'Reports' }` — sets the page title in the browser tab.

## 8. Common Pitfalls

- **SSR and Recharts.** Recharts uses browser APIs. The chart component MUST be a client component (`"use client"`). If you see a `window is not defined` error, the component is server-rendered.
- **Hydration mismatches.** The chart component renders on the client only. Use `useEffect` to defer rendering until mounted (or use a `dynamic` import with `ssr: false` if needed).
- **Date format.** The `startDate` and `endDate` are `YYYY-MM-DD` strings. The `formatDate` helper in `analytics-service.ts` handles the conversion. Do NOT pass JavaScript Date objects to the API directly (they serialize differently).
- **Filter URL state.** The filter state is in React, not in the URL. If the user refreshes the page, the filters reset to "today". Future enhancement: persist via URL params (e.g., `?startDate=...&endDate=...&serviceId=...`).
- **Debounce timing.** 300ms is a good default. Too short → many requests on rapid typing. Too long → feels sluggish.
- **Empty state vs loading state.** Loading shows skeletons. Empty (after fetch returns no data) shows the empty message. They are distinct — don't conflate.
- **Permission check at the page level.** The page MUST check `report:view` and redirect if missing. Do NOT rely on the sidebar hiding the nav item — a user with the URL can bypass the sidebar.
- **Initial data load.** The page fetches the initial data server-side. The client then refetches on filter change. The initial data must match the default filter values (today's date, no service/counter filter) so the first render shows real data.
- **Service and counter lists.** The service/counter lists passed to the page are the FULL lists (not filtered by anything). The user selects from the full list. Inactive services/counters are excluded at the query level.
- **Recharts responsive container.** Wrap the chart in `<ResponsiveContainer width="100%" height={300}>` to make it responsive to the parent width.
