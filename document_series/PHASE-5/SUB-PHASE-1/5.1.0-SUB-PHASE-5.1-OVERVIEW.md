## 1. Sub-Phase Purpose

Sub-Phase 5.1 is the **observability layer** of the Smart Queue Management System. It transforms the system from a "working but opaque" deployment into a "working and measurable" deployment. Where Phases 1–4 built features, Sub-Phase 5.1 makes those features observable to their operators: administrators can answer questions like "What's our average wait time today?", "Which counter is the slowest?", "When is our busiest hour?", and "What's the no-show rate?".

The strategic goal of this sub-phase is to enable **data-driven operational decisions**. Without analytics, the system is a black box — tickets are issued and served, but no one knows if the queue is performing well. With analytics, the system becomes self-aware: it accumulates the metrics that allow administrators to identify bottlenecks, optimise counter staffing, and quantify improvements over time.

The work in this sub-phase has three layers, matching the three task plan documents:

1. **Collection** (5.1.1) — gather the data. The `QueueDailySnapshot` table is already populated with some metrics by the daily reset in Phase 2.3.3. This sub-phase **extends** the daily reset to include ALL metrics defined in the master plan's Section 8.2 (`totalIssued`, `totalServed`, `totalNoShow`, `totalCancelled`, `totalWaiting`, `averageWaitMinutes`, `averageServiceMinutes`, `peakHour`), adds **incremental metric recording during the day** so the dashboard reflects near-real-time data, and exposes the metrics via the `GET /api/reports` endpoint.
2. **Presentation** (5.1.2) — show it in a dashboard. The `/reports` page renders four KPI cards, a tickets-by-hour bar chart, a per-service performance table, and a per-counter performance table. Date range picker, service filter, and counter filter allow the administrator to focus on the slice of data they need. The page is behind the `report:view` permission.
3. **Export** (5.1.3) — let administrators take it out of the system. The `GET /api/reports/export` endpoint returns a properly formatted CSV file with one row per service per day. The "Export CSV" button on the dashboard initiates the download. The endpoint is behind the stricter `report:export` permission. The architecture also defines an interface for future scheduled email reports, but the scheduler is NOT implemented in this phase (a documented future enhancement).

When this sub-phase is complete:

- After a business day with multiple tickets, the `QueueDailySnapshot` contains correct `totalIssued`, `totalServed`, `totalNoShow`, `totalCancelled`, `totalWaiting`, `averageWaitMinutes`, `averageServiceMinutes`, and `peakHour` values.
- The `GET /api/reports` endpoint returns analytics data for any date range with optional service and counter filters.
- The reports dashboard at `/reports` shows the KPIs and charts for the selected period and updates when the date range, service filter, or counter filter changes.
- The CSV export endpoint returns a properly formatted CSV with the correct columns and escaping.
- The "Export CSV" button on the dashboard downloads the file.
- A user without `report:view` cannot access the reports page (403).
- A user without `report:export` cannot call the export endpoint (403).
- Every report generation and export action is recorded in the `AuditLog` (as `REPORT_GENERATED` and `REPORT_EXPORTED` events).

Three problems are solved in this sub-phase, in order:

1. **Queue analytics data collection** — the `QueueDailySnapshot` population logic, the incremental metric recording during the day, the average wait time and service time calculation methodology, the peak hour calculation, the analytics query API endpoints, and the `lib/analytics-service.ts` centralised module with pure metric calculation functions. (Document 5.1.1.)
2. **Reports dashboard & visualization** — the `/reports` page layout, the KPI cards, the tickets-by-hour bar chart, the per-service performance table, the per-counter performance table, the date range picker, the service/counter filters, the loading/empty state handling, and the integration of the existing `DataTable` component for the tabular views. (Document 5.1.2.)
3. **Data export & scheduled reports** — the `GET /api/reports/export` endpoint with proper CSV generation, the "Export CSV" button on the dashboard, the `lib/report-export.ts` utility module, and the future hook for automated scheduled email reports (interface only, no scheduler). (Document 5.1.3.)

When this sub-phase is complete, **Phase 5.1 is finished**. The system has its analytics and reporting layer. Sub-Phase 5.2 (system hardening) can begin — it adds rate limiting and security headers that protect the new endpoints (5.1.3 in particular), caches the slowly-changing data that 5.1.1 queries, and adds the audit log viewer that surfaces the new `REPORT_GENERATED` and `REPORT_EXPORTED` events.

## 1.1 Why This Sub-Phase Comes First in Phase 5

Sub-Phase 5.1 precedes 5.2 (hardening) and 5.3 (deployment) because:

- **The reports dashboard (5.1.2) needs hardening.** Rate limiting (5.2.1) protects the report endpoints from abuse. Security headers (5.2.1) protect the report page. Caching (5.2.2) speeds up the report queries. Building the reports first and hardening them afterwards is cleaner than hardening an empty page and then adding features.
- **The CSV export endpoint (5.1.3) needs rate limiting and audit logging.** Building the export before the rate limiter means the export is temporarily unprotected. The `REPORT_EXPORTED` audit log entries are surfaced by 5.2.3's viewer.
- **The data collection in 5.1.1 must be in place before 5.1.2 (the dashboard) can show real data.** The dashboard without data is a placeholder.
- **The CSV export in 5.1.3 shares infrastructure with the reports dashboard.** Both query the same `QueueDailySnapshot` and `Ticket`/`TicketEvent` data. Implementing both in parallel is fine after 5.1.1 is complete.
- **The audit log writes from 5.1.3 use the existing helper.** 5.2.3's audit log viewer surfaces them.

## 1.2 Why This Sub-Phase Comes After Phase 4

Sub-Phase 5.1 depends on the operational surface from Phase 4 in the following ways:

- **The `Notification` and `NotificationReply` tables (populated by 4.1 and 4.3) are queryable by 5.1.1** for "notifications dispatched per officer per day", "replies received per officer per day", "average time from notification to reply". These are future report metrics; the tables exist and are populated by Phase 4 but no report consumes them yet.
- **The `CounterStatusEvent` and `CounterOfficer.currentStatus` (populated by 4.2.1) are queryable by 5.1.1** for "counter closure events per day" and "average closure duration". Future report metrics.
- **The `BroadcastMessage` and `OFFICER_REPLY` events (emitted by 4.3.2 and 4.3.1) are queryable by 5.1.1** for "broadcast volume per shift". Future report metric.
- **The Phase 2.3.3 daily reset already populates a partial `QueueDailySnapshot`.** 5.1.1 extends this to include all the missing fields.

## 1.3 What This Sub-Phase Is NOT

- It is **not** real-time analytics. The reports are static pages with on-demand refresh (no live updating charts). Real-time dashboards are a future enhancement.
- It is **not** automated scheduled email reports. The interface for scheduling is defined in 5.1.3 (function signature, configuration storage), but the scheduler is NOT implemented. An administrator manually triggers the export by clicking the button.
- It is **not** a BI tool. The reports are summary KPIs and tables, not drill-down analytics with cross-filtering and pivoting.
- It is **not** historical data analysis beyond what is stored. The system retains `QueueDailySnapshot` records indefinitely (or until cleanup, which is a future enhancement). No data warehouse, no OLAP cube.
- It is **not** rate limiting. Phase 5.2.1 adds rate limits to the report endpoints.
- It is **not** the audit log viewer. Phase 5.2.3 adds the viewer UI; 5.1.3 writes the audit entries.
- It is **not** PostgreSQL migration. Phase 5.3.1 migrates the database; 5.1 queries the same data regardless of provider.
- It is **not** performance optimisation. Phase 5.2.2 adds caching for the report queries.
- It is **not** machine learning or NLP. "Most common reply keywords" is a future enhancement requiring NLP.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to distribute the complexity across the three task documents. The split is:

- **5.1.1** owns the **data collection layer**. It extends the daily reset (Phase 2.3.3) to populate ALL `QueueDailySnapshot` fields, adds the incremental metric recording during the day, implements the `lib/analytics-service.ts` module with pure calculation functions, and exposes the metrics via the `GET /api/reports` endpoint. It does **not** know about the dashboard UI (5.1.2), the CSV export (5.1.3), or the charts. It is the data backend.
- **5.1.2** owns the **presentation layer**. It creates the `/reports` page, the KPI cards, the bar chart, the per-service/per-counter tables, the date range picker, the filters, and the loading/empty states. It does **not** know about the calculation logic (5.1.1 owns that) or the export (5.1.3 owns that). It consumes the `GET /api/reports` endpoint as a black box.
- **5.1.3** owns the **export layer**. It creates the `GET /api/reports/export` endpoint, the CSV generation logic, the export button on the dashboard, and the future scheduled reports interface. It does **not** know about the KPI cards or charts (5.1.2 owns that) or the calculation logic (5.1.1 owns that). It uses the same `lib/analytics-service.ts` module as 5.1.2.

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 5.1.1 — Queue Analytics Data Collection

**Owns:** The `QueueDailySnapshot` complete population logic (extending the daily reset in `src/lib/queue-reset.ts` from Phase 2.3.3), the incremental metric recording during the day (in-memory `ServiceDailyCounters` map), the average wait time and service time calculation methodology, the peak hour calculation, the `lib/analytics-service.ts` centralised module with pure metric calculation functions, the `GET /api/reports` endpoint, the Zod schemas in `src/schemas/report.schema.ts`, and the route protection.

**Does NOT own:** The reports dashboard UI (owned by 5.1.2), the CSV export (owned by 5.1.3), the audit log viewer (owned by 5.2.3), the rate limiting (owned by 5.2.1), the caching (owned by 5.2.2).

**Key outputs:**

- **Extended `src/lib/queue-reset.ts`** — the daily reset module from Phase 2.3.3 is EXTENDED to populate ALL `QueueDailySnapshot` fields. The Phase 2 implementation computed `totalIssued`, `totalNoShow`, and `peakHour`; 5.1.1 adds `totalServed`, `totalCancelled`, `totalWaiting`, `averageWaitMinutes`, and `averageServiceMinutes` (the latter two require the `completedAt` and `servedAt` fields that may not have been populated in Phase 2 due to deferred ticket state transitions). The extension is purely additive — no changes to existing field computation.
- **`src/lib/analytics-service.ts`** — the centralised analytics module:
  - **`async function calculateServiceDailyMetrics(serviceId: string, businessDate: Date): Promise<ServiceDailyMetrics>`** — pure server-side function that takes a service and a business date and returns all 8 metrics for that combination. Queries `Ticket`, `TicketEvent`, and (for `peakHour`) `TicketEvent` with `eventType = 'CALLED'`. Returns nulls for metrics that require data not yet present (e.g., `averageServiceMinutes` when no tickets are completed).
  - **`async function calculateServicePerformanceRows(startDate: Date, endDate: Date, serviceIds?: string[]): Promise<ServicePerformanceRow[]>`** — returns one row per service for the date range, aggregated across days. Used by 5.1.2's per-service table.
  - **`async function calculateCounterPerformanceRows(startDate: Date, endDate: Date, counterIds?: string[]): Promise<CounterPerformanceRow[]>`** — returns one row per counter for the date range. Used by 5.1.2's per-counter table. Counts tickets that called at the counter (via `TicketEvent.eventType = 'CALLED'`), no-shows at the counter, and closure events (via `CounterStatusEvent`).
  - **`async function calculateTicketsByHour(startDate: Date, endDate: Date, serviceId?: string): Promise<HourlyTicketCount[]>`** — returns an array of 24 hour buckets (0-23) with the count of tickets issued in each hour. Used by 5.1.2's bar chart.
  - **`async function calculateReportKpiSummary(startDate: Date, endDate: Date, serviceId?: string, counterId?: string): Promise<ReportKpiSummary>`** — returns the 4 KPI values: total tickets, average wait, no-show rate, busiest hour.
  - **`async function getCurrentServiceCounters(serviceId: string, businessDate: Date): Promise<ServiceDailyCounters>`** — returns the in-memory running counters for a service on the current business date. Used by the increment functions below.
  - **`function incrementServiceCounter(serviceId: string, event: TicketEventType): void`** — called by ticket state-change paths (issuance, call, no-show, etc.) to keep the in-memory counters in sync with the database. Best-effort (failures logged but never thrown). 5.1.1 does NOT modify existing ticket handlers — the increments are added as a parallel write in 5.1.1's own tracking module, called from the existing handlers via a thin integration layer. The integration approach: the existing handlers (Phase 2.2.1 for issuance, Phase 2.3.1 for call/recall, Phase 2.3.2 for no-show) call `incrementServiceCounter(serviceId, eventType)` after their transactions commit. This is the ONLY change to the existing handlers.
  - **`function resetServiceCountersForDate(businessDate: Date): void`** — called by the daily reset to clear the in-memory counters after they're flushed to the `QueueDailySnapshot`. Idempotent.
  - **`function getReportData(startDate: Date, endDate: Date, serviceId?: string, counterId?: string): Promise<ReportData>`** — the main entry point for the `GET /api/reports` endpoint. Combines the KPI summary, per-service breakdown, per-counter breakdown, and hourly chart data into a single `ReportData` object. Single function call for the endpoint — the function handles the parallel `Promise.all` of the four sub-calculations.
  - **Module-scope state:** `const serviceDailyCounters: Map<string, ServiceDailyCounters>` — the in-memory running counters. Survives across requests within a single Node.js process. Reset by the daily reset. Cleared by a manual reset.
  - **Module-scope constants:** `PEAK_HOUR_RANGE_START = 0`, `PEAK_HOUR_RANGE_END = 23`, `BUSIEST_HOUR_THRESHOLD = 5` (a service with fewer than 5 calls has no meaningful "busiest hour").
- **Integration with existing ticket handlers** — the increment calls are added to:
  - **Phase 2.2.1 (ticket issuance):** `incrementServiceCounter(serviceId, 'ISSUED')` after the transaction commits.
  - **Phase 2.3.1 (call/recall):** `incrementServiceCounter(serviceId, 'CALLED')` after each call (recall does NOT increment — the ticket was already counted as CALLED).
  - **Phase 2.3.2 (no-show):** `incrementServiceCounter(serviceId, 'NO_SHOW')` after the no-show transaction commits.
  - **Future ticket state transitions** (Phase beyond 5.1, when `SERVING` and `COMPLETED` transitions are implemented): `incrementServiceCounter(serviceId, 'SERVED' | 'COMPLETED' | 'CANCELLED')` after each. These are documented but NOT implemented in 5.1.
  - The increments are best-effort and NEVER throw. A failure in `incrementServiceCounter` is logged via `console.error` and does NOT fail the ticket operation. This is the same discipline as the audit log writes.
- **`GET /api/reports` endpoint** at `src/app/api/reports/route.ts`:
  - `GET` handler. Guarded with `withPermission(handler, PERMISSION_REPORT_VIEW)`. The `report:view` permission is consumed here for the first time (defined in master plan Section 10.4, seeded in Phase 1.3.1).
  - Query params: `startDate` (YYYY-MM-DD, required), `endDate` (YYYY-MM-DD, required), `serviceId?` (cuid, optional), `counterId?` (cuid, optional).
  - Validated via `reportsQuerySchema` (Zod).
  - Calls `getReportData(startDate, endDate, serviceId?, counterId?)` from `@/lib/analytics-service`.
  - Returns `{ success: true, data: ReportData }`.
  - **Errors:** `VALIDATION_ERROR` (422) for invalid dates (start after end, range too large, etc.), `FORBIDDEN` (403) for non-admin.
  - **Route config:** `runtime = 'nodejs'`, `dynamic = 'force-dynamic'`.
- **`src/schemas/report.schema.ts`** — Zod schemas:
  - `reportsQuerySchema` — `{ startDate: string (YYYY-MM-DD, required), endDate: string (YYYY-MM-DD, required), serviceId?: string.cuid(), counterId?: string.cuid() }`. Validates that `endDate >= startDate` and that the range is not more than 90 days (defensive — large ranges are slow).
  - `reportDateRangeSchema` — for the date range picker form.
  - `serviceFilterSchema` — for the service filter dropdown.
  - `counterFilterSchema` — for the counter filter dropdown.
  - `reportExportQuerySchema` — same as `reportsQuerySchema` but also accepts `format: 'csv'` (default `'csv'`).
- **`ReportData` type** at `src/types/report.types.ts`:
  - `kpi: ReportKpiSummary` — the 4 KPI values.
  - `services: ServicePerformanceRow[]` — per-service breakdown.
  - `counters: CounterPerformanceRow[]` — per-counter breakdown.

### 6.5 Report Calculation Methodology Discipline

- **Average wait time:** arithmetic mean of `(calledAt - issuedAt) / 60000` across all `Ticket` records for the service where `calledAt IS NOT NULL` and the ticket's `businessDate` is in the range. Excludes no-show'd and cancelled tickets. Excludes tickets still waiting at the time of the query. Result is a `number` in minutes (Float). Returns `null` if no called tickets exist.
- **Average service time:** arithmetic mean of `(completedAt - servedAt) / 60000` across all `Ticket` records where both `completedAt` and `servedAt` are non-null and the ticket's `businessDate` is in the range. Returns `null` if no completed tickets exist. **Note: in Phase 5.1, the `COMPLETED` and `SERVING` ticket state transitions are NOT yet implemented (deferred from Phase 2.3). Therefore, `averageServiceMinutes` will be `null` for all services until those transitions are implemented in a future phase.** The calculation logic IS in place; the data is missing.
- **Peak hour:** hour (0-23 in `APP_TIMEZONE`) with the highest count of `TicketEvent` records where `eventType = 'CALLED'` for the service on the date. Returns `null` if no `CALLED` events exist or if the total is below `BUSIEST_HOUR_THRESHOLD` (5 calls). Bucket by `eventTime` (which is in UTC) converted to `APP_TIMEZONE` hour.
- **No-show rate:** `totalNoShow / totalIssued` (Float, 0-1). Returns `0` if `totalIssued = 0`. The dashboard displays as a percentage.
- **Busiest hour:** the peak hour (0-23). The dashboard displays as "HH:00 - HH:59" range. Returns `null` if no calls.
- **In-progress ticket exclusion:** tickets in `WAITING`, `CALLED`, `RECALLED`, or `SERVING` status at the time of the query are NOT counted in `totalServed`, `averageWaitMinutes`, or `averageServiceMinutes`. They are counted in `totalIssued` (they were issued). They may be counted in `totalWaiting` (for the daily snapshot) but not for live queries.
- **Date range boundary:** the `startDate` is inclusive (00:00:00.000 in `APP_TIMEZONE`), the `endDate` is inclusive (23:59:59.999 in `APP_TIMEZONE`). Conversion to UTC is done at the database boundary.
- **Timezone discipline:** all hour bucketing uses `APP_TIMEZONE` (per the master plan Section 17). The Prisma `businessDate` field is a UTC `Date`; the conversion to `APP_TIMEZONE` hour is done in `calculatePeakHour`.

### 6.6 Incremental Metric Recording Discipline (NEW — Phase 5.1 specific)

The in-memory `serviceDailyCounters` map tracks per-service counters as tickets progress through their lifecycle:

- **Storage:** `Map<string, ServiceDailyCounters>` at module scope in `src/lib/analytics-service.ts`. The key is `serviceId`. The value is the `ServiceDailyCounters` object.
- **Lifecycle:**
  - On `ISSUED`: `totalIssued += 1`, `hourlyIssued[hour] += 1`.
  - On `CALLED`: `waitTimes.push((calledAt - issuedAt) / 60000)`. (Recall does NOT push — the ticket was already counted.)
  - On `NO_SHOW`: `totalNoShow += 1`.
  - On `SERVED` / `COMPLETED` / `CANCELLED`: increment the corresponding counter (future, when these transitions are implemented).
- **Flush:** at daily reset (Phase 2.3.3 extended by 5.1.1), the `serviceDailyCounters` map is flushed to the `QueueDailySnapshot` records. The map is then cleared (in-memory).
- **Best-effort:** all `incrementServiceCounter` calls are wrapped in try/catch — failures logged via `console.error` but never propagated. A failure in the analytics tracking does NOT fail the ticket operation.
- **Module-scope singleton:** the map survives across requests within a single Node.js process. The `globalThis.__serviceDailyCounters` pattern (similar to the SSE manager singleton from 3.1) is used to survive Next.js dev hot-reload.
- **Initialisation:** the map is empty on startup. The first request that triggers an increment creates the entry. The daily reset creates entries for all services (so they exist in memory even if no tickets were issued).
- **Manual reset behaviour:** a manual reset via `/api/admin/reset-queue` clears the map and re-initialises entries for all services.
- **Process restart behaviour:** on process restart (e.g., server reboot), the map is empty. The `QueueDailySnapshot` is still populated correctly by the daily reset (because the reset uses database queries, not the in-memory map, for the persisted metrics). The in-memory map is a real-time tracker; the persisted snapshot is the authoritative record.

### 6.7 CSV Format Discipline

- **Encoding:** UTF-8 with no BOM (browsers handle UTF-8 transparently).
- **Line endings:** `\r\n` (CRLF, per RFC 4180).
- **Header row:** included (the first row is the column names).
- **Quoting:** fields containing commas, double quotes, or newlines are wrapped in double quotes. Internal double quotes are doubled (e.g., `She said "hi"` becomes `"She said ""hi"""`).
- **Date format:** ISO 8601 date (`YYYY-MM-DD`) for the `Date` column. Floats formatted to 1 decimal place. Null values rendered as empty strings.
- **Row count:** one row per service per day in the date range. If the range spans 7 days and there are 3 services, the CSV has 21 data rows (plus header).
- **Filename:** `report-YYYY-MM-DD-YYYY-MM-DD.csv` (or `report-YYYY-MM-DD.csv` if start === end). The format is `formatReportFilename(startDate, endDate)`.
- **Empty data:** if the range has no data, the CSV has just the header row (no data rows). The dashboard shows a toast or message before download in this case.

- **Follow the order: 5.1.1 → 5.1.2 → 5.1.3.** Build the data layer first, then the presentation, then the export.
- **Use the existing patterns.** The codebase has established patterns for: API routes, Zod schemas, audit log writes, permission guards, transaction boundaries, response envelopes, component composition. Match these patterns.
- **Verify each step before moving to the next.** Run `yarn build`, `yarn lint`, `yarn type-check` after major changes.

### 8.2 Specific Guidance for 5.1.1

- **Extending the daily reset is additive.** Do NOT modify the existing `runDailyReset` logic from Phase 2.3.3. Add new field calculations AFTER the existing snapshot population.
- **The in-memory counter map is module-scope.** Use the `globalThis.__serviceDailyCounters` pattern (like the SSE manager) to survive Next.js hot-reload.
- **The `getReportData` function is the single entry point for the endpoint.** It calls the four sub-calculations in parallel via `Promise.all`.
- **The `ReportData` type is the contract between 5.1.1, 5.1.2, and 5.1.3.** All three documents must agree on the type shape. The type is defined in `src/types/report.types.ts` and re-exported by 5.1.1.
- **Date range validation is in the Zod schema.** The schema enforces `endDate >= startDate` and `endDate - startDate <= 90 days`. The route handler does NOT need to re-validate.

### 8.3 Specific Guidance for 5.1.2

- **The dashboard is a composition of pre-existing patterns.** The `DataTable` component is reused. The shadcn `Card`, `Skeleton`, `Select` components are reused. The Recharts `BarChart` is new — install it via `yarn add recharts` if not already present.
- **The filter state is local to the client component.** No URL params, no cookies. The filter state resets on page reload (a future enhancement could persist via URL params).
- **The skeleton loading state is shadcn `Skeleton`** (not a custom loading component).
- **The empty state is a simple message + icon.** No custom empty state component.
- **The sidebar nav update is a single line.** Add a new `<NavItem>` to the sidebar with the `BarChart3` icon and `/reports` link.

### 8.4 Specific Guidance for 5.1.3

- **The CSV is generated server-side.** The client does NOT construct the CSV.
- **The export endpoint returns a `NextResponse` with the CSV body.** Use `new NextResponse(csvString, { headers: { 'Content-Type': 'text/csv', 'Content-Disposition': 'attachment; filename="..."' } })`.
- **The "Export CSV" button uses an anchor element with the `download` attribute.** Do NOT use `fetch` + `Blob` + `URL.createObjectURL` — the anchor approach is simpler and more reliable.
- **The future scheduled reports interface is exported but not consumed.** The functions are in `lib/report-export.ts` but no code calls them. They are placeholders for the future scheduler.
- **CSV escaping is the responsibility of `escapeCsvField`.** Every field that goes into the CSV must be escaped.

### 8.5 Stub Recommendations

- **For 5.1.1:** Start by creating the `ReportData` type, the `getReportData` function, and the Zod schemas. These are the contract for 5.1.2 and 5.1.3 — having them in place first allows parallel development.
- **For 5.1.2:** Start by creating the page and the client component with hard-coded initial data. Then build the sub-components. Finally, wire the filter changes to the fetch.
- **For 5.1.3:** Start by creating `lib/report-export.ts` with the pure CSV generation functions. These are testable in isolation. Then create the export endpoint. Finally, add the button to the dashboard.

### 8.6 Common Pitfalls

- **Time zone bugs.** The `businessDate` field is UTC. The hour bucketing uses `APP_TIMEZONE`. Convert at the boundary; do NOT mix UTC and `APP_TIMEZONE` in the same query.
- **N+1 queries in the report calculations.** Use `Promise.all` to parallelise the four sub-calculations. Use Prisma's `include` and `select` to fetch only the fields needed.
- **Float formatting.** JavaScript floats can have many decimal places (`0.30000000000000004`). Format to 1 decimal place for display (`0.3`).
- **CSV escaping.** A comma in a service name (`Smith, Jones & Co.`) must be escaped. A double quote in a message (`She said "hi"`) must be escaped. Test with realistic data.
- **Rate of `REPORT_GENERATED` audit writes.** Every fetch of `GET /api/reports` writes an audit entry. If the dashboard is high-frequency, the audit log fills up. 5.1.2 is on-demand (user-triggered), so this is acceptable. If a future enhancement adds polling, debounce or batch the audit writes.
- **Missing `runtime` and `dynamic` exports.** Every API route that uses Prisma or `auth()` MUST export `runtime = 'nodejs'` and `dynamic = 'force-dynamic'`. Without these, the build fails or the route hangs.
- **Permission string typos.** `PERMISSION_REPORT_VIEW` and `PERMISSION_REPORT_EXPORT` are defined in master plan Section 10.4. The constants in `src/lib/permissions.ts` are the source of truth — import them, do NOT hard-code the strings.
