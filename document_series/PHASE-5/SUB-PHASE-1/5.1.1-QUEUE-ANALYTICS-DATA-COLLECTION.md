## 1. Document Purpose

This document delivers the **data collection layer** for the analytics and reporting sub-phase. It extends the existing daily reset mechanism (Phase 2.3.3) to populate ALL fields of the `QueueDailySnapshot` table per master plan Section 8.2, adds incremental metric recording during the day so the dashboard reflects near-real-time data, implements the centralised analytics calculation module in `src/lib/analytics-service.ts`, and exposes the metrics via the `GET /api/reports` endpoint.

This document is the **first document in Sub-Phase 5.1**. Documents 5.1.2 (reports dashboard) and 5.1.3 (data export) both consume the `getReportData` function and the `ReportData` type defined here. This document is the data backend — it does not implement any user-facing UI.

The strategic value of this document is that it **transforms the system from a black box to an observable system**. Before 5.1.1, the `QueueDailySnapshot` table was populated with a subset of the master plan's metrics (some fields were always 0 or `null` because the relevant ticket state transitions were not yet implemented). After 5.1.1, the snapshot is the authoritative historical record of the queue's daily performance, and the `/api/reports` endpoint serves live queries for any date range, service, and counter combination.

When this document is complete:

- After a business day with multiple tickets, the `QueueDailySnapshot` contains correct values for `totalIssued`, `totalServed`, `totalNoShow`, `totalCancelled`, `totalWaiting`, `averageWaitMinutes`, `averageServiceMinutes`, and `peakHour`.
- The in-memory `serviceDailyCounters` map is kept in sync with the database as tickets are issued, called, and no-show'd during the day.
- The `lib/analytics-service.ts` module exposes pure calculation functions that compute metrics for any date range.
- The `GET /api/reports` endpoint accepts `startDate`, `endDate`, `serviceId?`, `counterId?` and returns the full `ReportData` payload.
- The `REPORT_GENERATED` audit log entry is written for every report fetch.
- The `report:view` permission is enforced at the endpoint.

This document does **not** implement the dashboard UI (owned by 5.1.2), the CSV export (owned by 5.1.3), the rate limiting (owned by 5.2.1), the caching (owned by 5.2.2), or the audit log viewer (owned by 5.2.3).

**Specific instructions:**

Create the file with the following exports:

- **`interface ReportData`** — the full report payload returned by `getReportData` and the `GET /api/reports` endpoint:
  - `kpi: ReportKpiSummary`.
  - `services: ServicePerformanceRow[]`.
  - `counters: CounterPerformanceRow[]`.
  - `hourly: HourlyTicketCount[]` — always 24 entries (one per hour), even if the count is 0.
  - `startDate: string` — ISO 8601 date (`YYYY-MM-DD`).
  - `endDate: string` — ISO 8601 date.
  - `serviceId: string | null`.
  - `counterId: string | null`.
  - `generatedAt: string` — ISO 8601 datetime.
- **`interface ReportKpiSummary`** — the 4 KPI values:
  - `totalTickets: number`.
  - `averageWaitMinutes: number | null`.
  - `noShowRate: number` — Float, 0-1 (NOT a percentage). The dashboard multiplies by 100 for display.
  - `busiestHour: number | null` — 0-23 in `APP_TIMEZONE`. `null` if no calls.
- **`interface ServicePerformanceRow`** — one row per service:
  - `serviceId: string`.
  - `serviceCode: string`.
  - `serviceName: string`.
  - `totalIssued: number`.
  - `totalServed: number`.
  - `totalNoShow: number`.
  - `noShowRate: number`.
  - `averageWaitMinutes: number | null`.
  - `averageServiceMinutes: number | null`.
  - `peakHour: number | null`.
- **`interface CounterPerformanceRow`** — one row per counter:
  - `counterId: string`.
  - `counterName: string`.
  - `counterNumber: number`.
  - `totalServed: number` — tickets that were called at this counter.
  - `totalNoShow: number` — tickets that were no-show'd at this counter.
  - `averageServiceMinutes: number | null`.
  - `closureEvents: number` — count of `COUNTER_CLOSED` events in the date range.
- **`interface HourlyTicketCount`** — one bucket per hour:
  - `hour: number` — 0-23 in `APP_TIMEZONE`.
  - `count: number` — tickets issued in this hour.
- **`interface ServiceDailyCounters`** — the in-memory counter map values:
  - `totalIssued: number`.
  - `totalServed: number` (always 0 until Phase 2 implements the transition).
  - `totalNoShow: number`.
  - `totalCancelled: number` (always 0 until Phase 2 implements the transition).
  - `waitTimes: number[]` — array of wait times in minutes for `CALLED` events.
  - `serviceTimes: number[]` (always empty until Phase 2 implements the transition).
  - `hourlyIssued: Record<number, number>` — 24 buckets (0-23).
- **`const BUSINESS_DATE_RANGE_MAX_DAYS: 90`** — the maximum date range allowed (per the Zod schema below).

### Step 3: Create `src/lib/analytics-service.ts` — Analytics Module

**Goal:** Provide the centralised analytics calculation module with pure functions and the in-memory counter map.

**Files affected:** `src/lib/analytics-service.ts` (new file).

**Specific instructions:**

Create the file with:

- **Imports:** `db` from `@/lib/db`, `APP_TIMEZONE` from `@/lib/env` (or similar). The `ServiceDailyCounters`, `ReportData`, etc. types from `@/types/report.types`.
- **Module-scope state:**
  - `declare global { var __serviceDailyCounters: Map<string, ServiceDailyCounters> | undefined; }`.
  - `const serviceDailyCounters: Map<string, ServiceDailyCounters> = globalThis.__serviceDailyCounters ?? (globalThis.__serviceDailyCounters = new Map())`. The `globalThis` pattern ensures survival across Next.js dev hot-reload.
- **Module-scope constants:**
  - `BUSIEST_HOUR_THRESHOLD = 5`.
  - `HOURS_IN_DAY = 24`.
  - `MS_PER_MINUTE = 60_000`.
- **`function getOrCreateServiceCounters(serviceId: string): ServiceDailyCounters`** — internal helper. Gets the counters for a service or creates a fresh entry if absent. Used by the increment functions.
- **`async function calculateServiceDailyMetrics(serviceId: string, businessDate: Date): Promise<ServiceDailyMetrics>`** — calculates the 8 metrics for a single service on a single business date. Queries:
  - `db.ticket.findMany({ where: { serviceId, businessDate }, select: { status, issuedAt, calledAt, servedAt, completedAt } })`.
  - `db.ticketEvent.count({ where: { ticket: { serviceId, businessDate }, eventType: 'CALLED' } })` (for the wait time count).
  - `db.ticketEvent.findMany({ where: { ticket: { serviceId, businessDate }, eventType: 'CALLED' }, select: { eventTime: true } })` (for peak hour).
  - Computes the metrics in JavaScript (filter, reduce, mean). Returns `{ totalIssued, totalServed, totalNoShow, totalCancelled, totalWaiting, averageWaitMinutes, averageServiceMinutes, peakHour }`.
- **`async function calculateServicePerformanceRows(startDate: Date, endDate: Date, serviceIds?: string[]): Promise<ServicePerformanceRow[]>`** — returns one row per service for the date range. Iterates over the services (filtered by `serviceIds` if provided), calls `calculateServiceDailyMetrics` for each day in the range, aggregates the results across days. Returns the rows.
- **`async function calculateCounterPerformanceRows(startDate: Date, endDate: Date, counterIds?: string[]): Promise<CounterPerformanceRow[]>`** — returns one row per counter for the date range. For each counter (filtered by `counterIds` if provided):
  - Queries `db.ticketEvent.count({ where: { eventType: 'CALLED', ticket: { counterId }, eventTime: { gte: startDate, lte: endDate } } })` for `totalServed`.
  - Queries `db.ticketEvent.count({ where: { eventType: 'NO_SHOW', ticket: { counterId }, eventTime: { gte: startDate, lte: endDate } } })` for `totalNoShow`.
  - Queries `db.ticket.findMany({ where: { counterId, completedAt: { not: null }, businessDate: { gte: startDate, lte: endDate } }, select: { servedAt: true, completedAt: true } })` for average service time.
  - Queries `db.counterStatusEvent.count({ where: { counterId, status: 'CLOSED', createdAt: { gte: startDate, lte: endDate } } })` for `closureEvents`.
- **`async function calculateTicketsByHour(startDate: Date, endDate: Date, serviceId?: string): Promise<HourlyTicketCount[]>`** — returns 24 buckets. Queries `db.ticket.findMany({ where: { businessDate: { gte: startDate, lte: endDate }, ...(serviceId && { serviceId }) }, select: { issuedAt: true } })`. For each ticket, converts `issuedAt` to `APP_TIMEZONE` hour and increments the bucket. Returns the array of 24 `{ hour, count }` entries (always 24 entries, even if count is 0).
- **`async function calculateReportKpiSummary(startDate: Date, endDate: Date, serviceId?: string, counterId?: string): Promise<ReportKpiSummary>`** — returns the 4 KPI values. Queries:
  - `db.ticket.count({ where: { businessDate: { gte: startDate, lte: endDate }, ...(serviceId && { serviceId }) } })` for `totalTickets`.
  - `db.ticket.findMany({ where: { businessDate: { gte: startDate, lte: endDate }, calledAt: { not: null }, ...(serviceId && { serviceId }) }, select: { issuedAt: true, calledAt: true } })` for average wait time.
  - `db.ticket.count({ where: { businessDate: { gte: startDate, lte: endDate }, status: 'NO_SHOW', ...(serviceId && { serviceId }) } })` for no-show rate.
  - `db.ticketEvent.findMany({ where: { eventType: 'CALLED', eventTime: { gte: startDate, lte: endDate }, ...(serviceId && { ticket: { serviceId } }) }, select: { eventTime: true } })` for busiest hour (via `calculatePeakHour`).
- **`function calculatePeakHour(calledEventTimes: Date[], appTimezone: string): number | null`** — pure function. Buckets the event times by `APP_TIMEZONE` hour, returns the hour with the highest count. Returns `null` if total is below `BUSIEST_HOUR_THRESHOLD` or if `calledEventTimes` is empty. **Important:** `APP_TIMEZONE` is passed in (not read from env) to keep the function pure and testable.
- **`async function getCurrentServiceCounters(serviceId: string, businessDate: Date): Promise<ServiceDailyCounters>`** — returns the in-memory counters for a service on the current business date. If the service has no entry, returns a fresh entry (initialised to zeros).
- **`function incrementServiceCounter(serviceId: string, eventType: 'ISSUED' | 'CALLED' | 'NO_SHOW'): void`** — updates the in-memory counters. Best-effort: wraps the body in try/catch, logs failures via `console.error`, never throws. The function is intentionally synchronous (the in-memory map is in-process).
  - For `ISSUED`: increments `totalIssued` and the current hour's `hourlyIssued` bucket.
  - For `CALLED`: pushes the wait time (in minutes) to `waitTimes` if both `issuedAt` and `calledAt` are provided. (Defensive: if the event metadata doesn't include the timestamps, skip the push.)
  - For `NO_SHOW`: increments `totalNoShow`.
- **`function resetServiceCountersForDate(businessDate: Date): void`** — clears the in-memory map and re-initialises entries for all services (queried from the database). Called by the daily reset and the manual reset. Idempotent.
- **`async function getReportData(startDate: Date, endDate: Date, serviceId?: string, counterId?: string): Promise<ReportData>`** — the main entry point for the endpoint. Calls the four sub-calculations in parallel via `Promise.all`:
  - `const [kpi, services, counters, hourly] = await Promise.all([...])`.
  - Returns `{ kpi, services, counters, hourly, startDate: formatDate(startDate), endDate: formatDate(endDate), serviceId: serviceId ?? null, counterId: counterId ?? null, generatedAt: new Date().toISOString() }`.
- **`function formatDate(date: Date): string`** — internal helper. Returns `YYYY-MM-DD` in `APP_TIMEZONE`.

**Important notes:**

- All `db.*` calls are READ-ONLY. No transactions, no writes (except the `incrementServiceCounter` which is in-memory only).
- The function signatures match the type definitions in Step 1.
- The module exports the functions for use by the endpoint and the daily reset extension.

## 8. Common Pitfalls

- **Time zone bugs.** The `businessDate` field is UTC. The hour bucketing uses `APP_TIMEZONE`. The `calculatePeakHour` function must convert UTC `eventTime` to `APP_TIMEZONE` hour. Use `Intl.DateTimeFormat` with `timeZone: APP_TIMEZONE` and `hour12: false` to extract the hour. Do NOT use `getUTCHours()` — that returns UTC hour, not `APP_TIMEZONE` hour.
- **N+1 queries.** The per-service and per-counter calculations iterate over multiple records. For 10 services over 30 days, that's 300 calls to `calculateServiceDailyMetrics`. Consider batching with a single query that returns all the data, then aggregating in JavaScript. The current per-service approach is acceptable for small datasets but may need optimisation for 5.2.2.
- **In-memory map survives hot reload but not process restart.** If the server restarts, the map is empty. The `QueueDailySnapshot` is still correct (the daily reset queries the database, not the map). The map is a real-time tracker; the snapshot is the authoritative record.
- **Float formatting.** `0.30000000000000004` is a valid JavaScript result. Format to 1 decimal place at the boundary (in the calculation functions, not in the database).
- **Missing `runtime` and `dynamic` exports.** Every API route that uses Prisma or `auth()` MUST export both. Without them, the build fails or the route hangs.
- **Permission string typos.** Import `PERMISSION_REPORT_VIEW` from `@/lib/permissions`. Do NOT hard-code `'report:view'`.
- **Increment failures must NOT fail the ticket operation.** All `incrementServiceCounter` calls are wrapped in try/catch by the function itself (best-effort). Do NOT add additional error handling at the call site.
- **Range too large.** 90 days is the maximum. A range of 91 days returns 422. If a user needs more data, they must paginate manually (multiple requests with shorter ranges).
- **Cross-day tickets.** A ticket issued at 23:59:59 on day 1 and called at 00:00:01 on day 2 has its `businessDate` set to day 1 (at issuance time). The `calledAt` is on day 2. The wait time is still `(calledAt - issuedAt) / 60000` — it crosses day boundaries but is correctly attributed to day 1's `businessDate`.
