## 1. Document Purpose

This document delivers the **export layer** for the analytics and reporting sub-phase. It introduces the `GET /api/reports/export` endpoint that returns a CSV file, the `src/lib/report-export.ts` utility module that generates the CSV content, the "Export CSV" button on the reports dashboard, and the interface for future scheduled email reports.

This document is the **third and final document in Sub-Phase 5.1**. It consumes the `getReportData` function and the `ReportData` type from 5.1.1, and it adds its button into the dashboard layout created by 5.1.2. The export is the third pillar of the analytics layer: **collection** (5.1.1) → **presentation** (5.1.2) → **export** (5.1.3).

The strategic value of this document is that it **lets administrators take the data out of the system**. A CSV download is a familiar, universal format — it opens in Excel, Google Sheets, or any text editor. Administrators can use it for ad-hoc analysis, monthly reports, or sharing with stakeholders who don't have system access. The future scheduled reports interface (defined in this document but NOT implemented) sets the architectural foundation for automated email delivery.

When this document is complete:

- An administrator can click the "Export CSV" button on the reports dashboard.
- A CSV file downloads with the name `report-YYYY-MM-DD-YYYY-MM-DD.csv`.
- The CSV has the columns: `Date`, `Service Code`, `Service Name`, `Total Issued`, `Total Served`, `Total No Show`, `Total Cancelled`, `Total Waiting`, `Average Wait (min)`, `Average Service (min)`, `Peak Hour`.
- One row per service per day in the date range.
- Fields with commas, quotes, or newlines are properly escaped.
- The export endpoint enforces the `report:export` permission (stricter than `report:view`).
- Every export writes a `REPORT_EXPORTED` audit log entry.
- The `buildScheduledReportJob` and `getScheduledReportRecipients` functions exist in `lib/report-export.ts` as the interface for a future scheduler.

This document does **not** implement the data collection (owned by 5.1.1), the dashboard UI (owned by 5.1.2), the actual scheduler (the interface is defined but the scheduler is NOT implemented — it's a future enhancement), the rate limiting (owned by 5.2.1), the audit log viewer (owned by 5.2.3), or any other export format (PDF, Excel are future enhancements).

### Step 1: Create `src/lib/report-export.ts` — CSV and Scheduled Reports Utilities

**Goal:** Provide the pure CSV generation functions and the future scheduled reports interface.

**Files affected:** `src/lib/report-export.ts` (new file).

**Specific instructions:**

Create the file with the following exports:

- **`function escapeCsvField(value: unknown): string`** — pure function. Escapes a single field:
  - If `value === null || value === undefined`: returns `''` (empty string).
  - If `value` is a `number` and `Number.isFinite(value)`: returns `String(value)`.
  - If `value` is a `boolean`: returns `value ? 'true' : 'false'`.
  - Otherwise, converts to `String(value)`.
  - If the string contains a comma, double quote, newline (`\n` or `\r`), or carriage return: wraps in double quotes.
  - Internal double quotes are doubled (e.g., `She said "hi"` → `"She said ""hi"""`).
  - Returns the escaped string.
- **`function generateReportCsv(reportData: ReportData): string`** — pure function. Produces the CSV content:
  - Builds the header row: `Date,Service Code,Service Name,Total Issued,Total Served,Total No Show,Total Cancelled,Total Waiting,Average Wait (min),Average Service (min),Peak Hour`.
  - Joins the header with `\r\n`.
  - For each service in `reportData.services` AND for each day in the range (`startDate` to `endDate`):
    - The `Date` column is the day in `YYYY-MM-DD` format.
    - The `Service Code` and `Service Name` are from the service row.
    - The numeric metrics are formatted: integers as-is, floats to 1 decimal place, nulls as empty.
    - The `Peak Hour` is `null` → empty, otherwise `String(hour).padStart(2, '0') + ':00'`.
    - All values are passed through `escapeCsvField`.
  - Joins the rows with `\r\n`.
  - Returns the complete CSV string (with a trailing `\r\n` if there are data rows).
  - **Important:** the current `ReportData.services` is an aggregated row per service for the date range (not per service per day). The CSV needs per-service-per-day rows. To achieve this, 5.1.3 either:
    - (a) Fetches per-day data by calling `getReportData` for each day in the range (multiple calls), OR
    - (b) Extends `getReportData` to return per-day rows (the cleanest approach).
  - **Recommended approach (b):** modify 5.1.1's `calculateServicePerformanceRows` to also accept a `byDay: boolean` option. When `byDay === true`, returns one row per service per day. The export calls `getReportData` with `byDay: true`. **This is the cleanest separation.** However, modifying 5.1.1 from 5.1.3 is a dependency from 5.1.3 to 5.1.1 — 5.1.1 should be modified BEFORE 5.1.3 starts. The 5.1.1 document does NOT include this option; 5.1.3 adds it via a minor extension. See the **Update 5.1.1** sub-step below.
- **`function formatReportFilename(startDate: string, endDate: string): string`** — pure function. Returns:
  - If `startDate === endDate`: `report-YYYY-MM-DD.csv`.
  - Otherwise: `report-YYYY-MM-DD-YYYY-MM-DD.csv`.
- **`async function getScheduledReportRecipients(): Promise<string[]>`** — reads `SystemSetting` key `reports.scheduled_recipients`:
  - `db.systemSetting.findUnique({ where: { key: 'reports.scheduled_recipients' } })`.
  - If not found or `value` is empty: returns `[]`.
  - Otherwise: parses the value as a comma-separated string, trims each entry, filters out empty strings. Returns the array.
- **`async function getScheduledReportFrequency(): Promise<'daily' | 'weekly' | 'monthly' | null>`** — reads `SystemSetting` key `reports.scheduled_frequency`:
  - `db.systemSetting.findUnique({ where: { key: 'reports.scheduled_frequency' } })`.
  - If not found or value is not one of `'daily' | 'weekly' | 'monthly'`: returns `null`.
  - Otherwise: returns the value.
- **`async function buildScheduledReportJob(options: ScheduledReportJobOptions): Promise<ScheduledReportJob>`** — builds a job description object:
  - `options.recipients` defaults to the result of `getScheduledReportRecipients()`.
  - `options.frequency` defaults to the result of `getScheduledReportFrequency()`.
  - Generates a unique job ID (use `crypto.randomUUID()` or a cuid).
  - Sets `status: 'PENDING'`, `scheduledAt: new Date()`.
  - Returns the `ScheduledReportJob` object.
  - **The function does NOT execute the job.** It builds the description. A future scheduler will pick up `PENDING` jobs and execute them (email delivery).
- **Type definitions** (in this file or imported from `src/types/report.types.ts`):
  - `interface ScheduledReportJobOptions { startDate: Date; endDate: Date; serviceId?: string; counterId?: string; recipients?: string[]; frequency?: 'daily' | 'weekly' | 'monthly' | null }`.
  - `interface ScheduledReportJob { id: string; query: { startDate: string; endDate: string; serviceId: string | null; counterId: string | null }; recipients: string[]; frequency: 'daily' | 'weekly' | 'monthly' | null; scheduledAt: string; status: 'PENDING' | 'EXECUTING' | 'COMPLETED' | 'FAILED' }`.
- **Imports:** `db` from `@/lib/db`, `ReportData` from `@/types/report.types`.

**Update 5.1.1 (minor extension):**

If `calculateServicePerformanceRows` does not support per-day aggregation, the agent extends it. The extension is OPTIONAL — the alternative is to call `getReportData` once per day. The extension is cleaner. Modify `src/lib/analytics-service.ts`:

- Add an optional parameter `byDay?: boolean = false` to `calculateServicePerformanceRows`.
- When `byDay === true`, the function iterates over each day in the range and calls `calculateServiceDailyMetrics` for each (service, day) combination. Returns one row per service per day.
- Update `getReportData` to accept a `byDay` option and pass it through to `calculateServicePerformanceRows`.
- The default behaviour (`byDay: false`) is unchanged — the dashboard still receives aggregated rows.

This minor extension to 5.1.1 is acceptable. The 5.1.1 document does not explicitly cover it, but the extension is backward-compatible.

### Step 3: Create `src/app/api/reports/export/route.ts` — The Export Endpoint

**Goal:** Expose the CSV export via HTTP.

**Files affected:** `src/app/api/reports/export/route.ts` (new file).

**Specific instructions:**

Create the file with:

- **Route configuration exports:**
  - `export const runtime = 'nodejs'`.
  - `export const dynamic = 'force-dynamic'`.
- **`async function GET(request: Request)`** (default export):
  - **Step 1 — Guard:** `const handler = withPermission(async (request, { session }) => { ... }, PERMISSION_REPORT_EXPORT)`. The handler receives the authenticated session.
  - **Step 2 — Parse query:** parse the URL search params with `reportExportQuerySchema`. On validation failure → return JSON error with HTTP 422.
  - **Step 3 — Compute dates:** `const startDate = new Date(query.startDate + 'T00:00:00.000Z')`, `const endDate = new Date(query.endDate + 'T23:59:59.999Z')`.
  - **Step 4 — Fetch data:** `const startTime = Date.now(); const data = await getReportData(startDate, endDate, query.serviceId, query.counterId, { byDay: true })`. The `byDay: true` flag ensures per-service-per-day rows. (If 5.1.1 was not extended, omit this option and fetch per-day in 5.1.3.)
  - **Step 5 — Generate CSV:** `const csv = generateReportCsv(data)`. Pure function, fast.
  - **Step 6 — Build filename:** `const filename = formatReportFilename(query.startDate, query.endDate)`.
  - **Step 7 — Audit log:** `writeAuditLog({ action: 'REPORT_EXPORTED', actorId: session.user.userId, entity: 'Report', entityId: null, description: 'Report exported as CSV', metadata: { startDate: query.startDate, endDate: query.endDate, serviceId: query.serviceId ?? null, counterId: query.counterId ?? null, format: query.format, rowCount: <count of data rows>, fileSizeBytes: Buffer.byteLength(csv, 'utf8'), durationMs: Date.now() - startTime } })`. Best-effort — wrap in try/catch.
  - **Step 8 — Return:** `return new NextResponse(csv, { status: 200, headers: { 'Content-Type': 'text/csv; charset=utf-8', 'Content-Disposition': `attachment; filename="${filename}"`, 'Cache-Control': 'no-store' } })`. The `Content-Disposition` header triggers a browser download with the specified filename.

**Important:** The response is a raw CSV string, NOT a JSON envelope. This is intentional — the browser expects the CSV content directly when triggered by the `<a download>` click.

## 8. Common Pitfalls

- **Response envelope.** The export endpoint returns a RAW CSV string, NOT a JSON envelope. The browser expects raw CSV when triggered by `<a download>`. If you wrap the CSV in `{ success: true, data: csvString }`, the downloaded file will be invalid CSV.
- **CSV escaping.** A service name with a comma (`Smith, Jones & Co.`) MUST be escaped (wrapped in double quotes). A counter name with a double quote (`Counter "A"`) MUST have the internal quotes doubled. Test with realistic data including special characters.
- **UTF-8 encoding.** Ensure the `Content-Type` header includes `charset=utf-8`. The CSV should be UTF-8 without BOM.
- **CRLF line endings.** Use `\r\n` (not `\n`) between rows, per RFC 4180.
- **Permission string typos.** Import `PERMISSION_REPORT_EXPORT` from `@/lib/permissions`. Do NOT hard-code `'report:export'`.
- **Filename collisions.** If two exports have the same date range, they have the same filename. This is acceptable — the browser adds a `(1)` suffix for duplicates. Future enhancement: include a timestamp in the filename.
- **Button slot vs button implementation.** 5.1.2 creates the slot (a comment placeholder). 5.1.3 fills it. If 5.1.3 is implemented before 5.1.2, the import fails. The recommended order is 5.1.2 → 5.1.3.
- **The `byDay` extension to 5.1.1.** If implemented, ensure the dashboard still receives aggregated rows by default (the dashboard does NOT pass `byDay: true`).
- **Audit log failures must NOT fail the export.** The audit log write is best-effort (try/catch around the call). If the audit log fails, the export still succeeds and returns the CSV.
- **Large date ranges.** A 90-day range with 10 services produces 900 CSV rows. This is small. A 365-day range would be 3,650 rows — still manageable. If a user requests a much larger range in the future, consider streaming the response.
- **`Content-Disposition` quoting.** If the filename contains special characters (it shouldn't, but defensively), wrap it in double quotes: `filename="report-2026-06-20.csv"`.
