## 1. Document Purpose

This document delivers the **critical seam realisation** of the Smart Queue Management System. The `broadcastEvent()` function whose signature was locked in Phase 2.1.1 (as a no-op stub) finally receives a real body in 3.1.3. After this document is complete:

- Every call to `broadcastEvent('global', 'TICKET_CALLED', payload)` from any Phase 2 endpoint (2.2.1 ticket issuance, 2.3.1 call/recall, 2.3.2 no-show, 2.3.3 daily reset) actually delivers the event to every connected SSE client within a second.
- The Phase 2 code is **not modified** — the function signature is preserved exactly.
- The `EVENT_ROUTING` constant declares which event types go to which channels, as a single data structure that future agents (Phase 4) can extend.
- The `broadcastRoutedEvent()` convenience wrapper resolves dynamic channel references (e.g., `counter:<counterId>`) and dispatches via the routing table.
- The display board (3.2), audio system (3.3), officer dashboard (4.2.3), and security screen (4.3.3) all become implementable — they simply subscribe via `useSSE()` (from 3.1.2) and the events arrive.

This is the moment when **the queue becomes a live experience** for waiting customers. Until 3.1.3 lands, every event produced by Phase 2 is a phantom — written to the database and logged but never delivered to anyone. After 3.1.3 lands, every event becomes a real, observable, audible, visual event in the waiting room.

This document does **not** build the display board (owned by 3.2), the audio system (owned by 3.3), the officer dashboard (owned by 4.2.3), the FCM dispatch (owned by 4.1), or the broadcast message UI (owned by 4.3). It builds the single function that all of those consumers depend on.

**Goal:** Establish the data structure for which event types go to which channels.

    - `TICKET_COMPLETED` → `['global']` (no source emits this in 3.1; route is defined for forward-compatibility).
    - `BROADCAST_MESSAGE` → `['global', 'security']` (no source emits this in 3.1; route is defined for forward-compatibility with Phase 4.3).
    - `DAILY_RESET` → `['global']`.
    - `COUNTER_OPENED` → `['global']` (no source emits this in 3.1; route is defined for forward-compatibility with Phase 4.2.1).
    - `COUNTER_CLOSED` → `['global']` (same).

- **Verify the table covers every `SseEventType` value.** TypeScript's exhaustiveness check (via the `Record<SseEventType, ...>` type) ensures no event type is missing.

### Step 3: Implement the Real `broadcastEvent()` Function

**Goal:** Replace the no-op stub with the real broadcasting logic.

**Files affected:** `src/lib/events.ts` (modified).

**Specific instructions:**

- **Replace the no-op implementation of `broadcastEvent` with the real one.** The signature is preserved exactly.
- **New implementation:**
  - Signature: `export function broadcastEvent(channel: string | string[], eventType: SseEventType, payload: SseEventPayload[SseEventType]): void`.
  - **Imports:** `sseManager` from `@/lib/sse-manager`. Import `SseChannel`, `SseEventType`, `SseEventPayload` from `@/types/sse.types`.
  - **Body:**
    1. **Normalise the channel input.** If `channel` is a string, wrap it in an array: `const channels = Array.isArray(channel) ? channel : [channel]`.
    2. **Build the envelope** via `buildEnvelope(eventType, payload)`.
    3. **For each channel in `channels`:** call `sseManager.sendToChannel(channel, envelope)`.
       - The manager handles the no-op fallback (empty channel set → silent return) and the SSE wire format serialization.
       - The write is non-blocking; `sendToChannel` returns synchronously after queueing the writes on the event loop.
    4. **Log via `console.debug`** with the event `id`, `type`, the channel(s), and the total client count across all channels:
       - Compute `totalClients = channels.reduce((sum, ch) => sum + sseManager.getChannelClientCount(ch), 0)`.
       - Log: `console.debug('[broadcastEvent]', { id: envelope.id, type: envelope.type, channels, totalClients })`.
       - **NEVER log the payload contents** — PII risk (officer names, ticket numbers, customer-related metadata).
  - **Error handling:** wrap the body in try/catch. On error, log via `console.error` (NOT `console.debug` — failures need visibility) and return. Do NOT throw. Event publishing is best-effort; failures must not crash the calling business logic (the Phase 2 ticket call, recall, no-show, etc. must still succeed even if event delivery fails).
- **Preserve the export shape:** `export function broadcastEvent(...)`. The function is the default export shape used by Phase 2 (`import { broadcastEvent } from '@/lib/events'`).
- **Document** at the top of the function:
  - "The signature is locked by Phase 2.1.1. Do NOT modify the signature."
  - "Phase 2 call sites are unchanged. This document implements the body; Phase 2 code is already calling it correctly."
  - "This function is synchronous and non-throwing. Failures are logged to console.error."

### Step 4: Implement the `broadcastRoutedEvent()` Convenience Wrapper

**Goal:** Provide a higher-level helper that resolves dynamic channel references from the routing table.

**Files affected:** `src/lib/events.ts` (modified).

**Specific instructions:**

- **Add `export function broadcastRoutedEvent(eventType: SseEventType, payload: SseEventPayload[SseEventType], context?: BroadcastContext): void`:**
  - **Imports:** none additional (uses the same imports as `broadcastEvent`).
  - **Body:**
    1. **Look up the routing templates** via `EVENT_ROUTING[eventType]`. If undefined, log via `console.warn` ("Unknown event type in broadcastRoutedEvent, defaulting to no broadcast") and return. (Defensive — the `Record<SseEventType, ...>` type prevents this at compile time, but the runtime check guards against any tampering.)
    2. **Resolve each template** to a concrete channel name via `resolveChannelTemplate(template, context)` (helper below).
    3. **Call `broadcastEvent(resolvedChannels, eventType, payload)`** to do the actual delivery.
- **Add a private helper `resolveChannelTemplate(template: RouteTemplate, context?: BroadcastContext): string | null`:**
  - If `template === 'global'`, returns `'global'`.
  - If `template === 'security'`, returns `'security'`.
  - If `template === 'counter:<counterId>'`:
    - If `context?.counterId` is undefined, log via `console.warn` ("counter event emitted without counterId in context; skipping counter channel") and return `null`.
    - Otherwise, returns `'counter:<counterId>'` (substituting the actual counter ID).
  - Otherwise (unknown pattern), log via `console.warn` ("Unknown channel template") and returns `null`.
  - The `null` return signals that this template should be skipped in the resolved channels list.
- **Filter out `null` resolutions** in the wrapper before calling `broadcastEvent`.

## 8. Common Pitfalls

1. **Modifying the `broadcastEvent` signature.** This is the cardinal sin of this document. Phase 2 code calls `broadcastEvent(channel, eventType, payload)`. Changing the signature breaks every Phase 2 call site. Fix: preserve the signature exactly. Type-narrow the implementation but not the public interface.
2. **Forgetting to wrap in try/catch.** If `sendToChannel` throws (e.g., due to a closed controller), the error propagates to the Phase 2 business logic, breaking the ticket call/recall/no-show. Fix: try/catch around the entire body. Never throw.
3. **Logging payload contents.** PII risk. Fix: log only `id`, `type`, `channels`, `totalClients`. Never the payload.
4. **Forgetting the no-op fallback.** Broadcasting to a channel with no connected clients should be silent. The manager handles this (Step 2 of 3.1.1's sendToChannel). The agent confirms the implementation relies on this — no manual check is needed in `broadcastEvent`.
5. **Adding new `SseEventType` variants.** The discriminated union is frozen in 3.1. If the agent feels tempted to add a variant, that is a signal they are out of scope. New variants come from future sub-phases.
6. **Mutating the routing table at runtime.** The `EVENT_ROUTING` constant is a static, read-only data structure. Phase 4 may add entries (via extending the constant in their respective documents), but no runtime mutation.
7. **Treating `broadcastRoutedEvent` as a replacement for `broadcastEvent`.** It is a higher-level helper. Phase 2 code continues to call `broadcastEvent` directly (their signatures differ — `broadcastRoutedEvent` takes an event type and looks up the channels). The agent confirms Phase 2 call sites are NOT modified.
8. **Not handling the dynamic context.** `broadcastRoutedEvent('COUNTER_STATUS_CHANGED', payload, { counterId: undefined })` must NOT crash — it should log a warning and skip the counter channel. The `resolveChannelTemplate` returns `null` and the wrapper filters out `null`s.
9. **Routing table missing entries.** If a future `SseEventType` is added but not routed, events of that type silently go nowhere. TypeScript's `Record<SseEventType, ...>` enforces coverage at compile time. The agent confirms the table compiles without errors.
10. **Hot-reload loses the manager.** The 3.1.1 manager uses `globalThis.__sseManager` to survive hot-reload. If the agent accidentally removes the persistence, the SSE connections drop on every code change. The agent reads 3.1.1 carefully and does NOT touch the manager.

### 9.1 Critical Implementation Notes

- **The `broadcastEvent` signature is SACRED.** It was locked by Phase 2.1.1. The agent MUST NOT change the parameter order, the parameter types (in TypeScript: `channel: string | string[]`, `eventType: SseEventType` is acceptable tightening but not `eventType: string` if the discriminated union is now available), the return type (`void`), or the export shape (`export function broadcastEvent`).
- **Phase 2 source files MUST NOT be modified.** The seam is preserved by leaving the call sites unchanged. If the agent finds themselves editing `src/lib/ticket-service.ts` or `src/lib/queue-reset.ts` to "fix" a call to `broadcastEvent`, they are out of scope.
- **The function MUST NOT throw.** Event publishing is best-effort. The calling business logic must succeed even if event delivery fails.
- **The function MUST log via `console.debug`, never `console.log`.** Debug-level logging keeps the production console clean.
- **The `EVENT_ROUTING` table MUST cover every `SseEventType`.** TypeScript's exhaustiveness check enforces this. If the table is missing an entry, the build fails. The agent confirms all Phase 2 event types plus all forward-seam types are present.
- **`broadcastRoutedEvent` is a convenience, not a replacement.** Phase 2 continues to call `broadcastEvent`. Future sub-phases that emit events MAY prefer `broadcastRoutedEvent` (because the routing table decides the channels for them). The agent does NOT retrofit Phase 2 code to use `broadcastRoutedEvent`.
- **The payload is the discriminated union variant.** The TypeScript narrowing ensures the right payload shape for each event type. The agent does NOT need to cast.
- **The `<counterId>` placeholder in the routing table is a literal string** (with angle brackets). The `resolveChannelTemplate` function substitutes it at broadcast time.
- **The implementation is synchronous and fast.** No async, no `await`. The actual TCP writes happen on the next event loop tick.

### 9.3 What to Do When Stuck

- If the event does not arrive on the SSE test page, check:
  1. Is the SSE client connected? (`curl -N /api/sse/global` should show the connection).
  2. Does `sseManager.getChannelClientCount('global')` return > 0? (Add a temporary debug endpoint to check.)
  3. Is `broadcastEvent` being called? (Check the `console.debug` log for `[broadcastEvent]`.)
  4. Is the manager's `sendToChannel` writing to the controller? (Add a temporary `console.debug` in `sendToChannel` to trace.)
- If TypeScript complains about `EVENT_ROUTING` missing entries, add the missing entries (every `SseEventType` must be present).
- If `broadcastRoutedEvent` throws because of an unresolved template, check that the `context` parameter includes the required field (e.g., `counterId` for events routed to `counter:<counterId>`).
- If the Phase 2 endpoint call returns success but no event arrives, the issue is in the broadcasting path (manager write, SSE wire format, client parsing). The agent traces through `broadcastEvent → sendToChannel → sendToClient → controller.enqueue`.
