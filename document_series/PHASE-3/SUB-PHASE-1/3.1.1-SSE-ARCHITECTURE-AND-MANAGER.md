## 1. Document Purpose

This document delivers the **server-side plumbing** of the real-time system. It introduces the **SSE manager singleton** at `src/lib/sse-manager.ts` — a module-scope object that holds all active SSE connections keyed by channel name, with methods to register, remove, and broadcast to clients. It introduces the **`/api/sse/[channel]` route handler** that accepts new connections, sets the correct SSE response headers, registers the client with the manager, and tears the connection down on disconnect. It introduces the **heartbeat interval** that sends a comment-type `": heartbeat\n\n"` to every connected client every 30 seconds (preventing proxy/load-balancer idle timeouts). It introduces the **channel auth helper** that enforces per-channel access rules (the `global` channel is public; `counter:[counterId]` requires the officer; `security` requires the security officer role).

When this document is complete:

- A client can `GET /api/sse/global` and receive an open SSE stream.
- The stream sends an initial `event: connected` message, then a `": heartbeat\n\n"` comment every 30 seconds.
- Closing the client connection removes the client from the manager (no leaks).
- Connecting to `/api/sse/counter/<counterId>` as an unauthenticated request is rejected with HTTP 401.
- Connecting to `/api/sse/counter/<counterId>` as the wrong officer is rejected with HTTP 403.
- The manager survives Next.js dev hot-reload via `globalThis` persistence.
- The manager's API surface is ready to be called by 3.1.3's `broadcastEvent()` implementation.

This document does **not** write the real `broadcastEvent()` body (owned by 3.1.3), build the `useSSE()` hook (owned by 3.1.2), implement any display or audio consumer (3.2 and 3.3), or implement FCM (Phase 4.1).

### Step 1: Define the `SSEClient` and `SSEEnvelope` Types

**Goal:** Establish the TypeScript types for the manager's connection record and the on-the-wire event envelope.

**Files affected:** `src/lib/sse-client.ts` (new file).

**Specific instructions:**

Create `src/lib/sse-client.ts` containing:

- **`SSEClient` interface** — the connection record held in the manager:
  - `id: string` — a UUID generated on registration. Used for diagnostics.
  - `channel: string` — the channel name the client is subscribed to (e.g., `'global'`, `'counter:abc123'`).
  - `controller: ReadableStreamDefaultController<Uint8Array>` — the Web Streams API controller used to push data to the client. The TypeScript type is the standard `ReadableStreamDefaultController` from the DOM lib (`lib: ["dom"]` in `tsconfig.json` should already include it; if not, the document specifies the type assertion pattern).
  - `connectedAt: Date` — when the client registered. Used for diagnostics.
  - `userId: string | null` — the session user's ID (null for anonymous connections on the `global` channel). Used for channel auth decisions and logging.
- **`SSEEnvelope` interface** — the envelope shape every event conforms to:
  - `type: string` — the event type (matches an `SseEventType` value from `@/types/sse.types`).
  - `id: string` — a UUID generated at broadcast time.
  - `timestamp: string` — ISO 8601 datetime string in UTC.
  - `payload: Record<string, unknown>` — the event-specific payload (typed via the discriminated union from `@/types/sse.types`).
- **Re-export the `ReadableStreamDefaultController` type** as `SSEController` for convenience (so consumers don't need to import the DOM type directly).
- Document at the top of the file that this is the single source of truth for SSE connection and envelope shapes. Any new connection-related type must be added here.

### Step 2: Create the SSE Manager Singleton

**Goal:** Build the in-process connection manager that holds all active SSE clients and provides the lifecycle methods.

**Files affected:** `src/lib/sse-manager.ts` (new file).

**Specific instructions:**

Create `src/lib/sse-manager.ts` containing:

- **Type definitions (local, not exported):**
  - `ChannelMap = Map<string, Set<SSEClient>>` — the primary data structure, keyed by channel name.
- **The `SseManager` class** (or factory function — the document specifies the chosen pattern; a class is preferred for clarity):
  - **Private state:**
    - `channels: ChannelMap` — the connection map.
    - `heartbeatInterval: NodeJS.Timeout | null` — the active heartbeat handle, or null when no clients are connected.
    - `heartbeatIntervalMs: number` — the heartbeat interval in milliseconds (default 30000).
  - **Constructor:** takes no arguments (or an options object with `heartbeatIntervalMs`). Initialises `channels = new Map()` and `heartbeatInterval = null`.
  - **Method `registerClient(channel: string, client: SSEClient): void`:**
    - If `!this.channels.has(channel)`, `this.channels.set(channel, new Set())`.
    - `this.channels.get(channel)!.add(client)`.
    - Call `this.ensureHeartbeat()` (starts the interval if not running).
    - Log via `console.debug`: `SSE client registered: channel=${channel}, clientId=${client.id}, totalOnChannel=${this.channels.get(channel)!.size}`.
  - **Method `removeClient(client: SSEClient): void`:**
    - If the channel exists in `this.channels`, `this.channels.get(channel)!.delete(client)`.
    - If the channel's set is now empty, `this.channels.delete(channel)`.
    - If `this.channels.size === 0`, call `this.stopHeartbeat()`.
    - Log via `console.debug`: `SSE client removed: channel=${client.channel}, clientId=${client.id}, remainingOnChannel=${this.channels.get(client.channel)?.size ?? 0}`.
  - **Method `sendToClient(client: SSEClient, envelope: SSEEnvelope): void`:**
    - Serialises the envelope to SSE format (the `id`, `event`, `data` lines + terminator — see Step 3 for exact format).
    - Encodes the string to UTF-8 bytes via `TextEncoder`.
    - Calls `client.controller.enqueue(bytes)`.
    - **No-op if the controller is closed** — wrap in try/catch because a closed controller throws on `enqueue`. Log via `console.debug` (not error — this is expected during shutdown).
    - **Note:** the actual SSE format serialization (the wire format) is detailed in 3.1.3's `broadcastEvent()` implementation. This method takes an already-formatted `string` (or `Uint8Array`) parameter, or takes the envelope and serialises it — the document specifies which. The recommended approach: this method takes the envelope and serialises it, so callers don't need to know the wire format.
  - **Method `sendToChannel(channel: string, envelope: SSEEnvelope): void`:**
    - Looks up `this.channels.get(channel)`.
    - If the set is empty or undefined, returns silently (no-op fallback).
    - Iterates over the set, calls `sendToClient` for each. Uses `Array.from(set)` to avoid mutation during iteration (clients may be removed concurrently).
    - Logs via `console.debug`: `SSE broadcast: channel=${channel}, type=${envelope.type}, id=${envelope.id}, clients=${clients.length}`.
  - **Method `getChannelClientCount(channel: string): number`:**
    - Returns `this.channels.get(channel)?.size ?? 0`.
  - **Method `getAllChannels(): string[]`:**
    - Returns `Array.from(this.channels.keys())`.
  - **Method `getStats(): { totalClients: number, channels: Record<string, number> }`:**
    - Returns aggregate stats for diagnostics. Useful for the debug page in 3.1.2.
  - **Private method `ensureHeartbeat(): void`:**
    - If `this.heartbeatInterval` is not null, return.
    - Otherwise, set `this.heartbeatInterval = setInterval(() => this.tickHeartbeat(), this.heartbeatIntervalMs)`.
    - Note: `setInterval` does NOT block the event loop; it's a non-blocking timer.
  - **Private method `tickHeartbeat(): void`:**
    - For every channel, for every client, write the heartbeat comment `: heartbeat\n\n` to the controller.
    - The heartbeat is a **comment** in SSE format (lines starting with `:` are ignored by `EventSource` clients but keep the connection alive at the proxy level).
    - Use `controller.enqueue(textEncoder.encode(': heartbeat\n\n'))`.
    - Wrap each client write in try/catch — if a client write fails, remove the client from the manager.
  - **Private method `stopHeartbeat(): void`:**
    - If `this.heartbeatInterval` is not null, call `clearInterval(this.heartbeatInterval)` and set `this.heartbeatInterval = null`.
- **Singleton instance with hot-reload persistence:**
  - Use `globalThis.__sseManager` to persist the manager across Next.js dev hot-reloads. Without this, every code change in development would tear down all active SSE connections.
  - Pattern:
    - Declare `declare global { var __sseManager: SseManager | undefined }` to add the type to `globalThis`.
    - In the export, check `if (!globalThis.__sseManager) { globalThis.__sseManager = new SseManager() }`.
    - Export the singleton: `export const sseManager = globalThis.__sseManager!`.
- **Default export shape:** the singleton is the default export. Class is named export for testability.
- Document at the top of the file that:
  - This is the **single source of truth** for active SSE connections.
  - The manager is process-local (multi-server requires Redis Pub/Sub — future).
  - Hot-reload persistence is required and must not be removed.

### Step 3: Implement SSE Wire Format Serialization

**Goal:** Provide a small helper that serialises an `SSEEnvelope` to the wire format specified by the SSE standard and Master Plan Section 11.2.

**Files affected:** `src/lib/sse-client.ts` (extended) or a new `src/lib/sse-format.ts`.

**Specific instructions:**

Create a function (either in `sse-client.ts` or a new `sse-format.ts`):

- **`function formatSSEMessage(envelope: SSEEnvelope): Uint8Array`** — returns the encoded bytes for the SSE message.
- The format (exactly, per the SSE spec and Master Plan Section 11.2):
  - `id: <envelope.id>\n`
  - `event: <envelope.type>\n`
  - `data: <JSON.stringify(envelope)>\n`
  - `\n` (blank line terminating the message)
- Multi-line `data` is permitted by the SSE spec (`data:` can be repeated for multi-line content), but our payloads are JSON strings with no newlines, so a single `data:` line is sufficient.
- Encode the resulting string via `TextEncoder().encode(...)` to produce `Uint8Array`.
- **Helper function `formatSSEComment(comment: string): Uint8Array`** — formats an SSE comment (line starting with `:`). Used for the heartbeat.
  - Format: `:<space><comment>\n\n`.
  - For the heartbeat: returns the bytes for `: heartbeat\n\n`.
- Export both functions.

### Step 4: Create the Channel Auth Helper

**Goal:** Provide a single helper that the route handler calls to check whether a connection on a given channel is allowed.

**Files affected:** `src/lib/sse-auth.ts` (new file).

**Specific instructions:**

Create `src/lib/sse-auth.ts` containing:

- **Type definitions:**
  - `interface ChannelAuthResult { authorized: boolean, reason?: string, userId?: string }` — the result of the auth check. `reason` is included on rejection for logging only (not exposed to the client).
- **Function `async function authorizeSseChannel(channel: string, request: NextRequest): Promise<ChannelAuthResult>`**:
  - **Channel pattern parsing:** split the channel string on `:`. The first segment is the channel "kind" (`global`, `counter`, `security`); subsequent segments are the channel "id" (e.g., the counter ID).
  - **`global` channel:** always returns `{ authorized: true, userId: null }` (public).
  - **`counter:<counterId>` channel:**
    - Calls `auth()` to get the session. If no session, returns `{ authorized: false, reason: 'No active session' }`.
    - Queries `db.counterOfficer.findFirst({ where: { userId: session.user.userId, counterId: <counterId> } })`. If no record, returns `{ authorized: false, reason: 'Not assigned to counter' }`.
    - Returns `{ authorized: true, userId: session.user.userId }`. (The `isOnDuty` flag is NOT checked here — the SSE stream is allowed even if the officer is not currently on duty. The officer dashboard may want live updates even when off duty for diagnostic purposes.)
  - **`security` channel:**
    - Calls `auth()` to get the session. If no session, returns `{ authorized: false, reason: 'No active session' }`.
    - Checks whether the session user has the `SECURITY_OFFICER` role. The exact check depends on how roles are stored on the session (from Phase 1.2.1's type augmentation) — the document specifies the exact lookup: `session.user.roles.includes('SECURITY_OFFICER')` or via the `Role` table join.
    - Returns `{ authorized: true, userId: session.user.userId }` if the user has the role; `{ authorized: false, reason: 'Not a security officer' }` otherwise.
  - **Unknown channel patterns:** returns `{ authorized: false, reason: 'Unknown channel pattern' }`. This is a defensive default; new patterns must be added to the function explicitly.
- **Channel name validation:** the function ALSO validates the channel name format before any auth check. The channel must match `^[a-z]+(:[a-zA-Z0-9_-]+)?$` (lowercase letters for the kind, optional colon-separated ID with alphanumerics + dash + underscore). Invalid channels are rejected with `{ authorized: false, reason: 'Invalid channel name' }`. This prevents injection attacks via the URL.

### Step 5: Create the SSE Route Handler

**Goal:** Expose the SSE endpoint at `/api/sse/[channel]` that accepts connections, registers them with the manager, and tears them down on disconnect.

**Files affected:** `src/app/api/sse/[channel]/route.ts` (new file).

**Specific instructions:**

Create the file containing:

- **`GET` handler:**
  - Signature: `async function GET(request: NextRequest, { params }: { params: { channel: string } }): Promise<Response>`.
  - Steps:
    1. **Validate the channel name** via `authorizeSseChannel` (which performs both validation and auth in one call). If not authorized, return a JSON error response:
       - For `Invalid channel name`: HTTP 400, `{ success: false, error: { code: 'VALIDATION_ERROR', message: 'Invalid channel name.' } }`.
       - For `No active session` (counter/security channels): HTTP 401, `{ success: false, error: { code: 'UNAUTHORIZED', message: 'Authentication required.' } }`.
       - For `Not assigned to counter` / `Not a security officer`: HTTP 403, `{ success: false, error: { code: 'FORBIDDEN', message: 'You do not have access to this channel.' } }`.
       - For `Unknown channel pattern`: HTTP 404, `{ success: false, error: { code: 'NOT_FOUND', message: 'Unknown channel.' } }`.
    2. **Create the `ReadableStream`:**
       - `const stream = new ReadableStream({ start: (controller) => { ... }, cancel: () => { ... } })`.
       - Inside `start(controller)`:
         - Generate a `clientId` via `randomUUID()`.
         - Read the user ID from the auth result (null for `global`).
         - Build the `SSEClient` record: `{ id: clientId, channel: params.channel, controller, connectedAt: new Date(), userId }`.
         - Call `sseManager.registerClient(params.channel, client)`.
         - **Send the initial "connected" event:** `controller.enqueue(formatSSEMessage({ type: 'CONNECTED', id: randomUUID(), timestamp: new Date().toISOString(), payload: { clientId, channel: params.channel } }))`. (The `CONNECTED` event type is a server-internal event, not part of the `SseEventPayload` discriminated union — it is only sent on connection establishment, not via `broadcastEvent`. The agent defines a separate `SSE_CONNECTED` constant for it.)
       - Inside `cancel()`:
         - Call `sseManager.removeClient(client)`.
    3. **Build the `Response`:**
       - `return new Response(stream, { headers: { 'Content-Type': 'text/event-stream', 'Cache-Control': 'no-cache, no-transform', 'Connection': 'keep-alive', 'X-Accel-Buffering': 'no' } })`.
- **Route handler options:**
  - Mark the route as `dynamic = 'force-dynamic'` (export `const dynamic = 'force-dynamic'`) so Next.js does not attempt to cache the SSE response.
  - Mark the route as `runtime = 'nodejs'` (export `const runtime = 'nodejs'`) — the SSE manager uses Node's `setInterval` and `globalThis`, which are not available in the edge runtime.
- **Error handling:**
  - Wrap the handler body in try/catch. On unexpected error, return HTTP 500 with the standard envelope.
  - Do NOT wrap the stream in try/catch — once the stream is handed to the `Response`, errors are propagated via the stream's `cancel()` callback.

### Step 6: Handle Counter-Specific Channel URL (Optional Path Approach)

**Goal:** Decide between two path schemes and implement the chosen one.

**Files affected:** Possibly `src/app/api/sse/counter/[counterId]/route.ts` (alternative path).

**Specific instructions:**

The route handler from Step 5 uses the URL pattern `/api/sse/[channel]` where the channel string itself contains the colon (e.g., `counter:abc123`). Next.js dynamic segments do NOT allow colons in URL segments by default (Next.js may URL-encode them or reject them depending on the version).

**Two options:**

- **Option A (recommended):** Use the single `[channel]` route and accept that the channel segment in the URL must use a dash or other separator (e.g., `/api/sse/counter-abc123`). The handler converts the dash back to a colon internally. The client URL is `/api/sse/counter-abc123`; the handler translates to channel `counter:abc123`.
- **Option B:** Use a nested directory structure `/api/sse/counter/[counterId]/route.ts` with a separate route handler for counter channels. The handler constructs the channel string `counter:<counterId>` internally.

The document specifies the chosen approach. **Recommended: Option A** for simplicity (one route handler, one auth function). The dash-to-colon conversion is straightforward.

### Step 8: Optional — Instrumentation Hook for Heartbeat Bootstrap

**Goal:** Decide whether `src/instrumentation.ts` needs to be modified to bootstrap the heartbeat, or whether the lazy approach (start on first client) is sufficient.

**Files affected:** `src/instrumentation.ts` (possibly modified, or unchanged).

**Specific instructions:**

- The lazy approach (start heartbeat on first client registration, stop when last client disconnects) is **recommended** and requires NO changes to `instrumentation.ts`.
- The eager approach (start heartbeat on app boot) wastes CPU when no clients are connected and requires:
  - Importing `sseManager` from `@/lib/sse-manager`.
  - Calling `sseManager.ensureHeartbeat()` (or a public equivalent) on boot.
  - This is NOT recommended for Phase 3.1.
- **The document specifies: do NOT modify `instrumentation.ts` for the heartbeat.** The lazy approach is the chosen implementation.

### Step 10: End-to-End SSE Connection Verification

**Goal:** Confirm the SSE stream opens correctly, the heartbeat fires, and channel auth works.

**Files affected:** None (verification only).

**Specific instructions:**

- Start the dev server: `yarn dev`.
- **Test 1 — global channel opens:**
  - In a terminal, run `curl -N http://localhost:3000/api/sse/global`.
  - Verify: within 1 second, an `event: connected` line appears, followed by `data: {...}`.
  - Verify: every 30 seconds, a `: heartbeat` comment appears.
  - The connection stays open until you close the terminal.
- **Test 2 — connection cleanup:**
  - Open the curl in Test 1. Note the client ID from the `connected` event payload.
  - Press `Ctrl+C` to close the curl.
  - In another terminal, query the manager stats via a temporary debug endpoint (the agent adds a `GET /api/debug/sse-stats` route protected for admins, OR logs stats via `console.debug` in the heartbeat tick — the document specifies which).
  - Verify: the closed client's ID is no longer in the active clients list.
- **Test 3 — heartbeat fires:**
  - With the curl open from Test 1, wait 35 seconds.
  - Verify: at least one `: heartbeat` comment appears within the 30-35 second window.
- **Test 4 — invalid channel name rejected:**
  - `curl -N http://localhost:3000/api/sse/invalid%20channel%21`.
  - Verify: HTTP 400 with `VALIDATION_ERROR`.
- **Test 5 — counter channel requires auth:**
  - Pick a counter ID from the database. `curl -N http://localhost:3000/api/sse/counter-<counterId>`.
  - Verify: HTTP 401 with `UNAUTHORIZED`.
- **Test 6 — counter channel with wrong officer:**
  - Log in as a user who is NOT assigned to the chosen counter. Make the curl request with their session cookie.
  - Verify: HTTP 403 with `FORBIDDEN`.
- **Test 7 — counter channel with correct officer:**
  - Log in as the user who IS assigned to the counter. Make the curl request.

## 8. Common Pitfalls

1. **Forgetting `globalThis` persistence.** Without it, every Next.js dev hot-reload tears down all active SSE connections. Symptom: connections drop every few seconds during development. Fix: add the `globalThis.__sseManager` pattern.
2. **Starting the heartbeat on app boot.** This wastes CPU when no clients are connected. Fix: lazy start in `ensureHeartbeat()`, lazy stop in `stopHeartbeat()`.
3. **Forgetting the SSE response headers.** Without `X-Accel-Buffering: no`, nginx (and many reverse proxies) buffers the SSE stream and clients see delayed events. Fix: include all four headers exactly.
4. **Using `EventSource`'s built-in reconnection.** The client hook (3.1.2) must NOT use `EventSource.onerror` to drive reconnection — the built-in reconnection has no backoff control. This document doesn't touch the client side; the warning is for 3.1.2's agent.
5. **Forgetting `runtime = 'nodejs'`.** Without it, Next.js may attempt to run the route in the edge runtime, which forbids `setInterval` and Prisma. The route will throw at registration time. Fix: export `const runtime = 'nodejs'`.
6. **Forgetting `dynamic = 'force-dynamic'`.** Without it, Next.js may attempt to cache the SSE response, which breaks the stream. Fix: export `const dynamic = 'force-dynamic'`.
7. **Logging payload contents.** PII risk — ticket numbers, officer names, etc. are PII in some jurisdictions. Fix: log only `type`, `id`, `channel`, `clientCount`. Never log the full payload.
8. **Mutating the channel set during iteration.** When broadcasting, iterating over a `Set` while clients may be removed (e.g., due to controller errors) leads to "Cannot iterate over modified set" errors. Fix: `Array.from(set)` before iterating.
9. **Writing to a closed controller.** Calling `controller.enqueue` on a closed controller throws. Fix: wrap each `enqueue` in try/catch and silently ignore errors.
10. **Not handling the `cancel()` callback.** Without it, disconnected clients remain in the manager indefinitely, leaking memory and skewing connection counts. Fix: always wire `cancel()` to call `removeClient()`.

## 9. AI Agent Guidance

This section is specifically for AI agents picking up this task plan document.

### 9.1 Critical Implementation Notes

- **The SSE manager must use `globalThis` for hot-reload safety.** This is non-negotiable. Without it, every code change during development kills every active SSE connection. The pattern is documented in Step 2.
- **The heartbeat is lazy.** Do not start it on app boot. Start it on the first `registerClient` call. Stop it when the last client disconnects.
- **The route handler must set `runtime = 'nodejs'` and `dynamic = 'force-dynamic'`.** These exports are required for the SSE stream to function correctly in Next.js 14+.
- **The route handler must NOT call `broadcastEvent()`.** That is 3.1.3's job. This document only sets up the connection; publishing events is the next sub-phase step.
- **Channel auth is enforced by the route handler, not the middleware.** The middleware allow-list is for "reachability" (anonymous users can hit the endpoint). The route handler decides "authorization" (whether the specific channel is accessible to the specific user).
- **No `AuditLog` writes.** The audit log is for user actions, not system events. SSE connection establishment is too high-frequency to audit and provides no actionable information.
- **The `SECURITY_OFFICER` role check must use the session's role array.** The exact mechanism depends on Phase 1.2.1's type augmentation — read the existing session shape before implementing the check.
- **The heartbeat comment must be exactly `: heartbeat\n\n`.** Two newlines at the end (the second terminates the SSE message). Some agents miss the second newline; the result is that the comment is treated as part of the previous message by some parsers.
- **`controller.enqueue` may throw if the controller is closed.** Always wrap in try/catch. The client may have disconnected while the broadcast was in flight.

### 9.2 What the Agent Should NOT Do

- Do not write to `AuditLog` from any path.
- Do not call `broadcastEvent()` from the route handler — that's 3.1.3's job. This document only establishes the connection.
- Do not modify the `broadcastEvent` signature.
- Do not add new `SseEventType` variants.
- Do not modify the Prisma schema.
- Do not start the heartbeat on app boot (lazy approach only).
- Do not implement rate limiting on SSE connections.
- Do not log payload contents (PII risk).
- Do not use `setInterval` without a corresponding `clearInterval` (the lazy start/stop pattern is required).
- Do not iterate over a `Set` while mutating it — copy to an array first.

### 9.3 What to Do When Stuck

- If `curl -N` shows no output, the route handler may have thrown before the stream was returned. Check the dev server console for the error.
- If the connection opens but no heartbeat fires after 30 seconds, the heartbeat interval may not be started. Verify `ensureHeartbeat()` is called from `registerClient()`.
- If the manager's connection count keeps growing after clients disconnect, the `cancel()` callback is missing or `removeClient()` is not being called. Add a `console.debug` in `removeClient()` to trace.
- If `auth()` throws in the route handler, the route may be running in the edge runtime (which forbids Prisma). Confirm `runtime = 'nodejs'` is exported.
- If the manager state resets on every code change, the `globalThis` persistence is missing. Add the `declare global` block and the `globalThis.__sseManager` check.
