## 1. Sub-Phase Purpose

Sub-Phase 3.1 is the **plumbing** that makes every later real-time feature possible. It is the first sub-phase of Phase 3 and has one job: turn the `broadcastEvent()` no-op stub from Phase 2.1.1 into a working server-to-client event delivery pipeline, and give the client side a hook that consumes it.

Three problems are solved in this sub-phase, in order:

1. **Server-side event delivery infrastructure** — the SSE manager singleton that holds all active client connections, the `/api/sse/[channel]` route handler that accepts new connections and tears them down on disconnect, the heartbeat ping that keeps idle connections alive through proxies and load balancers, and the canonical event envelope format. (Document 3.1.1.)
2. **Client-side event subscription** — the `useSSE()` React hook that opens an `EventSource`, parses incoming events into the typed envelope, invokes a caller-provided callback for each event, and reconnects automatically with exponential backoff when the connection drops. The client-side discriminated union in `src/types/sse.types.ts` is the single source of truth for what event types exist and what their payloads look like. (Document 3.1.2.)
3. **The real `broadcastEvent()` implementation** — replacing the no-op stub from Phase 2.1.1 with the actual function body that constructs an event envelope, looks up the channel(s) in the SSE manager, and writes the event to every connected client. The event routing rules (which event types go to which channels) are defined as a data structure. (Document 3.1.3.)

When this sub-phase is complete:

- A client can connect to `/api/sse/global` from a browser and receive a stream of events.
- Any Phase 2 endpoint that calls `broadcastEvent('global', 'TICKET_CALLED', payload)` (from 2.3.1) actually delivers the event to every connected display board within a second.
- The officer dashboard, the security screen, and the audio announcement system — all future consumers — can subscribe via the same `useSSE()` hook with their own channel and event-type filter.
- The seam between Phase 2 (event producers) and Phase 3 (event delivery) is real.

After Sub-Phase 3.1, **Phase 3's infrastructure is complete**. The remaining Phase 3 work is purely consumer-side: the display board (3.2) renders the events, and the audio announcement system (3.3) plays them out loud.

---

## 1.1 Why This Sub-Phase Comes First

Sub-Phases 3.2 and 3.3 cannot exist without 3.1:

- The display board (3.2) is a `useSSE()` consumer on the `global` channel.
- The audio system (3.3) is also a `useSSE()` consumer on the `global` channel (specifically listening for `TICKET_CALLED` and `TICKET_RECALLED`).
- The officer dashboard (Phase 4.2.3) will subscribe to the `counter:[counterId]` channel.
- The security officer screen (Phase 4.3.3) will subscribe to the `security` channel.

Sub-Phase 3.1 produces the infrastructure that every later real-time consumer depends on. Building 3.2 or 3.3 before 3.1 would mean building a UI with no data flowing into it.

## 1.2 What This Sub-Phase Is NOT

- It is **not** a UI / display / audio work. No display board, no bell, no TTS, no kiosk view of the queue. That is 3.2 and 3.3.
- It is **not** push notifications (FCM). Phase 4.1.
- It is **not** counter temporary closure. Phase 4.2.1.
- It is **not** the officer reply / broadcast message system. Phase 4.3.
- It is **not** multi-server SSE with Redis Pub/Sub. The in-process singleton is sufficient for single-server deployment (the documented deployment). Horizontal scaling is documented as a future concern in master plan Section 17.
- It is **not** rate limiting, audit log viewer, or PostgreSQL migration. Phase 5.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to distribute the complexity across the three task documents. The split is:

- **3.1.1** owns the **server-side plumbing**. It introduces the SSE manager singleton, the route handler, the heartbeat, the connection lifecycle, and the channel auth helper. It does **not** write the real `broadcastEvent()` body (that's 3.1.3's job) and does **not** build the client hook (that's 3.1.2's job).
- **3.1.2** owns the **client-side consumer**. It introduces the `useSSE()` hook, the exponential backoff logic, the event parsing, and a small test page that proves the hook works. It depends on 3.1.1 because it needs the route handler to be live in order to test against it.
- **3.1.3** owns the **real `broadcastEvent()` implementation** and the **event routing table**. It is the **critical seam realisation** — the function whose signature was locked in 2.1.1 finally gets a real body. It depends on 3.1.1 (the manager) and 3.1.2 (so the function can be exercised against a live client).

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 3.1.1 — SSE Architecture & Manager

**Owns:** The SSE manager singleton, the `/api/sse/[channel]` route handler, the heartbeat interval, the connection lifecycle, the channel auth helpers, and the SSE response header discipline.

**Does NOT own:** The real `broadcastEvent()` implementation (owned by 3.1.3), the `useSSE()` hook (owned by 3.1.2), the display board UI (3.2), the audio system (3.3).

**Key outputs:**

- `src/lib/sse-manager.ts`:
  - A module-scope singleton that survives Next.js hot-reload via a `globalThis` cache.
  - Internally a `Map<string, Set<SSEClient>>` keyed by channel name.
  - Methods: `registerClient(channel, client)`, `removeClient(client)`, `sendToClient(client, event)`, `sendToChannel(channel, event)`, `getChannelClientCount(channel)`, `getAllChannels()`.
  - **Heartbeat:** the manager lazily starts a `setInterval` on first client registration. The interval sends a comment-type `": heartbeat\n\n"` to every connected client every 30 seconds. The interval is a no-op if no clients are connected.
  - **Hot-reload safety:** uses `globalThis.__sseManager` to persist across Next.js dev hot-reloads. The agent confirms this is required (Next.js reloads server modules on every file change in dev; without `globalThis` persistence the manager loses all connections on every reload).
- `src/lib/sse-client.ts`:
  - Type definitions: `SSEClient` interface (`{ id: string, channel: string, controller: ReadableStreamDefaultController, connectedAt: Date }`), `SSEEnvelope` interface (the envelope shape).
- `src/app/api/sse/[channel]/route.ts`:
  - `GET` handler that:
    1. Validates the channel name (alphanumeric + colon-separated segments only; rejects anything else with HTTP 400).
    2. Performs channel-level auth based on the channel pattern (global = public; counter:[id] = session user must be the assigned officer; security = session user must have `SECURITY_OFFICER` role).
    3. Creates a `ReadableStream` whose `start(controller)` registers the client and sends the initial "connected" event; whose `cancel()` removes the client.
    4. Sets SSE response headers: `Content-Type: text/event-stream`, `Cache-Control: no-cache, no-transform`, `Connection: keep-alive`, `X-Accel-Buffering: no`.
  - The route does **not** itself call `broadcastEvent()` — it only sets up the connection. The manager and `broadcastEvent()` (built in 3.1.3) do the publishing.
- `src/lib/sse-auth.ts`:
  - `async function authorizeSseChannel(channel: string, session: Session | null): Promise<{ authorized: boolean, reason?: string }>` — encapsulates the per-channel auth logic.
- `src/instrumentation.ts`:
  - Extended from the 2.3.3 version to also start the SSE heartbeat interval lazily (actually the heartbeat is lazy — it starts on first client registration, not on app boot — so `instrumentation.ts` may not need to change; the document specifies).

**Master Plan sections implemented:** Sections 4.4 (SSE Event Bus Architecture), 9.3 (SSE endpoint), 11.1 (channel design — server side), 11.2 (envelope format), 11.5 (heartbeat), 17 (decisions on multi-server SSE future).

### 4.2 Document 3.1.2 — Client SSE Hook & State Management

**Owns:** The `useSSE()` React hook, the automatic reconnection with exponential backoff, the envelope parsing on the client, the test page, and the finalised client-side discriminated union (the type-level single source of truth for what events exist).

**Does NOT own:** The server-side SSE manager (owned by 3.1.1), the real `broadcastEvent()` body (owned by 3.1.3), the display board UI (3.2), the audio system (3.3).

**Key outputs:**

- `src/types/sse.types.ts` (extension, from 2.1.1 + 2.2.1 + 2.3.1 + 2.3.2 + 2.3.3):
  - The `SseEventPayload` discriminated union is **frozen** at the set of variants Phase 2 already defined. No new variants are added in 3.1.
  - A new exported type `SseEnvelope<T extends SseEventType>` that pairs the `SseEventPayload[T]` with the envelope fields: `{ type, id, timestamp, payload }`.
  - A `SseEventTypeMap` constant that maps each `SseEventType` to its payload variant. Used by `useSSE()` for type-safe callback parameters.
- `src/hooks/use-sse.ts` (note the kebab-case file):
  - The `useSSE()` hook (camelCase export name).
  - Signature: `useSSE<T extends SseEventType>(channel: string, options?: { onEvent?: (envelope: SseEnvelope<T>) => void, filter?: T | T[] }): { status: 'connecting' | 'open' | 'reconnecting' | 'closed', reconnectAttempts: number, lastEventAt: Date | null }`.
  - Implementation: opens `EventSource` on mount, parses each incoming event as JSON, validates the envelope shape, checks the type filter, invokes `onEvent` if it passes. On `EventSource.onerror`, closes the current connection and schedules a reconnection attempt using exponential backoff (1s, 2s, 4s, 8s, max 30s). The attempt count is held in a `useRef` so it survives re-renders.
  - Cleanup on unmount: closes the `EventSource` and clears any pending reconnection timer.
  - **`onerror` handling:** the hook listens for `EventSource.onerror` but does NOT use the built-in `EventSource` reconnection (the built-in reconnection has no backoff control). The hook implements its own.
  - **Hook rules compliance:** the hook can only be called from inside a React component or another hook (it uses `useEffect`, `useRef`, `useState`).
- `src/components/debug/sse-debug-panel.tsx`:
  - A small "use client" component that displays the current SSE connection status (`connecting` / `open` / `reconnecting` / `closed`), the reconnect attempt count, and a scrollable log of the last 20 events received.
- `src/app/(debug)/sse-test/page.tsx`:
  - A simple test page that renders the `SseDebugPanel` and subscribes to the `global` channel. This page is **temporary** — it is removed (or made admin-only) once Phase 3.2 ships the display board. For Phase 3.1, it is the verification surface.
  - Route is NOT in the `(dashboard)` group — it is a top-level debug route under `(debug)` (a new route group introduced in 3.1.2 for testing/verification pages).
- **Public allow-list:** the debug page route is added to the middleware's public allow-list (from Phase 1.2.3) so it can be accessed without authentication. (The display page in 3.2 will also be public; this is the first public route added to the SSE consumer side.)

**Master Plan sections implemented:** Sections 11.2 (envelope — client-side parsing), 11.5 (client-side reconnection), client half of Section 11.1.

### 4.3 Document 3.1.3 — Event Broadcasting System

**Owns:** The real `broadcastEvent()` implementation, the event routing table, the integration with Phase 2 call sites (no source changes — Phase 2 is already calling the function), and the forward-seam documentation for Phase 4 routing rules.

**Does NOT own:** The SSE manager itself (owned by 3.1.1), the `useSSE()` hook (owned by 3.1.2), the display board UI (3.2), the audio system (3.3), the FCM dispatch (Phase 4.1), the broadcast message creation UI (Phase 4.3).

**Key outputs:**

- `src/lib/events.ts` (modified — was a no-op stub from 2.1.1):
  - **`broadcastEvent(channel: string | string[], eventType: SseEventType, payload: SseEventPayload[SseEventType]): void`** — the real implementation.
  - Steps:
    1. Look up the channels from the input. If a single string, wrap in an array.
    2. For each channel, call `sseManager.sendToChannel(channel, envelope)`.
    3. The manager's `sendToChannel` writes the SSE-formatted message to every connected client on that channel.
    4. The function returns synchronously (the write is non-blocking; the actual TCP write happens on the next event loop tick).
    5. **No-op fallback:** if a channel has no connected clients, the function returns silently. Code that emits events does not have to check whether anyone is listening.
  - **Envelope construction:**
    - `type: eventType`
    - `id: randomUUID()` (Node built-in `crypto.randomUUID()`)
    - `timestamp: new Date().toISOString()` (ISO 8601 string)
    - `payload: payload` (typed via the discriminated union)
  - **SSE message format:** the envelope is JSON-serialised and written as:
    - `id: <envelope.id>\n`
    - `event: <envelope.type>\n`
    - `data: <JSON.stringify(envelope)>\n`
    - `\n` (blank line terminating the message)
  - The exact format matches the SSE spec and matches Master Plan Section 11.2.
  - **Logging:** in development, log via `console.debug` that the event was emitted (channel + type + id + size of payload). Never log the payload contents (PII risk).
- `src/lib/events.ts` (constants section):
  - **`EVENT_ROUTING: Record<SseEventType, SseChannel[]>`** — the routing table. For Phase 3.1, the entries are:
    - `TICKET_ISSUED` → `['global']`.
    - `TICKET_CALLED` → `['global']` (Phase 4 will extend this to `['global', 'counter:<counterId>']` when the officer dashboard subscribes).
    - `TICKET_RECALLED` → `['global']`.
    - `TICKET_NO_SHOW` → `['global']`.
    - `DAILY_RESET` → `['global']`.
  - The other event types Phase 2's enum defined (`COUNTER_OPENED`, `COUNTER_CLOSED`, `TICKET_SERVED`, `TICKET_COMPLETED`, `BROADCAST_MESSAGE`, `QUEUE_UPDATED`, `NOTIFICATION_RECEIVED`, `OFFICER_REPLY`) are NOT emitted in Phase 3.1. They are listed in the routing table as forward-seam comments (`// Phase 4.x will add: BROADCAST_MESSAGE → ['global', 'security']`) so future agents see the intent.
- **Convenience wrapper:** `broadcastRoutedEvent(eventType: SseEventType, payload: SseEventPayload[SseEventType], context: { counterId?: string }): void` — looks up the routing table, resolves any dynamic channel references (e.g., `counter:<counterId>` becomes `counter:<context.counterId>`), and calls `broadcastEvent` with the resolved channel list.
  - **Why two functions:** `broadcastEvent` is the low-level primitive (callers pass the channel(s) explicitly). `broadcastRoutedEvent` is the high-level helper (callers pass event type and dynamic context; the routing table decides the channels).
  - **Phase 2 code stays unchanged.** Phase 2 calls `broadcastEvent('global', 'TICKET_CALLED', payload)`. Phase 3.1.3 makes that call actually do something. There are no Phase 2 source changes required.
- **Integration test:** a small admin-only endpoint (or a temporary test script) that calls `broadcastRoutedEvent('TICKET_CALLED', payload, { counterId: '...' })` and verifies the event arrives on the connected test page from 3.1.2.

**Master Plan sections implemented:** Sections 4.4 (broadcast part of SSE Event Bus), 11.1 (channel design and routing), 11.2 (envelope construction), 11.3 (`TICKET_CALLED` payload — the actual delivery, since 2.3.1 only defined the shape), 11.4 (forward seam for `BROADCAST_MESSAGE`), 17 (decisions on routing table as data).

**Build order:** `3.1.1` first (the manager and route handler are prerequisites for everything else). `3.1.2` second (the hook can be developed against the live route from 3.1.1). `3.1.3` third (the real `broadcastEvent()` body is the last piece; it requires both the manager from 3.1.1 and a consumer from 3.1.2 to verify against).

**Why this order:**

- **3.1.1 first.** The route handler and manager are the physical transport. 3.1.2 needs them to subscribe; 3.1.3 needs them to publish.
- **3.1.2 after 3.1.1.** The hook opens a connection to the route handler. Without the route, the hook has nothing to connect to (development can mock the EventSource, but the test page must hit the real endpoint for a meaningful verification).
- **3.1.3 last.** The `broadcastEvent()` body needs the manager (3.1.1) to write through, and the verification page (3.1.2) to confirm the events actually arrive. Building it last ensures the integration is testable end-to-end.

**What depends on this sub-phase:**

- **Phase 3.2 (display board)** consumes `TICKET_CALLED`, `TICKET_RECALLED`, `TICKET_NO_SHOW`, `COUNTER_OPENED`, `COUNTER_CLOSED`, `DAILY_RESET` via `useSSE('global', ...)`.
- **Phase 3.3 (audio system)** consumes `TICKET_CALLED` and `TICKET_RECALLED` via `useSSE('global', { filter: ['TICKET_CALLED', 'TICKET_RECALLED'] })`.
- **Phase 4.2.3 (officer dashboard)** consumes `TICKET_QUEUED`, `QUEUE_UPDATED`, `COUNTER_STATUS_CHANGED`, `NOTIFICATION_RECEIVED` via `useSSE('counter:<counterId>', ...)`.
- **Phase 4.3.3 (security screen)** consumes `BROADCAST_MESSAGE` and `OFFICER_REPLY` via `useSSE('security', ...)`.

## 6. Cross-Document Standards

These standards apply to every document in Sub-Phase 3.1. The conventions from Phase 1, Phase 2.1, 2.2, and 2.3 (folder naming, import paths, TypeScript, Zod validation, error handling, env vars, git commits, audit logging, broadcast event seam) all carry forward unchanged. The standards below are **new or specific to Sub-Phase 3.1**.

### 6.1 Folder and File Naming

- All folders use kebab-case for multi-word names.
- Component files use PascalCase (`SseDebugPanel.tsx`).
- Hook files use kebab-case (`use-sse.ts`) — this is the convention used by `use-kiosk-reset.ts` from 2.2.2 and matches the `@/hooks/*` directory convention. (Note: the master plan's Section 7 file structure lists `useSSE.ts` with a capital S; in practice the codebase uses kebab-case for hook files. The documents specify the file path explicitly to avoid ambiguity.)
- Non-component, non-hook modules use camelCase (`sse-manager.ts`, `sse-auth.ts`, `events.ts`).
- The SSE manager file is `src/lib/sse-manager.ts` (matches Master Plan Section 7).
- The hook file is `src/hooks/use-sse.ts` (matches the codebase convention; the master plan's `useSSE.ts` reference is treated as informal).
- Route handlers use `route.ts`.
- Test / debug pages live under `(debug)` route group: `src/app/(debug)/<page>/page.tsx`.

### 6.2 Import Conventions

- All project-internal imports use the `@/` alias.
- The `broadcastEvent` function is imported from `@/lib/events`.
- The `SseChannel`, `SseEventType`, `SseEventPayload`, `SseEnvelope` types are imported from `@/types/sse.types`.
- The `auth()` helper is imported from `@/lib/auth` (Phase 1.2.1) for channel auth.
- The `SseManager` singleton is imported from `@/lib/sse-manager` for any module that needs to write events (currently only `events.ts`).
- The `useSSE` hook is imported from `@/hooks/use-sse` for any client component that subscribes to events.

### 6.3 TypeScript Standards

- All exported functions and components have explicit type annotations.
- No `any` types in production code.
- The `SseEventPayload` discriminated union (defined in 2.1.1, extended by 2.2.1 and 2.3.x) is **frozen** in 3.1. No new variants are added in 3.1. New variants will be added by future sub-phases as they introduce new event types — those documents will specify the additions explicitly.
- The `SseEnvelope<T>` generic type is a Phase 3.1 addition. It is exported from `@/types/sse.types`.
- The `SseEventTypeMap` constant is a Phase 3.1 addition. It maps each `SseEventType` string to the corresponding payload type.
- The `useSSE()` hook is generic over `T extends SseEventType`. When a caller passes a `filter` of `['TICKET_CALLED']`, the callback's `envelope` parameter is typed as `SseEnvelope<'TICKET_CALLED'>` — the payload is correctly typed without the caller needing to cast.

### 6.4 Response and Event Envelope Standard

Every API route in this sub-phase follows the master plan's standard envelope. SSE event payloads follow the envelope standard from Master Plan Section 11.2 (and Section 11.3 for `TICKET_CALLED`):

- **SSE message format (what is written to the wire):**
  - `id: <envelope.id>\n`
  - `event: <envelope.type>\n`
  - `data: <JSON.stringify(envelope)>\n`
  - `\n` (blank line terminating the message)
- **Envelope JSON shape:**
  - `type: string` (the `SseEventType`)
  - `id: string` (UUID v4)
  - `timestamp: string` (ISO 8601 datetime in UTC)
  - `payload: object` (the discriminated variant for the event type)
- **SSE response headers (set by the route handler):**
  - `Content-Type: text/event-stream`
  - `Cache-Control: no-cache, no-transform`
  - `Connection: keep-alive`
  - `X-Accel-Buffering: no` (disables nginx buffering for SSE)

### 6.5 Error Handling Standards

- The SSE route handler returns standard HTTP errors for connection-time failures (400 for invalid channel, 401 for unauthenticated when channel requires auth, 403 for unauthorized channel access, 404 for unknown channel pattern, 500 for unexpected errors).
- Once a connection is established, the server cannot "return an error" over SSE — the connection is closed with an appropriate HTTP status if an error occurs during setup, or the stream is closed cleanly via `controller.close()` if the manager is shut down.
- The `useSSE()` hook does NOT throw. It returns a `status` enum and an `onEvent` callback. The consumer is responsible for handling connection errors (e.g., showing a "Reconnecting..." UI).
- The `broadcastEvent()` function does NOT throw. It logs to `console.error` if a write fails and continues. Phase 5 may add structured error logging.

### 6.6 Audit Logging Standard

- The SSE manager does NOT write to `AuditLog`. Connection events are high-frequency and not auditable at that granularity.
- The `broadcastEvent()` function does NOT write to `AuditLog`. Event publishing is not an auditable user action.
- The `useSSE()` hook does NOT write to `AuditLog` (client-side; no DB access anyway).
- No new `AuditAction` types are added in this sub-phase.

### 6.7 Broadcast Event Seam (Sub-Phase 3.1 Specific)

- **Phase 2 call sites are NOT modified.** Every `broadcastEvent('global', '<EVENT>', payload)` call from Phase 2 (2.2.1, 2.3.1, 2.3.2, 2.3.3) becomes functional when 3.1.3 ships. The signature is unchanged.
- **Phase 3.1.3 introduces `broadcastRoutedEvent()`** as a higher-level wrapper. Future sub-phases that emit events should prefer `broadcastRoutedEvent(eventType, payload, context)` over `broadcastEvent` when the channels can be determined from the event type. The low-level `broadcastEvent` remains available for cases where the caller needs explicit channel control (rare).
- **Phase 4 extensions to the routing table** are documented as forward-seam comments in `src/lib/events.ts`. When Phase 4.1 adds FCM, the document specifies exactly where to extend the table.

### 6.8 Environment Variable Standards

- No new environment variables are introduced by this sub-phase.
- `APP_TIMEZONE` (introduced in 2.2.1) is NOT consumed by 3.1.
- `NEXTAUTH_SECRET` and the other auth env vars (from Phase 1.1.1) are consumed by the channel auth helper (via `auth()`).
- An optional `SSE_HEARTBEAT_INTERVAL_MS` env var MAY be introduced (default 30000). The document specifies whether it is introduced or hard-coded.

### 6.9 Module-Scope Singleton Discipline

The SSE manager is a module-scope singleton. The discipline:

- The manager is held in `globalThis.__sseManager` to survive Next.js dev hot-reload. (Without this, every code change in dev would kill every active SSE connection.)
- The manager's state is process-local. Multi-server deployments require Redis Pub/Sub (future, out of scope).
- The manager exposes a synchronous API (`registerClient`, `removeClient`, `sendToChannel`) — these are non-blocking and return immediately.
- The heartbeat `setInterval` is held by the manager and cleared on `dispose()` (currently never called — the manager lives for the lifetime of the process).

### 6.10 Client-Side Hook Discipline

The `useSSE()` hook is subject to the React rules of hooks:

- Called from inside a component or another hook.
- Has a stable `channel` parameter (changing `channel` mid-render would re-subscribe; the hook handles this by tearing down and re-subscribing).
- Cleans up on unmount (the `EventSource` is closed, the reconnection timer is cleared).
- Multiple subscriptions in a single component are achieved by calling `useSSE()` multiple times — one per channel.
- NOT called inside loops or conditions (React rule).

### 6.11 Commit Standards

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`.
- Each task plan document corresponds to one or more commits.
- Suggested commit boundaries:
  - End of 3.1.1: `feat: add SSE manager singleton and route handler with heartbeat and channel auth`
  - End of 3.1.2: `feat: add useSSE client hook with exponential backoff reconnection and SSE test page`
  - End of 3.1.3: `feat: implement broadcastEvent with routing table — Phase 2 call sites become functional`
- The 3.1.3 commit is the **critical seam realisation commit** — after it lands, every Phase 2 event call actually delivers.

### 8.1 Read These Documents First (In Order)

4. **This document** (3.1.0) — the sub-phase overview. The standards in Section 6 are binding for all three task plan documents.
5. **The relevant prior task plan** (2.1.1 for the no-op stub, 2.3.1 for the `TICKET_CALLED` payload shape) — to understand exactly what the existing code does.

### 8.2 Critical Implementation Notes

- **The `broadcastEvent` signature must NOT change.** Phase 2 code calls `broadcastEvent(channel, eventType, payload)`. Phase 3.1.3 implements the body; it does NOT modify the signature. If the agent feels tempted to rename or restructure, that is a violation of the seam.
- **The `SseEventPayload` union is frozen in 3.1.** Adding new variants here is out of scope (future sub-phases will add them when they introduce new event types).
- **The SSE manager must survive Next.js hot-reload.** Use `globalThis.__sseManager`. Without this, every code change in development tears down every active SSE connection.
- **The heartbeat interval must be lazily initialised.** Starting it on app boot wastes CPU when no clients are connected. Start it on the first `registerClient` call, and stop it when the last client disconnects.
- **The `useSSE()` hook must use its own reconnection logic.** The built-in `EventSource` reconnection has no backoff control. The hook implements exponential backoff manually.
- **Channel auth is per-channel, not global.** The route handler must inspect the channel name and apply the right auth rule. Do not implement a single global auth check.
- **The public allow-list must be extended** to include the SSE endpoint and the debug page. Phase 1.2.3's middleware needs updating.
- **No `AuditLog` writes from any 3.1 path.** SSE connection events are too high-frequency to audit; event publishing is not an auditable user action.

### 8.3 What the Agent Should NOT Do

- Do not write to the `AuditLog` table from any 3.1 path.
- Do not modify the Prisma schema.
- Do not add new `SseEventType` variants (the union is frozen).
- Do not modify the `broadcastEvent` signature.
- Do not implement Redis Pub/Sub (multi-server is future).
- Do not implement FCM (Phase 4.1).
- Do not build the display board UI (3.2), the audio system (3.3), or the officer dashboard (4.2.3).
- Do not add rate limiting (5.2.1).
- Do not write to `AuditLog` from any path in this sub-phase.

### 8.4 What to Do When Stuck

- If the SSE stream does not appear in `curl -N`, check the response headers — missing `Content-Type: text/event-stream` is the most common cause.
- If `useSSE()` reconnects in a tight loop, the `EventSource.onerror` is firing — verify the `console.error` log and check the network panel. Often it is a CORS issue or a 401 from the route handler.
- If `broadcastEvent` does not deliver, check that the manager's `sendToChannel` is being called — add a temporary `console.debug` in `events.ts` to trace.
- If the heartbeat fires but no events arrive, the issue is in `sendToChannel`'s SSE formatting (likely the `\n\n` terminator is missing or malformed).

### 8.5 Verification Commands the Agent Should Run

- `yarn build` — must succeed.
- `yarn lint` — must be clean.
- `yarn type-check` — must be clean.
- `curl -N http://localhost:3000/api/sse/global` — should show `event: connected` followed by heartbeat comments every 30 seconds.
- Open `http://localhost:3000/sse-test` in a browser — should show "Connected" within a second.
- Trigger a `broadcastEvent` from any server-side script (or via a temporary `/api/debug/broadcast` endpoint) — should appear in the browser's debug panel within a second.
