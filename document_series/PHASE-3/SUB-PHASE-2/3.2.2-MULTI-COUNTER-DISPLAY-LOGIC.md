## 1. Document Purpose

This document delivers the **multi-counter display logic** for the display board — the layer that turns the single-counter stub from 3.2.1 into a full real-time, multi-counter, animated display. It introduces the **responsive grid layout** that adapts to 1, 2, 4, 8, or more active counters, the **per-counter in-memory recent calls history buffer** (capped at `DisplayBoard.maxDisplayedTickets`), the **transition animation** on a new ticket call, the **counter closure state rendering** (greyed-out card with "Temporarily Closed" label), and the **recall / no-show visual handling** for the "now serving" slot.

When this document is complete:

- The grid layout adapts correctly to any number of active counters (1-2 → 2 cols, 3-4 → 2 cols, 5-8 → 3 cols, 9+ → 4 cols).
- Each counter card maintains an in-memory buffer of the last N called tickets, where N is `DisplayBoard.maxDisplayedTickets`.
- On a new `TICKET_CALLED` event, the new ticket number animates into place (~300ms transition).
- The recent calls history is ordered most-recent-first and capped at `maxDisplayedTickets`.
- A counter marked as `closed` shows the "Temporarily Closed" state on the display.
- A `TICKET_RECALLED` event does NOT add a duplicate to the recent calls history.
- A `TICKET_NO_SHOW` event removes the ticket from "now serving"; if `autoAdvanced = true`, the auto-advanced ticket becomes the new "now serving".
- A `DAILY_RESET` event clears all "now serving" and recent calls history entries.
- The reducers in `src/lib/display-state.ts` (stubs in 3.2.1) are fully implemented.

This document does **not** build the display page itself (owned by 3.2.1), the `DisplayBoard` CRUD or admin UI (owned by 3.2.3), or the audio system (Phase 3.3).

### Step 2: Implement the Full `applyTICKET_CALLED` Reducer

**Goal:** Replace the stub `applyTICKET_CALLED` in `src/lib/display-state.ts` with the full implementation that respects the buffer cap.

**Files affected:** `src/lib/display-state.ts` (modified).

**Specific instructions:**

- Locate the `applyTICKET_CALLED` stub.
- **Implement the full logic:**
  - Extract the `payload` from the `envelope`. The `TICKET_CALLED` payload fields (from Phase 2.3.1 + master plan Section 11.3):
    - `ticketId`, `ticketNumber`, `serviceName`, `serviceId`, `counterId`, `counterName`, `counterNumber`, `officerName`, `calledAt`.

  - Return a new `state` object (use immutable update — do not mutate the original).
- Document that the reducer is pure (no side effects) and returns a new state object.

### Step 3: Implement the Full `applyTICKET_RECALLED` Reducer

**Goal:** Implement the recall reducer so it does NOT add a duplicate to the recent calls history.

**Files affected:** `src/lib/display-state.ts` (modified).

**Specific instructions:**

- Locate the `applyTICKET_RECALLED` stub.
- **Implement the logic:**
  - Extract the `counterId` from the payload.

  - Return the unchanged `state`.

### Step 4: Implement the Full `applyTICKET_NO_SHOW` Reducer

**Goal:** Implement the no-show reducer with auto-advance visual handling.

**Files affected:** `src/lib/display-state.ts` (modified).

**Specific instructions:**

- Locate the `applyTICKET_NO_SHOW` stub.
- **Implement the logic:**
  - Extract `counterId`, `autoAdvanced`, and `autoAdvancedTicketNumber` from the payload (the `TICKET_NO_SHOW` payload from Phase 2.3.2 includes these fields).
  - Set `state.nowServing[counterId] = null`.
  - **If `autoAdvanced === true`:**
    - Build a `TicketDisplayData` from the payload fields:
      - `ticketId`: `null` (the auto-advanced ticket's ID is not in the no-show payload — the document specifies: the auto-advance is announced via a separate `TICKET_CALLED` event from `callTicket`. The display board waits for that event rather than building the data from the no-show payload alone).
    - **Alternative approach (simpler, recommended):** set `state.nowServing[counterId] = null` and rely on the subsequent `TICKET_CALLED` event for the auto-advanced ticket to populate the new "now serving". This avoids data duplication and keeps the state machine consistent.
    - The document specifies the chosen approach: **rely on the subsequent `TICKET_CALLED` event**. The no-show reducer simply sets `nowServing = null`. When `callTicket` emits `TICKET_CALLED` for the auto-advanced ticket (which happens after the no-show transaction commits, per Phase 2.3.2's design), the `applyTICKET_CALLED` reducer populates the new "now serving".
  - **If `autoAdvanced === false`:** set `state.nowServing[counterId] = null`. The "now serving" slot shows the placeholder `—`.
  - Return the new state.
- Document that this approach relies on the event ordering: `TICKET_NO_SHOW` arrives before the auto-advance `TICKET_CALLED`. This ordering is guaranteed by Phase 2.3.2's design (no-show commits first, then auto-advance runs as a separate transaction and emits its own `TICKET_CALLED`).

## 8. Common Pitfalls

1. **Mutating state directly.** React state must be updated immutably. Always create a new object (e.g., `{ ...state, nowServing: { ...state.nowServing, [counterId]: ticket } }`). Direct mutation (`state.nowServing[counterId] = ticket`) will cause stale state and missed re-renders.
2. **Forgetting the buffer cap.** The recent calls buffer must be capped at `board.maxDisplayedTickets`. Without the cap, the array grows unbounded over a long session.
3. **Adding a duplicate to recent calls on recall.** The recall is the same ticket being re-announced. Do NOT add it to `recentByCounter`. The reducer returns the unchanged state.
4. **Wrapping the auto-advance ticket in the no-show event.** The no-show event's payload includes `autoAdvancedTicketNumber` but NOT the full ticket data. Do NOT try to build a `TicketDisplayData` from the no-show payload — rely on the subsequent `TICKET_CALLED` event from the `callTicket` auto-advance path.
5. **Not respecting `prefers-reduced-motion`.** The transition animation may be jarring for users with motion sensitivity. Add a media query that disables the animation.
6. **Re-rendering all counters on every state change.** Memoize `DisplayTicketBlock` so only the affected counter re-renders. The grid container itself can re-render freely.
7. **Forgetting to clean up the broadcast auto-dismiss timer.** If the user navigates away mid-broadcast, the timer fires on an unmounted component, causing a React warning. Always clean up the timer on unmount.
8. **Hard-coding the `maxDisplayedTickets` value.** Read it from the snapshot (which reads it from the `DisplayBoard` record). Hard-coding `10` defeats the purpose of the configuration.
9. **Using CSS keyframes instead of transitions for the animation.** Keyframes play once on mount; transitions play when the property changes. For a `key`-based remount, both work, but transitions are more idiomatic for this use case (and respect `prefers-reduced-motion` more cleanly).
10. **Not handling the empty state of `recentTickets`.** When a counter has no recent calls, the list renders nothing. Do NOT show a "No recent calls" placeholder — the display should look clean.

## 9. AI Agent Guidance

This section is for the AI agent that will implement this document. It supplements the general guidance in Sub-Phase 3.2 Section 8.

### 9.1 Why Recall Does Not Add to Recent Calls

The recall event is the same ticket being re-announced (e.g., the customer didn't hear the first call). The audio system (3.3) re-plays the announcement; the visual is unchanged. If the recall added a duplicate to the recent calls list, customers would see the same ticket number appearing twice in quick succession, which is confusing. The recall reducer returns the state unchanged.

### 9.2 Why No-Show Relies on the Subsequent `TICKET_CALLED` Event

The no-show event from Phase 2.3.2 includes `autoAdvanced: boolean` and `autoAdvancedTicketNumber: string | null`, but it does NOT include the full ticket data needed for the display (officer name, service name, etc.). Building a partial `TicketDisplayData` from the no-show payload would require an extra DB fetch. Instead, the no-show reducer simply sets `nowServing = null`, and the subsequent `TICKET_CALLED` event (emitted by the auto-advance `callTicket` operation) provides the full data. This keeps the event ordering simple and avoids data duplication.

If `autoAdvanced = false` (no auto-advance), the `TICKET_CALLED` event does NOT arrive, and the "now serving" slot stays as `—`. This is the correct visual: the counter has no ticket in progress.

### 9.3 Why the Transition Animation Uses `key` + CSS Transition

The animation triggers when the ticket number changes. Using React's `key` prop on the inner element causes React to unmount and remount the element when the key changes. The remount triggers the CSS `transition` from `opacity-0 scale-95` to `opacity-100 scale-100`, creating the fade-in effect.

Alternative approaches (e.g., using a CSS animation with `forwards` fill mode, or a JavaScript animation library) are also valid but more complex. The `key` + transition approach is idiomatic, lightweight, and respects `prefers-reduced-motion` via a media query.

### 9.4 Why the Grid Has a Maximum of 4 Columns

The master plan's Section 6.4 specifies the grid layout with 2-4 columns. With 9+ counters, 4 columns is the maximum because:

- Each ticket number needs to be readable from across a waiting room.
- 4 columns × ~3 rows = 12 counters visible at once (sufficient for most installations).
- Beyond 16 counters, vertical scroll is acceptable (the display board is large enough to show 4 columns × 4+ rows).

### 9.5 Why the Grid Scrolls for Many Counters

When the counter count exceeds what fits in the viewport (typically >16 counters), vertical scroll is enabled. This is a deliberate trade-off: the display board prioritises showing each counter's "now serving" ticket in a large, readable format. Horizontal scrolling or pagination would force the operator to interact with the board, which is undesirable for a display-only screen.

### 9.6 How to Verify the Multi-Counter Logic Without a Real Display

For automated testing, the agent can:

- Use `curl` to trigger events via a temporary `/api/debug/broadcast` endpoint (from 3.1.2).
