## 1. Document Purpose

This document delivers the **display configuration system** ‚Äî the API endpoints, admin UI, and URL scheme for managing `DisplayBoard` configurations. A super-admin can create, edit, list, and delete board configurations, each with its own theme color, logo, marquee message, TTS settings, and announcement template. The display page at `/display` (built in 3.2.1) reads the configuration from the snapshot endpoint and applies it. The URL scheme allows launching a specific board (`/display?boardId=xxx`) or the default board (`/display`).

When this document is complete:

- A super-admin can open `/settings/display`, see all display boards in a table, create a new board, edit any board, mark a board as default, and delete boards.
- The `DisplayBoard` CRUD API endpoints are functional (`GET/POST /api/display-boards`, `GET/PATCH/DELETE /api/display-boards/[boardId]`).
- The `isDefault` invariant is enforced: only one board may have `isDefault = true` at any time.
- The URL scheme works: `/display` loads the default board, `/display?boardId=xxx` loads the specific board, and missing/invalid `boardId` falls back to default.
- Every `DisplayBoard` mutation writes an `AuditLog` row.
- The `AuditAction` union is extended with `DISPLAY_BOARD_CREATED`, `DISPLAY_BOARD_UPDATED`, `DISPLAY_BOARD_DELETED`, `DISPLAY_BOARD_DEFAULT_CHANGED`.

This document does **not** build the display page itself (owned by 3.2.1), the multi-counter grid logic (owned by 3.2.2), or the audio system (Phase 3.3).

### Step 2: Create the `displayBoard` Zod Schemas

**Goal:** Define validation schemas for the `DisplayBoard` create and update payloads.

**Files affected:** `src/schemas/display-board.schema.ts` (new file).

**Specific instructions:**

Create `src/schemas/display-board.schema.ts` containing:

- **`displayBoardCreateSchema`** ‚Äî all fields except `id`, `createdAt`, `updatedAt`:
  - `name: string` ‚Äî required, min 1, max 100 chars.
  - `isDefault: boolean` ‚Äî optional, defaults to `false`.
  - `maxDisplayedTickets: number` ‚Äî required, integer, min 1, max 50.
  - `announcementEnabled: boolean` ‚Äî optional, defaults to `true`.
  - `bellEnabled: boolean` ‚Äî optional, defaults to `true`.
  - `ttsEnabled: boolean` ‚Äî optional, defaults to `true`.
  - `ttsLanguage: string` ‚Äî required, validated as BCP-47 (regex: `^[a-z]{2}(-[A-Z]{2})?$` or use a Zod refinement).
  - `ttsRate: number` ‚Äî required, min 0.1, max 10.
  - `ttsPitch: number` ‚Äî required, min 0, max 2.
  - `ttsVolume: number` ‚Äî required, min 0, max 1.
  - `announcementTemplate: string` ‚Äî required, min 1, max 500 chars.
  - `themeColor: string` ‚Äî optional, validated as hex color (regex: `^#[0-9A-Fa-f]{6}$`).
  - `logoUrl: string` ‚Äî optional, validated as URL (use `z.string().url()`).
  - `customMessage: string` ‚Äî optional, max 500 chars.
- **`displayBoardUpdateSchema`** ‚Äî same fields as create, but all optional (PATCH semantics). At least one field must be provided (use `.refine((data) => Object.keys(data).length > 0, { message: 'At least one field must be provided for update.' })`).
- **Inferred TypeScript types:** `DisplayBoardCreateInput`, `DisplayBoardUpdateInput`.
- Document that these schemas are reused on the server only (the admin UI calls them indirectly via the form component).

### Step 4: Create the Create Endpoint `POST /api/display-boards`

**Goal:** Expose the create board endpoint with `isDefault` invariant enforcement.

**Files affected:** `src/app/api/display-boards/route.ts` (modified ‚Äî extends the file from Step 3).

**Specific instructions:**

- Open the file from Step 3.
- **Add the `POST` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_SYSTEM_CONFIGURE)`.
  - Reads the session via `auth()` to get the actor's user ID (for audit logging).
  - Parses and validates the body with `displayBoardCreateSchema`.
  - Wraps the create logic in a Prisma `$transaction`:
    1. **If `isDefault: true`:** unset `isDefault` on all existing boards (`db.displayBoard.updateMany({ where: { isDefault: true }, data: { isDefault: false } })`).
    2. Create the new board (`db.displayBoard.create({ data: validatedBody })`).
  - After the transaction commits, write an `AuditLog` row:
    - `action: 'DISPLAY_BOARD_CREATED'`.
    - `entity: 'DisplayBoard'`.
    - `entityId: <new board id>`.
    - `actorId: <session user id>`.
    - `description: 'Created display board "<name>"'`.
    - `metadata: { isDefault: <boolean> }`.
  - **If `isDefault: true` was set on the new board,** also write a `DISPLAY_BOARD_DEFAULT_CHANGED` audit log row (after the transaction).
  - Returns `{ success: true, data: <created board> }` with HTTP 201.
  - On Zod validation failure: returns HTTP 422 with `VALIDATION_ERROR`.
  - On unexpected error: returns HTTP 500 with `INTERNAL_ERROR`.
- Document that the `isDefault` invariant is enforced inside the transaction (atomic with the create).

### Step 6: Create the Update Endpoint `PATCH /api/display-boards/[boardId]`

**Goal:** Expose the update board endpoint with `isDefault` invariant enforcement.

**Files affected:** `src/app/api/display-boards/[boardId]/route.ts` (modified ‚Äî extends the file from Step 5).

**Specific instructions:**

- Open the file from Step 5.
- **Add the `PATCH` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_SYSTEM_CONFIGURE)`.
  - Reads `boardId` from route params.
  - Reads the session via `auth()` for audit logging.
  - Parses and validates the body with `displayBoardUpdateSchema`.
  - Verifies the board exists (`db.displayBoard.findUnique({ where: { id: boardId } })`). If not found: returns 404.
  - Wraps the update logic in a `$transaction`:
    1. **If `isDefault: true` is being set:**
       - Unset `isDefault` on all OTHER boards (`db.displayBoard.updateMany({ where: { isDefault: true, NOT: { id: boardId } }, data: { isDefault: false } })`).
    2. **If `isDefault: false` is being set on a board that was previously the default** (this is the case of un-defaulting an existing default):
       - No additional action needed; the transaction updates the board's `isDefault` to `false`.
       - **Edge case:** if this was the only default board, after the update there is no default. The display page's fallback message handles this ("No default display board configured").
    3. Update the board (`db.displayBoard.update({ where: { id: boardId }, data: validatedBody })`).
  - After the transaction commits, write audit log rows:
    - **Always:** `DISPLAY_BOARD_UPDATED` with the changed fields in `metadata` (compute the diff between the old and new board).
    - **If `isDefault` changed:** also write `DISPLAY_BOARD_DEFAULT_CHANGED` with the old and new default board IDs.
  - Returns `{ success: true, data: <updated board> }` with HTTP 200.

### Step 7: Create the Delete Endpoint `DELETE /api/display-boards/[boardId]`

**Goal:** Expose the delete board endpoint with proper audit logging.

**Files affected:** `src/app/api/display-boards/[boardId]/route.ts` (modified ‚Äî extends the file from Step 5).

**Specific instructions:**

- Open the file from Step 5.
- **Add the `DELETE` handler:**
  - Wrapped with `withPermission(handler, PERMISSION_SYSTEM_CONFIGURE)`.
  - Reads `boardId` from route params.
  - Reads the session via `auth()` for audit logging.
  - Verifies the board exists. If not found: returns 404.
  - **Records whether the board is the default** before deletion (for audit logging).
  - Deletes the board (`db.displayBoard.delete({ where: { id: boardId } })`).
    - The document specifies: **hard delete** is used (no soft-delete flag). The `DisplayBoard` table has no FK from other tables; deleted boards simply cannot be referenced by `/display?boardId=xxx` (the page falls back to default with a console warning).
  - After the delete commits, write audit log rows:
    - **Always:** `DISPLAY_BOARD_DELETED` with the board's name and ID in metadata.
    - **If the deleted board was the default:** also write `DISPLAY_BOARD_DEFAULT_CHANGED` with `previousDefaultId: boardId, newDefaultId: null`.
  - Returns `{ success: true, data: null }` with HTTP 200.
  - **Document the edge case:** deleting the only default board leaves the system with no default. The next request for `/display` (no `boardId`) returns a snapshot with `board: null`, and the display page shows the "No display board configured" fallback. This is acceptable behavior; the admin is informed via the audit log.

## 8. Common Pitfalls

1. **Forgetting the `isDefault` invariant.** Without the invariant, multiple boards could be marked as default simultaneously, breaking the `/display` fallback behavior. The invariant MUST be enforced inside the transaction.
2. **Logging the audit row inside the transaction.** The audit log helper is best-effort. If it fails inside the transaction, the transaction rolls back. The audit log MUST be written AFTER the transaction commits.
3. **Soft-deleting display boards.** Soft-delete complicates the URL scheme (`/display?boardId=xxx` for a soft-deleted board returns 404 either way). Hard delete is simpler and acceptable for Phase 3. If the operator wants undo, the audit log provides a forensic trail.
4. **Not handling the no-default-board case.** After deleting the default board, `/display` (no `boardId`) returns a snapshot with `board: null`. The display page must render the fallback message. Test 8 verifies this.
5. **Using a `Switch` for booleans without a label.** Switches in shadcn require a label or they appear unlabeled. The agent must include a label or `aria-label`.
6. **Not validating `ttsLanguage` as BCP-47.** A typo like `enUS` (no dash) should be rejected. The Zod regex `^[a-z]{2}(-[A-Z]{2})?$` enforces the format.
7. **Forgetting to set `dynamic = 'force-dynamic'` on the API routes.** The endpoints must read fresh data from the DB; without the export, Next.js may cache.
8. **Forgetting to update the `AuditAction` union before writing audit logs.** If the action type is not in the union, TypeScript errors out. Step 1 must be done first.
9. **Using `useState` without cleanup in the form's submit handler.** If the user navigates away mid-submit, the response handler may run on an unmounted component. Use `AbortController` or check `isMounted` before calling `setState`.
10. **Not memoizing the form's field components.** With 14 fields, the form re-renders on every keystroke. The agent wraps the form in `React.memo` or uses uncontrolled inputs with refs.
11. **Returning the wrong HTTP status on create.** The convention is HTTP 201 for `POST` that creates a resource. The agent returns 201 on success, 200 on `GET`/`PATCH`, 204 on `DELETE` (or 200 with `{ success: true, data: null }` per the master plan's envelope standard ‚Äî the document specifies: 200 with the standard envelope is acceptable).

## 9. AI Agent Guidance

This section is for the AI agent that will implement this document. It supplements the general guidance in Sub-Phase 3.2 Section 8.

### 9.1 Why the `isDefault` Invariant Must Be in a Transaction

Two concurrent admin sessions could attempt to set different boards as default simultaneously. Without a transaction, both updates could succeed, resulting in two default boards (or one default board being silently overridden). The transaction ensures the invariant is preserved at every commit.

The Prisma transaction looks like:

1. `db.displayBoard.updateMany({ where: { isDefault: true, NOT: { id: targetBoardId } }, data: { isDefault: false } })`.
2. `db.displayBoard.update({ where: { id: targetBoardId }, data: { isDefault: true } })`.

Both operations are inside `$transaction([...])` so they commit atomically.

### 9.2 Why the Audit Log Is Best-Effort

The audit log writes happen AFTER the transaction commits. If the audit log write fails (e.g., DB connection drops), the main action is still recorded in the database; only the audit log row is missing. This is acceptable because:

- The audit log is for forensics, not enforcement.
- The main action (board create / update / delete) is the source of truth.
- The audit log helper already catches errors and logs them via `console.error` (per Phase 1.3.3's discipline).

If the audit log were inside the transaction and failed, the main action would roll back ‚Äî undesirable.

### 9.3 Why Hard Delete Is Used (Not Soft Delete)

The `DisplayBoard` table has no FK from other tables (other than possibly `BroadcastMessage.targetDisplayBoardId`, but that field is nullable). Hard delete is acceptable because:

- The count is small (< 20 boards expected).
- Deleted boards cannot be referenced by `/display?boardId=xxx` (the page falls back to default).
- The audit log provides a forensic trail.

If soft delete is needed in the future, the agent can add an `isArchived` field to the model and update the snapshot endpoint to skip archived boards. This is deferred to Phase 5 or later.

### 9.4 Why the URL Scheme Is `?boardId=xxx` (Not Path-Based)

The URL scheme uses a query parameter (`/display?boardId=xxx`) instead of a path segment (`/display/[boardId]`) because:

- The display page is at `/display` (not under a dynamic segment).
- Query parameters are easier to extend (e.g., `?boardId=xxx&theme=dark`).
- Path-based routing would require a separate route, complicating the layout structure.

The chosen approach: query parameter `boardId` is optional; absent means "use default".

### 9.5 Why the Form Has Helper Text for Placeholders

The `announcementTemplate` field uses placeholders (`{number}`, `{counter}`, `{service}`). Without helper text, the admin would not know what placeholders are available or how to use them. The form shows the available placeholders below the textarea, with the default template pre-filled as an example.

### 9.6 How to Verify the Configuration System Without an Admin UI

For automated testing, the agent can:

- Use `curl` to call the API endpoints directly with a super-admin session cookie.
- Use Prisma studio to inspect the DB state.
- Verify audit log rows via Prisma studio.

For manual testing, the agent logs in as a super-admin, navigates to `/settings/display`, and exercises the UI.
