## 1. Document Purpose

This document delivers the **visual public face** of the queue system: the main display board at `/display`. It introduces the full-screen dark-themed display page, the audio context unlock overlay (the operator's click-to-enable gesture), the three core display components (`DisplayTicketBlock`, `MarqueeMessage`, `DisplayClock`), the snapshot endpoint that fetches the initial state, the SSE subscription wiring that keeps the board live, and the `BroadcastBanner` layout slot that Phase 4.3 will populate with broadcast messages.

When this document is complete:

- Opening `/display?boardId=xxx` in a browser shows the configured display board with the correct theme color, logo, and marquee message.
- The full-screen dark-themed layout (`display-bg` background, top bar, main section, bottom marquee) fits in the viewport without scroll.
- The audio context unlock overlay is shown on first load and dismissed on click.
- After dismissing the overlay, the display board subscribes to the `global` SSE channel via `useSSE()`.
- `TICKET_CALLED` events received via SSE update the page within a second.
- The snapshot endpoint returns the initial state (active counters, latest CALLED tickets, board configuration).
- The `/display` route is added to the middleware's public allow-list.

This document does **not** handle the multi-counter grid breakpoints or the recent calls history buffer (owned by 3.2.2), the `DisplayBoard` CRUD or admin UI (owned by 3.2.3), or the actual audio playback (Phase 3.3).

### Step 3: Create the Snapshot Helper `src/lib/display-snapshot.ts`

**Goal:** Provide a single server-side function that fetches the initial state for the display page.

**Files affected:** `src/lib/display-snapshot.ts` (new file).

**Specific instructions:**

Create `src/lib/display-snapshot.ts` containing:

- **`async function getDisplayBoardById(boardId: string): Promise<DisplayBoardConfig | null>`**:
  - Queries `db.displayBoard.findUnique({ where: { id: boardId } })`.
  - If the board is not found, returns `null`.
  - Otherwise, maps the Prisma record to `DisplayBoardConfig` (omitting `createdAt` / `updatedAt`).
- **`async function getDefaultDisplayBoard(): Promise<DisplayBoardConfig | null>`**:
  - Queries `db.displayBoard.findFirst({ where: { isDefault: true } })`.
  - Returns the mapped `DisplayBoardConfig` or `null`.
- **`async function getActiveCounters(): Promise<CounterDisplayData[]>`**:
  - Queries `db.counter.findMany({ where: { isActive: true }, orderBy: { number: 'asc' } })`.
  - Maps each to `CounterDisplayData` (status hard-coded to `'open'` in this document; 3.2.2 extends this).
- **`async function getServingTickets(): Promise<Record<string, TicketDisplayData>>`**:
  - For each active counter, fetches the latest CALLED ticket:
    - Uses `db.ticket.findFirst({ where: { counterId, status: { in: ['CALLED', 'RECALLED', 'SERVING'] } }, orderBy: { calledAt: 'desc' }, include: { service: true } })`.
    - Maps to `TicketDisplayData`. The `officerName` comes from a join to `User` via `CounterOfficer.officerId` (the document specifies this join ‚Äî it's the most-recent `CounterOfficer` for the counter who called the ticket).
  - Returns a map keyed by counter ID.
- **`async function getRecentTicketsForCounter(counterId: string, max: number): Promise<TicketDisplayData[]>`**:
  - Queries the most recent N CALLED `TicketEvent` records for this counter on the current business date, joined to the ticket for full info.
  - The query: `db.ticketEvent.findMany({ where: { counterId, eventType: 'CALLED' }, orderBy: { createdAt: 'desc' }, take: max, include: { ticket: { include: { service: true } } } })`.
  - For each event, builds a `TicketDisplayData` from the joined ticket + event metadata.
  - Returns the array, most-recent-first.
  - Note: `TicketEvent` does NOT have a direct relation to `Ticket` via Prisma's auto-generated schema ‚Äî the agent checks the schema and uses the appropriate query. If no direct relation, the query uses `tx.ticket.findUnique({ where: { id: ticketEvent.ticketId }, ... })` per event (less efficient but correct). The document specifies: prefer a Prisma relation if available; otherwise use a per-event fetch. Performance is not critical at this scale.
- **`async function getDisplaySnapshot(boardId: string | null): Promise<DisplaySnapshot>`**:
  - If `boardId` is provided, calls `getDisplayBoardById(boardId)`. If null, calls `getDefaultDisplayBoard()`. If both are null, returns `{ board: null, counters: [], servingTickets: {}, recentTickets: {} }`.
  - Fetches active counters in parallel with serving tickets and recent tickets (Promise.all for performance).
  - Returns the assembled `DisplaySnapshot`.
- Document that the function is the single entry point for the display page's initial state. The snapshot endpoint (Step 4) wraps this function with HTTP transport.

### Step 4: Create the Snapshot Endpoints

**Goal:** Expose the snapshot function via two GET endpoints: one for a specific board ID, one for the default board.

**Files affected:**
- `src/app/api/display-boards/snapshot/[boardId]/route.ts` (new file).
- `src/app/api/display-boards/snapshot/default/route.ts` (new file).

**Specific instructions:**

**`src/app/api/display-boards/snapshot/[boardId]/route.ts`**:

- `GET` handler.
- Reads `boardId` from route params.
- Calls `getDisplaySnapshot(boardId)`.
- If `board === null` (board not found), returns HTTP 404 with `{ success: false, error: { code: 'NOT_FOUND', message: 'Display board not found.' } }`.
- Otherwise, returns HTTP 200 with `{ success: true, data: snapshot }`.
- **No `withPermission` guard** ‚Äî the snapshot endpoint is public (the display page is public).
- Route exports: `export const runtime = 'nodejs'` (Prisma access) and `export const dynamic = 'force-dynamic'` (snapshot changes when the DB changes).

**`src/app/api/display-boards/snapshot/default/route.ts`**:

- `GET` handler.
- Calls `getDisplaySnapshot(null)` (resolves the default board).
- If `board === null` (no default board exists), returns HTTP 404 with `{ success: false, error: { code: 'NOT_FOUND', message: 'No default display board configured.' } }`.
- Otherwise, returns HTTP 200 with the snapshot.
- Same route exports as above.

Document that both endpoints are public and unauthenticated. The middleware allow-list (Step 9) ensures reachability; the route handlers return the data without checking sessions.

## 8. Common Pitfalls

1. **Rendering the `AudioContext` server-side.** The `AudioContext` is a browser API and is NOT available in Node.js. The overlay component MUST be a `"use client"` component that lazily creates the `AudioContext` on mount. If the agent tries to create it in a server component, the build will fail or the runtime will throw.
2. **Forgetting to add the `DisplayBoard` color tokens.** Without the tokens in `tailwind.config.ts`, the `text-display-accent` class will not compile, and the ticket number will be invisible (defaulting to black on black).
3. **Hydration mismatch on the clock.** If the server's `new Date()` differs from the client's `new Date()` (timezone, second drift), React will warn about hydration mismatch. Fix: use `suppressHydrationWarning` on the time element, or use `useState(() => new Date())` with a client-only first render.
4. **Not handling the missing default board.** If no `DisplayBoard` with `isDefault: true` exists, the page returns a 404 from the snapshot endpoint and renders nothing. The page must handle this gracefully (fallback message: "No display board configured").
5. **SSE subscription on the server.** The `useSSE` hook MUST be called from a client component. The `DisplayPageClient` is the client wrapper; the server component (`page.tsx`) only fetches the snapshot and passes it as a prop.
6. **Using `setInterval` without cleanup.** The clock and the broadcast banner countdown both use `setInterval`. Both MUST clean up the interval on unmount or React will warn about state updates on unmounted components.
7. **Forgetting the SSE filter.** The `useSSE` call MUST specify the `filter` parameter to only receive events the display board cares about. Without the filter, the page receives every event on the `global` channel (including future events from Phase 4) and the dispatcher's switch statement ignores unknown types ‚Äî wasteful but not buggy.
8. **Logging payload contents.** PII risk ‚Äî ticket numbers, officer names, etc. Log only `type`, `id`, `channel`, `clientCount`. Never log the full payload.
9. **Not memoizing the `DisplayTicketBlock`.** With 4-8 counters, re-rendering all blocks on every state change is wasteful. Wrap the component in `React.memo` so only the affected counter re-renders.
10. **Misnaming the snapshot endpoint path.** Next.js dynamic segments are URL-based. `/api/display-boards/snapshot/[boardId]` captures the boardId segment. If the agent uses a query parameter instead, the routing changes.
11. **Hard-coding the `maxDisplayedTickets` default to 10 in the reducer stubs.** The actual cap should come from `state.board.maxDisplayedTickets`. The stub uses `10` as a placeholder; 3.2.2 replaces it with the dynamic value.

### 9.1 Critical Implementation Notes

- **The display page is public.** No auth, no session, no permissions. The `/display` route is in the middleware allow-list.
- **The audio unlock overlay is the gate for the session.** While `unlocked = false`, the overlay covers the entire viewport. SSE events may arrive and update the in-memory state, but the visual is hidden. Once `unlocked = true`, the overlay dismisses and the buffered state is visible.
- **The `AudioContext` is created ONCE on mount.** Not on every render, not on every event. The context is held in a `useRef` and reused.
- **The `quems:audio-unlocked` custom event is the handoff to Phase 3.3.** Phase 3.3.1's bell system listens for this event and starts preloading the bell audio. The agent implementing this document does NOT need to know how 3.3 consumes the event ‚Äî just dispatch it.

- **The snapshot endpoint is the bridge between server-rendered initial state and client-side live updates.** The page fetches the snapshot server-side, hydrates the client state, then subscribes to SSE for live updates. There is no client-side initial fetch.
- **`useSSE('global', ...)` is the ONLY SSE subscription on the display page.** Do not create a second `useSSE` call for a different channel ‚Äî the display board only consumes `global`.
- **The display page does NOT call `broadcastEvent`.** The page is a consumer, not a producer.
- **The display page does NOT write to `AuditLog`.** The page is read-only.
- **The middleware allow-list must be updated BEFORE testing.** Without the allow-list, the middleware redirects unauthenticated users to `/login` and the display page is unreachable.
