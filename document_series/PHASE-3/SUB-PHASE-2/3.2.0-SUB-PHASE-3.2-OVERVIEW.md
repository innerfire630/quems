## 1. Sub-Phase Purpose

Sub-Phase 3.2 is the **visual public face** of the queue system. It builds the main display board — the full-screen, dark-themed, high-contrast view that customers in the waiting area see. The display board shows every active counter, the ticket number each counter is currently calling, a recent history of tickets for each counter, a scrolling marquee for custom messages, and a live clock. It is the first **consumer** of the real-time infrastructure built in Sub-Phase 3.1 — every ticket call action an officer takes at the counter (Phase 2.3) flows through the SSE manager (3.1.1) and the `useSSE()` hook (3.1.2) into this display board.

When this sub-phase is complete:

- A super-admin can create, edit, and delete `DisplayBoard` configurations through an admin UI.
- An operator opens `/display` (or `/display?boardId=xxx`) in a browser and sees a full-screen dark-themed display board with the audio context unlock overlay.
- After dismissing the unlock overlay, the display board subscribes to the `global` SSE channel via `useSSE()`.
- Every ticket call action at the counter updates the display board within a second — the affected counter's "now serving" changes, the new ticket number animates into place, and the recent calls history grows.
- Counter closure states are visually reflected. Recent calls history is maintained per counter.
- Configuration changes (theme color, logo, marquee message, TTS settings) take effect on page reload.

Three problems are solved in this sub-phase, in order:

1. **The display board page itself** — the full-screen layout, the audio context unlock overlay, the `DisplayTicketBlock` / `MarqueeMessage` / `DisplayClock` components, the initial state snapshot fetched on page load, and the SSE subscription that keeps everything live. (Document 3.2.1.)
2. **The multi-counter logic** — the responsive grid layout that adapts to 1, 2, 4, 8, or more active counters; the per-counter in-memory recent calls history buffer of the last N tickets; the transition animation on a new call; the counter closure state rendering; the recall and no-show handling for "now serving". (Document 3.2.2.)
3. **The display configuration system** — the `DisplayBoard` CRUD API endpoints, the admin UI at `/(dashboard)/settings/display/`, the URL scheme (`/display` vs `/display?boardId=xxx`), the `isDefault` invariant, and the configuration propagation. (Document 3.2.3.)

When this sub-phase is complete, **the public-facing real-time experience is half-built**: customers can SEE their ticket calls. Sub-Phase 3.3 (next) adds the audio dimension so they can HEAR their ticket calls.

## 1.1 Why This Sub-Phase Comes After 3.1

The display board is fundamentally a `useSSE()` consumer. It cannot render real-time events without the SSE infrastructure built in 3.1. The display board subscribes to the `global` channel and filters for the event types it cares about (`TICKET_CALLED`, `TICKET_RECALLED`, `TICKET_NO_SHOW`, `COUNTER_OPENED`, `COUNTER_CLOSED`, `DAILY_RESET`). Without the manager and the hook, this sub-phase would have nothing to render.

## 1.2 Why This Sub-Phase Comes Before 3.3

Sub-Phase 3.3 (the audio announcement system) is also a `useSSE()` consumer. It listens for the same events. Both 3.2 and 3.3 share the **audio context unlock overlay** (the operator clicks once to unlock audio for the session — both visual display and audio announcement need this). Sub-Phase 3.2.1 builds the unlock overlay as part of the display board page; Sub-Phase 3.3.1 wires the actual audio playback into that overlay's dismissal. Building 3.2 first ensures the unlock overlay exists and is testable in isolation before 3.3 adds the audio on top of it.

## 1.3 What This Sub-Phase Is NOT

- It is **not** the audio announcement system (bell, TTS, announcement queue). Phase 3.3.
- It is **not** push notifications (FCM). Phase 4.1.
- It is **not** counter temporary-closure toggle from the officer side. Phase 4.2.1. (Phase 3.2.2 DOES render closed counters on the display board, but the API for closing a counter is Phase 4.)
- It is **not** the officer dashboard layout. Phase 4.2.3.
- It is **not** the broadcast message creation UI (only the display-side render slot is reserved). Phase 4.3.
- It is **not** the security officer screen. Phase 4.3.3.
- It is **not** the reports dashboard. Phase 5.
- It is **not** rate limiting or PostgreSQL migration. Phase 5.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to distribute the complexity across the three task documents. The split is:

- **3.2.1** owns the **display board page and components**. It introduces the page itself, the audio context unlock overlay, the three display components (`DisplayTicketBlock`, `MarqueeMessage`, `DisplayClock`), the snapshot endpoint for initial state, the SSE subscription wiring, and the `BroadcastBanner` layout slot. It does **not** handle multi-counter grid breakpoints or the recent calls history buffer (that's 3.2.2) and does **not** build the `DisplayBoard` CRUD or admin UI (that's 3.2.3).
- **3.2.2** owns the **multi-counter display logic**. It introduces the responsive grid breakpoints, the per-counter in-memory recent calls history buffer, the transition animation, the counter closure state rendering, and the recall / no-show handling for "now serving". It depends on 3.2.1 because the components, page, and SSE subscription must exist.
- **3.2.3** owns the **display configuration system**. It introduces the `DisplayBoard` CRUD API endpoints, the admin UI for managing board configurations, the URL scheme, and the `isDefault` invariant. It depends on 3.2.1 because the display page reads the configuration (so the schema and API must be stable for the page to consume). It can be developed in parallel with 3.2.2 once 3.2.1 is complete.

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 3.2.1 — Display Board UI & Layout

**Owns:** The display page, the audio context unlock overlay, the three display components, the snapshot endpoint, the SSE subscription wiring, and the `BroadcastBanner` layout slot.

**Does NOT own:** The multi-counter grid breakpoints or recent calls history buffer (owned by 3.2.2), the `DisplayBoard` CRUD or admin UI (owned by 3.2.3), the actual audio playback (Phase 3.3).

**Key outputs:**

- `src/app/display/page.tsx` — the main display page (server component shell + client sub-components):
  - Reads `boardId` from `searchParams`. If absent, resolves the default board. If `boardId` is provided but the board is missing, falls back to default with a console warning.
  - Fetches the snapshot (initial state) via the snapshot endpoint described below.
  - Renders the page shell: top bar (48px tall) with brand / logo (left) and clock (right), main content area (counter grid, populated by 3.2.2), bottom marquee, and the `BroadcastBanner` overlay slot.
  - Subscribes to the `global` SSE channel via `useSSE()` with the appropriate event filter.
  - Holds the in-memory state: per-counter "now serving" ticket and recent calls history (the buffer logic is owned by 3.2.2; the page exposes the state hooks for 3.2.2 to fill in).
  - The full-screen audio context unlock overlay is rendered ABOVE all other content and blocks event-driven state updates until dismissed.
- `src/components/display/audio-unlock-overlay.tsx` — the full-screen overlay component:
  - Visible on first page load (state: `unlocked = false`).
  - Shows the brand / logo (if configured), a "Click anywhere to enable" message, and a visual hint (animated icon).
  - On click: calls `audioContext.resume()`, sets `unlocked = true`, dispatches a custom event so 3.3.1's bell system can subscribe to it.
  - Once dismissed, never shown again for the session (held in `useState`).
  - The overlay is the FIRST interactive element in the DOM order — focus traps inside it, but doesn't need keyboard navigation (the click is the unlock gesture).
- `src/components/display/display-ticket-block.tsx` — the per-counter ticket card:
  - Props: `{ counter: CounterDisplayData; nowServing: TicketDisplayData | null; recentTickets: TicketDisplayData[]; counterStatus: 'open' | 'closed' }`.
  - Renders the counter name and number at the top, the currently serving ticket number in large display typography (`font-size: 5rem`, `font-weight: 900`, `letter-spacing: -0.02em`, color `display-accent`), and the recent calls history below.
  - If `counterStatus = 'closed'`: applies a greyed-out style, shows "Temporarily Closed" label, hides the "now serving" slot (but the recent calls history remains visible).
  - If `nowServing === null`: shows a placeholder (e.g., "—") for the ticket number.
- `src/components/display/display-clock.tsx` — the live clock:
  - Props: none (reads time client-side via `useState` + `setInterval`).
  - Renders the current time (HH:MM:SS in 24-hour format) and the current date (e.g., "Monday, June 20, 2026") in the top bar's right side.
  - Updates every second.
  - Uses `Intl.DateTimeFormat` with `APP_TIMEZONE` for the locale-aware date.
- `src/components/display/marquee-message.tsx` — the scrolling custom message:
  - Props: `{ message: string | null }`.
  - If `message` is null or empty, renders nothing (no marquee, no reserved space).
  - Otherwise, renders a continuously-scrolling horizontal marquee using CSS animation (`@keyframes marquee-scroll` from `translateX(100%)` to `translateX(-100%)`).
  - Uses the `display-text` color on `display-bg` background.
  - The animation pauses on hover (so the operator can read the message).
- `src/components/display/broadcast-banner.tsx` — the broadcast message overlay slot:
  - Props: `{ message: string | null; senderName: string | null; displaySeconds: number | null }`.
  - If `message` is null, renders nothing.
  - Otherwise, renders a centered banner at the top of the main content area with the message text, sender name, and a countdown timer.
  - After `displaySeconds` elapses (default 10), the banner auto-dismisses.
  - This component is the layout slot for broadcast messages. Phase 4.3 will populate the `message` prop by subscribing to `BROADCAST_MESSAGE` events.
- **Snapshot endpoint:** `src/lib/display-snapshot.ts` — the server-side function that fetches the initial state:
  - `async function getDisplaySnapshot(boardId: string | null): Promise<DisplaySnapshot>` — fetches the board config, all active counters, and the latest CALLED ticket for each counter.
  - Called from the page's server component on first render.
  - Returns a snapshot with: `{ board: DisplayBoardConfig | null, counters: CounterDisplayData[], servingTickets: Map<counterId, TicketDisplayData> }`.
  - If no default board exists, the function returns `{ board: null, counters: [], servingTickets: new Map() }` and the page renders a fallback (e.g., "Display board not configured").
- **SSE subscription wiring:**
  - The page (or a top-level client component within it) calls `useSSE('global', { filter: ['TICKET_CALLED', 'TICKET_RECALLED', 'TICKET_NO_SHOW', 'COUNTER_OPENED', 'COUNTER_CLOSED', 'DAILY_RESET', 'BROADCAST_MESSAGE'], onEvent: (envelope) => { ... } })`.
  - Each event type triggers a specific state update on the page (3.2.1 implements the update dispatcher; 3.2.2 may extend the dispatcher with recent calls buffer logic).
- **Middleware allow-list extension:** the `/display` route is added to the public allow-list (no auth required). The display board is public — anyone in the waiting area can see it.
- **No `AuditLog` writes from any display path.** The display board is read-only; the only mutations are configuration changes (owned by 3.2.3), which DO write to `AuditLog`.

**Master Plan sections implemented:** Sections 6.2 (display color tokens), 6.3 (display typography), 6.4 (display layout — top bar, main section, bottom marquee), 6.5 (display components — `DisplayTicketBlock`, `BroadcastBanner`, `DisplayClock`, `MarqueeMessage`), 8.2 (reads `DisplayBoard` model), 9.3 (no new endpoints; the existing snapshot endpoint), 11 (event consumption on the `global` channel), 12.2 (audio context unlock overlay — visual layer only; audio playback is 3.3).

### 4.2 Document 3.2.2 — Multi-Counter Ticket Display Logic

**Owns:** The responsive grid layout, the per-counter recent calls history buffer, the transition animation on new calls, the counter closure state rendering, and the recall / no-show handling for "now serving".

**Does NOT own:** The page itself and the SSE subscription wiring (owned by 3.2.1), the `DisplayBoard` configuration (owned by 3.2.3), the audio announcements (Phase 3.3).

**Key outputs:**

- `src/lib/display-state.ts` — the in-memory state management helpers:
  - `type DisplayState` — the state shape held in the display page's React state: `{ counters: Map<counterId, CounterDisplayData>, nowServing: Map<counterId, TicketDisplayData>, recentByCounter: Map<counterId, TicketDisplayData[]>, counterStatus: Map<counterId, 'open' | 'closed'> }`.
  - `function applyTICKET_CALLED(state, envelope): DisplayState` — pure reducer that updates `nowServing` and prepends to `recentByCounter` (FIFO, capped at `maxDisplayedTickets`).
  - `function applyTICKET_RECALLED(state, envelope): DisplayState` — same as `applyTICKET_CALLED` (the recall keeps the same ticket in "now serving"; the audio system re-plays; visually, the recent calls history gets a duplicate entry marked by the recall — or alternatively, recall is a no-op for the visual state and the recent calls history does not get a duplicate). The document specifies the chosen behavior: recall keeps the same ticket in "now serving" but does NOT add a duplicate to recent calls (the recall is the same call being re-announced).
  - `function applyTICKET_NO_SHOW(state, envelope): DisplayState` — pure reducer that removes the ticket from `nowServing`. If the event payload indicates `autoAdvanced = true` and provides `autoAdvancedTicketNumber`, the next waiting ticket is fetched and becomes the new "now serving" (the document specifies: the no-show event payload from 2.3.2 includes the auto-advanced ticket number, so the display board can show the new "now serving" immediately without a separate API call).
  - `function applyCOUNTER_OPENED(state, envelope): DisplayState` — sets the counter status to `'open'`.
  - `function applyCOUNTER_CLOSED(state, envelope): DisplayState` — sets the counter status to `'closed'`.
  - `function applyDAILY_RESET(state, envelope): DisplayState` — clears all `nowServing` and `recentByCounter` entries. The counters map remains (the counter definitions don't change at reset; only the queue state does).
- `src/components/display/display-counter-grid.tsx` — the responsive grid container:
  - Props: `{ counters: CounterDisplayData[]; children: <DisplayTicketBlock /> per counter }`.
  - Applies a CSS grid layout with breakpoint-driven column counts:
    - 1-2 counters: 2 columns.
    - 3-4 counters: 2 columns.
    - 5-8 counters: 3 columns.
    - 9+ counters: 4 columns.
  - Uses Tailwind responsive classes (`grid-cols-2 md:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4` based on counter count) or a conditional render of the right class.
  - Each grid cell is a `<DisplayTicketBlock>` populated from the display state.
- `src/components/display/transition-wrapper.tsx` — the animation wrapper:
  - Wraps the "now serving" slot of each counter card.
  - On a state change to `nowServing` for a given counter (via React's `key` prop on the ticket number element), applies a brief scale-up / fade-in animation (~300ms).
  - Uses CSS `transition: transform 300ms ease-out, opacity 300ms ease-out` and `animation` keyframes.
- `src/components/display/recent-calls-list.tsx` — the recent calls history list:
  - Props: `{ tickets: TicketDisplayData[]; maxItems: number }`.
  - Renders a scrollable list (or fixed-height list, no scroll) of the last N tickets, ordered most-recent-first.
  - If the list exceeds `maxItems`, only the most recent `maxItems` are rendered.
  - Each item shows the ticket number (smaller than "now serving" — `font-size: 1.5rem`, `font-weight: 700`) and a small timestamp (relative time via `Intl.RelativeTimeFormat`).
- **Counter closure visual state:**
  - When `counterStatus[counterId] === 'closed'`, the `DisplayTicketBlock` component (built in 3.2.1) receives `counterStatus: 'closed'` and applies a greyed-out style (opacity 0.5), shows a "Temporarily Closed" label, hides the "now serving" slot, and keeps the recent calls history visible.
  - The `applyCOUNTER_CLOSED` reducer (from `display-state.ts`) sets the status. The actual `COUNTER_CLOSED` event is emitted by Phase 4.2.1's counter status toggle API; for Phase 3.2 testing, the agent manually sets a counter's status in the DB and verifies the rendering.
- **Recall visual handling:**
  - `applyTICKET_RECALLED` keeps the same ticket in `nowServing`. The "now serving" slot shows the same ticket number — no change in the visual.
  - The recent calls history does NOT get a duplicate entry (recall is a re-announcement of the same ticket, not a new ticket call).
  - The audio system (3.3) re-plays the announcement.
- **No-show visual handling:**
  - `applyTICKET_NO_SHOW` removes the ticket from `nowServing`.
  - If `autoAdvanced = true` (from the event payload), the auto-advanced ticket becomes the new `nowServing`. The `TICKET_NO_SHOW` payload from 2.3.2 includes `autoAdvancedTicketNumber` (string | null), so the display board can render the new "now serving" immediately.
  - If `autoAdvanced = false` or no auto-advance occurred, `nowServing` becomes `null` (placeholder "—").
- **Buffer in-memory only:** the recent calls history is held in React state and initialized from the snapshot. It is NOT persisted across page reloads. (The snapshot endpoint on page load fetches the most recent CALLED tickets from the database, so a refresh restores the history within the limits of what the DB query returns.)
- **Snapshot fetch for recent history on page load:** the snapshot endpoint (from 3.2.1) is extended to also return the last N CALLED tickets per counter, where N is `DisplayBoard.maxDisplayedTickets`. The query: for each active counter, fetch the most recent N `TicketEvent` records of type `CALLED` on the current business date, sorted by timestamp descending.

**Master Plan sections implemented:** Sections 6.4 (grid layout), 6.5 (display components), 8.2 (reads `DisplayBoard.maxDisplayedTickets`), 11 (event consumption).

### 4.3 Document 3.2.3 — Display Configuration & Customization

**Owns:** The `DisplayBoard` CRUD API endpoints, the admin UI for managing display board configurations, the URL scheme handling on the display page, the `isDefault` invariant enforcement, and the audit logging for configuration changes.

**Does NOT own:** The display page itself (owned by 3.2.1), the multi-counter grid logic (owned by 3.2.2), the audio system (Phase 3.3).

**Key outputs:**

- `src/app/api/display-boards/route.ts` — list and create endpoints:
  - `GET /api/display-boards` — lists all display boards. Guarded with `withPermission(handler, PERMISSION_SYSTEM_CONFIGURE)`. Returns `{ success: true, data: DisplayBoard[] }`. Pagination not required for Phase 3 (the count is expected to be small, < 20).
  - `POST /api/display-boards` — creates a new display board. Guarded with `withPermission(handler, PERMISSION_SYSTEM_CONFIGURE)`. Validates the body with `displayBoardCreateSchema`. On `isDefault: true`, the handler unsets the previous default in the same transaction. Returns the created board.
- `src/app/api/display-boards/[boardId]/route.ts` — detail, update, delete endpoints:
  - `GET /api/display-boards/[boardId]` — returns the board. Guarded with `withPermission(handler, PERMISSION_SYSTEM_CONFIGURE)`.
  - `PATCH /api/display-boards/[boardId]` — updates the board. Guarded with `withPermission(handler, PERMISSION_SYSTEM_CONFIGURE)`. Validates with `displayBoardUpdateSchema`. On `isDefault: true`, the handler unsets the previous default in the same transaction. Returns the updated board.
  - `DELETE /api/display-boards/[boardId]` — soft-deletes the board (sets `isActive: false` or removes the record — the document specifies which). The document chooses hard delete (the count is small and there's no FK from other tables; deleted boards cannot be referenced by `/display?boardId=xxx` — the page falls back to default with a console warning). Guarded with `withPermission(handler, PERMISSION_SYSTEM_CONFIGURE)`. Returns `{ success: true, data: null }`.
- `src/app/api/display-boards/snapshot/[boardId]/route.ts` — the snapshot endpoint consumed by the display page (3.2.1):
  - `GET /api/display-boards/snapshot/[boardId]` — returns the display snapshot. NOT guarded by `withPermission` (the display page is public). Instead, the endpoint reads the board by ID and returns the snapshot. If the board doesn't exist, returns 404 and the page falls back to default.
  - Also: `GET /api/display-boards/snapshot/default` (or just `GET /api/display-boards/snapshot` with no ID) — returns the snapshot for the default board.
  - Document specifies whether the snapshot is one endpoint with optional ID or two endpoints. The chosen approach: `GET /api/display-boards/snapshot/[boardId]` for specific; `GET /api/display-boards/snapshot/default` for default. Both are public.
- `src/schemas/display-board.schema.ts` — Zod schemas for the board payloads:
  - `displayBoardCreateSchema` — all fields from the `DisplayBoard` model except `id`, `createdAt`, `updatedAt`. Optional `isDefault` (defaults to `false`).
  - `displayBoardUpdateSchema` — same as create, but all fields optional (PATCH semantics).
  - Validation rules:
    - `name` is required, min 1, max 100 chars.
    - `maxDisplayedTickets` is integer, min 1, max 50 (reasonable upper bound).
    - `ttsLanguage` is a valid BCP-47 tag (regex check or use a library).
    - `ttsRate`, `ttsPitch` are floats, ranges per master plan Section 8.2 (`ttsRate: 0.1-10`, `ttsPitch: 0-2`).
    - `ttsVolume` is float, range 0-1.
    - `announcementTemplate` is required, max 500 chars.
    - `themeColor` is a valid hex color (e.g., `#3B82F6`).
    - `logoUrl` is a valid URL (if present).
- `src/app/(dashboard)/settings/display/page.tsx` — the list page:
  - Server component that fetches all boards and renders them in a table (using shadcn `Table`).
  - Columns: name, isDefault, maxDisplayedTickets, ttsLanguage, themeColor, actions (edit, delete, set as default).
  - "Create Display Board" button (links to `/settings/display/new`).
  - Empty state if no boards exist (with a CTA to create one).
- `src/app/(dashboard)/settings/display/new/page.tsx` — the create page:
  - Renders the `DisplayBoardForm` component (described below) with `mode: 'create'`.
- `src/app/(dashboard)/settings/display/[boardId]/page.tsx` — the edit page:
  - Server component that fetches the board and renders the `DisplayBoardForm` in `mode: 'edit'` with initial values.
- `src/components/admin/display-board-form.tsx` — the form component:
  - "use client" component.
  - Props: `{ mode: 'create' | 'edit'; initialValues?: DisplayBoard; boardId?: string }`.
  - All fields from the `DisplayBoard` model (with appropriate input types — number inputs for numerics, color picker for `themeColor`, switch for booleans, select for `ttsLanguage`, textarea for `announcementTemplate`).
  - On submit: calls the appropriate API endpoint (`POST /api/display-boards` for create, `PATCH /api/display-boards/[boardId]` for edit).
  - On success: redirects to `/settings/display` (list) with a success toast.
  - On error: shows the error message in a toast.
  - The `isDefault` switch is labelled clearly: "Set as default display board". When toggled on, a warning text appears: "This will unmark the current default board."
- **Audit log writes:**
  - `createDisplayBoard` → `AuditLog` with `action: 'DISPLAY_BOARD_CREATED'`, `entity: 'DisplayBoard'`, `entityId: <new board id>`, `description: 'Created display board "<name>"'`.
  - `updateDisplayBoard` → `DISPLAY_BOARD_UPDATED` with the diff in metadata.
  - `deleteDisplayBoard` → `DISPLAY_BOARD_DELETED`.
  - `setAsDefault` → `DISPLAY_BOARD_DEFAULT_CHANGED`.
  - These action types are added to the `AuditAction` union by this document (additive extension).
- **URL scheme handling** (referenced from 3.2.1, implemented in this document):
  - `/display` (no `boardId`): the page server component calls the snapshot endpoint with `default` and renders the default board.
  - `/display?boardId=xxx`: the page server component reads `searchParams.boardId`, calls the snapshot endpoint for that board. If 404, falls back to default with a console warning.
- **`isDefault` invariant enforcement:**
  - On create with `isDefault: true`: in the same transaction, set all other boards' `isDefault` to `false` before setting the new board's `isDefault` to `true`.
  - On update with `isDefault: true`: same logic — unset the previous default.
  - On update with `isDefault: false` (and the board was previously default): the board loses its default status. The next request for the default board will return 404 (no default). The display page handles this by showing a "No default display board configured" fallback.
  - On delete: if the deleted board was default, the next request for default returns 404 (no fallback). The display page's fallback message is the same.
  - **Edge case:** deleting the only board that was default leaves the system with no default. The document specifies: deletion of the default board is allowed, but a warning is shown in the UI ("You are deleting the default board. The display will fall back to 'no configuration' until a new default is set."). The deletion is logged with `DISPLAY_BOARD_DELETED` AND `DISPLAY_BOARD_DEFAULT_CHANGED` (the default changed from this board to null).
- **No real-time config push:** configuration changes take effect on the next page load. There is no live push of config updates via SSE. (The display page reads config on mount.)
- **No `display:manage` permission:** the configuration is guarded by the existing `PERMISSION_SYSTEM_CONFIGURE` (super-admin only). No new permission is added.

### 6.7 Audio Context Discipline

The audio context unlock is built in 3.2.1 but the actual audio playback is 3.3.1. The discipline:

- The `AudioContext` is created once on the display page's mount and held in a `useRef` (or React state).
- The context starts in `suspended` state.
- The unlock overlay's click handler calls `audioContext.resume()` and dispatches a custom event (`window.dispatchEvent(new CustomEvent('quems:audio-unlocked'))`).
- After resumption, the context remains in `running` state. The display page (or 3.3.1's bell system) listens for the `audio-unlocked` event and proceeds with audio playback.
- The SSE event rendering (the display state updates) is gated on the unlock state. While `unlocked = false`, events are buffered (in the React state setters) but not rendered to the visible UI — the unlock overlay covers everything. Once `unlocked = true`, the overlay dismisses and the buffered state becomes visible. (Alternative: the events are received and processed during the locked phase, but the UI is hidden by the overlay — simpler and acceptable.)
- The unlock gesture is one-time per page load. The `AudioContext` is not re-suspended after the initial unlock (modulo browser behavior on tab backgrounding).

### 6.8 Error Handling Standards

- The display page itself does not throw to the user. SSE connection errors are handled by the `useSSE()` hook (returns a `status` state). The page may render a small "Reconnecting..." indicator when `status !== 'open'`.
- The snapshot endpoint returns 404 if the requested board doesn't exist. The display page handles 404 by falling back to the default board (or showing a "No display board configured" message if no default exists).
- The admin endpoints follow the standard envelope and error codes (401 for no session, 403 for no permission, 422 for validation errors, 404 for missing board, 500 for unexpected errors).
- The `DisplayBoardForm` component shows server-side errors via shadcn `Toast`.

### 6.9 Audit Logging Standard (Sub-Phase 3.2 Specific)

- **Display board mutations write `AuditLog` rows.** This is the ONLY path in 3.2 that writes to `AuditLog`.
  - `DISPLAY_BOARD_CREATED` — new action type added to `AuditAction` union.
  - `DISPLAY_BOARD_UPDATED` — new action type.
  - `DISPLAY_BOARD_DELETED` — new action type.
  - `DISPLAY_BOARD_DEFAULT_CHANGED` — new action type (logged when the default flag is toggled, including when it changes from a deleted board).
- **SSE consumption does NOT write to `AuditLog`.** The display board is a high-frequency reader; no audit log.
- **Display board read paths do NOT write to `AuditLog`.** The snapshot endpoint, the display page itself — no audit log.

### 6.10 Broadcast Event Seam (Sub-Phase 3.2 Specific)

- 3.2 does NOT emit any events. The display board is read-only. All event emission is owned by Phase 2 and Phase 4 code paths.
- 3.2 consumes events from the `global` channel via `useSSE()`. The event types it filters for:
  - `TICKET_CALLED` (update "now serving" + recent calls history).
  - `TICKET_RECALLED` (no visual change to "now serving"; audio re-play is 3.3's responsibility).
  - `TICKET_NO_SHOW` (remove from "now serving"; if auto-advanced, set new "now serving").
  - `COUNTER_OPENED` (counter status to `'open'`).
  - `COUNTER_CLOSED` (counter status to `'closed'`).
  - `DAILY_RESET` (clear all "now serving" and recent calls history).
  - `BROADCAST_MESSAGE` (populate the `BroadcastBanner` overlay).
- The `broadcastRoutedEvent()` helper (from 3.1.3) is the preferred call for any future code path that needs to emit these events. The display board itself does NOT call `broadcastRoutedEvent` or `broadcastEvent`.

### 6.11 Environment Variable Standards

- No new environment variables are introduced by this sub-phase.
- `APP_TIMEZONE` (introduced in 2.2.1) is consumed by the `DisplayClock` for locale-aware date/time formatting.

### 6.12 Transaction Boundaries

- The `isDefault` invariant logic (in 3.2.3) runs inside a Prisma `$transaction`:
  1. Set the previous default board's `isDefault` to `false` (if `isDefault: true` is being set on a new/updated board).
  2. Set the target board's `isDefault` to `true`.
  3. Insert the audit log row (the audit log helper is best-effort; if the audit log write fails inside the transaction, the transaction is NOT rolled back — the document specifies this is acceptable because the audit log is for forensics, not enforcement).
- All `DisplayBoard` mutations are atomic with respect to the default invariant.

### 6.13 Commit Standards

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`.
- Each task plan document corresponds to one or more commits.
- Suggested commit boundaries:
  - End of 3.2.1: `feat: add display board page with audio unlock overlay and SSE subscription`.
  - End of 3.2.2: `feat: add multi-counter grid layout and recent calls history buffer`.
  - End of 3.2.3: `feat: add DisplayBoard CRUD API and admin UI with isDefault invariant`.
