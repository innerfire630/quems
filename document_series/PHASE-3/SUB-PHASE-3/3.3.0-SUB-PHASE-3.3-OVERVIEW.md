## 1. Sub-Phase Purpose

Sub-Phase 3.3 is the **audible public face** of the queue system. When a ticket is called, the system plays a bell chime followed by a Text-to-Speech (TTS) announcement that names the ticket number and counter. The announcement system is what makes the queue useful in a noisy waiting area where the display board alone is not enough — for customers whose attention is not on the screen, for visually impaired customers, and for environments where sound is the primary notification channel.

Sub-Phase 3.3 closes Phase 3 by adding the audio dimension to the visual display built in Sub-Phase 3.2. The display board and the audio system are tightly coupled — both are consumers of the same `global` SSE events, both share the audio context unlock overlay (the operator clicks once to enable audio for the session — both the visual page and the audio system become live from that moment onward), and both are mounted on the same `/display` page.

When this sub-phase is complete:

- A customer sitting in the waiting area hears the bell chime followed by the spoken announcement within a fraction of a second of seeing the ticket number appear on the display board.
- Multiple ticket calls arriving close together are queued and played in strict bell → TTS sequence, one after another, with no overlap.
- The TTS announcement respects the configured template, language, rate, pitch, and volume from `DisplayBoard`.
- Disabling the bell, the TTS, or all announcements via the configuration correctly skips the corresponding audio.
- Navigating away from the display board mid-announcement cancels the in-progress audio cleanly.

Three problems are solved in this sub-phase, in order:

1. **The bell sound** — the Web Audio API integration for playing `bell.mp3` from `public/sounds/`, with pre-decoded `AudioBuffer` for low-latency playback, `AudioContext` lifecycle management, and the audio context unlock pattern from 3.2.1. (Document 3.3.1.)
2. **The TTS announcement** — the `SpeechSynthesis` API integration for playing the spoken announcement, voice selection based on the configured BCP-47 language tag, template substitution (`{number}`, `{counter}`, `{service}`), and the `utterance.onend` promise that signals completion. (Document 3.3.2.)
3. **The announcement queue and sequencing** — the `useAnnouncement()` React hook that ties the SSE event subscription to the audio playback, FIFO queue with single-flight guard, strict bell → TTS sequence, cancellation on unmount, bounded queue size, and the wiring into the display page. (Document 3.3.3.)

When this sub-phase is complete, **Phase 3 is finished**. The system has a working public display board (customers SEE ticket calls) and a working audio announcement system (customers HEAR ticket calls). Sub-Phase 3.3 is the last piece of the public-facing real-time experience.

## 1.1 Why This Sub-Phase Comes After 3.2

The audio system is a `useSSE()` consumer (just like 3.2). It listens for `TICKET_CALLED` and `TICKET_RECALLED` events on the `global` channel and triggers the announcement sequence. But the audio system also depends on:

- The **audio context unlock overlay** built in 3.2.1 — the overlay is the operator's gesture that resumes the `AudioContext` from `suspended` to `running`. Without this overlay (and the `quems:audio-unlocked` custom event it dispatches), the audio system cannot play anything because the browser will block the playback.
- The **`DisplayBoard` configuration model** built in 3.2.3 — the bell/TTS/announcement enable flags, the TTS language, rate, pitch, volume, and the announcement template are all fields on `DisplayBoard`. The audio system reads these from the snapshot fetched by the display page.
- The **`DisplayPageClient` component** built in 3.2.1 — the announcement hook is consumed by this component (the same client component that renders the visual display). The hook is mounted alongside the SSE subscription and the display state reducers.

Sub-Phase 3.2.1 builds the page shell, 3.2.3 builds the configuration model that 3.3 reads from, and the audio context unlock overlay is the gate that 3.3 hooks into.

## 1.2 Why This Sub-Phase Closes Phase 3

Phase 3 is the "real-time display and audio" phase. Without audio, the system has a working display but is incomplete — half the value proposition (the audible call-to-action) is missing. With audio, the system provides a complete real-time experience.

Sub-Phase 3.3 is also a non-trivial piece of engineering — the browser audio context lifecycle, the Web Audio API quirks for pre-decoded buffers, the TTS API quirks for voice loading across browsers, and the announcement queue with strict sequencing are all real engineering problems that deserve their own sub-phase. Splitting them out from 3.2 keeps 3.2 focused on the visual rendering and 3.3 focused on the audio playback.

## 1.3 What This Sub-Phase Is NOT

- It is **not** push notifications (FCM). Phase 4.1.
- It is **not** the officer dashboard layout. Phase 4.2.3.
- It is **not** the broadcast message creation API or the security officer screen. Phase 4.3.
- It is **not** the reports dashboard. Phase 5.
- It is **not** rate limiting, audit log viewing, or PostgreSQL migration. Phase 5.
- It is **not** an "announcement from the officer side". The audio system is exclusively a consumer on the `/display` page — officers do not trigger announcements; ticket call actions trigger them automatically.

### 2.3 The "Distributed Document" Principle

The user has explicitly asked to distribute the complexity across the three task documents. The split is:

- **3.3.1** owns the **bell sound**. It introduces the Web Audio API integration, the `AudioBuffer` preload strategy, the `AudioContext` lifecycle (including the unlock listener that bridges from 3.2.1's overlay), the `playBell()` promise-returning function, and the volume / enable flag handling. It does **not** know about TTS, the announcement queue, or SSE events — it just plays a chime when asked.
- **3.3.2** owns the **TTS announcement**. It introduces the `SpeechSynthesis` API integration, the voice selection logic, the template substitution, the `speakAnnouncement()` promise-returning function, and the `onend` / `onerror` handlers. It does **not** know about the bell, the queue, or SSE events — it just speaks a string when asked.
- **3.3.3** owns the **announcement queue and sequencing**. It introduces the `useAnnouncement()` React hook, the FIFO queue, the single-flight processor that chains bell → TTS via promises, the SSE event subscription, the cancellation logic, and the integration into `DisplayPageClient`. It depends on 3.3.1 (for `playBell`) and 3.3.2 (for `speakAnnouncement`).

An AI agent picking up any one of these three documents should be able to complete it independently, knowing only that the previous documents in the dependency chain are done.

## 4. Document Breakdown

### 4.1 Document 3.3.1 — Bell/Chime Audio Integration

**Owns:** The Web Audio API integration for bell playback, the pre-decoded `AudioBuffer` strategy, the `AudioContext` lifecycle (including the bridge from 3.2.1's unlock overlay), the `playBell()` promise-returning function, the volume and enable flag handling, and the `useAudioUnlock` listener hook.

**Does NOT own:** The TTS announcement (owned by 3.3.2), the announcement queue (owned by 3.3.3), the SSE subscription (owned by 3.3.3).

**Key outputs:**

- `src/lib/audio-bell.ts` — the Web Audio API integration:
  - `async function loadBellBuffer(audioContext: AudioContext): Promise<AudioBuffer>` — fetches `/sounds/bell.mp3`, decodes into an `AudioBuffer` via `audioContext.decodeAudioData()`. Caches the result at module scope so subsequent calls return the cached buffer.
  - `async function playBell({ audioContext, buffer, volume, onEnded }): Promise<void>` — creates an `AudioBufferSourceNode` from the buffer, sets its volume via a `GainNode`, connects it to the destination, starts playback, and returns a Promise that resolves when the `onended` event fires.
  - `function isAudioContextRunning(audioContext: AudioContext): boolean` — helper that returns `audioContext.state === 'running'`. Used by 3.3.3 to decide whether to resume before playing.
  - `async function ensureAudioContextRunning(audioContext: AudioContext): Promise<void>` — if `audioContext.state === 'suspended'`, calls `audioContext.resume()`. Logs a warning if resume fails.
- `src/hooks/use-audio-unlock.ts` — the listener for the unlock event:
  - A "use client" hook that uses `useEffect` to add a `window.addEventListener('quems:audio-unlocked', handler)` on mount.
  - The handler is invoked when the operator dismisses the audio context unlock overlay (built in 3.2.1).
  - Returns a boolean `isAudioUnlocked` state that the consumer can use to gate audio playback.
  - Cleanup: removes the event listener on unmount.
- `public/sounds/bell.mp3` — referenced and validated; the file was placed by 2.2.3 as a placeholder (valid MP3, possibly silent). This document verifies the file exists and is a valid audio file. If the agent is replacing it with a real chime, this is the time to do so (the document specifies a 1-2 second chime with minimal file size).
- **AudioBuffer caching:** the decoded `AudioBuffer` is held in a module-scope variable so it is loaded once per page session. Subsequent `playBell()` calls reuse the cached buffer.
- **AudioContext reuse:** the `AudioContext` instance is created once (passed in as a parameter) and reused. The audio context unlock from 3.2.1 establishes the `running` state; the bell system relies on that state being `running` for the duration of the page session.
- **Volume control:** the bell volume reads `DisplayBoard.ttsVolume` (master volume for all announcement audio). The bell is played at `Math.max(0, Math.min(1, ttsVolume))` — clamped to [0, 1] per the Web Audio API.
- **Bell enable flag:** if `DisplayBoard.bellEnabled === false` or `DisplayBoard.announcementEnabled === false`, `playBell()` returns immediately (resolves to `void`) without playing any sound. The announcement queue (3.3.3) is responsible for checking this flag before calling `playBell()`.
- **No SSE subscription in 3.3.1.** The bell system is event-agnostic — it just plays a sound when asked. The SSE subscription is owned by 3.3.3.

**Master Plan sections implemented:** Sections 12.1 (bell part of the trigger sequence), 12.2 (browser audio unlock — the visual overlay is 3.2.1; the audio context lifecycle is 3.3.1).

### 4.2 Document 3.3.2 — Browser TTS API Integration

**Owns:** The `SpeechSynthesis` API integration, the announcement template substitution, the voice selection logic, the `speakAnnouncement()` promise-returning function, the utterance event handlers, the TTS configuration flags, and the browser quirk handling.

**Does NOT own:** The bell (owned by 3.3.1), the announcement queue (owned by 3.3.3), the SSE subscription (owned by 3.3.3).

**Key outputs:**

- `src/lib/audio-tts.ts` — the TTS integration:
  - `function substituteTemplate(template: string, placeholders: { number, counter, service }): string` — replaces `{number}`, `{counter}`, `{service}` placeholders. Unknown placeholders are left as-is (so admins can see they made a typo).
  - `async function loadTtsVoices(): Promise<SpeechSynthesisVoice[]>` — returns a Promise that resolves when `speechSynthesis.getVoices()` returns the available voices. Resolves immediately if voices are already loaded; otherwise waits for the `voiceschanged` event with a 3-second timeout fallback.
  - `function selectVoice(voices: SpeechSynthesisVoice[], language: string): SpeechSynthesisVoice | null` — finds the first voice matching the BCP-47 tag (exact match preferred, then language-only fallback). Logs a warning if no match.
  - `async function speakAnnouncement({ text, language, rate, pitch, volume, onStart?, onError? }): Promise<void>` — constructs a `SpeechSynthesisUtterance`, applies the configuration, sets `onstart` / `onend` / `onerror` handlers, calls `speechSynthesis.speak(utterance)`, and returns a Promise that resolves on `onend` or rejects on `onerror`.
  - `function isTtsAvailable(): boolean` — helper that returns `typeof window !== 'undefined' && 'speechSynthesis' in window`. Used by 3.3.3 to skip TTS on browsers that don't support it.
  - `function cancelTts(): void` — calls `speechSynthesis.cancel()`. Used by 3.3.3's cleanup logic.
- **TTS enable flag:** if `DisplayBoard.ttsEnabled === false` or `DisplayBoard.announcementEnabled === false`, `speakAnnouncement()` returns immediately (resolves to `void`) without speaking. The announcement queue (3.3.3) is responsible for checking this flag before calling `speakAnnouncement()`.
- **TTS config from `DisplayBoard`:** `ttsLanguage`, `ttsRate` (clamped to 0.1-10), `ttsPitch` (clamped to 0-2), `ttsVolume` (clamped to 0-1), `announcementTemplate` (default: "Now serving ticket {number} at {counter}"). All values are read from the snapshot on page load; configuration changes require a page reload (consistent with the 3.2 display board config discipline).
- **Browser quirk handling:**
  - Chrome / Edge (desktop): voices populate synchronously, full support.
  - Safari (desktop): voices may populate asynchronously; the `voiceschanged` event handler is essential.
  - Firefox: TTS support varies by platform; the system gracefully skips if `speechSynthesis` is unavailable.
  - Mobile browsers: TTS may be limited; the system handles missing voices by falling back to the first available voice or skipping TTS gracefully (only the bell plays).
- **No SSE subscription in 3.3.2.** The TTS system is event-agnostic — it just speaks a string when asked. The SSE subscription is owned by 3.3.3.

**Master Plan sections implemented:** Sections 8.2 (DisplayBoard TTS settings), 12.1 (TTS part of the trigger sequence), 12.3 (TTS template and placeholders).

### 4.3 Document 3.3.3 — Announcement Queue & Sequencing Logic

**Owns:** The `useAnnouncement()` React hook, the FIFO announcement queue, the single-flight processor, the strict bell → TTS sequence enforcement, the SSE event subscription for the announcement trigger, the cancellation logic on unmount, the queue size bound, and the integration into `DisplayPageClient`.

**Does NOT own:** The bell (owned by 3.3.1), the TTS (owned by 3.3.2), the visual display (owned by 3.2.1), the SSE manager and `useSSE()` hook (owned by 3.1).

**Key outputs:**

- `src/hooks/use-announcement.ts` — the React hook:
  - Signature: `useAnnouncement({ displayBoard, audioContext, isAudioUnlocked }): void` — runs side effects only, no return value.
  - **SSE subscription:** uses `useSSE('global', { filter: ['TICKET_CALLED', 'TICKET_RECALLED'], onEvent: (envelope) => { ... enqueue ... } })`.
  - **Queue state:** `const [queue, setQueue] = useState<AnnouncementJob[]>([])` where `AnnouncementJob` is `{ envelope: SseEnvelope, ticketDisplay: TicketDisplayData }` (or a similar denormalized shape).
  - **Enqueue behavior:** on each `TICKET_CALLED` / `TICKET_RECALLED` event, build the announcement text (template substitution), append to the queue. If the queue exceeds `MAX_ANNOUNCEMENT_QUEUE_SIZE` (default 5), drop the oldest entry with a `console.warn`.
  - **Processor:** an async function `processQueue` that picks the first item from the queue, plays the bell (calls `playBell()` from 3.3.1), waits for completion, then speaks the announcement (calls `speakAnnouncement()` from 3.3.2), waits for completion, then recurses. Uses a `useRef` flag `isProcessingRef` as the single-flight guard.
  - **Strict sequence:** the processor `await`s `playBell()` before calling `speakAnnouncement()`. No overlap, no skipping, no reordering.
  - **Cancellation on unmount:** the hook's `useEffect` cleanup function calls `speechSynthesis.cancel()`, clears the queue state, and unsubscribes from SSE (handled by `useSSE`'s own cleanup).
  - **Audio context gating:** the processor only runs if `isAudioUnlocked === true`. While audio is locked, events are received but the queue is not processed (the events would be lost otherwise; the queue persists, and when unlock happens, the queued events are processed in FIFO order).
  - **Display board config gating:** the hook reads `displayBoard.announcementEnabled`, `displayBoard.bellEnabled`, `displayBoard.ttsEnabled`, `displayBoard.announcementTemplate`, `displayBoard.ttsLanguage`, `displayBoard.ttsRate`, `displayBoard.ttsPitch`, `displayBoard.ttsVolume` from the prop. If `announcementEnabled === false`, the hook does nothing (events are filtered out at the SSE level or no-op'd at the queue level).
- **Integration into `DisplayPageClient`:** the existing component from 3.2.1 is modified to:
  - Import `useAnnouncement` from `@/hooks/use-announcement`.
  - Pass `displayBoard` (the board config from the snapshot), `audioContext` (the `AudioContext` from the unlock overlay), and `isAudioUnlocked` (a new state variable).
  - The hook is called alongside the existing SSE subscription (the visual one) — both subscribe to `global` with different filters (visual: `['TICKET_CALLED', 'TICKET_RECALLED', 'TICKET_NO_SHOW', 'COUNTER_OPENED', 'COUNTER_CLOSED', 'DAILY_RESET', 'BROADCAST_MESSAGE']`; audio: `['TICKET_CALLED', 'TICKET_RECALLED']`).
- **No new server endpoints.** The audio system is entirely client-side — no API routes are added in 3.3.3.
- **No `AuditLog` writes.** The audio system is a consumer; playback is not a user action that needs audit logging.

**Master Plan sections implemented:** Section 12.1 (full announcement sequence — bell, TTS, queue, sequencing), 6.5 (`useAnnouncement` hook in the hooks inventory).

```
   │
   └── 3.3.3  Announcement Queue & Sequencing Logic

```

**Build order:** `3.3.1` first (the bell audio is the simplest of the three and provides the foundation that 3.3.2 reuses the audio context for). `3.3.2` second (TTS uses the same audio context lifecycle as the bell but is a separate API surface). `3.3.3` last (depends on both 3.3.1 and 3.3.2 to orchestrate the announcement sequence).

**Why this order:**

- **3.3.1 first.** The bell audio is independent of TTS — it only needs the `AudioContext` and the unlock event. Building it first establishes the audio context lifecycle pattern that 3.3.2 reuses.
- **3.3.2 after 3.3.1.** The TTS announcement uses the same `AudioContext` lifecycle (the `SpeechSynthesis` API does not directly use `AudioContext`, but the unlock pattern applies conceptually — the audio system is unlocked as a whole). Building TTS after the bell means the audio context discipline is already established.
- **3.3.3 last.** The queue and sequencing depend on both the bell and TTS being available. The queue is the orchestration layer.

**Parallel opportunity:** `3.3.1` and `3.3.2` could theoretically be developed in parallel because they are independent API integrations (Web Audio API vs SpeechSynthesis API). However, the audio context lifecycle is shared, so it is cleaner to do them sequentially. The agent develops them in the specified order.

**What depends on this sub-phase:**

- **Phase 4.2.3 (officer dashboard):** if the officer dashboard wants audible call notifications (the master plan doesn't explicitly require this, but the option is available), it can reuse `useAnnouncement()` from this sub-phase. The hook is reusable; the snapshot configuration is per-component.
- **Phase 5.1 (reports):** the audio system does not write to any analytics tables — it is purely client-side. The reports dashboard queries `QueueDailySnapshot`, not the announcement hook.

### 6.1 Folder and File Naming

- All folders use kebab-case for multi-word names.
- Library files use camelCase (`audio-bell.ts`, `audio-tts.ts`).
- Hook files use kebab-case (`use-audio-unlock.ts`, `use-announcement.ts`); hook exports use camelCase (`useAudioUnlock`, `useAnnouncement`). This matches the convention established by `use-sse.ts` and `use-permission.ts` in earlier sub-phases.
- The bell audio file is at `public/sounds/bell.mp3` (placed by 2.2.3). No new public assets are added.
- The audio libraries live at `src/lib/audio-bell.ts` and `src/lib/audio-tts.ts`.
- The audio hooks live at `src/hooks/use-audio-unlock.ts` and `src/hooks/use-announcement.ts`.

### 6.2 Import Conventions

- All project-internal imports use the `@/` alias.
- The `useSSE` hook is imported from `@/hooks/use-sse` (built in 3.1.2).
- The `SseEventType`, `SseEnvelope`, `SseEventPayload` types are imported from `@/types/sse.types`.
- The `DisplayBoardConfig` type is imported from `@/types/display.types` (built in 3.2.1).

### 6.3 TypeScript Standards

- All exported functions and components have explicit type annotations.
- No `any` types in production code.
- The audio libraries use the browser's built-in `AudioContext`, `AudioBuffer`, `AudioBufferSourceNode`, `GainNode`, `SpeechSynthesis`, `SpeechSynthesisUtterance`, and `SpeechSynthesisVoice` types (from the DOM lib).
- The `AnnouncementJob` type at `src/hooks/use-announcement.ts`: `{ id: string; envelope: SseEnvelope<'TICKET_CALLED' | 'TICKET_RECALLED'>; ticketNumber: string; counterName: string; serviceName: string }` (denormalized for queue processing — the full payload is not needed after enqueue).
- The `UseAnnouncementOptions` type at `src/hooks/use-announcement.ts`: `{ displayBoard: DisplayBoardConfig | null; audioContext: AudioContext | null; isAudioUnlocked: boolean }`.
- The `MAX_ANNOUNCEMENT_QUEUE_SIZE` constant is defined at the top of `use-announcement.ts` as a module-scope `const`. Default value: `5`.

### 6.4 Audio Context Discipline (NEW — Phase 3.3 specific)

The `AudioContext` is a per-page-instance resource. The rules:

- The display board creates ONE `AudioContext` on mount and reuses it for the entire session. The context is held in a `useRef` or state at the `DisplayPageClient` level.
- The context starts in `suspended` state and is resumed on the operator's click (the unlock gesture from 3.2.1's overlay).
- After resumption, the context remains in `running` state for the rest of the session. If it is suspended again (e.g., due to a long tab background), the announcement system attempts to resume it before each playback. The `ensureAudioContextRunning()` helper from 3.3.1 handles this.
- The unlock gesture is one-time per page load. The `quems:audio-unlocked` custom event fires once.
- The audio context unlock overlay is shown **before any SSE event is rendered on the display board AND before any announcement is played**. While audio is locked, the announcement queue accumulates events but does not process them.

These rules are non-negotiable. They are browser security policy, not design preferences.

### 6.5 TTS Voice Caching (NEW — Phase 3.3 specific)

The TTS voice list (`speechSynthesis.getVoices()`) is cached at the module level after the first `voiceschanged` event. The cache is invalidated only on page reload. This avoids repeated synchronous calls to `getVoices()` on every announcement, which can be slow on some browsers.

### 6.6 Announcement Queue Discipline (NEW — Phase 3.3 specific)

The announcement queue is a **single-instance** resource per display board page. The rules:

- The queue is held in React state (`useState<AnnouncementJob[]>`) — not in global state, not in local storage, not in the database. It is purely ephemeral.
- The queue is bounded. The maximum size is `MAX_ANNOUNCEMENT_QUEUE_SIZE` (default 5).
- The processor is a single async function with a single-flight guard (a `useRef<boolean>` flag). Only one announcement plays at a time.
- On unmount, the queue is cleared and the in-progress audio is cancelled. This is a hard cleanup — no announcements continue after the page is gone.

These rules are essential because audio playback that continues after the page is unmounted is a bug, not a feature.

### 6.7 Error Handling Standards

- Audio playback errors are handled gracefully:
  - If `playBell()` throws (e.g., `AudioContext` is suspended), the error is logged via `console.warn` and the processor moves to the TTS stage (or skips to the next item).
  - If `speakAnnouncement()` throws (e.g., `SpeechSynthesisErrorEvent`), the error is logged via `console.warn` and the processor moves to the next item.
  - The queue is not blocked by a single failed announcement.
- `AudioContext.resume()` failures are logged via `console.warn`. The bell system retries once on the next event; if the context is permanently broken, no announcements play (silent failure is preferable to crashing the page).
- TTS `voiceschanged` event handlers that never fire (some browsers never populate voices) fall back to the first available voice after a 3-second timeout. If no voices are available, TTS is skipped silently.

### 6.8 Audit Logging Standard (Sub-Phase 3.3 Specific)

- **No `AuditLog` writes from any 3.3 path.** The audio system is a client-side consumer. Playback is not a user action that needs audit logging.
- **No `TicketEvent` writes from any 3.3 path.** The audio system does not modify ticket state — it only observes events that 3.1.3's `broadcastEvent` already wrote.
- **No `broadcastEvent` calls from any 3.3 path.** The audio system is read-only.

### 6.9 Broadcast Event Seam (Sub-Phase 3.3 Specific)

- 3.3 does NOT emit any events. The audio system is read-only.
- 3.3 consumes events from the `global` channel via `useSSE()`. The event types it filters for:
  - `TICKET_CALLED` — triggers the bell + TTS sequence.
  - `TICKET_RECALLED` — also triggers the bell + TTS sequence (recall re-plays the announcement for the same ticket).
- The audio system does NOT consume `TICKET_NO_SHOW`, `COUNTER_OPENED`, `COUNTER_CLOSED`, `DAILY_RESET`, or `BROADCAST_MESSAGE`. These are visual-only events (the display board consumes them; the audio system does not).

### 6.10 Environment Variable Standards

- No new environment variables are introduced by this sub-phase.
- The bell file path `/sounds/bell.mp3` is a relative URL — it works in both development and production because `public/` is served at the root.

### 6.11 Transaction Boundaries

- No database transactions in this sub-phase. The audio system is entirely client-side.

### 6.12 Commit Standards

- Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`.
- Each task plan document corresponds to one or more commits.
- Suggested commit boundaries:
  - End of 3.3.1: `feat: add bell audio integration with AudioBuffer preload and AudioContext lifecycle`.
  - End of 3.3.2: `feat: add browser TTS integration with template substitution and voice selection`.
  - End of 3.3.3: `feat: add announcement queue and useAnnouncement hook with strict bell-to-TTS sequencing`.

### 8.2 General Guidance

- **Do not skip steps.** Sub-Phase 3.3 is mostly sequential within each document (each step's output is the next step's input). 3.3.1 must be done before 3.3.2 (shared audio context discipline). 3.3.1 and 3.3.2 must be done before 3.3.3 (the queue depends on both).
- **Do not add features beyond the document's scope.** The temptation to "just add per-board bell customization" or "just add a volume slider" is real — resist it. Those are owned by later phases or future enhancements.
- **The audio system is read-only.** It does not call `broadcastEvent()`. It does not write to `AuditLog` or `TicketEvent`. It does not call any mutating API.
- **The audio system is client-side only.** No server endpoints are added in 3.3. The audio is played in the browser using `AudioContext` and `SpeechSynthesis`.
- **Verify after every step.** The verification checklist at the end of each document is a hard gate.
- **When the master plan is ambiguous, choose the simpler option.** If a configuration choice has two reasonable interpretations, pick the one that is easier to change later.
- **Commit at the end of each document** using the commit message specified in the document.
- **Document any deviation.** If the agent makes a choice that differs from this document, it must note the deviation in the final summary so the human reviewer can approve or reject it.
- **Browser quirks are real.** Audio APIs behave differently across browsers. Test in Chrome and Safari at minimum. Mobile browsers may have limited support — design for graceful degradation, not feature parity.
- **The audio context unlock is a browser security policy.** There is no way to bypass it. The overlay from 3.2.1 must be dismissed before any audio plays. This is not a design preference; it is a browser requirement.

### 8.3 What the AI Agent Should NOT Do

- Should not implement FCM push notifications. Phase 4.1.
- Should not implement the officer dashboard. Phase 4.2.3.
- Should not implement broadcast message creation. Phase 4.3.
- Should not implement the reports dashboard. Phase 5.
- Should not add server-side endpoints for audio — the audio is entirely client-side.
- Should not modify the SSE infrastructure (3.1) or the display board (3.2). Read-only consumption.
- Should not add an admin UI for audio settings — the audio settings are already on `DisplayBoard` (3.2.3).

### 8.4 Browser-Specific Notes

- **Chrome (desktop, mobile):** Full support for `AudioContext` and `SpeechSynthesis`. Voices populate synchronously on desktop; asynchronously on mobile.
- **Safari (desktop, mobile):** `SpeechSynthesis` may take a moment to populate voices — the `voiceschanged` event is essential. `AudioContext` works but may suspend more aggressively on tab background.
- **Firefox:** `SpeechSynthesis` support varies by platform — works on most desktop installations, limited on some mobile builds. The system gracefully skips TTS if unavailable (only the bell plays).
- **Edge:** Same as Chrome (Chromium-based).
- **Mobile browsers (any):** Audio may be muted by default if the device is in silent mode. There is no programmatic way to override the device's mute switch — this is a hardware/OS constraint.

### 8.5 The Audio Context Unlock is Mandatory

The `AudioContext` cannot be created in `running` state in modern browsers — it starts in `suspended` state and requires a user gesture (a click, a keypress) to resume. The unlock overlay from 3.2.1 IS this user gesture. The audio system in 3.3.1, 3.3.2, and 3.3.3 MUST NOT play any audio before the operator clicks the overlay. This is enforced by:

- The `useAnnouncement` hook in 3.3.3 checks `isAudioUnlocked` before processing the queue.
- The `useAudioUnlock` hook in 3.3.1 listens for the `quems:audio-unlocked` event and updates state.
- The `AudioUnlockOverlay` component in 3.2.1 dispatches the event on click.

If any of these three are missing or bypassed, the audio system will silently fail (the browser throws "AudioContext was not allowed to start" errors). The agent MUST verify the unlock flow works end-to-end.
