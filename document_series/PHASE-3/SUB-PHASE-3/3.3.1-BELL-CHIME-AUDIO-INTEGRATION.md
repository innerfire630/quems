## 1. Document Purpose

This document delivers the **first half of the audio announcement system**: the bell chime that plays before each TTS announcement. It introduces the Web Audio API integration that loads and plays `bell.mp3` from `public/sounds/`, the `AudioBuffer` preloading strategy that ensures sub-500ms playback latency, the `AudioContext` lifecycle management that cooperates with the unlock overlay from 3.2.1, and the `useAudioUnlock` hook that bridges the visual overlay's click to the audio system's readiness.

When this document is complete:

- The bell sound file at `/sounds/bell.mp3` is fetched and decoded once into a cached `AudioBuffer` on page load.
- After the operator dismisses the audio context unlock overlay (built in 3.2.1), the `useAudioUnlock` hook flips `isAudioUnlocked` to `true`.
- A caller can invoke `playBell()` and receive a Promise that resolves when the bell's `onended` event fires (typically 1-2 seconds later).
- The bell volume respects `DisplayBoard.ttsVolume` (the master volume for all announcement audio).
- If `DisplayBoard.bellEnabled` or `DisplayBoard.announcementEnabled` is `false`, `playBell()` resolves immediately without playing any sound.

This document does **not** implement the TTS announcement (owned by 3.3.2), the announcement queue (owned by 3.3.3), or the visual display (owned by 3.2). The bell system is event-agnostic — it just plays a sound when asked.

### Step 3: Implement `loadBellBuffer()`

**Goal:** Fetch `bell.mp3`, decode it into an `AudioBuffer`, and cache the result.

**Files affected:** `src/lib/audio-bell.ts` (modified).

**Specific instructions:**

- **Body of `loadBellBuffer`:**
  - If `cachedBuffer !== null`, return `cachedBuffer` immediately (cached).
  - If `loadPromise !== null`, return `loadPromise` (in-flight load deduplication).
  - Otherwise, start a new load:
    1. Set `loadPromise = (async () => { ... })()`.
    2. Inside the IIFE:
       - `const response = await fetch(BELL_AUDIO_URL)`. If the response is not OK (status not 200-299), throw an error with the status code.
       - `const arrayBuffer = await response.arrayBuffer()`.
       - `const audioBuffer = await audioContext.decodeAudioData(arrayBuffer)` — this is the Web Audio API's async decoder. Note: in some TypeScript DOM lib versions, `decodeAudioData` is callback-based; the agent uses the Promise-based form (which is supported in all modern browsers).
       - Set `cachedBuffer = audioBuffer`.
       - Return `audioBuffer`.
    3. Wrap in try/catch: on error, log via `console.error('Failed to load bell audio:', error)` and rethrow so the caller knows.
    4. Clear `loadPromise` in a `finally` block (so a failed load can be retried).
  - Return the promise from the IIFE.
- **Logging:**
  - On successful load, log via `console.debug('Bell audio loaded:', { duration: audioBuffer.duration, sampleRate: audioBuffer.sampleRate, numberOfChannels: audioBuffer.numberOfChannels })` — useful for diagnostics.
- **SSR safety:**
  - The function uses `fetch` and `AudioContext` — both browser-only. The function is NOT called server-side; it is called from client components only. No SSR guard is needed in the function body, but callers MUST NOT call it from server components.
- **Caching rationale:**
  - Decoding an MP3 takes 10-50ms (depending on file size and device). Caching the result avoids re-decoding on every bell play.
  - The cache is in-memory only (not localStorage — `AudioBuffer` is not serializable).

Document that this function is called once on page load (in `DisplayPageClient`'s `useEffect`) and the cached buffer is reused for all subsequent bell plays.

### Step 4: Implement `playBell()`

**Goal:** Play the cached bell buffer at the configured volume and return a Promise that resolves on `onended`.

**Files affected:** `src/lib/audio-bell.ts` (modified).

**Specific instructions:**

- **Body of `playBell`:**
  - **Volume clamping:** `const safeVolume = Math.max(0, Math.min(1, volume))` — clamp to [0, 1] per the Web Audio API spec.
  - **Buffer guard:** if `buffer === null` or `buffer === undefined`, log via `console.warn('playBell called with no buffer')` and resolve immediately.
  - **Create source node:** `const source = audioContext.createBufferSource()` and `source.buffer = buffer`.
  - **Create gain node:** `const gainNode = audioContext.createGain()` and `gainNode.gain.value = safeVolume`.
  - **Connect:** `source.connect(gainNode)` and `gainNode.connect(audioContext.destination)`.
  - **Start playback:** `source.start(0)` — start immediately (no offset).
  - **Return a Promise:**
    - Resolve when `source.onended` fires.
    - Reject (or just resolve — the document specifies: resolve) if the source errors out. The agent chooses: resolve on error (don't break the announcement queue) OR reject (let the caller handle). The chosen behavior: **resolve on error** with a `console.warn` log. The bell is non-critical — if it fails, the TTS still plays.
    - Use `new Promise<void>((resolve) => { source.onended = () => resolve(); })`.
    - For error handling, attach `source.onerror` similarly (also resolve).
  - **Cleanup:** the source node is automatically garbage-collected after `onended` fires (the Web Audio API handles this).
- **Pre-condition:** the caller (3.3.3's queue) is responsible for ensuring the `AudioContext` is in `running` state before calling `playBell`. The function does NOT attempt to resume the context — that is the responsibility of `ensureAudioContextRunning()` (Step 6) and the caller's gating logic.
- **Browser quirk:** some browsers fire `onended` slightly before or after the actual audio stops. The function does not need to be precise — the queue's strict sequence is enforced by `await`ing the promise.

Document that `playBell` is a thin wrapper around `AudioBufferSourceNode` with a Promise interface. The caller `await`s the promise to enforce the strict bell → TTS sequence.

### Step 5: Implement `isAudioContextRunning()`

**Goal:** Provide a synchronous helper to check the audio context's state.

**Files affected:** `src/lib/audio-bell.ts` (modified).

**Specific instructions:**

- **Body of `isAudioContextRunning`:**
  - Return `audioContext.state === 'running'`.
- **State values:**
  - `'suspended'` — the context exists but cannot produce audio (initial state, or after a long tab background).
  - `'running'` — the context is active and can produce audio (after unlock).
  - `'closed'` — the context has been closed and cannot be used (should not happen in our flow, but defensively handled).
- **Usage:** the announcement queue (3.3.3) calls this before each playback to decide whether to attempt a resume.

Document that this is a simple state check, but it is exported as a separate function so the caller doesn't have to know the state-string values.

### Step 6: Implement `ensureAudioContextRunning()`

**Goal:** Resume the audio context if it is in `suspended` state.

**Files affected:** `src/lib/audio-bell.ts` (modified).

**Specific instructions:**

- **Body of `ensureAudioContextRunning`:**
  - If `audioContext.state === 'running'`, return immediately (no-op).
  - If `audioContext.state === 'closed'`, log via `console.warn('AudioContext is closed and cannot be resumed')` and return.
  - If `audioContext.state === 'suspended'`, call `await audioContext.resume()`.
    - Wrap in try/catch: on error, log via `console.warn('Failed to resume AudioContext:', error)` and return.
  - After resume, verify the state changed: if `audioContext.state !== 'running'`, log a warning. (Some browsers may not actually resume if the user has not interacted with the page — but in our flow, the unlock overlay click counts as the interaction.)
- **Return:** the function returns `void`. The caller does not need the state — it can call `isAudioContextRunning()` afterward if it cares.
- **Browser quirk:** on some browsers (notably Safari on iOS), the context may suspend again after the tab is backgrounded for a long time. The queue calls this function before each playback to recover.

Document that this is the "self-healing" helper for the audio context. The primary unlock is via the overlay click; this is the fallback for browser auto-suspension.

### Step 7: Create the `useAudioUnlock` Hook

**Goal:** Build the listener for the `quems:audio-unlocked` custom event dispatched by 3.2.1's overlay.

**Files affected:** `src/hooks/use-audio-unlock.ts` (new file).

**Specific instructions:**

Create `src/hooks/use-audio-unlock.ts` containing:

- **Imports:**
  - `useEffect`, `useState` from `react`.
- **Hook signature:**
  - `function useAudioUnlock(): { isAudioUnlocked: boolean }` — no parameters, returns an object with the boolean state.
- **State:**
  - `const [isAudioUnlocked, setIsAudioUnlocked] = useState(false)` — starts as `false` (audio is locked on page load).
- **Effect:**
  - Inside `useEffect`, define the handler:
    - `const handler = () => { setIsAudioUnlocked(true); }`.
  - `window.addEventListener('quems:audio-unlocked', handler)`.
  - Cleanup function (returned from `useEffect`):
    - `window.removeEventListener('quems:audio-unlocked', handler)`.
  - Effect dependencies: `[]` (mount/unmount only — the listener is registered once for the page session).
- **"use client" directive:**
  - The file MUST have `"use client"` at the top. The hook uses `window` and `addEventListener` — browser-only APIs.
- **SSR safety:**
  - The hook is a "use client" hook — it never executes server-side. Even if a server component accidentally imports it, Next.js will throw an error.
- **No re-render storm:**
  - The handler updates state once (on the unlock click). After that, the state never changes again. The component re-renders once on unlock and then is stable.

Document that this hook is the bridge between the visual overlay (3.2.1) and the audio system (3.3.1 / 3.3.3). It MUST be called by `DisplayPageClient` and the result passed to `useAnnouncement` (3.3.3).

### Step 9: Handle Edge Case — Bell File Missing or Invalid

**Goal:** Gracefully handle the case where `bell.mp3` cannot be fetched or decoded.

**Files affected:** `src/lib/audio-bell.ts` (modified).

**Specific instructions:**

- **In `loadBellBuffer`:**
  - If `fetch` throws (network error), the function rejects. The caller (preload effect) catches and logs a warning. The bell does not play, but the TTS still works.
  - If `decodeAudioData` throws (invalid MP3), the function rejects. Same handling.
  - If the response is not OK (e.g., 404), `fetch` does not throw by default. The agent explicitly checks `response.ok` and throws if false.
- **In `playBell`:**
  - If the buffer is `null` or invalid, the function logs a warning and resolves immediately. The TTS sequence proceeds.
- **User-facing behavior:**
  - The display board continues to work visually (the audio is a non-critical addition).
  - The TTS announcement still plays if configured.
  - A `console.warn` message informs the developer of the issue.

Document that the bell is a "nice to have" — the system degrades gracefully if it fails.

### Step 10: Handle Edge Case — AudioContext Auto-Suspended by Browser

**Goal:** Recover from automatic context suspension (e.g., after tab backgrounding).

**Files affected:** `src/lib/audio-bell.ts` (modified, already partially done in Step 6).

**Specific instructions:**

- **The `ensureAudioContextRunning()` function from Step 6 handles this.**
- **Document the policy:** the announcement queue (3.3.3) is responsible for calling `ensureAudioContextRunning()` before each playback. The bell library itself does not auto-recover — it is the caller's responsibility.
- **Why the caller is responsible:** the recovery may require a fresh user gesture (some browsers). If the audio context cannot be recovered, the caller should skip the bell and proceed to TTS (or skip both and log a warning). This decision is owned by the queue, not the bell library.
- **Document in `playBell`'s JSDoc:**
  - "Pre-condition: the caller is responsible for ensuring `audioContext.state === 'running'` before calling `playBell`. If the context is suspended, the bell will not play. Use `ensureAudioContextRunning()` to attempt recovery before calling."

Document that the recovery flow is owned by the queue (3.3.3). The bell library provides the building blocks; the queue orchestrates.

## 8. Common Pitfalls

1. **Creating `AudioContext` in a server component.** `AudioContext` is browser-only. The agent MUST ensure the context is created in a "use client" component. The pattern is: `DisplayPageClient` creates the context, passes it down.
2. **Calling `loadBellBuffer` before the unlock overlay is dismissed.** The audio context starts in `suspended` state. Calling `decodeAudioData` on a suspended context may still work (decoding doesn't require playback), but it is cleaner to load the buffer after unlock. The agent loads the buffer as soon as the audio context is available, regardless of state — decoding works regardless.
3. **Forgetting to cache the `AudioBuffer`.** Without caching, the bell re-fetches and re-decodes on every play, causing unnecessary network traffic and latency spikes. The module-scope cache is essential.
4. **Forgetting to deduplicate concurrent loads.** If two callers call `loadBellBuffer` before the first completes, two fetches happen and the second overwrites the first cache. The `loadPromise` deduplication prevents this.
5. **Using the callback-based `decodeAudioData`.** Modern browsers support both callback and Promise forms. The Promise form is cleaner and what the agent should use.
6. **Creating a new `AudioContext` on every render.** The context is a heavy resource. Create once, reuse forever.
7. **Calling `source.start()` on the same node twice.** The Web Audio API throws if you call `start()` twice on the same `AudioBufferSourceNode`. The agent always creates a new source node per `playBell` call.
8. **Not handling the `closed` state of `AudioContext`.** If the context is closed (e.g., page is being torn down), `resume()` throws. The agent checks `state === 'closed'` and skips.
9. **Forgetting to remove the event listener in `useAudioUnlock`.** The listener leaks if not removed. The cleanup function in `useEffect` is critical.
10. **Auto-suspension by the browser.** Some browsers suspend the audio context when the tab is backgrounded for a long time. The `ensureAudioContextRunning` function is the recovery path — the queue (3.3.3) is responsible for calling it before each playback.
11. **Trying to bypass the unlock with autoplay policy tricks.** Browsers detect and block these attempts. The unlock overlay from 3.2.1 IS the user gesture; there is no programmatic workaround.
12. **Forgetting the "use client" directive.** `useAudioUnlock` uses `window` and `addEventListener`. Without "use client", the build fails or the hook throws at runtime.

## 9. AI Agent Guidance

This section is specifically for AI agents picking up this task plan document.

### 9.1 Critical Implementation Notes

- **The bell file is a pre-existing asset.** `public/sounds/bell.mp3` was placed by 2.2.3. The agent validates it but does NOT generate a new one unless the validation fails (and the agent documents the choice to replace it).
- **The `AudioContext` is created at the `DisplayPageClient` level.** This document does NOT create `AudioContext` directly — the instance is passed in. If the audio context is currently held inside `AudioUnlockOverlay`, the agent refactors to lift the state up to `DisplayPageClient`.
- **The bell library is event-agnostic.** It does NOT subscribe to SSE. It does NOT know about `TICKET_CALLED`. The queue (3.3.3) is responsible for triggering `playBell` in response to events.
- **The `AudioBuffer` cache is in-memory only.** It is reset on page reload. There is no localStorage caching (the buffer is not serializable).
- **The bell is non-critical.** If it fails to load, the system continues with TTS only. The agent does NOT make the bell a hard requirement.
- **The unlock overlay click IS the user gesture.** The `quems:audio-unlocked` event fires once per page load. The audio system waits for this event before playing anything.

### 9.2 What the AI Agent Should NOT Do

- Should not implement TTS. 3.3.2.
- Should not implement the announcement queue. 3.3.3.
- Should not subscribe to SSE in this document. SSE subscription is 3.3.3.
- Should not modify the audio unlock overlay. 3.2.1.
- Should not add new display board fields or schema changes.

### 9.3 Browser Compatibility Notes

- **Chrome / Edge (Chromium):** `AudioContext` works as expected. Auto-suspend on tab background may occur but `resume()` recovers.
- **Firefox:** `AudioContext` works. Auto-suspend is less aggressive than Chrome.
- **Safari (macOS):** `AudioContext` works. Auto-suspend is more aggressive — the `ensureAudioContextRunning` fallback is more important.
- **Safari (iOS):** `AudioContext` works but is heavily restricted. Background tabs are aggressively suspended. The agent tests in Safari if possible.
- **Mobile Chrome / Firefox:** Generally work but may have device-specific quirks (audio output through earpiece vs speaker, mute switch behavior, etc.).
