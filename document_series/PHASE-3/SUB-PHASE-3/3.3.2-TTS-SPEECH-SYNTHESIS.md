## 1. Document Purpose

This document delivers the **second half of the audio announcement system**: the spoken announcement that follows the bell chime. It introduces the browser's native `SpeechSynthesis` API integration, the announcement template substitution (`{number}`, `{counter}`, `{service}` placeholders), the voice selection logic based on the configured BCP-47 language tag, the `SpeechSynthesisUtterance` configuration (rate, pitch, volume), and the `speakAnnouncement()` Promise-returning function that the announcement queue (3.3.3) uses to speak the configured template.

When this document is complete:

- The `speakAnnouncement()` function constructs a `SpeechSynthesisUtterance` from the `DisplayBoard.announcementTemplate` with the correct placeholders substituted.
- The utterance is spoken using the voice that best matches `DisplayBoard.ttsLanguage` (with a graceful fallback if no exact match).
- The utterance's `onend` event fires when speaking completes — `speakAnnouncement()` returns a Promise that resolves on this event.
- The utterance respects `ttsRate`, `ttsPitch`, and `ttsVolume` from the configuration.
- If `DisplayBoard.ttsEnabled` or `DisplayBoard.announcementEnabled` is `false`, `speakAnnouncement()` resolves immediately without speaking.
- Browser quirks (Chrome vs Safari voice loading, mobile TTS limitations) are handled gracefully.

This document does **not** implement the bell (owned by 3.3.1), the announcement queue (owned by 3.3.3), or the SSE subscription (owned by 3.3.3). The TTS system is event-agnostic — it just speaks a string when asked.

## 2. Scope and Boundaries

### 2.1 In Scope

- Creating `src/lib/audio-tts.ts` with the SpeechSynthesis API integration:
  - `substituteTemplate(template, placeholders)` — replaces `{number}`, `{counter}`, `{service}` in the template string.
  - `loadTtsVoices()` — returns a Promise that resolves with the available voices, handling the async voice loading on Safari.
  - `selectVoice(voices, language)` — finds the best matching voice for the configured language.
  - `speakAnnouncement({ text, language, rate, pitch, volume })` — speaks the text and returns a Promise that resolves on `onend`.
  - `isTtsAvailable()` — synchronous check for `speechSynthesis` global.
  - `cancelTts()` — calls `speechSynthesis.cancel()`.
- Template substitution for the three documented placeholders: `{number}` (ticket number), `{counter}` (counter display name), `{service}` (service name).
- Voice selection logic: exact BCP-47 match first, then language-only fallback, then first available voice.
- TTS configuration from `DisplayBoard`: `ttsLanguage`, `ttsRate`, `ttsPitch`, `ttsVolume`, `announcementTemplate`.
- TTS enable flag handling (`ttsEnabled`, `announcementEnabled`).
- Browser quirk handling: `voiceschanged` event for Safari, timeout fallback if voices never load.
- Utterance event handlers: `onstart`, `onend`, `onerror`.

### Step 2: Implement `substituteTemplate()`

**Goal:** Replace `{number}`, `{counter}`, `{service}` placeholders in the announcement template.

**Files affected:** `src/lib/audio-tts.ts` (modified).

**Specific instructions:**

- **Body of `substituteTemplate`:**
  - `let result = template`.
  - `result = result.replace(/\{number\}/g, placeholders.number)`.
  - `result = result.replace(/\{counter\}/g, placeholders.counter)`.
  - `result = result.replace(/\{service\}/g, placeholders.service)`.
  - Return `result`.
- **Why `replace` with global regex:** the same placeholder may appear multiple times (e.g., "Ticket {number}, please proceed to {counter}. Your ticket {number} is ready."). The `g` flag replaces all occurrences.
- **Unknown placeholders:** if the template contains a placeholder like `{foo}` that is not in the three known placeholders, the function leaves it as-is (so the admin can see the typo in the announcement). Document this behavior.
- **Empty values:** if any of `placeholders.number`, `placeholders.counter`, `placeholders.service` is an empty string, the placeholder is replaced with an empty string. The admin should validate the template to avoid awkward announcements like "Now serving ticket at" (missing number).
- **Length cap:** the result string is not capped — TTS engines handle long strings fine. The admin's `announcementTemplate` is capped at 500 chars (per the `displayBoardCreateSchema` Zod validation from 3.2.3).

Document that this is a pure string transformation — no audio is involved.

### Step 3: Implement `loadTtsVoices()`

**Goal:** Return the available TTS voices, handling async voice loading on Safari.

**Files affected:** `src/lib/audio-tts.ts` (modified).

**Specific instructions:**

- **Body of `loadTtsVoices`:**
  - If `cachedVoices !== null`, return `Promise.resolve(cachedVoices)` (cached).
  - If `!isTtsAvailable()`, return `Promise.resolve([])` (TTS not supported).
  - Return a `new Promise<SpeechSynthesisVoice[]>((resolve) => { ... })`:
    - Inside the executor:
      1. Get the current voices: `const initial = speechSynthesis.getVoices()`.
      2. If `initial.length > 0`, set `cachedVoices = initial` and resolve immediately.
      3. Otherwise (voices not loaded yet), set up a `voiceschanged` listener:
         - `const handler = () => { const voices = speechSynthesis.getVoices(); cachedVoices = voices; speechSynthesis.removeEventListener('voiceschanged', handler); resolve(voices); };`.
         - `speechSynthesis.addEventListener('voiceschanged', handler)`.
      4. Set a timeout fallback (3000ms = 3 seconds):
         - `setTimeout(() => { if (!cachedVoices) { cachedVoices = speechSynthesis.getVoices(); speechSynthesis.removeEventListener('voiceschanged', handler); resolve(cachedVoices); } }, 3000)`.
- **Why the timeout:** some browsers (rare, but it happens) never populate voices. The timeout prevents the announcement queue from waiting forever.
- **Logging:** on successful cache population, log via `console.debug('TTS voices loaded:', { count: voices.length })`. On timeout, log via `console.warn('TTS voices failed to load within 3 seconds; falling back to default voice')`.
- **Browser quirk:** Chrome populates voices synchronously; Safari populates asynchronously. The function handles both.
- **Cache invalidation:** the cache is held for the page session. There is no manual invalidation — if the user changes language at runtime (not supported in Phase 3), a page reload is required to refresh the voice list.

Document that this function is called once on page load (in the announcement queue's setup) and the cached voices are reused for all subsequent announcements.

### Step 4: Implement `selectVoice()`

**Goal:** Find the best matching voice for the configured language.

**Files affected:** `src/lib/audio-tts.ts` (modified).

**Specific instructions:**

- **Body of `selectVoice`:**
  - If `voices.length === 0`, return `null` (no voices available).
  - **Exact match:** find the first voice where `voice.lang === language` (exact BCP-47 match). Example: `language = 'en-US'`, `voice.lang = 'en-US'`.
  - **Language-only fallback:** if no exact match, find the first voice where `voice.lang.startsWith(language.split('-')[0])`. Example: `language = 'en-US'`, `voice.lang = 'en-GB'` (both start with `'en'`).
  - **First available fallback:** if no language match, return the first voice in the list.
  - **Log when falling back:** if the function had to fall back to a non-exact match, log via `console.warn('TTS: no exact voice match for language "{language}", falling back to "{voice.lang}"')`.
- **Why language-only fallback:** a user configured `en-US` may still get acceptable speech with `en-GB` (similar enough to understand). Falling back to the first available voice (which might be in a completely different language) would be confusing.
- **Return value:** `SpeechSynthesisVoice | null`. The caller (3.3.3's `speakAnnouncement`) uses the voice if non-null, otherwise lets the browser pick a default.

Document that this is a pure function — no audio is played, no async work, no state mutation.

### Step 5: Implement `speakAnnouncement()`

**Goal:** Speak the configured announcement and return a Promise that resolves on `onend`.

**Files affected:** `src/lib/audio-tts.ts` (modified).

**Specific instructions:**

- **Body of `speakAnnouncement`:**
  - **TTS availability check:** if `!isTtsAvailable()`, log via `console.warn('TTS not available; skipping announcement')` and resolve immediately.
  - **Volume/rate/pitch clamping:**
    - `const safeRate = Math.max(0.1, Math.min(10, options.rate))` — TTS rate range per master plan.
    - `const safePitch = Math.max(0, Math.min(2, options.pitch))` — TTS pitch range per master plan.
    - `const safeVolume = Math.max(0, Math.min(1, options.volume))` — TTS volume range per master plan.
  - **Construct utterance:** `const utterance = new SpeechSynthesisUtterance(options.text)`.
  - **Apply configuration:**
    - `utterance.lang = options.language` (BCP-47 tag like `'en-US'`).
    - `utterance.rate = safeRate`.
    - `utterance.pitch = safePitch`.
    - `utterance.volume = safeVolume`.
  - **Voice selection (best-effort):**
    - Try `selectVoice(speechSynthesis.getVoices(), options.language)` synchronously.
    - If a voice is returned, set `utterance.voice = voice`.
    - Do NOT `await loadTtsVoices()` here — the caller is responsible for pre-loading voices. If voices are not yet loaded, the browser uses its default voice for the configured language.
  - **Event handlers:**
    - `utterance.onstart = () => { options.onStart?.(); }`.
    - `utterance.onend = () => { resolve(); }`.
    - `utterance.onerror = (event) => { options.onError?.(event); resolve(); }` — resolve on error (don't break the queue).
  - **Return Promise:** wrap in `new Promise<void>((resolve) => { ... })`. Resolve on `onend` or `onerror`.
  - **Call `speechSynthesis.speak(utterance)`** at the end (after the handlers are attached).
- **Why resolve on error:** TTS errors are non-critical — the queue should proceed to the next announcement. A `console.warn` log inside `onerror` documents the failure.
- **Browser quirk:** some browsers queue utterances. If two `speak()` calls happen in rapid succession, the second may wait for the first to complete. The single-flight guard in 3.3.3 prevents this by serializing calls. The TTS library itself does NOT serialize — the queue does.
- **No retries:** if `speakAnnouncement` fails, it does not retry. The caller (queue) decides whether to re-queue or skip.

Document that this is the core TTS function. The caller MUST ensure `isTtsAvailable()` returns `true` before calling (or accept the silent skip).

### Step 6: Implement `isTtsAvailable()`

**Goal:** Synchronous check for `speechSynthesis` global.

**Files affected:** `src/lib/audio-tts.ts` (modified).

**Specific instructions:**

- **Body of `isTtsAvailable`:**
  - Return `typeof window !== 'undefined' && 'speechSynthesis' in window`.
- **Why the SSR check:** `window` is not defined server-side. Even though the function is only called from client components, a defensive check prevents runtime errors if a server component accidentally imports the module.
- **Browser coverage:**
  - Chrome / Edge / Opera: full support.
  - Firefox: support varies by platform (works on Windows, macOS, Android; limited on Linux).
  - Safari (macOS, iOS): full support.
- **Behavior when false:** `speakAnnouncement` skips silently. The bell (3.3.1) still plays (no dependency on TTS). The display board still works visually.

Document that this is a cheap synchronous check — call it before any TTS operation to fail fast.

### Step 7: Implement `cancelTts()`

**Goal:** Cancel any in-progress or queued TTS utterances.

**Files affected:** `src/lib/audio-tts.ts` (modified).

**Specific instructions:**

- **Body of `cancelTts`:**
  - If `!isTtsAvailable()`, return immediately.
  - Call `speechSynthesis.cancel()`.
- **Behavior:**
  - Cancels the current utterance (if any).
  - Removes all pending utterances from the queue.
  - The current utterance's `onend` or `onerror` may fire (with an `interrupted` or `canceled` error event).
- **Usage:** called by the announcement queue (3.3.3) on unmount to clean up.
- **No error handling:** `speechSynthesis.cancel()` does not throw in normal usage. Wrap in try/catch defensively.

Document that this is the cleanup function — the queue calls it when the display board unmounts.

### Step 8: Handle Edge Case — `speechSynthesis` Available but Voices Empty

**Goal:** Gracefully handle the case where TTS is supported but no voices are installed.

**Files affected:** `src/lib/audio-tts.ts` (modified, already partially done in Steps 3 and 4).

**Specific instructions:**

- **In `loadTtsVoices`:** if the timeout fires and voices are still empty, return `[]` (empty array). The caller's `selectVoice` returns `null`. The utterance uses the browser's default voice (which may or may not produce sound — depends on platform).
- **In `speakAnnouncement`:** if `selectVoice` returns `null`, do NOT set `utterance.voice` — let the browser pick. Log a debug message: `console.debug('TTS: using browser default voice (no matching voice installed)')`.
- **User-facing behavior:** the announcement may speak in an unexpected language or may not speak at all. The bell still plays. The display board continues to work visually.
- **Documentation:** the admin should ensure the operating system has the configured language installed (e.g., `en-US` voice pack on Windows). This is an OS-level concern, not a software issue.

Document that this is a "best effort" — the system degrades gracefully when voices are unavailable.

### Step 9: Handle Edge Case — Long Text or Special Characters

**Goal:** Ensure the template substitution produces clean text for the TTS engine.

**Files affected:** `src/lib/audio-tts.ts` (modified).

**Specific instructions:**

- **In `substituteTemplate`:**
  - Trim leading/trailing whitespace from the result: `return result.trim()`.
  - Collapse multiple spaces into one: `result = result.replace(/\s+/g, ' ')` (so "ticket   A001   at   counter 2" becomes "ticket A001 at counter 2").
  - Document this normalization in the JSDoc.
- **Special characters:** do NOT escape special characters — the TTS engine handles them natively. The template is plain text.
- **Unicode:** the TTS engine supports the configured language's characters. If the template contains characters not in the language, the TTS engine may skip them or mispronounce them. Document this as an admin concern.

Document that the normalization is purely cosmetic — it makes the spoken output cleaner without changing the meaning.

## 8. Common Pitfalls

1. **Calling `speechSynthesis.speak()` before voices are loaded.** On Safari, `getVoices()` returns an empty array initially; calling `speak()` immediately uses the browser's default voice (which may not match the configured language). The fix: call `loadTtsVoices()` first and `await` it. The queue (3.3.3) is responsible for this.
2. **Forgetting to handle the `voiceschanged` event.** Chrome loads voices synchronously; Safari loads them asynchronously. Without the `voiceschanged` listener, Safari may use the wrong voice.
3. **Setting `utterance.voice` after `speechSynthesis.speak()`.** The voice must be set before calling `speak()`. The agent confirms the order.
4. **Not clamping rate/pitch/volume.** The Web Speech API accepts arbitrary values, but extreme values produce unintelligible speech. Clamping to the documented ranges prevents this.
5. **Resolving on `onerror` instead of rejecting.** Rejecting would break the announcement queue (it would propagate up via the `await` and stop processing). The design is: TTS errors are non-critical — log and continue. The agent implements `resolve()` on error, NOT `reject()`.
6. **Not handling the `interrupted` error event.** When `speechSynthesis.cancel()` is called (during cleanup), the in-progress utterance fires `onerror` with `error = 'interrupted'`. This is expected behavior — the agent does not log this as an error.
7. **Trying to speak while another utterance is in progress.** Some browsers queue utterances, others interrupt. The agent relies on the single-flight guard in 3.3.3 to serialize calls. The TTS library itself does NOT serialize.
8. **Using `SpeechSynthesisUtterance` as a React component prop.** `SpeechSynthesisUtterance` is not a serializable object. The agent creates it inside `speakAnnouncement`, not outside.
9. **Assuming Firefox supports TTS.** Firefox support varies by platform. The `isTtsAvailable` check handles this — if TTS is unavailable, the announcement is skipped silently.
10. **Trusting the configured language without verifying a matching voice exists.** If the admin configures `ttsLanguage = 'fr-FR'` but no French voice is installed, the agent uses the first available voice (which may be in another language). The agent documents this behavior — the admin should install the language pack on the OS.
11. **Using `setTimeout` for the `voiceschanged` fallback without clearing it.** The 3-second timeout fallback sets a `setTimeout`. If voices load just before the timeout fires, the timeout still fires (the cached check prevents double-resolve, but it's wasteful). The agent stores the timeout ID and clears it when voices load.
12. **Forgetting to remove the `voiceschanged` listener.** After voices load, the listener is no longer needed. The agent removes it inside the handler (as documented in Step 3).

## 9. AI Agent Guidance

This section is specifically for AI agents picking up this task plan document.

### 9.1 Critical Implementation Notes

- **The TTS library is event-agnostic.** It does NOT subscribe to SSE. It does NOT know about `TICKET_CALLED`. The queue (3.3.3) is responsible for triggering `speakAnnouncement` in response to events (after the bell has played).
- **The TTS library does NOT own the audio context.** The bell library (3.3.1) owns the `AudioContext`. TTS uses `speechSynthesis` directly, which has its own browser-managed audio routing. The two systems do NOT interfere with each other.
- **TTS is non-critical.** If it fails (no voices, browser doesn't support it, user has muted the device), the system continues with the bell only. The agent does NOT make TTS a hard requirement.
- **The placeholder contract is fixed.** `{number}`, `{counter}`, `{service}` are the only supported placeholders. Adding new placeholders requires extending the `TemplatePlaceholders` interface and updating `substituteTemplate`.
- **The default template is `"Now serving ticket {number} at {counter}"`.** This is per master plan Section 12.3. The admin can change it via the display board configuration UI (3.2.3).

### 9.2 What the AI Agent Should NOT Do

- Should not implement the bell. 3.3.1.
- Should not implement the announcement queue. 3.3.3.
- Should not subscribe to SSE in this document.
- Should not modify the audio context unlock overlay. 3.2.1.
- Should not add new `DisplayBoard` fields.
- Should not call `broadcastEvent` from any path.
- Should not use `SpeechSynthesisUtterance` as a React state value (it's not serializable and may cause render loops).

### 9.3 Browser Compatibility Notes

- **Chrome / Edge / Opera (desktop, mobile):** Full TTS support. Voices populate synchronously. `voiceschanged` is a no-op (voices are already loaded).
- **Safari (macOS):** Full support. Voices populate asynchronously — the `voiceschanged` handler is essential.
- **Safari (iOS):** Full support but with platform restrictions (TTS may be muted if the device is in silent mode). Background tabs are aggressively suspended.
- **Firefox (Windows, macOS, Android):** Generally works. Some Linux distributions lack TTS engines — `isTtsAvailable` returns `false`.
- **Mobile Chrome (Android):** Works with Google TTS engine. The user must have Google TTS installed and configured.

### 9.4 Verification Priority

The agent verifies these critical paths:
- `speakAnnouncement` actually speaks (Step 12, Test 2) — if this fails, nothing else matters.
- Template substitution produces correct output (Step 12, Test 3).
- Enable flags skip cleanly (Step 12, Tests 6 and 7).

The agent does NOT skip these. They are the hard gates.

### 9.5 The TTS Library is Half of the Announcement

The TTS is the second half of the announcement. The bell (3.3.1) is the first half. The queue (3.3.3) chains them together. The agent's job in this document is to ensure TTS works reliably — the queue handles the orchestration. Resist the temptation to add queue logic here.

### 9.6 No "use client" Needed at the Module Level

`src/lib/audio-tts.ts` is a library file, not a React component. It does NOT have a "use client" directive. The "use client" directive applies to the consumer (the announcement queue in 3.3.3 and `DisplayPageClient`). The library is imported by client components; the directive is inherited.
