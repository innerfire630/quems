## 1. Document Purpose

This document delivers the **orchestration layer** of the audio announcement system: the `useAnnouncement()` React hook that ties the SSE event subscription to the bell + TTS playback, the FIFO queue that handles multiple simultaneous ticket calls, the single-flight guard that ensures strict bell → TTS sequencing, the cancellation logic that cleans up on unmount, and the integration into the display board page (`DisplayPageClient` from 3.2.1).

When this document is complete:

- Every `TICKET_CALLED` and `TICKET_RECALLED` event received on the `global` SSE channel triggers a bell chime followed by a spoken announcement.
- Multiple ticket calls arriving close together are queued in FIFO order and played one after another in strict bell → TTS sequence, with no overlap.
- The queue never exceeds its maximum size (default 5). Older events are dropped if the queue is full.
- Navigating away from the display board mid-announcement cancels the in-progress audio cleanly.
- The system respects the configured flags (`announcementEnabled`, `bellEnabled`, `ttsEnabled`) from `DisplayBoard`.
- The system respects the audio context unlock from 3.2.1 — events are buffered while audio is locked and processed when unlocked.

This document does **not** implement the bell (owned by 3.3.1) or the TTS (owned by 3.3.2). This document USES those libraries to orchestrate the announcement sequence.

- **Type definitions (local):**
  - `interface AnnouncementJob { id: string; ticketNumber: string; counterName: string; serviceName: string; enqueuedAt: number; }` — a denormalized queue entry.
    - `id`: a unique identifier (e.g., the event's envelope ID, or a generated UUID).
    - `ticketNumber`: the ticket number (e.g., `'A005'`).
    - `counterName`: the counter display name (e.g., `'Counter 2'`).

- **Hook signature (stub):**
  - `export function useAnnouncement(options: UseAnnouncementOptions): void` — runs side effects only, no return value.
- **Document the design:**
  - The hook is a "manager" hook — it owns state (the queue), effects (the SSE subscription, the processor), and refs (the single-flight guard).
  - It does NOT return any value to the caller. The caller (3.2.1's `DisplayPageClient`) just calls the hook and the audio plays in the background.

### Step 2: Implement the FIFO Queue State

**Goal:** Establish the React state for the announcement queue.

**Files affected:** `src/hooks/use-announcement.ts` (modified).

**Specific instructions:**

- **Inside the hook body:**
  - `const [queue, setQueue] = useState<AnnouncementJob[]>([])`.
- **Enqueue function (defined inside the hook body, memoized with `useCallback`):**
  - `const enqueue = useCallback((job: AnnouncementJob) => { setQueue(prevQueue => { const newQueue = [...prevQueue, job]; if (newQueue.length > MAX_ANNOUNCEMENT_QUEUE_SIZE) { const dropped = newQueue.shift(); console.warn(`Announcement queue overflow: dropping ${dropped?.ticketNumber}`); } return newQueue; }); }, [])`.
  - Note: the function uses the functional form of `setQueue` to read the current queue (avoids stale closure).
- **Queue shift (for the processor to take the first item):**
  - `const dequeue = useCallback((): AnnouncementJob | null => { let next: AnnouncementJob | null = null; setQueue(prevQueue => { if (prevQueue.length === 0) return prevQueue; next = prevQueue[0]; return prevQueue.slice(1); }); return next; }, [])`.
  - Note: the dequeue returns `null` if the queue is empty. The processor checks for `null` and stops.
- **Queue clear (for unmount):**
  - `const clearQueue = useCallback(() => { setQueue([]); }, [])`.

Document that the queue is FIFO. New events are appended; the processor takes from the front.

### Step 3: Implement the Single-Flight Processor

**Goal:** Build the async function that plays bell → TTS in strict sequence, with a single-flight guard.

**Files affected:** `src/hooks/use-announcement.ts` (modified).

**Specific instructions:**

- **Inside the hook body, add refs:**
  - `const isProcessingRef = useRef(false)` — the single-flight guard. `true` while the processor is running.
  - `const processorPromiseRef = useRef<Promise<void> | null>(null)` — the in-flight processor promise (so the caller can `await` it if needed; not currently used externally but useful for testing).
- **Processor function (defined inside the hook body):**
  - `const processQueue = useCallback(async () => { ... }, [queue, ...dependencies...])`.
  - **Body:**
    1. **Single-flight check:** `if (isProcessingRef.current) { return; }` — if already processing, return immediately.
    2. **Mark as processing:** `isProcessingRef.current = true`.
    3. **Wrap in try/finally:** the `finally` block resets `isProcessingRef.current = false` so the next event can trigger processing.
    4. **Loop:**
       - `let job = dequeue()`.
       - `if (job === null) break` — queue is empty, exit the loop.
       - **Ensure audio context is running:** `if (options.audioContext) { await ensureAudioContextRunning(options.audioContext); }`. If the context cannot be resumed, the bell and TTS may not play — log a warning.
       - **Bell stage (only if `bellEnabled && announcementEnabled && options.bellBuffer !== null && isAudioContextRunning(options.audioContext)`):**
         - `try { await playBell({ audioContext: options.audioContext, buffer: options.bellBuffer, volume: options.displayBoard.ttsVolume }); } catch (err) { console.warn('Bell playback failed:', err); }`.
         - Wrap in try/catch — bell errors are non-critical.
       - **TTS stage (only if `ttsEnabled && announcementEnabled && isTtsAvailable() && isAudioContextRunning(options.audioContext)`):**
         - Compute the announcement text: `const text = substituteTemplate(options.displayBoard.announcementTemplate, { number: job.ticketNumber, counter: job.counterName, service: job.serviceName })`.
         - `try { await speakAnnouncement({ text, language: options.displayBoard.ttsLanguage, rate: options.displayBoard.ttsRate, pitch: options.displayBoard.ttsPitch, volume: options.displayBoard.ttsVolume }); } catch (err) { console.warn('TTS playback failed:', err); }`.
         - Note: `speakAnnouncement` already resolves on error (does not reject). The try/catch is for unexpected throws (e.g., if `speechSynthesis` itself is undefined at runtime).
       - **Loop continues** until the queue is empty.
    5. **Reset:** `isProcessingRef.current = false` (in `finally`).
- **The processor does NOT block on `await options.audioContext`** — it checks `options.audioContext` for null and returns if so.
- **No retries:** if a single announcement fails, the loop continues to the next one. No retry, no exponential backoff.

Document that this is the core of the audio system. The processor is the only place that calls `playBell` and `speakAnnouncement`.

### Step 4: Trigger the Processor on Enqueue and on Unlock

**Goal:** Ensure the processor starts whenever a new event is enqueued OR the audio context is unlocked.

**Files affected:** `src/hooks/use-announcement.ts` (modified).

**Specific instructions:**

- **Inside the hook body, add a `useEffect`:**
  - Dependencies: `[isAudioUnlocked, audioContext, bellBuffer, displayBoard, queue]`.
  - **Effect body:**
    - **Gating checks:** if any of the following is false, return (don't process):
      - `isAudioUnlocked === true` (audio must be unlocked).
      - `audioContext !== null` (audio context must exist).
      - `bellBuffer !== null` (bell buffer must be loaded; bell stage is skipped if buffer is null, but the processor still runs).
      - `displayBoard !== null` (display board config must be loaded).
      - `displayBoard.announcementEnabled === true` (master switch).
      - `queue.length > 0` (there's something to process).
    - **Call `processQueue()`:** do NOT `await` it (it's a fire-and-forget side effect; the processor manages its own lifecycle).
- **Why this works:**
  - When a new event is enqueued, the queue length increases, the effect re-runs, and the processor starts.
  - When the audio context is unlocked, the effect re-runs (because `isAudioUnlocked` changed), and any buffered events are processed.
  - When `displayBoard` config is loaded (after the snapshot fetch), the effect re-runs.

Document that the processor is event-driven via React's `useEffect` dependency tracking — there is no manual trigger function.

### Step 5: Subscribe to SSE for `TICKET_CALLED` and `TICKET_RECALLED`

**Goal:** Wire the hook to the SSE event stream for the audio announcement trigger.

**Files affected:** `src/hooks/use-announcement.ts` (modified).

**Specific instructions:**

- **Inside the hook body, use `useSSE`:**
  - `useSSE<'TICKET_CALLED' | 'TICKET_RECALLED'>('global', { filter: ['TICKET_CALLED', 'TICKET_RECALLED'], onEvent: (envelope) => { ... } })`.
  - The hook returns `{ status, reconnectAttempts, lastEventAt, totalEventsReceived }` — these are NOT used by this hook (they are for diagnostics only). The agent may destructure and discard them, or pass them up to a debug component.
- **The `onEvent` callback:**
  - Filter is already applied by `useSSE`, so every envelope here is `TICKET_CALLED` or `TICKET_RECALLED`.
  - Build an `AnnouncementJob`:
    - `const job: AnnouncementJob = { id: envelope.id, ticketNumber: envelope.payload.ticketNumber, counterName: envelope.payload.counterName, serviceName: envelope.payload.serviceName, enqueuedAt: Date.now() }`.
  - Call `enqueue(job)`.
- **No deduplication:** `TICKET_RECALLED` for the same ticket creates a new job. The recall re-plays the announcement (the visual state does not change, but the audio does). The agent does NOT deduplicate — recalls are intentional.
- **No deduplication between `TICKET_CALLED` and subsequent `TICKET_RECALLED` for the same ticket:** the recall SHOULD re-announce. If the admin wants to suppress recalls, that's a future feature.

Document that the SSE subscription is the trigger. The audio system is purely event-driven.

### Step 6: Implement the Cleanup on Unmount

**Goal:** Cancel in-progress audio, clear the queue, and unsubscribe from SSE on unmount.

**Files affected:** `src/hooks/use-announcement.ts` (modified).

**Specific instructions:**

- **Inside the hook body, add a `useEffect` with empty dependencies:**
  - `useEffect(() => { return () => { /* cleanup */ }; }, [])`.
- **Cleanup body:**
  1. `clearQueue()` — empty the queue state. Note: `clearQueue` is wrapped in `useCallback`, so the cleanup can reference it via closure.
  2. `cancelTts()` — cancel any in-progress TTS utterance.
  3. The SSE subscription is cleaned up automatically by `useSSE`'s own cleanup function (the hook closes the `EventSource`).
  4. The processor's in-flight `await` will eventually resolve (the `await` for `playBell` resolves on `onended`; the `await` for `speakAnnouncement` resolves on `onend` or `onerror` after `cancelTts`). The `isProcessingRef.current` flag may remain `true` briefly until the loop iteration completes, but this is harmless because the component is unmounted and the loop will exit naturally.
- **Why no bell cancellation:** the bell uses an `AudioBufferSourceNode`. There is no direct "stop and clean up" API for the source node beyond `source.stop()`, but the source is GC'd after `onended`. Calling `cancelTts()` does NOT cancel the bell. The bell may continue playing for up to its duration (1-2 seconds) after unmount. This is acceptable — the audio is brief.
- **Alternative for strict cancellation (not implemented in Phase 3):** hold a reference to the current `AudioBufferSourceNode` and call `source.stop()` on unmount. This requires the bell library (3.3.1) to expose the source. For Phase 3, the brief post-unmount bell is acceptable.

Document that the cleanup is essential. Without it, the announcement queue persists in memory and audio may continue after navigation.

### Step 7: Handle Audio Context Suspension Mid-Playback

**Goal:** Detect when the audio context is suspended (e.g., tab backgrounded) and skip or abort cleanly.

**Files affected:** `src/hooks/use-announcement.ts` (modified).

**Specific instructions:**

- **Inside the processor loop (Step 3), before each bell and TTS stage:**
  - `if (!isAudioContextRunning(options.audioContext)) { console.warn('Audio context not running; attempting resume'); await ensureAudioContextRunning(options.audioContext); if (!isAudioContextRunning(options.audioContext)) { console.warn('Audio context resume failed; skipping bell + TTS for this announcement'); continue; /* skip to next iteration */ } }`.
- **The skip is silent:** the announcement is dropped (not retried, not buffered). The visual display still updates via the SSE stream from 3.2.1's subscription.
- **Why skip instead of buffer:** the queue is for in-session announcements. If the audio context cannot be resumed (e.g., the user navigated away and back), the system cannot play audio. The next announcement may work (if the context is recovered).

Document that this is a best-effort skip. The visual display is the source of truth for "what was called" — the audio is a courtesy.

### Step 10: Handle the SSE Reconnection Scenario

**Goal:** Ensure the queue continues to work after a network reconnect.

**Files affected:** `src/hooks/use-announcement.ts` (modified).

**Specific instructions:**

## 8. Common Pitfalls

1. **Not awaiting `playBell` before calling `speakAnnouncement`.** Without the `await`, both play simultaneously — the TTS starts while the bell is still ringing. The strict sequence is broken. The agent MUST `await`.
2. **Not using the single-flight guard.** Without it, two concurrent `processQueue` calls can both proceed — overlapping bells and TTS. The `isProcessingRef` is essential.
3. **Forgetting to call `enqueue` from the `onEvent` callback.** If the callback does not enqueue, events are received but never played.
4. **Using `useState`'s non-functional form in the processor.** The processor reads the queue length to decide whether to start. Using `if (queue.length > 0)` directly (without the functional form) reads a stale value. The effect-based trigger handles this, but the processor's dequeue uses the functional form internally.
5. **Cleanup that doesn't call `cancelTts`.** Without `cancelTts`, the TTS utterance continues after unmount — the browser may play audio for a page the user navigated away from. The cleanup function is critical.
6. **Cleanup that doesn't clear the queue.** Without clearing, the next mount may process stale events.
7. **Blocking the SSE subscription with a slow processor.** If the processor takes 30 seconds to drain a queue (e.g., many events during a network reconnect), the SSE subscription may back up. The `useSSE` hook handles this correctly (it continues receiving events while the processor drains), but the agent should be aware.
8. **Not handling the `audioContext === null` case.** If the audio context is not yet created (e.g., before the overlay is mounted), the processor crashes on `audioContext.state`. The agent guards with `if (audioContext === null) return;`.
9. **Playing audio before the user gesture.** If the processor runs while `isAudioUnlocked === false`, the bell and TTS may be blocked by the browser's autoplay policy. The gating `useEffect` prevents this — the processor does not start until unlocked.
10. **Queue overflow silent.** If the queue overflows without a `console.warn`, the admin has no way to know that events are being dropped. The `console.warn` is essential for diagnostics.
11. **Recall deduplication.** The agent might think recalls should be deduplicated (the same ticket is called again). They should NOT be — recalls are intentional re-announcements. The audio system plays them.
12. **Using `useEffect` with `[queue.length]` as the only dependency.** The effect must also re-run when `isAudioUnlocked`, `audioContext`, `bellBuffer`, or `displayBoard` change — not just when the queue changes. The agent includes all in the dependency array.
13. **Not handling `displayBoard === null` in the processor.** If the processor runs while the display board is null (initial state), it tries to read `displayBoard.announcementTemplate` and crashes. The guard is essential.
14. **Trying to play bell + TTS in parallel for "speed".** The master plan explicitly forbids this — the bell must complete before TTS starts. Parallelism is a bug, not an optimization.

## 9. AI Agent Guidance

This section is specifically for AI agents picking up this task plan document.

### 9.1 Critical Implementation Notes

- **The strict sequence is non-negotiable.** Bell completes → TTS starts. Bell ends → TTS ends → next bell starts. No parallelism, no overlap. The `await` in the processor is the enforcement mechanism.
- **The single-flight guard is non-negotiable.** Without it, two concurrent processors both dequeue, both play — overlapping audio. The `isProcessingRef` is the only protection.
- **The audio context unlock is the gate.** The processor MUST NOT run while `isAudioUnlocked === false`. The gating `useEffect` enforces this. Bypassing it (e.g., by calling `playBell` directly from the `onEvent` callback) breaks the browser's autoplay policy.
- **The cleanup is non-negotiable.** Without `cancelTts()` on unmount, audio continues after the user navigates away. The `useEffect` cleanup is the only protection.
- **The queue is bounded.** `MAX_ANNOUNCEMENT_QUEUE_SIZE = 5` is the upper limit. Larger queues cause audio backlogs that confuse the waiting area. The drop-oldest behavior is intentional.
- **No priority promotion.** Events are strict FIFO. A `TICKET_CALLED` from 5 minutes ago is NOT promoted to the front of the queue. Phase 3 does not implement priority — keep it simple.

### 9.2 What the AI Agent Should NOT Do

- Should not implement the bell library. 3.3.1.

### 9.3 The Hook is a Manager, Not a Component

The `useAnnouncement` hook is a manager — it owns state, refs, and effects, but it does NOT render anything. The caller (`DisplayPageClient`) just calls the hook and the audio plays in the background. The hook returns `void`.

This is different from typical React hooks (which return values). The agent should not be surprised — it's a deliberate design choice for side-effect-only logic.
